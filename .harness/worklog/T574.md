## What I did

- `src/workflows/plex-language-fix/contracts.ts`: replaced `plexLanguageEvaluateContract`'s trivial
  always-`ok:true` check with a real pre-mutation assertion. It reads the `plex-language-evaluate`
  ledger read-only via `ledgerSuccessRows` (from `stages/ledger.ts`, already used by `apply.ts`) and
  asserts every row with `detail.status === 'change'` has a numeric `detail.proposedAudio.streamId`
  AND a non-null `detail.currentAudio` — the exact malformation `apply.ts` (~line 54) currently skips
  silently at runtime. An empty ledger or an all-`'skip'` ledger passes trivially. On violation, the
  offending `itemKey`(s) are listed in the failing `checks[].actual`, the top-level `violations[]`,
  and the `sample`, so the gate detail page names exactly which row is malformed. Kept the existing
  `fromChecks(checks[], sample)` shape. Updated the contract's `shape.expectations` label/detail and
  top-level `description` to describe the real bar instead of the old "ledger is readable" minimum.
  `plexLanguageDiscoverContract`/`plexLanguageResolveContract` are unchanged (still the sanctioned
  trivial gates).
- `src/workflows/plex-language-fix/lib.test.ts`: added 5 tests — empty ledger passes, a
  valid change+skip mix passes, a `'change'` row missing `proposedAudio.streamId` fails and names the
  itemKey, and a `'change'` row with a null `currentAudio` fails and names the itemKey. Each test
  writes to the ledger via `markWorkItem` with a unique `randomUUID()`-suffixed key (test files each
  run in their own process against a fresh per-PID scratch DB, so no cross-file collision).
- `src/workflows/plex-language-fix/CLAUDE.md`: added a new section documenting that the evaluate→apply
  gate is now a real pre-mutation check (not the trivial minimum), what it asserts, and why (mirrors
  the exact malformation `apply.ts` already guards against at runtime).
