# T298 — retire DynamoDB from workouts-sync, local JSON history accumulator

## What changed

- `src/jobs/workouts-sync/stages/hevy-sync.ts`: removed `dynamoPut` import + the
  `writeWorkoutToDynamo`/`DynamoPutter` DynamoDB write path entirely. Added a local,
  full-history JSON accumulator: `readWorkoutsHistory(path)` / `appendWorkoutsHistory(path,
  existing, newWorkouts)`, defaulting to `data/out/workouts-history.json` (job-local,
  gitignored via the existing `src/jobs/**/data/` rule). `runHevySync` now takes
  `{ fetchPage?, historyPath? }` (dropped `putItem`/`workoutsTable`/`exercisesTable`),
  determines "new" the same way as before (`isWorkItemDone`/`markWorkItem` keyed by workout
  id), and appends only the genuinely new workouts each run.
- `src/jobs/workouts-sync/stages/hevy-sync.job.ts`: updated description (no more DynamoDB
  wording).
- `src/jobs/workouts-sync/workouts-sync.workflow.ts`: schedule changed from `'0 6 * * *'`
  (daily) to `'0 6 1 * *'` (monthly, 1st at 06:00); doc comment + description updated to
  describe the local JSON accumulator and note the planned T299 analysis stage.
- `.env.example`: removed `WORKOUTS_TABLE`/`EXERCISES_TABLE` (confirmed via grep — nothing
  else in the codebase references them after this change) and reworded the AWS-creds
  section header since it's no longer tied to a specific workflow.
- `CLAUDE.md` / `README.md`: updated the `workouts-sync` paragraph/bullet to describe the
  local-JSON-history behavior, monthly schedule, and the still-unbuilt T299 follow-up.
