## What I did
Fixed `plex-language-fix`'s self-referential `inputKeys()` trap: `discoverInputKeys()` in
`src/workflows/plex-language-fix/stages/discover.ts` used to call `ledgerSuccessRows(JOB_NAME)`
ŌĆö a read-back of this job's own prior `work_items` success rows. After "Clear output data"
wipes the ledger, that call returned `[]`, so a manually-limited run selected zero roots and
silently no-op'd instead of re-discovering the library.

Replaced it with a LIVE Plex library walk:
- Extracted the section/item/leaf enumeration logic that used to live inline in `runDiscover`
  into a shared `walkLibraryFiles(opts, hooks)` helper. `runDiscover` now calls it with hooks
  that reproduce its original `ctx.log`/`ctx.progress` narration exactly, then iterates the
  returned entries to do the ledger-write/rootAllowed/isWorkItemDone logic it always did.
- `discoverInputKeys(opts)` is now `async`, calling the same `walkLibraryFiles` with no hooks
  and mapping the returned entries to file keys.
- Both paths go through `lib.ts`'s `fetchSections`/`fetchSectionItems`/`fetchItemDetail`/
  `fetchAllLeaves`, which already route every Plex read through `callService('plex', ...)`
  (confirmed complete repo-wide by T578) ŌĆö so no direct `plexGet` calls were added here.
- Removed the now-unused `ledgerSuccessRows` import from `discover.ts` (still used by
  `resolve.ts`/`evaluate.ts`/`apply.ts`/`contracts.ts` ŌĆö `ledger.ts` itself is untouched).
