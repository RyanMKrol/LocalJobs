# T017 — Manually dismiss stuck items

## Goal
Add a MANUAL-ONLY way to permanently dismiss a stuck work item (genuinely bad
data that will never process) so it drops off the dashboard front page / stuck
list. Must be distinct from "unstick" (which retries) and never automatic.

## Design
- New `work_items.status` value: `dismissed` (no DDL needed — `status` is a free
  TEXT column; updated the schema comment).
- `dismissWorkItem(jobName, itemKey)` in `store.ts`: `UPDATE … SET
  status='dismissed' … WHERE status='failed'`. The exact opposite of
  `unstickWorkItem`, which `DELETE`s a failed row to retry it. Dismiss *keeps* the
  row but parks it, so it neither retries nor shows as stuck.
- `stuckItems`/`stuckCount` already filter `status='failed'`, so a dismissed item
  drops off both automatically — no change needed there.
- `isWorkItemDone` now treats `dismissed` (like `success`) as done → never
  reprocessed on a re-run.
- API: `POST /api/stuck/dismiss { job, key }` → `{ ok, dismissed }`.
- Dashboard: `api.dismiss` + a `✕ Dismiss` button next to `↻ Unstick` on the
  Overview stuck-items table, behind a `window.confirm` (it's permanent).

## Manual-only guarantee
Nothing in the run/schedule path calls `dismissWorkItem`; it's invoked solely
from the dashboard control via the API. `dismiss` only acts on a currently-failed
row, so it can't park a success.

## Verification (observed)
- `npx tsc --noEmit` → clean.
- `npm test` → all green, incl. new `dismissWorkItem` block in
  `src/db/store.test.ts`:
  - dismiss updates the row to `dismissed` (returns 1);
  - dismissed item leaves `stuckItems()` and drops `stuckCount` 2→1;
  - `isWorkItemDone(...,4)` returns true for a dismissed item;
  - no-op on a success / already-dismissed / unknown key (returns 0);
  - distinct from unstick: `unstickWorkItem` won't delete a dismissed row, but a
    still-stuck sibling can be unstuck.
- `npm --prefix dashboard run build` → succeeds.
