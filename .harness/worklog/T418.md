# T418 — tmdb-season-check / plex-seasons-notify: fail the run on genuine per-item failure

## What changed

- `src/workflows/missing-tv-seasons/stages/season-check.ts` (`tmdb-season-check`): added a `failed`
  counter, incremented it in the catch block's non-quota branch (the `show.tmdbId == null`
  "unverifiable" branch and the `QuotaExceededError` soft-stop branch are both excluded, as
  specified). At the end of `runSeasonCheck`, after writing `missing-seasons.json` and logging the
  summary, throws `` `${failed}/${shows.length - unverifiable.length} show(s) failed their TMDB
  check this run — see logs above` `` when `failed > 0`.
- `src/workflows/missing-tv-seasons/stages/notify.ts` (`plex-seasons-notify`): when the digest push
  returns `{ ok: false }`, logs an error line and throws `Digest push failed — <error>` BEFORE
  marking any `(tmdbId, season)` pair notified — so an undelivered digest no longer gets silently
  recorded as delivered and the affected seasons are retried on the next run.

## Tests

- New `src/workflows/missing-tv-seasons/stages/season-check.test.ts` covers: one genuine per-show
  TMDB failure alongside a successful show → `runSeasonCheck` rejects naming the failed count, and
  `missing-seasons.json` still captured the successful show; all-success case resolves normally;
  an unverifiable (no `tmdbId`) show alongside a successful one resolves normally (not counted as
  failed); a service-quota-exhausted first call (`QuotaExceededError`, forced via
  `syncService({ name: 'tmdb', dailyCap: 0 })`) resolves normally (soft-stop, not a failure).
- Extended `src/workflows/missing-tv-seasons/stages/notify.test.ts` with a case where the injected
  `push` returns `{ ok: false }`: `runNotify` rejects, and the run's newly-detected pair is NOT
  marked done in the ledger (`isWorkItemDone` still `false`).

### A process-freezing gotcha in `season-check.test.ts`

`src/core/plex-client.ts` reads `TMDB_API_TOKEN` into a module-level `const` at import time. In the
full `npm test` run, `src/core/plex-client.test.ts` (sorted before this workflow's tests) imports
that module first with no token set, freezing the const empty for the rest of the single-process
test run — no `process.env.TMDB_API_TOKEN = …` inside a later-loaded test file can undo that
per-process freeze once it's happened. Since `plex-client.ts` is out of this task's scope (only
`season-check.ts`/`notify.ts` + tests), the test file works around it by re-invoking itself in a
freshly-spawned child process (`execFileSync` with `TMDB_API_TOKEN` set on `env` before the process
— and therefore the module — exists at all). The child still stubs `global.fetch` itself and never
makes a live TMDB/Plex call — fully hermetic, just isolated from the parent test process's already-
frozen module state.
