# T169 — Editable workflow maxConcurrency (like schedule/enabled)

Made a workflow's bounded-parallelism cap user-editable from the **Workflows →
[workflow]** detail page, mirroring the T135 schedule mechanism end-to-end:
user-owned, override-flagged, code-reconciled.

## Changes
1. **Schema/migration (T098 rule):** added `max_concurrency` (INTEGER, NULL = use
   code default) + `max_concurrency_overridden` (INTEGER NOT NULL DEFAULT 0) to the
   `workflows` table in `src/db/schema.sql` (fresh DBs) and as additive `ALTER TABLE`
   migrations in `src/db/index.ts` (existing DBs). No index on the new columns → no
   bootstrap-index trap.
2. **Store:** `upsertWorkflowStmt` now seeds `max_concurrency` from the manifest and
   preserves an override on re-sync via
   `CASE WHEN max_concurrency_overridden = 1 THEN max_concurrency ELSE excluded.max_concurrency END`
   (exactly the schedule pattern). `syncWorkflow` passes `def.maxConcurrency ?? null`.
   Added `updateWorkflowConcurrency(name, n)` (sets value + flips override; throws on
   a non-integer / `< 1` as a defensive backstop). `WorkflowRow` gained both columns.
3. **Executor:** added exported `DEFAULT_WORKFLOW_CONCURRENCY` + helper
   `effectiveWorkflowConcurrency(def)` = DB `max_concurrency` ?? `def.maxConcurrency`
   ?? default. `runWorkflow` reads it FRESH each run (was `def.maxConcurrency ?? DEFAULT`),
   so an edit takes effect next run with no restart.
4. **API:** `POST /api/workflows/:name/concurrency { maxConcurrency }` — validates
   positive integer ≥ 1 server-side (400 otherwise), 404 unknown workflow, behind the
   existing mutation guard. `GET /api/workflows/:name` now returns
   `effective_max_concurrency`. Added `api.updateWorkflowConcurrency` client method +
   `Workflow` type fields in `dashboard/app/lib/api.ts`.
5. **Dashboard:** added a **Max concurrency** row to the workflow detail page with the
   same inline Edit → number input → Save affordance as Schedule (shows effective value
   + a "(default)" hint when not overridden, inline error on invalid). 3s poll refreshes.
6. **Docs:** updated CLAUDE.md (new editable-property convention note + T156 paragraph)
   and README (workflows section + dashboard map).

## Tests added
- `src/db/store.test.ts`: `updateWorkflowConcurrency` persists + flips override; a
  code-sync PRESERVES an overridden value but REFRESHES a non-overridden one; rejects
  0 / negative / non-integer; unknown workflow → undefined.
- `src/api/server.test.ts`: valid value 200 + persisted + overridden + surfaced on GET;
  0 / -1 / 1.5 / "x" → 400; unknown workflow → 404.
- `src/db/migrate-existing-db.test.ts`: existing-DB guard asserts both new columns are
  added with NULL / 0 defaults.
