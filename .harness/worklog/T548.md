# T548 — Show the daemon-down message on the five detail pages instead of a blank skeleton

2026-07-14

## What I did

Five dashboard detail pages destructured only `data` from `usePoll` and discarded `error`,
so when the daemon API is unreachable they rendered a permanently-blank skeleton instead of
the `⚠ Cannot reach the daemon API ({error}).` message the list pages (workflows, integrations,
logs) already show. Fixed all five by destructuring `error` and rendering the identical
markup/class (`<p className="muted">⚠ Cannot reach the daemon API ({error}).</p>`) right below
each page's back-link:

- `dashboard/app/workflows/[name]/page.tsx` — the `usePoll(() => api.workflow(name), ...)` call.
- `dashboard/app/workflow-runs/[id]/page.tsx` — both polls (`api.workflowRun(id)` and the
  workflow-definition poll `api.workflow(run?.workflow_name ...)`); the message shows if
  either errors (`error || pError`).
- `dashboard/app/jobs/[name]/page.tsx` — the `api.job(name)` poll (the run-history and stuck-item
  polls are left alone — the task's own spec singles out the primary job-detail fetch).
- `dashboard/app/workflow-runs/[id]/gates/[producer]/[key]/page.tsx` — the `api.workflowRun(id)`
  poll (the gate inspection poll is unaffected, matching the pattern above).
- `dashboard/app/workflows/[name]/gates/[producer]/[key]/page.tsx` — the `api.workflowGate(...)`
  poll.

## Fixtures / living artifact

Extended `dashboard/scripts/_dashboard-harness.mjs` with 5 new FLOWS entries
(`daemon-down-workflow-detail`, `daemon-down-workflow-run-detail`, `daemon-down-job-detail`,
`daemon-down-gate-run-scoped`, `daemon-down-gate-definition-scoped`) plus a new exported helper
`failDaemonFetch(page, pathPrefix)`.

Note on approach: my first attempt used `page.route(...).fulfill({status: 500})`, which is the
existing pattern for the `tv-recs-empty` flow. That works functionally, but Chromium's own
Network-domain integration logs a `"Failed to load resource: the server responded with a status
of 500"` console error for ANY real HTTP 500 response — including one served via
`route.fulfill()` — and `visual-check.mjs` treats every console error as a hard failure,
regardless of whether the page itself rendered correctly. Confirmed this by running
`visual-check.mjs` with the route-based approach: it produced the correct screenshot (the
message rendered) but still reported `daemon-down-*` as FAILed because of the console error.

Since `visual-check.mjs` is out of this task's scope (can't relax its console-error check),
`failDaemonFetch` instead monkey-patches `window.fetch` in-page via `page.addInitScript` before
the reload, so the failing response for the targeted path is built purely in JS
(`new Response(...)`) and never touches the browser's real network stack — no resource load, no
console error, same `usePoll` error-path behavior (`!res.ok` → `throw` → `setError`).
