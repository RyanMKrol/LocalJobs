# T290 — stocks-watch second stage

## What was built

- `src/jobs/stocks-sync/stages/stocks-watch.ts` — reads `data/out/portfolio.json`,
  computes `gainPct` per position, and implements the "re-scan + notification-log"
  idempotency pattern via the `work_items` ledger keyed by ticker under job name
  `stocks-watch`:
  - fresh breach (`gainPct >= 30`, not yet notified) → included in this run's digest,
    ledger row marked `success`.
  - sustained breach (already notified) → skipped, no re-notify.
  - drop back below 30% after being notified → ledger row reset to `skipped`
    (`isWorkItemDone` treats `skipped` as NOT done — the framework's existing
    "retry when quota resets" semantics repurposed here as "re-arm the alert"),
    so a later re-cross notifies again.
  - all fresh breaches in one run are combined into exactly ONE `push()` call
    (title + one line per ticker with sign/gain/prices), never one push per ticker.
- `src/jobs/stocks-sync/stages/stocks-watch.job.ts` — job wrapper, no `inputKeys`
  (it re-scans the whole current portfolio every run, same shape as the
  missing-tv-seasons notify stage).
- `src/jobs/stocks-sync/stages/stocks-watch.test.ts` — covers all 5 Done-when
  scenarios (a–e) with an injected push capture fn + synthetic portfolio.json,
  hermetic (scratch DB via `npm test`'s `LOCALJOBS_DB`).
- `stocks-sync.workflow.ts` — added `{ job: 'stocks-watch', dependsOn:
  ['stocks-snapshot'] }` to the `jobs` list (same `dependsOn` shape as
  `places.workflow.ts` / `tv-recs.workflow.ts`).
- Updated the `stocks-sync` paragraph in root `CLAUDE.md` and the `stocks-sync`
  bullet in `README.md` to describe the 2-stage DAG and the notify-once-per-
  breach-episode / reset-on-drop-below-threshold behaviour.
