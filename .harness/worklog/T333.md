# T333 — fix non-scrolling Recommendations panel (`.movie-gaps-scroll`)

## Root cause (independently re-verified, not just trusted from the idea note)

`.movie-gaps-scroll` (`dashboard/app/globals.css`) is a flex column with `max-height: 520px;
overflow-y: auto;`. Its child `.panel` has `overflow: hidden` (for border-radius clipping). Per
the flexbox spec, a flex item's automatic minimum size collapses to 0 when the item's own
`overflow` isn't `visible` — so `.panel` had no content-based minimum and the flex algorithm
shrank it to fit inside 520px instead of growing past it. Net effect: `.movie-gaps-scroll`'s own
`scrollHeight === clientHeight` always, so there was never anything to scroll — matching the
report (wheel does nothing, rows get cut off).

Confirmed empirically by driving the dashboard with the new `movie-recs-scrolled` /
`tv-recs-scrolled` FLOWS entries in `visual-check.mjs`:
- With `flex-shrink: 0` REMOVED (bug reproduced): both new flows failed with
  `page.waitForFunction: Timeout 30000ms exceeded` — `scrollTop` never moved after a real
  `page.mouse.wheel()` interaction.
- With the fix restored: both flows passed, and the `movie-recs-scrolled.png` screenshot visibly
  shows DIFFERENT rows than the baseline `workflow-movie-recs.png` screenshot (later films by
  TMDB rating scrolled into view), proving genuine scroll movement, not just a passing CSS
  property.

## Fix

`dashboard/app/globals.css`: added `flex-shrink: 0;` to the `.movie-gaps-scroll .panel` rule, with
a comment explaining why it's load-bearing (the overflow:hidden / flex auto-min-size interaction).

Corrected scope per the task note: `.movie-gaps-scroll` is used ONLY by `MovieRecsManager` and
`TvRecsManager` (`dashboard/app/workflows/[name]/page.tsx`) — confirmed via
`grep -n "movie-gaps-scroll" dashboard/app/workflows/\[name\]/page.tsx`, exactly 2 matches.
`MovieGapsManager` (franchise gaps) and `MissingSeasonsManager` (missing TV seasons) render their
tables in a bare `.panel` with no `.movie-gaps-scroll` wrapper — left untouched, no scroll
behaviour added to them.

## Fixture changes (`dashboard/scripts/_dashboard-harness.mjs`)

- `movieRecs.recommendations` bumped from 4 active + 1 ignored to 15 active + 1 ignored rows
  (real-world magnitude per the task note) — the previous 4-row fixture never overflowed 520px, so
  the bug (and any fix) was untestable through it.
- `tvRecs.recommendations` bumped similarly, from 3 active + 1 ignored to 15 active + 1 ignored.
- Two new `FLOWS` entries: `movie-recs-scrolled` (`/workflows/movie-recommendations`) and
  `tv-recs-scrolled` (`/workflows/tv-recommendations`). Each: waits for
  `.movie-gaps-scroll table`, calls `scrollIntoViewIfNeeded()` THEN reads `boundingBox()` (in that
  order — doing it the other way round leaves `mouse.move()` targeting stale/off-viewport
  coordinates), moves the mouse to the container's centre, drives a real
  `page.mouse.wheel(0, 400)`, and asserts via `page.waitForFunction(() =>
  .movie-gaps-scroll.scrollTop > 0, {timeout: 5000})` — mirroring the existing
  `overview-theme-toggle` flow's assertion-via-waitForFunction pattern. This throws/fails hard on
  regression, exactly as required.
