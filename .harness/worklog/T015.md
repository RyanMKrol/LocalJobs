# T015 worklog — Framework: shared headless-browser helper (src/core/browser.ts)

## Attempt 1 — 2026-06-21

### What was done (behaviour-preserving extraction)

The perfumes fetch (`src/jobs/perfumes/fetch.ts`) had an inline `launchContext`
that hard-coded the Cloudflare-beating launch recipe: persistent on-disk profile,
real-Chrome channel with a bundled-chromium fallback, anti-automation flag +
realistic UA/viewport/locale, and stale-`Singleton*`-lock cleanup. Promoted that
verbatim into a reusable helper.

- **New `src/core/browser.ts`:**
  - `launchPersistentBrowser(opts, launch?)` — the extracted launch. Same defaults
    as the old inline code (headless, UA `Chrome/126.0…`, viewport 1280×1800,
    locale `en-GB`, `--disable-blink-features=AutomationControlled`, channel pass +
    fallback, lock cleanup). Takes an **injectable launcher** (defaults to
    `chromium.launchPersistentContext`) so it's testable with zero real browser.
  - Exported `DESKTOP_CHROME_UA` and `PROFILE_LOCK_FILES` constants.
  - `jitteredDelayMs(base, maxJitter, rng?)` — the jittered-pacing learning, with
    an injectable rng for deterministic tests. (Perfumes paces via the
    `fragrantica` service, so this is for jobs that pace their own loop.)
- **`src/jobs/perfumes/fetch.ts`:** `launchContext` now delegates to
  `launchPersistentBrowser({ profileDir, headless, channel, log })`. Dropped the
  now-unused `rmSync` / `chromium` imports. No behaviour change — same options,
  same fallback, same `[fetch] …` log prefix on the fallback notice.
- **`src/jobs/perfumes/fragrantica.service.ts`:** doc-comment only — clarifies it
  is the *pacing* half and `core/browser` is the *launch* half of the strategy.
- **New `src/core/browser.test.ts`:** 7 tests via the injectable launcher — option
  defaults, override + extra-arg merge, channel pass-through, `Singleton*` lock
  clearing (real tmp dir), channel→bundled-chromium fallback (first call throws),
  error propagation when no channel, and `jitteredDelayMs` bounds/rounding/clamp.
