# T525 — guard three unhandled-failure channels in scheduler/daemon/executor

Date: 2026-07-13

## What I did

Three independent guards, as specced:

1. **`src/core/scheduler.ts`** — `scheduleWorkflow`'s cron callback body is now wrapped
   in a `try/catch`. croner's default `catch:false` means a rejecting callback would
   otherwise surface as an `unhandledRejection`; now a failure is logged loudly via
   `console.error('[scheduler] workflow <name> fire failed:', err)` and swallowed, so
   one workflow's fire failure can never take down the scheduler/daemon.
   `rescheduleWorkflow` reuses the same `scheduleWorkflow` callback, so both the
   initial and re-registered crons are covered by one fix.

   Added a test-only injection seam (`_setRunWorkflowForTest`, mirroring the existing
   `_setOnSettleThrowHookForTest` / `_setFetchForTest` patterns already in this repo)
   and a test-only manual-fire helper (`_triggerWorkflowCronForTest`, wrapping
   croner's own public `.trigger()`) so a unit test can force a rejection through the
   real registered cron callback without spawning a real workflow run.

2. **`src/daemon.ts`** — `main()` now installs `process.on('unhandledRejection', ...)`
   and `process.on('uncaughtException', ...)` as a last-resort safety net, alongside
   the existing SIGTERM/SIGINT handlers. Both log loudly with a `[daemon] …` prefix
   and the error/stack. Deliberate, documented policy: `unhandledRejection` logs and
   CONTINUES (usually a localized async fault — one bad promise shouldn't kill
   everything); `uncaughtException` logs and `process.exit(1)`s (a synchronous throw
   escaped all handling, so the process may be in a corrupt state — better to let
   launchd restart a clean daemon than limp on).

3. **`src/core/executor.ts`** — the `rl.on('line')` (stdout NDJSON) and
   `errRl.on('line')` (stderr) handlers now wrap their store-touching body
   (`addLog`/`setProgress`) in a `try/catch`. A synchronous store throw (e.g.
   SQLITE_BUSY under contention) now drops at most that one line — logged via a
   `console.error` fallback that is itself wrapped in a `try/catch` so it can never
   throw out of the handler — instead of crashing the daemon. The child process and
   the rest of the run are unaffected.

## Tests added

- `src/core/scheduler.test.ts`: new block asserts that a rejecting `runWorkflow`
  (injected via `_setRunWorkflowForTest`) triggered through the REAL registered cron
  callback (`_triggerWorkflowCronForTest`, using croner's own `.trigger()`) resolves
  without throwing, logs via `console.error` (name + message asserted), and leaves a
  different workflow's cron registration unaffected.
- `src/core/executor.test.ts`: the fake child script gained a `badlog*` branch that
  emits a `log` event whose `message` is a plain object — better-sqlite3 can only
  bind string/number/bigint/buffer/null, so `addLog`'s bind throws synchronously
  inside the line handler. New test asserts the run still completes `success`, the
  very next log line is still recorded (proving only the one bad line was dropped),
  and a fallback `console.error` fired.

## Manual confirmation (daemon.ts global handlers)

Per the task's own guidance, this is asserted by inspection rather than a spawned
real daemon (spawning `main()` would require binding real ports / opening the real
scheduler against the registry, out of proportion for a unit test):

- `process.on('unhandledRejection', (reason) => { console.error('[daemon] unhandled rejection:', reason, ...); })`
  — logs and returns; no `process.exit`, so the daemon keeps running.
- `process.on('uncaughtException', (err) => { console.error('[daemon] uncaught exception — exiting:', err, ...); process.exit(1); })`
  — logs and exits 1, so launchd restarts a clean process.

Both have an inline comment directly above documenting the exit-vs-continue policy
and why.
