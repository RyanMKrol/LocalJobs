# T453 — plex-language-fix: 4-stage per-item DAG replacing whole-library re-scan

2026-07-07

## What I did

Replaced `plex-language-fix`'s single `plex-language-scan` → `plex-language-apply` DAG with:

```
plex-language-discover → plex-language-resolve → plex-language-evaluate → plex-language-apply
```

- **`plex-language-discover`** (new, root stage — declares `inputKeys()`, so the workflow is
  limitable for the first time): walks the configured Plex sections and records every file
  (`${itemRatingKey}::part${partId}`) on the `work_items` ledger, with `tmdbId` extracted from the
  title's own Plex Guid (no TMDB call). Always re-walks Plex fresh so new files are found, but never
  re-marks an already-known file.
- **`plex-language-resolve`** (new): looks up each not-yet-resolved file's show/movie true original
  language via TMDB, wrapped in `callService('tmdb', () => lookupLanguageDetail(...), { cacheKey:
  'tmdb-language:<type>:<tmdbId>' })` (T451's opt-in response cache) so a show with many episodes in
  the same run makes one real TMDB call, not one per episode.
- **`plex-language-evaluate`** (new): fetches each file's live current Plex stream selection and
  decides `'change'` vs `'skip'` (collapsed `FileStatus` from the old 3-value
  `'change'|'already-correct'|'no-match'`), reusing `evaluatePart`/`pickAudioCandidate` from `lib.ts`
  unchanged. Records `currentAudio`/`currentSubtitle` too so apply can build a real before/after log
  without a second live fetch.
- **`plex-language-apply`** (rewritten): reads eligible `'change'` files from the evaluate/discover
  ledgers instead of a `data/out/language-scan.json` file (deleted, along with `plex-language-scan`
  and its types `LanguageScanFile`/`ShowOrMovieEntry`). PERMANENT idempotency — once a file is applied
  it's never automatically re-touched, even if evaluate later re-flags it, verified with a dedicated
  test.

Deleted `stages/scan.{ts,job.ts,test.ts}`. Rewrote `contracts.ts` with 3 new gate contracts (all deliberately
trivial `check()`s reading the ledger row count via the new `stages/ledger.ts`, per the repo's "trivial
gate is an acceptable minimum" rule). Updated `contracts.ts`/config.ts accordingly (dropped `scanOut`).
Fully rewrote the workflow's own `CLAUDE.md`, and fixed the root `CLAUDE.md`'s workflow-table one-liner,
which was still stale from before T452 (referenced the already-removed `plex-language-no-track-flag`).

## Scope-driven deviation, documented

`src/db/store.ts` has no generic "every success row for a job" query, and this task's scope is locked
to `src/workflows/plex-language-fix/**` + `CLAUDE.md` — so `stages/ledger.ts` reads the already-exported
`db` handle from `src/db/index.ts` directly (read-only), rather than widening scope to add a store.ts
export. Documented as a deliberate, confined exception in both the workflow's `CLAUDE.md` and the file's
own header comment.

## Testing seam

None of discover/resolve/evaluate's `run*` functions call Plex/TMDB directly — each takes an injectable
`opts` (mirroring apply.ts's pre-existing `PlexClientOverrides` pattern), defaulting to the real `lib.ts`
functions. Tests inject fakes; no live Plex/TMDB call, no `global.fetch` monkey-patching, no TMDB token
env-var workaround needed.

Added `lib.test.ts` (evaluatePart tie-breaking with the new 2-value status, migrated from the deleted
`scan.test.ts`) plus `discover.test.ts`, `resolve.test.ts`, `evaluate.test.ts`, and a rewritten
`apply.test.ts`.

**Important finding while testing:** this repo's whole test suite shares ONE scratch SQLite DB across
every `*.test.ts` file in a single process (`npm test` → `scripts/run-tests.ts`). Because these new
stages each process the WHOLE pending backlog of a shared `work_items` ledger (by design — that's the
point of "process once, ever"), a raw aggregate call-count/array-length assertion in a test is fragile:
it silently also counts leftover pending/retryable work seeded by *other* test files (including earlier
tests in the same file, e.g. a still-retryable `'failed'` row). Fixed by scoping every such assertion by
the specific key/tmdbId/ratingKey under test (a `Map` keyed by that identity) instead of a raw counter —
this is now correct regardless of what else is in the shared ledger. Confirmed via a standalone
reproduction harness before and after the fix, and via the full `npm test` run.
