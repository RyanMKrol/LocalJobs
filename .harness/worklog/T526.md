# T526

2026-07-13: Fixed the manifest-discovery walk in `src/api/server.ts` used by
`findWorkflowDataOut` (backing both `POST /api/workflows/:name/reset-output` and
`POST /api/workflows/reset-output-all`):

- `walkForWfFiles` now skips any directory literally named `data` before recursing
  (mirrors `src/workflows/registry.ts`'s `findFiles`), so a manifest cloned under
  `projects-sync/data/repos/...` is never discovered, imported, or executed.
- `findWorkflowDataOut` no longer returns `null` on the first name-matching
  manifest whose `data/out` is missing — it keeps walking and returns the first
  match whose `data/out` actually exists, only returning `null` after exhausting
  every candidate.
- `deleteDataOutContents`'s path guard now requires a genuine `.../data/out`
  boundary (trailing separator or end-of-string), not just a substring match, so
  it can't be fooled by e.g. `.../data/output-x`.

Added a regression test to `src/api/server.test.ts` using a synthetic fixture
rooted under the real `WORKFLOWS_ROOT` (the const is computed once at module
load from `import.meta.url` and can't be redirected) — a real workflow folder
with `data/out/` plus a name-duplicate "clone" manifest nested under
`.../data/repos/...` with no `data/out` and an import-time side effect (writes a
marker file). Asserts: the clone's marker is never written (never imported), the
real `data/out` is resolved (not null, not the clone), and
`deleteDataOutContents` on it reports `filesRemoved > 0` and actually removes the
file — pinning against the exact `ok:true/filesRemoved:0` false-success symptom
described in the spec.

Checks run: `npx tsc --noEmit` (clean), `npm test` (265/265 passing, including
the new test — `LOCALJOBS_TEST=1 npx tsx src/api/server.test.ts` run directly
too, 109/109 assertions in that file).

Scope: only `src/api/server.ts` and `src/api/server.test.ts` touched, per task
scope. No change to `src/workflows/registry.ts` (the structural fix considered
in the spec's part (c) is deliberately deferred — out of scope).
