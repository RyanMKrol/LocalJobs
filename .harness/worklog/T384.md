# T384: overall=true mode for GET /api/workflow-runs/:id/stage-io

## What was built

Extended `GET /api/workflow-runs/:id/stage-io` in `src/api/server.ts` to accept `overall=true`
as an alternative to `?job=<job>`. `overall` takes precedence when both are present (noted in
a code comment). When `overall=true`:

- Resolves the run's workflow DAG (`getWorkflowJobs` + `buildDag`) and reads `dag.waves[0]`
  (root wave — the workflow's own idempotency/potency key stage(s)).
- Resolves the EFFECTIVE terminal job(s) exactly like the unified Output section
  (`output-items` handler): `def?.outputJob && refs.some(...) ? [def.outputJob] : lastWave`.
- Calls the generalized `stageIoLists(outputJobs, rootWave, run.id)` from T383 and returns
  `{ inputs, outputs, predecessorJobs: rootWave, outputJobs, job: '__overall__' }`.

`?job=<job>` (no `overall`) is untouched — same code path, same shape as before.

## Verification observed

- `npx tsc --noEmit` — clean.
- `npm test` — 168/168 passing, including two new tests:
  - `overall=true — inputs match the DAG root wave for a fan-in workflow` (reuses the existing
    `sio-api-wf` 3-stage fan-in fixture: root `sio-api-a`, terminal `sio-api-c`).
  - A new `t384-override-wf` fixture (mirrors T348's `t348-override-wf` outputJob pattern):
    `t384-last` (the raw DAG last wave) never records ledger rows; `t384-first` is both the

    override (`['t384-first']`), not the empty raw terminal wave (`['t384-last']`), and
    `outputs` contains the override job's row.
