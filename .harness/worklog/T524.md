# T524 — 2026-07-13

## What was done

Two coordinated fixes to close the "workflow run wedged at status='running' forever"
hole:

1. **`src/core/workflow-executor.ts` (`runWorkflowInner`)** — widened the `try` to
   start immediately after `createWorkflowRun`/`activeWorkflowRuns.set` (was:
   only wrapping the `repeatUntilStable` cycle loop) and added a `catch` that:
   - `controller.abort()`s the run's `AbortController` so any in-flight member
     child is hard-killed via the existing SIGTERM→SIGKILL path.
   - Logs the escape loudly to the workflow run log (`error` level).
   - Calls `finishWorkflowRun(workflowRunId, 'failed')` — guarded by a `finished`
     boolean so it can never double-finish a run that already finished on the
     normal path.
   - Returns `{ workflowRunId }` (does not rethrow) so a scheduler/API caller
     that doesn't itself `.catch()` the promise doesn't crash.
   The `finally { activeWorkflowRuns.delete(workflowRunId) }` still runs on every
   path (moved to wrap the whole try/catch), so the run is always deregistered
   and immediately re-runnable.

   Added a test-only seam, `_setOnSettleThrowHookForTest` (mirrors notifier.ts's
   `_setFetchForTest`), invoked at the top of the `onSettle` callback right before
   `rollUpWorkflowProgress` — lets a unit test simulate the DB write throwing
   (e.g. a SQLITE_BUSY under real contention) without corrupting the scratch DB.
   Always a no-op (`null`) in production.

2. **`src/core/dag.ts` (`executeDag`)** — the ready-launch chain
   (`hooks.runOne(job).then(...).finally(...)`) had no `.catch`, so a REJECTING
   `runOne` (an unexpected throw, not a resolved `'failed'`) rejected the whole
   `executeDag` promise — poisoning the entire DAG run (no further stage settles,
   dependents never cascade-skip cleanly, and the rejection is exactly what used
   to escape `runWorkflowInner`). Added `.catch((): RunStatus => 'failed')`
   scoped to `runOne` ONLY (chained before `.then((s) => settle(job, s, false))`,
   not wrapping the `settle` call itself) — so a throw is now treated identically
   to a resolved `'failed'`: the node settles failed, its dependents cascade-skip,
   sibling stages keep running, and `executeDag` resolves normally instead of
   rejecting. Deliberately NOT catching a rejection from `settle`/`onSettle`
   itself (a genuine DB-write failure there) — that's left to propagate and is
   handled by fix (1) as defence in depth.

## Tests added

- `src/core/dag.test.ts`: new `executeDag` case — a rejecting `runOne` for one
  node settles that node `'failed'`, its dependent cascades to `'skipped'`, an
  independent sibling still runs to `'success'`, and `executeDag` resolves
  (doesn't reject).
- `src/core/workflow-executor.test.ts`: two new cases using the new
  `_setOnSettleThrowHookForTest` seam —
  - a single-member workflow whose `onSettle` throws once (simulated
    SQLITE_BUSY): asserts `runWorkflow` resolves (doesn't reject), the
    `workflow_runs` row ends `status='failed'` with `finished_at` stamped, the
    throwing hook fired exactly once, the escape is logged exactly once, the
    run is removed from the active-run registry, `workflowRunInProgress` is
    `false` afterward, and — the actual proof of "no daemon restart needed" —
    a SECOND `runWorkflow` call for the SAME workflow immediately afterward is
    NOT skipped as "already running" and completes normally.
  - a two-member workflow (one throws in `onSettle`, one hangs like the
    existing cancellation-test fixture): asserts the run ends `'failed'` AND
    the hung sibling is hard-killed (settles `'cancelled'`, not left `'running'`
    forever) — proving the `AbortController` is genuinely aborted on the throw
    path, not just logged.

- Debugged one flaky-looking test failure during development: the sibling-kill
  test's poll loop broke on the FIRST sighting of the member run row rather than
  waiting for it to reach a terminal status, so it sometimes observed the row
  mid-transition (`status='running'`, before the SIGTERM→SIGKILL path finished)
  and failed. Fixed by polling until the status is no longer `'running'`. Not a
  product bug — confirmed via a throwaway debug script that the actual kill
  completes in well under a second.
