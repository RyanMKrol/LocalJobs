# T010 worklog — Perfumes: handle empty Fragrantica notes sections in markdown

## Attempt 1 — 2026-06-21

### Problem
Some Fragrantica pages come back with an empty notes pyramid. The build stage is
LLM-driven (Claude reads the parsed `fragJson` and writes the markdown profile),
so the prompt previously said only "notes.top/heart/base: from the notes pyramid"
+ a generic "never invent notes". With an empty pyramid that's ambiguous — Claude
was also told to "research the rest (web search)", risking a fabricated/web-sourced
pyramid, or silently dropping the section.

### Decision (honest handling)
- **Section is never dropped, never fabricated — explicitly marked empty.**
- Parse side: normalize `notes` into a canonical `{top,heart,base}` pyramid where
  every tier is always present as an array; an empty pyramid is written back as
  `{top:[],heart:[],base:[]}` (present but explicitly empty) and logged as a warn.
- Build side: the notes-mapping clause in the build prompt now branches — an empty
  pyramid yields an EXPLICIT directive to keep the tiers as empty arrays, NOT
  fabricate/web-research a substitute, and state in the Olfactory Profile that the
  breakdown was unavailable. A populated pyramid gets the normal map-through clause.

### Changes
- `src/jobs/perfumes/parse.ts`: new exported `NotesPyramid`, `normalizeNotes()`,
  `notesEmpty()`; `applyNormalizedNotes()` mutates the parsed data + logs, called
  in `runParse` before writing the frag JSON.
- `src/jobs/perfumes/build.ts`: new exported `notesMappingClause(fragJson)` (never
  throws on malformed JSON → treated as empty); wired into `buildPrompt`.
- `src/jobs/perfumes/build.test.ts`: new — covers normalize shape/trim, empty vs
  populated `notesEmpty`, the empty-section fixture (asserts EMPTY / empty arrays /
  do-NOT-fabricate / unavailable), the populated fixture (normal clause, no EMPTY
  branch), and malformed-JSON fallback.
- `README.md`: perfumes pipeline description updated to mention empty-notes handling.
