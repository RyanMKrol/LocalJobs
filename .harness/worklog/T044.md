# T044 — Reorganize jobs into stages/ + pull services out to self-contained src/services/

## What I did

### Part 1 — Services → top-level, self-contained `src/services/`
- `git mv` the four service defs out of the workflow folders into a new sibling
  `src/services/`: `claude-cli.service.ts` + `fragrantica.service.ts` (ex-perfumes),
  `gemini.service.ts` + `google-places.service.ts` (ex-places). Fixed
  `'../../core/types.js'` → `'../core/types.js'`.
- Made each **self-contained** (Option B) — they import NOTHING from a workflow's
  `config.ts`; each reads its own limits from env with the SAME defaults so
  behaviour + `.env` are unchanged:
  - `claude-cli`: unchanged (`ratePerMinute: 20`).
  - `fragrantica`: `minIntervalMs`/`maxJitterMs` from `PERFUMES_FETCH_DELAY_MS ?? 12000`
    / `PERFUMES_FETCH_JITTER_MS ?? 6000` (perfumes `config.ts` KEEPS those fields —
    `fetch.ts` still uses them; the service just reads the same env independently).
  - `gemini`: `ratePerMinute: GEMINI_RATE_PER_MIN ?? 10`, `monthlyCap:
    PLACES_LLM_MONTHLY_CAP ?? 2000`, `dailyCap: PLACES_LLM_DAILY_CAP ?? floor(monthly/30)`,
    `paid: true`.
  - `google-places`: `ratePerMinute: PLACES_RATE_PER_MIN ?? 30`, `monthlyCap:
    PLACES_ENRICH_MONTHLY_CAP ?? 1000`, `dailyCap: PLACES_ENRICH_DAILY_CAP ??
    floor(monthly/30)`, `paid: true`.
  - The `daily = monthly/30` math moved into `src/services/lib.ts`
    (`DAILY_SPEND_DIVISOR` + `dailyFromMonthly()`).
  - Kept existing env var names only — did NOT add new aliases, so `.env.example`
    is unchanged.
- Registry (`src/jobs/registry.ts`): now discovers `*.service.ts` from BOTH
  `resolve(__dirname,'..','services')` (guarded with `existsSync`) AND `src/jobs/`
  (so a private job may still colocate a service). Job/workflow discovery + the
  orphan-job check are unchanged.
- Places `config.ts` shed the now-service-owned caps: removed
  `monthlyCap`/`dailyCap` from `llmConfig` and `enrichConfig` and removed
  `DAILY_SPEND_DIVISOR`. Kept `resolveConfig`'s caps (it's `job_usage`-governed,
  not a service) and perfumes `fetchDelayMs`/`fetchJitterMs`. Verified `enrich.ts`
  / `llm-enrich.ts` already read caps off the SERVICE object (`svc.dailyCap` /
  `svc.monthlyCap`), so removing the config caps is safe. Updated the
  `places.workflow.ts` doc comment to point at `src/services/lib.ts`.
- Moved the cap-math assertions out of `src/jobs/places/config.test.ts` into a new
  `src/services/caps.test.ts` that asserts each PAID service seeds `dailyCap =
  floor(monthlyCap/30)` + the month-of-daily-runs-fits-monthly invariant + that
  `capStatus`/`recordUsage` stops at the daily cap. `places/config.test.ts` keeps
  the (still places-specific) daily-cron schedule assertion.
