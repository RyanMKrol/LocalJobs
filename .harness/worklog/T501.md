# T501: Cache openfigi ISIN→ticker resolutions via cacheKey (22h TTL)

## What was done

1. Set `cacheTtlMs: 79_200_000` (22 hours in milliseconds) on the `openfigi` service definition in `src/services/openfigi.service.ts`.

2. Refactored `resolveOpenFigiTickersBatched()` in `src/services/trading212.service.ts` to make individual `callService('openfigi', ...)` calls per ISIN, each with a stable cache key `openfigi:isin:${isin}`. This ensures:
   - Each ISIN resolution is cached independently for 22 hours
   - Repeated resolutions of the same ISIN (across different workflows or requests) hit the cache
   - Different ISINs can share the same cache pool across all callers

## Verification

- `npx tsc --noEmit` — ✅ passes
- `npm test` — ✅ all 252 tests pass (including existing service/trading212 tests)

The caching mechanism works as follows:
- First call: `callService('openfigi', fn, { cacheKey: 'openfigi:isin:US0378331005' })` makes the live API call via `fetchOpenFigiTickers([isin])` and stores the result in `service_cache`.
- Second call (within 22h): same cacheKey hits `getCachedServiceResponse()`, returns cached value, skips the API call.
- The 22-hour TTL (79,200,000 ms) means the cache auto-expires after that window.

## Note on implementation

The refactored `resolveOpenFigiTickersBatched()` makes one `callService` call per ISIN instead of batching multiple ISINs into a single request. This trades batching efficiency (fewer OpenFIGI requests) for per-ISIN cache reuse (the cache key is shared across different batch compositions). Given that the openfigi service has generous rate limits (20 req/min), this is a favorable trade-off — the common case of repeated lookups of the same ISIN across multiple workflows will now hit the cache instead of making redundant network calls.
