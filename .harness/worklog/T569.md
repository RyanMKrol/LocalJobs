# T569 — migrate all 16 *.service.ts onto defineService()

2026-07-14

## What I did

- Converted the `ServiceDefinition` object in all 16 `src/services/*.service.ts`
  files to `defineService()` from `./lib.ts` (built in T568). Each definition
  shrank to a declarative ~10-line block; every file's per-service doc comment
  above the definition is unchanged.
- Preserved every service's exact pre-existing env-var name and default so
  effective limits are byte-identical:
  - Services whose env names already matched the `envPrefix`-derived convention
    (finnhub, github, hevy, lastfm, openfigi, tmdb, trading212,
    trading212-instruments, dynamodb) use `envPrefix` + `{ fallback }`.
  - Services with bespoke pre-existing names (gemini's `PLACES_LLM_MONTHLY_CAP`/
    `PLACES_LLM_DAILY_CAP`, google-places' `PLACES_ENRICH_MONTHLY_CAP`/
    `PLACES_ENRICH_DAILY_CAP`/`PLACES_RATE_PER_MIN`, fragrantica's
    `PERFUMES_FETCH_DELAY_MS`/`PERFUMES_FETCH_JITTER_MS`) use the `{ env, fallback }`
    override form.
  - gemini, google-places, dynamodb use the `'monthly/30'` dailyCap sentinel
    (still overridable via their existing `*_DAILY_CAP` env var, per the bespoke
    `{ env, fallback: 'monthly/30' }` form).
- Fixed `vercel.service.ts`, the one definition missing `rateLimitSource`
  (`defineService()` makes it a required field, a compile error to omit) — added
  an honest source string explaining the caps are our own conservative estimates,
  not a documented Vercel deploy-rate ceiling.
- Left the colocated client-helper code in `trading212.service.ts` (~230 lines),
  `dynamodb.service.ts` (~180), and `openfigi.service.ts` (~80) untouched — only
  their `ServiceDefinition` object at the bottom was converted, per the owner's
  2026-07-12 decision (split-file is the go-forward standard for NEW services
  only; the three existing colocated services are not refactored).
- Added `src/services/effective-limits-snapshot.test.ts`:
  - Imports all 16 services and asserts their effective
    `ratePerMinute`/`dailyCap`/`monthlyCap`/`minIntervalMs`/`maxJitterMs`/
    `timeoutMs`/`cacheTtlMs` equal a frozen pre-migration snapshot (values
    computed from each service's original hand-written definition — verified by
    manually re-deriving `dailyFromMonthly` for gemini/google-places/dynamodb:
    66/33/1666 respectively).
  - Asserts `vercel.rateLimitSource` is now non-empty.
  - A poisoned-env regression test: spawns a child process re-importing
    `vercel.service.ts` with `VERCEL_MONTHLY_CAP='2,000'` and asserts it throws at
    load, naming the var (proves `envInt()`, not a silent `NaN`-producing bare
    `Number(...)`, governs the read).
  - Sanity-checked the snapshot test actually catches drift by temporarily
    perturbing a value in `google-places.service.ts` and confirming the test
    failed, then restored it.
- Updated `src/services/CLAUDE.md` with two new sections: "Authoring a service —
  use `defineService()`" (documents the factory, envPrefix vs. bespoke `{ env }`
  overrides, the `monthly/30` sentinel, and where the regression tests live) and
  "Client-helper convention: split-file is the go-forward standard for NEW
  services" (documents the go-forward convention and explicitly notes the three
  existing colocated services are deliberately not refactored).
