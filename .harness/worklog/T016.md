# T016 worklog — Framework: first-class pipeline progress percentages

## Attempt 1 — 2026-06-21

### Problem
Pipeline `progress` was only updated at stage settle time (`settled/total*100`),
so a pipeline showed coarse whole-stage steps and, while a single long member job
ran, sat flat (the perfumes pipeline effectively showed 0% for long stretches).
It also accumulated `settled` across `repeatUntilStable` cycles → could exceed
100%.

### Change
Made pipeline progress a first-class, real-time roll-up from member-job progress.

- **`src/db/store.ts`**
  - Added `rollUpPipelineProgress(pipelineRunId, message?)`: denominator =
    member count from `pipeline_jobs`; each stage contributes a fraction in [0,1]
    — terminal run = 1, running run = its `progress`/100, not-yet-started = 0.
    Picks the latest run per member job by `rowid` (UUID ids aren't ordered), so
    it tracks the current `repeatUntilStable` cycle. `message` omitted → updates
    only the percentage (keeps the last stage message); given → also sets
    `progress_msg`. No new column needed — denominator comes from `pipeline_jobs`.
  - Hooked `setProgress`: after writing a member run's progress, if the run has a
    `pipeline_run_id`, it rolls the parent pipeline up live. (Parent/daemon is
    still the sole DB writer — `setProgress` runs in the executor.)
- **`src/core/pipeline-executor.ts`** — `onSettle`/`onSkip` now call
  `rollUpPipelineProgress(...)` instead of `setPipelineProgress(settled/total)`.
  This also caps repeat-cycle progress at 100 (DB-driven, deduped per job).
- **`src/core/types.ts`** — JSDoc on `PipelineRunRow.progress` documenting the
  roll-up semantics.
- **`src/core/pipeline-executor.test.ts`** — new test
  *"progress roll-up: pipeline % is computed from member progress over the total
  stage count, in real time"*: 4 stages → asserts 0% with no runs, 25% after one
  terminal stage, **38%** after a mid-stage member reports 50% **via `setProgress`
  alone** (real-time hook, no explicit roll-up call), and 50% after it hits 100%.
  Also strengthened the all-success test to assert final progress == 100.

### Verification (observed)
- `npx tsc --noEmit` — clean.
- `npm test` — all green; `pipeline-executor.test.ts` now 10 tests pass
  (incl. the new roll-up test); full suite "✓ all tests passed".
- No `dashboard/` files changed → dashboard build not required. Progress is
  surfaced via the existing `pipeline_runs.progress` column the dashboard already
  renders (Pipeline run detail), now meaningful instead of flat.
- No live paid-API calls made.

### Trade-offs (recorded in .harness/LIMITATIONS.md)
Non-monotonic under `repeatUntilStable` (sawtooth at cycle boundaries);
denominator trusts current `pipeline_jobs` (fine in single-process daemon);
one extra correlated subquery per member progress event (negligible at scale).
