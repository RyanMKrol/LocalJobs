# T280 — Stream the loop's Claude output live (decide + apply the streaming flag)

`needs-human` gated task; owner explicitly asked to attempt it interactively this session.

## Decision (owner-made, after an empirical correction to the spec)

The spec's own recommendation (`--verbose`, "lighter touch") turned out to be **wrong** — tested
empirically before implementing. Piping `claude -p` output to a file (exactly what `loop.sh` does
via `tee "$out"`, `.harness/scripts/loop.sh:417`) revealed:

- **`--verbose` alone**: with a long-running prompt (~15s), the output file stayed at a flat
  ~157 bytes (just an initial "no stdin data" warning) for the ENTIRE run, then jumped straight to
  the full ~12KB of content all at once right as the process exited. It does **not** flush
  incrementally when stdout isn't a TTY — the exact "looks frozen" problem this task exists to fix.
- **`--output-format stream-json`**: with the same prompt, the output file grew in real
  increments while text was still being generated (0 → 5.7KB → 6.8KB across polls at t=3s/6s/9s,
  well before the process exited) — genuinely live.

Reported this back to the owner (the spec's premise was wrong, not just a minor detail), and they
chose to switch to `--output-format stream-json` over sticking with the spec's original
recommendation.

## Verification

- **Wiring**: `LOOP_SOURCE_ONLY=1 bash -c 'source .harness/scripts/loop.sh; echo "$FLAGS[@]"'` →
  `--dangerously-skip-permissions --output-format stream-json --verbose`. Confirms harness.env's
  default reaches the actual `FLAGS` array `run_claude` passes to `claude -p`.
- **Live growth**: confirmed above (the empirical test that changed the decision) — this IS the
  "during a real loop build, `.claude-out` is observed to grow incrementally" check from the spec's
  Done-when, run standalone against the real `claude -p` invocation (not a synthetic mock), polling
  file size every 3s during an actual multi-paragraph generation.
- **Rate-limit detection still fires — REASONED, not empirically triggered** (no real usage limit
  was hit during testing; per the spec, reasoning from the format is an accepted substitute):
  - `RL_HARD_RE` (the unconditional check, `loop.sh:424`) is checked with plain `grep -qiE` against
    the raw output bytes regardless of format — grep doesn't parse JSON, so a phrase match works
    the same whether the text sits in a plain-text stream or inside a JSON string value, AS LONG AS
    the limit notice's wording is still present verbatim somewhere in the byte stream.
  - Confirmed via the same live test that `stream-json` embeds full, unencoded assistant-message
    TEXT inside its JSON events (directly observed the complete printing-press essay's prose, word
    for word, inside the JSON stream) — text content is not obfuscated or re-encoded in a way that
    would break a phrase match. Since the existing code comments (`loop.sh:403`) establish that the
    limit notice IS an ordinary assistant message (not a CLI-level system error), the same
    text-preservation applies to it.
  - **New finding, NOT wired up (out of scope for this task, flagging for a possible follow-up):**
    `stream-json` mode also emits a distinct, ALWAYS-present `rate_limit_event` metadata object per
    request (e.g. `{"type":"rate_limit_event","rate_limit_info":{"status":"allowed",
    "rateLimitType":"five_hour",...}}`) on every run, limited or not. Checked this does NOT cause a
    false positive: `RL_HARD_RE`'s phrases don't match this event's field names/values, and the
    broader `RL_RE` (which WOULD match "rate_limit" as a substring) is only consulted when
    `rc != 0` (`loop.sh:425`), so a normal successful run — which always emits this event and always
    exits 0 — never trips it. A future task could use this structured `status` field for more
    reliable limit detection than prose-matching, but that's a bigger change than this task's scope
    (verify existing detection, not replace it).
- `npx tsc --noEmit` — clean (no TS/JS touched by this task; ran anyway per Definition of Done).
- `npm test` — unaffected (no test-covered code touched); not re-run since this task only edits
  `.harness/config/harness.env` and `TASKS.json`'s `scope` field, confirmed via `git diff --stat`.

## Addendum — REVERTED after a real loop run (final resolution)

The isolated test above (a standalone `claude -p` call, output captured to a file, byte counts
polled) proved `stream-json` flushes incrementally — that part was correct. What it didn't surface:
`--verbose --output-format stream-json`'s raw JSONL, watched LIVE in a real `supervise.sh` foreground
terminal during an actual loop run (T322), is a firehose — hook-lifecycle events, per-turn session
config dumps, base64 `thinking` signatures, tool-call/tool-result JSON blobs — with the readable
content (assistant text, tool names) buried in it. The owner ran it for real, judged it "too much...
not really useful," and asked whether a better option existed before reverting.

Presented a middle ground (pipe the raw stream through a `jq` filter in `loop.sh` to show only
assistant text + tool calls live, while still teeing the raw JSON to `.claude-out` for
`RL_HARD_RE`/`RL_RE` to grep) versus a full revert. The owner chose to **revert entirely** rather
than take on that extra complexity right now.

**Final state: `CLAUDE_FLAGS` removed from `.harness/config/harness.env` — back to `loop.sh`'s own
built-in default (`--dangerously-skip-permissions` only, no streaming flag).** Verified by sourcing
`loop.sh` and confirming `$FLAGS[@]` == `--dangerously-skip-permissions`. The original "looks frozen
for 10–20 minutes" problem this task set out to fix is UNRESOLVED — accepted trade-off, since neither
tested flag combination gave both liveness AND readability without more engineering than the owner
wanted to invest right now. If revisited, the `jq`-filter approach above is the documented next idea,
not a dead end.
