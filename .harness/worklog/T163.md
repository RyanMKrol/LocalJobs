# T163 ‚Äî Run-limit selects 0 roots when downstream work is un-attempted

## Problem
A limited manual run (`POST /api/workflows/:name/run { limit: N }`) of a multi-stage
workflow could select **0 roots** and no-op while reporting success, even with a large
backlog. Reproduced on places: 1,753 resolved but only 131 enriched ‚Üí ~1,622
resolved-but-unenriched places were the real outstanding work, but the selector couldn't
see them.

## Root cause
`selectPendingRoots` marked a root pending iff `!isWorkItemDone(entryStage, root) ||
rootHasOutstandingWork(members, root)`. For a resolved-but-not-enriched root the entry
(resolver) IS done (clause 1 false) and `rootHasOutstandingWork` only matched **existing**
not-done rows ‚Äî a downstream stage that simply **hasn't been attempted has NO row at all**,
so a *missing* row wasn't counted outstanding. Root looked "fully done" ‚Üí not selected ‚Üí
empty allowlist ‚Üí every stage's `ctx.rootAllowed()` false ‚Üí nothing ran.

## Fix
1. **`src/db/store.ts`** ‚Äî redefined "pending" by propagation through the **terminal stage**:
   - New `isRootPending(members, terminalJobs, root, minAttempts)`:
     1. any retryable not-done row anywhere ‚Üí pending;
     2. else a *done* terminal-stage row (`rootReachedTerminal`) ‚Üí fully done;
     3. else a "gave-up" marker (`rootHasTerminalBlocker`: an `ignored` or
        retry-exhausted `failed` row at any stage) with no retryable work ‚Üí stuck below
        terminal, treated as **done** so the selector can't livelock on an
        unprogressable root;
     4. else un-attempted downstream work (entry done, later stage has NO row) ‚Üí pending.
   - `selectPendingRoots` signature changed: `entryJob` param replaced by `terminalJobs`
     (the DAG's last wave). It now only references terminal/member sets.
2. **`src/core/workflow-executor.ts`** ‚Äî pass `dag.waves[last]` as `terminalJobs`; and
   when a limited run computes an empty `selected_roots` from a non-empty candidate set,
   log a WARN ("... 0 originating inputs were selectable ‚Äî N candidate(s), all already
   complete ...") instead of silently succeeding.
3. **`src/jobs/places/stages/enrich.ts` + `llm-enrich.ts`** ‚Äî split exclusion reasons:
   `notDone` (still needs work) vs `todo` (within the run's selected roots). When
   `todo===0` but `notDone>0`, log "0 to do this run ‚Äî N outstanding but none in this
   run's selected roots" (a WARN) instead of the misleading "every resolved place is
   already done".
4. **`CLAUDE.md`** ‚Äî documented the corrected terminal-propagation pending semantics +
   the empty-selection warning.

## Tests (hermetic, scratch DB)
- `src/db/store.test.ts` ‚Äî updated existing `selectPendingRoots` calls to the new
  signature; added a T163 block covering: (1) entry-done-no-downstream-row ‚Üí pending
  (the regression), (2) terminal-done ‚Üí not selected, (3) retryable downstream row ‚Üí
  pending (preserves T094), (4) middle-stage exhausted-failed ‚Üí done (no livelock),
  (4b) ignored downstream ‚Üí done, plus mixed-set ordering.
- `src/core/workflow-executor.test.ts` ‚Äî added two executor-level tests with pre-seeded
  ledgers: a backlog (entry done, terminal un-attempted) selects those roots (not 0) and
  emits no warning; an all-complete candidate set selects 0 and logs the empty-selection
  WARNING.
