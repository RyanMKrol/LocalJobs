# T544 — clear the setTimeout(setBusy(false)) after unmount

2026-07-15

## What I did

Both `finally { setTimeout(() => setBusy(false), 1200); }` call sites left a
1.2s timer running unconditionally, which called `setState` after unmount if
the user navigated away before it fired (a React setState-after-unmount
warning path):

- `dashboard/app/workflows/[name]/page.tsx:136` (`run()`)
- `dashboard/app/workflow-runs/[id]/page.tsx:60` (`cancel()`)

Evaluated routing through the R05-a `useAction` hook
(`dashboard/app/components/useAction.ts`) first, per the task's stated
preference — but `useAction` clears its `busy` state immediately in its
`finally` block (no delay), so reusing it as-is would drop the required
~1.2s "Started…"/"Cancelling…" transient busy window entirely, which the
task explicitly requires preserving. Went with the simpler, behavior-exact
fix instead: a `useRef<ReturnType<typeof setTimeout> | null>` per component
holds the pending timer id, and a `useEffect` cleanup on unmount
(`() => { if (busyTimerRef.current) clearTimeout(busyTimerRef.current); }`)
clears it. The `finally` block now stores the timer id in the ref instead of
firing a bare, unref'd `setTimeout`. No change to the transient busy-state
UX — `setBusy(true)` still fires synchronously and `setBusy(false)` still
fires ~1.2s later on the happy path; the only behavioral change is that the
timer is cancelled if the component unmounts first.
