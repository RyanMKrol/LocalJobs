# T479 — Decide which services should adopt response caching (cacheKey) + TTLs

**Decision: adopt caching on EVERY cacheable (`category:'api'`) service, TTL = 22 hours for all,
deliberately independent of the work_items ledger.** Authored 11 buildable follow-up tasks
(T500–T510), one per service. Decision + authoring task; no caching implemented here.

## Policy (owner, 2026-07-11)

- **Cache basically every service that makes a traditional API call**, ≥1h TTL — don't gate it on a
  spend/latency "is it worth it" analysis. Rationale: caching is extremely cheap, and during
  development APIs get called repeatedly; a stable-key API response is safe to reuse.
- **Caching lives SEPARATELY from the `work_items` ledger — on purpose.** The ledger is mutable dev
  state (it can have bugs, gets reset during development); a stable-key API response rarely changes
  the same way. So caching is valuable alongside the ledger even where it doesn't "beat" it. This
  reverses the initial framing (cache only where it beats the per-item ledger skip) — rejected.
- **TTL = 22 hours for all** (owner's refinement of "at least an hour"): a daily-scheduled job runs
  ~24h apart, so a 22h TTL guarantees the cache has expired before the next daily run (always fresh
  in prod) while still catching every intra-day and dev re-run. `cacheTtlMs: 79_200_000`.

## Mechanism (already shipped — T451/T476)

`callService(name, fn, { cacheKey })` caches `fn`'s result in the `service_cache` table when
`cacheKey` is set AND `def.category === 'api'` (`src/core/services.ts:127`). TTL = `def.cacheTtlMs ??
SERVICE_CACHE_TTL_MS`. So adoption per service = set `cacheTtlMs` on the service def + pass a stable
`cacheKey` at each read call site. `cacheKey` is per-call-site, not a service-level default.

## Notable per-service judgment calls (owner-confirmed)

- **gemini (T505):** cache-key includes a **prompt-version tag** (`:v<N>`) so editing `buildPrompt`
  invalidates stale summaries. The response is a web-grounded, non-deterministic generation
  deliberately FROZEN for the TTL — fine for a summary, and a pure dev-convenience since the places
  ledger already makes gemini a once-ever call per place in prod.
- **github (T506):** the `github` service also routes `git clone`/`fetch`/`reset` — those are
  side-effecting, NOT cacheable; only the repo-listing API reads get a cacheKey.
- **trading212 (T509):** read-only per the broker rule; 22h TTL means daily prod runs always fetch
  fresh, cache only bites intra-day/dev.
