# T329 — Move TASKS.json/IDEAS.md + owner-owned overlay files into .harness/tracking/

Follow-up to T327, done interactively in the same session at the owner's request: "the
human-done/manual-fail/reviews files and even TASKS.json and IDEAS.md" were still loose files at
the `.harness/` top level after T327's docs/config/ledgers/scripts reorg — move them into a
`.harness/tracking/` folder so they're not cluttering the top level.

## What moved

`.harness/tracking/`: `TASKS.json`, `IDEAS.md` (gitignored), `human-done.json`, `manual-fail.json`,
`reviews.json`.

**Deliberately did NOT move:** `tasks/`, `worklog/`, `ledgers/` (already grouped by kind, per
T327) stay at the `.harness/` top level. `CLAUDE.md` stays too (Claude Code auto-load).

## The real coupling this exposed (why this wasn't just a file move)

`src/api/server.ts`'s `readTaskSpec`/`readWorklogContent`/`readTaskBuildFailures` all took a
`baseDir` parameter that was ALWAYS `dirname(<the backlog file's path>)` — i.e. the code hard-assumed
`TASKS.json`'s own directory directly contains `tasks/`, `worklog/`, and `ledgers/` as siblings, and
even derived the REPO ROOT itself as `dirname(baseDir)` inside `readTaskSpec` (used to resolve the
JSON `spec` field, a repo-root-relative path like `.harness/tasks/T001.md`). Moving `TASKS.json` one
level deeper into `tracking/` while leaving `tasks/`/`worklog/`/`ledgers/` in place broke ALL of
this silently (wrong repo-root resolution, wrong tasksDir/worklogDir/ledgersDir) — a real functional
regression, not just docs.

**Fix:** decoupled "where the backlog file lives" from "where tasks/worklog/ledgers live". Added a
`HARNESS_DIR` constant (the real, unmoved `.harness/` directory) and renamed the three functions'
second parameter from `baseDir` to `harnessDir` — now ALWAYS the real `.harness/` dir, passed
explicitly rather than derived from the backlog path. `createApiServer`'s `opts` gained a
`harnessDir` override (defaulting to `HARNESS_DIR`) so tests that model `.harness/` as a flat temp
dir (backlogPath + tasks/ + worklog/ + ledgers/ all siblings, mirroring the PRE-move layout) can
still point `harnessDir` at their own fixture root instead of the real production `.harness/`.
`reviewsPathFor`/`humanDonePathFor`/`manualFailPathFor` needed NO changes — those 3 files move
WITH `TASKS.json`, so "sibling of the backlog file" stays true for them.

## Incident during verification (flagged to the owner, deferred per their instruction)

While re-running the `mark-failed.sh --undo` sanity check after the move, I reused `T278` as an
"arbitrary safe test id" (it had no manual-fail entry during T327's earlier verification pass).
Between then and this pass, `T278` had gained a REAL manual-fail entry — sitting **uncommitted** in
the working tree (no prior commit in `git log` ever added it) — presumably from a dashboard action
during the session. My `--undo` test cleared it for real. Because I still had the `git mv` renames
for the 4 tracking files staged at the same time, `mark-failed.sh`'s bare `git commit` (no pathspec)
swept both into one commit: `4d207d1 manual-fail: clear T278 [skip ci]`. `NO_PUSH=1` was set, so this
never reached the remote before being caught — but the original `reason` text for T278's manual-fail
entry is unrecoverable (never committed anywhere). Flagged to the owner; they asked to defer
investigating/restoring it and just get this task's real work committed+pushed, which is what this
commit does — `4d207d1` predates it and is included as part of normal history, not reverted or
rewritten. **Lesson for future sessions:** never reuse a "known-safe" test id across verification
passes without re-checking current state immediately before — overlay files are live, owner-editable
state, not fixtures.
