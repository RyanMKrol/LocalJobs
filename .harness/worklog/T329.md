# T329 — clickable sortable headers on IoPanel

## What changed

- Extracted the `SortTh` component (previously a private helper hardcoded to
  `RecSortCol` inside `dashboard/app/workflows/[name]/page.tsx`) into a shared,
  generically-typed component at `dashboard/app/components/SortTh.tsx`
  (`SortTh<Col extends string>`), per the T268 shared-component convention.
  `active` accepts `Col | null` so a table can start with no active sort column.
- `workflows/[name]/page.tsx` now imports the shared `SortTh` (+ `SortDir`)
  instead of defining its own; `MovieRecsManager`/`TvRecsManager` unchanged
  otherwise (still use their own `sortRecs`/`RecSortCol`, which are
  rec-manager-specific).
- `IoPanel` (`dashboard/app/workflow-runs/[id]/page.tsx`) gets local
  `sortCol`/`sortDir` state (`IoSortCol = 'input' | 'state' | 'output' |
  'outputState'`) and a `handleSort` callback mirroring the existing
  toggle-asc-desc-on-repeat-click semantics. Default: no active sort column
  until the owner clicks a header (first click on a fresh column sorts desc,
  same as the existing pattern) — documented inline as a comment.
- New `sortIoRows` helper sorts by `localeCompare` on: input label (falls back
  to key via the existing `itemLabel` helper), input status, output key, output
  status. Applied AFTER the existing state-filter chips (`filteredByState` →
  `sortIoRows` → `filtered`), so sort composes correctly with filtering and with
  per-job scoping (T313/T314) — switching jobs/filters doesn't retain a stale
  sort on now-absent data since the sort re-derives from the current `io`/`filter`
  each render.
- Table headers replaced with `<SortTh>` instances; Output/Output-state headers
  only render when `!singleStage`, matching the pre-existing conditional column
  rendering.
- Reused `.sort-th`/`.sort-th-active` CSS as-is — no new classes needed, so
  `globals.css` was not touched.
