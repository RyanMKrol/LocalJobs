# T253 — workouts-sync workflow: Hevy → DynamoDB

## What was built

- `src/services/hevy.service.ts` — Hevy API rate-limit/quota service (20 rpm, 500/day, 5000/month; env-overridable).
- `src/jobs/workouts-sync/workouts-sync.workflow.ts` — workflow manifest: daily at 06:00, single `hevy-sync` stage, maxConcurrency 1.
- `src/jobs/workouts-sync/stages/hevy-sync.job.ts` — JobDefinition wrapper.
- `src/jobs/workouts-sync/stages/hevy-sync.ts` — implementation:
  - Paginates `https://api.hevyapp.com/v1/workouts?page=&pageSize=10` via `callService('hevy', ...)`.
  - Skips workout ids already in the `work_items` ledger (keyed by Hevy workout id).
  - Writes each workout to `WORKOUTS_TABLE` and each exercise as `{workoutId}_{index}` to `EXERCISES_TABLE` via `callService('dynamodb', () => dynamoPut(...))`.
  - Injectable `fetchPage` + `putItem` dependencies for hermetic testing.
- `src/jobs/workouts-sync/stages/hevy-sync.test.ts` — 10 unit tests; all hermetic (stub fetcher + stub putter + scratch DB; no live AWS writes, no live Hevy calls).
- `CLAUDE.md` / `README.md` — workouts-sync added to the workflow list and file map.
- `.env.example` — `HEVY_API_KEY` was already present (added in T252 env pass).
