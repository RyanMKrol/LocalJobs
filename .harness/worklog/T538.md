# T538 — Run the three dashboard test suites

2026-07-13

## What was done

1. `scripts/run-tests.ts` now also walks `dashboard/app/` for `*.test.ts` files
   (in addition to the existing `src/` + top-level `scripts/`) — this wires the
   pure, no-browser dashboard component suites
   (`OutputRenderer.test.ts`, `StageIoLists.test.ts`) into root `npm test`.
   `dashboard/scripts/` (nav-check.test.ts + mobile/visual-check) is a separate
   path never walked by this — stays local-only by construction.
2. Added a `test` script to `dashboard/package.json` that runs the same suites
   directly via the root's `tsx` binary (`../node_modules/.bin/tsx …`) — avoids
   adding `tsx` as a new dashboard devDependency / touching
   `dashboard/package-lock.json` (out of scope for this task), while still
   giving `npm --prefix dashboard test` a standalone entry point.
3. Added a CI step (`.github/workflows/ci.yml`) — "Test dashboard" — running
   `npm --prefix dashboard test` after the "Build dashboard" step. Also moved
   "Install (dashboard)" earlier (right after "Install (root)", before
   "Typecheck"/"Test") so the root `Test` step's new `dashboard/app` walk can
   resolve `react`/`react-dom`/`react-markdown` from `dashboard/node_modules`.
4. Documented the local-only (nav-check, mobile-check, visual-check — all
   Playwright/real-browser-driven) vs CI (the two pure component suites, plus
   the two new pure-helper suites) split in a new "Testing" section in
   README.md (CLAUDE.md is out of this task's scope, so the split is
   documented there instead of the file-map table).
5. Added the missing pure-helper test suites per the owner's 2026-07-12
   decision (not deferred):
   - `dashboard/app/ui.test.ts` — `cronToEnglish`, `resolveMode`, `backFrom`,
     `fmtDuration`, `fmtRelative`.
   - `dashboard/app/components/MarkdownModal.test.ts` — `parseFrontmatter`.
