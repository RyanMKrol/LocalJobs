# T368 — Add validation gate to workouts-sync's 2 stages

## What was built
- `src/jobs/workouts-sync/contracts.ts`: new `workoutsHistoryContract(file = defaultHistoryPath)`
  factory (key `workouts-history`), mirroring the `places`/`stocks-sync` `fromChecks` idiom.
  Validates `data/out/workouts-history.json` is a readable JSON file whose top-level value is a
  plain array; if non-empty, every entry must have a non-empty string `id` and an `exercises`
  array. An empty array is treated as legitimate (fresh install, no Hevy workouts synced yet) —
  same leniency as `stocks-sync`'s portfolio contract, not places' hard non-empty rule.
  No `config.ts` was introduced (per the spec's guidance) — the contract imports
  `defaultHistoryPath` directly from `./stages/hevy-sync.js`.
- Wired the contract: `hevy-sync.job.ts` declares `produces: [workoutsHistoryContract()]`,
  `workouts-progress.job.ts` declares `consumes: [workoutsHistoryContract()]`.
- `src/jobs/workouts-sync/contracts.test.ts`: synthetic-fixture tests (temp dir, no live Hevy
  call) covering pass (non-empty array, empty array) and fail (missing file, invalid JSON,
  non-array top-level, entry missing `id`) cases, plus a `buildDag`/`deriveGates` test asserting
  exactly 1 gate for the `hevy-sync → workouts-progress` edge.
