# T156 — Workflow manager: run independent stages in parallel (raise default concurrency from 1)

## What I changed

The DAG executor (`executeDag`) already supported bounded parallelism — it launches
every ready stage up to `concurrency` at a time. The only thing forcing strict
sequential execution was `runWorkflow` passing `concurrency: def.maxConcurrency ?? 1`.

1. **`src/core/workflow-executor.ts`** — added `DEFAULT_WORKFLOW_CONCURRENCY = 4`
   (with a doc comment) and changed the executeDag call to
   `concurrency: def.maxConcurrency ?? DEFAULT_WORKFLOW_CONCURRENCY`. So independent
   same-wave stages now run concurrently (up to 4) by default; strictly-linear
   workflows are unaffected (only one stage is ever ready at a time).
2. **`src/core/types.ts`** — updated the `maxConcurrency` doc comment: default is now
   4 (was 1); a workflow can raise it for a wide fan-out or set `1` to force serial.
3. **`src/jobs/movies/movies.workflow.ts`** — changed `maxConcurrency: 1` → `4` and
   rewrote the header comment. The movies DAG has `franchise-gaps` + 8 recommender
   branches all depending ONLY on `movie-snapshot`, so they now fan out 4-at-a-time
   once the snapshot finishes instead of running one-after-another. Capped at 4 (not
   8) to protect the Mac Mini — each branch is its own child process invoking Claude.
4. **`CLAUDE.md`** — documented the new default + the `maxConcurrency` override + the
   service-quota-is-the-governor note in the architecture bullets.

## Service coordination under parallelism (verified by reasoning + existing tests)

Parallelism does NOT loosen spend governance. Paid/metered stages route their calls
through `callService` → the shared SQLite `service_usage` meter, which coordinates
rate limits + monthly quotas GLOBALLY across processes (it's keyed by service name,
not by process). So N parallel branches hitting `claude-cli`/`gemini`/`tmdb` still
respect the one shared limit; a hit quota throws `QuotaExceededError` regardless of
how many branches run at once. The service quota is the governor, not the concurrency
cap. (`src/core/services.test.ts` already covers the cross-process quota soft-fail;
the cap only bounds how many children spawn, never the metered ceiling.)

## Tests

- **New** `workflow-executor.test.ts` test: `parallelism (T156): independent stages
  run concurrently by DEFAULT; maxConcurrency:1 forces sequential; a dependent still
  waits`. It runs the SAME 4 slow stages (a new `slow-*` fake-child that succeeds
  after 250ms) under three shapes and asserts RELATIONSHIPS between wall-times
  (robust to machine speed, since the constant `--import tsx` per-child startup
  inflates the serial run ~4× but the concurrent run only ~1×):
    - default (cap 4) is far faster than `maxConcurrency:1` (`parDefault*1.8 < parSerial`),
    - the serial run took ≈ the sum of all four stages (`> SLOW_MS*3`),
    - the dependency-gated run adds a second wave (`parDep > parDefault`) yet stays
      under fully-serial (`parDep < parSerial`) — proving deps overlap AND the
      dependent waits.
- Added `slow-a..slow-d` members + a `slow*` branch to the fake-child script.
- The existing `dag.test.ts` already proves `executeDag` overlaps up to `concurrency`
  and is strictly serial at `concurrency: 1` — left as-is (covers the executor layer).

## Pre-existing failure fixed (in scope: src/jobs/movies)

`npm test` was RED on clean `main` BEFORE my change (confirmed via `git stash`):
`src/jobs/movies/stages/notify.test.ts` threw at run 1 —
`The input did not match /2 franchise gaps detected/. Input: '🎬 2 franchise gaps · 🍿 8 recommendations'`.

Root cause: an environment-specific test-isolation leak (NOT related to T156). The
"Hermetic" GAPS-only runs (1–5) passed an isolated `gapsFile`/`reportDir` but NOT
`recsFile`/`historyFile`, so `runNotify` fell back to the REAL production
`moviesConfig.recsOut` on this Mac Mini — which holds 8 real recommendations — making
the digest recs-aware and the title machine-dependent. On CI (no `data/`) the file is
absent so it passed; locally it failed. Fixed by passing isolated non-existent
`recsFile: noRecsFile` + `historyFile: noHistoryFile` (under the test's temp dir) to
the 5 gaps-only `runNotify` calls, so they deterministically see zero recs as intended.
