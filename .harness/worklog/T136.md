# T136 — Durable reviews: owner-owned `.harness/reviews.json`, commit+push under the loop lock

## ⚠️ Why this task was reverted THREE times (the real root cause)

This task had **three** prior build commits today — `cd0e953`, `be952f4`,
`d9abce7` — each **auto-reverted by the loop** ~21 minutes later
(`0e9e23a`, `cdefb01`, `0ecab37`). The earlier worklog blamed a "CI/infra
hiccup". **That was wrong.** The implementation was always correct and green
locally; the revert had nothing to do with the code.

**The actual cause: the build commit MESSAGE contained the literal string
`[skip ci]`.** All three commit messages described the feature with a line like
"…the POST endpoint writes it atomically AND commits+pushes it `[skip ci]` under
the SAME mkdir lock…". GitHub Actions scans the **commit message** for CI-skip
tokens (`[skip ci]`, `[ci skip]`, `[no ci]`, `[skip actions]`, `[actions skip]`),
and on a match it **creates NO workflow run at all**.

Confirmed empirically with `gh`:
- `gh run list` shows a CI run (all `success`) for every commit on `main`
  EXCEPT the three T136 build commits — for `d9abce7`/`be952f4`/`cd0e953` the
  query `select(.headSha==<sha>)` returns **0 runs**.
- The loop's `wait_ci_green` polls for a run matching the pushed SHA for up to
  `CI_TIMEOUT=1200s` (20 min). No run ever appeared, so it returned
  *indeterminate*, and the loop's `if wait_ci_green; then … else revert` treated
  that the same as red → `git revert HEAD` + push. Hence the exact ~21-min gap on
  all three attempts, and hence the revert commits themselves DID get green CI
  runs (their messages don't contain the token).

**The fix is purely the commit message: it must NOT contain any CI-skip token.**
(File CONTENTS may contain `[skip ci]` freely — GitHub only scans the message,
not the diff — so CLAUDE.md/HARNESS.md/README/this worklog keep the token in
prose, and the daemon's own runtime reviews commit deliberately keeps
`reviews: <id> reviewed=<bool> [skip ci]` so its trivial data commits skip CI.)

## Note for the loop / future builders

When a commit's SUMMARY or BODY needs to mention the `[skip ci]` marker, do NOT
write the bare token — it suppresses the CI run for that push and the loop will
revert the commit after a 20-min no-run timeout. Describe it instead (e.g.
"a CI-skipping marker") in the commit message. File contents are fine.
