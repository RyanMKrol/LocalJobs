## What's implemented (spec coverage)

1. **Owner-owned store** — `.harness/reviews.json` (committed, NOT gitignored;
   `.harness/reviews.json.tmp*` IS ignored). One-time migration applied to the real
   files: `reviewed` stripped from all 136 tasks in `TASKS.json` (verified
   `grep -c '"reviewed"'` → 0; `jq '.tasks|length'` → 136; every other field
   preserved), reviews.json seeded `{}` (no task was `reviewed:true`). `git
   check-ignore .harness/reviews.json` → not ignored.

2. **Overlay** — `readBacklog(path, reviewsPath)` reads `readReviews()` and sets
   `reviewed = reviews[id]?.reviewed ?? false`; tasks no longer carry the field.
   `specContent` inlining unchanged.

3. **Atomic reviews write** — `setReviewEntry` (pure, field-scoped, stamps `at`,
   empty raw → `{}`), `writeReviewEntry` (temp-file + rename). `migrateReviewsOut`
   (pure one-time migration helper, generic).

4. **Commit+push under the loop lock** — new `src/core/repo-lock.ts`:
   `resolveRepoPaths` derives `<git-common-dir>/<basename(root)>-loop.lock`
   byte-identically to `loop.sh` (documented on both sides); `acquireRepoLock`
   is the mkdir-based lock with pid file + stale-pid reclaim + idempotent release.
   `commitReviewsFile` stages ONLY `.harness/reviews.json` explicitly, commits
   `reviews: <id> reviewed=<bool> [skip ci]`, then fetch+rebase+push with a
   bounded 3-try retry; failed push → non-fatal `warning`; lock released in
   `finally`; whole phase timeout-bounded. In-daemon POSTs serialized by an
   in-process mutex (`serializeReview`). `execFile`/`execFileSync` arg arrays only.

5. **API** — `POST /api/backlog/:id/reviewed` validates `T\d+` (400 otherwise),
   writes atomically + commits+pushes, returns `{ ok, id, reviewed, committed,
   pushed, warning? }`. Behind the existing loopback/token guard.

6. **Dashboard** — `api.ts markReviewed` return type gains
   `committed?/pushed?/warning?`; the Backlog page surfaces a small inline note on
   `committed && pushed === false`.

7. **Docs** — HARNESS.md §8 (reviewed lives in owner-owned reviews.json; overlay +
   commit+push under the loop lock; writers fully decoupled), CLAUDE.md (T124
   wording rewritten; new-task JSON no longer carries `reviewed`; repo-lock.ts in
   the file map), README.md (Mark reviewed now durable + pushed + conflict-free),
   LIMITATIONS.md (T136 supersedes the T124 race/durability note).

## Hardening added this attempt

`git()` in `server.ts` now runs with `GIT_TERMINAL_PROMPT=0` and
`GIT_SSH_COMMAND='ssh -oBatchMode=yes'`, so a `git push` can never block on an
interactive credential / SSH prompt and pin the loop lock until the timeout —
belt-and-braces with the existing timeout bound. Pure defense; no behaviour change
on the success path.

## Notes

- The local commit is the durability guarantee; push is best-effort with retry. A
  daemon review-commit while the loop holds the lock for a whole build simply waits
  up to its timeout then defers with a non-fatal warning — the entry is already on
  disk and goes out on the next successful push. Acceptable per spec.
- Reviews.json is a disjoint git path from everything the loop commits
  (TASKS.json / worklog) → merges are always clean; the two writers can't conflict.
