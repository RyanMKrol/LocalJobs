# T040 — Mobile dashboard styling pass

## Goal
Make the dashboard responsive at an iPhone-17-class (~402px) viewport, with an
automated headless-browser check covering every page that asserts no horizontal
overflow / no boundary-crossing, run locally with page-by-page results recorded.

## What I built

### 1. Responsive CSS (`dashboard/app/globals.css`)
Added one `@media (max-width: 640px)` block (covers ~402px) plus two global
wrapping rules:
- `.mono`, `.logs` → `overflow-wrap: anywhere` so long URLs / file paths / item
  keys can't widen the page.
- Mobile block: tighter `.container` padding; header `.inner` + nav wrap; stat
  cards 4→2 columns; **`.panel:has(> table) { overflow-x: auto }`** so every
  table-bearing panel scrolls sideways *within itself* (page never scrolls);
  smaller `th/td` padding; `.kv` collapses to a single stacked column; DAG nodes
  drop their 156px min-width.

### 2. Explicit viewport (`dashboard/app/layout.tsx`)
Added `export const viewport = { width: 'device-width', initialScale: 1 }` so the
mobile media queries actually engage on phones.

### 3. Automated check (`dashboard/scripts/mobile-check.mjs`)
- Boots a production `next start -p 4799`, drives headless Chromium (Playwright,
  already a root dep) at a 402×874 mobile viewport.
- **Hermetic:** intercepts `**/api/**` and serves synthetic in-process fixtures
  (no daemon, no SQLite, no paid calls). Fixtures are stuffed with long/adversarial
  strings (long cron, long URLs, long item keys, long errors) to stress layout.
- Assertions per page: (1) `documentElement.scrollWidth - clientWidth <= 1px`
  (no horizontal page overflow); (2) no element with `overflow-x: visible` has
  `scrollWidth - clientWidth > 1px` (no content crossing its own boundary —
  intentional `overflow-x:auto` scroll panels and `.logs` are correctly exempt).
- Local-only, NOT added to CI (per task). Run: `node dashboard/scripts/mobile-check.mjs`.

### Teeth-test (proving the check isn't a no-op)
Temporarily stripped the `@media` block, rebuilt, re-ran: the check FAILED loudly,
e.g. overview **281px** horizontal overflow with 6 boundary-crossing elements
(`.statcards` +70px, the stuck-items `.panel` +302px, the runs `.panel` +177px,
`header.site` +55px); `/workflows` +197px; `/jobs/...` +141px; etc. Restored the
block + rebuilt → back to 0px / all pass. So the assertions genuinely catch
overflow.

## Notes / decisions
- The DB page's intentional cell ellipsis (`overflow:hidden; text-overflow:ellipsis`
  with a `title` tooltip) is by-design truncation and correctly NOT flagged — the
  check only flags content spilling an `overflow-x: visible` box.
- Did NOT restart launchd agents (headless build task — the loop owns push/CI;
  no live daemon to restart in this context).
- Trade-off recorded in `.harness/LIMITATIONS.md`; docs updated in `README.md`
  (new "Mobile styling check" section + responsiveness note) and `CLAUDE.md`
  (file map + a "Dashboard must stay mobile-responsive" convention).
