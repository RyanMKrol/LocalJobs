# T135 — Editable workflow schedule (persisted, live, reconciled)

## Goal
Make a workflow's cron schedule editable from `/workflows/<name>`: persisted, user-owned
(reconciled across code-sync like `enabled`/service limits), and applied to the live
scheduler with NO daemon restart.

## What I built
1. **DB + reconcile** (`src/db/schema.sql`, `src/db/index.ts`, `src/db/store.ts`):
   - Added `schedule_overridden INTEGER NOT NULL DEFAULT 0` to the `workflows` CREATE TABLE
     (fresh DBs) + an additive `ALTER TABLE` migration in `openDb()` (existing DBs, per the
     T098 rule — no bootstrap index on the new column).
   - `upsertWorkflowStmt` now reconciles:
     `schedule = CASE WHEN schedule_overridden = 1 THEN schedule ELSE excluded.schedule END`
     (exactly the `limits_overridden` pattern), so a code-sync preserves a user override.
   - `updateWorkflowSchedule(name, schedule)` sets `schedule` + flips `schedule_overridden = 1`;
     blank/empty input → `null` (manual-only). Returns the row, or undefined for unknown wf.
   - `schedule_overridden` surfaced on `WorkflowRow`.
2. **Scheduler** (`src/core/scheduler.ts`):
   - `startScheduler` now reads the EFFECTIVE schedule from the DB row (override-or-default),
     not `def.schedule` directly. Added `effectiveSchedule()` helper.
   - `rescheduleWorkflow(name, schedule)` stops+removes the existing `Cron`, and if non-null
     registers a fresh one (def looked up from the registry, same fire callback). Null →
     unregistered (manual-only). Live, no restart. `nextWorkflowRun` still reads the live cron.
3. **API** (`src/api/server.ts`):
   - `POST /api/workflows/:name/schedule { schedule }` — guarded by the global loopback/token
     mutation check. 404 unknown wf; blank → manual-only; validates a non-empty value with
     `new Cron(expr, { paused: true })` and returns 400 on a bad expression (never crashes the
     daemon). On success: `updateWorkflowSchedule` → `rescheduleWorkflow` → `{ ok, schedule, next_run }`.
4. **Dashboard** (`dashboard/app/lib/api.ts`, `dashboard/app/workflows/[name]/page.tsx`,
   `dashboard/app/globals.css`):
   - `api.updateWorkflowSchedule(name, schedule)` — bespoke fetch that surfaces the server's
     400 `error` body as the thrown message (the generic `post` discards it).
   - Schedule row now has an inline Edit → text input (prefilled, blank=manual-only) → Save/Cancel.
     Shows the 400 validation error inline; closes on success and lets the 3s poll refresh the
     CronBadge + Next run. Note: `usePoll` (in out-of-scope `ui.tsx`) has no `refetch`, so I
     relied on the existing 3s poll rather than touch ui.tsx.
   - New understated CSS: `.schedule-view/.schedule-edit/.schedule-input/.schedule-edit-link/
     .schedule-err/.btn-sm/.btn-ghost`. Phone-width safe (flex-wrap).
   - `Workflow.schedule_overridden?` added to the api type.

## Tests added
- `src/db/store.test.ts`: reconcile preserves an overridden schedule across re-sync; set +
  clear-to-null; unknown wf → undefined.
- `src/db/migrate-existing-db.test.ts`: asserts `schedule_overridden` column exists + defaults 0
  on a pre-seeded old-shape DB (regression guard stays green).
- `src/core/scheduler.test.ts`: `rescheduleWorkflow` registers a cron for a manual-only wf,
  clears one to manual-only, and swaps an existing cron's next-run.
- `src/api/server.test.ts`: valid cron → 200 + persisted + overridden + next_run + live cron;
  empty → manual-only (null schedule/next_run, cron removed); invalid → 400 (not persisted, no
  cron); unknown wf → 404.
