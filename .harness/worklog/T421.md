# T421 ‚Äî places: throw at run end when any item genuinely fails (resolve/enrich/llm-enrich)

## What changed

Applied the exact patches from the spec to all three stages:

- `resolve.ts` (`runResolve`): after persisting `resolved.json` and logging the
  RESOLVER SUMMARY, throws `${failedThisRun}/${todo.length} place(s) failed to
  resolve this run ‚Äî see logs above` when `mismatch + no_place_id + error > 0`.
- `enrich.ts` (`runEnrich`): after the ENRICH SUMMARY log, throws
  `${failCount}/${okCount + failCount} place(s) failed to enrich this run ‚Äî see
  logs above` when `failCount > 0`.
- `llm-enrich.ts` (`runLlmEnrich`): after the LLM-ENRICH SUMMARY log, throws
  `${fail}/${ok + fail} place(s) failed to LLM-enrich this run ‚Äî see logs above`
  when `fail > 0`.

`places-ingest` left untouched per the spec's note (it has no per-item `'failed'`
ledger outcome; `ingest.job.ts` already throws on a failing `ValidationReport`).

## Tests added (none of these 3 files had existing test coverage)

- `resolve.test.ts` ‚Äî monkey-patches `chromium.launch` (mutates the shared object
  `playwright`'s named export exposes, so `resolve.ts`'s own import sees the same
  patched function ‚Äî no real browser/network). Fakes a page whose
  `page.on('response', cb)` callback can be fed a fabricated `ChIJ...` place_id
  string to control per-item success/failure. Covers: (1) one failing + one
  succeeding place ‚Üí `runResolve` rejects naming `1/2`; (2) all succeed ‚Üí resolves
  normally; (3) the pre-loop "usage cap already reached" soft-stop (dailyCap
  forced to 0) ‚Üí resolves normally (0 items ever attempted, so no throw).
- `enrich.test.ts` ‚Äî monkey-patches `global.fetch` (the bare `fetch` call in
  `fetchPlaceDetails`). Covers: (1) one HTTP-500 + one success ‚Üí rejects naming
  `1/2`; (2) all succeed ‚Üí resolves normally; (3) nothing left to enrich ‚Üí resolves
  normally; (4) the google-places service quota already exhausted before any item
  is attempted (`registerService('google-places', { dailyCap: 0 })`) ‚Üí resolves
  normally.
- `llm-enrich.test.ts` ‚Äî monkey-patches `global.fetch` (confirmed via a standalone
  probe script that `@google/genai`'s `generateContent` ultimately calls the bare
  global `fetch`, so no real Gemini call happens). The fake responds with valid
  JSON prose for "good" cids (matched by a marker baked into the prompt body) and
  JSON-free prose for "bad" ones, which `parseResult` rejects ‚Üí the genuine
  catch-all failure branch. Covers: (1) one failing + one succeeding place ‚Üí
  rejects naming `1/2`; (2) all succeed ‚Üí resolves normally; (3) the gemini
  service quota already exhausted before any item is attempted
  (`registerService('gemini', { dailyCap: 0 })`) ‚Üí resolves normally.

All three files mutate `placesConfig`/`enrichConfig`/`llmConfig`/`resolveConfig`
in place (temp paths, temp caps) for the duration of each test and restore the
originals in a `finally`, rather than refactoring the run functions to accept an
injected options object ‚Äî matches the "no unnecessary abstraction" rule and
keeps the diff to exactly what the spec asked for.
