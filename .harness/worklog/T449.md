# T449 — Add rateLimitSource to ServiceDefinition + document all current services

## What was built

- `ServiceDefinition.rateLimitSource?: string` (src/core/types.ts) — manifest-owned provenance
  label for a service's rate/quota numbers, mirroring `category` (T305): no `_overridden` column,
  always refreshed from code on sync.
- `services.rate_limit_source TEXT NOT NULL DEFAULT ''` added to `src/db/schema.sql` (fresh DBs)
  and an additive `ALTER TABLE` migration in `src/db/index.ts` (existing DBs) — no index, per the
  T098 bootstrap-index rule.
- `src/db/store.ts`: `upsertServiceStmt` now inserts/unconditionally refreshes `rate_limit_source`
  on every `syncService` call (same treatment as `description`/`category`/`paid` — not the
  `CASE WHEN limits_overridden` treatment the three limit columns get); `ServiceRow` carries the
  new field.
- `dashboard/app/lib/api.ts`'s `Service` type gained `rate_limit_source: string` so it round-trips
  through the existing `GET /api/services` endpoint (no server.ts change needed).
- All 13 current `src/services/*.service.ts` files (the 12 named in the spec plus
  `trading212-instruments`, which had already landed from T425) now set an honest
  `rateLimitSource` string distinguishing documented-with-link vs. empirically-observed vs.
  explicit-guess, grounded in each file's own existing comments.
