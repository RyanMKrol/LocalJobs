# T285 — user-editable, code-reconciled per-workflow notification toggle

## What I built

Mirrored the T169 `maxConcurrency` mechanism end-to-end for a new `notifyEnabled`
workflow property (binary on/off only, default ON):

1. `src/core/types.ts` — `WorkflowDefinition.notifyEnabled?: boolean`.
2. `src/db/schema.sql` — `notify_enabled` (default 1) + `notify_enabled_overridden`
   (default 0) columns on `workflows`.
3. `src/db/index.ts` — additive `ALTER TABLE` migration for existing DBs (no index
   on the new columns, per the T098 rule).
4. `src/db/store.ts` — `upsertWorkflowStmt` reconciles `notify_enabled` the same way
   as `max_concurrency` (override wins over manifest on re-sync);
   `updateWorkflowNotifyEnabled(name, enabled)` persists + flips the override flag.
5. `src/core/workflow-executor.ts` — `effectiveWorkflowNotifyEnabled(def)` (DB
   override ?? manifest ?? true), read fresh in `runWorkflowInner` right before the
   single T189 aggregate `notifyWorkflow(...)` call; when false it logs
   `Notifications disabled for workflow "<name>" — skipping push notification`
   instead of calling notifyWorkflow.
6. `src/api/server.ts` — `POST /api/workflows/:name/notify { notifyEnabled }`
   (boolean validated, 400/404 as appropriate, same mutation guard as
   `/toggle`/`/schedule`/`/concurrency`); `GET /api/workflows/:name` now also
   returns `effective_notify_enabled`.
7. `dashboard/app/lib/api.ts` — `updateWorkflowNotify` client method + `Workflow`
   type fields.
8. `dashboard/app/workflows/[name]/page.tsx` — click-to-toggle "Notifications" row,
   same affordance as the existing `enabled` toggle (no separate Save button).
9. `CLAUDE.md` / `README.md` — documented the new property alongside
   schedule/maxConcurrency.

## Tests added

- `src/db/store.test.ts`: `updateWorkflowNotifyEnabled` set/override-reconcile
  across `syncWorkflow`, unknown-workflow → undefined.
- `src/api/server.test.ts`: POST `/notify` valid value (200, persisted, surfaced on
  GET), non-boolean bodies rejected 400, unknown workflow 404.
- `src/core/workflow-executor.test.ts`: a workflow with `notifyEnabled: false` skips
  the actual push (verified via the existing `_setFetchForTest` ntfy interception —
  zero push titles) but logs the skip line; `updateWorkflowNotifyEnabled` override
  is reflected immediately in `effectiveWorkflowNotifyEnabled`.
- `src/db/migrate-existing-db.test.ts`: pre-existing workflow row ends up with
  `notify_enabled = 1` (default ON) and `notify_enabled_overridden = 0` after
  migration.

## Notes

- This is a binary on/off switch only, per the spec — no "only notify on failure"
  granularity was added (deliberately deferred).
- The gate sits at the single T189 aggregate `notifyWorkflow` call site in
  `runWorkflowInner`, not per-stage (there is no per-stage notification anymore
  since T189).
