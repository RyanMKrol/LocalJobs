# T426 — gate vercel-redeploy through a real `vercel` service

## What was built

- New `src/services/vercel.service.ts`: a self-contained `ServiceDefinition` named `vercel`,
  `category: 'cli-tool'` (mirroring `claude-cli`'s precedent for a CLI-tool invocation, not an
  HTTP API), `paid: false`, conservative `dailyCap`/`monthlyCap` (defaults 3/30, env-overridable
  via `VERCEL_DAILY_CAP`/`VERCEL_MONTHLY_CAP`). No `ratePerMinute`/`minIntervalMs` — a deploy
  takes minutes by itself, so a per-call throttle is meaningless; the day/month caps are the sole
  governor.
- `vercel-redeploy.job.ts`'s `runVercelRedeploy` now wraps the whole spawn-and-await-exit
  operation (extracted into a `deployOnce` helper) in `callService('vercel', ...)` (an injectable
  `callServiceFn` opt defaults to the real `callService`, matching this repo's existing
  DI-for-tests convention). A thrown `QuotaExceededError` is caught and logged as a clean WARN
  ("no deploy attempted today, budget exhausted") instead of failing the run — mirrors the
  existing soft-skip for an unconfigured `RYANKROL_CO_UK_PATH`.
- Tests added to `vercel-redeploy.test.ts`:
  - "gates the deploy through callService("vercel", ...)" — injects a spy `callServiceFn` and
    asserts it's called exactly once with the service name `'vercel'`.
  - "soft-skips with a warn log (not a throw) when the vercel service quota is exhausted" —
    injects a `callServiceFn` that throws `QuotaExceededError` and asserts the deploy never
    spawns and a warn log is emitted instead of a rejection.
- `.env.example`: documented `VERCEL_DAILY_CAP`/`VERCEL_MONTHLY_CAP` (commented out, showing
  defaults) under the existing `vercel-daily-redeploy workflow` section.
- `src/workflows/vercel-daily-redeploy/CLAUDE.md`: added a short section describing the new
  service gating and removed the stale "not yet gated... T426 still pending" note.

## A real bug caught + fixed during verification

Running the FULL `npm test` suite (not just this file) surfaced 3 pre-existing test failures in
`vercel-redeploy.test.ts` that passed when the file was run in isolation. Root cause: elsewhere in
the full suite the job registry discovers and registers every `*.service.ts` (including the new
`vercel.service.ts`), so by the time `vercel-redeploy.test.ts` runs, `callService('vercel', ...)`
is REAL (not a no-op for an unregistered service) and shares the scratch DB's real
`service_usage` quota counter across every test in the file that doesn't stub it. With
`dailyCap: 3`, the 4th+ real (unstubbed) call in the file silently hit the quota and — because of
the new catch-and-warn behaviour I just added — returned cleanly instead of exercising the actual
spawn/timeout/streaming logic those tests are about. Fixed by giving the pre-existing
spawn-behaviour tests (that don't care about service gating) an explicit no-op `bypassCallService`
stub, so they stay hermetic and don't consume/depend on the shared quota. Full suite now passes
210→213 tests, 0 failures.
