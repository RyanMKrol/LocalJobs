# T037 — Drop standalone jobs: everything is a pipeline

## Goal
Remove the standalone (non-pipeline) job concept. Every job must belong to a
pipeline declared in an explicit `*.pipeline.ts` manifest (a lone job = a one-stage
pipeline with its own manifest — no implicit wrapping). A job with no manifest is a
config error that fails loud at load. Pipelines own scheduling + the enable toggle.

## What I changed
- **`src/jobs/registry.ts`** — added pure, exported `orphanJobNames(jobs, pipelines)`
  (job names belonging to no pipeline). After discovery, the registry computes
  orphans and **throws** if any exist, so the daemon refuses to start with an
  orphaned job. Reworded `memberJobNames` doc.
- **`src/core/scheduler.ts`** — removed the standalone-job scheduling path entirely
  (`scheduleJob`, the `jobs`/`memberJobNames` loop, the per-job `crons` map). The
  scheduler now only registers crons for pipelines. `nextRun(jobName)` is kept (the
  job API view calls it) but always returns `null` — jobs never have their own cron.
- **`src/core/scheduler.test.ts`** — rewritten for the pipeline-only world: scheduled
  pipelines get a cron, the enabled one fires (overlap-skipped via a pre-seeded
  running pipeline run), the disabled one is gated at fire-time and creates no runs,
  a manual (null-schedule) pipeline gets no cron, and a member job never gets its own
  cron (`nextRun` → null).
- **`src/core/job-membership.test.ts`** (new) — unit tests for `orphanJobNames`
  (member ok, lone one-stage pipeline ok, orphan reported, zero-pipelines → all
  orphans, multi-claim ok).
- **`dashboard/app/pipelines/page.tsx`** — removed the entire "Standalone jobs"
  section + its now-unused `jobs()` poll, `standaloneJobs` filter, and `StatusBadge`
  import.
- **Docs** — README.md (Triggering/Adding-a-job now require a pipeline manifest;
  removed the Standalone-jobs dashboard mention), CLAUDE.md (architecture bullet +
  diagram, file map for scheduler/registry, "How to add a job" now adds a manifest,
  new Conventions rule), and `.harness/LIMITATIONS.md` (per-job enabled/schedule are
  now vestigial — left in place to avoid touching the out-of-scope job detail page +
  schema columns).

## Scope decisions
- `src/db/store.ts` and `src/api/server.ts` were in scope but needed no change: the
  job toggle endpoint + `setJobEnabled` are still used by the (out-of-scope) job
  detail page, so I left them. Their effect on scheduling is gone (the pipeline gates
  runs now), which is the real "standalone path" removed. Logged as a vestige in
  LIMITATIONS rather than ripping out an out-of-scope UI page + DB columns.
- Did not touch `JobDefinition.schedule` / `jobs.schedule` / `jobs.enabled` columns
  (types.ts + schema.sql are out of scope). The fields are simply ignored now.
