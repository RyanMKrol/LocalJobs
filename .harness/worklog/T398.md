# T398 worklog

## Root cause
`.panel + .panel { margin-top: 28px; }` (globals.css) is a block-flow sibling-margin rule
intended for stacked panel lists (category-grouped Workflows/Integrations pages), but CSS's
adjacent-sibling combinator matches by DOM adjacency regardless of layout mode — so it also
applied inside `.grid.cards` on the Overview page. `.grid` has no explicit `align-items`, which
defaults to `stretch` for grid items, so every row's track height is set by the tallest item's
margin box; the very first `.panel` card (no preceding sibling, no margin) kept its full content
height and set the row height, while every other card got the 28px top margin eaten into its
stretched content box, visually compressing it.

## Fix
Added a scoped override in `dashboard/app/globals.css`:
```css
.grid.cards > .panel + .panel { margin-top: 0; }
```
placed right after the existing `.grid`/`.cards` rules, with a comment explaining why `.grid`
layouts already space via `gap` and must not inherit the block-flow sibling margin.

## Fixture
Added a `claude-warmer`-shaped workflow entry (name `claude-warmer`, category
`regular-maintenance`, schedule `*/30 * * * *`, `jobs: singleStageMembers`) as the FIRST entry
in the `/api/workflows` fixture array in `dashboard/scripts/_dashboard-harness.mjs`, reproducing
the exact real-world reported symptom (claude-warmer is alphabetically first among the owner's
real workflows).
