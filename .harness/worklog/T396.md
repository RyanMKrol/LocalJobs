# T396 — fleet-wide run-all endpoint

## What I built
- `POST /api/workflows/run-all` in `src/api/server.ts` (registered right after the
  existing single-workflow `POST /api/workflows/:name/run` handler): iterates every
  workflow from `listWorkflows()`, skips any with `workflowRunInProgress(name) ===
  true` (recorded as `{ status: 'skipped', reason: 'already running' }`), and
  fire-and-forgets `runWorkflow(def, 'manual', ...)` for the rest — `{ limit }` when
  the workflow has a stage declaring `inputKeys()` (limitable), or `{}` (unlimited)
  otherwise. Optional body `{ limit }` defaults to 3, validated as a positive
  integer (400 otherwise). Responds `202` with `{ ok, totalWorkflows, startedCount,
  skippedCount, limit, results[] }` and logs one summary line via `console.log`.
  Behind the same `authoriseMutation` guard as every other mutating endpoint.
  Includes disabled workflows (manual runs already bypass `enabled`, matching
  `reset-output-all`'s behaviour).
- `dashboard/app/lib/api.ts`: added `api.runAllWorkflows(limit?)` client helper
  POSTing to the new endpoint, typed to match the response shape.

## Tests
- New unit test block in `src/api/server.test.ts` (mirrors the T105 guard test
  fixture pattern): seeds `ra-lim-wf` (limitable), `ra-plain-wf` (non-limitable),
  and `ra-busy-wf` (marked running via `createWorkflowRun` + cleaned up via
  `finishWorkflowRun`). Asserts the busy workflow is skipped with the right reason,
  the limitable workflow starts with `limited: true, limit: 3`, the non-limitable
  one starts with `limited: false, limit: null`, and the aggregate counts /
  `results[]` shape are correct. A second test asserts a negative limit is
  rejected with 400.
