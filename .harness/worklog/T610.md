# T610 — Add GET /api/cache/rows endpoint exposing service_cache row detail

2026-07-16

## What was done

- Added `GET /api/cache/rows[?service=<name>]` to `src/api/server.ts`'s route table,
  right after the existing `GET /api/cache` (row-counts) entry.
- It calls `listServiceCacheRows(serviceName)` (added by T609) and, per row, computes a
  server-side `live: boolean` by comparing `Date.now() - Date.parse(row.cached_at)`
  against that row's service's resolved `cacheTtlMs`, looked up via `getServiceDef` from
  `src/core/services.ts` (the same in-process service registry `effectiveLimits`/
  `effectiveServiceTimeoutMs` consult). A service with no configured `cacheTtlMs` treats
  every row as live (nothing would ever purge it), per the owner-confirmed spec.
- Response shape: `{ rows: [{ service_name, cache_key, cached_at, response_json, live }] }`
  — `response_json` is returned as the raw string from the store, unparsed (a UI concern).
- Read-only GET, no loopback/token mutation guard — mirrors the existing `GET /api/cache`.
- Note: `service_cache.cached_at` is written via `new Date().toISOString()` in
  `setCachedServiceResponse` (a full ISO string with trailing `Z`), NOT via SQLite's
  `datetime('now')` — so, unlike some other tables in this file, no `+'Z'` parse
  workaround is needed/correct here. Confirmed against `getCachedServiceResponse`'s own
  `Date.parse(row.cached_at)` in `src/db/store/services.ts`, and against a failing test
  first (appending `'Z'` broke the "fresh row is live" case with `Date.parse` returning
  `NaN` on a doubled-`Z` string).

## Tests

Added two new tests to `src/api/server.test.ts` (T610 block, after the existing T478
cache tests): seeds `t610-svc-a` (cacheTtlMs 60_000ms, one fresh + one row backdated 5
minutes past TTL) and `t610-svc-b` (no cacheTtlMs, one row backdated 24h) directly via
`setCachedServiceResponse` + a direct `service_cache.cached_at` UPDATE (same pattern as
`src/core/services.test.ts`'s existing TTL-expiry test), then asserts:
- unfiltered `GET /api/cache/rows` returns all 3 rows, each with all 5 fields present
  and the correct types, `live: true` for the fresh row, `live: false` for the stale
  row, and `live: true` for the no-TTL row regardless of age.
- `response_json` round-trips via `JSON.parse` to the original value.
- `GET /api/cache/rows?service=t610-svc-a` returns only that service's rows.
