# T139 ‚Äî Run-scope the workflow-run Input‚ÜíOutput panel (`work_item_runs` linkage)

## What was built

The run-page Input‚ÜíOutput panel previously read the ENTIRE `work_items` ledger for the
first/last-wave jobs and joined by `root_key` with NO run filter ‚Äî so it showed every input the
workflow had EVER processed and apologised for it ("First-cut mapping: reflects the global
work-item ledger‚Ä¶"). Now it is genuinely scoped to the run you're viewing.

1. **`work_item_runs` linkage table** (`src/db/schema.sql`) ‚Äî append-only
   `(workflow_run_id, job_name, item_key, root_key, at)`, `UNIQUE(workflow_run_id, job_name,
   item_key)`, + `idx_work_item_runs_run`. It's a BRAND-NEW table so the index lives in the schema
   bootstrap (columns exist on creation for fresh AND existing DBs ‚Üí does NOT hit the T098 trap;
   verified by running `migrate-existing-db.test.ts`, which seeds an OLD-shape DB and survives
   because `CREATE TABLE IF NOT EXISTS` adds the table).

2. **`markWorkItem`** (`src/db/store.ts`) ‚Äî new optional `workflowRunId` opt defaulting to
   `process.env.LOCALJOBS_WORKFLOW_RUN_ID || null`. When present, after the `work_items` upsert it
   INSERT-OR-REPLACEs a linkage row using the SAME resolved `root_key`. Existing call sites are
   UNCHANGED (env default); a standalone run (no env) records nothing.

3. **`workItemIoRows(first, last, workflowRunId?)`** now returns `{ rows, scoped }`. With a run id:
   takes the distinct `root_key`s from `work_item_runs` for that run, resolves each root's input
   (first-wave ledger) + output (last-wave ledger) from the cumulative ledger (so an output from an
   EARLIER run still shows). No linkage ‚Üí empty + `scoped:false` (NEVER a global dump). With no run
   id ‚Üí legacy un-scoped behaviour preserved. New `workflowHasRunLinkage(name)` helper.

4. **API** `GET /api/workflow-runs/:id/io` passes `run.id` in, returns `scoped` + `emptyReason`
   (`no-new` | `pre-feature` | null, derived via `workflowHasRunLinkage`) + an honest `note`
   ("One output is shown per input‚Ä¶" ‚Äî no more "first cut").

5. **Dashboard** (`workflow-runs/[id]/page.tsx`) ‚Äî IoPanel reworded (no "first cut" anywhere); the
   caveat is now a clean `.io-footnote` (new minimal CSS, subtle top-border footnote) shown only
   when there ARE rows; empty states render "This run processed no new items" or "Per-run
   input/output isn't recorded for runs created before this feature."

## Trade-off recorded

`.harness/LIMITATIONS.md` ‚Äî the "no new items" vs "pre-feature" empty-state distinction is a
heuristic (a pre-feature run on a workflow that has since had linked runs is mislabelled "no new
items"). Cosmetic; both messages explain the empty panel and neither dumps the ledger.
