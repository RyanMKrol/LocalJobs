## 2026-07-03 — built

### What changed

- `dashboard/app/lib/api.ts`: added `format?: string` to `WorkflowRunOutput` — the render
  form the backend (T262) declares for a fetched output item (`'markdown'` when unset).
- `dashboard/app/components/WorkflowOutputSection.tsx`:
  - Extracted the markdown-specific rendering (frontmatter parse + `ReactMarkdown`) out
    of the modal shell into `MarkdownOutputBody`.
  - Added `RawOutputBody` — a plain `<pre>` fallback for any form with no dedicated
    renderer yet, so a new form is usable the moment its backend support lands, even
    before a bespoke renderer is written.
  - Added `OUTPUT_RENDERERS: Record<string, (props) => ReactElement>` — the dispatch
    table keyed by `WorkflowRunOutput.format`, currently `{ markdown: MarkdownOutputBody }`.
    `renderOutputBody` picks the matching renderer or falls back to `RawOutputBody`.
  - Renamed `MarkdownModal` → `OutputModal` (now format-agnostic: renders loading /
    not-found / delegates to `renderOutputBody`).
  - Renamed `openMarkdown` → `openOutput` (the fetch call itself was already
    format-agnostic; only the naming was markdown-specific).
- Adding a new form (e.g. T263's structured size table) is now a two-line addition: a
  new renderer function + one `OUTPUT_RENDERERS` entry — no other code changes.

### No regression

`places`/`perfumes` output items have no `detail.format` recorded, so T262's backend
defaults an absent format to `'markdown'` — they hit the exact same `MarkdownOutputBody`
renderer with byte-identical markup/behaviour (frontmatter table, `ReactMarkdown` +
`remark-gfm` body) as before this refactor.
