# T312 — Add a global Logs nav tab + page

## What I built

- `dashboard/app/lib/api.ts`: added `GlobalLogLine` (matching the backend
  `GET /api/logs` shape verbatim) and `api.globalLogs(params)`, backed by a new
  local `buildQuery()` helper (no existing multi-param query-string builder was
  present, so this is now the pattern others can reuse).
- `dashboard/app/logs/page.tsx` (new): client page polling `api.globalLogs` via
  the shared `usePoll` hook at 5000ms (matches the interval other list-style
  pages already use, e.g. workflows/services). Filter controls: level pills
  (info/warn/error, multi-toggle, reusing `.io-filter-bar`/`.io-filter-chip`),
  an "Errors only" quick toggle (pure UI sugar — sets `levels=['error']`, no
  new backend param), a job/workflow scope `<select>` (options populated from
  `api.jobs()`/`api.workflows()`, no new endpoints), a debounced (400ms)
  free-text search box, and a 24h/7d window selector. Log lines render
  newest-first reusing the existing `.logs`/`.lvl-*` classes from the per-run
  logs view, each linking to `/runs/<runId>` or `/workflow-runs/<workflowRunId>`
  when present. A "Load more" button calls `before=<nextCursor>` and appends
  results; disabled when `nextCursor` is null. Empty state: "No log lines match
  the current filters."
- `dashboard/app/layout.tsx`: added `<a href="/logs">Logs</a>` after Backlog in
  the top nav (one-line addition, no restructuring).
- `dashboard/app/globals.css`: added a `select` base style (none existed yet,
  matching the existing `input[type=text]` dark-theme idiom) and two small
  layout classes for the logs filter panel wrapper. Reused `.io-filter-bar`/
  `.io-filter-chip`/`.logs`/`.lvl-*`/`.panel`/`.btn` — no duplicate pill/badge
  CSS.
- `dashboard/scripts/_dashboard-harness.mjs`: added a `/api/logs` fixture with
  6 synthetic rows covering both `source: 'job'` and `source: 'workflow'` and
  all three levels; added `{ name: 'logs', path: '/logs' }` to `PAGES`; added a
  `logs-errors-only` FLOWS entry that clicks the "Errors only" toggle.

## Notes

- The backend's `level` query param is a single comma-joined string (not
  repeated `level=` params) — `api.globalLogs` passes `levels.join(',')`,
  matching `src/api/server.ts`'s parsing (`levelParam.split(',')`).
- `job` and `workflow` scope filters are mutually exclusive per the backend
  (400 if both set); the page's single `scopeType` select enforces this by
  construction (only one of `job`/`workflow` is ever set at a time).
- Stayed strictly within the task's scope list; no other files touched besides
  this worklog.
