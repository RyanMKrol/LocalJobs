# T272 — migrate workflow-run page to shared UI components

## What changed
`dashboard/app/workflow-runs/[id]/page.tsx`: the top-of-page workflow-run status badge
(previously a raw `<span className={`badge ${run.status}`}>{statusLabel(run.status)}</span>`)
now uses the shared `<StatusBadge status={run.status as RunStatus} />` from `ui.tsx` (already
imported and already used for the member-run rows on this same page). Cast to `RunStatus` since
`run.status` is `WorkflowRunStatus` (a subset save `partial`, which `STATUS_LABELS`/`STATUS_EMOJI`
already cover) — no runtime behaviour change.

## What was NOT swapped, and why
- `RunButton` doesn't apply: this page has no run-trigger action, only a Cancel button (danger
  variant with different semantics — no shared "danger action" component exists anywhere in the
  codebase to swap into).
- The `run_limit` chip (`<span className="badge queued">N inputs limit</span>`) is a `.badge`-class
  informational chip with custom text, not an enum-state pill — `Pill` wraps the `.pill` CSS class
  family (on/off, reviewed/unreviewed, etc.), which is a different visual system. Forcing it into
  `Pill` would change its class/appearance.
- The `IoPanel` table's `inputStatus`/`outputStatus` badges (`<span className={`badge ${status}`}>{status}</span>`)
  render the RAW status string (e.g. `success`, `failed`, `unknown`), not `statusLabel(status)`.
  `StatusBadge` renders `statusLabel(status)` (e.g. "Succeeded") plus an emoji slot — swapping these
  would change the visible text, violating the like-for-like requirement. Left as-is.
- `WorkflowOutputSection` (T205's shared component) is used by the workflow DEFINITION page for its
  generic output section — this run-detail page has its own distinct `IoPanel`/`MarkdownModal`
  keyed off run-scoped `work_item_runs` data, a different data shape; not a duplicate to collapse.
