# T294 — surface build-attempt failure history for pending backlog tasks

## What I built

- `src/api/server.ts`: added `TaskBuildFailures` type + `readTaskBuildFailures(id, baseDir)`,
  mirroring the safety/shape style of `readWorklogContent` — reads `.harness/failures.jsonl`
  (JSON-Lines, tolerates a missing file / blank lines / malformed rows), filters lines matching
  the task id, and aggregates to `{ count, latestKind, latestDetail, latestAt }` (latest picked
  by greatest `ts`), or `null` when there are no matching rows.
- Wired it into `readBacklog()`: each task gets a `buildFailures` field spread on when non-null,
  same conditional-spread pattern as `specContent`/`worklogContent`.
- `dashboard/app/backlog/page.tsx`: `BacklogTask` (in `lib/api.ts`) is out of this task's scope, so
  I read `buildFailures` defensively off the raw payload via a local `getBuildFailures()` helper
  and a local `TaskBuildFailures` type, instead of editing `lib/api.ts`. Renders a new amber
  `Pill kind="blocked"` (`⚠ N failed attempt(s)`, tooltip = `latestKind: latestDetail`) next to the
  existing buildable/human pills for any non-terminal task carrying `buildFailures`.
- `dashboard/app/globals.css`: added `.pill.blocked` — `--yellow`-based, visually distinct from
  both `.pill.buildable` (amber/green-ish) and `.pill.failed` (red).
- `dashboard/scripts/_dashboard-harness.mjs`: gave the synthetic `T001` (a Ready/buildable task) a
  `buildFailures` value so the fixture + visual-check actually exercise the new pill.

## Scope note

`dashboard/app/lib/api.ts` (the `BacklogTask` interface) is NOT in this task's scope, so I
deliberately avoided editing it — see the local type/cast in `page.tsx` instead. If a future task
wants `buildFailures` as a first-class typed field on `BacklogTask`, that file needs to be added
to scope.
