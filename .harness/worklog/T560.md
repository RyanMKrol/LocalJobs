2026-07-14

## What I did

Behavior-preserving prep for the future shared-notify extraction (R01): aligned the
tv-recs recommendation-history rows on `movies`' richer `{ tmdbId, title, year, at }`
shape.

- `src/workflows/tv-recs/stages/tv-recs-notify.ts` — `appendHistory` now pushes
  `{ tmdbId: r.tmdbId, title: r.title, year: r.year, at: now.toISOString() }` per new
  rec, identical to `movies/stages/notify.ts`'s `appendHistory` (was `{ title, year }`).
- `src/workflows/tv-recs/types.ts` — `RecsHistoryFile.recommended` row type now carries
  `tmdbId` + `at`. Made both OPTIONAL (`tmdbId?`, `at?`) so the type honestly reflects
  that LEGACY 2-field `{title,year}` rows lack them — new rows always write all four; the
  existing try/catch parse stays and reading a legacy row can't crash (nothing downstream
  reads tmdbId/at). Confirmed nothing downstream relies on the old shape: the history file
  is fed back into branch prompts as "already suggested" context, bounded by
  history-context / recent-window limits, and those consume only title/year (unchanged).
- `src/workflows/tv-recs/CLAUDE.md` — documented the aligned schema + legacy tolerance in
  the notify-stage section.
- movies UNTOUCHED (already has the target shape); no shared module extracted (that's a
  separate unit).
