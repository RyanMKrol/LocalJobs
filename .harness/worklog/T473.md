# T473 — add stage-io fixtures + PAGES coverage for remaining workflows

2026-07-10

## What was done

Extended `dashboard/scripts/_dashboard-harness.mjs` (the single hermetic test harness
shared by `mobile-check.mjs` and `visual-check.mjs`) so every shipped workflow (per
`src/workflows/*/*.workflow.ts`) has a `/stage-io` fixture set AND a `PAGES` entry
exercising its run-detail page (`/workflow-runs/<id>`), matching the existing
convention set by `stockDigest*`/`movieRecs*`/`singleStage*`.

Re-derived the full workflow list from the real manifests (15 total) rather than
trusting the task's guess:

**Already covered before this task:** `places` (id `1`), `movie-recommendations`
(`movie-recs-run`), `stocks-sync` (`stocks`), `stock-digest` (`stock-digest-run`),
`claude-warmer` (`claude-warmer-run`), plus a generic `skipped` fixture.

**Missing coverage — confirmed by reading each `*.workflow.ts` manifest for its real
DAG shape (member names + dependsOn), then added:**
- `missing-tv-seasons` — linear: `plex-tv-snapshot -> tmdb-season-check -> plex-seasons-notify`
- `workouts-sync` — linear: `hevy-sync -> workouts-progress`
- `listening-digest` — single stage: `lastfm-digest`
- `projects-sync` — linear: `github-sync -> project-summarize`
- `plex-space-saver` — single stage: `plex-space-saver-scan`
- `plex-language-fix` — 4-stage with a fan-in: `discover -> resolve -> evaluate
  (depends on both discover+resolve) -> apply`
- `plex-profiles` — single stage: `plex-profiles-build`
- `vercel-daily-redeploy` — single stage: `vercel-redeploy`
- `tv-recommendations` — `tv-snapshot` fanning out to 8 `tv-rec-*` branches ->
  `tv-rec-merge` -> `tv-recs-notify` (mirrors the existing `movie-recommendations`
  fan-out fixture shape)
- `perfumes` — linear: `perfumes-find-url -> perfumes-fetch -> perfumes-parse -> perfumes-build`

For each: added `<name>Members` (job list matching the real DAG), `<name>WorkflowRun`
+ run jobs, `<name>StageIo` (per-stage inputs/outputs keyed by job name) +
`<name>StageIoOverall` (root-wave inputs / terminal-wave outputs), wired into
`fixtureFor` under `/api/workflow-runs/<name>-run` (both `/stage-io` and the bare
run) AND `/api/workflows/<name>` (needed so the run-detail page's `DagFlow` — which
fetches the workflow DEFINITION via `api.workflow(run.workflow_name)`, not just the
run — renders the real member/dependency shape instead of falling through to the
generic `workflow()` fixture, which defaults to `places`' 3-node shape). This second
part wasn't obvious from the task's initial hint about "existing conventions" and
was caught only by visually inspecting the first screenshot batch (see below) — every
new run-detail page's DAG graph was rendering the placeholder `places` nodes instead
of the workflow's own stages until the `/api/workflows/<name>` fixture was added
alongside the `/api/workflow-runs/<name>-run` ones.

Added 10 new `PAGES` entries (`workflow-run-<name>`) pointing at each new
`/workflow-runs/<name>-run` route, `waitFor: ['.rf-dag-node']` matching the existing
convention.
