# T611 — Add cache row-browsing UI to the admin page's Cached responses panel

2026-07-16

## What was done

- `dashboard/app/lib/api.ts`: added `serviceCacheRows(service?)` next to the existing
  `serviceCacheCounts`/`clearServiceCache`, calling `GET /api/cache/rows[?service=]` (the
  endpoint T610 already shipped) and returning
  `{ rows: Array<{ service_name, cache_key, cached_at, response_json, live }> }`.
- `dashboard/app/admin/page.tsx`: extended the existing "Cached responses" panel's
  per-service count table with a per-row **Browse/Hide** toggle button. Clicking it
  fetches `serviceCacheRows(serviceName)` (cached in local state per service, only
  fetched once per session unless re-expanded) and renders a nested table directly
  below that service's row: cache key, cached at, a `<Pill kind={row.live ? 'on' : 'off'}>`
  Live/Stale badge, and a per-row `<details>` block that pretty-prints `response_json`
  (`JSON.parse` + `JSON.stringify(..., null, 2)`, falling back to the raw string on a
  parse failure — same fallback pattern as `JsonOutputBody` in
  `components/OutputRenderer.tsx`; that component isn't in this task's scope so the
  small helper `prettyPrintJson` was added locally in `admin/page.tsx` instead of
  importing it).
- The existing count table, the `cacheTotal` row, and the separate "Clear all cached
  responses" danger-zone panel are untouched — purely additive.
- New local state (`expandedCacheService`, `cacheRowsByService`, `cacheRowsLoading`,
  `cacheRowsErr`) follows the same `useState` pattern already used on this page.

## Dashboard harness (living artifact)

- `dashboard/scripts/_dashboard-harness.mjs`: added a synthetic `GET /api/cache/rows`
  fixture (two gemini rows: one fresh/live, one backdated/stale) alongside the existing
  `/api/cache` fixture.
- Added a new FLOWS entry `admin-cache-browse-expanded` (path `/admin`) that clicks the
  "Browse" button then expands the first row's `<details>`, so visual-check captures the
  interactive expanded state (not just the baseline page).
