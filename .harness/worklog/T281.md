# T281 — Surface skipped/no-op run status in the dashboard run-detail view

## Finding

Traced the run-detail view (`dashboard/app/workflow-runs/[id]/page.tsx`) and the
shared status rendering (`dashboard/app/ui.tsx`) against T258's backend status
model. The dashboard already fully supports the `skipped` status with **no code
change needed** in either scoped file, because earlier tasks already built the
generic machinery T258 needed:

- `RunStatus` (member run status) already includes `'skipped'` (`dashboard/app/lib/api.ts`).
- `STATUS_LABELS`/`STATUS_EMOJI`/`StatusBadge` in `ui.tsx` already render `skipped`
  as its own "Skipped" pill with a distinct emoji (⤼).
- `globals.css` already has `.badge.skipped` (grey, distinct from `.badge.success`
  green) and `.dag-node.skipped { opacity: .55 }`.
- The top-level workflow-run status badge (`run.status as RunStatus`,
  `workflow-runs/[id]/page.tsx:404`) was already cast to `RunStatus` (T272), so a
  `'skipped'` `WorkflowRunStatus` from T258 renders correctly at runtime.
- `completedStages` counting (`r.status !== 'queued' && r.status !== 'running'`)
  already treats `skipped` as completed, not stuck mid-run.
- The Input→Output panel (`IoPanel`) is scoped to items this run actually advanced
  via `work_item_runs` linkage (T139). A noop/skipped run advances nothing, so
  `io.length === 0` and the panel renders the existing "This run processed no new
  items." empty state — it never shows items as freshly "succeeded" for a skipped run.

So this task is a **verification task**: confirm the existing UI surfaces T258's
new status correctly, and add the missing visual-check coverage per the Living
Artifact rule (no run/fixture previously exercised a `skipped` run-detail page).

## Changes made (harness only, in scope per rule: visual-check scripts always in-scope)
