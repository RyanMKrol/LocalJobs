# T616 — Workflow Output section: show 'View' for non-markdown (JSON/path) outputs too

2026-07-16

## What I did
- `src/db/store/workItems.ts`: `workflowTerminalItems()` now computes a new
  format-agnostic `viewable` boolean on each `OutputItem` — true when
  `detail.markdown` is a non-empty string OR (`detail.format` + `detail.path`
  are both non-empty strings). Kept `hasMarkdown` unchanged, added `viewable`
  alongside (per the owner-confirmed field naming in the spec).
- `dashboard/app/lib/api.ts`: mirrored the new `viewable` field on the client
  `WorkflowOutputItem` interface with a doc comment.
- `dashboard/app/components/WorkflowOutputSection.tsx`: the "View" button gate
  now checks `item.viewable` instead of `item.hasMarkdown`, so a JSON/path
  (T262 form) output item gets a working View button too. `openOutput()`/
  `OutputModal` needed no change — they already dispatch through
  `renderOutputBody`/`OUTPUT_RENDERERS`, which already handles the `json` form.
- `dashboard/scripts/_dashboard-harness.mjs`: updated the `places`
  `/output-items` fixture so each row carries the new `viewable` field
  (matching real backend semantics): the markdown item is
  `hasMarkdown:true, viewable:true`, the no-artifact item is
  `hasMarkdown:false, viewable:false`, and the JSON item is now realistically
  `hasMarkdown:false, viewable:true` (previously fudged as `hasMarkdown:true`
  just to make the button render pre-fix). The existing `workflow-output-section-json-popover`
  FLOWS entry (added for T458) already clicks that JSON row's View button and
  waits for the modal + `<pre>` body — no new FLOWS entry was needed, it now
  exercises the real gating path end-to-end.
