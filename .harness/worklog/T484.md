# T484 — 2026-07-10

Migrated `places`'s `resolveInputKeys()` off the self-referential
`data/out/places.json` read onto a live query against the raw Google Takeout
CSVs (`data/raw/Saved/*.csv`), routed through `callService('fs', ...)` (the
T483 fs pseudo-service).

## What changed

- `src/workflows/places/parse.ts` — added two shared helpers:
  - `listSavedCsvFiles(savedDir)` — the CSV-file discovery step (was inline in
    `places-ingest`'s `runIngest`).
  - `collectResolvableCids(savedDir)` — walks the CSVs and returns the full CID
    list for every row that would be "resolvable" (has a URL, is a
    `/maps/place/` URL, carries a CID feature id) — the exact same per-row
    filter `places-ingest` applies when building its normalized place records.
- `src/workflows/places/stages/ingest.ts` — `runIngest` now calls the shared
  `listSavedCsvFiles` instead of its own inline `readdirSync(...).filter(...).sort()`,
  so ingest and `resolveInputKeys` can never walk a different file set.
  `runIngest`'s own row loop, narration, and validation-report semantics are
  otherwise untouched.
- `src/workflows/places/stages/resolve.ts` — `resolveInputKeys()` is now
  `async` and calls `callService('fs', () => collectResolvableCids(placesConfig.savedDir))`
  directly against the raw CSVs, instead of reading back `data/out/places.json`
  (this workflow's own prior output). `JobDefinition.inputKeys` already accepts
  `string[] | Promise<string[]>`, so no other call site needed a change.
- `src/workflows/places/CLAUDE.md` — documented the new live-source root
  selection and the shared CSV-parse helpers.
