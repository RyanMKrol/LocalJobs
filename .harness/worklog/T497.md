# T497 — Add a user-set 'certified' flag on workflows (schema + store + API)

2026-07-10

## What I did
- `src/db/schema.sql`: added `certified INTEGER NOT NULL DEFAULT 0` to the `workflows`
  table with a comment explaining it's a plain user-set flag with no code/manifest
  source — no `_overridden` companion column, no CASE-based preserve logic needed.
- `src/db/index.ts`: added the additive `ALTER TABLE workflows ADD COLUMN certified
  INTEGER NOT NULL DEFAULT 0` migration, guarded by a `PRAGMA table_info` column-
  existence check, placed after the existing workflow-column migrations.
- `src/db/store.ts`: added `certified: number` to `WorkflowRow`, and
  `setWorkflowCertified(name, certified)` — a plain `UPDATE ... SET certified = ?`,
  returning the updated row via `getWorkflow` or `undefined` if no row matched.
  Confirmed `upsertWorkflowStmt`/`syncWorkflow` never reference `certified`, so a
  re-sync can never touch it either way.
- `src/api/server.ts`: added `POST /api/workflows/:name/certify` right after the
  existing `/notify` block — validates the body is strictly boolean (400 otherwise),
  404s on unknown workflow, calls `setWorkflowCertified`, returns `{ ok: true,
  certified }`. Confirmed `GET /api/workflows` and `GET /api/workflows/:name` need
  no changes — both already spread the raw row (`...p`), so `certified` is
  automatically exposed.
- `dashboard/app/lib/api.ts`: added `certified?: number` to the `Workflow` interface
  and `updateWorkflowCertified(name, certified)` client helper mirroring
  `updateWorkflowNotify`'s shape.
- `CLAUDE.md`: added a new bullet in the workflow-property conventions block
  documenting `certified`, explicitly calling out it's simpler than
  schedule/maxConcurrency/notifyEnabled (no `_overridden` column, no reconcile).

## Tests added
- `src/db/migrate-existing-db.test.ts`: new regression test asserting an old-shape
  DB (no `certified` column) opens cleanly via the real `openDb()` and ends with
  `certified` defaulting to 0.
- `src/db/store.test.ts`: new block exercising `setWorkflowCertified` — toggle on,
  toggle off, survives an unrelated `syncWorkflow` re-sync untouched, and returns
  `undefined` for an unknown workflow name.
- `src/api/server.test.ts`: new block exercising `POST /api/workflows/:name/certify`
  — valid boolean persists + is visible on both the GET detail and GET list
  payloads, non-boolean bodies (`1`, `'x'`, `null`) → 400, unknown workflow → 404.
