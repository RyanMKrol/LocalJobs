# T460 — listening-digest: add trailing 3-month digest alongside current-month

2026-07-08

## What I did

- `config.ts`: added `trailingPeriod: '3month'` alongside the existing `period: '1month'`
  (shared `singleTrackAlbumRatio`, no second ratio config).
- `stages/listening-digest.ts`:
  - `makeTopAlbumsFetcher`/`makeTopTracksFetcher` now return fetchers parameterized by
    `period: string` instead of hardcoding `listeningDigestConfig.period`, so the same
    Last.fm request logic serves both passes. `TopAlbumsFetcher`/`TopTracksFetcher` types
    updated to `(period: string) => Promise<...>`.
  - `runListeningDigest` now runs a second pass after the existing 1-month pass (unchanged
    filename/ledger key/detail): fetches top albums/tracks for `trailingPeriod` via the same
    `callService('lastfm', ...)` pattern, filters with the same ratio, renders via
    `renderDigestMarkdown` with `period: trailingPeriod` and a monthLabel suffixed
    `(Trailing 3 Months)`, writes `listening-digest-<YYYY-MM>-3month.md`, and records a second
    `markWorkItem` row under the same job name with key `<YYYY-MM>-3month`.
  - `ctx.progress` now reports 50 after the first pass and 100 after the second; verbose
    `ctx.log` narration mirrors the first pass for the second.
  - Both passes still throw on any Last.fm/write failure (no swallowed second-pass failure).
- `stages/listening-digest.test.ts`: updated `albumsFetcher`/`tracksFetcher` stubs to accept
  (and ignore) a `period` arg to match the new fetcher shape; added
  `periodAwareAlbumsFetcher`/`periodAwareTracksFetcher` helpers that return different data per
  period. Added two new tests: (1) a single run writes both files with period-correct content
  and both ledger keys marked done and both surfacing via `workflowTerminalItems`; (2) a
  same-month re-run overwrites both files in place without duplicating either ledger row.
  All prior tests kept passing unmodified in behavior/assertions.
- `stages/listening-digest.job.ts`: updated `description` to narrate the two-file, two-period
  behavior.
- Docs (all in scope): `src/workflows/listening-digest/CLAUDE.md`, root `CLAUDE.md`'s
  shipped-workflows table row, and `README.md`'s listening-digest job-list paragraph all
  updated to describe the two-output-file (current-month + trailing-3-month) behavior.

No new job/stage/DAG member was added — still a single-job, single-stage workflow. No new
`dependsOn` edge, so no new gate is required (unchanged from the workflow's existing gate
coverage).
