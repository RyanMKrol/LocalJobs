# T460 — listening-digest writes a trailing 3-month digest alongside the current-month one

2026-07-08

## What was done

Same `lastfm-digest` job now runs the fetch/filter/render pipeline twice per run
(`period=1month` and `period=3month`), producing a second markdown file and ledger row
alongside the existing current-month digest. No new job, stage, or DAG edge.

- `config.ts`: added `trailingPeriod: '3month'` alongside the existing `period: '1month'`;
  `singleTrackAlbumRatio` stays shared across both passes.
- `stages/listening-digest.ts`: `TopAlbumsFetcher`/`TopTracksFetcher` now take an explicit
  `period: string` parameter, and `makeTopAlbumsFetcher`/`makeTopTracksFetcher` build their
  Last.fm request params from the passed-in period instead of hardcoding
  `listeningDigestConfig.period`. `runListeningDigest` calls each fetcher twice — once for
  `period` (unchanged filename `listening-digest-<YYYY-MM>.md`, unchanged ledger key
  `<YYYY-MM>`) and once for `trailingPeriod` (new file
  `listening-digest-<YYYY-MM>-3month.md`, new ledger key `<YYYY-MM>-3month`, heading suffixed
  "(Trailing 3 Months)"). Both passes route through `callService('lastfm', …)` for
  rate-limit/quota coordination. A second `markWorkItem` row is recorded under the same job
  name with key `<YYYY-MM>-3month`. `ctx.progress` now reports 50 after the first pass and
  100 after the second; verbose `ctx.log` narration mirrors the first pass for the second.
  Both passes still throw on any Last.fm/write failure (no swallowed second-pass failure).
- `stages/listening-digest.test.ts`: updated `albumsFetcher`/`tracksFetcher` stubs to accept
  (and ignore) a `period` arg to match the new fetcher shape; added
  `periodAwareAlbumsFetcher`/`periodAwareTracksFetcher` helpers that return different data per
  period. Added a new test proving a single run writes both files with period-correct content
  and both ledger keys marked done and both surfacing via `workflowTerminalItems`. Extended the
  existing same-month re-run test to assert the trailing-3-month file is also overwritten in
  place rather than duplicated. All prior tests kept passing unmodified in behavior/assertions.
- `stages/listening-digest.job.ts`: updated `description` to narrate the two-file, two-period
  behavior.
- Docs (all in scope): `src/workflows/listening-digest/CLAUDE.md`, root `CLAUDE.md`'s
  shipped-workflows table row, and `README.md`'s listening-digest job-list paragraph all
  updated to describe the two-output-file (current-month + trailing-3-month) behavior.

No new job/stage/DAG member was added — still a single-job, single-stage workflow. No new
`dependsOn` edge, so no new gate is required (unchanged from the workflow's existing gate
coverage).

## Checks run

- `npx tsc --noEmit` — clean, no errors.
- `npm test` — full suite green: 242 tests, 242 pass, 0 fail.

No live/paid external calls were made — all tests use injected stub fetchers.
