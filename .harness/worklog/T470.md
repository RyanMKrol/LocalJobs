# T470 — Evaluate a cross-workflow job-sharing mechanism

**Decision: DECLINE for now. Keep the existing convention — each workflow gets its own
(duplicate) job; do NOT build a shared-job mechanism.** No code, no doc changes, no follow-up
tasks authored. Decision-only task.

## How the question was reframed (important)

The original spec framed "job sharing" as a RUNTIME mechanism: the executor recognising a job
already ran this cycle in another workflow and REUSING its output instead of re-running. That is the
expensive/complex version and was NOT what the owner wanted.

The owner's actual model: share a job's *definition* (one file, one `name`, one set of code) while
each workflow still runs it **independently** — if two workflows fire at once the shared job genuinely
runs twice, once per workflow. "A job is just logic; the ledger driving a workflow should be
orthogonal to the job."

## What the survey found (3 parallel exploration agents, 2026-07-11)

**Duplication clusters across workflows:**
1. Plex TV snapshot — `missing-tv-seasons/plex-tv-snapshot` vs `tv-recs/tv-snapshot` (free Plex reads).
2. Plex movie snapshot — `movies/movie-snapshot` vs planned `missing-movies/plex-movie-snapshot` (free).
3. Trading212 portfolio — `stocks-sync/stocks-fetch` vs `stock-digest/stock-portfolio-fetch`. Both call
   the SAME `fetchPortfolio()` in `src/services/trading212.service.ts` — the fetch logic is ALREADY
   shared via the service; only the thin job wrapper is duplicated. Deliberately decoupled (per
   `stock-digest/CLAUDE.md`).
4. Plex library list scans — `plex-space-saver` / `plex-profiles` / `plex-language-fix` — only ~40%
   alike; genuinely different work, not one job.
5. TMDB rec-merge — `movies/rec-merge` vs `tv-recs/tv-rec-merge` — shared LOGIC with DIFFERENT inputs
   (movie vs TV); a shared-helper refactor, not a shared job.

**The mechanism would be small but not free.** The one load-bearing change is the overlap guard:
`hasActiveRun(job_name)` (`src/db/store.ts`) is keyed by job name alone, so a shared job's 2nd
workflow is recorded `skipped` (and `skipped` blocks downstream in `executeDag`). Fix = scope the
guard to per-`workflow_run_id` (the `runs` table already has that column). Plus minor cleanup: the
job→workflow map is last-write-wins (`map.set`, `src/api/server.ts`) so a shared job would be
mislabeled; the `/jobs/[name]` view assumes a single parent; docs assert "belongs to ONE workflow"
in ~5 places.

**The real complication — the ledger is namespaced by job name.** `work_items` is keyed
`(job_name, item_key)`, and a job's `run()` records/reads it under its own `name`. So a job's `name`
is both its identity AND its idempotency-state namespace. Sharing one job across two workflows shares
that ledger:
- Re-scan-fresh jobs (the snapshots — never consult "already done?") → sharing is harmless.
- Item-loop jobs (`isWorkItemDone`-gated) → would MISBEHAVE: workflow B skips items workflow A already
  marked done. Unsafe until the ledger is namespaced (a larger, deferred change).

## Why decline

- **Logic reuse is already solved.** Shared API/logic lives in `src/services/*` and plain helper
  functions that every workflow imports (cluster 3 proves it — the fetch is already shared). The
  duplication that remains is thin job wrappers.
- **The only clean shared-*job* candidates are the two Plex snapshots**, both free — so the payoff is
  code-DRY on a small, cheap surface, not spend or correctness.
- **The mechanism drags in the ledger-namespace coupling** and is only safe for re-scan jobs today;
  doing it "properly" (namespacing the ledger per workflow) is a much larger change with no current
  demand.
- Owner's call (2026-07-11): "too complicated for the benefit — just keep them separate."

## Consequences / no-ops

- The duplicate-job convention STANDS. `movies` → `missing-movies` split (T468/T469) proceeds with a
  duplicate `plex-movie-snapshot` exactly as T467's record specifies — no change.
- The "every job belongs to (at least) one workflow" docs stay correct as written; NO doc edits.
- No follow-up tasks authored.
- If ever a future split needs to share a genuinely expensive/rate-limited upstream, OR the wrapper
  duplication becomes a real maintenance burden, revisit — and prefer extracting a shared HELPER
  FUNCTION (zero framework change, each workflow keeps its own job/ledger) before reaching for the
  shared-job-identity mechanism.
