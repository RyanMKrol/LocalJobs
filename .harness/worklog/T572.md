# T572 — soft-pause on Claude rate limit (workouts-progress, stock-digest-build, project-summarize)

2026-07-15

## What I did

Three Claude-narrating stages were treating `result.rateLimited` (already surfaced by
`src/services/claude.ts`'s `ClaudeResult`) as an ordinary failure — burning their retry budget
against a limit that won't clear within a retry window, and (for `project-summarize`) continuing
to hammer every remaining repo in the loop after the limit was hit.

Added a `result.rateLimited` branch **before** each stage's existing generic `if (!result.ok) throw`
handler, mirroring the wording/shape already used in `movies/stages/recommend.ts` (the recommender
branches already treat a rate-limited Claude call as a soft no-op):

- **`workouts-sync/stages/workouts-progress.ts`** (single-artifact stage): on `rateLimited`, logs a
  `warn` ("Claude rate/usage limit hit — pausing this stage, will retry next run") and `return`s
  early — no markdown written, no `work_items` ledger mark, no throw. The month is left un-done so a
  later scheduled/manual run resumes it.
- **`stock-digest/stages/stock-digest-build.ts`** (single-artifact stage): same shape — soft `return`
  before writing the digest markdown or marking the ledger.
- **`projects-sync/stages/project-summarize.ts`** (per-repo loop): on `rateLimited`, logs the same
  warn and `break`s the loop — the current repo is NOT marked `failed` and the remaining repos are
  left for the next run. Since `break` (not `throw`) exits the loop before the `catch` block, this
  correctly does not increment the loop's own `failed` tally, so it can't trip the existing T416
  "throw if `failed > 0`" end-of-run rule — a rate-limit pause is a soft-stop, not a per-item failure.

A **non**-rate-limit failure (`result.ok === false`, `rateLimited === false`) is untouched — it still
throws/fails exactly as before in all three stages.

## Tests

Added one test per job (all hermetic — scratch DB, no live Claude/network calls):
- `workouts-progress.test.ts`: injects a `rateLimited: true` `runClaudeFn`, asserts no markdown
  written, ledger item left un-done, and a `warn`-level log line is emitted. Existing "throws when
  Claude fails" test (non-rate-limit) still passes unchanged.
- `stock-digest-build.test.ts`: same shape — asserts no digest markdown written, ledger item left
  un-done, warn logged. Existing "throws when the Claude call fails" test still passes.
- `project-summarize.test.ts`: 3-repo catalog where the 2nd repo's Claude call reports
  `rateLimited: true` — asserts the 3rd repo is never attempted (loop broke), the rate-limited repo
  has NO ledger row (not marked failed), the 1st repo still succeeded, and a warn was logged. Existing
  "fails the whole run when one of two repos fails" (non-rate-limit) test still passes.

(Each new rate-limit test uses a distinct month/week key from the other tests in its file, since the
scratch DB persists across `it()` blocks in the same file and an already-claimed period/week key
would make the "item left un-done" assertion spuriously fail against a row a prior test had already
marked done.)
