# T287 — projects-sync: drop DynamoDB, write a local catalog + weekly schedule

## What changed
- `src/jobs/projects-sync/stages/github-sync.ts`: removed `dynamoPut`/`DynamoPutter`/
  `PROJECTS_TABLE` entirely. Kept `fetchAllRepos` + `filterAndSortRepos` unchanged. Added
  `repoToCatalogEntry` (trimmed projection: repoId, name, fullName, description, url,
  language, topics, pushedAt, defaultBranch), `writeCatalog`/`CatalogWriter` (writes
  `data/out/projects.json`), and `githubSyncInputKeys` (reads repo ids back from the
  last-written catalog, exposed as the job's `inputKeys()` so it stays the limitable root).
  `runGithubSync` now writes the catalog once then records one `success` work_items row per
  repoId (no more per-repo Dynamo put/fail path).
- `src/jobs/projects-sync/config.ts` (new): job-local `data/out` + catalog path, resolved via
  `resolve(here, ...)` per convention.
- `src/jobs/projects-sync/stages/github-sync.job.ts`: updated description, wired
  `inputKeys: githubSyncInputKeys`.
- `src/jobs/projects-sync/projects-sync.workflow.ts`: schedule changed `0 5 * * *` → `0 5 * * 0`
  (weekly, Sunday 05:00); doc comment + description updated to say weekly / local catalog, no
  DynamoDB.
- `src/jobs/projects-sync/stages/github-sync.test.ts`: rewritten — stub `CatalogWriter` in place
  of the stub `DynamoPutter`, added tests for `repoToCatalogEntry` field mapping and
  `repoIdsFromCatalog`; `runGithubSync` tests assert the catalog writer receives the filtered/
  sorted entries and that a `work_items` success row exists per repo id (via the scratch DB,
  `LOCALJOBS_DB` from `npm test`) — never touches the real job `data/` tree since the writer is
  injected.
- `.env.example`: removed `PROJECTS_TABLE=`.
- `CLAUDE.md` / `README.md`: `projects-sync` paragraph/bullet updated — local catalog file, no
  DynamoDB, weekly (not daily) schedule.

`src/services/dynamodb.service.ts` is left in place — `workouts-sync` still uses it.
