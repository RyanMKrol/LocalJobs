# T112 ‚Äî Fix workflow-run stage status flicker + perfumes self-looping

## Root causes (confirmed from code, matching the live-log diagnosis)

**(1) Self-looping / never "stable".** `runWorkflowInner`'s `repeatUntilStable`
loop only stopped on `workflowRetryableCount(...) === 0`. That count is
`COUNT work_items WHERE status='failed' AND attempts < minAttempts`. A
genuinely-unfindable input (`perfumes-find-url` key `amouage-jubilation-40`,
failed, attempts=2, "no Fragrantica page found") is `failed AND attempts<4`, so
it's counted retryable EVERY cycle ‚Äî but its attempts never increment across
cycles (it isn't actually re-advanced; in practice the Claude CLI hits a
usage/rate limit and `runFindUrl` breaks before re-attempting it, leaving it
frozen). Count is permanently 1 ‚Üí loop spins to `maxCycles=40`, re-running all 4
stages each cycle for nothing.

**(2) Status flicker (succeeded‚Üírunning‚Üísucceeded).** `listRunsForWorkflowRun`
ordered by `started_at` only. `started_at` is second-granularity, so during fast
cycling an earlier cycle's settled (success) run and the current cycle's fresh
`running` run can share a second; with no tiebreaker the order is undefined and
the dashboard's last-write-wins "latest per stage" could pick the stale success
over the live running run.

**(3) Notification spam (ntfy 429).** `notifyStage` was fired on every
`onSettle`/`onSkip`, i.e. every stage every cycle (~40√ó4=160 pushes) ‚Üí ntfy
rate-limits.

## Fixes

- **No-forward-progress early stop** (`src/core/workflow-executor.ts` +
  `src/db/store.ts`). New `workflowProgressSignature(members, minAttempts)`
  snapshots the member ledger (row count + summed attempts + retryable count);
  pure `noForwardProgress(prev, cur)` returns true when a whole cycle left the
  signature unchanged AND retryable didn't drop. The cycle loop now breaks early
  in that case (logs a warn pointing to unstick/ignore) instead of spinning to
  `maxCycles`. Robust regardless of *why* a "retryable" item isn't advancing.
- **Per-stage notification dedup.** `runWorkflow` keeps a `lastNotifiedStatus`
  map and only calls `notifyStage` when a stage's status changes from the last
  push, while `repeatUntilStable`. Single-cycle workflows (empty map) still
  notify once per stage ‚Äî unchanged. Aggregate `notifyWorkflow` always sent.
- **Deterministic latest-run ordering.** `listRunsForWorkflowRun` now orders by
  `(started_at, rowid)`; `rowid` increases with creation so the genuinely-latest
  run sorts last ‚Üí dashboard status derivation is correct (no false succeeded).
  Updated the page comments in `dashboard/app/workflow-runs/[id]/page.tsx`.

## Tests (regression)

Added to `src/db/store.test.ts`:

- **status flicker**: two same-job runs in one workflow run, started_at forced

  the newer running run sorts last and the dashboard derivation yields `running`.

## Notes / trade-offs

The no-progress stop is a heuristic (immediate single-cycle break), documented in
`.harness/LIMITATIONS.md`: a transient external outage where a job advances
nothing in the ledger would stop a cycle early (resumes fine next run, since work
is idempotent). Jobs should advance the ledger on every real attempt. Did NOT
change `perfumes-find-url`'s freeze behaviour itself ‚Äî the framework-level stop is
the robust place to fix this and covers all causes; the find-url freeze is an
environmental Claude rate-limit, not a code bug.
