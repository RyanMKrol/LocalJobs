# T564 — consolidate the duplicated `readPortfolio` helper

## 2026-07-14

### What I did
R07 dedup: the tolerant `readPortfolio(path): NormalizedPosition[]` helper (reads a
`NormalizedPosition[]` snapshot from a JSON path, returns `[]` on missing/empty/malformed
file) was duplicated verbatim in three stage files. Consolidated to a single definition.

- **New single definition:** `src/workflows/stock-digest/lib.ts` now exports
  `readPortfolio`. Used the `Array.isArray`-guarded form (the form the two stock-digest
  copies already had) — strictly more tolerant than stocks-watch's un-guarded copy, and
  behaviourally identical for real portfolio data (a valid array), which is all the file
  ever contains. `lib.ts` already imports the shared `NormalizedPosition` type's home
  (`src/services/trading212.service.js`), so a portfolio-read helper is a natural fit.
- **Deleted the three local copies** and pointed all three call sites at the single def:
  - `stock-digest/stages/stock-sector-lookup.ts` → `import { weekKey, readPortfolio } from '../lib.js'`
    (also dropped its now-unused `NormalizedPosition` import).
  - `stock-digest/stages/stock-digest-build.ts` → `import { weekKey, weekLabel, readPortfolio } from '../lib.js'`
    (also dropped its now-unused `readFileSync` import).
  - `stocks-sync/stages/stocks-watch.ts` → `import { readPortfolio } from '../../stock-digest/lib.js'`
    (also dropped its now-unused `readFileSync` import).
- Updated the two tests that imported `readPortfolio` from a stage
  (`stock-sector-lookup.test.ts`, `stock-digest-build.test.ts`) to import it from `../lib.js`.
  Their existing `readPortfolio` test blocks (missing/empty/array cases) stay and act as the
  regression net for the shared definition.

### Cross-workflow import decision
`stocks-watch` is in a *different* workflow (`stocks-sync`) from `stock-digest/lib.ts`. The
spec's preferred single-def location is `stock-digest/lib.ts` and the Overview explicitly
frames it as "a small shared stocks lib both workflows import". The ideal coupling-free home
would be the shared services layer (alongside `NormalizedPosition` in
`src/services/trading212.service.ts`), but that file is **out of this task's scope**, so I
could not place it there. Given (a) the scope includes `stocks-watch.ts` and excludes every
services/core file — signalling the author expected `stocks-watch` itself to be edited, not the
service — (b) the Overview endorses both workflows importing the shared stocks lib, and (c) the
repo already contains cross-workflow imports of pure helpers/types (`missing-movies` ↔ `movies`),
I chose `stock-digest/lib.ts` as the single home and had `stocks-watch` import from it. This is a
pure read helper, not a runtime data dependency between the two workflows' outputs (the thing
`stock-digest/CLAUDE.md`'s "No inter-workflow dependency" rule actually forbids — reading
`stocks-sync`'s `portfolio.json`), which is untouched.

### Docs
