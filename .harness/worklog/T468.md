## Scope-driven deviation (deliberately NOT a full physical/API relocation) — read before touching this area again

The T467 design record (`.harness/worklog/T467.md`) says a full physical move of `notify.ts` to
`missing-movies/` requires updating `src/api/server.ts`'s imports
(`NOTIFY_JOB as MOVIE_GAPS_JOB, gapKey` from `../workflows/movies/stages/notify.js`, and the
`GET /api/movie-gaps` endpoint's direct read of `moviesConfig.gapsOut`). **`src/api/server.ts` was
NOT in this task's scope list.** Rather than violate the hard scope gate, I kept two small
compatibility shims so `server.ts` needed zero changes and still behaves correctly:

1. `src/workflows/movies/stages/notify.ts` (now the recs-only job) still exports `NOTIFY_JOB =
   'movie-gaps-notify'` and `gapKey()` — UNUSED by its own recs-only logic, kept purely so
   `server.ts`'s existing import path resolves. `missing-movies/stages/notify.ts` defines its own
   identical copy of both constants as the actual source of truth for the live gaps-notify job.
2. `src/workflows/movies/config.ts`'s `moviesConfig.gapsOut` is now an alias:
   `gapsOut: missingMoviesConfig.gapsOut` — so `server.ts`'s `GET /api/movie-gaps` endpoint (which
   reads `moviesConfig.gapsOut` directly) keeps reading the REAL current file location
   (`missing-movies/data/out/franchise-gaps.json`, where `franchise-gaps` now actually writes)
   without a code change there.

Both are called out with comments at the point of definition and in both workflows' `CLAUDE.md`s.
A follow-up task that puts `src/api/server.ts` in scope should: point its `NOTIFY_JOB`/`gapKey`
import and its `moviesConfig.gapsOut` read at `missing-movies/stages/notify.js` /
`missing-movies/config.ts` directly, then delete the two compat shims above.

**Also NOT done (same reason, dashboard/ out of scope):** `MovieGapsManager` (the franchise-gaps
ignore/report UI section, `POST /api/movie-gaps/*`) still renders on the
`movie-recommendations` workflow detail page, not `missing-movies`'s — moving it needs
`dashboard/app/workflows/[name]/page.tsx` in scope, which this task didn't have. The T467 design
record listed this dashboard split as in-scope for "the build pass," but the actual scope grant for
T468 didn't include `dashboard/`; I flagged this explicitly rather than silently under-delivering
or breaking the scope gate. The `/api/movie-gaps/*` endpoints themselves are completely unaffected
either way (ledger keyed by job_name/item_key, not workflow membership) — only the dashboard
section's location is stale. Both workflows' `CLAUDE.md`s call this out. Recommend a follow-up
task with `dashboard/` in scope to move the section.
