# T486 — fix projects-sync self-referential inputKeys()

2026-07-10

## What I did

`githubSyncInputKeys()` in `src/workflows/projects-sync/stages/github-sync.ts` used to read back
`data/out/projects.json` — the file the SAME stage (`runGithubSync`) writes — returning `[]` on any
read error including file-not-found. On a reset/fresh checkout this made every candidate root look
"already complete" to the T094 limited-run root selector, even though nothing had ever synced.

Changed it to call the live GitHub API directly, reusing the existing `fetchAllRepos` fetcher (which
already routes each page through `callService('github', ...)`, T424) instead of reading the derived
catalog file. It now takes an injectable `ReposFetcher` (defaulting to `fetchAllRepos`) for testability,
mirroring the pattern already used by `runGithubSync`'s `opts.fetchRepos`. Applies the same
filter/sort as the real sync so the candidate id set matches what `writeCatalog` would produce.

Updated `src/workflows/projects-sync/CLAUDE.md` to document the new live-source `inputKeys()` and why
the old read-back approach was self-referential.

## Tests added (`github-sync.test.ts`)

- `githubSyncInputKeys` returns repo ids from a live fetch, simulating a reset with no catalog file
  on disk (never touches the filesystem in the test at all — proves it no longer depends on
  `data/out/projects.json` existing).
- Returns `[]` without attempting a fetch when `GITHUB_USERNAME` is unset.
- Returns `[]` (not throws) if the live fetch fails.
