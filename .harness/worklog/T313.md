# T313

Added an optional `?job=<name>` query param to `GET /api/workflow-runs/:id/io`.

- No `job` param: behavior unchanged; response now also carries neutral defaults
  `selectedJob: null, scopedProducerJobs: [], scopedConsumerJobs: []`.
- `job=<root job>` (no predecessors): self-pairs producer/consumer as `[job]`/`[job]`.
- `job=<mid-DAG job>`: producer = its direct `dependsOn` (via `dag.dependencies`),
  consumer = `[job]`. `workItemIoRows` itself needed no changes — it was already
  generic over which job sets are joined; only the route's call arguments differ.
- `job=<unknown>`: 400 with `{ error: 'unknown job "<X>" for this workflow' }`.

Tidied `workItemIoRows`'s JSDoc in `src/db/store.ts` to describe both call shapes
(whole-workflow first/last wave, and single-stage predecessor(s)->itself).

Added test coverage in `src/api/server.test.ts` (T313 block): root-job self-pair,
mid-job scoped pairing (asserts `outputJob`/`outputStatus` reflect the mid stage's
own ledger row, not the terminal stage), unknown job -> 400, and a regression
assertion that the no-`job` response's new fields default correctly.

Verified: `npx tsc --noEmit` and `npm test` both green (85/85 passing). No files
under `dashboard/` touched. Only `src/api/server.ts` and `src/db/store.ts` (in
scope) plus the test file and this worklog were changed.
