# T323 — Admin nav tab + bulk "delete all workflow output" page

## What was built
- `dashboard/app/layout.tsx`: added `<a href="/admin">Admin</a>` as the last nav tab (after Logs).
  The nav (now 6 items) no longer fits on one line at mobile width, so the `<nav>` element
  got explicit inline styles (`width:100%`, `boxSizing:border-box`, `display:flex`,
  `flexWrap:wrap`, `rowGap:6`) so its links wrap onto extra rows within the header instead of
  overflowing the page. This is scoped to `layout.tsx` deliberately — `globals.css` (which owns
  the shared mobile media-query block) is out of this task's scope, so the fix lives as an inline
  style on the specific element that changed.
- `dashboard/app/admin/page.tsx` (new): single danger-zone panel — "Delete all workflow output".
  Requires typing the exact phrase `DELETE ALL OUTPUT` into a text input before the destructive
  button (`disabled={!canDelete || busy}`, `canDelete = confirmText === CONFIRM_PHRASE`) enables —
  no native `confirm()` used. On click calls `api.resetAllWorkflowsOutput()` (already existing
  client helper, wired to `POST /api/workflows/reset-output-all` from the T322 dependency), shows
  a busy state ("Deleting…"), and on success renders a results table (workflow / reset|skipped
  pill / detail) plus an aggregate summary line. Errors surface inline.
- `dashboard/scripts/_dashboard-harness.mjs`: added `PAGES` entry `{ name: 'admin', path: '/admin' }`,
  a synthetic fixture for `/api/workflows/reset-output-all` (one reset + one skipped result), and
  a `FLOWS` entry `admin-confirm-typed` that fills the confirmation input and screenshots the
  enabled/dangerous button state.

## Verification performed
- `npx tsc --noEmit` — clean.
- `npm test` — 153/153 passing (no new tests needed; page has no non-trivial logic beyond a
  string-equality gate already visible in the reviewed code).
- `npm --prefix dashboard run build` — succeeds; `/admin` compiles as a static route.
- `node dashboard/scripts/visual-check.mjs` — all 37 captures pass. Read `admin.png` (baseline: nav
  shows "Admin" as the 6th tab, wrapped to a second line at this capture width; danger panel
  renders with full explanation text, the input empty, and the delete button in a muted
  pink/disabled look) and `admin-confirm-typed.png` (input filled with `DELETE ALL OUTPUT`, button
  now solid red — visually and unambiguously distinct from the disabled state). Both confirmed
  correctly painted, nothing blank/clipped/overlapping.
- `node dashboard/scripts/mobile-check.mjs` — all 19 pages pass at 402px width, 0px horizontal
  overflow, including `/admin`. Note: adding the 6th nav tab initially broke wrapping on every
  page (5px page-level overflow) two different ways — (1) a nested flex container's intrinsic
  width, as computed by its own flex-item parent, doesn't account for its own `flex-wrap`, so a
  `flex-basis:100%`-only fix still rendered wider than the container; (2) once `width:100%` forced
  the correct box, JSX's whitespace-collapsing between adjacent `<a>` tags removed the implicit
  inline-wrap opportunity between them entirely (no text/whitespace node between elements = no
  break point), so the 6 anchors rendered as one unbreakable run. Landed on `display:flex` +
  `flexWrap:wrap` + explicit `width:100%`/`box-sizing:border-box` on the `<nav>` itself, which
  sidesteps both issues (flex items always get an implicit wrap point, and the explicit width
  overrides the nested-flex intrinsic-sizing quirk).

## Scope note
Only `dashboard/app/layout.tsx`, `dashboard/app/admin/**`, and the always-allowed
`dashboard/scripts/_dashboard-harness.mjs`/worklog were touched — `globals.css` was deliberately
left untouched per the task's hard scope gate, even though the repo convention normally keeps all
responsive CSS there; the nav-wrap fix lives as an inline style scoped to the exact element that
needed it.
