# T401 — perfumes reads live from PerfumeRatings DynamoDB table

## What changed
- `src/services/dynamodb.service.ts`: added a paginated `dynamoScan(table, params?)` helper
  (loops on `LastEvaluatedKey`), mirroring the existing thin-wrapper style of `dynamoGet`/
  `dynamoQuery`/etc.
- `src/jobs/perfumes/config.ts`: added `perfumeRatingsTable` (env `PERFUMES_RATINGS_TABLE`,
  default `PerfumeRatings`); removed `inputFile`.
- `src/jobs/perfumes/types.ts`: `PerfumeInput` gained an optional `fragranticaUrl?: string`.
- `src/jobs/perfumes/lib.ts`: `loadPerfumes()` is now `async`, reads live via
  `callService('dynamodb', () => dynamoScan(perfumesConfig.perfumeRatingsTable))`, and maps
  `id/title/designer/type/fragranticaUrl` → `id/name/brand/concentration/fragranticaUrl`.
  A malformed item (missing id/title/designer/type) is skipped with a `console.warn` rather
  than crashing the whole load.
- All 5 call sites (`find-url.job.ts`'s `inputKeys()`, `find-url.ts`, `parse.ts`, `fetch.ts`,
  `build.ts`) now `await loadPerfumes()`.
- `find-url.ts`: when a perfume's `fragranticaUrl` is present and matches the accepted
  Fragrantica URL shape, it's recorded as an immediate success (`detail.seeded: true`) and the
  Claude web-search call is skipped entirely. Items without one fall through to the existing
  Claude-search path unchanged.
- Deleted `src/jobs/perfumes/data/raw/perfumes.json` (already gitignored, never tracked — no
  git-history implication) and confirmed via grep nothing references `inputFile` anymore.
- `.env.example`: documented `PERFUMES_RATINGS_TABLE` alongside the existing shared `AWS_*` vars
  (no new credential vars needed).
