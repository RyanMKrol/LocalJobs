# T053 — Database view: surface useful canned queries (not a SQL editor)

## What I built
Added a set of **named, server-side, strictly read-only** canned queries to the
Database view, alongside the existing raw table browser. NOT a free-form SQL
editor: the client only ever picks a query by `id`; each query's SQL is a fixed,
hand-written `SELECT` defined in `store.ts`. No client text reaches a query.

### Queries (catalogue in `src/db/store.ts` → `CANNED_QUERIES`)
- `recent-failed-runs` — last 50 runs with status failed/timeout/cancelled.
- `stuck-ignored-items` — per job, count of failed (stuck) + ignored work items.
- `work-item-status` — full per-(job,status) work_items breakdown.
- `service-spend` — per service: calls today/this month vs daily/monthly caps
  (same UTC `start of day`/`start of month` windows as `serviceCallsToday/Month`).
- `workflow-run-outcomes` — last 50 workflow runs with status/trigger/duration.

### Layers
- `src/db/store.ts`: `CANNED_QUERIES` registry + `listCannedQueries()` (metadata)
  and `runCannedQuery(id)` (returns `{id,title,description,columns,rows}` or null
  for unknown id). Columns derived from the first row.
- `src/api/server.ts`: `GET /api/db/queries` (catalogue) and
  `GET /api/db/queries/:id` (run one; 404 on unknown id). Id is the sole input.
- `dashboard/app/lib/api.ts`: `dbQueries()` / `dbQuery(id)` + types.
- `dashboard/app/db/page.tsx`: a "Common queries" panel (buttons per query) above
  a "Browse tables" section. Extracted shared `ResultTable`. Table browser
  unchanged in behaviour.
- `src/db/store.test.ts`: seeds a failed run, failed/ignored/success work items,
  a service call, and a workflow run; asserts the catalogue (≥5, all well-formed,
  expected ids), that every query runs + returns well-formed shape, the specific
  rows each query should surface, and that unknown / injection-y ids → null.
