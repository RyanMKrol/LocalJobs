# T314 — per-stage job filter pills on the workflow-run IO panel

## What I built

Wired the dashboard up to the (already-existing) backend `?job=<jobName>` param on
`GET /api/workflow-runs/:id/io`:

- `dashboard/app/lib/api.ts`: extended `WorkflowIo` with `selectedJob` /
  `scopedProducerJobs` / `scopedConsumerJobs`; `api.workflowRunIo(id, job?)` now
  appends `?job=` when given.
- `dashboard/app/workflow-runs/[id]/page.tsx`: moved the IO poll INTO `IoPanel`
  itself (it now takes `{ runId, members }` instead of `{ runId, data }`), added
  `selectedJob` state, and a new pill row (`.io-job-filter-bar`) above the existing
  state-filter pill row — one 'All stages' pill plus one pill per workflow member
  job (only rendered when `members.length > 1`). Clicking a pill sets `selectedJob`
  (triggering a genuine re-fetch via the new query param) and resets the state
  filter to 'all'. `singleStage` is now derived from `scopedProducerJobs`/
  `scopedConsumerJobs` when a job is selected (collapses to the existing
  single-column view for a root stage), and the Input/Output column headers are
  relabeled to name the actual jobs when scoped and not single-stage.
- `dashboard/app/globals.css`: added `.io-job-filter-bar` / `.io-job-filter-chip`
  (rectangular chips, mono font) as a visually distinct second filter row, separate
  from the existing pill-shaped `.io-filter-bar` state chips.
- `dashboard/scripts/_dashboard-harness.mjs`: `fixtureFor` now takes a second
  `searchParams` arg; the `/io` route branches on `job=places-enrich` to return a
  new `workflowIoScopedToEnrich` fixture with genuinely different rows/statuses
  (`selectedJob: 'places-enrich'`, `scopedProducerJobs: ['places-resolve']`,
  `scopedConsumerJobs: ['places-enrich']`) from the default `workflowIo` fixture.
  Added a `io-panel-job-scoped` FLOWS entry that clicks the `places-enrich` pill
  (via `:text-is()` to avoid ambiguously matching `places-enrich-with-llm`) and
  waits for the re-fetch to settle before the screenshot.
