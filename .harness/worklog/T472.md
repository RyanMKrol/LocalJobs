# T472 — 2026-07-10

## What was done

Fixed the false-positive boundary-crossing heuristic in `dashboard/scripts/mobile-check.mjs`'s
`measure()` that caused T388 to block: React Flow's internal `.react-flow__renderer`/
`.react-flow__pane` layers report `overflow-x: visible` and a large `scrollWidth` from their
unzoomed canvas coordinate space, but they sit two levels inside `.dag-flow-wrap`/`.react-flow`
ancestors that clip with `overflow: hidden` — so nothing actually paints outside the viewport.

Added `isClippedByAncestor(el)` in `measure()`: walks up an element's ancestor chain to `<body>`
and treats the element as clipped (not a real violation) if any ancestor between it and `<body>`
has a non-`visible` `overflow-x` AND that ancestor's own bounding rect right edge is within the
viewport (i.e. the ancestor isn't itself already flagged as overflowing). The existing
document-level `docOverflow` check is untouched (it already correctly reports 0px for the
claude-warmer-run page).

Added a `claude-warmer-run` `PAGES` entry (`workflow-run-claude-warmer`,
`/workflow-runs/claude-warmer-run`) to `dashboard/scripts/_dashboard-harness.mjs` — previously it
only existed in `FLOWS` for visual-check, which was the specific gap that let this false positive
go unnoticed by mobile-check.
