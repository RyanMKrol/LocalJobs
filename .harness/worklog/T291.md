# T291 — Workflows list page: read-only Notifications column

## What changed
- `src/api/server.ts`: `GET /api/workflows` now computes `effective_notify_enabled` per
  workflow via `effectiveWorkflowNotifyEnabled(wfDef)` (falling back to
  `p.notify_enabled !== 0` when no definition is found), mirroring the exact logic already
  used by `GET /api/workflows/:name`.
- `dashboard/app/workflows/page.tsx`: added a read-only "Notifications" `<th>`/`<td>`
  column right after "Enabled", rendering `<Pill kind={p.effective_notify_enabled ? 'on' :
  'off'}>` — no click handler, copies the existing "Enabled" pill pattern. Bumped the
  empty-state `colSpan` from 7 to 8 for the new column.
- `dashboard/scripts/_dashboard-harness.mjs`: added `effective_notify_enabled: true` to the
  base workflow fixture and `effective_notify_enabled: false` to the `perfumes` override, so
  the workflows-list screenshot exercises both pill states.
- `src/api/server.test.ts`: added a test seeding a workflow with manifest
  `notifyEnabled: false` (not user-overridden) and asserting the list endpoint's
  `effective_notify_enabled` for that workflow is `false`.

## Verification performed
- `npx tsc --noEmit` — clean.
- `npm test` — all suites pass (72 tests across 19 suites incl. the new T291 test).

- `node dashboard/scripts/mobile-check.mjs` (402px viewport) — all 13 pages pass, 0px

  overflow on the workflows-list page (it fits, and `.panel:has(> table){overflow-x:auto}`
  already covers any wide-table case).

## Notes
No new interactive state was introduced (the column is read-only, no FLOWS entry needed).
