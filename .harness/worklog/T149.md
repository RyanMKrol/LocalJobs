# T149 — Plex client LAN-scan fallback (DHCP-resilient host resolution)

## What I built
Made `src/jobs/plex/client.ts` resilient to a changed Plex DHCP IP.

- Added `resolvePlexHost(deps?)`: a **process-cached** resolver (resolves at most once
  per daemon run, with concurrent-call dedup via a shared in-flight promise).
  1. **Configured host first** — probes `PLEX_HOST` via `/identity` and accepts it only
     if it answers as a Plex and (when `PLEX_MACHINE_ID` is set) the returned
     `machineIdentifier` matches — never latches onto the wrong Plex.
  2. **LAN scan fallback** — `enumerateSubnetHosts()` derives each local IPv4 `/24`
     from `os.networkInterfaces()` and emits both `https://` and `http://` `:32400`
     candidates; `scanForPlexHost()` probes them with bounded concurrency, a per-probe
     timeout, and an **overall wall-clock cap** (deadline) so a no-Plex LAN fails fast.
     Accepts the first answering Plex, machine-id-gated when set.
  3. On a scan hit it logs `✓ found Plex at <host> — set PLEX_HOST=<host> in .env …`.
  4. When neither configured host nor scan finds a Plex it throws the existing clear
     `set PLEX_HOST` error (mentions PLEX_MACHINE_ID when set).
- `plexGet` now `await resolvePlexHost()` instead of using `plexConfig.host` directly;
  the error message references the resolved host.
- The real probe `probePlexIdentity(host, timeoutMs)` GETs `<host>/identity` with the
  `X-Plex-Token`, returns the `machineIdentifier` (or null), never throws — uses the
  SAME scoped self-signed-cert agent (no global TLS disable). The probe is **injectable**
  (`PlexProbe`) for hermetic testing; `resetPlexHostCacheForTests()` clears the cache.
- Tuning via env (defaults): `PLEX_PROBE_TIMEOUT_MS=1500`, `PLEX_SCAN_CONCURRENCY=32`,
  `PLEX_SCAN_CAP_MS=20000`.
- `config.ts`: updated the `machineId` doc (no longer "future"); `.env.example`:
  documented `PLEX_MACHINE_ID` (makes the scan safe) + the scan-tuning vars.
- Docs: README plex paragraph + a CLAUDE.md convention bullet (T149).

### Pre-existing, OUT-OF-SCOPE local-only failure (NOT introduced by T149)

(`/2 franchise gaps detected/`). I confirmed via `git stash` that it fails on a clean

gitignored `data/` is absent → 0 recs → title `… detected` → the test PASSES. So this is
a non-hermetic-test artifact that only fails on the owner's machine with real data
present; CI (which the DoD mirrors) is green. Left untouched to stay within T149's scope
— worth a follow-up to make that movies test override `recsFile`/`recsHistoryFile`.
