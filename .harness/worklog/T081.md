# T081 — Workflow-run progress: count completed stages only (no in-flight partial credit)

## What changed
`rollUpWorkflowProgress` (`src/db/store.ts`) previously gave a still-running member
partial credit: `fraction += TERMINAL ? 1 : progress/100`, producing confusing
fractional percentages (e.g. 2% while one of four jobs was 7% in). Changed it to
count ONLY completed (terminal) stages:

- Each member in a terminal state (`success`/`failed`/`timeout`/`cancelled`/`skipped`)
  contributes 1; a still-running or not-yet-started member contributes 0.
- `pct = (completed / total) * 100`, total still = `workflow_jobs` member count.
- Result: bar stays 0% until the first stage finishes, then steps in 100/N
  increments (4 jobs → 0/25/50/75/100). `finishWorkflowRun` success=100 unchanged.

Updated the doc comment on the function to describe completed-stage stepping.

## Tests
- `src/db/store.test.ts`: added a dedicated `rollUpWorkflowProgress` block asserting
  0% with no runs, 25% after one terminal stage, that an in-flight member at 50%
  does NOT move the bar (stays 25%), 50% after a second terminal stage, and that a
  `failed` member is terminal and counts toward stepping (75%).
- `src/core/workflow-executor.test.ts`: the existing roll-up test asserted the OLD
  in-flight behaviour (mid-stage member rolled up to 38%). Updated it to the new
  whole-stage stepping (in-flight stays at 25%; finishing ru-b steps to 50%). This
  test would otherwise have failed under the new code, so updating it was required.
