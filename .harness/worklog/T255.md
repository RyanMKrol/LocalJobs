# T255 — listens: one-time Last.fm history backfill

## What was built

Added a backfill workflow (`listens-backfill`) with a single manual-only job (`lastfm-backfill`) inside `src/jobs/listens-sync/`.

**Key design decisions:**
- Shares the same `work_items` ledger entry key as the live sync (`JOB_NAME = 'lastfm-sync'`), so scrobbles synced by the backfill are automatically skipped by the live sync and vice versa — true shared idempotency.
- No `from` timestamp in the Last.fm API call — walks the full history from page 1 to `totalPages`.
- Processes and writes page-by-page (never buffers the whole history in memory).
- Paced via `callService('lastfm', ...)` — respects the 30 req/min rate limit.
- No Spotify enrichment in the backfill: the live sync handles enrichment for recent scrobbles; adding it to a full-history backfill would be prohibitively slow and would consume the Spotify rate limit for thousands of tracks.
- Manual-only workflow (`schedule: null`) — meant to be run once.
- 6-hour timeout (allows ~1000 pages at 30 req/min).

**Files added:**
- `src/jobs/listens-sync/stages/listens-backfill.ts` — core logic (`runListensBackfill`, `makeBackfillFetcher`)
- `src/jobs/listens-sync/stages/listens-backfill.job.ts` — `JobDefinition` for `lastfm-backfill`
- `src/jobs/listens-sync/listens-backfill.workflow.ts` — `WorkflowDefinition` for `listens-backfill` (manual-only)
- `src/jobs/listens-sync/stages/listens-backfill.test.ts` — hermetic tests (11 test cases, no live API/AWS)
