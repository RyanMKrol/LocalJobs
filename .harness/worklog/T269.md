# T269 — Migrate Overview page to the shared component library

## What I did
Like-for-like swap of the Overview page (`dashboard/app/page.tsx`) onto the T268 shared
component library. No rendered appearance or behaviour change.

### Applicable shared components on this page
Surveyed the page for what the T268 library (`<RunButton>`, `<Pill>`, …) actually covers:
- **Card "▶ Run" button** (workflow cards) — the one genuine run-trigger button → migrated to `<RunButton>`.
- **Pills** — the page uses NO `.pill` elements (statuses are rendered as `.badge` spans), so
  `<Pill>` does not apply here.
- **"Manage all…" button** — `.btn.secondary`, but it opens the stuck popover; it is not a run
  trigger, so no shared component applies.
- **Status badges** — the page renders raw `<span className="badge ...">` WITHOUT the emoji that the
  existing `StatusBadge` component adds; swapping would CHANGE appearance (add an emoji), so left as-is
  to keep the swap like-for-like. (StatusBadge also lives in `ui.tsx`, not the T268 `components/` lib.)

### The one behavioural subtlety
The Overview card run button is styled `.btn.secondary`, always enabled, labelled "▶ Run" — it does
NOT reflect running state (per CLAUDE.md, only the Workflows LIST + workflow DETAIL run buttons
disable-while-running). `<RunButton>` as created in T268 hardcoded `.btn.btn-run` (accent colour) and
a running-disable state, which would NOT be like-for-like here.

Fix: added an optional `variant?: 'run' | 'secondary'` prop to `<RunButton>` (default `'run'`, so all
existing/future `.btn-run` usages are unchanged). The Overview card now uses:
```tsx
<RunButton variant="secondary" isRunning={false} label="▶ Run" onClick={() => runWorkflow(p.name)} />
```
→ renders exactly `<button className="btn secondary">▶ Run</button>` (disabled=false → attr omitted,
title=undefined → omitted), identical to the original markup.

`<RunButton>` was not imported anywhere prior to this task, so the new default-valued prop is a
non-breaking, appearance-preserving extension.
