## T537 — make scripts/ typechecked+tested, fix 3 run-tests.ts defects

2026-07-13

### What I did

1. `tsconfig.json`: `include` now `["src/**/*.ts", "scripts/**/*.ts"]` — `scripts/*.ts`
   (including `plex-language-undo.ts`) is now covered by `npx tsc --noEmit`.
2. `scripts/run-tests.ts` walk now covers `[...walk('src'), ...walk('scripts')]` —
   `scripts/plex-language-undo.test.ts` is discovered and run by `npm test`.
3. Rewrote the runner to isolate each `*.test.ts` in its OWN child process
   (`spawn(tsxBin, [file])`) instead of a same-process dynamic `import()`. This fixes
   the verdict-ordering defect at the root: node:test's real pass/fail bookkeeping
   (and the `process.exitCode` it sets) happens asynchronously at THAT process's own
   exit, not as part of the promise returned by `await test(...)` — a same-process
   importer can print its banner (and even move on to the next file) before that
   bookkeeping has actually landed for a given file, so a real failure can be missed
   entirely. Per-file `child.exitCode` after the child has genuinely exited is the
   single source of truth. The final PASS/FAIL banner + `process.exit(...)` now runs
   only after every child has settled.
4. Added a per-file wall-clock timeout (`LOCALJOBS_TEST_TIMEOUT_MS`, default 60s):
   the child is SIGKILLed on timeout and the file is marked failed, rather than
   hanging the whole run. Verified empirically: a deliberately-hanging file
   (`await new Promise(() => {})`) is killed and reported failed within the
   configured window; a deliberately-failing `node:test` file makes the runner's
   LAST printed line the failure verdict.
5. `package.json`'s `test` script no longer hardcodes `LOCALJOBS_DB=/tmp/lj-test.db`
   (nor the `rm -f /tmp/lj-test.db*` that actively deleted a concurrent run's DB).
   Each spawned child process gets its own DB via `config.ts`'s existing
   `isTestEnv`/`resolveDbPath` per-PID scratch path (`lj-test-guard-<pid>.db` in
   `$TMPDIR`) — no code change needed there, it already supported this.

### Discovered pre-existing issue (documented, NOT fixed — out of this task's scope)

Running the full suite via two truly-concurrent `npm test` processes surfaced that
`src/api/server.test.ts` writes several fixtures to REAL, non-scratch, non-PID-unique
paths under the actual `src/workflows/**/data/` tree (e.g. a `moviesConfig.gapsOut`
real file backed up/restored, and a `perfumes/data/out/reports/__t262-test__.json`
fixture) — under genuine cross-process concurrency these race with another
process's own run and can spuriously fail. This is a PRE-EXISTING gap (unrelated to
the 3 runner defects this task targets — it was equally present before this change,
just never exercised because prior local runs were single-process) and is much
larger in scope than this task (many such fixtures scattered across a 2000+ line
test file). One instance I DID fix in-scope, since it was small and directly
analogous to the runner's own concurrency requirement: the T526
`findWorkflowDataOut` fixture in `server.test.ts` used a hardcoded folder name
(`__t526-fixture__`) under the real workflows root — now suffixed with
`process.pid` so two concurrent runs never collide on it. The remaining
`moviesConfig.gapsOut`-style real-path races are NOT fixed here (would need either
an env-var-overridable base path in workflow `config.ts` files, out of this task's
scope, or a broad rewrite of `server.test.ts`'s fixture strategy) — flagged as a
follow-up. The `stocks-fetch.test.ts` / `hevy-sync.test.ts` service-cache test
failures found along the way (in isolated per-process runs, these two tests failed
100% of the time, not flakily) were a genuine, unrelated pre-existing bug: neither
test file ever called `registerService(...)`, so in a real per-file process (no
registry scan, unlike the old shared-process runner where an earlier-loaded test
file's `registerService` call for the same service name could leak across files
via the shared module cache) `callService`'s `getServiceDef` missed and the
cache-key gating was silently skipped entirely. Fixed by registering the
`trading212`/`hevy` service defs directly in each test file — a legitimate,
in-scope test-file fix, and arguably a correctness improvement (removes a
hidden cross-test-file dependency that only "worked" by accident under the old
same-process runner).
