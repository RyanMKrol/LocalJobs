# T095 worklog — Workflow run: show input → output mapping (first stab)

## Approach

Added a first-cut IO mapping panel to the workflow-run detail page. Implemented
in three layers (store → API → dashboard) with no changes to the DB schema.

### store.ts — `workItemIoRows(firstWaveJobs, lastWaveJobs): IoRow[]`

Two queries (one for first-wave, one for last-wave work_items) joined in JS by
root_key: `last_stage.root_key = first_stage.item_key`. This handles both
example workflows correctly:
- **perfumes**: all stages key by `p.id`; root_key = item_key throughout → exact match.
- **places**: `places-ingest`/`cid-to-place-id-resolver` key by CID (root_key=CID);
  `enrich-with-llm` keys by place_id but root_key=CID → join by root_key works.

Fan-out: collapses to the first matching output (Map insert order, keyed by root_key).

### server.ts — `GET /api/workflow-runs/:id/io`

Builds the workflow DAG from `getWorkflowJobs`, picks `waves[0]` (first wave) and
`waves[waves.length-1]` (last wave), calls `workItemIoRows`, and returns `{ io, firstWave, lastWave, note }`.
Read-only (DB only), safe to poll.

Single-stage workflows (first == last wave) are handled — the UI collapses
output columns in that case.

### dashboard — `IoPanel` component on the workflow-run page

Polls `GET /api/workflow-runs/:id/io` at 5 s. Renders a table of input → output
rows with status badges. Extracts `detail.name` for a friendly display label.
The panel footer displays the first-cut limitation note from the API.

## Known limitations (per task spec)

- **Not per-run-scoped**: work_items is global; the panel reflects the workflow's
  current ledger, not strictly this run's items. Per-run scoping arrives with
  further lineage work (T093/T094).
- **Fan-out collapses**: 1 input → multiple outputs is collapsed to one output row.
- **No live output links**: detail fields (e.g. `markdown` path) are shown via the
  name label but not linked (web UI can't open local file paths).

These are noted in the panel footer and in code comments.
