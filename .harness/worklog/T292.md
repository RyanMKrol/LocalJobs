# T292 — workflow `category` field (backend/schema half)

## What was built
- `WorkflowDefinition.category?: string` added to `src/core/types.ts`.
- `workflows.category TEXT NOT NULL DEFAULT ''` added to `src/db/schema.sql`.
- Additive `ALTER TABLE workflows ADD COLUMN category TEXT NOT NULL DEFAULT ''` migration in
  `src/db/index.ts`, guarded by a `PRAGMA table_info` check (mirrors the `notify_enabled`
  migration), no index added (T098 rule).
- `src/db/store.ts`: `WorkflowRow.category`; `upsertWorkflowStmt` always refreshes `category` from
  the manifest on every sync (`category = excluded.category`, no `_overridden` branching — unlike
  `schedule`/`max_concurrency`/`notify_enabled`); `syncWorkflow` passes `def.category ??
  'uncategorized'`.
- Set `category` on all 10 existing workflow manifests: `second-brain` (places, perfumes),
  `recommendations` (movie-recommendations, tv-recommendations, missing-tv-seasons),
  `regular-maintenance` (workouts-sync, listening-digest, projects-sync, claude-warmer,
  stocks-sync — the last one existed at build time).
- Documented the field + controlled values in root `CLAUDE.md`, next to the
  schedule/maxConcurrency/notifyEnabled conventions, making clear it's manifest-owned only (no
  `_overridden` column, no dashboard edit endpoint).
- `GET /api/workflows` and `GET /api/workflows/:name` need no `server.ts` change — both already
  spread/`SELECT *` the workflow row, so `category` flows through automatically (confirmed by
  reading `src/api/server.ts`'s workflow handlers; no live server run needed to verify this — it's
  the same mechanism the other manifest columns already use).

## Tests
- `src/db/store.test.ts`: new block seeds two `WorkflowDefinition`s (one with `category:
  'second-brain'`, one omitted) through `syncWorkflow`, asserts `getWorkflow`/`listWorkflows`
  reflect the manifest value or `'uncategorized'` fallback, then re-syncs with a DIFFERENT category
  value and asserts the stored value updates (no override/preservation, unlike schedule/concurrency/
  notifyEnabled).
- `src/db/migrate-existing-db.test.ts`: new test asserts the `category` column exists after
  migrating a pre-existing (old-shape) DB and defaults to `''`.
