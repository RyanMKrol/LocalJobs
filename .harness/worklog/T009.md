# T009 — Perfumes: populate accord percentages

## Problem
`perfumes-parse` emitted `accords: [{name, pct}]` with `pct` always `null`. The
accord strength on Fragrantica is the coloured bar's `width: NN%` — present in the
page **HTML** but stripped from the captured page **text** (innerText keeps only the
accord name), which is the only thing the Claude parse step reads. So strength was lost.

## What I did (scope: parse.ts, parse.job.ts, types.ts, parse.test.ts)
- `types.ts`: added an `Accord { name; pct: number|null }` interface.
- `parse.ts`:
  - `parseAccordPercents(html)` — pure, exported. Anchors to the `main accords`
    heading, then matches each bar div (`style="…width: NN%…"` wrapping
    `<span class="truncate">NAME</span>`), returns accords in page order (strongest
    first), names lowercased+normalised, pct rounded to nearest integer, deduped.
    Returns `[]` when there's no accords block.
  - `applyAccordPercents(data, id, ctx)` — reads the cached page HTML
    (`pages/<id>.html`, falling back to `pages-failed/<id>.html`), builds a
    name→pct map, and fills each Claude-extracted accord's `pct` by name. An accord
    with no matching bar — or no cached HTML — stays `null` (a genuine absence is
    never faked). Verbose `ctx.log` of filled/total + bars parsed.
  - Wired into `runParse` right before writing `<id>.json`.
- `parse.job.ts`: instructions note pct is filled from cached HTML bar widths.
- `parse.test.ts`: synthetic HTML fixture mirroring Fragrantica's real bar markup.
  Asserts: widths parsed+rounded strongest-first; a width bar lacking the `truncate`
  label is ignored; multi-word names normalised; no `main accords` block → `[]`;
  duplicate names dedupe to the strongest.

## Verification (observed)
- `npx tsc --noEmit` → clean (exit 0).
- `npm test` → all test files pass, incl. the new
  `src/jobs/perfumes/parse.test.ts` ("perfumes accord percentages parse from bar
  widths").
- Empirical, against **real cached HTML** under `data/out/pages-failed/` (no live
  scrape, no paid API):
  - `parseAccordPercents` on `diptyque-philosikos.html` →
    `green 100, woody 83, sweet 61, fresh 58, fruity 54, lactonic 52, coconut 51`
    (matches the page's `main accords` list + order).
  - `maison-violet-abime.html` → `woody 100, warm spicy 70, aromatic 57, …`.
  - `parfums-de-marly-althair.html` → `sweet 100, vanilla 97, warm spicy 86, …`.
  - End-to-end merge simulating Claude accords + real HTML →
    `[green:100, woody:83, coconut:51, nonexistent:null]` — confirms an accord
    absent from the page stays `null`.

## Known limitation / follow-up (recorded in .harness/LIMITATIONS.md)
`perfumes-fetch` persists only `<id>.txt` on the success path; page HTML is saved
only for diagnosed-failed pages. So in the normal flow no `<id>.html` exists and
accords keep `pct: null`. The parse-side slice is correct and activates the moment
HTML is cached. A follow-up (out of T009 scope — `fetch.ts` not in scope) should
have the fetch stage also write `<id>.html` alongside `<id>.txt`.
