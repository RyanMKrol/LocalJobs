# T327 — Reorganize .harness/ into subdirectories by kind (docs/config/ledgers/scripts)

This is a `needs-human` gated task; the owner explicitly asked for it to be attempted directly in
this session (interactively) rather than via the autonomous loop. The owner also revised the
starting-suggestion layout mid-task (see "Deviation from the starting suggestion" below).

## Final layout

```
.harness/
  CLAUDE.md                    (unchanged — Claude Code auto-loads a directory's own CLAUDE.md)
  README.md                    (unchanged location — see deviation below)
  TASKS.json, IDEAS.md, human-done.json, manual-fail.json, reviews.json   (unchanged — hardcoded
                                paths in src/api/server.ts, consolidate-ideas.mjs, /idea)

    integrate.sh, loop.sh, mark-failed.sh, postflight.sh, repo-lock.sh, supervise.sh,
    consolidate-ideas.sh, consolidate-ideas.mjs, policy.jq
```

## Deviation from the starting suggestion

The task spec's starting suggestion put `README.md` in `docs/` alongside `HARNESS.md`/`LIMITATIONS.md`
and left `designs/` where it was (already "grouped by kind" per the spec's own framing). Mid-task the
owner asked for two changes: (1) move `designs/` inside `docs/` (`.harness/docs/designs/`), and (2)
move `README.md` back OUT of `docs/` to the `.harness/` top level, alongside `CLAUDE.md`/`TASKS.json`.
Both were applied; `docs/HARNESS.md`'s and `docs/LIMITATIONS.md`'s existing relative links
(`./HARNESS.md`, `./LIMITATIONS.md`, `designs/...`) still resolve correctly since everything that
references `designs/` from within `docs/` stayed together as siblings. `.harness/README.md`'s own
links to `HARNESS.md`/`LIMITATIONS.md` were updated to `./docs/HARNESS.md` / `./docs/LIMITATIONS.md`
since it no longer lives in the same directory as them.

## Cross-references verified/updated (all items from the spec's risk list, plus one it missed)

1. **`repo-lock.sh`'s lock-path derivation** — unchanged logic (self-anchors via `BASH_SOURCE[0]`),
   confirmed still byte-identical to `loop.sh`'s inline `acquire_lock` and `src/core/repo-lock.ts`'s
   `resolveRepoPaths` (neither of the other two hardcodes `repo-lock.sh`'s own path, so moving it was
   safe by construction — verified by actually running `mark-failed.sh` and `consolidate-ideas.sh`,
   both of which exercise the lock end-to-end; see Verification).
2. **`mark-failed.sh`'s `source "$HERE/loop.sh"`** — self-relative, both files moved together into
   `scripts/` as siblings, so no change needed beyond usage-comment paths.
3. **`consolidate-ideas.sh`'s `source .harness/repo-lock.sh`** — this WAS cwd-relative (assumed
   invocation from repo root after its own `cd "$HERE/.."`, which only worked because it used to live
   directly at `.harness/`). Normalized to self-relative (`source "$HERE/repo-lock.sh"`) as the spec
   suggested while touching this file — `HERE` now derives the repo root via
   `git -C "$HERE" rev-parse --show-toplevel` and `cd`s there explicitly, so every `.harness/...`
   path later in the script (which never moved: `TASKS.json`, `tasks/`, `.pending-tasks/`) still
   resolves unchanged. Also updated `node .harness/consolidate-ideas.mjs` → `node "$HERE/consolidate-ideas.mjs"`.
4. **`consolidate-ideas.mjs`'s hardcoded `path.join(REPO, '.harness/...')` constants** — no change
   needed: `REPO` is derived via `git -C HERE rev-parse --show-toplevel` from the script's OWN
   directory, which resolves the repo root correctly regardless of subdirectory depth. Only the
   file's own usage-comment (`// Usage: node ...`) needed a path update.
5. **`src/api/server.ts`'s `BACKLOG_PATH`/`REVIEWS_PATH`/`HUMAN_DONE_PATH`/`MANUAL_FAIL_PATH`** —
   confirmed unchanged (none of those 4 files moved, per the spec's own "leave at top level" list).
6. **`.claude/commands/*.md` prose references** — updated in
   `local-jobs-mark-task-failed.md`, `local-jobs-loop-recover.md`, `local-jobs-pre-loop-checkin.md`,
   `local-jobs-convert-ideas.md` (paths to `loop.sh`/`supervise.sh`/`mark-failed.sh`/
   `consolidate-ideas.{sh,mjs}`/`outcomes.jsonl`/`harness.env`/`facets.json`/`HARNESS.md`).
7. **Root `CLAUDE.md` + `.harness/CLAUDE.md`** — updated in the same change (multiple path
   references to moved files across both).
8. **`dashboard/app/*` prose/comments referencing moved paths** — one hit
   (`dashboard/app/backlog/page.tsx`'s `TaskBuildFailures` doc-comment) — updated.

**One real cross-reference the spec's risk list did NOT anticipate, found by grepping actual code
(not just scripts) for the moved filenames:** `src/api/server.ts`'s `readTaskBuildFailures(id, baseDir)`
resolves `failures.jsonl` via `joinPath(baseDir, 'failures.jsonl')` where `baseDir` is `TASKS.json`'s
directory — this is REAL functional code (not a doc/comment), used by `GET /api/backlog`'s
`buildFailures` inlining and rendered on the Backlog page. Moving `failures.jsonl` into `ledgers/`
without updating this function would have silently broken that feature (the endpoint would just never
find the file — no crash, no test failure until someone noticed the UI stopped showing build-failure
history). Fixed: `joinPath(baseDir, 'ledgers', 'failures.jsonl')`, plus its test fixture
(`src/api/server.test.ts`'s T294 suite) now writes to `<baseDir>/ledgers/failures.jsonl`. This is
exactly the kind of thing the task's own "Done when" #1 asked to verify ("every cross-reference...
resolves correctly") — worth flagging since it shows the spec's enumerated list, while thorough, wasn't
exhaustive; a full-repo grep for each moved filename (not just following the spec's list) is what
caught it.

Also updated **`src/core/repo-lock.ts`** and **`src/services/claude.ts`** doc-comments (non-functional,
just accuracy) referencing `.harness/loop.sh` and `.harness/LIMITATIONS.md`.

**T331** (a separate PENDING, not-yet-built `needs-human` task auditing repo docs) had its `scope`
array and spec file (`tasks/T331.md`) hardcode 4 of the paths that moved (`HARNESS.md`,
`LIMITATIONS.md`, `designs/**` ×4 files). Since T331 hasn't been built yet, left stale it would have
sent a future builder looking for files at the old locations. Updated both via a field-scoped edit
(only `scope`, no other field touched) — this is forward-looking correctness, not historical-record
editing, so it doesn't conflict with the "worklogs/old task specs are historical" convention.
