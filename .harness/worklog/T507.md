# T507: Cache lastfm top-albums/top-tracks reads via cacheKey (22h TTL)

## What I did

1. **Added cacheTtlMs to lastfm service** (`src/services/lastfm.service.ts`):
   - Set `cacheTtlMs: 79_200_000` (22 hours, matching the convention used by other cached services like github, google-places, openfigi, trading212-instruments, finnhub, gemini)

2. **Added cacheKey to all 4 callService calls** in `src/workflows/listening-digest/stages/listening-digest.ts`:
   - 1-month top-albums: `lastfm:top-albums:${listeningDigestConfig.period}` (= "lastfm:top-albums:1month")
   - 1-month top-tracks: `lastfm:top-tracks:${listeningDigestConfig.period}` (= "lastfm:top-tracks:1month")
   - 3-month top-albums: `lastfm:top-albums:${trailingPeriod}` (= "lastfm:top-albums:3month")
   - 3-month top-tracks: `lastfm:top-tracks:${trailingPeriod}` (= "lastfm:top-tracks:3month")

3. **Fixed test failures caused by caching**:
   - The existing listening-digest tests were using injected mock fetchers, but the service layer now caches results across tests
   - Added `clearServiceCache('lastfm')` to the `beforeEach` hook in `listening-digest.test.ts` to clear the cache between test runs
   - This ensures each test gets fresh data from its mock fetcher, not cached results from a prior test

4. **Added comprehensive caching verification test** (`src/services/lastfm.service.test.ts`):
   - Test 1: Verifies `lastfm` service definition has `cacheTtlMs: 79_200_000` (22 hours)
   - Test 2: Verifies a repeated call with the same cacheKey is served from `service_cache` without invoking the fetch function twice
   - Follows the established pattern from `google-places.service.test.ts`
