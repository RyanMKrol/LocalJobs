# T105 — Enforce one active run per workflow (UI + API + executor)

## Goal
A workflow may have only ONE active run at a time (per-workflow, NOT a global lock —
different workflows still run concurrently). Enforce at all three layers and document.

## What I changed

### Executor (authoritative, race-safe guard) — `src/core/workflow-executor.ts`
- Added an in-process `startingWorkflows` Set<string> (keyed by workflow NAME) and
  exported `workflowRunInProgress(name) = startingWorkflows.has(name) || hasActiveWorkflowRun(name)`.
- `runWorkflow` now does a **synchronous check-and-claim** (no await between
  `workflowRunInProgress` and `startingWorkflows.add`) BEFORE its first await, then runs
  the existing body via an extracted `runWorkflowInner`, releasing the claim in a `finally`.
  - Race-safety: for an UNLIMITED run `createWorkflowRun` already happens synchronously
    before any await, so the DB `running` row exists immediately. For a LIMITED run there's
    an `await inputKeys()` before the run row is created — the in-process claim covers that
    window so two near-simultaneous limited starts can't both pass.
  - The loser returns the existing `{ workflowRunId: null, skipped: true, reason: 'already running' }`.

### API — `src/api/server.ts`
- `POST /api/workflows/:name/run` now checks `workflowRunInProgress(def.name)` immediately
  before firing `runWorkflow`, returning **409 Conflict** `{ error: '… already has an active
  run', running: true }` instead of a fire-and-forget 202 the executor would silently skip.

### Dashboard — `workflows/page.tsx` + `workflows/[name]/page.tsx`
- Both Run buttons (list per-row "▶ Run" and detail "▶ Run now") render a disabled
  **"Running…"** button (with tooltip) when `last_run.status === 'running'`. The detail
  endpoint already returns `last_run` via `workflowView`, as does the list endpoint.

### Tests
- `src/core/workflow-executor.test.ts`: new test — start workflow A (hangs via timeout-* fake),
  a 2nd start of A is refused (`skipped`, no 2nd run row), a DIFFERENT workflow B still starts
  while A runs; both cancelled + guard released after settle. Added `timeout-guard-a/-b` members.
- `src/api/server.test.ts`: new test — seed a `running` workflow_run row (no child spawned),
  assert `POST /run` returns 409 with the `already has an active run` message + `running:true`,
  then clean up the seeded row.

### Docs
- `CLAUDE.md`: file-map row for workflow-executor + a new architecture bullet "One active run
  per workflow (T105)" describing the synchronous claim, the API 409, and the disabled UI.
- `README.md`: "One run per workflow at a time" paragraph in the workflows-running section.
- `.harness/LIMITATIONS.md`: noted the guard is in-process (single-daemon), not a DB lock.
