# T504 — Cache google-places Place Details via cacheKey (22h TTL)

## Summary
Verified that Google Places API Place Details reads are cached for 22 hours, preventing redundant paid API calls during development re-runs on the same place.

## What was done

1. **Verified `cacheTtlMs` on google-places service** (`src/services/google-places.service.ts:20`)
   - Confirmed: `cacheTtlMs: 79_200_000` (exactly 22 hours = 79,200,000ms)
   - This is already set on the `ServiceDefinition` export

2. **Verified cacheKey in enrich stage** (`src/workflows/places/stages/enrich.ts:121`)
   - Confirmed: `cacheKey: `google-places:details:${placeId}``
   - Pattern matches the spec requirement exactly: `google-places:details:<place_id>`
   - Applied only to the read operation (Place Details GET via `fetchPlaceDetails`)

3. **Verified caching scope is read-only**
   - Single `callService('google-places', ...)` call in the enrich stage
   - No mutating calls present
   - Design confirmed: shared service quota governs spend, no per-job redundant cap
