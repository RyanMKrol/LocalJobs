# T027 — Validation gates: declare contracts on real jobs so gates fire

## Goal
T006 built the gate MECHANISM but no real job declared `produces`/`consumes`, so
`deriveGates` returned 0 for both pipelines and zero gates ever fired. Declare
SHAPE + NON-EMPTY `ArtifactContract`s on the perfumes + places pipeline stages so
gates derive and fire at each boundary.

## What I did
- Added `src/jobs/perfumes/contracts.ts` and `src/jobs/places/contracts.ts` —
  small **factory functions** returning `ArtifactContract`s. Each takes an
  optional path defaulting to the job's real `data/` artifact, so the jobs use the
  default while unit tests point at synthetic fixtures (the gitignored real `data/`
  is never needed to test).
- Contracts (key · check):
  - perfumes `fragrantica-urls`: id→URL map exists, non-empty JSON object, every
    value a Fragrantica URL.
  - perfumes `fragrantica-pages`: ≥1 non-empty `<id>.txt` captured page.
  - perfumes `fragrantica-data`: ≥1 parsed `<id>.json` with non-empty `name` +
    `notes` object.
  - places `places-normalized`: places.json with `source: google-takeout` and a
    non-empty `places[]` whose records have a string `name`.
  - places `resolved-place-ids`: resolved.json with a non-empty `resolved` map and
    ≥1 entry carrying a `placeId`.
  - places `enriched-places`: enriched.json with a non-empty `enriched` map and ≥1
    entry carrying a `placeId`.
- Wired them onto the job definitions:
  - `perfumes-find-url` produces urls; `perfumes-fetch` consumes urls / produces
    pages; `perfumes-parse` consumes pages / produces data; `perfumes-build`
    consumes data.
  - `places-ingest` produces normalized; `cid-to-place-id-resolver` consumes
    normalized / produces resolved-ids; `places-enrich` consumes resolved-ids /
    produces enriched; `enrich-with-llm` consumes enriched.
- Added `src/core/contracts.test.ts`: 21 assertions — each contract proven to
  PASS on a well-formed synthetic fixture and FAIL on empty/drifted/missing input,
  plus two tests that `deriveGates` yields 3 gates per pipeline (keys in boundary
  order), plus a key-stability test.
- Docs: README typed-artifact-contracts paragraph now notes both pipelines
  declare them (3 gates each, in `contracts.ts`); CLAUDE.md validation-gates
  convention points at the new `contracts.ts` factory modules + the SHAPE +
  NON-EMPTY intent; LIMITATIONS.md records the representative-sampling trade-off.

## Verification (observed)
- `npx tsc --noEmit` → clean.
- `npm test` → all green; `src/core/contracts.test.ts` reports `21 assertions
  passed`; `src/jobs/registry.test.ts` still loads cleanly (8 jobs, 4 services,
  2 pipelines).
- Empirical gate-derivation against the REAL registry-loaded jobs (throwaway
  tsx script, scratch DB, no commit, no live calls):
  - `PIPELINE perfumes: 3 gate(s) | find-url→fetch:fragrantica-urls,
    fetch→parse:fragrantica-pages, parse→build:fragrantica-data`
  - `PIPELINE places: 3 gate(s) | ingest→resolver:places-normalized,
    resolver→enrich:resolved-place-ids, enrich→llm:enriched-places`
  So a pipeline start log now reports `… · 3 gate(s) · …` (was 0). No
  dashboard change → no dashboard build needed. No live paid-API/scrape calls in
  verification.
