## T568 — 2026-07-14

Added `envInt(varName, fallback)` and `defineService(d)` to `src/services/lib.ts`.

- `envInt`: unset → fallback; set → strict `Number()` parse, throwing an `Error`
  naming `varName` on empty string, non-finite/NaN (catches `'2,000'`), or negative.
- `defineService`: builds a `ServiceDefinition` from a declarative input. Numeric
  fields (`ratePerMinute`, `dailyCap`, `monthlyCap`, `minIntervalMs`, `maxJitterMs`,
  `timeoutMs`, `cacheTtlMs`) accept:
  - a plain `number` — an already-resolved literal, no env lookup;
  - `{ fallback }` — read via `envInt` from an `envPrefix`-derived name (e.g.
    `${envPrefix}_RATE_PER_MIN`);
  - `{ env, fallback }` — read via `envInt` from an explicit bespoke env var name,
    so a service keeping a pre-existing name (gemini's `PLACES_LLM_MONTHLY_CAP`,
    fragrantica's `PERFUMES_FETCH_DELAY_MS`/`_JITTER_MS`, trading212-instruments'
    `TRADING212_INSTRUMENTS_MIN_INTERVAL_MS`) can end up with a byte-identical
    effective limit once migrated (not done in this task — that's R04b).
  - `dailyCap` additionally accepts the bare or fallback sentinel `'monthly/30'`,
    which resolves to `dailyFromMonthly(resolvedMonthlyCap)` (throws if
    `monthlyCap` wasn't also resolved).
  - `rateLimitSource` is a required field on the input type, so omitting it is a
    compile error (verified via a `// @ts-expect-error` test case) — this makes
    vercel's current omission a type error once it's migrated onto this helper
    (R04b), per the task.
