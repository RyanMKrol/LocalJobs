# T093: Design — workflow run-limits via framework-tracked input lineage

## What was done

Authored the design doc `.harness/designs/T093-workflow-run-limits.md` for
"run a workflow capped at N originating inputs". Doc-only task (scope is the
single design file). Wrote it concrete enough for the build task (T094) to
implement directly, grounded in the actual current code (read before writing):
`work_items` schema + `store.ts` ledger API, `executor.ts`/`workflow-executor.ts`/
`dag.ts`/`runJob.ts` run path, `db/index.ts` migrations, the `POST
/api/workflows/:name/run` endpoint, the perfumes + places job files, and the
dashboard Run control.

The doc covers every required section:

1. **Lineage data model** — additive nullable `root_key` + `parent_key` columns on
   `work_items` (+ `idx_work_items_root`); root-resolution rule in `markWorkItem`
   (explicit `rootKey` → inherit from `parentKey` → default `root_key = item_key`);
   `store.ts` additions (`markWorkItem` lineage opts, `distinctRoots`/`rootCount`,
   `selectPendingRoots`, `getWorkflowRunRoots`, `createWorkflowRun` carrying the
   limit).
2. **Limit semantics + selection** — first-N-pending roots, deterministic by the
   root stage's `inputKeys()` order; "pending" = entry-not-done OR any-descendant-
   not-done; selection happens once in the parent at run start and is frozen on the
   run row; every stage filters input to `root_key ∈ allowlist`; pre-root stages
   (e.g. `places-ingest`) run unfiltered.
3. **Invocation** — `POST /api/workflows/:name/run { limit? }` with validation +
   `hasRootStage` guard; `runWorkflow(def, trigger, { limit })`; thread
   `LOCALJOBS_WORKFLOW_RUN_ID` env to the child (child reads the frozen allowlist
   from the run row — it already has DB access); scheduler stays unlimited.
4. **Job contract** — `JobContext.selectedRoots()`/`rootAllowed()` built in
   `runJob.ts` from the env var + run row; lineage declared via `markWorkItem`
   opts; root enumeration reuses the existing `inputKeys()` hook (no new
   `JobDefinition` field).
5. **Backward compat** — unlimited ⇒ `rootAllowed` always true ⇒ filters are
   no-ops; default `root_key = item_key`; migration backfills existing rows;
   minimal per-stage edits spelled out for perfumes (same key — needs only
   `inputKeys()` + a filter clause) and places (key-changing `cid→place_id` — needs
   lineage on enrich/llm `markWorkItem`).
6. **UI** — limit number input beside ▶ Run now (rendered only when limitable);
   "limited · N" badge on the run-detail page; `api.runWorkflow(name, limit?)`.
7. **Migration** — additive `work_items` columns + backfill and `workflow_runs`
   `run_limit`/`selected_roots` in `db/index.ts` + `schema.sql` (column named
   `run_limit`, not the SQL keyword `limit`).

Plus: glossary, current-state facts the design relies on, three worked traces
(perfumes 1:1, places key-change, hypothetical fan-out), the smallest viable first
cut (with a truly-minimal perfumes-only fallback), risks/open questions (the
key-changing-stage lineage trap is the biggest), and a file-by-file touch list for
T094.
