# T005 — Unit tests for the framework core

## Goal
Add meaningful unit-test coverage for the framework core (previously only `dag.ts` +
store helpers were tested): executor (timeout-kill / retries / event parsing),
pipeline-executor (skip / aggregation / repeat-until-stable), scheduler (croner
respects `enabled`), the `callService` limiter (quota / throttle / min-interval),
and the notifier's header sanitiser.

## What I built
New `*.test.ts` files (discovered + run by `npm test`, self-asserting, scratch DB):

- **`src/core/notifier.test.ts`** — exercises the header sanitiser across ASCII,
  emoji, accented/CJK Unicode, control chars, trimming, and the all-non-ASCII →
  empty (caller falls back to `localjobs`) case, plus the printable-ASCII boundary.
- **`src/core/services.test.ts`** — `callService`: unregistered → ungated;
  no-limit → meters one row/call; monthly & daily quota → `QuotaExceededError`
  (retryable, fn not run); rate throttle (first call immediate, no-slot → throws
  on `maxWaitMs:-1`); min-interval spacing (second call throttles ~2s, `onThrottle`
  fires) + min-interval-takes-precedence timeout throw.
- **`src/core/scheduler.test.ts`** — swaps the registry `jobs`/`pipelines` arrays
  for fakes (so NO real metered job is ever scheduled), schedules an enabled +
  disabled job at `* * * * * *`, and asserts: both register (gate is at fire-time),
  the pipeline-member job gets no own cron, the ENABLED job fires (observed via the
  scheduler's "skipped: already running" log — pre-seeded running row means no child
  spawns/notifies), the DISABLED job never fires (no run rows), and `stopScheduler`
  clears the cron. Registry arrays restored in `finally`.
- **`src/core/executor.test.ts`** — points `config.runJobScript` at a fake
  NDJSON-emitting child (temp file) and drives the engine via `runJobForPipeline`
  (shares the attempt/spawn machinery but does NOT notify, so no osascript/ntfy):
  success + event parsing (log/progress/result + non-JSON stdout still logged),
  failure result → error recorded, retry (fail×2 then success → 3 attempts via a
  counter file), timeout-kill (hanging child SIGTERM'd → `timeout`), and both
  overlap guards (`runJob` skip without spawn; pipeline member → skipped run row).
- **`src/core/pipeline-executor.test.ts`** — runs real `runPipeline` over fake
  members: all-success → `success`; failed upstream → dependent skipped → `partial`;
  `repeatUntilStable` cycles twice while a seeded retryable work item keeps it from
  going stable; invalid DAG (cycle) → no members run, finished `failed` with an
  error log. ntfy force-disabled (empty topic) so no network call.

## Source change needed for testability (minimal, justified)
- **`src/core/notifier.ts`**: extracted the inline `headerSafe` closure into an
  exported pure `sanitizeHeader()` (so it can be asserted directly) and added an
  `'error'` handler to the `osascript` spawn. The latter is a real hardening bug
  fix: on a non-mac host (incl. CI ubuntu) `spawn('osascript')` emits an async
  `'error'` (ENOENT) with no listener, which would crash the job run; swallowing it
  is correct and also lets tests exercise the notifier path safely.
