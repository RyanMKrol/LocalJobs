# T152 — Relocate movie-gaps management into the movies workflow page

## What was built
Moved the franchise-gaps manage/ignore UI off its dedicated top-level page onto the
**movies workflow detail page**, and removed the global nav entry.

1. **Removed the top-level page + nav.**
   - Deleted `dashboard/app/movie-gaps/page.tsx` (and the now-empty folder) via `git rm`.
   - Removed `<a href="/movie-gaps">Movie gaps</a>` from `dashboard/app/layout.tsx`.
2. **Added a "Recommendations & gaps" management section** to
   `dashboard/app/workflows/[name]/page.tsx`:
   - New `MovieGapsManager` component (moved `groupByCollection` + the active/ignored
     gap tables + per-item ✕ Ignore control over from the deleted page).
   - Reuses the EXISTING API/client methods unchanged — `api.movieGaps()` (`GET
     /api/movie-gaps`) and `api.ignoreMovieGap()` (`POST /api/movie-gaps/:tmdbId/ignore`).
     No server.ts or api.ts changes were needed.
   - Gated with `{name === 'movies' && <MovieGapsManager />}` so it renders ONLY for the
     movies workflow (kept the simple string gate per the spec — the API surfaces only
     this one workflow's outputs, so a declarative signal would be over-engineering).
   - Section placed after the Runs table, consistent with the page's other `<h2>` sections;
     inner panels use `<h3>` so headings nest correctly under the page `<h1>`.
3. **Ignore-to-suppress preserved** — unchanged `ignoreSurfacedItem` path via the same
   endpoint; ignoring a gap still drops it from future reports + notifications.
4. **Docs updated in lockstep** — README.md (movies section) + CLAUDE.md (T145 ignore note)
   now describe the management UI as a section of `/workflows/movies`, not a top-level page.

### Note on a pre-existing, unrelated local test artifact (NOT caused by T152)
`src/jobs/movies/stages/notify.test.ts` fails **only in this local checkout** with
`/2 franchise gaps detected/` not matching `🎬 2 franchise gaps · 🍿 8 recommendations`.
Root cause: Run 1 of that test does not pass `recsFile`, so `runNotify` defaults to the
REAL `src/jobs/movies/data/out/recommendations.json` (8 fetched recs present locally),
leaking real data into the digest title. On CI — where `data/` is gitignored/absent — the
default file doesn't exist (0 recs) and the assertion passes. **Verified**: temporarily
moving the real recs artifacts aside (guaranteed-restore) made the whole notify test pass.
This is a pre-existing test-isolation bug from the T146 recs layer, OUTSIDE T152's scope
(`src/jobs/movies/stages/notify.test.ts` is not in scope), and does NOT affect CI or T152.
Did not touch it; flagged here for a follow-up (the test should pass an isolated `recsFile`
/`historyFile` in runs 1–5).
