# T059: Status labels — consistent success/failure wording (parity)

## What was done

Added a `statusLabel(status: string): string` function to `dashboard/app/ui.tsx` with a
centralized map of all status values to human-friendly display labels:

| Raw value   | Display label |
|-------------|---------------|
| `success`   | Succeeded     |
| `failed`    | Failed        |
| `running`   | Running       |
| `queued`    | Queued        |
| `timeout`   | Timed out     |
| `cancelled` | Cancelled     |
| `skipped`   | Skipped       |
| `partial`   | Partial       |
| `passed`    | Passed        |
| `pending`   | Pending       |

Updated every render site that was previously printing the raw status value:

- `dashboard/app/ui.tsx` — `StatusBadge` now calls `statusLabel(status)` instead of rendering raw
- `dashboard/app/page.tsx` — workflow card badge + recent runs table
- `dashboard/app/workflows/page.tsx` — last-run badge in the workflows table
- `dashboard/app/workflows/[name]/page.tsx` — run history table
- `dashboard/app/workflow-runs/[id]/page.tsx` — workflow-run header badge + gate state badges
- `dashboard/app/components/Dag.tsx` — DAG node status text

No underlying status values (DB, API, types) were changed — display only.
