# T328 — Convert IDEAS.md inbox bullets to numbered style (in lockstep with the consolidation parser)

This is a `needs-human` gated task; the owner explicitly asked for it to be attempted directly in
this session (interactively) rather than via the autonomous loop.

## What changed (both halves, landed together)

1. **`.harness/IDEAS.md` reformatted.** Its single remaining bullet (the Plex file-naming idea) is now
   `1. ` instead of `- `. The file header comment was also updated to document the numbered-bullet
   convention going forward, and to record the decision that numbers are **never renumbered or
   reused** after a bullet is converted/deleted — new bullets always continue from
   `(highest number ever used) + 1`. This is a deliberate judgment call (not asked about explicitly):
   renumbering on every consolidation run would add real complexity (rewriting every surviving
   bullet's marker each time) for a purely cosmetic property (numbers here are just a sequence marker,
   never referenced elsewhere the way task ids are), so gaps are accepted.
   - `.harness/IDEAS.md` is gitignored — this edit is on disk but there is nothing to commit for it.

2. **`.harness/consolidate-ideas.mjs`'s bullet-boundary parser updated.** Replaced the literal
   `lines[i].startsWith('- ')` checks (both the span-start test and the span-end lookahead) with a
   `NUMBERED_BULLET_RE = /^\d+\.\s/` regex test, matching `1. `, `2. `, `10. `, etc. Span-end logic
   (stop at the next bullet marker or the next `## ` heading) is unchanged. Chose numbered-ONLY
   support (not "both `- ` and numbered") since the spec explicitly reformats every existing bullet in
   the same change — there's no longer a mixed-format file to support, and supporting both would be
   dead code the moment this lands.

3. **`.claude/commands/idea.md` (the `/idea` append path) updated to match.** `/idea` now appends
   `<N>. <idea text>` instead of `- <idea text>`, where `N` = `(max existing bullet number in the
   file) + 1` (scanning all `<digits>. ` bullets under `## Inbox`, not just counting bullets — a
   deleted bullet leaves a gap, and the instructions explicitly call out not to just count). This
   resolves the spec's step 3 question: yes, the append path needed updating, and it's done in this
   same task (not deferred) — a fresh `/idea` capture right after this change would otherwise revert
   the file to mixed/broken format on its very next use.
