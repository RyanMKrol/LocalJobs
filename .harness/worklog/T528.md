# T528 — 2026-07-13

Fixed stocks-sync's gain-alert breach notification: the `<account:ticker>::notified`
episode row was previously written by `stocks-watch` the instant a fresh breach was
detected — one stage before the push actually happens in `stocks-notify`, and
`stocks-notify` ignored the `pushFn` result, logging "notification sent"
unconditionally. A failed/interrupted push between the two stages permanently lost
the alert (the episode looked "notified" even though nothing was delivered).

## Changes

- `src/workflows/stocks-sync/stages/stocks-watch.ts`: removed the
  `markWorkItem(WATCH_JOB, notifiedKey, 'success', ...)` call on fresh-breach
  detection. Everything else unchanged: `alreadyNotified` check, the
  reset-to-`skipped` path on drop-below-threshold, the unconditional per-run check
  row, and writing `fresh-breaches.json`. Also removed `formatBreachLine`/
  `buildDigest`, now dead duplicates of `stocks-notify`'s own copies (confirmed via
  repo-wide grep no other caller referenced watch's copies).
- `src/workflows/stocks-sync/stages/stocks-notify.ts`: captures `const res = await
  pushFn(...)`; throws `Breach push failed — <error>` on `!res.ok` before marking
  anything; on a verified-ok push, marks each fresh breach's
  `positionKey(account, ticker)::notified` row `success` (mirroring the detail shape
  `stocks-watch` used to write); the "notification sent" log now only fires when
  `res.ok` is true (mirrors `movies/stages/notify.ts`'s verify-then-mark pattern).

## Tests

- `stocks-notify.test.ts`: added (d) a failing `pushFn` throws AND leaves no
  `::notified` row for the fresh breach; (e) an ok `pushFn` marks the `::notified`
  row exactly once, and a second full pipeline run (watch + notify) with that row
  present does not re-add the position to `fresh-breaches.json` or send a duplicate
  push.
- `stocks-watch.test.ts`: updated case (b) to assert `stocks-watch` no longer writes
  the `::notified` row on fresh-breach detection. Cases (c) and (f), which rely on
  "already notified" state to prove a sustained/duplicate breach is suppressed, now
  call a new `simulateNotified(account, ticker)` helper (a direct `markWorkItem`
  call) to stand in for a completed `stocks-notify` run, since that responsibility
  moved out of `stocks-watch`.
