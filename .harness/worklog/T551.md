# T551 — api.ts path encoding, postWithServerError helper, modal error state

Date: 2026-07-14

## What I did
- `dashboard/app/lib/api.ts`:
  - `encodeURIComponent`'d the interpolated `name`/`job` in `job()`, `jobRuns()`,
    `stuck(job)`, `ignored(job)`, `workflow()`, `runWorkflow()`, `toggleWorkflow()`
    — matching every other endpoint that already did this.
  - Added a generic `postWithServerError<T>(path, body)` helper (fetch + JSON
    parse + `if (!res.ok) throw new Error(data.error || ...)`) and routed the five
    hand-rolled functions (`updateWorkflowSchedule`, `updateWorkflowConcurrency`,
    `updateWorkflowNotify`, `updateWorkflowCertified`, `updateJobTimeout`) through
    it, preserving each one's original return shape via the generic type param.
- `dashboard/app/components/StageIoLists.tsx`:
  - `ModalState` gained an `error` variant. `StageIoModal` now renders an in-modal
    `Failed to load output: <message>` line instead of the caller closing the
    modal on fetch failure. Both `StageIoBlock`/`StageIoOverallBlock`'s
    `openModal` callbacks now `.catch` into `{ loading: false, title, error }`
    instead of `setModal(null)`.
- `dashboard/app/components/WorkflowOutputSection.tsx`:
  - `OutputModal` gained an `error` prop rendering the same in-modal error line.
    `openOutput`'s catch now sets `{ item, result: null, error }` (keeping the
    modal open) instead of `setModal(null)`; `loading` is now
    `result === null && !error` so the error state doesn't show a spinner.
