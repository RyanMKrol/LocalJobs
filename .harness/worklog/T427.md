# T427 — Dashboard: convert internal navigation to next/link for real client-side SPA transitions

## What was done

Converted every internal (same-origin) `<a href="...">` across `dashboard/app/**` to
Next's `<Link href="...">` (`import Link from 'next/link'`), keeping every prop
(className/style/title/aria-label) unchanged. Genuinely external links — the TMDB
`https://www.themoviedb.org/...` links in `app/workflows/[name]/page.tsx` (all carrying
`target="_blank" rel="noreferrer"`) — were left as plain `<a>`, untouched.

Files converted: `layout.tsx` (brand + 6-item header nav — the highest-impact fix, since
it's rendered on every page), `page.tsx`, `ui.tsx` (`WorkflowRunBackLink` + the stuck-item
job link — `useMode`/`ThemeControls` untouched), `admin/page.tsx`, `workflows/page.tsx`,
`workflows/[name]/page.tsx` (only its 2 internal links; ~8 TMDB links left alone),
`workflows/[name]/gates/[producer]/[key]/page.tsx`, `workflow-runs/[id]/page.tsx`,
`workflow-runs/[id]/gates/[producer]/[key]/page.tsx`, `integrations/page.tsx`,
`components/DagFlow.tsx` (both the stage-node link and the gate-padlock link — both
internal, confirmed by reading `buildLayout`'s href construction), `logs/page.tsx`,
`runs/[id]/page.tsx`, `jobs/[name]/page.tsx`.

Confirmed every ambiguous case by reading the surrounding code before converting:
`ui.tsx`'s `fallback.href` (internal — always `/workflow-runs/...`, `/runs`, or a
job/workflow route), `DagFlow.tsx`'s `data.href`/`gateHref` (internal —
`/runs/`, `/jobs/`, `/workflow-runs/.../gates/...`, `/workflows/.../gates/...`),
`logs/page.tsx`'s `href` (internal — `/runs/${id}` or `/workflow-runs/${id}`),
`jobs/[name]/page.tsx`'s `back.href` (internal — resolved via `backFrom()`, defaults
to `/workflows`).

`app/backlog/page.tsx` has no `<a href>` at all (its "links" are click-driven
`DepIdLink` buttons that scroll/expand in place, not navigation) — nothing to convert
there.

## Hermetic nav check (new)

Added `dashboard/scripts/nav-check.test.ts` — a self-running (`npx tsx
dashboard/scripts/nav-check.test.ts`), Playwright-based check reusing the shared
harness (`startDashboard`/`waitForServer`/`routeApi` from `_dashboard-harness.mjs`, no
daemon/SQLite/paid calls). It plants `window.__t427Marker = true` after the first page
load, clicks an internal nav link, and asserts the marker SURVIVES (a real full-page
reload would wipe it) plus that the URL actually changed. Two assertions: clicking
"Workflows" in the header nav, and clicking the brand link back to `/`. Placed as a
`*.test.ts` file (always in scope per the task's scope gate) rather than a new `.mjs`
script, since the scope only allows editing the existing `_dashboard-harness.mjs`, not
adding new top-level scripts — this stays consistent with the repo's existing
self-running `*.test.ts` convention (`src/core/dag.test.ts` et al.) while reusing the
dashboard's hermetic harness.

Observed: both assertions pass —
```
✓ clicking a header nav link does not trigger a full page reload
✓ clicking the brand link navigates client-side back to Overview
✓ all 2 nav check(s) passed
```
