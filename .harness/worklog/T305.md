# T305 — service `category` field (backend/schema half)

## What was built

Mirrored T292 (workflow `category`) for `ServiceDefinition`/`services`:

- `src/core/types.ts`: `ServiceDefinition.category?: string`, documented with the 3
  controlled values (`cli-tool` | `website-scrape` | `api`).
- `src/db/schema.sql`: `services` table gets `category TEXT NOT NULL DEFAULT ''`.
- `src/db/index.ts`: additive `ALTER TABLE services ADD COLUMN category ...`
  migration for existing DBs, guarded by `PRAGMA table_info`, no index.
- `src/db/store.ts`: `ServiceRow.category`; `upsertServiceStmt` inserts +
  unconditionally refreshes `category` (no `_overridden` gating, matches
  `description`/`paid`); `syncService` passes `def.category ?? 'uncategorized'`.
- Set `category` on all 9 manifests:
  - `cli-tool`: `claude-cli`
  - `website-scrape`: `fragrantica`
  - `api`: `gemini`, `github`, `google-places`, `hevy`, `lastfm`, `tmdb`,
    `trading212`, `dynamodb`
- `CLAUDE.md`: new bullet documenting `category` as manifest-owned only, no
  dashboard edit UI, controlled values + fallback.

## Verification

- `GET /api/services` already spreads `...s` from `listServices()` in
  `src/api/server.ts` (confirmed by reading, no code change needed) — `category`
  flows through automatically now that it's a real column.
- Added a unit test in `src/db/store.test.ts` (T305 block) seeding two services
  through `syncService` (one with `category`, one without), asserting the synced
  row/listed row match the manifest value or fall back to `uncategorized`, and
  that re-syncing with a DIFFERENT category value updates the stored value (no
  override/preservation behavior).
- `npx tsc --noEmit` — clean.
- `npm test` — 85/85 pass (all suites), including the new T305 assertions.

No paid API calls made. No dashboard changes (T306 is the dependent task for the
rendering half).
