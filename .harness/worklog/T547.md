# T547 — Harden usePoll against stale/out-of-order responses

2026-07-14

## What I did

- `dashboard/app/ui.tsx`: extracted the non-React core of `usePoll` into a new
  exported `createPollController<T>()` (plus a small `PollDocumentLike`
  interface) so the ordering/visibility logic is testable without mounting a
  React tree.
  - Added a monotonically-increasing sequence counter (`seq`). Each `tick()`
    captures its own sequence number before awaiting `fn()`, and only calls
    `setData`/`setError` when `seq === mySeq` — i.e. it is still the newest
    in-flight tick. An older tick that resolves after a newer one is
    discarded and can never overwrite fresher data.
  - `intervalMs` is now included in `usePoll`'s effect dependency array
    (`[intervalMs, ...deps]`), so a dynamic interval takes effect on the next
    run of the effect (previously `intervalMs` changes were silently ignored
    since only `deps` was passed to `useEffect`).
  - Added `document.visibilityState` pause/resume: the interval is not
    started (and is torn down) while `document.visibilityState === 'hidden'`,
    and a `visibilitychange` listener stops the interval on hidden and, on
    becoming visible again, does an immediate `tick()` (refetch) before
    restarting the interval. Guarded with `typeof document !== 'undefined'`
    checks (and an injectable `doc` option) so it's a no-op/testable outside
    a browser DOM.
  - `usePoll` itself is now a thin wrapper: it creates a `createPollController`
    inside its `useEffect`, wires `setData`/`setError` from `useState`, and
    tears the controller down (which also detaches the visibility listener)
    on cleanup. `refetch()` now calls `controllerRef.current?.refetch()`.

- `dashboard/app/ui.test.ts`: added two pure tests against
  `createPollController` (no React rendering needed — the dashboard has no
  jsdom/testing-library dependency, and adding one would require a
  package.json change outside this task's scope):
  1. "discards a stale response that resolves after a newer one" — fires two
     ticks (constructor tick + a `refetch()`), resolves the *newer* call's
     deferred promise first, then the *older* one, and asserts the recorded
     data is only ever the newer value (the older, late resolution is
     discarded).
  2. "pauses its interval while hidden and resumes+refetches on visible" —
     injects a fake `PollDocumentLike` (controllable `visibilityState` +
     capturable `visibilitychange` handler) and fake `setIntervalFn`/
     `clearIntervalFn` stubs, then asserts: the interval starts once while
     visible; going hidden stops the interval and no further fetch happens;
     coming back visible triggers an immediate refetch and restarts the
     interval.
