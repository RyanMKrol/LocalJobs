# T014 worklog — Framework: manual prune of orphaned ledger rows

## Attempt 1 — 2026-06-21

### Goal
Build a framework feature to prune `work_items` whose `item_key` is no longer in
a job's current input — the case that orphaned the 3 old-id perfume rows after
their ids were corrected. Manual-only (never automatic), and surface what it
removed.

### Design (stayed within scope: store.ts, types.ts, server.ts, store.test.ts)
- **`JobDefinition.inputKeys?(): string[] | Promise<string[]>`** (types.ts) — a
  job optionally exposes its current input key-set (the `itemKey`s it would mark
  in the ledger; for the perfume jobs that's `perfumes.map(p => p.id)`). Doc note
  makes clear nothing in the run/schedule path calls it.
- **`orphanedWorkItems(jobName, currentKeys)`** (store.ts) — read-only preview:
  ledger rows for the job whose key ∉ current set. **`pruneOrphanedWorkItems(...)`**
  deletes them in a transaction and returns the removed rows so the caller can
  surface exactly what was pruned. Current keys kept untouched.
- **`POST /api/jobs/:name/prune`** (server.ts) — the manual trigger. Current set
  comes from request `{ keys: [...] }` (lets a job WITHOUT `inputKeys()` — like
  the perfume jobs today — be pruned by passing the corrected id set), else the
  job's `inputKeys()`. `{ dryRun: true }` previews. An empty current set is
  refused unless `{ force: true }` — guard so a misbehaving `inputKeys()` can't
  wipe the whole ledger. Logs the removal count.

The user clears the 3 orphaned perfume rows by POSTing the perfumes job's current
id set (or wiring `inputKeys()` later) — I did not touch their data (privacy /
out of scope) or run any deletion against the live DB.

### Tests (store.test.ts)
Added a T014 block: preview surfaces only orphans and mutates nothing; prune
removes orphans + returns them + keeps current; idempotent second prune is a
no-op; a `Set` works as the input set and an unknown key is harmless; empty set
orphans everything (store obeys; API guards).
