# T299 — workouts-progress: 6-month per-exercise progress report

## What was built

- `src/jobs/workouts-sync/stages/workouts-progress.ts`: reads
  `data/out/workouts-history.json` (the T298 accumulator), computes calendar-month
  baseline (6 months before current) and current (most recently completed month)
  periods, buckets sets per exercise (`exercise_template_id`), and computes best
  single set, total volume, and estimated 1RM (Epley) per period. Sets with a null
  `weight_kg`/`reps` are skipped from the metrics (duration/distance exercises); an
  exercise with no usable sets in either period is dropped entirely. Writes
  `data/out/progress-data.json` (raw comparison) then calls `runClaude` (shared
  helper from `src/services/claude.ts`, unmodified) to narrate the comparison into
  `data/out/workouts-progress.md`. Idempotent per calendar month via the
  `work_items` ledger, keyed by the current period's `YYYY-MM` — mirrors
  `listening-digest`'s pattern. Both output files use static names (no month in the
  filename), so a same-month re-run overwrites rather than duplicates.
- `src/jobs/workouts-sync/stages/workouts-progress.job.ts`: thin job wrapper.
- `workouts-sync.workflow.ts`: added `workouts-progress` as a second member with
  `dependsOn: ['hevy-sync']` (same `dependsOn` shape as `projects-sync`'s
  `project-summarize` stage).
- `workouts-progress.test.ts`: unit tests with fixed injected workout history
  (two workouts 6+ months apart) verifying period math (current = most recent
  completed month, baseline = 6 months earlier), and numeric correctness of all
  three metrics for 2 exercises (squat has data in both periods; bench has
  baseline-only and current-only data) plus exclusion of a March-dated workout
  (outside both windows) and of a duration-only (null weight/reps) exercise.
  Also verifies same-month re-run overwrites the static-named report file and
  ledger entry rather than duplicating, and that a missing history file / a
  failed Claude call are handled (no-op / throw respectively). Claude is
  injected via a fake `runClaudeFn` — no live CLI calls.

  follow-up manual spot-check for the owner (not blocking — the task's Definition
  of Done accepts the injected-fake-Claude unit test as sufficient, with a live
  check as "a reasonable one-time check" rather than a hard requirement).
