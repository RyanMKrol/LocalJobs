# T573 — Logging hygiene sweep: drop baked-in info:/warn: prefixes, pass real level; name stocks-watch ledger rows

2026-07-15

## What I did

(A) Swept every `ctx.log(...)` call in the 9 stage files under
`src/workflows/stocks-sync/stages/` and `src/workflows/stock-digest/stages/`:
- Removed literal `'info:'`/`'warn:'`/`'error:'` prefixes from every log message string.
- For every line that was previously `warn:`-prefixed text logged at the default info
  level, passed the real `'warn'` level arg instead (info is the default, so a plain
  info line just drops the prefix with no second arg).
- Explicitly fixed the two lines named in the spec:
  - `stocks-resolve-names.ts` (`could not resolve company name for Trading212 ticker...`)
    — now logs at `'warn'`.
  - `stock-portfolio-snapshot.ts` (`ticker resolution failed — continuing without
    isin/resolvedTicker...`) — now logs at `'warn'`.
- Also caught the same pattern in `stock-sector-lookup.ts` (4 warn-prefixed lines) and
  `stock-digest-build.ts` (4 warn-prefixed lines), which the spec's design note flagged
  as "at least two cases" without being exhaustive.
- Left `stocks-notify.ts`'s `res.ok ? 'info' : 'error'` ternary alone — it already
  passed a real level arg and carried no baked-in text prefix.
- Left one line in `stocks-watch.ts` ("dropped back below threshold, resetting ledger")
  at the default info level — it was never `warn:`-prefixed in the original text, just
  a normal state-transition log line, so no level change was warranted there.

(B) `stocks-watch.ts`'s per-run position-check `markWorkItem` (the unconditional
per-position ledger write) previously recorded `detail: { gainPct, breaching }` with no
name/ticker. Added `name: label, ticker: position.ticker, account: position.account`
alongside the existing `gainPct`/`breaching` fields, matching the shape the nearby
reset-row `markWorkItem` call already uses.

## Tests

- Extended `stocks-watch.test.ts`'s existing no-breach-run assertion to also fetch the
  per-run check ledger row via `getWorkItem` and assert `detail.name`/`detail.ticker`
  are populated (no longer blank-labeled).
- `stocks-resolve-names.test.ts` had a pre-existing assertion asserting the log message
  text started with the literal `'warn:'` prefix — since the prefix is now gone in
  favour of the real level arg, updated `fakeCtx` to also capture the level passed to
  `ctx.log`, and changed the assertion to check `level === 'warn'` instead of the
  removed text prefix. This is the only test that needed updating for the level-arg
  migration; no other test in either workflow's stage files asserted on message
  prefixes.
