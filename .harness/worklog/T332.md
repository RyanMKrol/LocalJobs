# T332 — rec-auteur / franchise-gaps node overlap on the run-view DAG

## Root cause (confirmed, not guessed)

Confirmed **Lead B**: in `buildLayout()` (`dashboard/app/components/DagFlow.tsx`), each node's
X was pinned to its true wave depth (`computeLayers`/`colX`) but its Y was taken straight from
`g.node(m.job_name).y` — dagre's own rank-scoped vertical position. Dagre's non-overlap guarantee
for Y only holds among nodes it placed in the *same rank*; since franchise-gaps is genuinely one
dagre-rank closer to its consumer (`movie-gaps-notify`, via `rec-merge`) than the 8 `rec-*`
branches are, dagre puts it in a *different* rank from them even though the X-override draws it
in the *same visual column*. Its dagre-assigned Y is only spaced apart from other nodes in *its
own* rank, not from the 8 branches it's actually drawn beside — so nothing prevented it landing
very close to whichever branch (rec-auteur, in the real 9-node fan-out) sat nearest on the Y axis
purely by rank-coincidence.

Instrumented by temporarily logging `g.node(name).y` per node for the real 12-member
`movie-recommendations` DAG (movie-snapshot, franchise-gaps, 8 `rec-*`, rec-merge,
movie-gaps-notify) built from `src/jobs/movies/movies.workflow.ts`'s member list: franchise-gaps'
raw dagre y sat within ~1px of rec-auteur's, while every other pair in the wave-1 column had the
full `NODE_H + NODE_SEP` (86px) gap dagre reserves within a single rank. This reproduces exactly
in BOTH the structural and run view's raw dagre math (Lead B is view-independent, confirming the
spec's note that it alone doesn't explain why only run view visibly overlapped) — Lead A's
run-view-only `.dag-node-status` line (extra ~14px of real height with no fixed `NODE_H` floor)
was the tipping factor that turned an already-near-zero baseline gap into a visible overlap only
when the status line was rendered (`showStatus = !!statusByJob`, true only in run view). Lead C
(the gate padlock `EdgeLabelRenderer` overlay) does not contribute to node overlap — it's a
separate absolutely-positioned element at edge midpoints, unrelated to node Y spacing.

## Fix

`dashboard/app/components/DagFlow.tsx`: re-derive each node's Y **within its own wave column**
instead of trusting dagre's rank-scoped Y. For each wave (the same grouping used for the X
override), the members drawn in that column are sorted by their raw dagre Y (preserving dagre's
crossing-minimised relative order) and then assigned evenly-spaced Y positions
(`i * (NODE_H + NODE_SEP)`). This guarantees every node that is actually drawn beside another node
gets the same reserved gap, regardless of which dagre rank it originally came from.

As a second, independent guard against Lead A: `.dag-node` in `dashboard/app/globals.css` now has
`min-height: 58px` (== `NODE_H`) with `box-sizing: border-box`, so the real DOM height (with or
without the run-view status line) can never fall below what dagre assumed when reserving vertical
space, and the status line's growth can no longer eat into the reserved `NODE_SEP` gap on its own.
