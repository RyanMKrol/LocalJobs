# T099 — Graph view: 5 toggleable validation-gate styles to reduce footprint

## Goal
The single chip-per-gate (T090, ⛒ producer · key) on the graph views takes too much
space. Produce FIVE distinct, more compact gate-display styles, let the user toggle
between them live, persist the choice. Must work on BOTH the workflow-run DAG and the
structure-only workflow graph, at desktop AND phone widths; every gate stays clickable
to its detail page and state colouring stays distinguishable. Evaluation aid — a
follow-up will hardcode the winner and remove the toggle.

## Starting state
The working tree held a PARTIAL interrupted attempt (`Dag.tsx`, `ui.tsx` modified). It
was ~90% there but did NOT compile:
- `useGateStyle()` was imported but never called in `Dag` → `style`/`setStyle`
  referenced but undefined (lines using `<GateStyleBar value={style} …>`, `GateMarks
  … style={style}`).
- The render was wrapped in a `<>…<div className="dag">` but the closing `</>` fragment
  tag was missing → JSX parse error.
- No CSS existed for any of the new `gs-*` / `gate-conn` styles or the selector bar.

## What I did (only the outstanding work vs doneWhen)
1. `Dag.tsx`: added `const [style, setStyle] = useGateStyle();` at the top of `Dag`;
   closed the fragment (`</div></>`). The pre-existing partial logic (RenderGate
   normalisation, `GateMarks`, `GateStyleBar`, connector special-case) was sound and
   left as-is.
2. `ui.tsx`: `GATE_STYLES` (source of truth: icon/dot/key/connector/lock) +
   `useGateStyle()` hook persisting to `localStorage['localjobs.gateStyle']`, SSR-safe
   (reads in an effect after mount; defaults to `icon`). This was already present from
   the partial attempt and is correct.
3. `globals.css`: added the `.gate-style-bar` selector styling and the five compact
   variants:
   - `.dag-gate.gs-icon` / `.gs-lock` — bare glyph, no chip frame, state colour inherited.
   - `.dag-gate.gs-key` — tiny pill of just the artifact key.
   - `.dag-gate.gs-dot` — 9px state-coloured dot (green/red/grey, dashed for structural).
   - `.dag-arrow-glyph.gate-conn.*` — colours the arrow glyph itself per state, clickable.
   Plus mobile-media-query overrides so the connector colour survives the `→`→`↓`
   glyph swap (the existing `.dag-arrow-glyph::after{color:grey}` would otherwise win).
4. Page files (`workflow-runs/[id]`, `workflows/[name]`) needed NO change — the selector
   is internal to `Dag`, so it appears on both views automatically whenever a DAG has
   gates (`hasGates`).
5. Docs: updated the gate-rendering paragraph in `CLAUDE.md` and added a trade-off note
   to `.harness/LIMITATIONS.md` (toggle is temporary, follow-up hardcodes the winner).
