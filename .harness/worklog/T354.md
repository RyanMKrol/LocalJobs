# T354 — fix stale log lines surviving a Logs page filter change

## Root cause confirmed

Server-side SQL filtering (`listGlobalLogs()` in `src/db/store.ts`) is correct — verified by
reading the query builder end to end; no change made there. The bug is client-side in
`dashboard/app/logs/page.tsx`: the "reset accumulated list on filter change" effect seeded `lines`
from `data` (the PREVIOUS filter's poll result, since the new fetch triggered by `usePoll` in the
same dependency-array flush hasn't resolved yet). The later "merge fresh data" effect only ADDS
rows never seen before — it never prunes — so the stale, wrong-filter rows it just seeded stuck
around permanently, mixed in with the newly-fetched correctly-filtered rows.

## Fix

- `dashboard/app/logs/page.tsx`: the filter-change effect now clears `setLines([])` /
  `setNextCursor(null)` instead of seeding from `data`. Added a code comment documenting the
  narrower known limitation (double filter change within one round trip could still transiently
  reintroduce stale rows) — explicitly left unfixed per the spec, since fixing it needs request
  cancellation/generation tracking, out of scope here.
- `dashboard/scripts/_dashboard-harness.mjs`: the `/api/logs` fixture handler now reads
  `searchParams.get('level')`, splits on `,`, and filters the static 6-line array to matching
  levels when present; returns the full set when the param is absent (mirrors
  `listGlobalLogs()`'s real behaviour).
