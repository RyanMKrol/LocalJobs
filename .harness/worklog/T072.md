# T072 — Fragrantica accord percentages still null — evaluate + fix end-to-end

## Root cause (confirmed)
The parser (`parse.ts::parseAccordPercents`) was already correct. The bug was
purely upstream in `fetch.ts`: on the **success path** it persisted only the page
innerText (`pages/<id>.txt`) and NEVER the raw HTML. The accord-bar widths — each
accord's strength as an inline `style="…width: NN%…"` — are dropped by innerText.
`applyAccordPercents` reads them back from `pages/<id>.html` (via `readPageHtml`),
which only ever existed for **failed** pages (`pages-failed/<id>.html`). So on
every successful capture there was no HTML to read → every `pct` fell back to
`null`. That's why Amouage Beach Hut Man (and all ~42 outputs) had null pcts
despite T009.

## Fix
- `fetch.ts`: on the success branch, also write `pages/<id>.html` (the full
  `page.content()`) alongside the `.txt`. One added `writeFileSync` + a comment;
  log line now reports html byte count too. The `.txt` is kept for the Claude
  parse step; the `.html` feeds `applyAccordPercents`.
- `parse.test.ts`: added a **real-markup** regression case mirroring the exact
  structure observed in the live fetch (incl. the trailing
  `/accords-search/?green=100&woody=64&…` link), asserting the bar widths win and
  the link's opacity-derived query-param weights (64/63/40) do NOT leak in.
- Docs: README perfumes-fetch/-parse description now notes HTML is persisted for
  the bar widths; `.harness/LIMITATIONS.md` notes old outputs only backfill on
  re-fetch.

The parser itself needed NO change — the live HTML matches the synthetic fixture
exactly (no selector drift).

## End-to-end proof (ONE frugal live fetch — Fragrantica is free, paid:false)
Throwaway script (in `/tmp`, not committed) launched the shared persistent
browser and fetched the real page
`https://www.fragrantica.com/perfume/Amouage/Beach-Hut-Man-46336.html`
(HTTP 200, 3.2 MB HTML), then ran `parseAccordPercents` on the live `page.content()`.

Raw bar widths in the HTML vs parsed pct (BEFORE this task: all null):

| accord       | width in HTML | parsed pct |
|--------------|---------------|------------|
| green        | 100%          | 100        |
| woody        | 75.0859%      | 75         |
| aromatic     | 74.3476%      | 74         |
| earthy       | 58.0808%      | 58         |
| mossy        | 48.0495%      | 48         |
| fresh        | 44.8007%      | 45         |
| fresh spicy  | 44.6586%      | 45         |

Every pct is non-null and matches the page bar width (rounded). The real markup
is `<div … style="…opacity: …; width: NN%;"><span class="truncate">NAME</span>` —
identical to the test fixture. The `/accords-search` link's different numbers were
correctly ignored (not `width:` styles / no `truncate` span). With the fetch fix,
`pages/<id>.html` now exists on the success path, so `applyAccordPercents` fills
these pcts into the parse output and they flow into the built profile.

Existing ~42 outputs stay null until re-fetched in production (see LIMITATIONS).
