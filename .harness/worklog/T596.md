# T596 — 2026-07-16

## What
`runWorkflowInner` (`src/core/workflow-executor.ts`) had a pre-`workflow_runs`-row
window where a throw was invisible to the dashboard: the DAG-build failure already
had its own try/catch that creates a row and marks it `failed`, but the newer
run-limit (T094) block just below it — the `await getJobDefinition(rootStage)!
.inputKeys!()` call for a limited manual run — had no such guard. If `inputKeys()`
rejected there (a live crawl's API call failing, a network error), the exception
escaped `runWorkflowInner` entirely before `createWorkflowRun` ever ran, and the
only trace was the fire-and-forget caller's `.catch((e) => console.error(...))` in
`src/api/server.ts` — nothing in the DB, nothing on the dashboard.

Wrapped the `inputKeys()` + root-selection block in its own try/catch that mirrors
the DAG-build pattern exactly: on a throw, `createWorkflowRun(def.name, trigger,
runLimit, null)` creates the row, `addWorkflowLog` records the captured error
message, `finishWorkflowRun(id, 'failed')` settles it, and the function returns
`{ workflowRunId: id }` instead of letting the rejection propagate.

## Tests
Added a fixture job `t596-throwing-root` (`inputKeys()` returns a rejected promise)
plus a paired `t596-cons` consumer in `src/core/workflow-executor.test.ts`, and a
new test: "T596: a limited run whose root stage inputKeys() throws still creates a
failed workflow_runs row with the error captured" — runs a LIMITED manual run
(`runWorkflow(def, 'manual', { limit: 1 })`, the only trigger that awaits
`inputKeys()` before the row exists), then asserts:
- `workflowRunId` is truthy (a row was created despite the pre-row throw)
- `getWorkflowRun(id).status === 'failed'`
- `getWorkflowLogs(id)` contains the thrown error's message
- `isWorkflowStarting`/`workflowRunInProgress` both cleared after the failure

Confirmed the test fails for the right reason before the fix (an unhandled
rejection from `runWorkflowInner`, no row created) and passes after.
