# T570 — github-sync respects run root selection

2026-07-14

## What I did
`src/workflows/projects-sync/stages/github-sync.ts`'s `runGithubSync` unconditionally
marked EVERY catalog entry in the `work_items` ledger, even on a limited manual run —
unlike `places-ingest`, which scopes its ledger writes to `ctx.rootAllowed`. Fixed by
computing `allowed = entries.filter((entry) => ctx.rootAllowed(entry.repoId))` right
after the catalog write, and looping/logging/progress-reporting over `allowed` instead
of `entries`. The catalog write (`writeCatalogFn(entries)`) and the
`entries.length === 0` early-return both still operate on the full `entries` list, so
`data/out/projects.json` stays complete regardless of the limit — only the ledger is
scoped. On an unlimited/scheduled run `ctx.rootAllowed` returns true for everything
(the set is null), so `allowed === entries` and behaviour is unchanged.

## Tests
Added a new test in `github-sync.test.ts`: drives a fake `ctx.rootAllowed` that allows
only one of three repo ids (via a `selectedRoots()` returning that single id as a
`Set`), asserts `markWorkItem` is only recorded for that repoId (checked via
`isWorkItemDone`/`getWorkItem`) while the catalog writer spy still receives all three
entries.
