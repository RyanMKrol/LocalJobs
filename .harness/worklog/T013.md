# T013 — Finish service migration: drop per-job caps + backfill service_usage

## Attempt 1 — 2026-06-21

### What the task wanted
Make the shared **service** quota the single spend governor for the two paid
places stages (remove the redundant per-job `capStatus`/`recordUsage`), make the
`QuotaExceededError` soft-fail path genuinely reachable, then backfill THIS
MONTH's `service_usage` from `job_usage` (idempotently) and run it in place.

### Changes
- **`src/jobs/places/enrich.ts`** (`places-enrich` → `google-places`)
  - Dropped `capStatus`/`recordUsage` imports + all `job_usage` cap logic.
  - Headroom/visibility now read from the `google-places` service quota
    (`getServiceDef` + `serviceCallsThisMonth/Today`); spend logs say
    `… (google-places service)`.
  - Removed the in-loop `usedMonth/usedToday >= cap` break branches — the
    `callService('google-places', …)` call throws `QuotaExceededError` when the
    service quota is hit and the existing `catch` breaks gracefully (now the live
    governor, previously shadowed by the job cap). Per-run `runLimit` kept.
- **`src/jobs/places/llm-enrich.ts`** (`enrich-with-llm` → `gemini`)
  - Same removal: dropped `capStatus`/`recordUsage`; per-call quota now solely the
    `gemini` service. Visibility line reads the service meter.
- **`src/db/store.ts`** — added `backfillServiceUsage(service, count)` (guarded
  no-op for `count <= 0`, single transaction).
- **`scripts/backfill-service-usage.ts`** (new) — tops up `service_usage` for the
  current month from `job_usage`, per mapping `google-places←places-enrich`,
  `gemini←enrich-with-llm`. Idempotent: adds only `max(0, jobMonth − svcBefore)`,
  self-verifies each service reconciles, exits non-zero on mismatch. Reads/writes
  only the local meter tables — NO API calls.
- **`src/db/store.test.ts`** — two new tests:
  - `backfillServiceUsage` idempotent top-up (5 job rows + 1 pre-existing service
    row → reconciles to 5; second run + negative count are no-ops, no double-count).
  - `QuotaExceededError` soft-fail loop: a `monthlyCap:2` service, a 4-item loop —
    only the 2 in-quota items run, the 3rd throws `QuotaExceededError`
    (`retryable`, `window:'monthly'`), the loop breaks, no usage recorded past the
    cap. This exercises the exact path the jobs now depend on as sole governor.
- Docs (same commit): CLAUDE.md spend-caps + services conventions (one-governor
  rule, service is sole governor, backfill pointer); README places example;
  `.harness/LIMITATIONS.md` (month-scoped backfill + unmetered dry-run trade-off).
