# T091 — Cancel a running workflow run (abort + kill children + API)

## What was built
Made an in-flight workflow run cancellable end-to-end.

1. **In-memory registry** (`src/core/workflow-executor.ts`): module-level
   `activeWorkflowRuns: Map<workflowRunId, AbortController>`, populated in
   `runWorkflow()` right after `createWorkflowRun()` and removed in a `finally`
   when the run settles. Both the scheduler and the manual
   `POST /api/workflows/:name/run` path flow through `runWorkflow`, so both register.
   Exported `cancelWorkflowRun(id)` (looks up + `abort()`, returns false if not
   active) and `isWorkflowRunActive(id)`.

2. **Abort threading**: `controller.signal` → `executeDag` (new `signal` hook in
   `DagExecHooks`) → `runJobForWorkflow(jd, workflowRunId, signal)` →
   `runAttempts(..., signal)` → `executeAttempt(..., signal)`.
   - `executeDag` (`src/core/dag.ts`): once `signal.aborted`, stops launching new
     stages (skips the ready-scan) and only drains in-flight members; a new
     `else if (aborted) break` ends the loop when nothing is in flight. Not-yet-
     started members are left absent from the status map (never spawn).
   - `executeAttempt` (`src/core/executor.ts`): an `abort` listener hard-kills the
     child via the existing timeout path (`SIGTERM` then `SIGKILL` after 3s) and a
     `cancelled` flag makes `close` resolve `AttemptOutcome.status = 'cancelled'`.
     A pre-aborted signal resolves `cancelled` without spawning at all.
   - `runAttempts`: `cancelled` is terminal — recorded via `finishRun(..,'cancelled')`
     and returned immediately (NOT retried); also returns early if aborted between
     attempts.

3. **Store transitions**: no new store fns needed — `finishRun`/`finishWorkflowRun`
   already accept `'cancelled'` (the status existed). `runWorkflow` breaks the cycle
   loop on abort and finalises with `'cancelled'` (authoritative over the member
   tally). Executor stays the sole DB writer.

4. **API** (`src/api/server.ts`): `POST /api/workflow-runs/:id/cancel` — guarded by
   the existing `authoriseMutation` (loopback/token). 404 if unknown, 409 if not
   `running` or not active in this process, else `abort()` + `{ ok: true }`.

## Tests added (all green)
- `executor.test.ts`:
  - aborting an in-flight (hanging, no-timeout) member kills the child promptly
    (<5s, proving real kill not early-resolve) and records `cancelled`.
  - a pre-aborted signal yields `cancelled` with NO run rows + no retry.
- `workflow-executor.test.ts` (FAKE child now hangs on `timeout*` names):
  - start a workflow whose first stage hangs, poll until active, `cancelWorkflowRun`,
    then assert: workflow run `cancelled`, killed member `cancelled`, downstream stage
    never spawned (no row), registry cleaned up, second cancel returns false.

## Verification observed
- `npx tsc --noEmit` — clean.
- `npm test` — all suites pass; executor 8/8, workflow-executor 12/12 (incl. the 3
  new cancellation tests). The hang-then-cancel tests would block ~60s if the child
  were not actually killed; they return in well under 5s, empirically confirming the
  SIGTERM/SIGKILL reap (no orphaned PID). No paid/live calls — fake child script only.
- Did NOT run the live daemon PID check (no daemon in headless harness); the
  hanging-child unit tests give equivalent empirical proof of the kill.
