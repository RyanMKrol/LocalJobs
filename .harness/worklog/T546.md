# T546 — keyboard accessibility for clickable spans + sortable headers

2026-07-15

## What I did
- `dashboard/app/components/SortTh.tsx`: the `<th onClick>` is now a plain `<th aria-sort=…>`
  wrapping a `<button type="button">` for the click/sort action — natively focusable +
  Enter/Space-activatable, no custom key handler needed. `aria-sort` reflects
  `ascending`/`descending`/`none` per the active column + direction. The button resets its
  chrome via an inline `style` (background/border/padding/margin, `font: inherit`, `color:
  inherit`, `cursor: pointer`) rather than a new `globals.css` class, since `globals.css` is
  outside this task's scope — so the header text renders visually identical to the old plain
  `<th onClick>`. `.sort-th`'s own `cursor: pointer` is untouched (harmless now that the click
  target is the inner button, not the cell).
- `dashboard/app/workflows/[name]/page.tsx`: the two `<span className="schedule-edit-link"
  onClick>` (Schedule, Max concurrency) became `<button type="button" className="btn-link
  schedule-edit-link" onClick>` — `btn-link` (existing class) strips button chrome,
  `schedule-edit-link` keeps the original font-size/colour/underline exactly as before.
  The three `<span className="toggle" onClick>` (Enabled, Notifications, Certified) gained
  `role="switch"`, `tabIndex={0}`, `aria-checked={…}`, and an `onKeyDown` that fires the same
  toggle handler on Enter/Space (mirrors the `CronBadge` pattern at `ui.tsx:246-258`). The
  inner `<input type="checkbox" readOnly>` got `tabIndex={-1}` so it's not a second stop in
  the tab order (the wrapping span is now the single focusable/announced control).
- `dashboard/app/jobs/[name]/page.tsx`: the Timeout Edit span got the same `btn-link
  schedule-edit-link` button treatment.
- No `_dashboard-harness.mjs` change needed — these are pure semantics/focusability changes
  with no new visual state; the existing `workflow`, `job`, `tv-recs-default-sort` (SortTh
  usage) PAGES entries already exercise the changed surfaces.
