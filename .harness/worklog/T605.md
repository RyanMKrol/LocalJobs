# T605 — one work_items row per movie/show in movie-snapshot / tv-snapshot

2026-07-16

## What I did

Replaced the T571-era single combined bulk `work_items` row (keyed by the run's ISO date) in both
`src/workflows/movies/stages/snapshot.ts` and `src/workflows/tv-recs/stages/tv-snapshot.ts` with one
row PER movie/show, mirroring `missing-tv-seasons/stages/snapshot.ts`'s `recordSnapshotLedger`
pattern exactly:

- Added `snapshotItemKey(item)` (prefers `tmdbId`, falls back to `ratingKey`) and
  `recordSnapshotLedger(items)` (one `markWorkItem(..., 'success', { detail: { name, tmdbId, year } })`
  call per item) to both files, exported for unit-testability.
- `runSnapshot` (movies) / `runTvSnapshot` (tv-recs) now call `recordSnapshotLedger(movies|shows)`
  in place of the old single `markWorkItem(JOB, dayKey(now), 'success', { detail: { name, count,
  path, format } })` call. `dayKey`/`markWorkItem`-by-date imports removed where no longer used.
- Idempotency unchanged: both stages still re-scan/re-compute fresh every run (no skip-if-done).
  Because the ledger key is now the item's own id rather than the run's date, a same-day OR
  cross-day re-run upserts the same per-item rows in place instead of accumulating stale
  date-keyed rows — strictly better than before, not a behaviour regression.
- Updated both stages' doc comments (no longer claim "ONE combined visibility row per run").
