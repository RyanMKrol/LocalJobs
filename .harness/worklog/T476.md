# T476: Make cache TTL configurable per service

## Summary
Completed successfully. Added an optional `cacheTtlMs?: number` field to `ServiceDefinition` that allows each service to declare its own response cache TTL instead of using a global 5-minute constant. Set Plex to use a 3-hour cache TTL since full-library crawls are expensive and contents change slowly.

## Changes made

1. **src/core/types.ts**
   - Added `cacheTtlMs?: number` field to `ServiceDefinition` interface
   - Documented as code-level field (like `minIntervalMs`/`maxJitterMs`), not dashboard-editable
   - Explained use case: expensive/slow services can use longer windows than cheap lookups

2. **src/core/services.ts**
   - Updated `callService` cache check to use `def.cacheTtlMs ?? SERVICE_CACHE_TTL_MS`
   - Falls back to existing 5-minute global constant when service doesn't set `cacheTtlMs`
   - No behavior change for services without the field

3. **src/services/plex.service.ts**
   - Set `cacheTtlMs: 3 * 60 * 60 * 1000` (10,800,000 ms = 3 hours)
   - Added comment explaining why: Plex library crawls are expensive, multi-hour-stale cache is acceptable

4. **src/core/services.test.ts**
   - Added test `cacheTtlMs: a service with custom cacheTtlMs uses that TTL instead of the global default`
   - Verifies cached entry older than global default but younger than custom TTL is still a cache hit
   - Entry is 7 minutes old; custom TTL is 10 minutes; global default is 5 minutes
   - Confirms entry serves from cache (fn not re-invoked) with custom TTL
