# T541 — resolve node at run time in launchd wrappers

2026-07-13

## What changed

- `scripts/start-daemon.sh` is now a generic wrapper (not daemon-only): it resolves a
  usable node bin dir fresh on **every launch** via `resolve_node_bin_dir()` — checks
  an explicit `NODE_BIN_DIR` override, then sources `~/.nvm/nvm.sh` and does `nvm use`
  (honouring `.nvmrc`, falling back to nvm's default alias), then common non-nvm
  locations (`/opt/homebrew/bin`, `/usr/local/bin`), then whatever `node` is already on
  `PATH`. With no args it runs `npm run daemon` (the orchestrator); with args it execs
  `npm "$@"` — this lets a second launchd agent reuse the exact same run-time
  resolution instead of baking its own frozen path.
- `scripts/install-launchd.sh` no longer computes or bakes `NODE_BIN_DIR` into the
  plist at install time — the plist's `EnvironmentVariables/PATH` is now just the
  minimal system path; `start-daemon.sh` prepends the resolved node dir at run time.
- `scripts/install-dashboard-launchd.sh` now routes the dashboard through
  `scripts/start-daemon.sh --prefix <dashboard-dir> run start` (via `/bin/bash`,
  mirroring the orchestrator plist) instead of invoking `$NODE_BIN_DIR/npm` directly —
  so it gets the same run-time node resolution and never bakes a frozen nvm path
  either. Also fixed the stale `http://localhost:3001` header comment to `4788`
  (matches the actual dashboard port used elsewhere in the file/repo).
- Left `launchctl load`/`unload` as-is (not converted to `bootstrap`/`bootout`) —
  the spec marks this as an open owner question (pending-questions #14), out of
  scope for this task without explicit confirmation.
