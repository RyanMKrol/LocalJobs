# T606: Add per-item work_items ledger recording to missing-movies stages

## Summary
Implemented per-item `work_items` ledger recording for the `missing-movies` workflow's two stages (`plex-movie-snapshot` and `franchise-gaps`), mirroring the working pattern from `missing-tv-seasons`. This provides visibility into the workflow's operations on the dashboard's Inputs & Outputs panel.

## What was done

### 1. plex-movie-snapshot stage (src/workflows/missing-movies/stages/snapshot.ts)

- Added import of `markWorkItem` from the store
- Added import of `PlexMovie` type for type safety
- Extracted `snapshotItemKey(movie)` helper function: keys by `tmdbId` if present, else falls back to `ratingKey`
- Extracted `recordSnapshotLedger(movies)` helper function: records one `work_items` row per movie with:
  - `job_name: 'plex-movie-snapshot'`
  - `item_key`: result of `snapshotItemKey()`
  - `status: 'success'` (all movies are successfully snapshotted, regardless of TMDB GUID presence)
  - `detail`: { name, tmdbId, year }
- Called `recordSnapshotLedger(movies)` after writing snapshot.json but before progress/finish logging
- Stage behavior unchanged: still re-scans fresh every run (no skip-if-done)

### 2. franchise-gaps stage (src/workflows/missing-movies/stages/franchise-gaps.ts)

- Added import of `markWorkItem` from the store
- Extracted `recordCollectionLedger(collectionId, collectionName, gaps)` helper function: records one `work_items` row per DISTINCT collection checked in pass 2 with:
  - `job_name: 'franchise-gaps'`
  - `item_key`: collection id as string
  - `status: 'success'` (a collection is "checked successfully" whether or not gaps were found)
  - `detail`: { name, gapsCount, gaps: [titles...] }
- Updated pass 2 loop to call `recordCollectionLedger()` for each successfully-fetched collection (with or without gaps)
- Updated pass 2 error handling: on collection fetch failure (non-quota), records a `failed` ledger row with:
  - `status: 'failed'`
  - `detail`: { name, error: <message> }
- Added `collectionFailed` counter for the summary
- At the end of the stage, if `collectionFailed > 0`, throws an error per the T416 item-loop convention:
  - Run is marked `failed`
  - Blocks downstream DAG dependents
  - Can be retried on the next run
  - Allows per-collection failures to be visible while still failing the run overall

**Design note on collection granularity:** Keying by collection id (not by individual gap) was chosen to match `missing-tv-seasons`' per-show (not per-missing-season) granularity. A collection with 3 gaps is one collection-level outcome, not 3 rows. This is appropriate because:
- Collections are the natural work unit: a single TMDB `/collection/{id}` call determines the outcome for that whole collection
- The dashboard's Inputs & Outputs panel then shows the workflow's structure: franchise-gaps → (one row per collection checked) → movie-gaps-notify (whose ledger is per-gap, a different concern)
- Maintainability: if in future we add per-gap actions (e.g. "ignore this specific gap from collection X"), the parent collection row remains the structural anchor
