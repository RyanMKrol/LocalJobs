# T308 — Hardcode dashboard to sunny-8bit + baloo, replace theme picker with sun/moon toggle

## Changes

- `dashboard/app/globals.css`:
  - Sunny-8bit's dark palette (was `html[data-theme="sunny-8bit"]`) is now the base
    `:root` palette (added later in the file so it overrides the original `:root`'s
    colours; a second `:root` block is valid CSS and wins by source order for the
    overlapping custom properties).
  - Its light palette (was `html[data-theme="sunny-8bit"][data-mode="light"]`) is now a
    plain `[data-mode="light"]` override.
  - Added `--font-display`/`--font-body: var(--font-baloo)` + `--heading-weight: 500`
    to the new base `:root` (font is hardcoded now — no `data-font` attribute is
    written anymore).
  - Found + fixed a correctness bug this change would otherwise have introduced: the
    old `default`-family light rule was `html[data-mode="light"]:not([data-theme]), ...`
    — with `data-theme` never written, `:not([data-theme])` now ALWAYS matches, and its
    specificity (0,2,1) beats the new bare `[data-mode="light"]` (0,1,0), so it would have
    silently overridden the sunny-8bit light palette back to the plain default look.
    Removed that branch, leaving only `html[data-theme="default"][data-mode="light"]`,
    which nothing sets anymore (dead, as intended — full removal deferred to T309).
  - The "joyful accents" block (emoji badges, rounded panels, hover-lift, table-row
    accent, success-badge glow) was gated on `html[data-theme]:not([data-theme="default"])`
    — also always-true-when-set/never-true-when-unset logic that would have gone dark
    once `data-theme` stopped being written. Since the dashboard is now permanently on
    the (non-default) sunny-8bit family, replaced all 8 occurrences of that selector
    with plain `html` so the accents are simply always on.
  - Simplified `.theme-controls`/`.theme-trigger`/etc CSS down to a single 32×32 icon
    button; removed the now-dead `.theme-modal`/`.theme-grid`/`.theme-opt`/`.theme-section*`/
    `.theme-motion` rules (the modal markup they styled no longer exists).
  - Left `pixel-picnic`/`default` family blocks and `data-font`/`data-motion` CSS rules
    in place, per the task's explicit scope note (T309 follow-up removes the dead CSS).
- `dashboard/app/layout.tsx`: dropped the `Space_Mono` font load (kept only `Baloo_2`);
  `PREPAINT` now only derives + sets `data-mode` from `localjobs.mode` (same
  stored-choice-or-`prefers-color-scheme` fallback as before), no longer touching
  `data-theme`/`data-font`/`data-motion`.
- `dashboard/app/ui.tsx`:
  - `useMode()` is now strictly binary (`ModeId = 'dark' | 'light'`, no `'system'`):
    untouched state follows `prefers-color-scheme` (and reacts live to OS changes,
    via a `matchMedia` listener that's only attached when nothing is stored yet);
    `toggle()` flips the effective mode and persists an explicit choice — matching the
    spec's "no way back to System once toggled".
  - `ThemeControls` now renders one icon button (☀️ in light mode / 🌙 in dark) that
    calls `toggle()` directly — no modal/popover.
  - `useTheme`/`useFont`/`useMotion`, `THEMES`/`FONTS`/`MODES` arrays, and
    `useHtmlPref` are left in place but unused (per spec, for T309 to remove) — the new
    toggle component does not import them.
- `dashboard/scripts/_dashboard-harness.mjs`: replaced the `overview-theme-picker` FLOWS
  entry (clicked `.theme-trigger` → waited for the now-gone `.theme-modal`) with
  `overview-theme-toggle`: clicks `.theme-trigger` and waits for `data-mode` to flip
  away from its pre-click value before the screenshot.
