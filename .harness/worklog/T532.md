# T532: timing-safe token comparison

## Implementation

Replaced the non-constant-time string comparison (`===`) in `authoriseMutation()` with a timing-safe constant-time comparison using `crypto.timingSafeEqual`. 

### Changes:

1. **Import:** Added `import { timingSafeEqual } from 'node:crypto'` at the top of `src/api/server.ts`

2. **Function:** Replaced the simple `return supplied === args.token || bearer === args.token` with a timing-safe comparison helper that:
   - Returns false early if the presented token is falsy (undefined/empty)
   - Returns false early if the byte lengths differ (prevents `timingSafeEqual` from throwing on unequal-length buffers)
   - Uses `crypto.timingSafeEqual` to do the constant-time comparison
   - Catches any exceptions and returns false (defense-in-depth)
   - Is applied to both the X-LocalJobs-Token and Authorization: Bearer values

### Semantics Preserved:

- Loopback callers always pass (unchanged)
- Empty configured token still rejects remote (unchanged)
- Both header styles (X-LocalJobs-Token and Bearer) still accepted (unchanged)
- Valid token still allows mutation (unchanged)
- Invalid/wrong token still rejects with false (unchanged)

## Why This Matters

Token comparison is a security-sensitive operation. A non-constant-time comparison (regular string `===`) is theoretically vulnerable to timing-side-channel attacks where an attacker could deduce the token byte-by-byte by measuring response times. While the practical risk is low on loopback, this is a standard hardening practice in security codebases. Using `crypto.timingSafeEqual` mitigates this attack vector by ensuring the comparison always takes the same amount of time regardless of which bytes differ.
