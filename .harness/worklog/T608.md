# T608 — buffer the busy state on Workflows-list + Overview Run buttons

2026-07-16

## What I did

`dashboard/app/workflows/page.tsx`'s `run()` and `dashboard/app/page.tsx`'s
`runWorkflow()` both cleared `busyWorkflows` for a workflow name IMMEDIATELY in
their `finally` block once the fire-and-forget `POST /api/workflows/:name/run`
resolved — racing the 3s `usePoll` refresh, since the DB run row isn't
guaranteed visible to a `GET /api/workflows` poll that fast. This caused the
"Started…" busy label to flash back to a plain clickable "Run" before the next
poll picked up the real `running` status (reported live on the claude-warmer
row in the Regular maintenance category table).

Applied the same fix `dashboard/app/workflows/[name]/page.tsx`'s `run()`
already uses (a `busyTimerRef` + 1200ms `setTimeout` before clearing `busy`),
adapted for the per-workflow `Set<string>` busy state both pages use instead
of a single boolean:

- Added a `busyTimersRef = useRef<Map<string, ReturnType<typeof setTimeout>>>(new Map())`
  to each page.
- In each page's run handler's `finally`, instead of immediately deleting the
  workflow name from `busyWorkflows`, clear any existing timer for that name
  and schedule a new `setTimeout(() => { delete from busyWorkflows }, 1200)`,
  storing the timer in the map keyed by name.
- Added a mount-cleanup `useEffect(() => () => { busyTimersRef.current.forEach(clearTimeout) }, [])`
  on both pages to avoid leaking timers across unmounts, mirroring the detail
  page's single-ref cleanup pattern but for the keyed collection.
- Did not touch `RunButton.tsx`, the `isRunning`/`busy` prop contract, or any
  other button/state logic.
