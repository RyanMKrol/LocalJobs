# T300 — split stocks-watch into check + notify stages

## What was wrong

`stocks-watch` (T290) only called `markWorkItem` in two narrow branches (a fresh breach, or
resetting a stale notified-flag). In the common case — nothing breaching, or an already-notified
position still above threshold — it called `markWorkItem` zero times, tripping the framework's
`hasJobAdvancedAnyItem` noop-detection even though it did real work (reading the portfolio and
computing `gainPct` for every position).

## What changed

Split `stocks-sync` into 3 stages:

1. **`stocks-watch`** (check) — for EVERY position, EVERY run, computes `gainPct` and calls
   `markWorkItem(WATCH_JOB, ticker, 'success', { detail: { gainPct, breaching } })`
   unconditionally, guaranteeing at least one ledger write whenever there's at least one position.
   "Already notified for this breach episode" is now tracked on a separate ledger key
   (`<ticker>::notified`) so the unconditional per-run success write doesn't erase that state.
   Writes `data/out/fresh-breaches.json` (this run's fresh breaches, `BreachLine[]`, empty if
   none) for the next stage.
2. **`stocks-notify`** (new) — reads `fresh-breaches.json`; if non-empty sends exactly ONE push
   (reusing `buildDigest`/`formatBreachLine`, now moved to `stocks-notify.ts` since that's where
   the push happens); if empty, does nothing — a legitimate noop.
3. Workflow DAG: `stocks-snapshot` → `stocks-watch` → `stocks-notify`.

## Verification

- `npx tsc --noEmit` — clean.
- `npm test` — all 80 tests pass, including:
  - `stocks-watch.test.ts` (rewritten): a no-breach run still records ledger activity for every
    checked position; asserts `hasJobAdvancedAnyItem(runId, 'stocks-watch') === true` in that
    scenario — the exact check the framework's noop-detection makes. Also covers fresh-breaches.json
    contents across breach/no-breach/reset/re-cross/multi-breach scenarios.
  - `stocks-notify.test.ts` (new): a run with no fresh breaches sends nothing AND
    `hasJobAdvancedAnyItem(runId, 'stocks-notify') === false` — confirming this stage's noop is the
    correct, desired outcome (only `stocks-watch`'s noop misclassification was the bug). Also
    reproduces the full idempotency chain end-to-end across both stages (fresh cross → notify once
    → still above → nothing → drop below → reset → re-cross → notify again), and confirms multiple
    simultaneous fresh breaches send exactly one combined push.
- No live/paid calls made — used synthetic portfolio.json fixtures + the scratch test DB only.
