## What I did
Wired RunButton's existing `busy` prop on both surfaces that trigger a workflow run, so the
button flips to the disabled "Started…" label the instant a click fires, instead of waiting for
the next 2–3s poll to see `last_run.status === 'running'`.

- `dashboard/app/workflows/page.tsx`: added a `busyWorkflows: Set<string>` state. `run(name)` now
  adds the name to the set synchronously, awaits `api.runWorkflow(name)` in `try`, and removes it
  in `finally` (reverts on error, hands off to poll-driven `isRunning` once observed). Passed
  `busy={busyWorkflows.has(p.name)}` to the row's `RunButton`.
- `dashboard/app/page.tsx`: same pattern for the Overview workflow cards (`busyWorkflows` state,
  `runWorkflow(name)` sets/clears busy around the API call). Also switched the card's
  `RunButton isRunning={false}` (hardcoded) to `isRunning={p.last_run?.status === 'running'}` so
  the card actually reflects a genuinely-in-progress run, matching the Workflows list's behaviour
  and the "hands off to poll-driven Running… state" requirement in the spec's Done-when.
- `RunButton.tsx` itself: unchanged, per the spec.
