# T322 — POST /api/workflows/reset-output-all bulk output-reset endpoint

## What was built (within scope: `src/api/server.ts` + test files)

- Added `POST /api/workflows/reset-output-all` in `src/api/server.ts`, registered
  BEFORE the existing `:name`-based `POST /api/workflows/:name/reset-output` route
  (matches `parts[1]==='workflows' && parts[2]==='reset-output-all' && parts.length===3`,
  so it can't be shadowed/misrouted).
- Iterates every workflow from `listWorkflows()`. For each:
  - If `workflowRunInProgress(name)` is true → SKIP (status `'skipped'`, reason
    `'active run in progress'`) — does NOT block the reset of any other workflow
    (best-effort across the fleet, not all-or-nothing).
  - Otherwise runs the exact same sequence as the single-workflow endpoint:
    `findWorkflowDataOut` → `resetWorkflowOutput` → `deleteDataOutContents`.
- Responds 200 with `{ ok, totalWorkflows, resetCount, skippedCount, results[] }`,
  each result entry carrying either the reset counts (`itemsDeleted`/`runsDeleted`/
  `wfRunsDeleted`/`filesRemoved`/`outDir`) or the skip `reason`.
- Logs one summary line via `console.log` naming reset vs skipped workflows and why.
- Sits behind the same global `authoriseMutation` loopback/token guard as every
  other POST route (applied once, globally, before routing — confirmed at
  `src/api/server.ts:1053`; no separate guard needed for this route).
- Added a unit test in `src/api/server.test.ts` (`T322` block) seeding two
  workflows in the scratch DB — one idle with a work item to reset, one with a
  `workflow_runs` row left in `'running'` status to simulate an active run — and
  asserting: the idle workflow is reset (counts > 0), the active-run workflow is
  skipped with the exact reason string, and its work item is left untouched
  (`isWorkItemDone` still true afterward).
