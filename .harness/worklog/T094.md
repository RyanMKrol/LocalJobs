# T094 — Build: workflow run-limits (input lineage + manual N-input cap)

**Result:** done (built + committed; not pushed).

Implemented the design in `.harness/designs/T093-workflow-run-limits.md`: a manual
workflow run can be capped to **N originating inputs**, and **all** fan-out of each
selected root runs to completion. Scheduled runs stay unlimited. Lineage is
framework-tracked via a `root_key` on `work_items`.

## What was built (layer by layer)

1. **DB** — `src/db/schema.sql`: `work_items.root_key`/`parent_key` + index
   `idx_work_items_root`; `workflow_runs.run_limit`/`selected_roots`. Additive,
   idempotent migration extracted into exported `migrateRunLimitLineage(db)` in
   `src/db/index.ts` (backfills existing rows to `root_key = item_key`).
2. **store.ts** — `markWorkItem` gains lineage opts + `resolveRootKey` rule
   (explicit `rootKey` → inherit `parentKey` row's root → default `item_key`);
   `WorkItemRow` gains `root_key`/`parent_key`; `selectPendingRoots`
   (first N pending roots, in input order); `getWorkflowRunRoots`;
   `createWorkflowRun(name, trigger, runLimit?, selectedRoots?)`.
3. **types + child + executor** — `JobContext.selectedRoots()`/`rootAllowed()`;
   `runJob.ts` reads `LOCALJOBS_WORKFLOW_RUN_ID` → builds the allowlist set;
   `executor.ts` threads the workflow run id into the child env;
   `workflow-executor.ts` `runWorkflow(def, trigger, { limit })` finds the root
   stage (first member with `inputKeys()`), selects + freezes roots on the run row,
   logs the limit.
4. **API** — `POST /api/workflows/:name/run { limit }` validates a positive integer
   and that the workflow is limitable (`400` otherwise); `workflowView` surfaces
   `limitable`.
5. **Example jobs** — perfumes: `inputKeys()` on `find-url.job.ts` (root stage) +
   `ctx.rootAllowed(p.id)` in all 4 stage filters (same-key, no lineage args needed).
   places: `inputKeys()` (resolver, keyed by cid) + `rootAllowed` filters; enrich +
   llm-enrich (keyed by place_id) pass `rootKey: cid` on `markWorkItem` so the cid
   root propagates across the key change.
6. **UI** — `dashboard/app/lib/api.ts` (`runWorkflow(name, limit?)`,
   `Workflow.limitable`, `WorkflowRun.run_limit`); limit number box beside ▶ Run now
   (only when `limitable`); `limited · N inputs` badge on the run-detail page.

## Deviations from the design (all immaterial — noted in the design doc)

- Migration extracted into a testable exported function (design had it inline).
- `distinctRoots`/`rootCount` not added (unused by the selection path).
- Places enrich/llm pass explicit `rootKey: cid` (rule 1) instead of relying on
  `parentKey` inheritance (rule 2); rule 2 is still implemented + unit-tested.

## Notes / trade-offs (recorded in `.harness/LIMITATIONS.md`)

- "Pending" re-selection = entry-not-done OR any descendant outstanding (a root with
  only stuck descendants is treated as done by a limited run).
- A limited `places` run reads roots from the existing `places.json`; on a brand-new
  DB it selects nothing until ingest has produced it — run unlimited once first.
- Stages before the root stage (places-ingest) run unfiltered under a limit (intended).
