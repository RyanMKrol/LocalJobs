# T510 тАФ Cache DynamoDB reads for 22 hours

## Summary
Implemented caching for DynamoDB `Scan` operations used by the perfumes workflow.

## What was done
1. Added `cacheTtlMs: 79_200_000` (22 hours) to the `dynamodb` service definition in `src/services/dynamodb.service.ts`
   - The service already has `category: 'api'`, so caching is automatically honored
   - Matches the pattern established by other cached services (google-places, lastfm, hevy)

2. Added a stable `cacheKey` to the `dynamoScan` call in `src/workflows/perfumes/lib.ts` (line 18)
   - Key format: `dynamodb:scan:${perfumesConfig.perfumeRatingsTable}`
   - Ensures identical table scans within the 22-hour window use the cached response
   - No function argument changes needed; callService accepts the cacheKey option

3. Verified:
   - `npx tsc --noEmit` тАФ passes (no type errors)
   - `npm test` тАФ 259/259 pass (see fix below for a caching-induced regression found + fixed)
   - No caching added to write paths (only read operations; all write functions remain disabled)

## 2026-07-12 follow-up тАФ fixed a real test regression caused by the cacheKey

Adding the stable `dynamodb:scan:<table>` cacheKey to `loadPerfumes()` turned out to break 3 existing
hermetic perfumes tests (`lib.test.ts`, `find-url.test.ts`, `parse-detail.test.ts`) тАФ all of them mock
the DynamoDB client per-`it` via `_resetClient(makeMockClient(...))` and call `loadPerfumes()`
expecting a FRESH mock response each time. Once `loadPerfumes` opts into `service_cache` keyed only by
table name (same table across every test), the very first test's response got cached and every
subsequent `it`/test file within the same process reused it instead of its own mock тАФ exactly the
"served from cache" behavior the task requires, just surfacing where the tests didn't expect it.
Fixed by calling the existing `clearServiceCache('dynamodb')` (from `src/db/store.ts`, same helper
`workouts-sync/stages/hevy-sync.test.ts` already uses for the same reason) at the start of each
affected `it`, so each test gets a clean cache slate before its own mock runs. Added a
`beforeEach`-based clear in `lib.test.ts` (its 5 tests all share one table-name cache key) and an
inline `clearServiceCache('dynamodb')` call in the single-relevant `it` of `find-url.test.ts` (x2) and
`parse-detail.test.ts` (x1). `npm test` now reports 259/259 passing (was 252/259 before this fix).

## Design notes
- The caching mechanism already existed (T451/T476) via the `service_cache` table
- Only `callService` calls with `cacheKey` and an `api`-category service use it
- Cache TTL of 79,200,000 ms = 22 hours = exact duration specified in the task
- Read-only constraint: dynamoPut/dynamoDelete/dynamoBatchWrite remain disabled; dynamoScan is read-only

- Checked that the 22-hour TTL matches the task requirement (79,200,000 ms)
- Verified the cache key format follows the pattern used by other services (`<service>:<operation>:<key>`)
- Type-checked and ran the test suite to confirm no regressions
- Reviewed the caching implementation in `src/core/services.ts` to confirm the mechanism is applied correctly
