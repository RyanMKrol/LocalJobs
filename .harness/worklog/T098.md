# T098 — Regression guard: migrations must run against an existing (non-fresh) DB

## Goal
Prevent the class of bug behind the T094 regression (fixed in 2748c58): a
`schema.sql` bootstrap statement referencing a column that only a LATER additive
migration adds. Invisible to CI because the unit suite always starts from a
fresh/empty scratch DB (so `schema.sql`'s `CREATE TABLE` already had the column),
while the live/existing DB crash-looped in `openDb()`.

## What I did
1. **Minimal enabling refactor** — `src/db/index.ts`: `openDb()` now takes an
   optional `dbPath` (`openDb(dbPath: string = config.dbPath)`). Daemon still uses
   the default (`export const db = openDb()` unchanged); the param exists solely so
   the regression test can drive the full bootstrap+migration path against a chosen
   file. No env juggling.
2. **Regression test** — `src/db/migrate-existing-db.test.ts`:
   - Seeds a realistic LEGACY snapshot: every core table in its PRE-migration shape
     — `work_items` without `root_key`/`parent_key` + no `idx_work_items_root`
     (pre-T094), `runs` without `workflow_run_id` (no `idx_runs_workflow`),
     `workflow_runs` without `run_limit`/`selected_roots` (pre-T094), `services`
     without `limits_overridden` (pre-T018), `jobs` WITH the dropped
     `schedule`/`enabled` columns (pre-T070) — plus representative rows incl. a
     legacy `'dismissed'` work_item.
   - Runs the REAL `openDb(dbPath)` against THAT file.
   - Asserts: openDb does NOT throw; `work_items` gains root_key/parent_key +
     `idx_work_items_root`; legacy rows backfilled `root_key = item_key`; `runs`
     gains `workflow_run_id` + `idx_runs_workflow`; `workflow_runs` gains
     run_limit/selected_roots (NULL = unlimited); `services` gains
     `limits_overridden`; `jobs` schedule/enabled dropped; `'dismissed'` →
     `'ignored'`; no data loss; idempotent on a second `openDb()`.
3. **Convention** — CLAUDE.md Conventions: "Schema bootstrap must NEVER reference a
   migration-added column (T098)", cross-referencing the `migrateRunLimitLineage`
   ALTER-then-CREATE-INDEX pattern.

## Verification (observed)
- `npx tsc --noEmit` → clean.
- `npm test` → all green; new file reports `11 existing-db migration test(s) passed.`
- **Proved it catches the ordering bug:** temporarily re-inserted the pre-fix
  buggy line `CREATE INDEX IF NOT EXISTS idx_work_items_root ON work_items(job_name, root_key);`
  into `schema.sql` and ran the test in isolation — it FAILED with
  `SqliteError: no such column: root_key` thrown from `openDb` at the
  `db.exec(schema)` step (index.ts:31), exactly the live crash. Restored the fixed
  `schema.sql` (grep count of the buggy line back to 0) and re-ran the full suite —
  green. No paid API calls.
