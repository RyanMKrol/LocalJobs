# T347 — always render the Output section

## Change

- `dashboard/app/components/WorkflowOutputSection.tsx`: removed the premature
  `if (!error && items.length === 0 && data !== undefined) return null;` guard.
  The component now always renders `<div className="output-section"><h2>Output</h2>...`
  plus the description paragraph, and then one of: the item table (items present),
  the previously-dead "No output yet. Run the workflow — produced items will appear
  here." message (loaded, zero items), or the error message. During loading
  (`data === undefined`) neither branch renders, so only the heading + description
  show while waiting — no incorrect empty-state flash.

## Living artifact update (`dashboard/scripts/_dashboard-harness.mjs`)

- Added a dedicated fixture route for `.../output-items`, placed before the
  generic `/api/workflows/` catch-all (which previously swallowed this route and
  always returned `{ workflow: workflow() }`, so `items` was always `[]` and the
  Output section was invisible in every prior screenshot):
  - `/api/workflows/places/output-items` → 2 synthetic `OutputItem` rows (one
    with `hasMarkdown: true` + a name, one with `name: null` + no markdown) —
    exercises the populated table + View button for the first time.
  - any other `.../output-items` path → `{ items: [], terminalJobs: [] }` — the
    guaranteed explicit empty state this task adds.
- Added a new `PAGES` entry `{ name: 'workflow-empty-output', path:
  '/workflows/workouts-sync', waitFor: ['.rf-dag-node'] }` to exercise the empty
  state on a real, non-`WORKFLOWS_WITH_SPECIFIC_MANAGERS` workflow.
