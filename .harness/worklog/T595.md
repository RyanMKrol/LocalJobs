# T595 — 2026-07-16

## What I did

Surfaced the existing in-process `startingWorkflows` claim (T105, `src/core/workflow-executor.ts`)
as a read-only, dashboard-visible "Starting…" signal for the window between a limited manual run
being accepted and its `workflow_runs` DB row being created (the window a slow root-stage
`inputKeys()` crawl — e.g. `plex-language-fix`'s `discoverInputKeys()` — can stretch to minutes).

- `src/core/workflow-executor.ts`: added `isWorkflowStarting(name)`, a pure read of the existing
  `startingWorkflows` Set — no new state, no schema/DB change. `workflowRunInProgress`/the T105
  guard semantics are completely untouched.
- `src/api/server.ts`: both `GET /api/workflows` and `GET /api/workflows/:name` now spread a
  `starting: boolean` field (sourced from `isWorkflowStarting`) onto each workflow payload.
- `dashboard/app/lib/api.ts`: added `starting?: boolean` to the `Workflow` type.
- `dashboard/app/workflows/page.tsx`: the Workflows list's Last Run cell shows a `Pill kind="starting"`
  "Starting…" chip (falling back to the normal last-run badge once the flag clears) and the Run
  button's `isRunning` now also considers `p.starting`.
- `dashboard/app/workflows/[name]/page.tsx`: the detail page's header shows the same "Starting…"
  pill next to the title, and the Run button shows a "Starting…" label / disabled state while
  `p.starting` is true.
- `dashboard/app/globals.css`: added `.pill.starting` (accent-blue, matching the existing
  `.badge.running` colour).
- `dashboard/scripts/_dashboard-harness.mjs`: added two FLOWS entries (`workflows-list-starting`,
  `workflow-detail-starting`) that override the relevant fixture endpoint with `starting: true` so
  visual-check captures the new state — required by the living-artifact rule since this adds a new
  interactive/transient UI state.

No database schema change of any kind, no new `workflow_runs` status value — purely reading an
existing in-process claim through a new field on two already-existing read endpoints.

## Tests added

- `src/core/workflow-executor.test.ts`: new fixture `t595-slow-root`/`t595-cons` (a root stage
  whose `inputKeys()` hangs until manually released) + a new test that starts a LIMITED manual run,
  polls until `isWorkflowStarting` flips true, asserts `workflowRunInProgress` is still true and
  that `lastWorkflowRunForWorkflow` returns `undefined` (proving the pre-DB-row gap this task
  targets), then releases the hang and asserts both flags settle correctly.
- `src/api/server.test.ts`: new fixture job/workflow with a hanging `inputKeys()`, hit through the
  REAL `POST /api/workflows/:name/run` (fire-and-forget) endpoint. Asserts both `GET /api/workflows`
  and `GET /api/workflows/:name` return `starting: true` during the gap and `starting: false` after
  the run settles.

Both tests were run BEFORE implementing `isWorkflowStarting`/the API fields — they failed as
expected (no such export / field absent), then passed once implemented.
