# T046 — Why does the claude-cli service have a per-minute rate limit?

Investigated together on 2026-06-23. Conclusion: the `ratePerMinute: 20` on `claude-cli`
is **inert and slightly misleading** — safe to remove.

**Findings:**
- **Origin:** added in T004 ("publish the perfumes job code", 2026-06-21) as part of the
  initial service definition — a defensive guess ("a gentle per-minute rate just avoids
  bursting"), NOT a response to any observed limit/problem.
- **Enforcement:** `callService` (`core/services.ts`) throttles to ≤20 calls/min via the
  SQLite `service_usage` meter (waits for a slot if exceeded).
- **Actual call pattern:** only the perfumes workflow uses `claude-cli`. Its three claude
  stages (find-url/parse/build) loop over items **sequentially** (`for … await`), perfumes
  is a **linear DAG**, and there's **one active run per workflow** (T105) — so **≤1 claude
  call is ever in flight**. Calls are long (`claudeTimeoutMs` 5 min; real calls tens of
  seconds+), so throughput is ~0.5–6/min — far below 20/min. **The cap never triggers.**
- **Doesn't mirror a real limit:** the true constraint on `claude -p` is the ~5-hour plan
  usage window (handled by the build loop's own backoff), not a per-minute rate. A
  per-minute cap is also the wrong tool for limiting *concurrent* subprocesses (it caps
  throughput, not concurrency).
- **Removal is lossless:** `callService`'s no-rate path still `recordServiceCall(name)`, so
  the Services page keeps showing claude-cli call counts; the service stays editable if a
  limit is ever wanted. Nothing tests the value.

**Decision: REMOVE the per-minute rate limit.** (If parallel claude calls are ever
introduced, the correct guard would be a max-concurrency limit, added then.) The removal is
tracked as a separate buildable task **T141**. Marking this investigation done + reviewed.
