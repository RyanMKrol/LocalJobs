# T451 — Opt-in 5-minute response cache for callService

2026-07-07

## What I did

- `src/db/schema.sql`: added a new `service_cache` table
  (`service_name`, `cache_key`, `response_json`, `cached_at`, PK on
  `(service_name, cache_key)`) — a brand-new table, no additive migration needed.
- `src/db/store.ts`: added `getCachedServiceResponse<T>(serviceName, cacheKey, ttlMs)`
  (returns `undefined` on miss or TTL expiry, else the parsed JSON value) and
  `setCachedServiceResponse(serviceName, cacheKey, value)` (upsert via
  `ON CONFLICT ... DO UPDATE`, matching the existing `upsertWorkflowStmt` idiom).
- `src/core/services.ts`: added `cacheKey?: string` to `CallServiceOpts` and a local
  `SERVICE_CACHE_TTL_MS = 5 * 60_000` constant. Right after resolving `def`,
  `callService` checks `opts.cacheKey && def.category === 'api'` — on a cache hit it
  returns immediately, skipping consumer recording, quota checks, rate limiting, and
  `fn()` entirely. On a miss (or when caching doesn't apply), the existing three
  `return fn()` call sites (min-interval throttle, rate throttle, no-limit) were
  replaced with a shared `runAndCache()` closure that awaits `fn()` and, only when
  caching applies, stores the result before returning it. No existing call site
  passes `cacheKey`, so behavior is byte-identical for every current caller.

## Checks run

- `npx tsc --noEmit` — clean.
- `npm test` — full suite: 230/230 passing (52 suites), including 5 new cases added
  to `src/core/services.test.ts`:
  - cache hit within TTL skips `fn` entirely and records no extra `service_usage` row
  - a different `cacheKey` is its own independent cache miss
  - a backdated `cached_at` (simulated TTL expiry via direct SQL UPDATE) re-invokes
    `fn` and refreshes the row
  - a `cli-tool`-category service ignores a passed `cacheKey` (always calls `fn`,
    writes no `service_cache` row)
  - a `website-scrape`-category service likewise ignores `cacheKey`
  All tests run against the scratch test DB via the existing `isTestEnv()` guard in
  `src/config.ts` — no production `data/jobs.db` touched, no paid API calls made.

test file `src/core/services.test.ts` (tests are always in-scope), plus this worklog.
