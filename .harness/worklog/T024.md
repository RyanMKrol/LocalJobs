# T024 — Remote-accessible dashboard via Tailscale

## Goal
Make the dashboard reachable over a Tailscale tailnet (not the open internet) via
`tailscale serve`, WITHOUT exposing the daemon API. Browser talks only to the
dashboard origin; the dashboard server (on the Mini) proxies `/api/*` to the
loopback API. Deliver the code + docs (machine-side `tailscale up`/`serve` is
operational, out of scope).

## What I changed
- `dashboard/next.config.js` — added a `rewrites()` (`beforeFiles`) that proxies
  `/api/:path*` → `${LOCALJOBS_API_UPSTREAM ?? 'http://127.0.0.1:4789'}/api/:path*`.
  `beforeFiles` so `/api/*` is ALWAYS proxied before Next's own routing (the app
  has no `/api` route, so `afterFiles` was being beaten to a 404 — see gotcha).
- `dashboard/app/lib/api.ts` — `API_BASE` now defaults to **same-origin** (`''`)
  instead of the absolute `http://127.0.0.1:4789`. The browser fetches `/api/*`
  from the dashboard's own origin → rewrite → loopback API. Still overridable via
  `NEXT_PUBLIC_API_BASE` for the directly-exposed-API path.
- `README.md` — new **Remote access over Tailscale** section: one-time
  `tailscale serve --bg 4788`, confirm Funnel is OFF, T023 guards stay in front;
  updated the security note + config table (`LOCALJOBS_API_UPSTREAM`,
  `NEXT_PUBLIC_API_BASE`).
- `.env.example` — documented the two new dashboard env vars (placeholders).
- `.harness/LIMITATIONS.md` — recorded the trade-off: the proxy hop is a loopback
  caller, so the T023 mutation token does NOT gate the proxied path → tailnet
  membership is the auth (Funnel must stay off).

## Verification (empirical, no paid calls)
- `npx tsc --noEmit` → clean.
- `npm test` → all tests passed (9 jobs / 4 services / 2 pipelines registry load).
- `npm --prefix dashboard run build` → BUILD OK; rewrite baked into
  `.next/routes-manifest.json` `beforeFiles`.
- **End-to-end proxy** against the real running daemon (read-only GETs only):
  started the freshly-built dashboard on a spare port (4790, since launchd owns
  4788) and confirmed it serves daemon data through its OWN origin:
  - `GET :4790/api/health` → `{"ok":true,"time":...}` (daemon JSON)
  - `GET :4790/api/jobs` → real jobs list
  - `GET :4790/api/runs?limit=1` → real run (query string + proxy both work)
  The browser-side request hit ONLY the dashboard origin; the dashboard proxied
  server-side to `127.0.0.1:4789`. API never bound off loopback.

## Gotcha (for next time)
- Next.js bakes `rewrites()` into `.next/routes-manifest.json` at **build** time,
  read by `next start` — must rebuild after editing `next.config.js`.
- `afterFiles` rewrites lost to Next's static `_not-found` for `/api/*`; switching
  to `beforeFiles` fixed it (always proxy, never let the app route it).
- The launchd dashboard (`com.ryankrol.localjobs-dashboard`) holds port 4788 with
  the OLD build, so a test `next start -p 4788` silently EADDRINUSE'd and curls hit
  the stale instance (false 404s). Test on a spare port. **Operator must rebuild +
  restart the dashboard agent** for this to take effect in prod.
