# T225 worklog — work-item `skipped` outcome + stop marking success with no output

## What was built

### Part 1: `skipped` outcome in `src/db/store.ts`

`WorkStatus` already included `'skipped'` in the type. Updated `isWorkItemDone` to
explicitly handle `skipped` — returns `false` (item is retried on next run when quota
resets). Added a clear comment explaining the semantics: soft-stopped, not done,
not stuck.

The `stuckItems` query already only matches `failed` rows, so `skipped` items
are not stuck by construction.

### Part 2: Record `skipped` for soft-stops in places stages

**`src/jobs/places/stages/enrich.ts`**: When `callService('google-places', ...)` throws
`QuotaExceededError` or when `res.rateLimited` is true, the current item is now recorded
as `skipped` before breaking. Attempts are NOT incremented (prior count used) since no
actual API call was completed.

**`src/jobs/places/stages/llm-enrich.ts`**: Same pattern — `QuotaExceededError` and
`isRateLimited(err)` both record `skipped` before breaking.

### Investigation: why ingest marks `success` with no enriched output

Ingest's `success` status is intentional per CLAUDE.md: it's a bulk first stage that
anchors the ledger for the I/O mapping. Its "real output" is cataloging — it successfully
parsed and recorded the item for downstream processing. The downstream stages (enrich,
llm-enrich) have separate ledger entries under their own job names.

The "no actual output" problem was that quota-stopped items in enrich/llm-enrich had NO
ledger row — making it invisible that those items were seen but skipped. The fix is adding
`skipped` rows for those items, not changing ingest's `success` status.
