# T603 — fix movie-recs/tv-recs notify-stage ledger job-name mismatch via outputJob

2026-07-16

## What was wrong

`movie-recs-notify` / `tv-recs-notify` record their "have I notified this?" ledger
under the decoupled recommender keyspace (`RECS_JOB = 'movie-recs'` / `'tv-recs'`,
`src/workflows/movies/recs.ts` / `src/workflows/tv-recs/recs.ts`), via
`markWorkItem(domain.recsJob, ...)` in `src/core/recommender/notify.ts`. But the
run-scoped Stage I/O endpoint (`GET /api/workflow-runs/:id/stage-io?job=<job>`)
queried `stageIoLists([jobParam], ...)` using the literal DAG member name
(`movie-recs-notify`), which never has any `work_items` rows — so that stage's
tab on the workflow-run page always showed an empty Outputs list even for a run
that genuinely notified recommendations.

## Fix (owner-confirmed direction: unify with `WorkflowDefinition.outputJob`, T348)

- `src/core/types.ts`: widened `outputJob`'s JSDoc — it can now name either an
  existing member job (stocks-sync-style: an earlier stage's ledger) OR a
  decoupled ledger job_name that isn't a DAG member at all (movie-recs/tv-recs-
  style): a terminal stage's own ledger, recorded under a shared keyspace name
  rather than its own DAG member name.
- `src/api/server.ts`:
  - `GET /api/workflow-runs/:id/stage-io?job=<job>` (the per-stage tab, not just
    `overall=true`): when the requested job is in the DAG's terminal wave and the
    workflow declares `outputJob`, look its ledger rows up under `outputJob`
    instead of the literal member name.
  - Dropped the `refs.some((r) => r.job === def.outputJob)` "must be an actual
    DAG member" guard from all three `outputJob` consumers (`output-items`,
    `stage-io?overall=true`, and the new per-job branch) — it was blocking the
    new decoupled-keyspace use case; a stale/misconfigured `outputJob` now just
    yields an empty (harmless) result, same as any other job_name with no rows.
- `src/workflows/movies/movies.workflow.ts` / `src/workflows/tv-recs/tv-recs.workflow.ts`:
  declared `outputJob: 'movie-recs'` / `outputJob: 'tv-recs'` on the workflow
  manifest (the notify stage — `movie-recs-notify` / `tv-recs-notify` — is
  already the DAG's terminal wave, so this governs both its own Stage I/O tab
  and the Overall tab).
- No data migration: the existing ledger rows stay under `movie-recs`/`tv-recs`;
  only the lookup changed.
- `notify.ts` / `recs.ts` (movies + tv-recs) needed NO changes — they already
  correctly key by `domain.recsJob`; only the *lookup* side was wrong. The
  ignore/un-ignore endpoints (`POST /api/movie-recs/:tmdbId/ignore` etc.) also
  already reference `'movie-recs'`/`'tv-recs'` directly and are untouched.

## Tests (test-first)

Added to `src/api/server.test.ts` (T603 block): a synthetic 2-stage workflow
(`t603-recs-wf`) whose `outputJob: 't603-recs'` names a job that is NOT a DAG
member, mirroring the real movie-recs/tv-recs shape. Confirmed the new test
FAILS (empty outputs) against the pre-fix `server.ts` via `git stash`, then
passes with the fix:
- `GET .../stage-io?job=t603-notify` returns the `t603-recs`-keyed ledger row as
  `outputs` (not empty) — the core T603 assertion.
- A non-terminal member (`t603-snapshot`) is unaffected by `outputJob` — proves
  the remap is scoped to the terminal wave only.

The existing T384 (`outputJob` as a real member, stocks-sync-style) and T348
(`output-items` override) tests still pass unchanged, confirming no regression
to the mechanism's original behaviour.
