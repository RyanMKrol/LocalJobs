# T353 — getWorkflowJobs DAG ordering

## What changed
`getWorkflowJobs()` in `src/db/store.ts` previously returned `workflow_jobs` rows in
SQLite's raw PK-index order (alphabetical by `job_name` within a workflow), not DAG
order. Fixed by reusing `buildDag()` from `src/core/dag.ts`: build the DAG from the
members' `dependsOn`, flatten `waves` to get a topological order, and sort the
members by that order. Falls back to the raw row order if `buildDag` throws
(malformed DAG — duplicate member / dangling depends_on), so the function itself
never throws.

## Verification
- `npx tsc --noEmit` — clean.
- `npm test` — 134/134 pass, including two new cases in `src/db/store.test.ts`:
  - a 3-stage linear chain named so alphabetical order would scramble it
    (`z-first` → `a-second` → `m-third`), asserting `getWorkflowJobs` returns DAG
    order, not alphabetical order.
  - a fan-in shape (two independent branches `fi-a`/`fi-b` feeding `fi-join`),
    asserting the join comes after both its dependencies (order between the two
    branches left unconstrained).
- Confirmed by reading `dashboard/app/workflow-runs/[id]/page.tsx`'s `IoPanel` that
  it maps directly over the `members` prop with no client-side re-sort — the fix at
  the source flows through automatically, no dashboard change needed.
- Confirmed the other `getWorkflowJobs()` consumers (`workflowView`,
  `gatesForWorkflow`, output-items/io endpoints, `resolveBulkScope`,
  `resetWorkflowOutput`, `memberWorkflowMap`) are either fixed by this change too
  (workflow.jobs) or already computed their own `buildDag` internally and are
  unaffected (a harmless no-op reordering of their input `refs`).
