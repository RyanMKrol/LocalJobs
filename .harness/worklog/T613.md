# T613: Add idempotencyNote field to WorkflowDefinition, synced + exposed like category

2026-07-16

## What was done

Mirrored the existing `category` field mechanism exactly (manifest-owned, code-synced on every
sync, no dashboard edit UI, no `_overridden` column) for a new `idempotencyNote` field:

- `src/core/types.ts`: added `idempotencyNote?: string` to `WorkflowDefinition`, doc comment
  states manifest-owned / code-synced / no override column, mirrors `category`.
- `src/db/schema.sql`: added `idempotency_note TEXT NOT NULL DEFAULT ''` column to the
  `workflows` table CREATE statement.
- `src/db/index.ts`: added an additive migration guarded by
  `hasColumn(db, 'workflows', 'idempotency_note')` (no index — doesn't hit the T098 trap).
- `src/db/store/workflows.ts`: added `idempotency_note` to `WorkflowRow`, to
  `upsertWorkflowStmt`'s INSERT column list/VALUES and its `ON CONFLICT ... DO UPDATE SET`
  clause (always refreshed from the manifest, no CASE/override logic); `syncWorkflow` reads
  `idempotencyNote: def.idempotencyNote ?? ''` (defaults to empty string, not a placeholder
  label, so an unauthored workflow renders nothing on the dashboard).
- `dashboard/app/workflows/[name]/page.tsx`: added a local `WorkflowWithIdempotencyNote` type
  (the `Workflow` type in `lib/api.ts` doesn't yet declare `idempotency_note` — same pattern
  the workflows list page already uses for `category` — this file is the only one in scope so
  I extended the type locally rather than touching the out-of-scope `lib/api.ts`), cast `p` to
  it, and rendered `{p?.idempotency_note ? <p className="sub">{p.idempotency_note}</p> : null}`
  directly below the info panel and above the `Graph` heading.
- `dashboard/scripts/_dashboard-harness.mjs`: added a synthetic `idempotency_note` string to the
  default `workflow()` fixture (used by the `/workflows/places` screenshot) so visual-check
  produces a screenshot of the rendered note (living-artifact rule for a UI-surface change).
- `src/db/store.test.ts`: added a test block (mirroring the existing `category` test) asserting:
  (1) a workflow synced with `idempotencyNote` persists it to `idempotency_note`, (2) omitting it
  persists an empty string, (3) a second sync with a changed `idempotencyNote` updates the stored
  value unconditionally (no preservation/override logic).
