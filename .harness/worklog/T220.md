# T220 worklog — DAG graph prototype (chooser)

## What was built

Three switchable DAG graph views behind a localStorage-persisted toggle ("Graph style" switcher
rendered above every workflow graph on the definition and run-scoped pages).

The switcher is exposed via `useDagOption()` + `DagOptionSwitcher` in `dashboard/app/ui.tsx` and
wired into the `Dag` wrapper in `dashboard/app/components/Dag.tsx`. The three implementations share
the same props interface (`DagProps`) as the original Dag component, so all call sites
(`/workflows/[name]`, `/workflow-runs/[id]`) receive all three options without any page changes.

---

## Option A — Wave Columns (`waves`, `DagWaves.tsx`)

**Library:** none (hand-rolled SVG + CSS flexbox — enhanced version of the original layout).

**How it shows waves/concurrency:**
- Each topological column is labelled "Wave N" above the nodes.
- When a wave contains >1 stage, a badge reads "N concurrent" (blue pill, e.g. "8 concurrent" for
  the movie-recommendations fan-out branches).
- Alternating translucent background bands (one per column) visually group stages by wave.
- Bezier edges from producer right-center → consumer left-center (identical to the original layout).

**Trade-offs:**
- Familiar to anyone who has used the existing view; lowest disruption.
- Concurrency is labelled but the horizontal node columns are the same as before — stages in the
  same column look *stacked* rather than side-by-side, so the "parallelism" is annotation-only.
- Best for workflows with a few wide waves (e.g. movie-recommendations fan-out).

---

## Option B — Swim Lanes (`swimlane`, `DagSwimlane.tsx`)

**Library:** none (hand-rolled SVG + CSS flexbox, but a completely different layout direction).

**How it shows waves/concurrency:**
- Waves become horizontal **rows** (swim lanes). Each row is labelled "Wave N" on the left with a
  "×N" multiplier badge when N > 1 stages share that wave.
- Stages in the same wave sit **side-by-side** in their row, making it immediately obvious they run
  at the same time — no annotation needed, the geometry is the proof.
- Alternating light/dark row backgrounds reinforce the grouping.
- Dependency edges route **top-to-bottom** (producer bottom-center → consumer top-center), always
  crossing row boundaries (since a proper DAG's edges always go from smaller wave to larger wave).

**Trade-offs:**
- The clearest representation of concurrency: parallel stages are literally next to each other.
- Wide fan-outs (8 recommender branches in the same lane) create very wide rows; the container
  scrolls horizontally via `overflow-x: auto`.
- On mobile, the label + nodes stack vertically per lane — still readable.
- Edge routing in a swimlane layout can cross multiple rows and overlap for complex graphs; the SVG
  bezier curves handle this but may look busy when many cross-cutting edges exist.

## Option C — React Flow (`flow`, `DagFlow.tsx`)

**Libraries:** `@xyflow/react` v12 (React Flow), `@dagrejs/dagre` (auto-layout engine).

**How it shows waves/concurrency:**
- `dagre` computes node positions with `rankdir: 'LR'`; each dagre **rank** corresponds to one wave.
  Stages in the same rank (same wave) are positioned at the same X coordinate and stacked vertically,
  so the graph spatially groups parallel stages without any extra annotation.
- Translucent **wave group background nodes** (custom `waveGroup` node type) are drawn BEHIND the
  stage nodes, each spanning its wave's stages with a dashed blue border and "Wave N · M concurrent"
  label in the top-left corner.
- Gate marks appear as React Flow edge labels (preserving the `state`-aware styling via inline CSS
  rather than CSS classes, since React Flow isolates node/edge DOM).
- Controls (zoom/fit-view) via React Flow's built-in `<Controls />` panel.
- Nodes are non-draggable (the DAG is defined by the manifest, not user layout).

**Trade-offs:**
- Most polished, interactive feel: pinch-to-zoom, fit-view button, smooth edge curves.
- React Flow ships ~200 KB of additional JS (included in the bundle as `@xyflow/react`).
- Wave group background nodes require careful position computation (dagre layout is needed before the
  groups can be sized); on very wide fan-outs the groups overlap slightly due to dagre's vertical
  node separation not exactly matching the group padding.
- Gate marks are edge `label` props (text) rather than the existing `.dag-gate` pill components,
  which means they lose the click-to-detail link; this is a known prototype limitation (the link
  href IS computed but React Flow's text-label system doesn't render anchor elements).
- CSS from `@xyflow/react/dist/style.css` is globally imported; conflicts are scoped via
  `.dag-flow-wrap .react-flow__*` overrides in `globals.css`.
