# T155 — Why the movies graph shows a gate under gaps-notify (not between two jobs)

Investigated with the owner (2026-06-24) against the live movies run `4b8ad18e`.

**Finding — a visualization shortcoming, NOT a scheduling bug.** The Dag layout (`Dag.tsx`) is
wave-based (a node's column = its distance from the root). The movies DAG places `franchise-gaps` at
wave 2 (alongside the 8 `rec-*` branches), but its only edge is `franchise-gaps →
movie-gaps-notify` at wave 4 — a **skip-wave** edge that bypasses `rec-merge` (wave 3). The Dag can't
draw a skip-wave edge as a single inter-wave arrow, so it falls back to rendering that edge's gate
chip UNDER the consumer (`movie-gaps-notify`) with no connector line. Result: `franchise-gaps` looks
co-located with — and feeding — the rec pipeline, when it actually bypasses `rec-merge` straight to
notify; the only hint of its real path is the orphaned "franchise-gaps" chip under notify.

**Scheduling verified correct (separately):** rec-merge waited for all 8 `rec-*` deps (last finished
17:25:59, rec-merge started 17:25:59); `franchise-gaps` is a legitimate independent parallel branch.
So there is no DAG/ordering bug — only the rendering misrepresents the branch structure.

**Decision:** real visualization bug → fix. Buildable follow-up **T175** filed: render
non-adjacent / skip-wave edges as visible connector lines with the gate mark on the edge, so
bypassing branches read correctly. T155 (investigation) is complete.
