# T458 — shared output-renderer dispatch (json/text renderers)

2026-07-08

## What I did

- Created `dashboard/app/components/OutputRenderer.tsx`, moving the renderer-dispatch
  machinery (`parseFrontmatter`, `MarkdownOutputBody`, `RawOutputBody`, `OUTPUT_RENDERERS`,
  `renderOutputBody`) out of `WorkflowOutputSection.tsx` into a shared module.
- Added a real `json` renderer (`JsonOutputBody`): `JSON.parse` + `JSON.stringify(…, null, 2)`
  inside a monospace `<pre>` (reusing `RawOutputBody`'s styling), falling back to the raw
  content on a parse failure (never throws/blanks).
- Added a `text` renderer (`RawOutputBody`, now also registered under the `text` key).
- Registered both in `OUTPUT_RENDERERS`; unrecognized formats still fall back to the raw
  renderer (e.g. `plex-space-saver`'s `size-table`, unchanged/out of scope).
- `WorkflowOutputSection.tsx` now imports `renderOutputBody` from the new module — no more
  local copy of the dispatch.
- `StageIoLists.tsx`: threaded `format` through the Stage I/O popover's modal state
  (`ModalState` now carries the full `WorkflowRunOutput`, not just `{content, truncated}`),
  and replaced the hardcoded `MarkdownModal` render with a new `StageIoModal` that calls
  `renderOutputBody` from the shared module — so `json`/`text` formats now route correctly
  on this surface too (previously always forced through the markdown viewer regardless of
  declared format).
- **Found + fixed a real pre-existing bug while adding regression tests**: the dispatch's
  `result.format && OUTPUT_RENDERERS[result.format]` short-circuited to `RawOutputBody` for
  an *unset* format, contradicting `WorkflowRunOutput`'s own doc comment ("'markdown' when
  unset") and the `MarkdownOutputBody` comment calling markdown "the default/legacy form".
  Fixed to `OUTPUT_RENDERERS[result.format ?? 'markdown']` so an unset format now correctly
  routes to markdown, as the spec's done-when explicitly requires ("dispatch still picks the
  markdown path for format: 'markdown'/unset").
- Added `dashboard/app/components/OutputRenderer.test.ts` (Node's built-in test runner, same
  convention as `StageIoLists.test.ts`) covering: json pretty-prints valid JSON with real
  indentation, json falls back gracefully (no throw) on invalid JSON, text renders raw
  content, an unrecognized format falls back to raw, markdown format renders through the
  markdown path, and an unset format also renders through markdown (regression + the fix
  above). All 6 pass. Needed an explicit `import React from 'react'` in `OutputRenderer.tsx`
  so the file also renders correctly under a direct `tsx --test` run (classic JSX transform
  outside Next's own automatic-runtime build) — harmless under the real Next build.
- Updated `dashboard/scripts/_dashboard-harness.mjs`: added a JSON-format output item
  (`placesLlmJsonOutput`, key `place:json-summary`, `detail.path` + no `detail.markdown`) to
  the `places-enrich-with-llm` stage's outputs (both the per-stage and Overall-tab StageIo
  fixtures) and to `places`'s `output-items` fixture; wired both `/output` fixture routes to
  return a real (unformatted) JSON string + `format: 'json'` for that item's key so the
  renderer's own pretty-printing is what's actually visible in the screenshot. Added two new
  FLOWS entries: `stage-io-json-output-popover` (clicks the JSON item's link on
  `/workflow-runs/1`'s Stage I/O panel) and `workflow-output-section-json-popover` (clicks
  the JSON item's View button on `/workflows/places`'s Output section).
- Updated root `CLAUDE.md`'s "Shared dashboard UI components (T268)" list: added an
  `OutputRenderer` entry describing what it owns, and touched up the `MarkdownModal`/
  `StageIoPanel` bullets to reference it.
