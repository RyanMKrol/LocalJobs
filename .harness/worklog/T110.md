# T110 — Workflow-run IO: expressive output (markdown preview + full popover)

## Goal
The T095 IO panel showed an output that was basically a copy of the input id. Make the
OUTPUT side expressive: preview the produced markdown artifact (title + excerpt) and open
the FULL markdown in a popover on click, served by a read-only, path-safe API.

## What I built

### Backend
- `src/db/store.ts` — added `workItemMarkdownPath(jobName, itemKey)`: reads the work item's
  `detail.markdown` (DB-only, no FS). Returns null when absent/unparseable.
- `src/api/server.ts`:
  - `GET /api/workflow-runs/:id/output?job=&key=` (read-only). Resolves the markdown path
    from `workItemMarkdownPath`, runs it through `safeOutputMarkdown`, reads the file, and
    returns `{ found, job, key, file, bytes, truncated, content }`. "No artifact" is a benign
    `200 { found: false }`; missing job/key → 400; unknown run → 404. 512 KB payload cap.
  - `safeOutputMarkdown(candidate)` path-safety guard (exported for tests): `resolve` +
    `realpathSync` (symlink-follow) + must stay under `JOBS_ROOT` (realpath of `src/jobs`) +
    must contain a `/data/out/` segment + must be a regular `.md` file. No paid/remote calls.
  - `isWithin(parent, child)` helper (exported), via `path.relative`.
- `src/jobs/perfumes/build.ts` — **harmonise with places**: now records `markdown: mdPath`
  in the success work item's detail (places-llm-enrich already did). This is what makes the
  endpoint generic — it only reads `detail.markdown`, no per-job path knowledge in the API.

### Frontend (`dashboard/`)
- `app/lib/api.ts` — `WorkflowRunOutput` type + `api.workflowRunOutput(id, job, key)`.
- `app/workflow-runs/[id]/page.tsx`:
  - `mdPreview(content)` — derives title (frontmatter `name:` → first `#` heading) + a short
    body excerpt.
  - `OutputCell` — lazily fetches each row's markdown ONCE, shows `📄 title · excerpt · file`,
    falls back to the bare key/detail when `found:false`. Clicking opens…
  - `MarkdownModal` — reuses the DB browser's `db-modal` styling; shows full markdown as
    readable monospace `<pre>` (no markdown-renderer dependency).
- `app/globals.css` — `.md-preview*` + `.md-preview-full` styles (and they pass mobile-check).

## Path-safety design
The API never trusts the stored path blindly: it must resolve+realpath to a real `.md` file
inside a job's own `data/out/` tree under `src/jobs`. Traversal (`..`), symlink escapes,
non-`.md`, files outside `data/out/`, and missing files all return `found:false` — never the
file. Pure local file read, safe to poll.
