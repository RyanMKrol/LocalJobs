# T187 — Design the ideas-vs-backlog two-step process

**Status:** design agreed AND implemented directly with the owner in this session (2026-06-25). The
owner chose to build it live rather than queue follow-up tasks.

## The problem

Today raw ideas get thrown at Claude Code in bulk and asked to become backlog tasks. With the
harness's now-high planning bar (per-task `## Do`/`## Done when` spec, `scope`, `dependsOn`,
`facets`, `verify`), a bulk dump produces rushed, low-quality task specs, and Claude loses track
juggling several half-formed ideas while also mid-build on something else.

## Agreed design

A deliberate **two-step** flow, with **ideas as a first-class harness concept**:

### Step 1 — capture: the ideas inbox (`.harness/IDEAS.md`, GITIGNORED)
- A dead-simple, **gitignored** markdown scratchpad — a single `## Inbox` list, one bullet per
  idea, one or two sentences, no schema, no planning. Lowest possible friction.
- **Gitignored**, consistent with `data/` folders: the *mechanism* (docs in the committed
  `.harness/CLAUDE.md`) travels with the harness for distribution; the raw ideas themselves stay
  local/private and never hit the public repo. Each project grows its own inbox.
- Capture = hand-edit the file, or tell Claude "add an idea: …" and it appends a bullet. That's it.

### Step 2 — convert: a TWO-PHASE interview (one idea at a time)
Conversion is its OWN harness process — **not** the bare `ralph-loop-add-to-backlog` skill. It leans
on that skill but adds a probing front-end, because an idea is typically barely formed:
- **Phase 1 — idea excavation.** Treat the idea as one vague sentence. Probe the owner first: what's
  the underlying itch/problem, what are they really after, rough shape, why it matters — BEFORE any
  task-shaping. Default to *more* questions here; assume nothing is fleshed out.
- **Phase 2 — task shaping.** Feed the now-understood idea into the existing
  `ralph-loop-add-to-backlog` interview (DoD, scope, dependsOn, facets, spec MD) → schema-correct
  task.
- **Discipline:** ONE idea per conversion, and ideally not while mid-build on something else (that
  context-juggling is the root problem this solves).

### Bookkeeping — delete on convert
Once an idea becomes a task, **delete it from `.harness/IDEAS.md`**. The resulting TASKS.json task
(+ its spec MD) is the record; the gitignored inbox stays a clean, transient surface. (No "Converted"
archive — the inbox is gitignored, so there'd be no git history of it anyway.)

### Where it lives / distribution
- Authoritative process documented in `.harness/CLAUDE.md` (committed → travels with the harness).
- Ergonomic entry points as project slash-commands (`/idea` capture, `/convert-idea` convert).
- Folding the conversion into the distributable `claude-skills` plugin (so other projects get it) is
  deferred to the harness-parity work in **T188** — v1 proves it out here first.

## Decisions locked (from the owner)
1. Ideas file: `.harness/IDEAS.md`, **gitignored**.
2. Conversion: ideas **baked into the harness** as a first-class concept; a dedicated two-phase
   convert process that **reuses** add-to-backlog but adds a Phase-1 idea-excavation probe.
3. Converted idea: **deleted** from the inbox.

## Implemented this session (no follow-up tasks needed)
- `.harness/IDEAS.md` — the gitignored inbox scaffold (`## Inbox`, format comment).
- `.gitignore` — `.harness/IDEAS.md` added to the Ralph-harness transient-scratch block.
- `.harness/CLAUDE.md` — new "Ideas inbox & the two-step flow" section (capture + two-phase convert
  + delete-on-convert + worked example; travels with the harness).
- `.claude/commands/idea.md` — `/idea <text>` capture command (append-only, no planning).
- `.claude/commands/convert-idea.md` — `/convert-idea` two-phase convert (excavate → add-to-backlog
  → delete the bullet).

Distribution into the `claude-skills` plugin remains tracked by **T188** (harness parity).
