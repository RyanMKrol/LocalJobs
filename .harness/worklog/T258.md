# T258 — Plumb skipped/no-op outcome

## What was built

Two linked bugs fixed:

### 1. Root re-selection bug (perpetual noop runs)

A root whose items all have `status = 'skipped'` (quota soft-stop from a prior run)
was being treated as "outstanding" by `rootHasOutstandingWork` and as "not blocked" by
`rootHasTerminalBlocker`, so `selectPendingRoots` kept re-selecting it every limited
run. The run would start, find nothing to do, and settle `success` silently.

**Fix in `src/db/store.ts`:**
- `rootHasOutstandingWork`: added `'skipped'` to the excluded-from-outstanding set
  (items soft-stopped by quota are NOT outstanding — they're done for now)
- `rootHasTerminalBlocker`: added `'skipped'` as a terminal blocker so a root with
  only quota-soft-stopped items is treated as "stuck" (like ignored/exhausted-failed)
  and not re-selected by limited runs

Unlimited scheduled runs still retry `skipped` items (`isWorkItemDone` returns false
for `skipped`), which is correct — quota resets nightly/monthly and the next run
should re-attempt.

### 2. Noop stage/run status (all items already done)

A successful stage with an empty `todo` list returned normally → status `success`.
This made it impossible to distinguish "did real work" from "had nothing to do".

**New functions in `src/db/store.ts`:**
- `hasJobAdvancedAnyItem(workflowRunId, jobName)`: queries `work_item_runs` for
  any row with the given (workflow_run_id, job_name) pair
- `workflowRunAdvancedAnyItem(workflowRunId)`: queries `work_item_runs` for any
  row with the given workflow_run_id (whole-run noop check)
- `setRunNoop(runId)`: updates a `runs` row status to `'skipped'` in-place
- `FinalWorkflowRunStatus = WorkflowRunStatus | 'skipped'`: extends the workflow
  run status type for the new outcome; `finishWorkflowRun` now accepts it

**Changes in `src/core/workflow-executor.ts`:**
- `isLimitableWorkflow = !!findRootStage(dag)`: computed once; noop detection
  only applies to limitable workflows (those with `inputKeys()`), because
  non-limitable workflows (plex, movies) re-scan on every run by design
- Per-stage noop detection in `runOne`: after a successful stage completes, if
  `isLimitableWorkflow && !hasJobAdvancedAnyItem(workflowRunId, job)`, the member
  run is marked `'skipped'` via `setRunNoop(runId)`. The `runOne` return value
  stays `'success'` so `executeDag` does not cascade-skip dependents
- Whole-run noop in status roll-up: `!anyRealWork && statuses.every(s==='success')`
  → workflow run status `'skipped'`. For notification purposes, `'skipped'` maps to
  `'success'` (benign, nothing failed)

## Tests added

- `src/db/store.test.ts`: T258 block tests `selectPendingRoots` excludes skipped

- `src/core/workflow-executor.test.ts`: T258 tests that a limitable workflow where

  workflows still settle `'success'` unaffected
