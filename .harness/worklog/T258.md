# T258 worklog — noop status model + root re-selection fix

## Root cause

Running the `places` workflow with a limit (e.g. 5) repeatedly selects the same
already-processed or quota-stuck places, does zero work, yet reports `succeeded`.

Two interacting causes:

1. **`skipped` WorkStatus keeps roots perpetually pending.** `isWorkItemDone`
   returns `false` for `skipped` (by design — quota soft-stop should retry). But
   `rootHasOutstandingWork` only excluded `success` / `ignored` / `failed past maxAttempts`
   from "outstanding". A root whose terminal stage is `skipped` therefore always
   looked outstanding → re-selected every limited run, every time.

2. **Pre-loop quota exhaustion leaves zero `work_item_runs` rows.** When the
   enrich/llm-enrich stage checked the service quota BEFORE the loop and found it
   exhausted, it logged a warning and returned — without marking any items. The
   executor's `hasNonNoopItemsInRun` check returned false (no rows), so it couldn't
   distinguish "nothing to do" from "real success".

## Fix

**New `noop` WorkStatus** (distinct from `skipped`):
- `skipped` = quota hit MID-attempt; item was being processed. Retryable, still
  selectable on the next limited run.
- `noop` = quota exhausted BEFORE any attempt. Retryable (`isWorkItemDone` returns
  false), but a terminal-stage `noop` row counts as "reached terminal" so the root
  is NOT perpetually re-selected.

### `src/db/store.ts`
- Added `'noop'` to `WorkStatus` type.
- `isWorkItemDone`: `noop` returns `false` (retryable, like `skipped`).
- `rootHasOutstandingWork`: excludes `noop` from "outstanding" work.
- `rootReachedTerminal`: includes `noop` as "reached terminal" (fixes re-selection).
- New `hasNonNoopItemsInRun(runId, jobName)`: returns `true` when the run advanced
  at least one non-noop item. Used by the executor to settle stages as `skipped`.
- `finishWorkflowRun` signature extended to accept `'skipped'`.

### `src/core/workflow-executor.ts`
- After `runJobForWorkflow` returns `success`, check `hasNonNoopItemsInRun`. If
  false → override run to `'skipped'` ("nothing to do").
- Workflow status computation: if ALL stages are `skipped` → workflow settles
  `'skipped'`. If some are `success` → `'partial'`. Otherwise existing rules.
- Local `WorkflowRunStatusOrSkipped` alias (can't extend `WorkflowRunStatus` in
  types.ts — out of scope); cast when calling out-of-scope `notifyWorkflow`.

### `src/jobs/places/stages/enrich.ts`
- Pre-loop quota exhaustion: mark all `todo` items as `noop` (instead of just
  logging and returning).

### `src/jobs/places/stages/llm-enrich.ts`
- Same pre-loop quota guard: mark all `todo` items as `noop`. Critical because
  `enrich-with-llm` is the terminal stage — `noop` here fixes root re-selection.

## Tests added (`src/db/store.test.ts`)

1. **T258 noop WorkStatus**: `isWorkItemDone` returns false; noop not in stuck
   list; terminal-stage noop root NOT re-selected; non-terminal noop root still
   pending; `skipped` still selectable.
2. **T258 `hasNonNoopItemsInRun`**: false when no items or all-noop; true when any
   non-noop item advanced; skipped counts as non-noop.
