# T038 — Rename 'pipeline' → 'workflow' everywhere

## What was done
Mechanical-but-wide rename of the "pipeline" concept to "workflow" across the whole repo.

### File / dir renames (git mv, tracking preserved)
- `src/core/pipeline-executor.ts` → `src/core/workflow-executor.ts` (+ `.test.ts`)
- `src/jobs/places/places.pipeline.ts` → `places.workflow.ts`
- `src/jobs/perfumes/perfumes.pipeline.ts` → `perfumes.workflow.ts`
- `dashboard/app/pipelines/` → `dashboard/app/workflows/`
- `dashboard/app/pipeline-runs/` → `dashboard/app/workflow-runs/`

### Identifier substitution
Case-preserving `perl -i` over all `*.ts`/`*.tsx`/`*.sql` under `src/` + `dashboard/app/`,
plus `README.md`, `CLAUDE.md`, and `dashboard/app/globals.css` (two CSS comments):
`PIPELINE→WORKFLOW`, `Pipeline→Workflow`, `pipeline→workflow`. This covered all 699
occurrences — types (`PipelineDefinition`→`WorkflowDefinition`, `PipelineRunRow`, etc.),
store fns (`syncPipeline`→`syncWorkflow`, `createPipelineRun`→`createWorkflowRun`, …),
the `*.pipeline.ts` discovery glob in `registry.ts`, API routes (`/api/workflows`,
`/workflow-runs`), dashboard pages/labels, the `workflow` field on jobs, schema table
names (`workflows`, `workflow_jobs`, `workflow_runs`, `workflow_run_logs`) + their
`workflow_name`/`workflow_run_id` columns, and all docs.

Verified 0 remaining `pipeline` (any case) in scope after the pass.

### DB migration (data-preserving)
Added `migrateWorkflowRename(db)` in `src/db/index.ts`, run **before** schema bootstrap
(so `CREATE TABLE IF NOT EXISTS workflow_*` finds the renamed-in-place populated tables
instead of making empty ones). It:
- `ALTER TABLE … RENAME TO` for the 4 tables (guarded: only if old exists & new doesn't)
- `ALTER TABLE … RENAME COLUMN` for `pipeline_name`→`workflow_name` (×2),
  `pipeline_run_id`→`workflow_run_id` (in `workflow_run_logs` and on `runs`)
- drops legacy indexes (`idx_pipeline_runs_name`, `idx_pipeline_run_logs`,
  `idx_runs_pipeline`) so the schema recreates them under workflow names
Idempotent (no-op on a fresh DB and on an already-migrated DB). `foreign_keys = ON`
means SQLite rewrites child FK references automatically during the renames.

### Tests
New `src/db/migrate-workflow.test.ts`: builds a throwaway DB with the legacy
`pipeline_*` layout + seed rows, runs `migrateWorkflowRename`, asserts tables/columns
renamed, **all rows preserved**, legacy indexes dropped, idempotent second run, and
fresh-DB no-op.

### Docs
README.md + CLAUDE.md renamed via the same pass. Added a T038 trade-off entry to
`.harness/LIMITATIONS.md` (clean break, no redirects; one-way migration).
