# T480 — missing-tv-seasons workflow ledger chaining

## What was built

Added work_items ledger rows to the first two stages of the missing-tv-seasons workflow, enabling the dashboard's per-run Inputs & Outputs panel to show real per-show and per-season data instead of being blank. Stage 3 (notify) already recorded ledger rows; stages 1-2 now do too, with explicit rootKey chaining to enable the dashboard's input→output lineage joining.

### Stage 1: plex-tv-snapshot
- Imports `markWorkItem` from `src/db/store.js`
- Records one ledger row per show in the snapshot with:
  - **Key**: `tmdbId` (as string) when present, else `ratingKey` (Plex id)
  - **Status**: always `'success'` (a show is successfully snapshotted whether it has a TMDB GUID or not)
  - **Detail**: captures show name, TMDB id (or null), and highest owned season
- No impact on re-scan behavior: stages 1-2 still re-compute fresh every run (ledger is for visibility, not idempotency)

### Stage 2: tmdb-season-check
- Imports `markWorkItem`
- Records one ledger row per show, keyed the same way stage 1 does, with explicit `rootKey` chaining
- Records ALL four outcomes:
  1. **Checked successfully, actionable** (`status='success'`): includes TMDB status, highest aired season, and list of complete missing seasons
  2. **Checked successfully, nothing missing** (`status='success'`): includes TMDB status, highest aired season, and empty completeMissingSeasons list
  3. **No tmdb:// GUID** (`status='failed'`): `reason: 'no tmdb:// GUID'`. **This now blocks the run** (desired behavior per spec)
  4. **Genuine TMDB call error** (`status='failed'`): includes error message
- **Critical behavior change**: shows with no tmdb:// GUID now block the run via the existing `if (failed > 0) throw` guard
- Updated the throw message to be accurate: `"${totalFailed} show(s) failed this run (${unverifiableFailedCount} unverifiable + ${failed} TMDB errors) — see logs above"`

### Stage 3: plex-seasons-notify
- Updated existing `markWorkItem` call to add `rootKey: String(s.tmdbId)` so the dashboard can chain stage 3's per-season ledger rows back through stage 2's per-show rows to stage 1's per-show rows
- No behavior change; only visibility improvement

## Behavior changes

1. **Blocking behavior**: Shows with no tmdb:// GUID now block the run until manually resolved via the dashboard's Unstick/Ignore controls. This is intentional per the spec.
2. **Error message**: The thrown error message at the end of stage 2 now clearly breaks down the count of unverifiable vs. TMDB-error failures, e.g., `"1 show(s) failed this run (1 unverifiable + 0 TMDB errors) — see logs above"`.
3. **Dashboard visibility**: The workflow-run Inputs & Outputs panel (StageIoPanel) now shows real per-show ledger rows for stages 1-2 instead of being blank, enabling the owner to see what each stage processed.

## Accepted limitations (deliberately out of scope per spec)

An unverifiable show with no tmdb:// GUID will keep blocking the weekly run until manually resolved. Neither Unstick nor Ignore provides a permanent fix — Unstick deletes the row so it fails fresh next run (the show still has no GUID), and Ignore marks it ignored but stage 2 does NOT currently consult the ledger before recomputing failed status, so it would be recounted as failed again. This is documented as an accepted rough edge and explicitly noted as out of scope for this task. A real fix (checking for ignored rows before counting toward failed) is deferred for future work if the owner actually hits this scenario.

  ORIGINAL test's quota-forcing (`syncService({ name: 'tmdb', dailyCap: 0 })`) never actually enforced

  same in-scope file, surfaced directly by the Done-when's own coverage requirement, so fixed here.
- **`stages/notify.test.ts`**: added `getWorkItem` assertions confirming each notified pair's `root_key`
