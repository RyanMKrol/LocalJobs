# T076 — Backlog DAG view: nicer node rendering + drop the 'next' indicator

## Scope
- `dashboard/app/components/Dag.tsx`
- `dashboard/app/backlog/page.tsx`
- `dashboard/app/globals.css`

## What I did

### (1) Nicer node rendering (`BacklogDag` in `Dag.tsx`)
Replaced the bare `id + status word` node with a styled card:
- header row: mono task **id** on the left + a pill-shaped **status badge** on the right
  (done · needs human · pending), coloured by state;
- the task **title** below (was previously hidden — only the id showed).

New CSS classes in `globals.css`: `.backlog-node` (width clamp 188–220px),
`.backlog-node-head`, `.backlog-node-id`, `.backlog-node-badge` (+ `.done` / `.needs-human`
colour variants), `.backlog-node-title` (wraps normally). Reuses the shared `.dag` /
`.dag-wave` / `.dag-arrow` / `.dag-node` / `.selected` scaffolding, so horizontal scroll
(`.backlog-dag-scroll`), click-to-select, and the selected-node ring all still work.

### (2) Removed the 'next' concept from the DAG view entirely
- `Dag.tsx`: dropped the `nextId` prop, the `id === nextId → 'running'` status class, and
  the `▶ next` node-status label.
- `page.tsx`: removed `findNextEligible` + the `nextId` local; dropped `isNext` from
  `DagDetailPanel` (and its blue border branch); removed the `· next` legend entry and
  added a `· pending` legend entry instead (with a new `.dag-legend.pending` dot);
  updated the now-stale `statusPill` comment that claimed "The DAG view still highlights
  the next-eligible task" (T073's decision — this task supersedes it).
- `globals.css`: removed the unused `.dag-legend.next` rule. (`.dag-node.running` is kept —
  still used by the *workflow* run DAG, not the backlog.)
