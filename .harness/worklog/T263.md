# T263 — Plex space-saver workflow

## What was built

New tracked workflow `src/jobs/plex-space-saver/` (weekly, single stage):

- `config.ts` — reuses `plexConfig` (host/token/machineId) and the existing
  `PLEX_MOVIE_SECTION`/`PLEX_TV_SECTION` env vars (no new env vars).
- `types.ts` — minimal Plex API response shapes + the `SizeBreakdownFile`/`SizeBreakdownItem` output shape.
- `lib.ts` — pure helpers: `itemBytes` (sum `Media[].Part[].size`), `formatBytes`,
  `buildMovieRows` (one row per movie), `buildShowRows` (one row per TV show,
  summing every episode across every season via `grandparentRatingKey`),
  `buildBreakdown` (combine + sort biggest-first).
- `stages/scan.ts` + `stages/scan.job.ts` — the only stage (`plex-space-saver-scan`):
  fetches the movie section, the TV section's shows, and its flat episode list
  (`type=4`) via the shared `plexGet`, computes the breakdown, writes
  `data/out/size-breakdown.json`, and records a `work_items` ledger row keyed by
  ISO week (`weekKey`, mirrors `stock-digest`) so a manual re-run the same week
  regenerates rather than duplicates. Report only — never flags/suggests deletions.
- `plex-space-saver.workflow.ts` — single-stage manifest, `category:
  'regular-maintenance'`, weekly schedule (`'0 6 * * 0'`). One stage → no DAG
  edge → no gate needed (per `src/jobs/CLAUDE.md`).
- `plex-space-saver.test.ts` — hermetic pure-logic tests (byte summation,
  formatting, one-row-per-movie/show, biggest-first sort, weekKey) — no live Plex.

## Output-form mechanism (T262)

The ledger row's `detail.format: 'size-table'` + `detail.path` point the
existing generic output-fetch endpoint (`resolveOutputForm` in
`src/api/server.ts`, unmodified) at the structured JSON breakdown, served via
`safeOutputFile`. The dashboard's `WorkflowOutputSection` has no dedicated
renderer for `'size-table'` yet, but its declared fallback (`RawOutputBody`)
renders any undeclared form's raw content — the file's own doc comment
explicitly anticipates this as T263's use case ("so a new form … is usable
before its own renderer is added"). So the structured JSON is visible with
**no dashboard changes**, which matters because `dashboard/` is out of this
task's scope.

**Known limitation (in-scope-compatible workaround, not fully solved):** the
Output section's LIST query, `workflowTerminalItems` in `src/db/store.ts`, only
shows a "View" button when `detail.markdown` is a truthy string — it predates
T262's format dispatch and wasn't updated for non-markdown forms. `src/db/store.ts`
is outside this task's scope to edit, so `detail.markdown` is ALSO set (to the
same JSON path) purely so the button renders; the fetch endpoint dispatches
correctly on `detail.format`/`detail.path` regardless of what `detail.markdown`
contains, so clicking "View" correctly serves the structured JSON, not a
(nonexistent) markdown render of that path. A follow-up task should update
`OutputItem`/`workflowTerminalItems` to gate on "any declared output form" so
this workaround can be dropped — flagging for a human, not filing it here.
