# T029 ‚Äî Per-item progress across all item-loop jobs

## Goal
Every job that processes N items should emit `ctx.progress(i/N*100)` + a per-item
`i/N` log line as it works, so the run % advances live (and rolls up into the
pipeline) instead of jumping 0‚Üí100 at the end. Cover ALL such jobs.

## Audit of the 8 registered job loops
- **places** (already compliant ‚Äî left untouched):
  - `ingest.ts` ‚Äî `ctx.progress(((i+1)/csvFiles.length)*95, ‚Ä¶)` per list + per-place logs.
  - `resolve.ts` ‚Äî `ctx.progress(((i+1)/todo.length)*100, ‚Ä¶)` + `[i+1/todo.length]` logs.
  - `enrich.ts` ‚Äî `ctx.progress((min(i+1,willAttempt)/willAttempt)*100, ‚Ä¶)` + `[i+1]` logs.
  - `llm-enrich.ts` ‚Äî `ctx.progress((min(i+1,todo.length)/todo.length)*100, ‚Ä¶)` + `[i+1]` logs.
- **perfumes** (the gap ‚Äî fixed here): `find-url.ts`, `fetch.ts`, `parse.ts`,
  `build.ts` all looped `for (const p of todo)` with NO `ctx.progress` per item
  and no `i/N` counter ‚Äî they only logged a final summary + per-item ‚úì/‚úó.

## Change
- Converted each perfumes stage loop to `for (const [i, p] of todo.entries())`,
  added a pre-item log `[stage] i/total ‚Üí label(p)` (narrates each item before the
  slow Claude/browser call), and a per-item progress update after each item.
- Added shared helper `reportItemProgress(ctx, done, total, suffix?)` in
  `perfumes/lib.ts` (pure; guards `total<=0` ‚Üí 100, never NaN) and routed all four
  stages through it. Denominator = `Math.min(todo.length, cap)` where
  `cap = runLimit>0 ? runLimit : Infinity` ‚Äî the count actually attempted this run.
- New unit test `perfumes/progress.test.ts` drives the helper directly (no config/
  DB/live calls): asserts one update per item, evenly-spaced strictly-increasing
  percentages landing exactly on 100, the `done/total ¬∑ suffix` status format, and
  the `total 0 ‚Üí 100%` (no divide-by-zero) edge.
- Docs: added a "item-loop jobs report progress per item" convention to CLAUDE.md.

## Why no integration test of the run functions
`scripts/run-tests.ts` imports every `*.test.ts` into ONE process, and
`perfumesConfig` resolves its input/output paths at import time to the real
(gitignored, CI-absent) `data/` dir with no env override. Driving `runFindUrl`
etc. would read real data (absent in a fresh CI checkout) and write into the
gitignored data dir ‚Äî not CI-safe. So the testable behaviour (the progress math
that every stage now shares) was extracted into `reportItemProgress` and unit
tested there. No live paid/CLI calls made.
