# T505: Cache gemini place-enrichment for 22 hours

**Built:** 2026-07-12

## Summary

Implemented caching for the gemini-based place enrichment in the places workflow. This prevents re-paying for the same place's summary generation during development when running enrich-with-llm multiple times.

## Changes Made

1. **`src/services/gemini.service.ts`**
   - Added `cacheTtlMs: 79_200_000` (22 hours) to the gemini ServiceDefinition
   - This aligns with the caching TTL used by google-places service

2. **`src/workflows/places/stages/llm-enrich.ts`**
   - Added cacheKey to the callService call at line 125: `` `gemini:place:${place.placeId}:v1` ``
   - The cacheKey includes:
     - Service prefix: `gemini:place:`
     - The place id (stable identifier)
     - Version tag `v1` for prompt invalidation
   - Added comment at `buildPrompt()` documenting that the version tag must be bumped when the prompt changes

3. **`src/workflows/places/stages/llm-enrich.test.ts`**
   - Added test to verify gemini service has the correct cacheTtlMs (79_200_000 = 22 hours)

## Caching Mechanism Verified

The existing caching infrastructure (T451/T476) is leveraged:
- callService() accepts a `cacheKey` option
- When a cacheKey is provided with an api-category service, responses are cached in the service_cache table
- Cache hits within the TTL return the cached value without invoking the function or incrementing usage
- Cache entries expire after 22 hours, allowing fresh calls on subsequent runs

## Notes

- The caching is purely a dev-convenience; in production the places work_items ledger already ensures gemini is called only once per place
- The `v1` version tag serves as a cache-invalidation mechanism for prompt changes — should be bumped to `v2`, etc. if buildPrompt() is modified
- No live paid API calls were made during verification (used existing test infrastructure)
