# T102 ‚Äî Workflow definition view: gate chips open a run-agnostic gate detail

## Problem
On the workflow DEFINITION view (`/workflows/[name]`), the structural-graph gate
chips (`StructuralGate` marks in `Dag.tsx`) linked into a SPECIFIC run's gate page
(`/workflow-runs/<lastRunId>/gates/...`). Wrong for a definition view ‚Äî a job node
there opens the read-only `/jobs/<name>`, not a run. The gate chip should behave the
same way: open a run-agnostic, definition-level view of the gate itself.

## What I built
1. **New read-only API** `GET /api/workflows/:name/gates/:producer/:key` (`src/api/server.ts`).
   Returns the structural gate from `gatesForWorkflow` (`deriveGates` + enriched
   producer/consumer contract descriptions) plus each side's declared `shape` ONLY.
   It does **not** run any contract `check()` ‚Äî so it touches no `data/` files and
   makes no paid/remote calls at all (purely static contract metadata). Returns 404
   for unknown workflow or unknown gate.
2. **api.ts**: added `StructuralGateDetail` / `GateShapeSide` types and the
   `api.workflowGate(name, producer, key)` client method.
3. **New page** `dashboard/app/workflows/[name]/gates/[producer]/[key]/page.tsx`:
   run-agnostic gate detail. Same left-to-right Produced ‚Üí Gate ‚Üí Consumed layout as
   the run-scoped page, but shows ONLY each contract's expected shape (summary ¬∑
   format ¬∑ expectations) ‚Äî no actuals, no run state. Each side links to the member
   job's `/jobs/<name>` page.
4. **Rewired `Dag.tsx`**: structural gate chips now take a `workflowName` prop
   (replacing the old `lastRunId`) and link to `/workflows/<name>/gates/...`. Run-view
   chips still use `workflowRunId` ‚Üí the run-scoped page, UNCHANGED. Updated the caller
   in `workflows/[name]/page.tsx` to pass `workflowName={name}` instead of `lastRunId`.
5. Docs updated in lockstep: CLAUDE.md (new bullet under the gate-page section) and
   README.md (paragraph on the definition-view gate page).
