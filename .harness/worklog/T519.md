# T519 — plex-space-saver shrink guard

2026-07-12

## What I did

Added a run-over-run "silent library shrink" safety net to the existing single-stage
`plex-space-saver-scan` job:

- `types.ts`: new `SizeBaselineFile` (`{ totalBytes, at }`).
- `config.ts`: `baselineOut` path (`data/out/size-baseline.json`) and `dropThresholdGb`
  (env `PLEX_SIZE_DROP_GB`, default `1` — absolute GB per the owner's decision in the
  spec, NOT a percentage).
- `lib.ts`: pure helpers `readBaseline`/`writeBaseline` (I/O, JSON round-trip) and
  `checkDrop(prior, currentTotalBytes, thresholdGb)` (pure decision logic — no prior
  baseline or `current >= prior` never exceeds; drop must exceed the threshold in
  bytes).
- `stages/scan.ts`: after writing the size breakdown as before, reads the prior
  baseline, logs the prior/current/drop/threshold, and — only when the drop exceeds
  the threshold AND this baseline hasn't already been alerted (notify-once via the
  `work_items` ledger, job `plex-space-saver-shrink-alert`, keyed by the baseline's
  `at` timestamp, mirroring `missing-tv-seasons/stages/notify.ts`) — sends exactly one
  critical push via `core/notifier`'s `push` (urgent priority, `rotating_light,warning`
  tags). The push function is injectable via `ScanOpts.push` (tests). The baseline is
  written at the end of every successful scan (alert or not) so the next run has a
  fresh prior total.
- Confirmed no service-layer caching was added to the Plex reads (the `plex` service
  declares no `cacheKey`/TTL already — no change needed).
- `CLAUDE.md` updated to document the baseline file, `PLEX_SIZE_DROP_GB`, and the
  alert behaviour end-to-end.

## Tests

Extended `plex-space-saver.test.ts` (synthetic, no live Plex call):
- `checkDrop` against prior 1000 GB / current 800 GB (200 GB drop) → exceeds a 1 GB
  threshold; prior 1000 GB / current 999.5 GB (0.5 GB drop) → does not exceed; stable
  and growing totals → never exceed; no prior baseline (first run) → never exceeds.
- Notify-once guard exercised directly against the scratch DB's `work_items` ledger
  (`isWorkItemDone`/`markWorkItem`) with a fake injected push function: first run
  against an exceeding baseline sends exactly one alert and marks the ledger; a second
  run against the SAME baseline sends zero further alerts.
