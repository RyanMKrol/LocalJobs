# T599 — fix dead `recordBranchLedgerRow` (branch-runner shadowing bug)

2026-07-16

## What was wrong

Every one of the 16 recommender-branch DAG jobs (8 movies + 8 tv-recs) reported
`status='success'` on a real run but never wrote a `work_items` ledger row.

Root cause: each branch's `*.job.ts` calls its domain's `makeBranchJob(id)`
(`movies/stages/recommend.ts` / `tv-recs/stages/recommend.ts`), which calls
`coreMakeBranchJob(domain, id, contracts)` in `src/core/recommender/branch.ts`.
The `JobDefinition` that function returns has its own `run(ctx)` that calls
`runBranch(ctx, domain, spec)` — by ordinary JS scoping this resolves to
`branch.ts`'s OWN local `runBranch`, NOT the domain wrapper's `runBranch` in
`recommend.ts` (the only place that called `recordBranchLedgerRow`). So the
ledger-writing code was dead for every real job execution — it only ran when
something called the domain wrapper's exported `runBranch` directly (tests).

## Fix

Added an optional `onBranchWritten?: (branchId, opts) => void` hook to
`BranchRunOpts` (`src/core/recommender/branch.ts`). The core `runBranch` now
calls it right after every `writeBranchFile` call (success path + all three
early-return skip/error paths). `makeBranchJob` gained a `MakeBranchJobOpts`
parameter (`onBranchWritten` + a test-only `runOpts`) and threads it into the
`runBranch` call inside the returned `JobDefinition.run(ctx)` — so the hook now
genuinely fires from the REAL job execution path.

Both domain wrappers (`movies/stages/recommend.ts`, `tv-recs/stages/recommend.ts`)
now pass their existing `recordBranchLedgerRow` as `onBranchWritten` when
building the job via `coreMakeBranchJob`. Their own exported `runBranch` (kept
for API-compat with existing tests/call sites) was simplified to route through
the SAME hook mechanism instead of a separate duplicated manual call — so there
is exactly one `runBranch` implementation actually executing per branch job.
`makeBranchJob(id, runOpts?)` also grew an optional test-only second param so a
test can inject a mock `runClaude` + fixture paths and drive the real
`run(ctx)` end-to-end without a live Claude call.

## Tests

Added `src/core/recommender/branch.test.ts`: drives
`makeBranchJob('rec-random-1', runOpts).run(ctx)` (movies) and
`makeBranchJob('tv-rec-random-1', runOpts).run(ctx)` (tv-recs) end-to-end with
an injected `runClaude`, then asserts `getWorkItem(branchId, dayKey(now))` now
returns a row with `status='success'` and the existing
`{ name, suggestions, path }` detail shape.

Verified the test catches the regression: stashed just the `branch.ts` change
(pre-fix) and re-ran the test in isolation — it failed with
`AssertionError: a work_items row exists for rec-random-1 after a real job
run(ctx)` (row was `undefined`), confirming it exercises the real bug, not just
the domain wrapper's already-correct standalone `runBranch`. Restored the fix
and re-ran — both assertions pass.
