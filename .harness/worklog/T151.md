# T151 — Rename the `plex` workflow → `missing-tv-seasons` + DB migration

## What I did

1. **Manifest** (`src/jobs/plex/plex.workflow.ts`): `name: 'plex'` → `name: 'missing-tv-seasons'`.
   Member job names (`plex-tv-snapshot`, `tmdb-season-check`, `plex-seasons-notify`) left
   unchanged — their `work_items` are keyed by job name, so renaming would orphan the ledger.
   The `src/jobs/plex/` folder was kept as-is (registry discovers by glob, not folder name).
   The description already only described behaviour, not the slug, so it stayed.

2. **DB migration** (`src/db/index.ts`): added `migrateRenamePlexWorkflow(db)`, called after
   the other additive migrations. Idempotent — guarded on the old `plex` workflows row existing.
   Renames `workflows.name`, `workflow_jobs.workflow_name`, and `workflow_runs.workflow_name`
   from `'plex'` to `'missing-tv-seasons'`. Because `foreign_keys = ON`, NO single ordering of
   the three UPDATEs avoids a transient orphan (child→parent or parent→child both violate
   mid-statement), so the migration toggles `foreign_keys = OFF` *outside* a transaction (PRAGMA
   is a no-op inside one), wraps the three updates in a transaction for atomicity, and restores
   `foreign_keys = ON` in a `finally`.

3. **Test** (`src/db/migrate-existing-db.test.ts`): seeded a pre-T151 `plex` workflow + a
   `workflow_jobs` member link + a `workflow_runs` history row into the legacy fixture, and
   added an assertion that after `openDb()` the workflow is renamed to `missing-tv-seasons`,
   no orphaned `plex` row lingers, and the member link + run history (status preserved) carry
   over under the new name. Bumped the `workflow_runs`/`workflows` row-count assertions (1→2).
   ⚠️ Gotcha hit: backticks in a SQL comment inside the JS template-literal `legacy.exec(\`…\`)`
   terminated the template literal (TS1005). Fixed by using single quotes in the comment.

4. **Docs / references**:
   - `README.md`: worked-examples list + the `**plex**` section header → `**missing-tv-seasons**`
     (noted "served at /workflows/missing-tv-seasons; was the plex workflow before T151"), and
     the movies section's "Same connectivity as **plex**" → **missing-tv-seasons**.
   - `CLAUDE.md`: project-intro bullet, the strictly-linear-workflows note, the re-scan
     idempotency variant, and the ignore-to-suppress note now name `missing-tv-seasons`
     (the `src/jobs/plex/` *folder* references were left — the folder name is unchanged).
   - `.env.example`: section header → "missing-tv-seasons workflow (Plex TV new-seasons audit)".
     `PLEX_*` env vars keep their names (they're about the Plex *server*, not the workflow id).

Confirmed no hardcoded `'plex'` workflow gating exists in `src/api` or `dashboard/app` (the
dashboard discovers workflows dynamically), so no dashboard change was needed.
