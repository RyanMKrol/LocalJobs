# T466 — Integrations page: editable per-service timeout + modest prominence bump

2026-07-08

## What I did
- `dashboard/app/lib/api.ts`: added `timeout_ms` to `Service` and `ServiceLimits`
  interfaces (backend support from T465 already exposes/accepts this field).
- `dashboard/app/integrations/page.tsx`:
  - Added a 4th editable "Timeout (ms)" column alongside Rate/min, Rate/day,
    Rate/month, using the same `toField`/`parseField`/`validField` blank-means-
    no-override idiom (extended the shared helpers, no new parallel copy needed
    since they're generic over `number | null`).
  - Wired the field through `startEdit`/`draft` state/`save` → `api.updateServiceLimits`.
  - Added an always-visible small mono line under each service's description
    showing the current effective timeout (`none` when unset/0) AND
    `rate_limit_source` inline, so a reader no longer has to open the Consumers
    popover to see where the rate-limit numbers came from. Kept scope narrow —
    no new dedicated per-service detail page/route.
  - Non-edit-mode row now also shows the current timeout as plain mono text
    (no bar — a bare number needs no visualization, per the spec).
- `dashboard/app/globals.css`: added `.service-meta` (wraps + breaks long
  `rate_limit_source` text instead of forcing overflow).
- `dashboard/scripts/_dashboard-harness.mjs`: added `timeout_ms` to the base
  `service()` fixture (15000) and a new `plex` service entry (free API,
  timeout_ms 8000, no overridden limits, with its own `rate_limit_source`) so
  the visual/mobile check harness exercises the new field. Reused the existing
  `services-edit-limits-row` FLOWS entry (already opens edit mode) rather than
  adding a redundant flow — it now naturally captures the new Timeout column.
