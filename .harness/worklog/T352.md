# T352 — fix stock-sector-lookup Finnhub ticker mismatch

## What was wrong
`stock-sector-lookup` sent Trading212's raw ticker (e.g. `AMD_US_EQ`) straight to Finnhub's
`profile2` endpoint, which only recognizes bare symbols (`AMD`). Finnhub doesn't error on an
unrecognized symbol — it returns `{}` — so the stage recorded every lookup as `'success'` with a
null industry, permanently poisoning the ledger (idempotency meant it was never retried even after
a code fix).

## What changed
- `src/jobs/stock-digest/stages/stock-sector-lookup.ts`: added `toFinnhubSymbol()` (strips trailing
  `_EQ` then a trailing 2-letter market code) and used it to build `fetchFinnhubProfile`'s request
  URL. `sectors[ticker]` / `markWorkItem` keys are unchanged (still the raw Trading212 ticker, so
  the map stays joinable with `portfolio.json`).
- Moved the `markWorkItem` call inside the `industry`/no-`industry` branch so an unresolved lookup
  is recorded `'failed'` (retryable, capped at `MAX_ATTEMPTS`) instead of `'success'`.
- `src/db/store.ts`: added `deleteNullDetailSuccessItems(jobName, field)` — a generic one-time
  recovery helper that deletes `success` rows whose `detail[field] === null`.
- `scripts/reset-stock-sector-lookup-null-successes.ts`: one-time script calling the above for
  `stock-sector-lookup`/`industry`, mirroring `backfill-service-usage.ts`'s style/header.
- `CLAUDE.md`: updated the `stock-digest` paragraph to document the ticker translation, the
  `'failed'`-not-`'success'` fix, and the new reset script.

## Verification performed
- `npx tsc --noEmit` — clean.
- `npm test` — 134/134 passing, including new unit tests:
  - `toFinnhubSymbol`: `AMD_US_EQ`→`AMD`, `MU_US_EQ`→`MU`, `VUSA_EQ`→`VUSA` (no market code),
    `BRK_B_US_EQ`→`BRK_B` (documented residual class-share limitation).
  - `fetchFinnhubProfile` requests the translated symbol (stubbed `global.fetch`).
  - Unresolved lookup (`{}` from Finnhub) → `'failed'`, not done at `maxAttempts=3`; resolved
    lookup path unchanged (`'success'`, done at `maxAttempts=1`).
  - `deleteNullDetailSuccessItems`: removes only null-field `success` rows for the target job,
    idempotent (second call returns `[]`), leaves other jobs' rows untouched.
- **Manually ran the reset script against a scratch DB** (`LOCALJOBS_DB=/tmp/t352-test.db`): seeded
  one poisoned row, ran the script — it printed `removed: AMD_US_EQ` and the row's `isWorkItemDone`
  flipped back to `false`; a second run correctly reported `0 removed — already reconciled`. Never
  run against the production `data/jobs.db`.
- **Live Finnhub verification (small sample, per CLAUDE.md's "Verify correctness" convention):**
  called `fetchFinnhubProfile` directly with the real `FINNHUB_API_KEY` for two currently-held
  tickers:
  - `AMD_US_EQ` → `Semiconductors`
  - `MU_US_EQ` → `Semiconductors`
  Confirms the ticker-translation fix actually resolves real industries where the pre-fix code
  returned nothing. Only 2 live calls made (Finnhub free tier is ~60 calls/min).
