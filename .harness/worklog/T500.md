# T500 — build 2026-07-12

Adopted response caching for the `tmdb` service.

## What changed
- `src/services/tmdb.service.ts`: added `cacheTtlMs: 79_200_000` (22h), matching the pattern
  already used by `gemini`/`google-places`.
- Added a stable `cacheKey` to every production `callService('tmdb', ...)` read site:
  - `src/workflows/plex-language-fix/lib.ts` — `` `tmdb:${path}` `` (unchanged: `resolve.ts`'s own
    `tmdb-language:<type>:<tmdbId>` cacheKey wrapping `lookupLanguageDetail` was left untouched).
  - `src/workflows/tv-recs/stages/tv-rec-merge.ts` — `` `tmdb:search:tv:${title}[:${year}]` ``.
  - `src/workflows/missing-tv-seasons/stages/season-check.ts` — `` `tmdb:/tv/${tmdbId}` `` and
    `` `tmdb:/tv/${tmdbId}/season/${n}` `` (2 call sites).
  - `src/workflows/missing-movies/stages/franchise-gaps.ts` — `` `tmdb:/movie/${id}` `` and
    `` `tmdb:/collection/${id}` `` (2 call sites).
  - `src/workflows/movies/stages/merge.ts` — `` `tmdb:search:movie:${title}[:${year}]` ``.
- Added `src/services/tmdb.service.test.ts` (mirrors the existing
  `google-places.service.test.ts` pattern): asserts `cacheTtlMs`/`paid`/`category` on the service
  definition, and proves a repeated `callService('tmdb-cache-test', fn, { cacheKey })` for the same
  path only invokes the live fetch fn once (cache hit on the second call) — a hermetic scratch-DB
  check, no live TMDB call.
