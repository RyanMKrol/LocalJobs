# T591: Skip certified workflows in the bulk reset-output-all admin action

## Summary

Implemented the certified-workflow exemption for the `POST /api/workflows/reset-output-all` bulk reset endpoint. This change mirrors the existing active-run-skip behavior but for the owner's `certified` flag (T497), which marks a workflow as "reviewed & settled" and should be protected from bulk resets.

## Changes Made

### 1. `src/api/server.ts` (handler update)
- Added a certified-check in the `POST /api/workflows/reset-output-all` handler (after the active-run check, maintaining order of precedence)
- When `wf.certified` is truthy, the workflow is skipped with `reason: 'certified'` instead of being reset
- Updated the handler comment to document the new certified-skip behavior alongside the existing active-run-skip behavior
- The console.log output and response handling already generically count/report skipped entries regardless of reason, so no changes needed there

### 2. `src/api/server.test.ts` (test suite)
- Added comprehensive test block `T591` that verifies:
  - A certified, non-running workflow is skipped with `reason: 'certified'` and its data is untouched
  - A non-certified, non-running workflow is reset as before (existing behavior)
  - An actively-running workflow is still skipped with `reason: 'active run in progress'` (existing behavior unchanged)
  - All three categories return correct counts and item deletion/preservation

### 3. `dashboard/app/admin/page.tsx` (UI description)
- Updated the Danger-zone panel descriptive text to mention that certified workflows are also skipped
- Changed from: "Workflows with an active run are skipped rather than reset."
- Changed to: "Workflows with an active run or certified flag are skipped rather than reset."

### 4. `dashboard/app/lib/api.ts` (type verification)
- Confirmed existing type definition already supports `reason: string` for the 'skipped' variant, so no type changes needed
