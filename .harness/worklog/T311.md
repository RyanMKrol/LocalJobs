# T311 — global cross-cutting log feed (backend/API only)

## What was built

- `src/db/schema.sql`: added `idx_run_logs_ts` (`run_logs(ts, id)`) and
  `idx_workflow_run_logs_ts` (`workflow_run_logs(ts, id)`) bootstrap indexes,
  right after the existing per-run/per-workflow-run indexes. Safe in bootstrap
  (not a migration) — `ts`/`id` are original columns on both tables since
  their `CREATE TABLE`, so no T098 trap.
- `src/db/store.ts`: added `GlobalLogFilter` / `GlobalLogLine` types and
  `listGlobalLogs(filter)`. Implementation queries `run_logs JOIN runs` and
  `workflow_run_logs JOIN workflow_runs` separately (skipping whichever side
  is excluded by `job`/`workflow`), each ordered `ts DESC, id DESC LIMIT
  limit`, then merges in JS by the total order `(ts DESC, source ASC ['job' <
  'workflow'], id DESC)` and slices to `limit`. Cursor is encoded as
  `` `${ts}|${source}|${id}` `` and decoded to build a "strictly after cursor"
  predicate per side, reusing the same total order — this is what makes
  same-second pagination gap/dup-free.
- `src/api/server.ts`: added `GET /api/logs` (level/job/workflow/q/windowHours
  /before/limit query params). `job`+`workflow` together → 400. Invalid level
  token → 400. `limit` is clamped into `[1, 500]` rather than rejected.
  Returns `{ logs, nextCursor }`.

## Notes / gotchas hit

- The test-runner (`scripts/run-tests.ts`) imports every `*.test.ts` file
  sequentially IN THE SAME PROCESS against ONE shared scratch DB — so an
  unscoped `listGlobalLogs()` call with a small `limit` and no `job`/`workflow`
  filter can get crowded out by other test files' rows (which use the real
  current timestamp via `datetime('now')` defaults). Fixed by seeding this
  file's fixture rows with a `ts` far in the future (`2099-...`) so they
  always sort at the very top regardless of what else has been inserted.
