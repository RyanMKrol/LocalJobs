## What I did

Widened the capture point to the branch's own `build()` call, so `input-sample`
ledger rows are the EXACT items a branch put in its Claude prompt, never a
recomputation:

1. `src/core/recommender/types.ts` — `BranchSpec.build()` now returns
   `{ prompt, sampledItems } | null` (added `BranchBuildResult<M>`) instead of
   a bare `string | null`; `null` still means "skip".
2. `src/workflows/movies/stages/branches.ts` /
   `src/workflows/tv-recs/stages/branches.ts` — every one of the 16 branch
   `build()` functions (8 per domain) now returns `sampledItems` alongside the
   prompt: the 3 `rec-random-N`/`tv-rec-random-N` branches report their
   `stratifiedSample(seed: 1000+n)` result; the 5 targeted branches
   (auteur/creator, canon, thin-genre, older-era, world) report their
   lens-filtered subset (`ownedByDirectors`/`ownedByCreators`/`ownedInGenres`/
   `ownedPreYear`/`ownedNonAnglophone`) — the SAME array reference used to
   build the prompt's owned-items block, not a re-derived one.
3. `src/core/recommender/branch.ts` — `runBranch` captures `built.sampledItems`
   right after `spec.build()` succeeds and threads it through
   `onBranchWritten(branchId, opts, sampledItems)` on every downstream path
   (Claude error, unparseable reply, success) — only the skip-no-targets path
   (build returned null) has no items to report. `collectBranchSuggestions`
   (the merge top-up loop) updated to read `.prompt` off the new return shape;
   it does NOT record sampled items (top-up calls are in-memory, no ledger
   writes). Kept core domain-agnostic — no `markWorkItem` import here.
4. `src/workflows/movies/stages/recommend.ts` /
   `src/workflows/tv-recs/stages/recommend.ts` — `recordBranchLedgerRow` now
   takes `sampledItems` and calls a new `recordInputSampleRows`, which writes
   one `work_items` row per item under the branch's OWN job name, keyed
   `<dayKey>::input::<index>` (a keyspace distinct from the `<dayKey>::<index>`
   per-suggestion output rows, so they never collide), with
   `detail: { kind: 'input-sample', id, title, year }` (`id` = the item's
   `ratingKey`).
5. `src/db/store/workItems.ts` — `stageIoLists` now splits a queried
   `outputJobNames` row set by `detail.kind === 'input-sample'`: those rows are
   excluded from `outputs` and folded into `inputs` alongside the real
   predecessor rows. No `src/api/server.ts` change needed — that endpoint
   already calls `stageIoLists([jobParam], predecessors, runId)` for a branch's
   own IO view, so the split happens transparently.
