# T203 worklog — Per-workflow "Clear output data" reset action

## What was built

Full-stack feature: a guarded "Clear output data" reset action for any workflow.

### Store (`src/db/store.ts`)
- `resetWorkflowOutput(workflowName)` — wraps everything in a single SQLite
  transaction; uses subquery-based deletes to avoid the 999-variable limit.
  Clears: `work_item_runs`, `work_items`, `run_logs`, `runs`,
  `workflow_run_logs`, `workflow_runs`. Returns counts of deleted rows.
- New `WorkflowResetResult` interface (exported).

### API (`src/api/server.ts`)
- `findWorkflowDataOut(name)` — scans `JOBS_ROOT` for `*.workflow.ts` files,
  dynamically imports each (fast — already in module cache from registry
  startup), matches by `default.name`, returns `dirname(file)/data/out` if
  it exists and is within `JOBS_ROOT`.
- `deleteDataOutContents(outDir)` — removes every entry under a `data/out`
  directory (not the directory itself). Two path-safety guards: must be within
  `JOBS_ROOT` AND must contain `/data/out/` in the path.
- `POST /api/workflows/:name/reset-output` — 404 unknown, 409 while a run
  is active (`workflowRunInProgress`), 200 with counts + `outDir` on success.
  Behind the standard loopback/token mutation guard.

### API client (`dashboard/app/lib/api.ts`)
- `resetWorkflowOutput(name)` method.

### Dashboard (`dashboard/app/workflows/[name]/page.tsx`)
- "Danger zone" section at the bottom of the workflow detail page.
- Left: description of what is cleared / what is preserved.
- Right: "Clear output data" button (`.btn.btn-danger`; disabled while a run
  is active). Confirmation via `window.confirm` before the API call.
- Shows a green success summary or red error message after the call.

### Tests (`src/api/server.test.ts`)
- `resetWorkflowOutput`: clears target workflow rows; leaves other workflow
  untouched; idempotent second call is a no-op.
- `deleteDataOutContents`: path-safety guards reject paths outside `JOBS_ROOT`
  or without `/data/out/` in the path.
- API 404 for unknown workflow.
- API 200 for a known workflow with no active run; response shape validated.

## Key design decisions

- **Subquery deletes** (not bulk IN lists) to avoid SQLite's 999-variable cap
  when a workflow has many runs.
- **Dynamic import scan** for `data/out` location — the registry knows job
  NAMES but not file paths; workflow folder names don't always match workflow
  names (e.g. `missing-tv-seasons` → `plex/`, `movie-recommendations` →
  `movies/`). Module cache makes this fast.
- **`service_usage` is never cleared** — lifetime spend meters must survive
  a reset or the quota protection becomes meaningless.
- **`data/raw` and settings preserved** — a reset wipes outputs only; raw
  input data, schedule/concurrency overrides, and service limits are untouched.
