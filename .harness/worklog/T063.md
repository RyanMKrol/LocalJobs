# T063 — Validation gate: dedicated page instead of in-page scroll

## Attempt 1 — 2026-06-22

### What was done

Created a dedicated gate detail page at
`dashboard/app/workflow-runs/[id]/gates/[producer]/[key]/page.tsx`.

The page:
- Fetches the existing `api.workflowRun(id)` data and filters to the specific gate
  by `producer` + `key` (URL-decoded)
- Shows: artifact key (mono), producer → consumer bridge, state badge, contract
  assertions (description)
- Shows a Logs section with links to producer run and consumer/violation run

**`Dag.tsx`** changes:
- `GateChips` now accepts a required `workflowRunId` prop
- Gate chip `href` changed from `#${gateAnchor(g)}` to
  `/workflow-runs/${workflowRunId}/gates/${encodeURIComponent(g.producer)}/${encodeURIComponent(g.key)}`
- `Dag` component gained an optional `workflowRunId` prop; `GateChips` only
  renders when `workflowRunId` is provided (preserving the no-chips behaviour of
  the structure-only `/workflows/[name]` view)
- Removed `gateAnchor` export (dead code — no longer used)

**`workflow-runs/[id]/page.tsx`** changes:
- `GatePanel` refactored: drops `id={gateAnchor(g)}` anchor attributes and
  replaces the Logs column with a "detail →" link to the gate's dedicated page
- `Dag` call now passes `workflowRunId={id}`
- Removed unused `gateAnchor` import

No new API endpoint was needed — the gate detail page reuses `api.workflowRun(id)`
which already returns `gates[]`.
