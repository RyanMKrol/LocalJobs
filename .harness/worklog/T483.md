# T483: Create fs pseudo-service for local-file-backed root inputKeys()

## Summary

Created a new `fs` pseudo-service (`src/services/fs.service.ts`) to govern local filesystem reads via the same `ServiceDefinition`/`callService` contract as all real external dependencies. This enables the eventual "every root's `inputKeys()` must call a named service" rule to have zero carve-outs.

## Work done

1. **Created `src/services/fs.service.ts`**
   - Modeled directly on the `plex.service.ts` reference pattern (free, local, owner-controlled, no real rate limits)
   - `paid: false`, no `ratePerMinute`/`dailyCap`/`monthlyCap`/`minIntervalMs`/`maxJitterMs`
   - `rateLimitSource` documents that this is the owner's own local disk with no external rate limit to model
   - `timeoutMs: 5_000` — a short timeout for near-instant local file reads
   - `category: 'local'` — a new category value (not forced into 'api')
   - Clear description + docstring explaining the conceptual stretch

2. **Created `src/services/fs.service.test.ts`**
   - Unit test confirming the service definition loads and has expected shape
   - Verifies name='fs', category='local', paid=false, no rate/quota fields
   - Test passes ✓

3. **Updated `dashboard/app/integrations/page.tsx`**
   - Added new `local` category to `CATEGORY_GROUPS` array
   - Positioned after `api` and before `uncategorized`
   - Service will auto-discover and appear on the Integrations page under "Local"

## Category decision

Chose to introduce a new `local` category value rather than force-fit into `api`. The controlled vocabulary was `cli-tool`, `website-scrape`, `api` — none of which accurately describe "local filesystem." The per-T305 convention (category is manifest-owned only, no dashboard edit UI, refreshed from code on sync) means adding a new category is clean and imposes no ongoing maintenance burden. A single service using it now, but it's the right semantic label for local sources.
