# T320 — projects-sync: real filesystem tool-access for project-summarize

## What changed
- New `src/jobs/projects-sync/claude-repo.ts`: `runClaudeWithRepoAccess(prompt, model, repoDir, effort?)`.
  Self-contained CLI spawn (duplicated envelope-parsing logic from `src/services/claude.ts`'s
  internal `spawnClaude` since that file is out of this task's scope and its exported `runClaude`
  contract must stay unchanged for perfumes/movies callers). Spawns with `cwd: repoDir` and
  `--add-dir <repoDir> --allowedTools Read Glob Grep` (read-only exploration tools only — no
  Bash/Write/Edit). Still routed through `callService('claude-cli', ...)`.
- `project-summarize.ts`: dropped `readRepoReadme`/README embedding; `buildSummaryPrompt` now only
  takes the catalog entry (no `readme` param) and tells Claude the repo is checked out at its cwd
  and to explore it. `runProjectSummarize` now calls `summarizeWithRepoAccess` (default
  `runClaudeWithRepoAccess`) with `(prompt, model, dest, effort)` instead of the old
  `summarize(prompt, model, effort)` / `ClaudeSummarizer` shape (which took no repoDir).
- `project-summarize.test.ts`: updated all injection points to `summarizeWithRepoAccess`; removed
  `readReadme` opt (no longer exists); added a test asserting the summarizer is called with the
  cloned repo dir, and a new `claude-repo invocation args` suite asserting `buildRepoAccessArgs`
  produces the correct `--add-dir`/`--allowedTools` args and excludes Bash/Write/Edit.
- Root `CLAUDE.md`: updated the projects-sync `project-summarize` description to describe the new
  invocation shape and explicitly note it's a different CLI arg/cwd shape from `runClaude`, while
  still sharing the same `claude-cli` service quota.

## Verification
- `npx tsc --noEmit` — clean.
- `npm test` — 91/91 passing (91 tests, 23 suites), including the new project-summarize tests and
  the perfumes/movies suites (unmodified, confirming `runClaude`'s existing callers are unaffected).
- No live paid/CLI calls made — all Claude calls in tests are stubbed via injected `summarizeWithRepoAccess`.

## Design note
`spawnClaude`'s spawn/parse logic in `src/services/claude.ts` was NOT factored into a shared helper
callable from both files, because doing so would require editing `src/services/claude.ts`, which is
outside this task's scope list. `claude-repo.ts` duplicates the (small, ~35-line) envelope-parsing
logic instead. Flagging this as a reasonable follow-up if a future task wants a single shared spawn
core parameterized by args/cwd.
