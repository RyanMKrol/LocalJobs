## What changed

`dashboard/app/workflow-runs/[id]/page.tsx`:
- Added `earlierAttemptsByStage(members)` — groups the full, uncollapsed `members`
  list by `job_name` and returns every run EXCEPT the latest one, oldest-first
  (mirrors `members`' own `(started_at, rowid)` ordering already returned by
  `listRunsForWorkflowRun`/`GET /api/workflow-runs/:id`).
- `latestByStage` output is still the primary row per stage in "Member runs"
  (unchanged for the common one-run-per-stage case).
- Added local `expandedStages` state (a `Set<string>` of job names) + a
  `toggleExpanded` helper.
- For any stage with earlier attempts, render a "▸/▾ N earlier attempt(s)" toggle
  row beneath its primary row; when expanded, render each earlier run with its
  own `StatusBadge`, relative timestamp, duration, and `logs →` link — same
  column shape as the primary row.
- No new CSS classes (task scope excludes `globals.css`) — used inline styles
  for the toggle button and indentation.

`dashboard/scripts/_dashboard-harness.mjs`:
- Extended the `/workflow-runs/1` fixture (the one backing the `workflow-run`
  PAGES entry) so `places-resolve` now has TWO runs: an earlier `failed` attempt
  (id `3`) followed by the current `success` (id `4`) — a real failed-then-
  succeeded pair, giving the new toggle real data. `places-enrich-with-a-long-
  job-name` (the original default `run()`) and `places-enrich` (still `failed`)
  are unchanged/still single-run, so the common case (no toggle) stays covered
  by the same fixture too.
- Added a `FLOWS` entry `workflow-run-earlier-attempts-expanded` that clicks the
  "earlier attempt" toggle button on `/workflow-runs/1` before the screenshot.
