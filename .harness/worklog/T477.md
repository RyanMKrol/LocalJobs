# T477 ‚Äî Cache Plex reads via callService cacheKey (3-hour TTL)

2026-07-15

## What I did

Wired a descriptive, collision-safe `cacheKey` (`plex:<request-path>`, mirroring the
`plex-language-fix/stages/resolve.ts` `tmdb-language:<type>:<tmdbId>` precedent) into every
genuinely idempotent Plex READ call site across the 6 scoped stage/lib files, so they now engage
the `plex` service's existing 3-hour cache TTL (T476) via `service_cache` (T451):

- `missing-tv-seasons/stages/snapshot.ts` ‚Äî the TV-section shows listing + flat episode listing.
- `tv-recs/stages/tv-snapshot.ts` ‚Äî the TV-section snapshot listing (new `plexFetch` seam feeds
  the default `fetchMeta`, which the pre-existing `fetchMeta` test override still fully bypasses).
- `movies/stages/snapshot.ts` ‚Äî the movie-section snapshot listing (same `plexFetch` seam shape).
- `plex-space-saver/stages/scan.ts` ‚Äî the movies/shows/episodes section listings. Deliberately
  changes this workflow's prior no-cache stance (documented in its CLAUDE.md) since the task
  explicitly scoped it in ‚Äî the workflow only runs weekly (well outside the 3h TTL) so the shrink
  guard is unaffected in normal operation; documented the one accepted trade-off (a manual re-run
  within 3h sees a cached total) in the workflow's CLAUDE.md.
- `plex-language-fix/lib.ts` ‚Äî `fetchSections`, `fetchSectionItems`, `fetchItemDetail`,
  `fetchAllLeaves` (the read-only primitives used by `discover`/`evaluate`). Each now takes an
  optional low-level `fetchPlex` override (default: the real `plexGet`) so the cache dedup can be
  unit-tested directly (these functions are normally swapped out wholesale by `discover.ts`/
  `evaluate.ts`'s own injectable seams, which would otherwise bypass `callService` entirely in
  tests). Confirmed via grep that `stages/apply.ts`'s mutating `plexPutStreams`/
  `triggerButlerBackup` calls remain untouched ‚Äî no cacheKey option was added there.
- `plex-profiles/stages/build.ts` ‚Äî `runBuild`'s movies/shows/episodes section listings AND its
  per-title `/library/metadata/<ratingKey>` detail fetch. Left `resolveInputKeys()`'s own two
  listing calls deliberately UNCACHED, per its existing doc comment ("a stale key list would
  misrepresent what the library currently contains") ‚Äî it feeds the manual run-limit's candidate
  root set (T094) and must stay live.

Each touched stage gained an injectable low-level Plex-fetch seam (`fetchPlex`/`plexFetch`,
naming matched to each file's existing convention) that stands in for the real `plexGet` while
still routing through the real `callService('plex', ...)`, so the cache mechanism itself ‚Äî not
just a bypassed mock ‚Äî is exercised in tests.

## Tests added

One new test (or test block) per touched stage/lib file proving a second call sharing a
`cacheKey` within the TTL does NOT re-invoke the underlying Plex GET mock:
- `missing-tv-seasons/stages/snapshot.test.ts` ‚Äî two `runSnapshot` calls, asserts both distinct
  paths (shows/episodes) were each fetched exactly once.
- `tv-recs/stages/tv-snapshot.test.ts` ‚Äî new describe block, two `runTvSnapshot` calls via the new
  `plexFetch` seam.
- `movies/stages/snapshot.test.ts` ‚Äî same shape as tv-recs.
- `plex-space-saver/stages/scan.test.ts` ‚Äî two `runScan` calls (with a no-op `push` override),
  asserts all three paths fetched once each.
- `plex-language-fix/lib.test.ts` ‚Äî four new tests, one per `fetch*` helper, each calling it twice
  via the new `fetchPlex` param.
- `plex-profiles/stages/build.test.ts` ‚Äî one `runBuild` test covering all 4 cached paths
  (3 listings + 1 detail fetch) across two calls.
