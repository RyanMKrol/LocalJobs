# T428 — Neuter DynamoDB write functions

## What was done
- `src/services/dynamodb.service.ts`: `dynamoPut`, `dynamoDelete`, `dynamoBatchWrite` now
  throw an explicit "disabled" error immediately on call, for every input (including
  `dynamoBatchWrite`'s previously no-op empty-array case). No code path through any of the
  three reaches `getClient().send(...)`. Signatures/exports and the AWS SDK write-command
  imports (`PutCommand`, `DeleteCommand`, `BatchWriteCommand`) are left in place so re-enabling
  later is a small, reviewable diff. `dynamoGet`/`dynamoQuery`/`dynamoScan` untouched.
- `src/services/dynamodb.service.test.ts`: updated the two `dynamoPut`/`dynamoDelete` tests and
  the two `dynamoBatchWrite` tests (over-25 and empty-array) to assert rejection with a
  `/disabled/` message instead of the old working/no-op behavior. All other tests unchanged.
- `src/services/CLAUDE.md`: added a "🚫 DynamoDB write functions are disabled by default"
  section documenting the policy, mirroring the existing broker read-only section.

## Verification observed
- `npx tsc --noEmit` — clean, no errors.
- `npm test` — 213/213 tests pass, 0 fail (includes the updated dynamodb service tests).
- Confirmed via grep before starting: no production call site references `dynamoPut`,
  `dynamoDelete`, or `dynamoBatchWrite` outside the service file and its test file, so this
  change has no downstream callers to update.

## Result
done T428
