## What changed

- `src/jobs/projects-sync/project.template.md`: reworded every section's placeholder
  guidance to explicitly ask for "a couple of substantive paragraphs" grounded in real
  repo exploration, instead of a one-line prompt. Headings/frontmatter keys unchanged.
- `src/jobs/projects-sync/stages/project-summarize.ts` (`buildSummaryPrompt`): added an
  explicit depth instruction block asking for substantive multi-paragraph prose per
  section, with an honest-brevity escape hatch for small/dormant projects, placed
  alongside (not replacing) the existing "NEVER invent facts" instruction.
- `project-summarize.test.ts`: added a unit test asserting the prompt contains the new
  depth instruction and still contains the honesty instruction.
- `CLAUDE.md`: updated the `projects-sync` description to note the expected per-section
  depth (T334) alongside the existing T319 template-contract description.

`runClaudeWithRepoAccess`, the CLI invocation shape, `--add-dir`/`--allowedTools`, the
`claude-cli` service routing, and `templateShapeViolations` are all untouched.

## Before/after evidence (no live Claude call — see reasoning below)

Inspected existing markdown files already sitting in
`src/jobs/projects-sync/data/out/*.md` (real prior output from this stage, pre-T334
prompt). They are exactly the terse pattern this task targets — e.g. `ExploreEvents.md`
and `GeneralSetupDocs.md` are 2-3 short sentences per pseudo-section, several files
don't even follow the current 7-heading template at all (they predate T319/T320
entirely, one output even opens with "Simple factual summary task, no tools needed." —
a paid Claude call visibly took a discovery shortcut). This is direct evidence of the
"a terse one-sentence-per-section response is a fully valid response Claude could
return" gap the task describes.

I did NOT trigger a fresh live `project-summarize` run to regenerate a sample, because:
- there is no cloned repo under `data/repos/` in this checkout (gitignored, not present),
  so a real run would require both a fresh `git clone` of a real GitHub repo AND a paid
  `claude-cli` service call — real spend against the monthly cap for a wording-only change.
- the change is pure prompt/template text (no new code path, no new parsing/validation
  logic) — `templateShapeViolations` and the invocation shape are unit-tested and
  unchanged, so there's no deterministic behavior here beyond "the prompt text asks for
  more" which is directly verifiable by reading the prompt output (asserted in the new
  test) rather than by judging LLM prose length, which the task's own Do section
  acknowledges isn't unit-testable.

Net effect expected on the next real run: `buildSummaryPrompt`'s new "couple of
substantive paragraphs... not a single generic sentence" instruction, combined with
`runClaudeWithRepoAccess`'s existing real read-only repo access (T320), should make a
1-2 sentence per-section reply (like the existing on-disk samples) noticeably less
likely, since the prompt now explicitly names that failure mode and asks Claude to draw
on specific files/patterns it read.
