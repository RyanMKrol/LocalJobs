# T506: Cache the GitHub repo-catalog API READS for 22 hours

## Completed

Successfully implemented caching for the GitHub repository listing API reads in the projects-sync workflow.

## Implementation

1. **GitHub service definition** (`src/services/github.service.ts`):
   - Added `cacheTtlMs: 79_200_000` (22 hours) to the `ServiceDefinition`
   - This enables opt-in caching for `api`-category services via the existing mechanism (T451/T476)

2. **Repository listing reads** (`src/workflows/projects-sync/stages/github-sync.ts`):
   - Updated the `ServiceCaller` type to accept an optional `CallServiceOpts` parameter
   - Added `cacheKey` to each paginated repo-listing API call: `` `github:repos:${user}:page:${page}` ``
   - The cacheKey is stable per endpoint + page, enabling 22-hour caching of repeated page fetches
   - Added import of `CallServiceOpts` type for proper typing

3. **Git operations** (`src/workflows/projects-sync/stages/project-summarize.ts`):
   - Left unchanged — the `cloneOrPullRepo` function continues to call `callService('github', ...)` WITHOUT a cacheKey
   - Git clone/fetch/reset are side-effecting operations that must never be cached

4. **Test coverage** (`src/workflows/projects-sync/stages/github-sync.test.ts`):
   - Added new test: "passes a stable cacheKey to callServiceFn keyed by endpoint + page for read caching"
   - Verifies that `fetchAllRepos` correctly passes the expected cacheKey format for each page
   - Test confirms caching pattern works correctly within the existing test infrastructure
