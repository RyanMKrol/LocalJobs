# T602 — merge stage writes one work_items row per final recommendation

2026-07-16

## What I found
Re-verified `src/core/recommender/merge.ts` (shared by movies' `rec-merge` and tv-recs'
`tv-rec-merge`, `RecommenderDomain.mergeStageName` carries the exact job name for each) against
the spec's premise: there was **no existing `markWorkItem` call anywhere in `runMerge`** — the
spec's "convert bulk marker to per-item" framing didn't match the current file; this was an
add-from-nothing, as the spec's own fallback note anticipated.

## What I did
- Added `markWorkItem` to the imports in `merge.ts`.
- After `balanced` is finalized (right after `recommendations.json` is written), loop over
  `balanced` and call `markWorkItem(domain.mergeStageName, recKey(r.tmdbId), 'success', { detail:
  { title, year, genre, reason, lens, tmdbRating } })` once per final recommendation — using the
  merge stage's OWN job name (`'rec-merge'` / `'tv-rec-merge'`), not `domain.recsJob` (that's the
  separate notified/ignored ledger keyspace keyed the same way but owned by the notify stage).
  `recKey` (from `pure.ts`) is reused so the key format matches what `notify.ts` already expects
  for a tmdbId-keyed ledger.
- `markWorkItem` is already an upsert keyed by `(job_name, item_key)`, so a same-day re-run with
  the same balanced set naturally updates the same K rows instead of duplicating them — no extra
  idempotency logic needed.

## Tests
Extended `merge.test.ts` with a new case: a fixture with 2 branch suggestions that both clear the
quality bar (distinct genres, so genre-cap doesn't interfere) → asserts `workItemCounts('ledgertest-merge').success === 2`
after one run, asserts the row keyed by `recKey(111)` carries the exact recommendation fields in
`detail`, then runs merge a second time on the same fixture and asserts the count is STILL 2 (no
duplication — the upsert idempotency required by "Done when").
