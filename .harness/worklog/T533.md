# T533: Fix LIKE wildcard injection in listGlobalLogs

## Overview
Fixed a security issue in `src/db/store/runs.ts` where the `listGlobalLogs` function was treating SQL LIKE wildcard characters (`%` and `_`) in the user's search string (`q` parameter) as wildcards instead of literal characters. This could allow patterns like `q=%` to match everything or `q=a_b` to match any character in place of `_`.

## Implementation

### Changes made:

1. **Added `escapeLike()` helper function** in `src/db/store/runs.ts`:
   - Escapes backslashes: `\` → `\\`
   - Escapes percent signs: `%` → `\%`
   - Escapes underscores: `_` → `\_`
   - This ensures these characters are treated as literals, not LIKE wildcards

2. **Updated both LIKE clauses**:
   - Job logs LIKE clause (line ~196): Now uses escaped query and `ESCAPE '\'` directive
   - Workflow logs LIKE clause (line ~247): Now uses escaped query and `ESCAPE '\'` directive

3. **Added test case** in `src/db/logs.test.ts`:
   - Tests that `q='100%'` finds only the literal `%` row, not all rows (which would happen if `%` was treated as a wildcard)
   - Tests that `q='file_name'` finds only the literal `_` row, not any single character in that position
   - Tests that `q='%'` does not match everything (full result set)
   - Verifies case-insensitive behavior is preserved

## Details

The fix handles the LIKE escape correctly:
- `ESCAPE '\'` tells SQLite that backslash is the escape character
- All special LIKE characters in the user input are escaped before being wrapped in `%…%`
- The parameterized query prevents SQL injection (already present, but now also prevents LIKE-injection via wildcards)
