# T363 — fix light-mode flash-to-dark hydration mismatch in `useMode()`

## Root cause (as given in the spec, reproduced live)

`useMode()` in `dashboard/app/ui.tsx` computed a DIFFERENT value in its `useState` lazy
initializer on the server vs. the client:

- Server (`typeof window === 'undefined'`): always `'dark'`.
- Client's first render during hydration: read the REAL persisted `localjobs.mode` from
  `localStorage` (e.g. `'light'`).

When the persisted mode differs from `'dark'`, this produces a genuine React hydration
content mismatch several component layers below `<html>` (in the header's mode-toggle
button's `aria-label`/`title`/emoji), which is NOT covered by `<html>`'s
`suppressHydrationWarning` (that only protects the exact element it's set on). React has
no boundary to localize the mismatch, so it discards and regenerates the whole document
on the client — wiping the pre-paint script's already-correct `data-mode="light"`
attribute in the process (it briefly becomes unset → falls back to `globals.css`'s dark
`:root` palette) before a later effect corrects it back. This is the "flash to dark, then
settle to light" the owner reported, and it reproduced on every full page load/navigation
where the persisted mode differs from `'dark'`.

## Fix

1. `useState<ModeId>` initializer is now a plain, unconditional literal — `useState<ModeId>('dark')`
   — with no `typeof window` branch and no `localStorage` read during initial render. This is
   IDENTICAL on the server and the client's first render, so there is nothing for React to flag
   as mismatched: no hydration error, no full-tree regeneration, and the pre-paint script's
   `data-mode` attribute is never touched during the initial render/hydration pass.
2. A new mount-only `useEffect` (`[]` deps) reads the real persisted `localjobs.mode` from
   `localStorage` and calls `setMode(real)` if it differs from the hardcoded default. This is a
   normal, contained client-only re-render (post-commit), not a hydration recovery.
3. **Additional fix beyond the spec's literal Do-list, found during verification:** the existing
   "apply mode to `data-mode`" effect (`useEffect(() => { ... }, [mode])`) unconditionally wrote
   `document.documentElement.setAttribute('data-mode', mode)` using whatever `mode` was on ANY
   render — including the very FIRST effect run, where `mode` is still the placeholder `'dark'`
   before the mount-reconcile effect corrects it. Since effects run in declaration order within
   the same commit, this meant the first commit still wrote `data-mode="dark"` (via this second
   effect) immediately after the reconcile effect's `setMode` call was merely *scheduled* — a
   self-inflicted flash of the exact same kind, just moved to be client-only/React-caused rather
   than an SSR/hydration issue. Confirmed this empirically with the Playwright repro below: without
   this additional guard, `data-mode` mutations during a `localStorage.mode === 'light'` navigation
   were `["dark", "light"]` — a real, if brief, wrong-palette flash. Fixed by skipping the
   attribute-application effect's FIRST invocation via a `useRef` guard (`skippedFirstApply`), since
   the pre-paint script has ALREADY applied the correct value before hydration ran at all — React
   only needs to apply the attribute on genuine subsequent mode changes (mount-reconcile's
   corrective `setMode`, `cycle()`, or an OS `prefers-color-scheme` change), never on the initial
   mount.
4. `resolveMode()`, the pre-paint script in `layout.tsx`, `ModeId`'s shape, and the toggle button's
   click-to-cycle interaction are all UNCHANGED. Only `useMode()`'s initial-render/mount logic in
   `dashboard/app/ui.tsx` was touched.

**Accepted trade-off (per spec point 4):** the header icon may show the hardcoded default (🌙,
dark) for one contained frame after mount before the mount-reconcile effect corrects it to the
real persisted icon. This is the industry-standard hydration-safe pattern (same approach
`next-themes` uses) and is a much smaller, contained artifact (one header icon) than the previous
full-page background flash — and, per point 3 above, no longer involves any `data-mode`
attribute write during that window either, so there is no flash of the wrong palette at all, only
(at most) a one-frame stale icon glyph.

## Scripted repro (primary proof)

Wrote an ad hoc Playwright script (`dashboard/scripts/_t363-repro-tmp.mjs`, deleted after use —
not part of the permanent suite) that:
1. Reused the existing hermetic harness (`_dashboard-harness.mjs`'s `startDashboard`/`waitForServer`/
   `routeApi`) to run a real `next start` on port 4799 with synthetic `/api/**` fixtures — no daemon,
   no real SQLite, no paid calls.
2. Installed a `page.addInitScript` that attaches a `MutationObserver` on
   `document.documentElement` watching the `data-mode` attribute (guarded to wait for
   `DOMContentLoaded` if `document.documentElement` isn't yet available), plus a `page.on('pageerror', ...)`
   listener — both installed before any page script runs, on every navigation.
3. Loaded `/`, set `localStorage.setItem('localjobs.mode', 'light')`, then did a REAL full-page
   `page.goto('/workflows')` (matching how the dashboard's plain `<a>` nav links behave).
4. Asserted (a) no hydration-mismatch `pageerror`, and (b) `data-mode` never observed as `null` or
   `"dark"` during the navigation.

**Pre-fix run (via `git stash -- dashboard/app/ui.tsx`, rebuild, rerun, then `git stash pop`):**
```

```
React minified error #418 is exactly "Hydration failed because the initial UI does not match what
was rendered on the server" — confirming this is a genuine hydration mismatch, not just a cosmetic
lag, and that the `data-mode` attribute really was transiently wiped to `null` (falling back to the
dark `:root` palette) during the client-side recovery. The fix eliminates both symptoms.

Also caught and fixed one intermediate regression during development (see fix step 3 above): an
early version of the fix (spec's literal Do-list only) still produced `["dark", "light"]` mutations
— a self-inflicted flash from the unguarded attribute-apply effect — before the `skippedFirstApply`
ref guard was added.
