# T420 ‚Äî perfumes stages must fail the run on failed>0 (T394/T416 convention)

## What changed

All four `perfumes` `.job.ts` wrappers (`find-url.job.ts`, `fetch.job.ts`, `parse.job.ts`,
`build.job.ts`) previously called their stage's `run*` function and discarded the returned
`StageResult`, so a run in which every item failed still recorded the job/run as `'success'`
(the confirmed live incident: `perfumes-build` cycle 1 logged `1/1 ¬∑ 0 ok, 1 failed` but only
cycle 2's coincidental retry masked it).

Each wrapper now captures the `StageResult` and throws a summarizing `Error` when
`result.failed > 0`, via a small exported `assertNoFailures(result)` helper per file (kept
local per-file per the spec's exact throw messages, but exported so the decision is directly
unit-testable without needing to force a live Claude/browser failure). A `rateLimited: true`
run with `failed === 0` is a soft defer and does NOT throw; a run with genuine failures throws
regardless of `rateLimited`.

No changes to the stage `.ts` implementations (`find-url.ts`, `fetch.ts`, `parse.ts`, `build.ts`)
‚Äî each already computed and returned the correct `StageResult`; the bug was purely that the
wrapper never inspected it.

## Tests

Added `src/workflows/perfumes/stages/job-throw.test.ts` ‚Äî imports each job's exported
`assertNoFailures` and exercises it against synthetic `StageResult` fixtures (no live network /
Claude CLI / browser calls, fully hermetic):
- `failed > 0` (mixed ok/failed, and all-failed) ‚Üí throws with the expected tally in the message.
- `failed === 0` (with pending work remaining) ‚Üí does not throw.
- `rateLimited: true` + `failed === 0` ‚Üí soft defer, does not throw.
- `rateLimited: true` + `failed > 0` ‚Üí still throws (genuine failures before/around a rate-limit
  stop are not masked by the rate limit).

## Verification

- `npx tsc --noEmit` ‚Äî clean.
- `npm test` ‚Äî 197/197 tests pass, including the new `job-throw.test.ts` suite.
- Manual spot-check of the motivating scenario: `assertNoFailures({ ok: 0, failed: 1, pending: 0,
  rateLimited: false })` for the `perfumes-build` case throws
  `"1/1 profile(s) failed to build this run ‚Äî see logs above"` ‚Äî exactly the shape needed so the
  executor now records the run `'failed'` instead of `'success'` for the observed
  `1/1 ¬∑ 0 ok, 1 failed` cycle-1 log line.
