# T386 — StageIoPanel as the default for every workflow

## What changed
- `dashboard/app/workflow-runs/[id]/page.tsx`: removed the `run?.workflow_name === 'stock-digest'`
  gate. `<StageIoPanel runId={id} members={workflow.jobs} />` now renders unconditionally whenever
  `workflow` has loaded. The old `IoPanel` component/`workItemIoRows`/`GET .../io` endpoint are left
  in place and unused by this page (deletion is T389's job).
- `dashboard/scripts/_dashboard-harness.mjs`:
  - Added `/api/workflow-runs/1/stage-io` fixtures (`placesStageIo` / `placesStageIoOverall`) —
    proves StageIoPanel against a STRICTLY-LINEAR 3-stage chain (places-resolve -> places-enrich ->
    places-enrich-with-llm, no fan-out/fan-in), reusing the existing `workflow-run` PAGES entry at
    `/workflow-runs/1`.
  - Added a new SINGLE-STAGE workflow fixture set (`claude-warmer`, one member `claude-warm`, no
    `dependsOn`): `/api/workflows/claude-warmer`, `/api/workflow-runs/claude-warmer-run`, and its
    `/stage-io` (+`overall=true`) responses. Added as a FLOWS entry (`stage-io-single-stage-overall`)
    rather than a PAGES baseline — see below for why.
  - Updated two now-stale FLOWS entries that assumed the old `IoPanel` still rendered at
    `/workflow-runs/1`: `workflow-run-output-table` now clicks StageIoPanel's
    `.stage-io-item-link` (was `.out-meta-link`); removed `io-panel-sorted-by-state` (StageIoPanel's
    list-based columns have no sortable `<table>` — that feature no longer exists on this page).
    `io-panel-job-scoped` needed no selector change (StageIoPanel reuses the same
    `.io-job-filter-bar`/`.io-job-filter-chip` classes) — only its comment was updated.

## A pre-existing DagFlow bug, kept out of scope
Adding `claude-warmer-run` (single DAG node) to `PAGES` made `mobile-check.mjs` fail: the single
`.rf-dag-node` at 402px viewport spills ~234px past the boundary ("claude-warmSucceeded"). This is a
pre-existing `DagFlow.tsx` layout issue for single-node graphs, not something introduced by this
task and not something in this task's scope (`dashboard/app/components/DagFlow.tsx` is outside the
allowed file list). Rather than either violating scope or leaving mobile-check red, I moved the
single-stage run page out of `PAGES` (which mobile-check iterates) and into `FLOWS` (which only
`visual-check.mjs` iterates, at desktop viewport) — so the single-stage shape still gets full visual
proof without tripping the phone-width regression that's really DagFlow's, not StageIoPanel's. Worth
a follow-up backlog task against `DagFlow.tsx` for single-node mobile layout.
