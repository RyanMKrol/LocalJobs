## Overview

The movie/tv recommendation top-up loop (T162) already avoids re-suggesting titles within the same
run (`considered`) and titles previously actually notified to the owner (`historyFile`, fed into
`alreadySuggested`) — but it does NOT feed the ledger's owner-**ignored** recommendations into that
same exclude list. This was empirically verified (2026-07-04, against real code + the real DB) as a
genuine, if currently-latent, gap: fix the exclude list so ignored recs are treated like historied
ones for both movies and tv-recs.

## Do

**Grounding — what was verified, and the confirmed gap (2026-07-04):**

- Within one run, `considered` (`src/jobs/movies/stages/merge.ts:298-302,334` /
  `src/jobs/tv-recs/stages/tv-rec-merge.ts` mirrors it) is passed as `exclude` into every re-prompt —
  correctly implemented already, not touched by this task.
- Across runs, `historyFile` (`moviesConfig.recsHistoryOut` / `tvRecsConfig.recsHistoryOut`) feeds
  `alreadySuggested = allHistoryTitles(historyFile, ...)` into every branch's prompt context
  (`merge.ts:222-233`, `tv-rec-merge.ts:220-223`, and — for the FIRST/initial branch pass, not just
  top-up — `recommend.ts:112,114` in both `movies/stages/` and `tv-recs/stages/`). `appendHistory` is
  called ONLY when a recommendation is newly `markWorkItem`'d `success` at notify time
  (`movies/stages/notify.ts:161`, `tv-recs/stages/tv-recs-notify.ts:103`) — i.e. history is a log of
  what was actually SENT to the owner, not every candidate ever surfaced.
- `verifyInto` (`merge.ts:175-179`, `tv-rec-merge.ts:178-182`) independently checks
  `isWorkItemDone(RECS_JOB, recKey(tmdbId), 1)` and drops anything already `success`/`ignored` in the
  `movie-recs`/`tv-recs` ledger — so an ignored item genuinely CANNOT slip into final output even if
  Claude re-suggests it. This part is correct; the issue is purely wasted PROMPT budget, not a
  correctness/output bug.
- **The confirmed gap:** `GET /api/movie-recs` / `GET /api/tv-recs` (`src/api/server.ts:1236-1257`,
  `:1276-1297`) read directly from the CURRENT `recommendations.json` — the merge stage's raw output —
  and `POST /api/movie-recs/:tmdbId/ignore` / the tv-recs equivalent (`:1262-1268`, `:1302+`) accept
  ANY tmdbId present there, with no requirement that the item has already been through the notify
  stage. `ignoreSurfacedItem` (`src/db/store.ts:944-951`) also stores **no `detail` at all** on a fresh
  ignore (its `INSERT` lists no `detail` column). So: if the owner ignores a recommendation BEFORE (or
  in the same run as, before) notify has processed it — e.g. notify hasn't run yet this cycle, or
  crashes between merge and notify — that title is (a) never in `historyFile` (`appendHistory` never
  ran for it) and (b) unrecoverable from the ledger row itself (no title stored). Net effect: it's
  correctly excluded from final OUTPUT (`verifyInto`'s `isWorkItemDone` check still catches it) but
  NOT excluded from the PROMPT — the top-up loop can waste a whole re-prompt round asking Claude to
  suggest it again before dropping it post-hoc, after already spending the Claude call.
- **One on-disk data point for the common case working correctly today:** the real `tv-recs` ledger
  has exactly one `ignored` row, tmdbId `6357` (`detail.name`: `"The Twilight Zone (older-era)"`), and
  the real `recs-history.json` DOES contain `{"title": "The Twilight Zone", "year": 1959}` — because
  that item was ignored AFTER already being notified (the common path). The `movie-recs` ledger
  currently has zero ignored rows (68 `success`, 0 `ignored`), so the pre-notify-ignore path has not
  been observed on disk — it's a latent gap the code structurally allows, not something caught in the
  act. Fix it anyway since the API demonstrably permits it and there's no reason to leave the window
  open.

**The fix — 8 files, all mirrored 1:1 between movies and tv-recs:**

1. **`src/db/store.ts`** — give `ignoreSurfacedItem(jobName, itemKey)` an optional third `detail?:
   unknown` param. When provided, JSON-serialize it into the row's `detail` column on both INSERT and
   UPDATE; when omitted, PRESERVE whatever `detail` the row already had rather than clobbering it to
   `NULL` (a row already `success` with a real `detail` — e.g. the existing T145/T209
   ignore-an-already-recommended-item path — must not lose its `name`/`markdown` on ignore). Add a
   companion read helper, e.g. `ignoredWorkItemDetails(jobName: string): { itemKey: string; detail:
   unknown }[]`, mirroring the existing `ignoredItems()` (`store.ts:991-996`) detail-parsing pattern,
   scoped to one job name. Leave the bulk `ignoreSurfacedItems` (`store.ts:959-974`, used only by the
   franchise-gaps bulk-dismiss UI) and `ignoredItemKeys` untouched — franchise gaps are TMDB-
   collection-derived, not Claude-suggested, so this whole top-up-prompt concern doesn't apply to
   them; do not fold gaps into this task.
2. **`src/api/server.ts`** — in `POST /api/movie-recs/:tmdbId/ignore` (~:1262-1268) and the tv-recs
   equivalent (~:1302+), before calling `ignoreSurfacedItem`, look up the matching entry for that
   `tmdbId` in the currently-loaded `recommendations.json` (the same file the sibling `GET` handler
   already reads) and, if found, pass `{ title, year }` as the new `detail` arg so a fresh (pre-notify)
   ignore captures a recoverable title. If the id isn't in the current file, omit `detail` — no title
   recoverable for that edge case is an acceptable, documented limitation, not a regression.
3. **`src/jobs/movies/stages/notify.ts`** (~:157) and **`src/jobs/tv-recs/stages/tv-recs-notify.ts`**
   (~:95) — when marking a newly-notified rec `success`, add explicit `title`/`year` fields alongside
   the existing `name`/`markdown` in the `detail` object passed to `markWorkItem`, so a LATER ignore of
   an already-notified item (the common case — see the real `6357` example above) also carries a
   clean, parseable title/year, not just the combined `"Title (lens)"` `name` string. Additive only —
   don't change `name`'s existing format; anything already reading `detail.name` must keep working.
4. **`src/jobs/movies/stages/recommend.ts`** and **`src/jobs/tv-recs/stages/recommend.ts`** — add a
   function (e.g. `ignoredSuggestionTitles(jobName: string): string[]`, formatted identically to
   `allHistoryTitles`'s `"Title (Year)"` / bare `"Title"` output) that reads the ledger's `ignored`
   rows via the new store helper and extracts `title`/`year` from `detail` when present, skipping rows
   with no usable title (e.g. legacy ignores that predate this fix). Do NOT attempt to parse the old
   `"Title (lens)"` `name` format — it isn't reliably splittable (lens values aren't years).
5. **`src/jobs/movies/stages/merge.ts`** and **`src/jobs/tv-recs/stages/tv-rec-merge.ts`**, PLUS the
   initial branch pass in `recommend.ts` itself (`runBranch`, movies `:112,114` / tv `~:111,112`) —
   wherever `alreadySuggested` is built from `allHistoryTitles(...)`, also concatenate +
   dedupe (`Set`-based) with `ignoredSuggestionTitles(RECS_JOB)` so BOTH the first branch pass and
   every top-up round get the combined exclude list, not just top-up.
6. Add/update unit tests: `recommend.test.ts` (both workflows) covering the new ignored-title function
   directly (a row with usable `detail.title`/`year` → included; a legacy row with no usable title, or
   only the old `name` format → skipped, not crashed) and `merge.test.ts` /
   `tv-rec-merge.test.ts` asserting that an item marked `ignored` (via the new detail-carrying path)
   WITHOUT ever being appended to the history file still shows up in the exclude/`alreadySuggested`
   list passed to the injected `topUp`/branch function on a top-up round — this is the regression test
   for the exact gap this task closes.
7. This is ordinary production job/API/db code, not `.harness/**` — `gate: null` is correct.

## Done when

- `ignoreSurfacedItem` accepts an optional `detail` arg and never clobbers an existing non-null
  `detail` when the arg is omitted — covered by a unit test (fresh ignore with detail → detail stored;
  ignoring an already-`success` row with existing detail, calling with no detail arg → prior detail
  preserved unchanged).
- A fresh ignore via `POST /api/movie-recs/:tmdbId/ignore` (and the tv-recs equivalent) for a tmdbId
  present in the current `recommendations.json` results in a ledger row whose `detail` carries a
  recoverable `title`/`year`.
- Both the initial branch pass (`recommend.ts`'s `runBranch`) and the top-up loop
  (`merge.ts`/`tv-rec-merge.ts`) build their exclude/`alreadySuggested` list from history titles AND
  currently-ignored ledger titles, for BOTH movies and tv-recs.
- A unit test demonstrates the originally-identified gap is closed: mark an item `ignored` in the
  `movie-recs`/`tv-recs` ledger with a title-carrying `detail` but WITHOUT ever appending it to the
  history file, run the merge stage's top-up path, and assert that title appears in the
  `exclude`/`alreadySuggested` list passed to the (injected, test) branch/topUp function.
- `npx tsc --noEmit` and `npm test` both pass.
- No `.env.example`/config change is required by this fix (no new env var). If one is introduced
  anyway, document it there — but the fix as scoped shouldn't need one.
