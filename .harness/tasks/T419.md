## Overview

`movie-recommendations`'s notify stage (`src/jobs/movies/stages/notify.ts`, `runNotify`) sends ONE
combined digest push for newly-detected franchise gaps + recommendations, then marks every one of
those items `'success'` in their respective "already-notified" ledgers (`movie-gaps-notify` /
`movie-recs`) so they're never re-announced. The push's own result IS checked today (`res.ok ? ... :
...`) and a failure is logged at `error` level — but that's where it stops: the code falls through
unconditionally and marks every new gap/rec notified (and appends recs to the history file) even
when the push failed. That means a failed digest permanently poisons the ledger: the owner never
actually received the notification, but the item is recorded as delivered and will never be retried
or resurfaced. This is exactly the class of bug the T394 convention targets — a genuine, already-
detected per-run failure that is logged but not propagated to fail the job's own run.

## Do

- In `src/jobs/movies/stages/notify.ts`'s `runNotify`, after calling `pushFn(...)`, if `res.ok` is
  false: do NOT mark any of `newGaps`/`newRecs` in their ledgers, do NOT call `appendHistory`, and
  `throw` a summarizing `Error` (e.g. naming the push error, how many gaps/recs would have been
  marked, and that the run is failing so the next run retries) instead of falling through.
- Only mark items notified / append history when `res.ok` is true, exactly as today.
- Keep the existing `ctx.log` lines (info/error) — the thrown error is in addition to, not instead
  of, the log narration.
- Do NOT change the "nothing new to notify" early-return path (`newGaps.length === 0 && newRecs.length
  === 0`) — that's a legitimate no-op success, not a failure.
- Do NOT touch `rec-merge` (`src/jobs/movies/stages/merge.ts`) — its per-suggestion/per-branch
  try/catch swallowing is a deliberate, documented resilience design (see its own doc comment: "it
  always writes a (possibly short) list and succeeds, so the notify stage runs") and is explicitly
  out of scope for this task.

## Done when

- `runNotify` throws when the digest push fails (`res.ok === false`), and in that case no ledger rows
  are written for `newGaps`/`newRecs` and the history file is not appended.
- A new/updated unit test in `src/jobs/movies/stages/notify.test.ts` (or wherever the existing notify
  tests live) injects a `push` stub that returns `{ ok: false, error: '...' }` and asserts: `runNotify`
  rejects/throws, and `markWorkItem`/ledger state for the affected gap(s)/rec(s) is unchanged (still
  not-done, i.e. would be picked up as "new" again on a subsequent call) and the history file is not
  written to.
- Existing tests covering the successful-push path still pass unchanged (ledger marked, history
  appended, no throw).
- `npx tsc --noEmit` and `npm test` are green.
