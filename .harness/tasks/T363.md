## Overview

With Light mode explicitly selected, every full dashboard page load/navigation briefly flashes to
the dark palette before settling back to Light. Root-caused and reproduced live with Playwright
against the real dashboard: `useMode()` in `dashboard/app/ui.tsx` computes a **different initial
value on the server than on the client** (its `useState` lazy initializer hardcodes `'dark'` when
`typeof window === 'undefined'`, but reads the real `localjobs.mode` value from `localStorage` on
the client) — which is a genuine React SSR/hydration mismatch, not just a stale first frame. This
task fixes the hydration mismatch at its source so React never has a reason to discard/regenerate
the page's DOM, which is what was wiping the pre-paint script's correct `data-mode` attribute.

## Do

**Confirmed root cause (do not re-investigate — reproduce to confirm, then fix):**

`useMode()` (`dashboard/app/ui.tsx`, ~lines 408-413) is:
```ts
const [mode, setMode] = useState<ModeId>(() => {
  if (typeof window === 'undefined') return 'dark';
  const stored = window.localStorage.getItem('localjobs.mode') as ModeId | null;
  return stored === 'dark' || stored === 'light' || stored === 'system' ? stored : 'system';
});
```
On the **server** (SSR, `typeof window === 'undefined'`) this always returns the hardcoded `'dark'`.
On the **client's first render during hydration**, the SAME initializer function runs again but now
has `window` available, so it reads the REAL persisted mode (e.g. `'light'`) straight from
`localStorage`. When the persisted mode isn't `'dark'`, the header's mode-toggle button
(`ThemeControls`, rendered from this hook's `mode`) renders DIFFERENT `aria-label`/`title`/emoji
text on the client than what the server rendered — a genuine content mismatch, several component
layers below `<html>` (`html > body > header > div > ThemeControls > div > button`), **well outside
what `<html>`'s `suppressHydrationWarning` prop protects** (React's `suppressHydrationWarning` only
covers the exact element it's set on, "one level deep" per React's own docs — it does nothing for a
mismatch this many levels down).

**Empirically confirmed via a live Playwright repro against a `next dev` instance of this dashboard**
(explicitly set `localStorage.setItem('localjobs.mode','light')`, then did a real full-page
`page.goto` to another route — a genuine full reload, since nav links are still plain `<a>` tags,
confirmed unchanged in `layout.tsx`): the browser throws **React hydration error "Hydration failed
because the server rendered text didn't match the client... this tree will be regenerated on the
client"**, whose printed component stack starts at `<html>` and ends at the mode button's
`aria-label`/emoji — i.e. the mismatch is real and React has no boundary to localize the
regeneration to just the button, so it discards and rebuilds the tree from `<html>` down on the
client. A `MutationObserver` on `<html>`'s `data-mode` attribute captured the DOM's actual
behavior during that regeneration: the attribute (already correctly set to `"light"` by the
pre-paint script BEFORE hydration) briefly becomes **`null`** (falls back to `globals.css`'s base
`:root` rule, which is the DARK palette — see the "THEME + FONT + ACCENTS (T308)" section: `:root`
is the dark palette and `[data-mode="light"]` is the override), then a separate, later mutation sets
it back to `"light"` once `useMode()`'s own `useEffect` re-applies it post-recovery. This is the
exact "flash to dark, then settle to light" the owner reported — and it happens on **every full page
load** where the persisted mode differs from `'dark'` (reproduced on the very first load of `/` too,
not just on the navigation to a second page) — it's just far more visually jarring when navigating
from an already-light page to another, versus a fresh cold browser-tab open.

**Why T343 didn't fix this (context, don't re-litigate):** T343 (done) scoped itself to the header
icon's `useState` initializer to fix a *cosmetic, one-frame-stale icon* (hardcoded `'dark'` default
rendering the wrong icon for one frame before its `useEffect` corrected it) — it deliberately did
NOT touch `resolveMode()` or the pre-paint script. T343's chosen fix — a **lazy initializer that
reads the CURRENT client-only source of truth (`document.documentElement`'s `data-mode` attribute at
the time)** — looked correct in isolation but reintroduced the same class of SSR-vs-client
divergence it was trying to paper over, just sourced from the DOM instead of a plain literal. T344
(later, unrelated: reintroducing System mode) rewrote `useMode()` again, keeping the same
SSR-diverges-from-client shape (now sourced from `localStorage` instead of the DOM attribute). Both
versions share the identical structural bug this task fixes: **the `useState` initializer must
return the IDENTICAL value on the server and on the client's first render, or React will treat it as
a genuine hydration mismatch** — a `useEffect` cannot rescue this after the fact for the FIRST
render's correctness, and (as proven above) an unprotected mismatch this many levels below `<html>`
triggers full-tree hydration recovery, wiping the pre-paint script's DOM work in the process.

**The fix — eliminate the SSR/client divergence in `useMode()`'s initial state, in
`dashboard/app/ui.tsx`:**

1. Change the `useState<ModeId>` initializer back to a **plain, unconditional literal** — the SAME
   value on the server and on the client's very first render, e.g. `useState<ModeId>('dark')` (no
   `typeof window`/`document` branch, no `localStorage`/DOM read inside the initializer at all). This
   guarantees zero divergence between what the server renders and what the client renders on its
   first pass, so React has nothing to flag as mismatched — no hydration error, no full-tree
   regeneration, and (critically) no reason to ever touch the `data-mode` attribute the pre-paint
   script already set correctly before hydration.
2. Keep (or extend, if needed) the existing `useEffect` as the ONLY place that reads the real
   persisted mode and reconciles both the React state (`setMode(...)`) and the DOM attribute — this
   already exists (the `mode !== 'system'` branch / the `matchMedia` branch), so it may need a small
   adjustment: on mount it should read `localStorage.getItem('localjobs.mode')` (mirroring the
   removed initializer logic) and `setMode` to the real value if it differs from the hardcoded
   default, THEN proceed with the existing attribute-setting / OS-listener logic. This makes the
   correction a normal, CONTAINED client-only re-render (no hydration recovery involved, since it
   runs after commit, same as any ordinary state update) — not a full-document rebuild.
3. Do **not** change `resolveMode()`, the pre-paint script in `layout.tsx`, `ModeId`'s shape, or the
   toggle button's click-to-cycle interaction — none of those are implicated; this is purely about
   making `useMode()`'s FIRST render SSR-safe.
4. Accept, and note explicitly in the worklog, the resulting trade-off: the header icon may show the
   hardcoded default (🌙) for one contained frame after mount before the effect corrects it to the
   real persisted icon — this is the industry-standard "hydration-safe" pattern for state that
   depends on `localStorage`/`document` (the same approach libraries like `next-themes` use). It is a
   MUCH smaller, more contained visual artifact (one small header icon) than the current bug (a
   full-page background-color flash caused by React discarding and rebuilding the whole document),
   and does not involve any DOM/attribute wipe on `<html>` at all — the `data-mode` attribute stays
   exactly as the pre-paint script set it, untouched, for the entire page lifetime after that.
5. If any JSDoc/comment above `useMode()` in `ui.tsx` describes the old (SSR-diverging) initializer
   behavior, update it to describe the new hydration-safe shape.
