## Do

Make a workflow's push notifications **user-editable and toggleable OFF per workflow**, exactly like
the existing **`enabled`** run-toggle, **`schedule`** (T135), and **`maxConcurrency`** (T169)
properties — user-owned, override-flagged, and code-reconciled (a later code-sync preserves the
user's value instead of reverting to the manifest default). **Default: notifications ON** for both
existing and newly-added workflows, so current behavior is unchanged unless the owner explicitly
opts a noisy workflow out. This is a **binary on/off switch only** — do NOT build a finer-grained
"only notify on failure" mode; that is a deliberately separate, deferred idea.

Mirror the T169 `maxConcurrency` mechanism end-to-end (read `.harness/tasks/T169.md` and its
resulting diff first for the literal pattern to copy):

1. **Type:** add `notifyEnabled?: boolean;` to `WorkflowDefinition` in `src/core/types.ts` (defaults
   to `true` when a workflow manifest omits it).
2. **Schema/migration (T098 rule — no index on a migration-added column):** add `notify_enabled`
   (INTEGER, NOT NULL DEFAULT 1) and `notify_enabled_overridden` (INTEGER NOT NULL DEFAULT 0)
   columns to the `workflows` table — in `src/db/schema.sql` for fresh DBs, and as an additive
   `ALTER TABLE` migration in `src/db/index.ts` for existing DBs. The manifest's `notifyEnabled`
   (default `true`) seeds `notify_enabled` on sync.
3. **Store:** add `updateWorkflowNotifyEnabled(name, enabled)` in `src/db/store.ts` that sets
   `notify_enabled = enabled ? 1 : 0` and flips `notify_enabled_overridden = 1`. In
   `upsertWorkflowStmt` (the sync reconcile), preserve the override the same way `max_concurrency`
   does: `notify_enabled = CASE WHEN notify_enabled_overridden = 1 THEN notify_enabled ELSE
   excluded.notify_enabled END`.
4. **Executor gates the notification call with the EFFECTIVE value:** in
   `src/core/workflow-executor.ts`'s `runWorkflow`, right before the aggregate
   `await notifyWorkflow(def.name, workflowRunId, notifyStatus, log)` call (~line 435, the single
   T189 aggregate call site), read the effective notify-enabled value (an
   `effectiveWorkflowNotifyEnabled(def)`-style helper — DB override when set, else the manifest's
   `notifyEnabled` default `true`, mirroring `effectiveWorkflowConcurrency(def)` at line 56/299) and
   skip the `notifyWorkflow` call when it's `false` — but still `log(...)` that the notification was
   skipped (per this repo's "log generously" convention), so the run's own log narrates why no push
   went out.
5. **API:** `POST /api/workflows/:name/notify` with `{ notifyEnabled: boolean }` — a mutating
   endpoint behind the SAME loopback/token guard as `/toggle`, `/run`, `/schedule`, `/concurrency`,
   `/limits`. Validate the body is a boolean and reject otherwise with **400**; 404 for an unknown
   workflow. Surface the effective `notify_enabled` in the `GET /api/workflows/:name` payload
   (alongside `effective_max_concurrency`) so the UI can display it. Add an
   `api.updateWorkflowNotify(name, notifyEnabled)` client method in `dashboard/app/lib/api.ts`.
6. **Dashboard:** on the workflow detail page (`dashboard/app/workflows/[name]/page.tsx`), add a
   **"Notifications"** row using the SAME click-to-toggle affordance as the existing `enabled` toggle
   (see `toggle()` at line ~766 and the `<span className="toggle" onClick={toggle}>` + checkbox at
   lines ~825-826) — clicking calls `api.updateWorkflowNotify(name, !currentValue)` directly, no
   separate Save button, and shows "notifications on (click to toggle)" / "notifications off (click
   to toggle)". The 3s poll refetches so it reflects the current effective value. Mobile-responsive,
   consistent styling with the existing enabled/schedule/concurrency rows.
7. **Docs:** update `CLAUDE.md` (notifyEnabled joins `enabled` + `schedule` + `maxConcurrency` +
   service limits as a user-editable, code-reconciled, override-flagged workflow property; note the
   gate is applied at the single T189 aggregate `notifyWorkflow` call site, not per-stage) and the
   README workflows/notifications section.

Tests: `updateWorkflowNotifyEnabled` persists + flips the override; a code-sync PRESERVES an
overridden value but refreshes a non-overridden one from the manifest (mirror the existing
`maxConcurrency`/schedule reconcile test); the API rejects a non-boolean body with 400; a workflow
run with `notify_enabled = 0` does NOT call `notifyWorkflow` (mock/spy it) but still logs the skip;
the migrate-existing-db guard passes with the two new columns.

## Done when

A workflow's push notifications can be toggled on/off from its detail page (click-to-toggle, like the
existing `enabled` switch), persisted as a user override that a code-sync preserves; both
existing and newly-added workflows default to notifications ON; `runWorkflow` reads the effective
value fresh each run (so an edit takes effect on the next run, no daemon restart) and gates ONLY the
single T189 aggregate `notifyWorkflow` call, logging when a notification is skipped; the API validates
the body (boolean, else 400) behind the mutation guard and exposes the effective value on the workflow
payload; `CLAUDE.md` + `README.md` document it; the new columns migrate cleanly on an existing DB;
`npx tsc --noEmit`, `npm test`, and `npm --prefix dashboard run build` are green; `node
dashboard/scripts/mobile-check.mjs` stays green; `node dashboard/scripts/visual-check.mjs` was run
after the build and the screenshots in `dashboard/scripts/visual-out/` (the workflow detail page)
confirm the new "Notifications" toggle row renders correctly, and
`dashboard/scripts/_dashboard-harness.mjs` is updated if the toggle changes what the baseline
`/workflows/<name>` screenshot shows.
