## Overview
Harden the shared usePoll hook so a slow or out-of-order response can never overwrite fresher data, and so a dynamic interval actually re-applies.

## Do
In dashboard/app/ui.tsx, usePoll (:282-310) has no ordering guard — an interval tick racing a manual refetch(), or a slow prior tick when latency exceeds the interval, can resolve out of order and let the OLDER response win setData/setError. Add a monotonically-increasing sequence ref: capture const seq = ++seqRef.current immediately before the await inside tick(), and only call setData/setError when seqRef.current === seq (i.e. this is still the newest in-flight tick). Also add intervalMs to the effect's dependency array (currently only the caller-supplied deps are passed), so a dynamic interval would take effect — latent today but correct. Add a pure hook test (dashboard/app/ui.test.ts, or the existing test file) that resolves an older tick AFTER a newer one and asserts the newer data is retained. Owner decision (2026-07-12): ALSO include the optional document.visibilityState poll-pause (Q04 item 4) — pause polling while the tab is hidden and resume (with an immediate refetch) on becoming visible.

## Done when
usePoll discards a stale (out-of-order) response via the sequence token and never overwrites newer data; intervalMs is in the effect deps; a pure hook test proves the older-response-after-newer case; npx tsc --noEmit, npm test, and npm --prefix dashboard run build are green. usePoll pauses its interval while document.visibilityState==="hidden" and resumes+refetches on visibilitychange back to visible.
