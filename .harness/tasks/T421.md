## Overview

Three of the four `places` workflow stages — `cid-to-place-id-resolver` (`resolve.ts`),
`places-enrich` (`enrich.ts`), and `enrich-with-llm` (`llm-enrich.ts`) — run a genuine per-item
processing loop, correctly mark each failed item `'failed'` in the `work_items` ledger, and
correctly track a local failed-count tally, but then **always return/exit normally regardless of
that tally**, so a run where every single item failed is still recorded as a `'success'` run. This
silently hides real breakage (e.g. a systemic resolver failure, a broken enrich field mask, a
parsing regression in `parseResult`) from the workflow's own run status and lets a broken
downstream stage spawn on bad/incomplete upstream data. Per the newly-decided convention (T394): an
item-loop job must `throw` at the end of `run()` if this run's own tally shows `failed > 0` (a
genuine failure, not a soft quota/rate-limit stop), so the run is correctly marked failed, blocks
downstream DAG dependents, and is retried via the job's existing `maxRetries` budget. The fourth
stage, `places-ingest`, does NOT have this gap (see `## Do` note below) and needs no change.

### Note on `places-ingest` — out of scope, no change needed

`ingest.ts`'s `recordIngestLedger` only ever marks items `'success'` — there is no per-item
`'failed'` outcome in this stage's ledger at all (a parse issue becomes a `ValidationIssue`
warn/error on the whole-run `ValidationReport`, not a per-place ledger failure). `ingest.job.ts`
already throws when `report.ok` is false (any error-level validation issue), which is the
equivalent hard-fail for this stage's failure shape. The T394 convention does not apply here — do
not add anything to `ingest.ts`/`ingest.job.ts` as part of this task.
