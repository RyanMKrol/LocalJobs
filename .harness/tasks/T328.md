The owner wants `.harness/IDEAS.md`'s `## Inbox` bullets reformatted from flat `- ` markdown bullets to numbered bullets (`1. `, `2. `, `3. `, ...).

This is a `needs-human` task because the bullet FORMAT is not purely cosmetic in this repo — it's a load-bearing parsing convention. `.harness/consolidate-ideas.mjs` (the Stage 3 consolidation pass for `/local-jobs-convert-ideas`) has a bullet-boundary parser that finds each top-level idea bullet's start/end by matching lines that literally `startsWith('- ')`:

```js
if (lines[i].startsWith('- ')) {
  const start = i;
  let j = i + 1;
  while (j < lines.length && !lines[j].startsWith('- ') && !lines[j].startsWith('## ')) j++;
  bulletSpans.push({ start, end: j, text: lines.slice(start, j).join('\n') });
  i = j;
}
```

This code (around line 161 of `.harness/consolidate-ideas.mjs` at the time this task was authored) walks the lines under the `## Inbox` header, and treats ANY line starting with `- ` as the start of a new bullet, continuing until the next `- `-prefixed line or the next `## ` heading. It is used to fuzzy-match each converted idea's recorded text against the live bullets in the file, so the matched span can be deleted once an idea becomes a task.

If `IDEAS.md`'s bullets are reformatted to numbered style (`1. `, `2. `, ...) WITHOUT also updating this parser, the parser will find ZERO bullet spans (nothing starts with literal `- ` any more). The fuzzy-match bullet-removal step already logs a warning and leaves a bullet in place when no match is found — so bullet removal would silently stop working entirely the next time `/local-jobs-convert-ideas` runs against a numbered-format `IDEAS.md`. Converted ideas would pile up in the inbox forever instead of being cleaned up.
