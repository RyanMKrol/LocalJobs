## Overview
Add a new read-only API endpoint that returns the actual service_cache row contents (via the listServiceCacheRows() store function from the prior task), so the dashboard admin page can browse what's cached, not just row counts.

## Do
In src/api/server.ts, near the existing `GET /api/cache` (~line 1515-1523, which calls serviceCacheCounts()), add a new route `GET /api/cache/rows` that accepts an optional `?service=<name>` query param, calls `listServiceCacheRows(url.searchParams.get('service') ?? undefined)` (import it alongside the existing `serviceCacheCounts`/`clearServiceCache` imports from '../db/store.js' or wherever those are currently imported from in server.ts — check the existing import line for serviceCacheCounts), and returns `json(res, 200, { rows: [...] })` where each row is `{ service_name, cache_key, cached_at, response_json, live }` — `response_json` exactly as returned by the store (raw string — do not parse/reformat it server-side; that's a UI concern). OWNER-CONFIRMED: compute `live: boolean` server-side per row by looking up that row's service_name in the service registry (the same registry `effectiveLimits`/`effectiveServiceTimeoutMs`-style helpers in src/core/services.ts already consult for a service's `cacheTtlMs`) and comparing `Date.now() - Date.parse(cached_at + 'Z')` (SQLite datetime() has no trailing Z, mirror the existing UTC-parse convention used elsewhere) against that service's resolved `cacheTtlMs` (a service with no configured TTL treats every row as live, since nothing would ever purge it). This is READ-ONLY — no loopback/token mutation guard needed (mirrors the existing GET /api/cache, which has none). Follow the existing route table pattern (a `{ method: 'GET', pattern: '/api/cache/rows', handler: ... }` entry in the `routes` array) — do not add an ad-hoc if-chain.

## Done when
`GET /api/cache/rows` (no query) returns `{ rows: [...] }` with every service_cache row, each including a computed `live: boolean` reflecting whether `cached_at` is still within that row's service's configured `cacheTtlMs`; `GET /api/cache/rows?service=<name>` returns only that service's rows, same shape. A new test in src/api/server.test.ts seeds ≥2 service_cache rows across ≥2 services (directly via the DB/store, same pattern as existing cache-related tests in that file if any exist — check for one near the existing GET /api/cache or POST /api/cache/clear tests), including at least one row old enough to be past its service's TTL and one within it, and asserts both the unfiltered and service-filtered responses return the expected rows with all 5 fields present and `live` computed correctly for both cases. `npx tsc --noEmit` and `npm test` are green.
