## Overview

On every workflow DAG graph, a validation gate's padlock mark should communicate its state (passed /
failed / pending) via a colored, softly-rounded BACKGROUND behind the icon — not just via the
padlock's own stroke/fill color as today, which is easy to miss on the dotted graph background at a
glance.

## Do

Work in `GateEdge` in `dashboard/app/components/DagFlow.tsx` (~lines 80-164, the custom edge that
renders the gate padlock via `EdgeLabelRenderer`). Today the state color pipeline is:

```tsx
const lockColor = isFailed ? 'var(--red)' : isPassed ? 'var(--green)' : 'var(--muted)';
const lockFill = `color-mix(in srgb, ${lockColor} 18%, transparent)`;
```

`lockColor` is applied to (a) the SVG `stroke` (the padlock outline) and (b) `lockFill`, an 18%-mix
tint applied ONLY to the padlock body's own `<rect fill={lockFill} />` inside the icon (T310) — that
is a fill *inside* the glyph, not a background *behind* it. The icon sits inside an
`EdgeLabelRenderer` wrapper `<div>` (~lines 127-137) that currently has a flat, state-agnostic
background:

```tsx
<div
  className="nodrag nopan"
  style={{
    position: 'absolute',
    transform: `translate(-50%, -50%) translate(${labelX}px,${labelY}px)`,
    pointerEvents: 'all',
    background: 'var(--panel-2)',
    borderRadius: 4,
    padding: '2px 3px',
    lineHeight: 0,
  }}
>
```

1. Change that wrapper `<div>`'s `background` from the static `'var(--panel-2)'` to a value derived
   from `lockColor` (the SAME three-way red/green/grey state value already computed above — do not
   introduce a new color mapping), at a modest, clearly-visible opacity via `color-mix`. Reuse the
   SAME 18% mix ratio T310 already established for the icon's internal fill (for visual consistency
   between the icon and its new background — `color-mix(in srgb, ${lockColor} 18%, transparent)`), OR
   pick a slightly higher ratio (e.g. 22-25%) if 18% reads as too faint once you look at the actual
   rendered screenshot — your call, but justify whichever you pick against the rendered screenshot,
   not just the number. Because the graph's own background is dotted/textured (`Background` /
   `BackgroundVariant.Dots` in the parent `DagFlow` component) rather than a flat panel color, the
   background needs enough opacity to read clearly as a distinct "badge" behind the icon at all three
   states, including the muted/grey pending state which is the easiest to end up indistinguishable
   from the graph background — confirm pending specifically renders as a visibly distinct grey chip,
   not just green/red.
2. Give the wrapper a `border-radius` that reads as "softly rounded" per the owner's own wording for
   this idea — a rounded-square/pill treatment sized to comfortably contain the icon with a few
   pixels of padding on each side (the existing `padding: '2px 3px'` may need a touch more room once
   there's a visible colored background to sit inside, so the corners don't look cramped against the
   icon — use your judgement looking at the rendered screenshot). Do not turn it into a full circle
   unless that's genuinely what reads best against the padlock's rectangular-ish silhouette; a
   rounded square/pill is the more likely fit but this is your call to make and document.
3. **Preserve the existing hover/failed interaction states rather than fighting them.** Today:
   - `.dag-gate-lock:hover { opacity: 1; transform: scale(1.25); }` (`dashboard/app/globals.css`
     ~line 268-269) scales the icon itself on hover.
   - `.dag-gate-lock--failed { opacity: 1; filter: drop-shadow(...); }` and
     `.dag-gate-lock--failed:hover { transform: scale(1.35); }` (~270-271).
   - `.dag-gate-lock--passed { opacity: 0.45; }` and `.dag-gate-lock--passed:hover { opacity: 0.85; }`
     (~272-273).
   These classes (`dag-gate-lock` / `dag-gate-lock--failed` / `dag-gate-lock--passed`) are applied to
   the inner `<a>`/`<span>` that wraps the SVG (~lines 120, 139-158), NOT to the outer wrapper `<div>`
   you're adding the background to. If the new colored background stays perfectly static while the
   icon inside it scales up on hover, the icon can visually escape/overflow its own badge, which will
   look broken. Make the background transition/scale together with the icon on hover — the simplest
   correct approach is to move the new background (and its border-radius) onto the SAME element the
   `dag-gate-lock*` classes are already on (the `<a>`/`<span>`, via inline style or new CSS driven by a
   CSS custom property set inline e.g. `style={{ '--lock-bg': lockFill }}` + a `background:
   var(--lock-bg)` rule alongside the existing `.dag-gate-lock` rules in `globals.css`) rather than on
   the separate outer `EdgeLabelRenderer` wrapper div — that way the existing `transform: scale(...)`
   hover rules naturally scale the background WITH the icon with no extra work. Keep the outer
   `EdgeLabelRenderer` wrapper div's own `background: 'var(--panel-2)'` as the surrounding "chip" if
   you want (it still gives the whole overlay a base contrast box against the dotted canvas) OR remove
   it if the new state-colored inner background makes it redundant — your call, but the padlock itself
   must end up visibly sitting inside a colored, rounded background whose color unambiguously encodes
   the gate state, and that background must transition/scale WITH the icon on hover, not stay static
   underneath it.
4. Add whatever new CSS rules you need to `dashboard/app/globals.css` alongside the existing
   `.dag-gate-lock` / `.dag-gate-lock--failed` / `.dag-gate-lock--passed` rules (~lines 267-273) rather
   than scattering new ad-hoc class names elsewhere. A `transition` on `background`/`background-color`
   matching the existing `opacity .15s, transform .15s` timing keeps the state-change (e.g. pending →
   passed once a run completes) feeling consistent with the rest of the icon's existing transitions.

Do NOT change: the padlock SVG's own geometry (`viewBox`, `path`/`rect` coordinates), `lockSize`,
`strokeWidth`, the three-way `lockColor`/`isFailed`/`isPassed` state derivation itself, or anything in
`buildLayout`/edge-data plumbing above `GateEdge` — this task is scoped purely to the visual
background treatment of the already-existing padlock mark.
