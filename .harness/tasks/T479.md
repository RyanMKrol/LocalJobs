## Overview
Once idea-11-a lands (per-service configurable cache TTL) and idea-11-b wires cacheKey into Plex reads, review every OTHER service (tmdb, google-places, gemini, dynamodb, github, fragrantica, claude-cli, hevy, lastfm, trading212, trading212-instruments, openfigi) to decide which of them would similarly benefit from response caching via callService's cacheKey option, and what TTL each should use.

## Do
For each registered service (src/services/*.service.ts), check: (a) is it already using cacheKey anywhere (tmdb already is, via plex-language-fix's resolve.ts), (b) does it make genuinely idempotent, repeatable-within-a-short-window calls where a cache would reduce real spend/latency (e.g. google-places or gemini calls keyed by a stable identifier that's looked up more than once), (c) if so, what TTL makes sense given how often the underlying data changes. Produce a short written recommendation (in the worklog) naming which services should get a follow-up cacheKey-adoption task and their target TTLs, or explicitly conclude none are worth it beyond tmdb/plex.

## Done when
A clear owner decision is recorded (in the worklog) on which services, if any, should get a follow-up task to adopt cacheKey caching, with target TTLs — this task does not itself implement any caching, only decides. A dependent buildable follow-up task should be authored (via a future backlog-authoring pass) for each service the owner confirms.
