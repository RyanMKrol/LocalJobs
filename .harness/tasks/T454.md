## Overview
No bulk-unignore mechanism exists anywhere in the codebase today — unignoreSurfacedItem (src/db/store.ts:1029) only reverses one item at a time, and there is no */unignore-bulk API route. This task adds the backend half (store function + routes) so a later task can wire up 'Un-ignore all' buttons in the dashboard.

## Do
In src/db/store.ts, add `export function unignoreSurfacedItems(jobName: string, itemKeys: string[]): number` directly below unignoreSurfacedItem (~line 1029), mirroring the existing ignoreSurfacedItems batch pattern (~line 1003): loop the given itemKeys, call unignoreSurfacedItem(jobName, key) for each (or run the equivalent DELETE inside a single transaction for efficiency, matching whatever pattern ignoreSurfacedItems already uses — read it first and mirror its shape exactly, including any transaction wrapping), and return the total count of rows actually deleted/affected. Add a `*.test.ts` unit test (this repo requires tests for new store behavior) exercising: unignoring a subset of ignored keys removes exactly those ledger rows and leaves others untouched; unignoring a key that was never ignored is a no-op (doesn't throw, contributes 0 to the count).

In src/api/server.ts, add four new mutating routes, one per existing ignore-bulk route, each accepting the SAME body shape as its ignore-bulk twin and calling unignoreSurfacedItems instead of ignoreSurfacedItems:
- POST /api/movie-gaps/unignore-bulk { tmdbIds: number[] } → unignoreSurfacedItems(MOVIE_GAPS_JOB, tmdbIds.map(gapKey))
- POST /api/missing-seasons/unignore-bulk { items: { tmdbId: number, season: number }[] } → unignoreSurfacedItems(PLEX_SEASONS_JOB, items.map(({tmdbId,season}) => pairKey(tmdbId, season)))
- POST /api/movie-recs/unignore-bulk { tmdbIds: number[] } → unignoreSurfacedItems(RECS_JOB, tmdbIds.map(recKey))
- POST /api/tv-recs/unignore-bulk { tmdbIds: number[] } → unignoreSurfacedItems(TV_RECS_JOB, tmdbIds.map(tvRecKey))
Each validates its body the same way its ignore-bulk sibling does (400 on malformed input) and sits behind the same loopback/token mutation guard. Each returns { ok: true, unignored: <count> }.
No dashboard/frontend changes in this task — it is backend-only, feeding a follow-up UI task.
