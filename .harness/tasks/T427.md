## Overview

The dashboard doesn't use client-side navigation at all today — every internal link, including the
persistent header nav, is a plain `<a href="...">`, which forces a full browser page reload (and a
full React remount) on every click. This is why the theme mode icon visibly flickers (moon → correct
icon) on EVERY navigation, not just on first page load: `ThemeControls`/`useMode` (`app/ui.tsx`) has
to initialize its React state with a hardcoded placeholder for hydration-safety (a deliberate, already
-documented T363 tradeoff — see the comment above `useMode`), and that placeholder-then-correct cycle
re-runs from scratch every time the whole app remounts. Converting internal links to Next.js's
`next/link` `<Link>` component gives genuine client-side transitions (the persistent layout, including
`ThemeControls`, stays mounted across navigation) — this is a near drop-in swap, not an architecture
rewrite, since Next's App Router already provides SPA-style navigation once `<Link>` is used.

## Do

### 1. Convert internal links to `next/link`

For every internal (same-origin, app-route) `<a href="...">` or `<a href={...}>` found across
`dashboard/app/**`, replace it with Next's `<Link href="...">` (`import Link from 'next/link'`).
Keep all existing className/style/other props on the element — this is a tag swap, not a redesign.
Leave every genuinely external link (an absolute `https://` URL to a different origin, e.g. the TMDB
links) as a plain `<a>` exactly as it is today, including its `target`/`rel` attributes.

Start with `app/layout.tsx`'s header nav (the highest-impact fix, since it's rendered on every page)
and the brand link, then work through the rest of the files listed above. For any `href` you're not
sure is internal vs external (e.g. `DagFlow.tsx`'s `data.href`, `logs/page.tsx`'s `href`, `ui.tsx`'s
`fallback.href`, `jobs/[name]/page.tsx`'s `back.href`), read enough of the surrounding code to confirm
before converting — do not guess.

### 2. Verify real client-side navigation, not just a visual check

Add a hermetic Playwright-based check proving a click on an internal nav link does NOT trigger a full
page reload. The clean way to prove this: inject a marker into `window` after the first page load
(e.g. `await page.evaluate(() => { window.__t427Marker = true; })`), click an internal nav link, then
assert `window.__t427Marker` is STILL `true` (`await page.evaluate(() => window.__t427Marker)`) — a
real page reload would wipe any in-memory JS global, so this only passes if navigation was genuinely
client-side. Add this as a new check in `dashboard/scripts/_dashboard-harness.mjs`'s shared
infrastructure (or a new small script reusing it — your call on the cleanest place, matching how
`mobile-check.mjs`/`visual-check.mjs` already reuse the shared harness) so it's reusable and consistent
with this repo's existing hermetic-dashboard-check pattern (synthetic `/api/**` fixtures, no daemon/
SQLite dependency).

### 3. LIVING ARTIFACT check

Per this repo's `layer:ui` convention, confirm whether this change requires any update to
`dashboard/scripts/_dashboard-harness.mjs`'s `PAGES`/`FLOWS` — it likely does NOT need new `FLOWS`
entries (no new interactive state is being added, just a navigation mechanism change), but confirm
this yourself rather than assuming.
