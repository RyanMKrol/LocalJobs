## Overview

`stocks-watch` (`src/workflows/stocks-sync/stages/stocks-watch.ts`) writes its per-run breach *check* row
with a useful `detail` (`{ gainPct, breaching }`) — that's already compliant with the "every stage's
success `detail` must describe what it produced" convention. But the SAME stage also writes a second,
separate ledger row per position — the `<positionKey>::notified` "have I already notified this breach
episode" marker — with **zero `detail` at all** on both its `'success'` and `'skipped'` writes. Anyone
inspecting that ledger entry (dashboard ledger views, `work_items` queries, future IO-panel surfacing)
sees a bare status with no explanation of what happened, even though the stage had computed the exact
gain/price numbers moments earlier in the same loop iteration.

## Do

Note: this is orthogonal to and does NOT conflict with T403 (which only touched `stocks-snapshot`'s
ledger granularity, already landed as one-row-per-day — `stocks-watch` is untouched by that change and
unaffected either way).

In `src/workflows/stocks-sync/stages/stocks-watch.ts`, inside `runStocksWatch`'s per-position loop
(currently around lines 99–122):

1. **Fresh-breach `'success'` write on the `notifiedKey`** (currently, ~line 111):
   ```ts
   markWorkItem(WATCH_JOB, notifiedKey, 'success');
   ```
   has no third argument at all — `detail` is entirely absent. Change to pass a `detail` describing
   the breach that was just recorded as notified, using data already computed in scope (`label`,
   `position.ticker`, `position.account`, `gain`, `position.averageBuyPrice`, `position.currentPrice`):
   ```ts
   markWorkItem(WATCH_JOB, notifiedKey, 'success', {
     detail: {
       name: `${label} — notified breach`,
       ticker: position.ticker,
       account: position.account,
       gainPct: gain,
       averageBuyPrice: position.averageBuyPrice,
       currentPrice: position.currentPrice,
     },
   });
   ```

2. **Reset `'skipped'` write on the `notifiedKey`** (currently, ~line 115):
   ```ts
   markWorkItem(WATCH_JOB, notifiedKey, 'skipped');
   ```
   also has no `detail`. Change to record why the reset happened:
   ```ts
   markWorkItem(WATCH_JOB, notifiedKey, 'skipped', {
     detail: {
       name: `${label} — reset (dropped below threshold)`,
       ticker: position.ticker,
       account: position.account,
       gainPct: gain,
     },
   });
   ```

3. Leave the existing per-run check row (`markWorkItem(WATCH_JOB, key, 'success', { detail: { gainPct,
   breaching } })`, ~line 120) **unchanged** — it already complies with the convention.

4. Leave `stocks-snapshot.ts` unchanged — its combined-day row already includes `positionCount`,
   `totalValue`, `resolvedCount`, and `markdown` (the produced file path); it is fully compliant
   post-T403.

5. Leave `stocks-notify.ts` unchanged — it deliberately records **no** `work_items` rows at all (a
   pure notify-trigger, per T348/T300's design, mirrored by the workflow manifest's `outputJob:
   'stocks-snapshot'` override) — there is nothing to add `detail` to.

6. Update any test in `src/workflows/stocks-sync/stages/stocks-watch.test.ts` only if a new assertion is
   added for the new `detail` shape (optional — existing tests don't assert `detail` contents today and
   won't break from this change, but adding a small assertion that the `notifiedKey` row's `detail` is
   non-null/contains `gainPct` after a fresh breach would directly guard the fix).
