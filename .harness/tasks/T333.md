## Root cause — CONFIRMED by direct empirical reproduction (do not re-derive from scratch; verify then fix)

Investigated this by building the dashboard and driving it with a standalone Playwright script against a synthetic 20-row `/api/movie-recs` fixture (enough rows to genuinely overflow 520px). Findings, precise and reproducible:

1. `.movie-gaps-scroll` (`dashboard/app/globals.css:178`) is `display:flex; flex-direction:column; max-height:520px; overflow-y:auto;` and wraps its content (a `<div className="panel">` containing the recommendations `<table>`) as a flex ITEM.
2. `.panel` (`dashboard/app/globals.css:87`) declares `overflow: hidden` (present for border-radius clipping across the whole app).
3. Per the CSS Flexbox spec, a flex item's automatic minimum size (the rule that normally stops a flex item shrinking below its own content size) collapses to `0` whenever that item's own `overflow` is not `visible`. Because `.panel` has `overflow: hidden`, it has NO content-based minimum — so the flex layout algorithm SHRINKS the `.panel` down to fit inside the 520px `.movie-gaps-scroll` container instead of letting it grow taller than 520px (which is what would need to happen for the outer `overflow-y: auto` to ever have something to scroll).
4. Net measured effect (Playwright, 20-row fixture): `.movie-gaps-scroll.scrollHeight === .movie-gaps-scroll.clientHeight === 520` — i.e. the scroll container's own scrollable range is ALWAYS zero, so `page.mouse.wheel()` over it does nothing (`scrollTop` stays `0` before and after), and even a PROGRAMMATIC `el.scrollTop = 100` has no effect (nothing to scroll to). This exactly matches the owner's report: the mouse wheel does nothing, AND rows visually cut off after a certain point — because the table is being clipped by `.panel`'s OWN `overflow: hidden`, not by the outer scroll container overflowing.
5. Verified fix: adding `flex-shrink: 0;` to the existing `.movie-gaps-scroll .panel { padding: 12px 16px; }` rule (`globals.css:180`) stops the panel being forced to shrink. Re-measured with the same 20-row fixture: `.panel.offsetHeight` grows to its natural content height (965px in the test), `.movie-gaps-scroll.scrollHeight` (965) now genuinely exceeds `.clientHeight` (520), and BOTH a programmatic `scrollTop` set AND a real `page.mouse.wheel()` interaction now move the content (`scrollTop` went `0 → 400` after a wheel event, confirmed with the mouse positioned over the container via `scrollIntoViewIfNeeded()` + `mouse.move()` — a real repro pitfall: `boundingBox()`/`mouse.move()` coordinates must be within the current VIEWPORT, not just the page; if the container is below the fold, `document.elementFromPoint()` at those coordinates returns `null` and the wheel event lands nowhere — always `scrollIntoViewIfNeeded()` first). Reverting the `flex-shrink: 0` reproduces the bug exactly (`scrollTop` stays `0` after `page.mouse.wheel()`).

**Apply this fix**: add `flex-shrink: 0;` to the `.movie-gaps-scroll .panel` rule in `dashboard/app/globals.css` (or an equivalent scoped rule with the same effect — the key requirement is that a `.panel` (or any overflow:hidden child) inside `.movie-gaps-scroll` must NOT be allowed to flex-shrink below its content height, so the OUTER container is the one that overflows and scrolls). Do your own independent check before committing to this (re-derive the measurement above, don't just trust this note blindly) — but this is the confirmed, verified root cause and fix, not a guess.
