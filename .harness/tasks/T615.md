## Overview
Re-attempt of T601, which was blocked (audit FAIL x2 at the top rung). The original spec wrongly assumed all 8 recommender branches build a single stratified sample (default 50) that could be recomputed after the fact, and its scope excluded the files where the real per-branch input is actually built. In truth only the 3 rec-random-N branches call stratifiedSample(seed:1000+n); the 5 targeted branches (rec-auteur/rec-canon/rec-thin-genre/rec-older-era/rec-world-cinema) build entirely different lens-filtered subsets (ownedByDirectors/ownedInGenres/ownedPreYear/ownedNonAnglophone) inside spec.build(...). The blocked builder — unable to touch branches.ts/types.ts and told to 'record the sampled item ids/titles' — recomputed a fresh stratified sample with a different seed/algorithm OUTSIDE build, producing per-branch-distinct-but-FABRICATED data that passed the size+subset unit test yet was correctly rejected by the auditor as not the ACTUAL set the branch was shown. This re-attempt widens scope to branches.ts (both workflows) + types.ts so the fix can intercept the genuine item set each branch selected, rather than reconstructing it.

## Do
Capture, per branch run, the EXACT owned-item set that branch actually put into its Claude prompt — the real subset produced inside spec.build(...), NOT a recomputed/re-sampled reconstruction. The correct approach is to have build EXPOSE the item set it selected so runBranch can record precisely those items. Concretely: (1) Change BranchSpec.build's contract in src/core/recommender/types.ts so, in addition to the prompt string, a branch reports the exact owned items it included (e.g. build returns { prompt, sampledItems } | null, or accepts a sink/callback the branch pushes its selected items into — pick the cleaner shape and keep the null-means-skip semantics). Do NOT recompute with a new seed or a genre-stratified sample for the targeted branches — that was exactly the bug. This is the same class of alternate-ledger-source problem outputJob solves for the notify stage's Outputs — reuse/generalize that mechanism if it's clean to do so, but do not over-engineer.
