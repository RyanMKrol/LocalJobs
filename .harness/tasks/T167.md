## Do

On a finished job's run detail page (`dashboard/app/runs/[id]/page.tsx`), the progress bar keeps
running its **moving "barber-pole" stripe animation** (diagonal stripes scrolling left→right) even
when the job is **100% done**. A scrolling animation implies work is in progress, which is
misleading for a completed run. Stop the animation once the run is terminal / at 100%.

- The animated stripe fill comes from the joyful themes (T142 — "animated progress-bar fills"),
  applied to the progress fill (`.progress > span`). It should only scroll while the run is actually
  IN PROGRESS (status `running`, below 100%).
- Fix: gate the scrolling animation on the run being in-progress — e.g. add a state class
  (`.progress.running` vs `.progress.done`, or condition on the run status), so a completed
  (`success`/`failed`/`cancelled`) or 100% bar shows a **static** fill. The stripe texture can stay
  as a static pattern; only the scrolling MOTION stops when done.
- The fill colour/state (e.g. green for success) is unchanged — only the motion stops. Keep the
  reduce-motion path (it should already not animate).
- Apply consistently to every progress bar that can represent a terminal run — the run detail page
  and the **workflow-run** progress bar (check both).
- Mobile-responsive.

## Done when

A finished / 100% run's progress bar no longer shows the scrolling barber-pole animation (it's
static once the run is terminal), while an in-progress run still animates; reduce-motion is
unaffected; the fix covers the run detail and workflow-run progress bars; `npm --prefix dashboard
run build` passes and `node dashboard/scripts/mobile-check.mjs` stays green.
