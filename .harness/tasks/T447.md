## Overview

The plex-space-saver workflow's dashboard Output section (and its workflow-run "Inputs & Outputs"
panel) currently shows "No output content found" / an empty preview modal instead of the real
size-breakdown data, even though the artifact file exists on disk and the run succeeded. The root
cause is NOT a bug in the output-serving code itself — it's that `work_items.detail` stores a raw,
**absolute filesystem path** frozen at write time, and the 2026-07-04 `src/jobs` → `src/workflows`
rename (commit `39bdafa`) moved every workflow's code+data directory, permanently orphaning every
already-recorded ledger row's stored path. Live confirmation:
`sqlite3 data/jobs.db "SELECT detail FROM work_items WHERE job_name='plex-space-saver-scan'"` returns
`.../src/jobs/plex-space-saver/data/out/size-breakdown.json` — a directory that no longer exists (the
real file now lives at `.../src/workflows/plex-space-saver/data/out/size-breakdown.json`). This is not
isolated to plex-space-saver: `SELECT job_name, count(*) FROM work_items WHERE detail LIKE
'%src/jobs/%' GROUP BY job_name` currently returns stale rows across **13 different jobs** —
`movie-gaps-notify` (348 rows), `movie-recs` (83), `tv-recs` (45), `plex-seasons-notify` (44),
`lastfm-digest`, `listening-digest`, `perfumes-build`, `plex-space-saver-scan`, `project-summarize`,
`stock-digest-build`, `stocks-fetch`, `stocks-snapshot`, `workouts-progress` (1 each) — meaning the
"View"/preview affordance is currently broken dashboard-wide for every item recorded before the
rename that hasn't been reprocessed since. `plex-space-saver-scan` happens to self-heal on its own
next weekly run (it re-records `detail` fresh every run), but many of the others (e.g.
`movie-gaps-notify`'s "have I notified this?" ledger, T144-style workflows) may never naturally
re-record an already-notified item's `detail`, so those rows could stay permanently broken.

This is exactly the kind of "coded so this can never happen again" fix the owner asked for: stop
storing a frozen absolute path in the ledger at all, and instead resolve a job's output artifact
**programmatically, by job name, at read time** — so a future workflow-folder rename (which this
repo's own history shows does happen) can never invalidate already-recorded rows again.

## Confirmed root cause (file:line)

- **Write side (the chokepoint):** `src/db/store.ts`'s `markWorkItem` (~line 454-488) is the SINGLE
  function every job calls to record a work item; `opts.detail` is `JSON.stringify`'d as-is at line
  475 with no path normalization. Every job author builds an absolute path themselves (e.g.
  `src/workflows/plex-space-saver/config.ts`'s `breakdownOut = resolve(dataDir, 'out',
  'size-breakdown.json')`, where `dataDir` is derived from `dirname(fileURLToPath(import.meta.url))`
  at module-load time) and passes it straight into `detail.markdown`/`detail.path`.
- **Read side:** `src/api/server.ts` has THREE guard functions that consume these stored paths —
  `safeOutputMarkdown` (~line 783), `safeOutputFile` (~line 817), and `resolveOutputForm` (~line 843)
  — all of which call `resolvePath(candidate)` (= Node's `path.resolve`, which for an absolute
  `candidate` just normalizes it — it does NOT re-root a stale absolute path). `WORKFLOWS_ROOT` (line
  709, `realpathSync(fileURLToPath(new URL('../workflows', import.meta.url)))`) is already computed
  **fresh, correctly, from the CURRENT on-disk location** every time the server process starts — it is
  NOT itself stale. The bug is purely that the STORED candidate path is stale, not the resolution
  constant.
- **Existing precedent for this exact "resolve fresh by name, don't trust a frozen path" pattern:**
  `findWorkflowDataOut(workflowName)` (server.ts ~line 727) already does this correctly for the T203
  "Clear output data" feature — it re-scans `WORKFLOWS_ROOT` for `*.workflow.ts` files and matches by
  the exported `default.name`, so it can never go stale across a rename. The fix below generalizes
  that same idea to `detail.markdown`/`detail.path`.

## Do

1. **Stop persisting frozen absolute paths.** In `src/db/store.ts`, add a `WORKFLOWS_ROOT`-equivalent
   constant (mirroring `src/api/server.ts`'s, resolved relative to `src/db/store.ts`'s own location —
   `realpathSync(fileURLToPath(new URL('../workflows', import.meta.url)))`) and a small helper (e.g.
   `toStoredPath(absPath: string): string`) that, given an absolute path under that root, returns it
   made **relative to `WORKFLOWS_ROOT`** (e.g. `plex-space-saver/data/out/size-breakdown.json`); a
   path NOT under the root is left untouched (defensive — don't break anything unexpected). Apply this
   in `markWorkItem` to `opts.detail.markdown` and `opts.detail.path` specifically (the two known
   path-bearing keys per the Output-form convention in `CLAUDE.md`) before `JSON.stringify`, so every
   job's existing call sites need ZERO changes — the normalization is transparent and centralized at
   the one chokepoint every job already calls.
2. **Resolve relative paths fresh, by root, at read time.** In `src/api/server.ts`, update
   `safeOutputMarkdown`, `safeOutputFile`, and `resolveOutputForm`'s candidate resolution so that a
   **relative** stored candidate (`!isAbsolute(candidate)`) is joined against the **freshly-computed**
   `WORKFLOWS_ROOT` (not `path.resolve`'s implicit `process.cwd()`, which only happens to work today
   because the daemon is launched from the repo root — an accidental correctness, not a guaranteed
   one) before the existing `realpathSync`/`isWithin`/`data/out` checks run unchanged. An **absolute**
   stored candidate (an old, pre-fix row) continues to resolve exactly as it does today — this task
   does not need to break backward compatibility for absolute rows, it just stops MAKING new ones.
3. **One-time migration to repair the currently-stale rows.** Add
   `scripts/backfill-fix-stale-output-paths.ts`, mirroring the idempotent, read/rewrite style of
   `scripts/backfill-service-usage.ts` (same top-comment convention: what it does, why, that it's
   idempotent, how to run it against the live DB vs. a scratch `LOCALJOBS_DB`). It should scan
   `work_items.detail` for rows whose `markdown`/`path` values are absolute paths that no longer exist
   on disk (`existsSync` false) but whose relative suffix (path relative to the OLD stale root prefix,
   or more simply: the path segment from the job's own folder name onward — e.g. everything from
   `plex-space-saver/…` onward) DOES resolve under the CURRENT `WORKFLOWS_ROOT`, and rewrite `detail`
   in place to the corrected (now-relative, per step 1's format) path. Log every row it fixes and any
   it can't confidently repair (leave those alone — don't guess). Run it once against the real
   `data/jobs.db` as part of verifying this task, and report how many of the 13 affected jobs / ~500+
   rows it corrected.
4. **Document the convention.** Add a short note to the root `CLAUDE.md`'s "Job resources are
   job-local" convention (and/or the Output-form convention section) and to
   `src/workflows/CLAUDE.md`, stating: `detail.markdown`/`detail.path` are stored **relative to the
   workflows root**, resolved fresh at read time — never store a raw absolute path — precisely so a
   future workflow-folder rename can't strand already-recorded ledger rows the way the `src/jobs` →
   `src/workflows` rename (T331) just did.
