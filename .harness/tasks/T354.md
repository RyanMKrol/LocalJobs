## Overview

When the owner selects a level filter pill (e.g. "warn") on the dashboard's Logs page
(`dashboard/app/logs/page.tsx`), previously-loaded log lines from BEFORE the filter changed keep
showing on screen instead of being cleared out — even though the server-side SQL query genuinely
filters correctly. Fix the client-side effect race so a filter change never leaves stale,
non-matching lines behind, and make the hermetic visual-check fixture level-aware so this exact
regression is actually visible in a screenshot going forward.

     do not change it.
   - The `usePoll(() => api.globalLogs(params), 5000, [levels.join(','), scopeType, scopeValue, q,

   - The "Reset the accumulated list whenever the filters (not pagination) change" effect
     (~lines 52-57):
     ```ts
     useEffect(() => {
       setLines(data?.logs ?? []);
       setNextCursor(data?.nextCursor ?? null);
     }, [levels.join(','), scopeType, scopeValue, q, windowHours]);
     ```
   - The "Merge in fresh newest-first data on each poll tick" effect (~lines 61-70), which only
     ADDS rows from `data.logs` that aren't already present in `lines` (deduped by `source:id`) —
     it never removes a row.

2. **Root cause (confirmed during investigation — this is NOT a server/SQL bug).** `GET /api/logs`
   (`src/api/server.ts`) and `listGlobalLogs()` (`src/db/store.ts:238-` onward) DO filter correctly
   server-side via a real SQL `WHERE rl.level IN (...)` clause (and the equivalent for workflow
   logs) — verified by reading the query builder end to end. The actual bug is a client-side React
   effect ordering/race:
   - Both the `usePoll` fetch effect and the "reset" effect above share the exact same dependency
     array, so both re-run in the same effect flush when any filter changes. `usePoll`'s effect
     KICKS OFF an async fetch (it does not block) using the NEW filter params; that fetch has not
     resolved yet.
   - Immediately after, in the same synchronous effect flush, the "reset" effect runs and reads
     `data` — but `data` state is still the RESULT OF THE PREVIOUS FETCH (from before the filter
     changed), because the new fetch's `setData` call hasn't happened yet (it's on the other side of
     a network round trip). So `lines` gets reset to the OLD filter's results, not the new one.
   - When the new, correctly-filtered fetch later resolves, the "merge" effect runs — but it only
     ADDS rows that aren't already present; it never prunes the stale rows the reset effect just
     seeded. Those stale rows (which may be any level, e.g. info) then sit in `lines` permanently,
     mixed in with the genuinely-filtered new rows, because nothing ever removes them.
   - This reproduces on EVERY filter change (level, scope, search text, or time window) — not only
     level — though the level pills make it most visually obvious, since the owner expects to see
     only warn-styled lines and instead sees a lingering mix.

3. **The fix**: change the "reset on filter change" effect to clear synchronously to an EMPTY list
   instead of seeding from the stale `data`:
   ```ts
   useEffect(() => {
     setLines([]);
     setNextCursor(null);
   }, [levels.join(','), scopeType, scopeValue, q, windowHours]);
   ```
   This is safe because the very next effect flush after the new fetch resolves runs the "merge"
   effect with `prev = []`, so `fresh = data.logs` in full and `lines` becomes exactly the new,
   correctly-filtered set — no stale row can survive a filter change. A brief "No log lines match
   the current filters." flash may appear for the (typically sub-second) round trip while the new
   fetch is in flight; that is an acceptable, minor trade-off for never showing wrong data. Do not
   try to eliminate the flash by any other means (e.g. re-introducing a stale-data fallback) — that
   would reintroduce the bug.

   Note for context — deliberately NOT fixed in this task (out of scope, a much narrower edge case
   than what's reported): if the owner changes filters twice within one network round trip, the
   FIRST fetch could still resolve after the SECOND reset and briefly reintroduce mismatched rows
   via the merge effect, since neither `usePoll` nor this component cancels in-flight requests or
   tags them with a request generation/sequence number. This window is far narrower than the
   guaranteed, screenshot-reproduced bug being fixed here and is left as a known limitation — add a
   one-line code comment noting it, but do not attempt to fix it (that would need request
   cancellation/generation tracking, a larger change).
