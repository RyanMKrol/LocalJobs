## Overview
Both POST /api/workflows/:name/run and POST /api/workflows/run-all (src/api/server.ts) dispatch `runWorkflow(...)` fire-and-forget with only `.catch((e) => console.error(...))` — if `runWorkflowInner` throws before it manages to create a `workflow_runs` row (e.g. during the pre-row `inputKeys()` wait this idea is about, or any other early failure), the failure is logged only to the daemon's own stdout/stderr and is completely invisible to the dashboard: no run row exists at all, so there's no record the attempt was ever made.

## Do
In `runWorkflowInner` (src/core/workflow-executor.ts), wrap the pre-row-creation work (the DAG build already has its own try/catch at the top that creates a row and records `failed` — the gap is specifically the newer `inputKeys()`/root-selection block added for T094 limited runs, between the DAG build and `createWorkflowRun`) so that ANY throw before the row exists still results in a `workflow_runs` row being created and marked `failed` with the error captured, mirroring the existing DAG-build-failure pattern just above it in the same function. This makes the fix apply uniformly alongside invisible-slow-crawl-b's 'Starting…' indicator — a genuinely thrown error during input-key resolution (a live crawl's API call rejecting, a network failure) becomes a visible failed run instead of a silent stdout line.

## Done when
Simulate `inputKeys()` throwing (add or extend a unit test in src/core/workflow-executor.test.ts using a fixture job whose `inputKeys()` rejects) for a limited manual run and confirm a `workflow_runs` row is created and ends in status `failed` with the thrown error captured in the run's log/error detail — not just logged to console. `npx tsc --noEmit` and `npm test` are green.
