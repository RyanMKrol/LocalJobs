## Overview
Cache the GitHub repo-catalog API READS (projects-sync's paginated repo listing) for 22 hours.
Outcome of the T479 caching-adoption decision; the caching mechanism already exists (T451/T476).
CRITICAL: the `github` service also routes the `git clone`/`fetch`/`reset` operations — those are
side-effecting, NOT cacheable reads, and must NOT get a cacheKey.

## Do
1. Set `cacheTtlMs: 79_200_000` (22 hours) on the `github` `ServiceDefinition` in
   `src/services/github.service.ts`.
2. Find the repo-listing API read calls under `src/workflows/projects-sync/stages/` — the paginated
   `fetchAllRepos` fetcher makes one `callService('github', ...)` reservation PER PAGE. Add a stable
   `cacheKey` to those READ calls only, keyed by endpoint + page:
   `` `github:repos:${user}:page:${page}` ``.
3. Do NOT add a cacheKey to the `git clone`/`git fetch`/`git reset` `callService('github', ...)`
   sites — caching a side-effecting git command is meaningless and wrong. Leave those untouched.
4. Cache reads only.

## Done when
- `cacheTtlMs: 79_200_000` is set on the `github` service def.
- The repo-listing READ calls pass a stable `github:repos:<user>:page:<n>` cacheKey; the git-operation
  calls are left WITHOUT a cacheKey.
- `npx tsc --noEmit` and `npm test` are green.
- Verified a repeated repo-list page fetch within 22h is served from `service_cache` (hermetic
  scratch-DB check) rather than a second live call.
