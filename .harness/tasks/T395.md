## Overview

A live, human-present walkthrough of the T094 "manual run limit" mechanism — how a workflow run
gets capped to N originating inputs, how the framework tracks which downstream items descend from
which input, and how that stays correct across a multi-stage DAG. This is a pure understanding
session, not a bug report: the mechanism is already built, working, and extensively documented — the
owner wants to sit with someone (or an agent) and walk through it end to end using the real
dashboard/API, not just read the docs alone.

## Do

This is a guided reading + live-demo session. Follow the order below; it goes from "what the owner
sees" down to "how it actually works," then closes with a real triggered example.

### 1. Start from the docs (5 min)

Read the root `CLAUDE.md` T094 section in full — search the file for `T094` (there are two hits: a
short cross-reference near the top-of-file gotcha about schema migrations, and the real writeup
under **Conventions → "Input lineage + manual run-limits (T094)"**, currently around line 1043).
Read straight through its nested T163 sub-bullet too ("Pending" is propagation through the TERMINAL
stage) — it's the one subtlety that isn't obvious from the feature name alone. This is the
already-agreed source of truth; the rest of this walkthrough is verifying the live system matches
it, not re-deriving it.

### 2. Read the code in this order (confirmed current as of this task's authoring — 2026-07-03)

1. **`src/db/store.ts`, `resolveRootKey`** (~line 434) and **`markWorkItem`** (~line 454) — the
   3-rule lineage resolution: explicit `rootKey` wins; else inherit the `parentKey` row's root; else
   the item is its own root (`root_key = item_key`). Point out that `perfumes` (same key every
   stage) never needs to pass `rootKey`, while `places`' `enrich`/`llm` stages do (they key by
   `place_id`, not the original `cid`).
2. **`src/db/store.ts`, `isRootPending`** (~line 583) and its three helpers immediately above it
   (`rootHasOutstandingWork`, `rootReachedTerminal`, `rootHasTerminalBlocker`) — this is the T163
   fix. Walk through the 4-branch decision it documents in its own comment: (1) any retryable
   not-done row anywhere → pending; (2) a done row at a TERMINAL-wave job → fully done; (3) a
   gave-up marker (ignored/skipped/retry-exhausted) with no retryable work → treat as done (avoids
   livelock); (4) otherwise → pending (this is the branch T163 added — a root whose downstream
   stage simply has no row yet used to be miscounted as "done").
3. **`selectPendingRoots`** (~line 605, same file) — freezes the first N pending roots, in
   `inputKeys()` order, onto the run row.
4. **`src/core/executor.ts`** — search `LOCALJOBS_WORKFLOW_RUN_ID` (~line 122-125) — how the frozen
   run id reaches every child process via env.
5. **`src/runJob.ts`** (~line 38-48) — how the child turns that env var into
   `ctx.selectedRoots()` / `ctx.rootAllowed(rootKey)` (a `null` set when unlimited, so
   `rootAllowed` is always `true`).
6. **`src/core/types.ts`** (~line 26-28) — the `JobContext` shape those two methods live on.
7. **`src/api/server.ts`** — search `limitable` (~line 955-965, the `GET /api/workflows/:name`
   payload field) and the `POST /api/workflows/:name/run` handler (~line 1582-1608) — where the
   `limit` body param is validated (must be a positive integer) and rejected with 400 if the
   workflow isn't limitable (no member declares `inputKeys()`).
8. **`dashboard/app/workflows/[name]/page.tsx`** — search `run-limit` (~line 673, 775, 786-794) —
   the small number box next to "▶ Run now" that only renders when `p?.limitable` is true, and
   posts `{ limit }` to the run endpoint.

At each stop, sanity-check the code still matches what `CLAUDE.md` claims (it did as of this task's
authoring — treat any drift you find as a real finding, not a doc bug to silently ignore; see "Done
when" below).

### 3. See which workflows are actually limitable right now

Only a workflow with at least one member job declaring `inputKeys()` is limitable. As of this
task's authoring that's: `places` (`resolve.job.ts`), `perfumes` (`find-url.job.ts`),
`projects-sync` (`github-sync.job.ts`), and `stocks-sync` (`stocks-snapshot.job.ts`). Confirm this
list is still current with:

```bash
grep -rl "inputKeys" src/jobs --include="*.job.ts" | grep -v /data/
```

(The `grep -v /data/` matters — `src/jobs/projects-sync/data/repos/**` contains a full nested clone
of this repo from a prior `projects-sync` run, including copies of these same `*.job.ts` files; see
the "Never let `data/` folders be scanned for code" gotcha in root `CLAUDE.md` if that's surprising.)

### 4. Trigger a real limited run together

`perfumes` is the cleanest live demo: it's the "same-key stages" example (no `rootKey` juggling to
explain away) and its per-item cost is a Fragrantica scrape + one Claude CLI call per item — cheap
relative to `places`, which burns paid Google Places / Gemini quota per item. Suggested script:

1. Open the `perfumes` workflow detail page on the real dashboard. Confirm the "Limit" number box
   is present next to "▶ Run now" (proves `limitable: true` reached the UI).
2. Pick `limit = 1` and run it. Watch the run's logs for the root-selection line
   (`runWorkflow` logs which root(s) it selected) and confirm exactly one root's whole DAG fan-out
   ran to completion (not just the first stage).
3. Inspect the DB directly to see the lineage columns land as documented:
   ```bash
   sqlite3 data/jobs.db "SELECT job_name, item_key, root_key, parent_key, status FROM work_items WHERE root_key = '<the selected root>' ORDER BY job_name;"
   sqlite3 data/jobs.db "SELECT id, run_limit, selected_roots FROM workflow_runs ORDER BY started_at DESC LIMIT 1;"
   ```
   Confirm `root_key` is identical across every stage's row for that item (same-key workflow), and
   that `selected_roots` on the run row is a 1-element JSON array matching it.
4. Run the SAME limited run again (`limit = 1`). Confirm it picks a **different** root this time
   (the first one is now fully done through the terminal stage, so `isRootPending` excludes it) —
   this is the T163 "propagation through the terminal stage" behavior made concrete.
5. Optional, if time allows: run with a limit larger than the number of remaining pending roots (or
   after everything is already done) and confirm the WARN log described in `CLAUDE.md` ("0
   originating inputs were selectable — N candidate(s), all already complete") actually appears
   instead of a silent false-success.

### 5. Close the loop

Ask directly: does the mechanism as demonstrated match what the owner expected from reading the
docs? Anything surprising, missing, or worth changing?
