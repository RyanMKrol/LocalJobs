## Overview

`src/services/dynamodb.service.ts` exports three write-capable helpers — `dynamoPut`, `dynamoDelete`,
`dynamoBatchWrite` — but a fresh repo-wide grep confirms nothing in production code calls any of them
today (the only real call site, `src/workflows/perfumes/lib.ts`'s `loadPerfumes()`, uses exclusively
the read-only `dynamoScan`). Per explicit owner request, neuter the three write functions so they
cannot be silently wired up and used by a future job without the owner first making a deliberate,
explicit decision to re-enable them — mirroring this repo's existing hard rule that broker/trading
APIs must stay strictly read-only (see root `CLAUDE.md`'s "🚫 Broker / trading APIs are READ-ONLY,
always" section and `src/services/CLAUDE.md`'s local reinforcement of it), but scoped here to three
named functions in one shared service file rather than a whole external integration.

  `dynamodb.service.ts` itself and `dynamodb.service.test.ts` — no production call site exists, so
  this change is safe with no downstream callers to update.

## Do

1. **Neuter `dynamoPut`, `dynamoDelete`, and `dynamoBatchWrite`** so each throws an explicit, clear
   error immediately instead of performing the AWS call — e.g.:
   ```ts
   export async function dynamoPut(
     table: string,
     item: Record<string, unknown>,
   ): Promise<void> {
     throw new Error(
       'dynamoPut is disabled — DynamoDB write functions are policy-gated off in this repo ' +
       '(see CLAUDE.md / src/services/CLAUDE.md). Explicit owner sign-off is required before ' +
       're-enabling this function.',
     );
   }
   ```
   Apply the same pattern to `dynamoDelete` and `dynamoBatchWrite`, each with a message naming the
   specific function. Keep the functions' existing signatures/exports unchanged (so a re-enable later
   is a small, reviewable diff — just restoring the body) — do NOT remove the functions, delete their
   exports, or strip the AWS SDK write-command imports (`PutCommand`, `DeleteCommand`,
   `BatchWriteCommand`) from the top of the file; leaving the imports in place keeps the "just restore
   the body" reversibility promise honest (an unused-import lint/type error is acceptable to leave as
   a visible reminder these are currently unused — do not silence it by removing the imports). Do NOT
   touch `dynamoGet`, `dynamoQuery`, or `dynamoScan` — they must keep functioning exactly as before.
   `dynamoBatchWrite`'s existing empty-array no-op (`if (items.length === 0) return;`) should be
   REMOVED (or moved after the throw, whichever reads more clearly) so that calling it with an empty
   array ALSO throws — the point is that no code path through these three functions ever reaches
   `getClient().send(...)`, not even a no-op fast path.

   - Change `'dynamoDelete calls send without throwing'` (line 86) similarly.
   - Update the two `dynamoBatchWrite` tests (lines 52-64) so both the over-25-items case and the
     empty-array case assert a rejection with the disabled-error message, rather than asserting the
     old max-25 / no-op behavior.
   - Leave every other test (`dynamoGet`, `dynamoQuery`, `dynamoScan`, the service-shape tests)
     unchanged — they cover the read-only paths, which are untouched.

3. **Document the policy** in `src/services/CLAUDE.md`, alongside the existing "🚫 Broker / trading
   APIs are READ-ONLY, always" section — add a short new section (e.g. "🚫 DynamoDB write functions

   `dynamodb.service.ts` are intentionally neutered (throw on call) until the owner explicitly
   re-enables them; only the read helpers (`dynamoGet`/`dynamoQuery`/`dynamoScan`) are live; re-
   enabling requires restoring the real function bodies as a deliberate, reviewed change, not a
