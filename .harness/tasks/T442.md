## Overview

Build a new, real `local-jobs` workflow, `plex-language-fix`, whose first stage scans the whole Plex
library and works out — for every show/movie whose TRUE original language (from TMDB) isn't English —
which audio/subtitle track SHOULD be selected by default, versus what's currently selected. This
replaces Plex's single global default-audio-language setting with a per-title, original-language-aware
default. This task builds the scan/report half only (read-only, no Plex mutation) as a proper,
scheduled workflow stage — porting and cleaning up validated logic from a one-off exploration script,
not designing the approach from scratch.

## Do

There is a **gitignored, repo-root scratch script** at `/plex-language-fix/` (NOT under `src/`) —
`lib.ts`, `types.ts`, `report.ts` — that already implements and VALIDATES this exact scan against the
owner's real Plex server (a live run found 1,758 proposed changes, 500 already-correct, 1,886 with no
track in the target language, and 23 "ambiguous" ties across 2,184 titles). Read those three files
first; port their logic into a real workflow rather than re-deriving it, but **do not just wrap/import
the scratch script** — rebuild it properly inside this repo's workflow conventions from that reference.

**Known bug to fix while porting:** the scratch `lib.ts`'s `pickAudioCandidate` flags a genuine
channel-count tie as `status: 'ambiguous'` and skips it. **The owner has explicitly said ambiguous
flagging should NOT exist in the final workflow** — every tie must resolve via the existing
best-judgment heuristic (highest channel count, tied-broken by lowest stream index — already
implemented, just currently gated behind the ambiguous branch) and be treated exactly like any other
`'change'`. When porting, **delete the `ambiguous` status entirely** — `FileStatus` should have only
`'change' | 'already-correct' | 'no-match'`, not 4 values.
