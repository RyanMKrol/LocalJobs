## Overview

Add a new, single-stage `plex-profiles` workflow that scans the owner's entire Plex library
(movies AND TV shows) and writes one markdown profile file per title to `data/out/`, sourced
purely from data Plex's own API already exposes — no LLM involved in this task. This is
deliberately phase 1 of a two-phase idea: phase 2 (optionally layering Claude-authored narrative
commentary on top, later, as its own follow-up task) is explicitly OUT OF SCOPE here.

## Do

**Context — this is a NEW, STANDALONE workflow**, not a stage bolted onto any of the four existing
Plex-touching workflows (`missing-tv-seasons`, `movie-recommendations` at `src/workflows/movies/`,
`tv-recommendations` at `src/workflows/tv-recs/`, `plex-space-saver`) — none of them cover BOTH
movies and shows with a per-title markdown output, and this workflow's scope (a markdown profile
of every title) is genuinely distinct from what any of them do (season-completeness audit,
franchise-gap/recommendation digests, disk-size breakdown).

**Confirmed live against the real Plex API this session** (via the shared `plexGet` helper in
`src/core/plex-client.ts`) — use these exact field names, don't re-derive them from scratch:

`GET /library/sections/<movieSectionId>/all` (movie list) returns, per item, at minimum:
`ratingKey`, `guid`, `slug`, `studio`, `title`, `titleSort`, `contentRating`, `contentRatingAge`,
`summary`, `rating` (critic 0-10), `audienceRating` (0-10), `year`, `tagline`, `duration` (ms),
`originallyAvailableAt`, `addedAt`/`updatedAt` (unix epoch seconds), `Media[]` (each with `Part[]`,
each `Part` carrying `size` in bytes — same field `plex-space-saver`'s `itemBytes()` already
sums), `Genre[]`, `Country[]`, `Director[]`, `Writer[]`, `Role[]` (cast) — each of the last 4 an
array of `{ tag: string }` (see `tags()` extraction in `src/workflows/movies/movies.ts` /
`src/workflows/tv-recs/tv-shows.ts` for the exact established pattern to copy).

`GET /library/metadata/<ratingKey>` (single-item DETAIL fetch — genuinely new territory: no
existing workflow calls this endpoint today, only the list endpoint above) returns a strict
superset of the list fields for that one item, PLUS: `Guid[]` (an array of `{ id: string }` where
`id` looks like `"imdb://tt28658276"` / `"tmdb://1146556"` / `"tvdb://358850"` — extract with the
SAME `extractTmdbId(guid)` pattern already used in `src/workflows/missing-tv-seasons/plex.ts`,
`src/workflows/movies/movies.ts`, `src/workflows/tv-recs/tv-shows.ts`, generalized here to also
pull the imdb/tvdb ids the same way, not just tmdb), `Rating[]` (an array of
`{ image: string, value: number, type: 'audience'|'critic' }` — the per-source imdb/rottentomatoes/
themoviedb ratings, distinct from the single top-level `rating`/`audienceRating`), and full
per-`Media`/`Part` technical detail including a `Stream[]` array (`videoCodec`/`videoResolution`,
`audioCodec`/`audioChannels`, language, etc. — see `Media[0].videoResolution`/`.videoCodec`/
`.container` for the simple summary fields worth surfacing; don't try to capture every `Stream`
sub-field).

TV show list/detail additionally carry: `originalTitle`, `leafCount` (episode count),
`childCount` (season count), `theme` (music track path — not useful for a text profile, skip it).
Show-level `Media`/`Part`/size is NOT present on the show object itself — mirror
`plex-space-saver`'s `buildShowRows()` approach: fetch the show's episodes via
`GET /library/sections/<tvSectionId>/all?type=4` (flat episode read, grouped by
`grandparentRatingKey`) and sum each episode's `itemBytes()`-equivalent for the show's total
library size.

**1. New folder `src/workflows/plex-profiles/`** — mirror the established multi-file workflow
layout (`config.ts`, `types.ts`, `lib.ts`, `plex-profiles.workflow.ts`, `stages/`,
`*.test.ts`, own `CLAUDE.md`):

- `config.ts`: read `PLEX_MOVIE_SECTION` (default `'4'`) / `PLEX_TV_SECTION` (default `'5'`) —
  the SAME env vars `movies`/`missing-tv-seasons`/`plex-space-saver` already read, no duplication.
  Add ONE new env var: `PLEX_PROFILES_RUN_LIMIT` (default `0` = unlimited), mirroring the
  `runLimit`/`PLACES_ENRICH_RUN_LIMIT`/`PERFUMES_RUN_LIMIT` convention in
  `src/workflows/places/config.ts` / `src/workflows/perfumes/config.ts` — caps how many
  new/changed titles are built in a single run so a large first-run backlog doesn't blow the
  job's `timeoutMs`; `0` processes everything found. Output dir resolved via `resolve(here, …)`
  to `data/out/movies/` and `data/out/shows/` (two subfolders under the job's own `data/out/`).
- `types.ts`: `PlexGuid { id?: string }`, `PlexTag { tag?: string }`, `PlexStream` (minimal:
  `videoResolution?`/`videoCodec?`/`audioCodec?`/`audioChannels?`/`language?` — only what you
  actually render), `PlexPart { size?: number }`, `PlexMedia { videoResolution?, videoCodec?,
  container?, Part?: PlexPart[] }`, `PlexRating { image?: string; value?: number; type?: string }`,
  `PlexMovieDetail` and `PlexShowDetail` interfaces carrying every field enumerated above that the
  markdown template (below) actually uses, and `PlexEpisodeMeta { grandparentRatingKey?: string |
  number; Media?: PlexMedia[] }` (for the show byte-sum, same shape as
  `src/workflows/plex-space-saver/types.ts`'s `PlexEpisodeMeta`).
- `lib.ts` (pure, no I/O — unit-tested): `extractGuidIds(guids)` → `{ tmdbId, imdbId, tvdbId }`
  (parse the `imdb://`/`tmdb://`/`tvdb://` prefixes — generalizes the existing single-purpose
  `extractTmdbId` pattern), `tags(arr)` → `string[]` (copy the existing helper verbatim from
  `movies.ts`/`tv-shows.ts`), `formatBytes(bytes)` (copy `plex-space-saver/lib.ts`'s implementation
  — small, duplicated-per-workflow helper, matches this repo's existing convention rather than a
  new cross-workflow import), `itemBytes(item)` (same copy, for summing a show's episodes),
  `slugFileName(ratingKey, slug)` → e.g. `"37974-the-430-movie-2024"` (Plex already returns a
  `slug` field on both movies and shows — reuse it directly rather than re-deriving one from the
  title), `buildMovieProfileMarkdown(meta: PlexMovieDetail): string`,
  `buildShowProfileMarkdown(meta: PlexShowDetail, totalBytes: number): string`.

  **Markdown template — deterministic, no LLM, so no shape-validation is needed (unlike
  `perfumes`/`projects-sync`'s Claude-output contracts) — but keep the shape FIXED and consistent
  movie-to-movie / show-to-show** so the corpus stays scannable, roughly:
  ```
  ---
  type: movie            # or "show"
  rating_key: <ratingKey>
  title: <title>
  original_title: <originalTitle>     # shows only, omit if absent
  year: <year>
  tmdb_id: <or omit>
  imdb_id: <or omit>
  tvdb_id: <or omit>
  content_rating: <contentRating>
  genres: [<Genre tags>]
  countries: [<Country tags>]
  studio: <studio>
  added_at: <ISO date from addedAt epoch * 1000>
  updated_at: <ISO date from updatedAt epoch * 1000>
  ---

  ## Summary
  <tagline (movies only), then summary>

  ## Cast & Crew
  - Director(s): ... / Writer(s): ...   (movies only)
  - Cast: first ~10 Role tags

  ## Ratings
  - Audience rating: <audienceRating>
  - Critic rating: <rating>            (movies only, if present)
  - Per-source: one line per Rating[] entry (image/type/value)

  ## Technical
  - Duration: humanized (from duration ms)
  - Movie: resolution/codec/container from Media[0], file size via formatBytes(Part[0].size)
  - Show: season count (childCount), episode count (leafCount), total library size via
    formatBytes(totalBytes) (summed across all episodes' Media/Part sizes)

  ## Source
  - Plex rating key + library section
  - (movies only) file path, from Media[0].Part[0].file — omit for shows (avoid dumping every
    episode's file path into one profile)
  ```
  This is a firm STARTING template, not a pixel-perfect spec — keep the frontmatter keys and `##`
  section names stable (future queryability, mirrors why `project.template.md` fixes its headings)
  but use your judgment on exact prose/formatting within each section.

- `stages/build.job.ts` + `stages/build.ts` (single stage — no DAG edges, so per
  `src/workflows/CLAUDE.md` / the `plex-space-saver` precedent, **no gate/contracts.ts needed**):
  job name `plex-profiles-build`. Mirror `plex-space-saver`'s `runScan` shape but idempotent
  per-title rather than re-scan-fresh-every-run (this is a BUILD, like `places`/`perfumes`/
  `projects-sync`, not an audit):
  1. Fetch the movie section list + TV section (shows) list + TV section flat episode list
     (`?type=4`) via `plexGet`, exactly like `plex-space-saver`'s `runScan`.
  2. Build the show → total-bytes map from the episode list (reuse `itemBytes`/grouping by
     `grandparentRatingKey`, same as `plex-space-saver`'s `buildShowRows`).
  3. Item key convention: `movie:<ratingKey>` / `show:<ratingKey>`. **Declare `inputKeys()`** on
     the job (returning every current `movie:<ratingKey>`/`show:<ratingKey>` key) so a manual run
     can be limited (T094), mirroring `places`'s `resolveInputKeys()` — this is a build workflow
     with potentially a large first-run backlog (the owner's whole library), exactly the situation
     the run-limit mechanism exists for.
  4. For each key (filtered by `ctx.rootAllowed(key)`, applying `PLEX_PROFILES_RUN_LIMIT` per the
     places/perfumes `runLimit` convention): check the `work_items` ledger — skip (log why) when a
     `success` row already exists AND its stored `detail.updatedAt` marker equals the item's
     CURRENT `updatedAt` from the list fetch (an unchanged-since-last-build skip, exactly mirroring
     `project-summarize`'s `pushedAt`-marker idiom in `src/workflows/projects-sync/` — no separate
     calendar-based skip). A brand-new key (no ledger row) or one whose `updatedAt` has moved
     always rebuilds.
  5. For each key that needs (re)building: call `GET /library/metadata/<ratingKey>` for the full
     detail object, build the markdown via `buildMovieProfileMarkdown`/`buildShowProfileMarkdown`,
     write it to `data/out/movies/<slugFileName>.md` / `data/out/shows/<slugFileName>.md`
     (`mkdirSync(..., {recursive:true})` first), then `markWorkItem(ctx, key, 'success', { detail:
     { name: title, markdown: <absolute path>, updatedAt: <the item's updatedAt> } })` — the
     `detail.markdown` convention (T110) means this surfaces automatically in the workflow's
     generic unified Output section (T205) with zero dashboard changes.
  6. **Item-loop progress**: `ctx.progress(i/total*100)` + a per-item `ctx.log` line as each
     title finishes (built vs skipped-unchanged), per the "item-loop jobs report progress per item"
     convention — use `Math.min(todo.length, runLimit || todo.length)` as the denominator.
  7. Verbose logging throughout per the root `CLAUDE.md` logging convention: what's about to
     fetch, counts per section, each item's decision (build/skip + why), and a detailed final
     summary (built count, skipped-unchanged count, output dir).
