## Overview

All four `perfumes` stages (`perfumes-find-url`, `perfumes-fetch`, `perfumes-parse`,
`perfumes-build`) already compute a per-run tally of `{ ok, failed, pending, rateLimited }`
(the `StageResult` shape in `src/workflows/perfumes/types.ts`) and return it from their
`run*` implementation function — but every `.job.ts` wrapper calls that function and
**discards the return value**, so a run in which every single item failed still ends
with `run()` resolving normally and the job/run recorded `'success'`. This is a live,
already-observed incident: `perfumes-build`'s cycle 1 logged `1/1 · 0 ok, 1 failed` yet
the run reported success (only cycle 2's coincidental retry masked it). Apply the T394
convention (to be formalized in root `CLAUDE.md` by T416): an item-loop job must `throw`
at the end of its own run if this run's own tally shows `failed > 0`, so the run is
correctly marked failed, downstream DAG dependents correctly block, and the job's own
`maxRetries` budget correctly kicks in (a retry skips already-done items and resumes at
the first not-yet-done one — no extra logic needed, this is pure `isWorkItemDone`
behaviour already in place).

## Do

The per-item `try/catch` / `markWorkItem(..., 'failed', ...)` / loop-`continue` pattern
inside each stage's item loop is UNCHANGED — one bad item must still not stop the loop
from attempting every other item this run. Only the FINAL decision (throw vs. return)
changes, and it belongs in each `.job.ts` wrapper (not the stage `.ts` impl) since the
wrapper is the one place that currently calls-and-discards the result and is the
natural single point of control over the job's `run()` outcome:

- **`src/workflows/perfumes/stages/find-url.job.ts:19`** — currently:
  ```ts
  async run(ctx) { await runFindUrl(ctx); },
  ```
  Change to capture the `StageResult` and throw when `failed > 0`:
  ```ts
  async run(ctx) {
    const result = await runFindUrl(ctx);
    if (result.failed > 0) {
      throw new Error(`${result.failed}/${result.ok + result.failed} perfume(s) failed to find a Fragrantica URL this run — see logs above`);
    }
  },
  ```
  Do NOT special-case `rateLimited` here — a rate-limited stop can coexist with
  `failed > 0` from items that genuinely failed *before* the rate limit was hit; those
  still deserve a throw. A `rateLimited: true` run with `failed === 0` must NOT throw
  (it's a soft defer-to-later-run, same as a `'skipped'` work item — not a failure).

- **`src/workflows/perfumes/stages/fetch.job.ts:12`** — currently:
  ```ts
  async run(ctx) { await runFetch(ctx); },
  ```
  Change identically:
  ```ts
  async run(ctx) {
    const result = await runFetch(ctx);
    if (result.failed > 0) {
      throw new Error(`${result.failed}/${result.ok + result.failed} page(s) failed to fetch this run — see logs above`);
    }
  },
  ```
  Note `runFetch` (`src/workflows/perfumes/stages/fetch.ts:32`) already has an early-return
  path (`if (todo.length === 0) return { ok: 0, failed: 0, pending: 0, rateLimited: false };`)
  — that's `failed: 0`, so it naturally doesn't throw. No change needed there.

- **`src/workflows/perfumes/stages/parse.job.ts:12`** — currently:
  ```ts
  async run(ctx) { await runParse(ctx); },
  ```
  Change identically:
  ```ts
  async run(ctx) {
    const result = await runParse(ctx);
    if (result.failed > 0) {
      throw new Error(`${result.failed}/${result.ok + result.failed} page(s) failed to parse this run — see logs above`);
    }
  },
  ```

- **`src/workflows/perfumes/stages/build.job.ts:11`** — currently:
  ```ts
  async run(ctx) { await runBuild(ctx); },
  ```
  Change identically — **this is the stage with the confirmed live incident**
  (cycle 1 logged `1/1 · 0 ok, 1 failed` but the run/cycle reported success):
  ```ts
  async run(ctx) {
    const result = await runBuild(ctx);
    if (result.failed > 0) {
      throw new Error(`${result.failed}/${result.ok + result.failed} profile(s) failed to build this run — see logs above`);
    }
  },
  ```

None of the four stage `.ts` implementation files (`find-url.ts`, `fetch.ts`, `parse.ts`,
`build.ts`) need to change — each already computes and returns the correct `StageResult`;
the bug is purely that the wrapper never looks at it.

## Done when

- Each of the four `.job.ts` files has a test (new `*.job.test.ts` alongside the existing
  `stages/*.test.ts` files, or an addition to an existing test file if more natural)
  proving: given a stage run whose returned `StageResult` has `failed > 0`, the job's
  `run(ctx)` throws (use `perfumesConfig.dryRun` / a stubbed `runClaude`/`callService`

  stages without live network/Claude calls). At least one test must also prove the
  inverse: `rateLimited: true` with `failed === 0` does NOT throw (the soft-defer case
  stays a normal return).
- Manual/log spot-check: re-run (or point at a scratch DB fixture reproducing) the exact
  `perfumes-build` scenario that motivated this task — a cycle where the stage's own log
  shows `N/N ... 0 ok, N failed` — and confirm the run/member job now shows `'failed'`,
  not `'success'`.
