## Overview

`tmdb-season-check` (`src/workflows/missing-tv-seasons/stages/season-check.ts`, the `missing-tv-seasons` workflow's
stage 2) runs a genuine per-show processing loop against TMDB, but its `catch` block treats EVERY
non-quota error identically — a deliberate, documented design ("One bad show must not fail the
whole audit") that logs a `warn` and moves on, with no tally and no way for the run to ever fail.
That collapses two very different situations into one: a single show genuinely having no usable
TMDB match (fine, expected, shouldn't fail the run) vs. a systemic problem — TMDB API drift, an
auth/network fault, a bug in `evaluateShow` — silently affecting many/all shows, which should be
visible as a failed run. Per the newly-decided convention (T394), an item-processing loop must
`throw` a summarizing `Error` at the end of `run()` if this run's own tally shows genuine
per-item failures (excluding intentional soft-stops like the existing `QuotaExceededError` break),
so the run is correctly marked failed, downstream (`plex-seasons-notify`) is blocked from acting on
an incomplete/degraded scan, and the stage retries via its own `maxRetries` budget.

A second, related gap exists one stage later: `plex-seasons-notify` (`src/workflows/missing-tv-seasons/stages/notify.ts`)
sends ONE digest push after building it, but if that push fails (`res.ok === false`) it only logs an
`error`-level line — it still unconditionally marks every newly-detected `(show, season)` pair
`'success'` in the ledger and returns normally. That means a failed delivery is silently swallowed
into an overall-success run AND the ledger now believes those seasons were already notified, so they
will never be retried/re-sent even though the owner was never actually told about them. This is the
same "don't let something fail this run without the run itself failing" principle T394 codifies.

### 1. `src/workflows/missing-tv-seasons/stages/season-check.ts` (job `tmdb-season-check`) — tally genuine per-show failures and hard-fail

In `runSeasonCheck`, the `catch` block (lines ~78-86) currently does:

```ts
} catch (err) {
  if (err instanceof QuotaExceededError) {
    ctx.log(`tmdb ${err.window} cap reached (${err.used}/${err.cap}) — stopping gracefully; next run resumes.`, 'warn');
    break;
  }
  const msg = err instanceof Error ? err.message.split('\n')[0] : String(err);
  // One bad show must not fail the whole audit — log + skip it.
  ctx.log(`  ✗ "${show.title}" (tmdb=${show.tmdbId}) — ${msg}`, 'warn');
}
```

Add a `let failed = 0;` counter alongside the existing `checked`/`tmdbCalls` counters (near line
46), increment it in the non-quota branch of the catch (`failed++;` right after the existing
`ctx.log('  ✗ ...')` line — keep that log, it's the per-item narration), and, at the very end of
`runSeasonCheck` — after the existing "SEASON-CHECK SUMMARY" logging block, right before the
function implicitly returns (after the closing `ctx.log('═════...')` line) — add:

```ts
if (failed > 0) {
  throw new Error(`${failed}/${shows.length - unverifiable.length} show(s) failed their TMDB check this run — see logs above`);
}
```

Do NOT touch: the `show.tmdbId == null` branch (an expected, non-failure "unverifiable" case, not a
failure — must NOT count toward `failed`), and the `QuotaExceededError` branch (an intentional
soft-stop that already `break`s the loop — must NOT count toward `failed` either). Keep the
"missing-seasons.json" write (`writeJsonFile(plexConfig.missingOut, out)`) happening BEFORE the new
throw, exactly like it does now — the file is still written with everything successfully checked so
far even on a failed run, only the run's own status changes.

### 2. `src/workflows/missing-tv-seasons/stages/notify.ts` (job `plex-seasons-notify`) — don't mark undelivered notifications as notified

In `runNotify`, after building and sending the digest (lines ~96-108):

```ts
const res = await pushFn(digest.title, digest.body, { job: 'plex', tags: 'tv', priority: 'default' });
ctx.log(res.ok ? `digest push sent — ${digest.title}` : `digest push FAILED (${res.error})`, res.ok ? 'info' : 'error');

for (const s of newShows) {
  for (const season of s.seasons) {
    markWorkItem(NOTIFY_JOB, pairKey(s.tmdbId, season), 'success', {
      detail: { name: `${s.title} S${season}`, markdown: reportPath },
    });
  }
}
```

Change this so the ledger is only marked notified when the push actually succeeded, and the run
fails (loudly, retryably) when it didn't:

```ts
const res = await pushFn(digest.title, digest.body, { job: 'plex', tags: 'tv', priority: 'default' });
if (!res.ok) {
  ctx.log(`digest push FAILED (${res.error}) — NOT marking ${totalNew} season(s) notified, so they're retried next run.`, 'error');
  throw new Error(`Digest push failed — ${res.error}`);
}
ctx.log(`digest push sent — ${digest.title}`);

for (const s of newShows) {
  for (const season of s.seasons) {
    markWorkItem(NOTIFY_JOB, pairKey(s.tmdbId, season), 'success', {
      detail: { name: `${s.title} S${season}`, markdown: reportPath },
    });
  }
}
```

The markdown report (`writeReport`) should still be written before the push attempt exactly as it
is today (unchanged) — only the ledger-marking + final outcome changes on a push failure.
