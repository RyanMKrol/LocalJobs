## Do

Build a unified **"output" section** on every workflow's detail page
(`dashboard/app/workflows/[name]/page.tsx`) — a single, consolidated, de-duplicated view of
everything that workflow has produced. This is a **straight build, NOT a prototype/chooser**: there
are NO switchable options and NO live toggle — implement ONE good consolidated view and ship it.

The model already exists in miniature: the movie-recommendations recommendation cards on
`/workflows/movie-recommendations`. Generalize that idea so **every** workflow detail page gets an
output section:
- **missing-tv-seasons** → all currently-missing TV seasons in one list/section.
- **movie-recommendations** → the recommendations (and franchise gaps) it has produced.
- **places** → the place profiles it has produced (link the markdown artifact via the existing
  `workItemMarkdownPath` / `GET /api/workflow-runs/:id/output` machinery).
- **perfumes** → the perfume profiles it has produced.

Hard requirements for the view:
- **Scoped to the workflow.** It shows only that workflow's output, on that workflow's detail page.
- **De-duped by a stable per-item key — NOT "latest run".** Output repeats across runs (the same
  movie recommended again next month, the same place re-enriched). Each item must appear **once**,
  collapsed by a stable per-item key, never as duplicate rows from multiple runs.
- **A per-workflow way to know the stable key.** Establish a simple, documented convention for what a
  workflow's "output items" are and what their stable key is. The natural fit in this codebase is the
  **terminal stage's `work_items` ledger**, which is already keyed uniquely by `(job_name, item_key)`
  — so `item_key` IS the stable de-dupe key, and the cumulative ledger is already de-duped by
  construction. Use that (or an equally-simple convention); do NOT build a heavyweight new framework.
  Document the chosen convention in `CLAUDE.md` so current and future workflows expose output the same
  way.
- Feed the view from a read-only endpoint (extend `src/api/server.ts`, query via `src/db/store.ts`);
  add the client call in `dashboard/app/lib/api.ts`. Keep it **mobile-responsive** and
  **theme-variable-driven** (all colours/fonts from the CSS custom properties in `globals.css`).

Scope: `dashboard/app/components/**`, `dashboard/app/workflows/[name]/page.tsx`,
`dashboard/app/lib/api.ts`, `dashboard/app/globals.css`, `src/api/server.ts`, `src/db/store.ts`,
`CLAUDE.md`.

## Done when

- Every workflow's detail page shows a consolidated **output section** scoped to that workflow,
  rendering its produced items as a unified card/list view (e.g. all missing TV seasons on
  `/workflows/missing-tv-seasons`).
- Items are **de-duped by a stable per-item key**, so output repeated across runs appears exactly
  once (verified for at least two example workflows — e.g. movie-recommendations + missing-tv-seasons).
- The per-workflow output / stable-key convention is implemented as a documented, reusable mechanism
  and described in `CLAUDE.md`.
- The view is mobile-safe and theme-driven. `npm --prefix dashboard run build`, `npx tsc --noEmit`,
  `npm test`, and `node dashboard/scripts/mobile-check.mjs` are green.
