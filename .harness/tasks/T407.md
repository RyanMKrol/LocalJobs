## Overview

The `perfumes` workflow's `fetch` and `parse` stages each write a real artifact to
disk (a captured Fragrantica page, a structured JSON extraction) but their success
`markWorkItem` calls only record `{ name }` — so the dashboard's Inputs & Outputs
panel can't show what either stage actually produced. Point each stage's `detail` at
its real output, per the "every stage's success `detail` must describe what it
produced" convention in the root `CLAUDE.md`.

  matched && !blocked`) already writes `data/out/pages/<id>.txt` (line 55) and
  `data/out/pages/<id>.html` (line 61) via `perfumesConfig.pagesDir` (see

  re-derive accord bar widths and isn't itself the "output" a human would want to
  preview. `textLength`/`htmlBytes` mirror the numbers already logged at
  `fetch.ts:64` so the ledger doesn't just point at a file but also gives a

- **`src/workflows/perfumes/stages/parse.ts:44`** — the success branch writes
  `data/out/fragrantica/<id>.json` at line 43 via `perfumesConfig.fragranticaDir`
  (`config.ts:18`), containing the Claude-extracted accords/notes/rating/votes/
  review themes (see `parsePrompt`, lines 59-84). Change:
  ```ts
  // old
  markWorkItem(PARSE_JOB, p.id, 'success', { attempts, detail: { name: label(p) } });
  // new
  const accordsCount = Array.isArray((data as { accords?: unknown }).accords)
    ? (data as { accords: unknown[] }).accords.length
    : 0;
  const notesTiers = (data as { notes?: { top?: unknown[]; heart?: unknown[]; base?: unknown[] } }).notes;
  const notesCount = notesTiers
    ? (notesTiers.top?.length ?? 0) + (notesTiers.heart?.length ?? 0) + (notesTiers.base?.length ?? 0)
    : 0;
  markWorkItem(PARSE_JOB, p.id, 'success', {
    attempts,
    detail: {
      name: label(p),
      format: 'json',
      path: join(perfumesConfig.fragranticaDir, `${p.id}.json`),
      accordsCount,
      notesCount,
      rating: (data as { rating?: number | null }).rating ?? null,
    },
  });
  ```
  Compute `accordsCount`/`notesCount`/`rating` from the already-normalized `data`
  object (after `applyAccordPercents`/`applyNormalizedNotes` have run, so `notes`
  is always `{top,heart,base}` arrays by this point) rather than re-reading the
  file. This surfaces the genuinely interesting "what did parsing extract" numbers
  directly in the ledger, not just a file pointer. Leave the FAILURE branch
  (`parse.ts:49`) untouched — `{ name, error }` is adequate for a parse failure.
