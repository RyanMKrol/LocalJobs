## Do

Make the autonomous loop's `claude` invocation **stream its output live** so a long run is
observably progressing, instead of looking frozen for 10–20 minutes.

**Background / problem.** `loop.sh` runs the builder as `claude -p … "${FLAGS[@]}"`, where
`CLAUDE_FLAGS` currently defaults to just `--dangerously-skip-permissions`
(`.harness/harness.env`, the `: "${CLAUDE_FLAGS:=--dangerously-skip-permissions}"` line). With no
streaming flag, `claude -p` **buffers all output until it exits**, so the tee'd
`.harness/worklog/.claude-out` stays **0 bytes** for the whole build and the `supervise.sh` terminal
shows nothing. Compounding it, the `claude` parent process sits near 0% CPU (the heavy work — `tsc`,
`npm test`, the dashboard build — runs in spawned child processes), so `ps` also looks idle. The
only current way to confirm a run is alive is to watch Claude's own JSONL session transcript under
`~/.claude/projects/-Users-ryankrol-Development-local-jobs/*.jsonl` grow — not ergonomic.

**This is a needs-human task because** it edits the loop's OWN invocation (a bad change degrades or
breaks visibility for *every* future run) and the output-format choice is an owner decision.

**Decide** between:
- `--verbose` — human-readable streaming in the terminal (lighter touch, **recommended**); or
- `--output-format stream-json` — machine-parseable but noisy in a terminal.

**Then apply** the chosen flag by adding it to the `CLAUDE_FLAGS` default in `.harness/harness.env`,
keeping the existing overridable form (`: "${CLAUDE_FLAGS:=--dangerously-skip-permissions <newflag>}"`),
so `.harness/worklog/.claude-out` fills live and `tail -f .harness/worklog/.claude-out` (and the
supervise terminal) show real-time progress.

## Done when

- `CLAUDE_FLAGS` in `.harness/harness.env` includes the chosen streaming flag, still in the

- During a real loop build, `.harness/worklog/.claude-out` is observed to **grow incrementally**
  (not stay 0 bytes until the end), i.e. `tail -f .harness/worklog/.claude-out` shows live activity.
- **Rate-limit detection still fires.** Confirm the streamed output still contains the usage/session

  10 on a limit (and `rl_reset_wait` still parses the reset time). Record in the worklog how this was
  confirmed (e.g. inspected a captured limit message, or reasoned from the chosen format's output).
