## Overview

Add a new `overall=true` mode to `GET /api/workflow-runs/:id/stage-io` that shows a workflow's OWN root/idempotency key(s) as "inputs" and its terminal output(s) as "outputs" — the owner's phrase: "the input should be the item potency key or the ledger key, and the output should be the output for that item potency key." This reuses the generalized `stageIoLists` from T383 (which must be done first), scoped to the workflow's DAG root wave and terminal wave instead of one stage's direct predecessors.

## Do

In `src/api/server.ts`, extend the existing `GET /api/workflow-runs/:id/stage-io` handler (around line 1912) to accept an `overall=true` query flag as an ALTERNATIVE to `?job=<job>`. When `overall=true`:

1. Resolve the run's workflow definition (`getWorkflowJobs(run.workflow_name)`, `buildDag(refs)`) exactly as the existing `/io` endpoint handler does (around line 1845-1853) to get `dag.waves[0]` (the root/first wave — these jobs' ledger rows are the workflow's own "potency key" / idempotency root) and `dag.waves[dag.waves.length - 1]` (the raw last wave).
2. Resolve the EFFECTIVE terminal job(s) the SAME way the unified Output section already does (`src/api/server.ts` around line 1544-1546: `const outputJobs = def?.outputJob && refs.some((r) => r.job === def.outputJob) ? [def.outputJob] : lastWave;`) — so a workflow with an `outputJob` manifest override (e.g. `stocks-sync`, whose true DAG terminal `stocks-notify` never records ledger rows) shows the SAME effective terminal stage here as it does in the unified Output section, not the raw DAG last wave.
3. Call the generalized `stageIoLists(outputJobs, dag.waves[0], run.id)` from T383 and return `{ inputs, outputs, predecessorJobs: dag.waves[0], outputJobs, job: '__overall__' }` (the `job` field distinguishes this from a real job name for any caller that logs/displays it).

`?job=<job>` (no `overall`) must keep working EXACTLY as it does today (T383 already made this the default path through the generalized function). `overall=true` together with `?job=` should be rejected with 400 (mutually exclusive) or `overall` should simply take precedence and ignore `job` — pick one and note it in a comment; either is acceptable, just be consistent and don't silently merge both modes' logic into an ambiguous third behavior.

## Done when

- A NEW test in `src/api/server.test.ts` proves the `outputJob` override is honored: build a fixture workflow (mirroring `src/jobs/stocks-sync/stocks-sync.workflow.ts`'s `outputJob: 'stocks-snapshot'` pattern — see root `CLAUDE.md`'s "outputJob override" note) where the raw DAG last wave records NOTHING but an earlier `outputJob`-named stage does, and assert `overall=true`'s `outputs` reflects the OVERRIDE job's rows, not an empty raw-last-wave result.
- A second test proves `overall=true`'s `inputs` matches `dag.waves[0]`'s ledger rows for a 3+ stage fan-in workflow (reuse the `sio-api-wf` fixture from `src/api/server.test.ts` if still present after T383, or an equivalent).
