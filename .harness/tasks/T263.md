## Do

Build a new **Plex "space saver" workflow** that scans the Plex library, computes how much disk each
title uses, and outputs a **biggest-first size breakdown** so the owner can see where space is going.
Create a new tracked workflow folder `src/jobs/plex-space-saver/` (published like the other Plex
example workflows; its `data/` stays gitignored, so the generated report is NEVER committed). Reuse
the shared Plex client `src/jobs/plex/client.ts` (`resolvePlexHost`/`plexGet` — DHCP self-heal, T149)
and the existing Plex env (`PLEX_HOST`/`PLEX_API_TOKEN`/optional `PLEX_MACHINE_ID`). Plex's API exposes
per-item media `Part` `size` bytes (via `/library/sections/<id>/all` + per-item metadata), so size is
obtained via the API (no filesystem walk).

Granularity (decided): **one row per title** — each **movie** stands alone (sum its media parts);
each **TV show** is a single row summing **every episode across every season**. Sort biggest-first,
show human-readable sizes (GB). **Report only** — do not flag or suggest deletions. Schedule
**weekly**. Idempotent/re-scan style like missing-tv-seasons (re-reads the library each run; it's an
audit, not a build).

Output: a size-breakdown artifact in the workflow's gitignored `data/out/`, surfaced in the workflow
detail page's Output section. Render it via the **declared output form** mechanism from T262 (a
structured size view / table — biggest-first), NOT plain prose; this workflow is the first concrete
consumer of T262's new form. (If a structured form is impractical, a size-sorted markdown table is the
fallback — but prefer the T262 form.)
