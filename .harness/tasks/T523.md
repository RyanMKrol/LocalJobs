## Overview
Make the workflow Run button feel instant: on click, immediately flip it to RunButton's existing disabled 'Started…' state before the API round-trip resolves, instead of waiting ~5s for the poll to observe last_run.status === 'running'.

## Do
RunButton (dashboard/app/components/RunButton.tsx) ALREADY supports an optimistic in-flight state: its `busy` prop shows a disabled 'Started…' label, distinct from the poll-derived `isRunning` ('Running…'). No change to RunButton.tsx itself is needed — the fix is to WIRE `busy` on the two surfaces that trigger runs. (1) Workflows list — dashboard/app/workflows/page.tsx (around line 125): the RunButton passes only `isRunning={p.last_run?.status === 'running'}` and calls `run(p.name)` (which fires api.runWorkflow and swallows errors). Add per-workflow busy tracking (e.g. a `useState<Set<string>>` or a `Record<string, boolean>` keyed by workflow name): set the workflow's busy true synchronously on click, await the api.runWorkflow call in a try/finally, and clear busy in `finally` (so it reverts on error and hands off to the poll-driven isRunning once a run row appears). Pass `busy={<isBusy(p.name)>}` to its RunButton. (2) Overview cards — dashboard/app/page.tsx (around line 196): the card's RunButton currently hardcodes `isRunning={false}` with no busy at all, and `runWorkflow(name)` swallows errors. Apply the same busy-state pattern there so clicking a card's Run button immediately shows 'Started…'. Keep the existing behaviour that the next poll surfaces the real running/last_run state. This is the whole fix — no genuinely-slow endpoint was found (the ~5s lag is purely poll latency), so no chooser/review pair is warranted and no endpoint/poll-interval change is made. Build the dashboard, run node dashboard/scripts/mobile-check.mjs, and run node dashboard/scripts/visual-check.mjs (baseline screenshots of 'workflows' and 'overview' still render; the busy state is transient/interaction-driven and needn't be captured as a new FLOW unless trivial to add). LOOK at the screenshots to confirm the Run buttons still render correctly.

## Done when
Clicking a workflow's Run button on BOTH the Workflows list and the Overview cards immediately flips it to the disabled 'Started…' (busy) state before the API call resolves, reverting on error (busy cleared in finally) and handing off to the poll-driven 'Running…' state once the run is observed; RunButton.tsx is unchanged (only its existing `busy` prop is now wired); and dashboard build + mobile-check + visual-check are green with the 'workflows' and 'overview' screenshots still rendering the Run buttons correctly.
