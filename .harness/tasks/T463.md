## Overview
Give Plex the same top-level, self-contained service entry every other shared external dependency (Gemini, Google Places, Fragrantica, Claude CLI, TMDB, ...) already has, so it shows up on the dashboard's Integrations page and gets automatic call-count metering + per-job consumer tracking through `callService`, even though — unlike those paid/rate-limited APIs — it is the owner's own local LAN server with no real rate limit or spend quota to enforce.

## Do
Read `src/services/tmdb.service.ts` and `src/services/google-places.service.ts` as the closest worked examples (a small, self-contained `ServiceDefinition`), and `src/core/plex-client.ts` for the connectivity it fronts. Create `src/services/plex.service.ts` default-exporting a `ServiceDefinition` named `'plex'`: `description` explaining it's the owner's local Plex Media Server (not a paid/rate-limited API); `category: 'api'`; `paid: false`; deliberately OMIT `ratePerMinute`/`dailyCap`/`monthlyCap`/`minIntervalMs`/`maxJitterMs` (a local LAN server has no meaningful rate limit or spend quota to model — `callService`'s existing 'no limit configured' branch already falls through to just metering + recording the call, which is exactly the visibility win wanted here); set `rateLimitSource` to something honest, e.g. 'Local LAN server the owner runs — no external rate limit or quota applies; this service exists purely for call-count visibility and per-job consumer tracking on the Integrations page.' Add a small `plex.service.test.ts` (mirroring an existing `*.service.ts` test if one exists, or a minimal shape/import test if not) confirming the definition loads and has the expected `name`/`paid`/`category` and no rate/quota fields set. Do NOT touch any call site yet (that's a separate dependent task) — this task only adds the definition so the registry picks it up (it auto-discovers every `*.service.ts` under `src/services/`) and syncs a `plex` row into the `services` table on next daemon start.

## Done when
`src/services/plex.service.ts` exists, exports a valid `ServiceDefinition` named `plex` with no rate/quota fields set, `npx tsc --noEmit` and `npm test` are green, and (by inspection/registry test coverage) the service is discoverable the same way every other `*.service.ts` is — i.e. `src/workflows/registry.test.ts` or an equivalent existing registry test continues to pass with the new file present. No behavior of any existing Plex call site changes in this task.
