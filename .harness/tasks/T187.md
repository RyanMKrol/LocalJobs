## Do

NEEDS-HUMAN process design. Align with the owner on a deliberate **two-step process** for getting
work into the backlog, replacing today's habit of throwing several raw ideas at Claude Code at once
(which loses track and produces rushed, low-quality task specs now that the harness sets a high
planning bar — per-task `## Do`/`## Done when` spec, `scope`, `dependsOn`, `facets`, `verify`).

The process to design and agree on:

1. **Ideas capture surface (separate from the backlog).** A single flat Markdown file (e.g.
   `.harness/IDEAS.md` — exact location TBD with the owner) holding a plain list where the owner
   jots one or two sentences per idea, with little ceremony. Low-friction, no schema, no planning —
   just a place to dump thoughts so they aren't lost and aren't interrupting in-flight work.

2. **A deliberate idea → backlog conversion step (an INTERVIEW).** A separate, focused step that
   takes ONE idea at a time off the list and, before producing anything, interviews the owner —
   clarifying questions on definition of done, scope, edge cases / nuance, and dependencies — then
   produces a schema-correct backlog task that flows through the existing
   `ralph-loop-add-to-backlog` skill. One idea per conversion, never a bulk dump.

Open questions to settle WITH the owner (this is why it's needs-human — don't pre-decide):
- Where the ideas file lives and its exact format (free list vs. light per-idea structure).
- How the conversion step is triggered and where it's documented — a new slash-command / skill, an
  extension of `ralph-loop-add-to-backlog`, or a documented manual ritual.
- How a converted idea is marked done / removed from `IDEAS.md` (and whether converted ideas are
  archived for traceability).
- How this interacts with the existing add-to-backlog skill (interview feeds INTO it, vs. replaces
  part of it).

Output of this task is an AGREED process description (captured in this task's worklog
`.harness/worklog/T187.md`), plus — if implementation is warranted — one or more **dependent
buildable follow-up tasks** drafted to implement the agreed process (the ideas file, the conversion
command/skill, the docs). The loop SKIPS this task; the owner returns to it deliberately later.
