## Do

PROTOTYPE the **workflow graph (DAG) view** with multiple graphing approaches so the owner can pick
the best one. This IS an **experimentation/chooser** task: build **2–3 distinct options** behind a
live switch so the owner can compare them on a real workflow page, then a paired review task (T206)
picks the winner and a follow-up (T207) hardcodes it.

**Why this exists:** the current graph (`dashboard/app/components/Dag.tsx` — a hand-rolled SVG bezier
layout) does NOT represent the workflow faithfully. The specific failures to fix:
- **Concurrency isn't shown.** Stages that run in parallel (independent stages sharing a dependency,
  up to the workflow's `maxConcurrency`) aren't visually distinguishable from sequential ones.
- **Waves aren't shown.** The DAG executes in waves (each wave = the set of stages whose deps are all
  satisfied). The view should make the wave structure legible — which stages belong to the same wave
  and therefore can run together — rather than an ambiguous node soup.
- More generally, the layout often mis-renders the actual `dependsOn` structure (e.g. the
  movie-recommendations fan-out of `franchise-gaps` + 8 recommender branches off `movie-snapshot`).

Each option must, for a given workflow:
- Render the real DAG: nodes = stages, edges = the actual `dependsOn` relationships, with the
  **gate marks** preserved on the correct producer→consumer edges (as today).
- **Clearly represent waves + concurrency** — that is the whole point of the prototype.
- Work for BOTH the structure-only definition view (`/workflows/[name]`) and the run-scoped view (with
  per-stage status / gate state), reusing the data the existing endpoints already provide
  (`GET /api/workflows/:name`, `GET /api/workflow-runs/:id`) — no new backend data should be needed,
  but a minimal read-only `src/api/server.ts` addition is acceptable if an option genuinely needs it.

Build the options with **real, different graphing libraries** (e.g. React Flow / `@xyflow/react`,
`dagre`/`elkjs` for layout, `cytoscape`, `vis-network`, `mermaid`) so the comparison is meaningful —
at least one should be a proper graph/layout library, not another hand-rolled SVG. Put them behind a
toggle persisted to `localStorage` (mirror the existing gate-style selector pattern in
`dashboard/app/ui.tsx`) so the owner can switch live on a workflow page. Keep every option
**mobile-responsive** and **theme-variable-driven**. **Document each option** (which library, how it
shows waves/concurrency, trade-offs) in `.harness/worklog/T220.md` so the reviewer (T206) knows what
each one is.

Scope: `dashboard/app/components/**`, `dashboard/app/workflows/[name]/page.tsx`,
`dashboard/app/workflow-runs/[id]/**`, `dashboard/app/ui.tsx`, `dashboard/app/globals.css`,
`dashboard/app/lib/api.ts`, `dashboard/package.json`, `src/api/server.ts`.
