## Do

Make a workflow's **`maxConcurrency`** user-editable from the **Workflows → [workflow]** detail page
(`/workflows/movies`), exactly like the editable **schedule** (T135) and the **enabled** toggle —
i.e. user-owned, override-flagged, and code-reconciled (a later code-sync preserves the user's value
instead of reverting to the manifest). This builds on T156, which raises the default concurrency and
has `runWorkflow` pass the effective value to `executeDag`.

Mirror the T135 schedule mechanism end-to-end:

1. **Schema/migration (T098 rule):** add `max_concurrency` (INTEGER) and `max_concurrency_overridden`
   (INTEGER default 0) columns to the `workflows` table — in `src/db/schema.sql` for fresh DBs, and
   as an additive `ALTER TABLE` migration in `src/db/index.ts` for existing DBs (no index on the new
   columns, so no bootstrap-index trap). The manifest's `maxConcurrency` seeds `max_concurrency` on
   sync.
2. **Store:** add `updateWorkflowConcurrency(name, n)` in `src/db/store.ts` that sets
   `max_concurrency = n` and flips `max_concurrency_overridden = 1`. In `upsertWorkflowStmt` (the
   sync reconcile), preserve the override the same way schedule does:
   `max_concurrency = CASE WHEN max_concurrency_overridden = 1 THEN max_concurrency ELSE
   excluded.max_concurrency END`. Validate `n` is a positive integer ≥ 1 (reject otherwise).
3. **Executor uses the EFFECTIVE value:** `runWorkflow` should pass the **effective** maxConcurrency
   to `executeDag` — the DB `max_concurrency` (which is the override when set, else the synced
   manifest value, else T156's default) — instead of reading `def.maxConcurrency` directly. So an
   edit takes effect on the **next run without a daemon restart** (runWorkflow reads it fresh each
   run, mirroring the live schedule/enabled checks).
4. **API:** `POST /api/workflows/:name/concurrency` with `{ maxConcurrency: N }` — a mutating
   endpoint behind the SAME loopback/token guard as `/toggle`, `/run`, `/schedule`, `/limits`.
   Validate server-side (positive integer ≥ 1) and reject an invalid value with **400** before it
   reaches the store. Surface the effective `max_concurrency` in the `GET /api/workflows/:name`
   payload so the UI can display it. Add the `api.updateWorkflowConcurrency` client method in
   `dashboard/app/lib/api.ts`.
5. **Dashboard:** on the workflow detail page (`dashboard/app/workflows/[name]/page.tsx`), add a
   **"Max concurrency"** row with the same inline-edit affordance as the Schedule row — show the
   current effective value, an **Edit** → number input → **Save** (and a clear error on an invalid
   value). The 3s poll refetches so it updates live. Keep it consistent with the Schedule/enable
   row styling; mobile-responsive.
6. **Docs:** update CLAUDE.md (maxConcurrency joins schedule + enabled + service limits as a
   user-editable, code-reconciled, override-flagged workflow property) and the README workflows
   section.

Tests: `updateWorkflowConcurrency` persists + flips the override; a code-sync PRESERVES an
overridden value but refreshes a non-overridden one (mirror the schedule reconcile test); the API
rejects ≤ 0 / non-integer with 400; the migrate-existing-db guard passes with the new columns.

## Done when

A workflow's max concurrency is editable from its detail page (number input + Save, like Schedule),
persisted as a user override that a code-sync preserves; `runWorkflow` uses the effective value so
an edit takes effect on the next run with no restart; the API validates it (≥ 1, else 400) behind
the mutation guard and exposes it on the workflow payload; CLAUDE.md + README document it; the new
columns migrate cleanly on existing DBs; `npx tsc --noEmit`, `npm test`, and `npm --prefix dashboard
run build` are green and `node dashboard/scripts/mobile-check.mjs` stays green.
