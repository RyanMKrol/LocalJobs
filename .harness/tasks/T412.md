## Overview

The `workouts-sync` workflow's first stage, `hevy-sync`, appends each newly-synced workout's full
data (title, exercises, sets) to the shared `data/out/workouts-history.json` file — real, substantial
work — but its success `markWorkItem` call records **no `detail` at all** (not even `{ name }`), so
the dashboard's Inputs & Outputs panel shows nothing for what the stage actually did. Fix it per the
"every stage's success `detail` must describe what it produced" convention in the root `CLAUDE.md`
("Job conventions" section) and `src/jobs/CLAUDE.md`. The second stage, `workouts-progress`, was
checked and is already compliant (see below) — no change needed there.

## Do

- **`src/jobs/workouts-sync/stages/hevy-sync.ts:159`** — inside the per-workout loop in
  `runHevySync`, the success call currently is:
  ```ts
  // old
  markWorkItem(JOB_NAME, workout.id, 'success');
  ```
  Since this stage appends to ONE shared history file (not a file per workout), `detail.path` isn't
  the right shape here — instead record a summary of the specific workout that was synced, using the
  real `HevyWorkout`/`HevyExercise` fields already defined at the top of this file (`title`,
  `start_time`, `exercises: HevyExercise[]`, each with `sets: HevySet[]`). Change to:
  ```ts
  // new
  const setCount = workout.exercises.reduce((sum, e) => sum + e.sets.length, 0);
  markWorkItem(JOB_NAME, workout.id, 'success', {
    detail: {
      name: workout.title,
      startTime: workout.start_time,
      exerciseCount: workout.exercises.length,
      setCount,
    },
  });
  ```
  Compute `setCount` once per workout right before the `markWorkItem` call (inside the existing `try`
  block, after the `newlyAppended.push(workout)` line at `hevy-sync.ts:158`) — don't add a second loop
  over `workout.exercises`.

- **`src/jobs/workouts-sync/stages/hevy-sync.ts:165`** (the `catch` block's failure call) — while the
  root convention's hard requirement is only for the SUCCESS branch, this failure call is currently
  bare (`markWorkItem(JOB_NAME, workout.id, 'failed')`) even though the catch block already has the
  workout and the error message (`msg`, computed two lines above at `hevy-sync.ts:163`) in scope. Bring
  it in line with how other stages in this codebase record failures (e.g. places/perfumes include
  `{ name, error }`):
  ```ts
  // old
  markWorkItem(JOB_NAME, workout.id, 'failed');
  // new
  markWorkItem(JOB_NAME, workout.id, 'failed', {
    detail: { name: workout.title, error: msg },
  });
  ```
  This is a small, low-risk, adjacent improvement in the same function — include it in the same
  change since it costs nothing extra and closes an obvious gap while you're already touching this
  exact call shape, but it is secondary to the success-branch fix above (which is the actual
  convention violation this task exists to fix).
