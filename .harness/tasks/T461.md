## Overview
The perfumes workflow's PerfumeRatings DynamoDB source now stores 8 more owner-authored fields per perfume than this repo reads — they're currently dropped on the floor before ever reaching PerfumeInput. This task is the read-side plumbing only: capture them onto the type, nothing downstream changes yet.

## Do
Extend `PerfumeInput` in src/workflows/perfumes/types.ts with 8 new OPTIONAL fields, deliberately renamed to avoid colliding with the build stage's existing Fragrantica/LLM-researched frontmatter keys of similar name: `description?: string` (free-text owner notes), `rating?: number` (the owner's own 0-10 score — this is a DIFFERENT field from any existing PerfumeInput key, but note it DOES map onto profile.template.md's already-existing-but-never-populated `rating: null # your own score, 1-10` frontmatter key — that mapping is Unit B's job, not this one), `dateAdded?: string` (raw `DD-MM-YYYY` string, from Dynamo's `date` — pass through verbatim, no date parsing/reformatting), `ownership?: 'Sample' | 'Travel size' | 'Full bottle'` (from Dynamo's `ownership`), `personalLongevity?: number` (0-8, from Dynamo's `longevity` — renamed with a `personal` prefix specifically so it's never confused with the template's existing community-vote-derived `longevity` enum field), `personalProjection?: number` (1-4, from Dynamo's `projection`), `personalSeasons?: string[]` (from Dynamo's `seasons` — renamed with a `personal` prefix so it's never confused with the template's existing community/LLM-researched `season` array field), `applicationSpots?: string[]` (from Dynamo's `applicationSpots`, no rename needed — no existing collision). Update `loadPerfumes()` in src/workflows/perfumes/lib.ts to destructure `description`, `rating`, `date`, `ownership`, `longevity`, `projection`, `seasons`, `applicationSpots` off each raw Dynamo item (alongside the existing `id`/`title`/`designer`/`type`/`fragranticaUrl` destructure) and map each onto its renamed PerfumeInput field, using the SAME lenient per-field pattern the existing `fragranticaUrl` handling already uses: a present-and-correctly-typed value is included (string fields via `typeof === 'string'`, number fields via `typeof === 'number' && Number.isFinite(...)`, array fields via `Array.isArray(...)` — element-level validation is NOT required, a shallow type check per field is enough), a wrong-typed or absent optional field is simply OMITTED from the resulting object (not set to `undefined` — genuinely absent, matching how `fragranticaUrl` is conditionally spread in today). Do NOT change the existing behaviour that a malformed REQUIRED field (`id`/`title`/`designer`/`type` not a string) still skips the whole item with the existing console.warn — that's unchanged; only these 8 new fields get the lenient per-field omission treatment. Update the existing `lib.test.ts`: its first test currently asserts a strict `assert.deepEqual` that the returned perfume EXCLUDES `rating`/`longevity` even though the mock item includes them — change that assertion to assert the new fields ARE captured under their renamed keys, and add: (1) a case where an optional field is present but the WRONG type (e.g. `rating: "five"`) and assert it's omitted from the result while the item itself is still returned (not dropped), and (2) a case where NONE of the 8 optional fields are present on the source item and assert none of the 8 keys exist on the resulting `PerfumeInput` (matching the existing 'omits fragranticaUrl when absent' test's style).

## Done when
`npm test` is green with lib.test.ts's `loadPerfumes` describe block covering: all 8 new fields mapped correctly by name (description/rating/dateAdded/ownership/personalLongevity/personalProjection/personalSeasons/applicationSpots) from a fixture Dynamo item that has every field; a wrong-typed optional field silently omitted without dropping the whole item; and an item with none of the 8 optional fields present producing a `PerfumeInput` where none of those 8 keys exist (checked via e.g. `assert.equal(p.rating, undefined)` per key, or an explicit `Object.keys` check, mirroring the existing fragranticaUrl-absent test). `npx tsc --noEmit` is clean — the new `PerfumeInput` fields are all optional so no existing caller of `PerfumeInput`/`loadPerfumes` (find-url.ts, build.ts, contracts.ts) needs to change or breaks.
