## Overview

Add shift-click range-select to the Backlog page's "Done" section checkboxes: click one row's
checkbox, then hold Shift and click another row's checkbox, and every checkable row between the two
(inclusive) gets selected — the standard file-manager/email/spreadsheet range-select pattern. This
makes it faster to select a batch of tasks for the existing "Mark N as reviewed" bulk action, which
today only supports one-by-one clicking.

## Do

**Where this lives:** `dashboard/app/backlog/page.tsx`. Selection state is
`const [selected, setSelected] = useState<Set<string>>(new Set())` (~line 260). The per-row checkbox
lives in the shared `CollapsibleRow` component (defined in the same file, ~line 57) — it renders an
`<input type="checkbox">` only when `selectable && !reviewed` (a row that's already reviewed has no
checkbox — reviewed rows can never be selected). `onCheck` is currently typed
`(id: string, val: boolean) => void` and is wired as `onCheck={toggleOne}` where `done.map(...)`
renders each row (~line 504-516). `toggleOne` (~line 356) does a plain add/delete on the `Set`. There
is also an existing "select all unreviewed" checkbox (~line 481-489) and the bulk action button
`{selected.size > 0 && <button>Mark N as reviewed</button>}` (~line 490-499) — both already work
end-to-end; this task only changes how a MULTI-row selection gets built, not what happens once it's
selected.

**Two judgment calls already made for you (well-established UX conventions — do not re-litigate):**
1. **Shift-click ADDS the range to the existing selection** (never replaces it) — the universal
   Gmail/Finder/Sheets convention. A prior, disjoint selection outside the range survives a
   shift-click elsewhere.
2. **The range is computed over the CURRENTLY RENDERED, CHECKABLE rows, in their rendered order** —
   i.e. index into the same ordered list of ids that `unreviewedVisible` already computes (~line
   345: `done.filter((t) => t.reviewed !== true)`), NOT the full unfiltered task list. If the "Show"
   review-status filter (All / Reviewed / Not reviewed) is active, the range only spans what's
   actually checkable on screen — a hidden row is simply not part of the range (this falls out
   naturally from indexing into the visible list rather than the full dataset, so no special-case
   logic is needed for the filter).

**Implementation approach — anchor tracking + shiftKey plumbed through the checkbox's `onClick`:**

1. Add `const [lastClickedId, setLastClickedId] = useState<string | null>(null)` in `Backlog()`
   alongside `selected` — this is the shift-click anchor (the row a *plain* click most recently
   landed on).
2. **Capturing the Shift modifier reliably:** a checkbox's React `onChange` event is NOT a
   `MouseEvent` and does not carry `shiftKey`, but the native `click` event that precedes it does. In
   `CollapsibleRow`, add a `useRef<boolean>(false)` (e.g. `shiftKeyRef`) and set it from the
   checkbox's existing `onClick` handler (`shiftKeyRef.current = e.shiftKey`, alongside the existing
   `e.stopPropagation()`), then read `shiftKeyRef.current` inside the existing `onChange` handler when
   calling `onCheck`. Change the `onCheck` prop's type to
   `(id: string, val: boolean, shiftKey?: boolean) => void` and call
   `onCheck?.(t.id, e.target.checked, shiftKeyRef.current)`.
3. Rewrite `toggleOne` in `page.tsx` to accept the third `shiftKey` argument:
   - If `shiftKey` is true AND `lastClickedId` is set AND `lastClickedId` is still present in the
     current `unreviewedVisible` id list: compute both ids' indices in
     `unreviewedVisible.map((t) => t.id)`, take the inclusive range between the smaller and larger
     index, and `add` every id in that range to the `selected` Set (a union — never removes anything,
     per judgment call 1). Do NOT toggle based on the clicked checkbox's own new `checked` value in
     this branch — a shift-click range always SELECTS the whole span.
   - Otherwise (a plain click, or shift-click with no usable anchor — e.g. the anchor row scrolled out
     of the current filter): fall back to the existing add/delete-by-`val` behavior.
   - In both branches, end by setting `lastClickedId` to the just-clicked row's id (the new anchor for
     the next shift-click, matching Gmail/Finder — every plain click and every shift-click updates the
     anchor to the row you just clicked).
4. **Reset the anchor wherever the existing code already resets `selected`** so it can never point at
   a row that's no longer meaningful: the review-status filter buttons already do
   `setSelected(new Set())` (~line 472) — add `setLastClickedId(null)` there too. `markSelectedReviewed`
   already does `setSelected(new Set())` on success (~line 371) — add `setLastClickedId(null)` there
   too (once a batch is marked reviewed, those rows lose their checkboxes entirely, so the anchor is
   stale).
5. No changes are needed to `toggleSelectAll`, the bulk "Mark N as reviewed" button, or any API call —
   this task is purely about how the `selected` Set gets populated.
