## Overview

Add a new `stocks-resolve-names` stage to `stocks-sync` that looks up each fetched position's
company name from Trading212's own metadata endpoint, and wire it into the DAG right after
`stocks-fetch`. This is step 1 of 2 in replacing `stocks-snapshot`'s ISIN/OpenFIGI ticker
resolution with a simpler name lookup — this task only ADDS the new stage; it does not yet remove
anything from `stocks-snapshot` (that's a separate follow-up task, T414). Nothing in `stocks-sync`
actually uses the resolved ticker computationally today (confirmed by inspection — it's purely a
display column), so a company **name** — which Trading212's own metadata endpoint already
returns — serves the same "make the report readable" purpose without needing OpenFIGI at all.

## Context (read before making any change)

Today `stocks-sync` is a 4-job DAG: `stocks-fetch` → `stocks-snapshot` → `stocks-watch` →
`stocks-notify`. `stocks-fetch` fetches raw positions and writes `data/out/raw-positions.json`.
`stocks-snapshot` currently reads that file, resolves each position's ISIN + real-world ticker via
`src/services/trading212.service.ts`'s `resolveTickers`/`resolveOpenFigiTickersBatched` (backed by
the `openfigi` service), and writes the final `data/out/portfolio.json`/`portfolio.md` report.

`src/services/trading212.service.ts` is a **shared, top-level service file** — it is ALSO used
independently by the completely separate `stock-digest` workflow (`src/jobs/stock-digest/stages/
stock-portfolio-snapshot.ts` and `stock-sector-lookup.ts`), which uses the SAME
`resolveTickers`/`resolveOpenFigiTickersBatched`/`fetchInstrumentsMetadata`/`NormalizedPosition`
exports for its OWN independent purpose (Finnhub industry classification). **This task must not
touch or break `stock-digest`'s use of ISIN/OpenFIGI resolution in any way** — do not remove,
rename, or otherwise change `resolveTickers`, `resolveOpenFigiTickersBatched`,
`fetchInstrumentsMetadata`, `Trading212Instrument`, or `NormalizedPosition`'s existing `isin?`/
`resolvedTicker?` fields. This task is purely additive to the shared service file.

### 4. New file: `src/jobs/stocks-sync/stages/stocks-resolve-names.ts`

`runStocksResolveNames(ctx, opts)`:
- Read raw positions from `stocksSyncConfig.rawPositionsJsonPath` (write a small local
  `readRawPositions`-equivalent reader in this file — do not remove or touch `stocks-snapshot.ts`'s
  existing `readRawPositions`/`RawPositionsReader`, which stays exactly as-is for now; a
  short-lived duplication across the two files is fine and expected here, T414 will clean it up).
- If there are zero raw positions: skip the ledger row entirely (nothing to resolve).
- Otherwise: read+validate `TRADING212_API_KEY_ID`/`TRADING212_API_SECRET_KEY` (throw if missing),
  call `fetchInstrumentsMetadata` via `callService('trading212', ...)` **at most once per run**
  (never once per position), build a `nameByTicker` map from the response's `name` field, and
  attach `name` to each position (a ticker not found in the metadata response gets `name:
  undefined` — soft miss, `ctx.log('warn: ...')` naming the unresolved ticker, never throw). **Do
  NOT call `resolveTickers` or `resolveOpenFigiTickersBatched` or read/write `isin` anywhere in
  this file** — this stage is Trading212-metadata-only, no OpenFIGI dependency at all.
- Write the named positions to `stocksSyncConfig.namedPositionsJsonPath`.
- Record ONE ledger row keyed by `dayKey(now)` (import `dayKey` from `./stocks-snapshot.js` — same
  same-key-stage convention as the rest of the pipeline, no `rootKey`/`parentKey` needed), with
  `detail: { name: \`Names resolved — ${key}\`, resolvedCount, totalPositions: positions.length,
  path: stocksSyncConfig.namedPositionsJsonPath, format: 'json' }` where `resolvedCount` is how
  many positions got a non-undefined `name`.
