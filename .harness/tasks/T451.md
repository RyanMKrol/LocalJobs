## Overview

Add an opt-in, short-TTL (5 minute) response cache to the shared `callService` helper, so a service
consumer can dedupe near-simultaneous identical calls to an `'api'`-category service without a fresh
network call each time.

## Do

In `src/core/services.ts`, add an optional `cacheKey?: string` field to the `CallServiceOpts`
interface (currently `{ maxWaitMs?, onThrottle? }`). Inside `callService(name, fn, opts)`, right after
resolving `const def = getServiceDef(name)` (around line 92) and before the existing
consumer-recording/quota/rate logic, add: if `opts.cacheKey` is set AND `def.category === 'api'`, call
a new `getCachedServiceResponse<T>(name, opts.cacheKey, SERVICE_CACHE_TTL_MS)` from `src/db/store.ts`
— if it returns a cached value (not `undefined`), return it immediately, skipping the
consumer-recording, quota checks, rate-limit throttling, and `fn()` entirely (a full short-circuit,
not just a metering skip). If there's no cache hit (or `cacheKey` is absent, or the service's category
isn't `'api'`), proceed exactly as today through the existing quota/min-interval/rate-limit/no-limit
branches — but wrap the ONE `fn()` invocation that finally succeeds in each of those branches so that,
when `opts.cacheKey` is set and `def.category === 'api'`, the result is stored via a new
`setCachedServiceResponse(name, opts.cacheKey, result)` from `src/db/store.ts` before returning it.
Define `const SERVICE_CACHE_TTL_MS = 5 * 60_000;` as a local constant in `src/core/services.ts` — a
flat, hardcoded value, NOT read from the `ServiceDefinition` and NOT configurable via the dashboard (no
new `services` table column, no new API endpoint).

In `src/db/schema.sql`, add a new table:
```sql
CREATE TABLE IF NOT EXISTS service_cache (
  service_name TEXT NOT NULL,
  cache_key TEXT NOT NULL,
  response_json TEXT NOT NULL,
  cached_at TEXT NOT NULL,
  PRIMARY KEY (service_name, cache_key)
);
```
This is a brand-new table (not a new column on an existing table), so it can be added directly to
`schema.sql` with no additive `ALTER TABLE` migration needed in `src/db/index.ts` (it does not hit the
T098 "migration-added column referenced at bootstrap" trap described in the root `CLAUDE.md`, since
there's no existing-DB backfill concern for a wholly new table).

In `src/db/store.ts`, add two new exported functions:
- `getCachedServiceResponse<T>(serviceName: string, cacheKey: string, ttlMs: number): T | undefined` —
  reads the row for `(service_name, cache_key)`; returns `undefined` if no row exists, or if
  `Date.now() - Date.parse(cached_at) > ttlMs`; otherwise `JSON.parse(response_json)` as `T`.
- `setCachedServiceResponse(serviceName: string, cacheKey: string, value: unknown): void` — upserts
  `(service_name, cache_key, JSON.stringify(value), new Date().toISOString())` into `service_cache`
  (use `INSERT ... ON CONFLICT(service_name, cache_key) DO UPDATE SET response_json = excluded.response_json,
  cached_at = excluded.cached_at`, matching this file's existing upsert style — see `upsertWorkflowStmt`
  for the pattern already used in this file).

## Done when

- `CallServiceOpts` has a `cacheKey?: string` field; `callService`'s behavior is UNCHANGED for every
  existing call site that doesn't pass `cacheKey` (byte-identical to today — no existing call site in
  the repo needs to change).
- Passing `cacheKey` to a service whose `ServiceDefinition.category !== 'api'` (e.g. `claude-cli`,
  which is `'cli-tool'`, or `fragrantica`, which is `'website-scrape'`) is a no-op: the real call
  always happens, nothing is cached, no `service_cache` row is written or read for that call.
- Passing `cacheKey` to an `'api'`-category service (e.g. `tmdb`, `gemini`, `google-places`): the FIRST
  call with a given `(service, cacheKey)` pair makes the real call and records the row; a SECOND call
  with the SAME `(service, cacheKey)` pair within 5 minutes returns the cached value WITHOUT invoking
  `fn` again (assert this with a test double `fn` that increments a call counter — the counter must
  stay at 1 after the second call) and without incrementing that service's `service_usage` call count
  for the second call.
- A third call after the TTL has elapsed (simulate by inserting a `service_cache` row with a
  `cached_at` older than 5 minutes, or by injecting a fake clock) re-invokes `fn` and refreshes the
  cached row.
- New unit tests in `src/core/services.test.ts` (this file already exists) cover: cache hit skips the
  real call, cache miss populates it, TTL expiry re-fetches, and a `cli-tool`/`website-scrape` service
  ignores a passed `cacheKey`. Tests must run against the scratch test DB (`LOCALJOBS_DB`/`isTestEnv()`
  guard already in `src/config.ts` — never the production `data/jobs.db`).
- `npx tsc --noEmit` and `npm test` both pass.
