## Overview
Every one of the 16 recommender-branch DAG jobs (8 in movies, 8 in tv-recs) reports status='success' on a real run, yet zero work_items rows exist for any of them — confirmed empirically. The domain wrapper's own ledger-writing code (recordBranchLedgerRow) is never invoked because of a same-name-different-function shadowing bug.

## Do
Confirmed call chain: each branch's `*.job.ts` (e.g. src/workflows/movies/stages/rec-random-1.job.ts) calls `makeBranchJob('rec-random-1')` imported from its own domain's `recommend.ts` (movies: line 112, tv-recs: line 111) — NOT the core one directly. That domain `makeBranchJob` calls `coreMakeBranchJob(domain, id, {...})` (src/core/recommender/branch.ts:213-230), whose returned JobDefinition's `run(ctx)` (branch.ts:227-229) calls `runBranch(ctx, domain, spec)` — which resolves, by ordinary JS scoping, to branch.ts's OWN LOCAL `export async function runBranch<M,P>(...)` at branch.ts:110, NOT the domain wrapper's `runBranch` in recommend.ts:59-63 (movies) / recommend.ts:61-65 (tv-recs), which is the ONLY place that calls `recordBranchLedgerRow`. So `recordBranchLedgerRow` (recommend.ts:73-89 movies, :75-91 tv-recs) is dead code for every real job execution — it only runs when something calls the domain wrapper's exported `runBranch` directly (e.g. a test). Fix this at the shared source, src/core/recommender/branch.ts, so both domains are fixed identically instead of patching each domain's dead wrapper: give `coreMakeBranchJob`'s options an optional `onSuggestionsWritten` (or similarly named) hook, or otherwise have `branch.ts`'s own `runBranch`/`makeBranchJob` invoke a domain-supplied post-write callback after it writes the branch's suggestions file (after the `writeBranchFile` calls inside branch.ts's `runBranch`, i.e. after each of the three early-return writes at lines ~150, ~161, ~172, and the final success write at line 177). Then wire movies/stages/recommend.ts and tv-recs/stages/recommend.ts's `makeBranchJob` to pass their existing `recordBranchLedgerRow` as that hook, and delete/simplify their now-redundant local `runBranch` wrapper (which currently duplicates coreRunBranch + calls the dead-code path) so there is exactly ONE runBranch implementation actually executing per branch job. Keep the exported `runBranch`/`makeBranchJob`/`parseSuggestions`/etc. names each domain file re-exports (per their own file-header comments, other call sites — recommend.test.ts, merge.ts's top-up loop via collectBranchSuggestions — depend on the stable API surface), but their behavior must now genuinely route through the fixed path. This unit is ONLY the wiring fix (making recordBranchLedgerRow actually execute with its EXISTING count-only detail shape) — extending it to one-row-per-suggested-title is the next unit (recs-ledger-rebuild-b), which depends on this one.
