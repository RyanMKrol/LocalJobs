## Do

Make the Plex client resilient to a changing Plex server IP. Today `src/jobs/plex/client.ts`
relies solely on `PLEX_HOST` and throws `Plex unreachable — set PLEX_HOST in .env` when it's
unset or unreachable. The owner's Plex server gets its IP via **DHCP**, so `PLEX_HOST` goes stale
whenever the lease changes, breaking the whole workflow at the snapshot stage (this exact failure
already happened). Add a **LAN-scan fallback** — the owner's public PlexUtils repo already has the
proven version (`src/remote/index.js` `resolvePlexHost` + `src/utils/remote.js` `scanForPlexHost`
/ `probePlexIdentity`) — use it as a reference, don't copy wholesale.

Implement a `resolvePlexHost()` in the plex client that:
1. **Tries the configured `PLEX_HOST` first** — confirm it's actually a live Plex by GETting
   `<host>/identity` (with the `X-Plex-Token`) and, if `PLEX_MACHINE_ID` is set, checking the
   returned `machineIdentifier` matches (so we never latch onto a different Plex). If it answers,
   use it.
2. **Falls back to scanning the local subnet** when the configured host is missing/unreachable:
   derive the machine's local IPv4 `/24`(s) from `os.networkInterfaces()`, probe each host on
   `:32400` at `/identity` (short per-probe timeout, bounded concurrency), and accept the first
   that answers as a Plex server — **gated on `PLEX_MACHINE_ID` when set** (only a server with that
   identifier is accepted). Both http and https (self-signed → the existing scoped
   `rejectUnauthorized:false` agent; never global TLS-disable).
3. **Caches the resolved host for the process** (resolve at most once per daemon run), and
   `plexGet` uses the resolved host instead of `plexConfig.host` directly.
4. On a successful scan, **log a clear heads-up** (e.g. "found Plex at <host> — set PLEX_HOST in
   .env to skip the scan next time"). If neither the configured host nor the scan finds a Plex,
   throw the existing clear, actionable error.

Notes:
- Keep `PLEX_MACHINE_ID` optional but **document it in `.env.example`** (names/placeholders only)
  and note it makes the scan safe (won't pick the wrong Plex). Add a short README/CLAUDE note that
  the Plex client self-heals a changed DHCP IP via a LAN scan.
- The scan can be slow on a big subnet — bound it (concurrency + per-probe timeout + an overall
  cap) so a no-Plex environment fails fast with the clear error rather than hanging the stage.
- **Testability:** factor the resolution so the network probe is injectable — a hermetic unit test
  drives `resolvePlexHost` with a fake probe to assert: configured-host-wins when reachable;
  falls back to scan when not; respects `PLEX_MACHINE_ID` (rejects a non-matching server);
  caches (probes once); and throws the clear error when nothing is found. The REAL subnet scan is
  the owner's live verification, not a unit test.
