## Overview
The gate-padlock icon on a workflow-run DAG diagram can render on top of / behind an unrelated intermediate stage's box instead of cleanly on the connecting line, whenever a stage depends on both its immediate predecessor AND an earlier ancestor in the same linear row (a 'skip-column' edge). Fix DagFlow.tsx's edge-label positioning so the padlock never lands inside another node's footprint, confirmed via an actual screenshot of the plex-language-fix workflow-run page (the real-world case that surfaced this).

## Do
In `dashboard/app/components/DagFlow.tsx`, the `GateEdge` component (~lines 81-169) positions its padlock label at the raw bezier-curve midpoint returned by `getBezierPath()` (`labelX`/`labelY`, used at line 136's `translate(${labelX}px,${labelY}px)`). For a same-row 'skip-column' edge — a source/target pair whose Y is equal but whose wave-column gap is 2+ (e.g. `plex-language-discover -> plex-language-evaluate`, which coexists with the `discover -> resolve -> evaluate` chain since `plex-language-evaluate`'s manifest declares `dependsOn: ['plex-language-discover', 'plex-language-resolve']` in `src/workflows/plex-language-fix/plex-language-fix.workflow.ts`), the geometric midpoint X falls exactly on an intervening node's own center X (worked example: discover right-edge=196, evaluate left-edge=596, midpoint=396 == resolve's center X exactly, per `buildLayout`'s `colX = 16 + wave*(NODE_W=180 + RANK_SEP=110)`). Because React Flow paints the nodes layer above the edges/EdgeLabelRenderer layer, the padlock (whose `lockBg` background is semi-transparent) ends up rendered underneath the opaque intermediate node box. Fix this in `GateEdge` (or a small helper it calls) by detecting when the label's default midpoint (at t=0.5 along the path) falls within the X-span (and Y-span, since same-row) of any OTHER node's rendered footprint, and in that case moving the label off that midpoint — the two acceptable approaches, pick whichever is cleaner to implement correctly: (1) recompute the label position at a different point along the same bezier (e.g. t=0.25 or t=0.75, whichever is farther from the nearest node's center) using `getBezierPath`'s curve math or a manual bezier-point interpolation, so the padlock slides along the existing line to a non-colliding point still visibly ON the connecting curve; or (2) if a clean non-colliding point along the existing nearly-straight bezier isn't reliably obtainable, bow the curve's midpoint vertically (offset `labelY` up/down by roughly half a node-height plus a small margin) so it clears the row entirely, either by passing custom control points into a manual quadratic/cubic bezier instead of `getBezierPath`'s default, or by post-hoc offsetting just the computed `labelY` for this one case (simpler, and the edge path itself doesn't need to visually bow to still be correct — only the label must not sit on the collision point). Do NOT change layout/positioning for edges that don't collide (the common producer->immediate-consumer case must render exactly as it does today — verify this specifically, since it's the vast majority of gates in the app and a regression there would be worse than the bug being fixed). The collision check should be general (based on actual node X/Y footprints already computed in `buildLayout`, not hardcoded to plex-language-fix's specific stage names), since root CLAUDE.md's own incident write-up notes this reproduces for ANY workflow with a same-row multi-hop dependency, not just plex-language-fix.
