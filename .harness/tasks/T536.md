## Overview
src/services/trading212.service.ts throws `Trading212 API error ${res.status}: ${await res.text()}` in fetchPortfolio (~line 123) and fetchInstrumentsMetadata (~line 149), embedding the FULL upstream error body verbatim in the thrown Error — which lands in run logs. Safe with Trading212 today, but if any upstream ever echoes auth material into an error body it would be logged in full. claude.ts already slices error text to ~300-400 chars.

## Do
In src/services/trading212.service.ts, slice the awaited error body to ~300 chars before interpolation in both throws (~lines 123 and 149): `const body = (await res.text()).slice(0, 300); throw new Error(\`Trading212 API error ${res.status}: ${body}\`);` — matching the `.slice(0, 300)` pattern in src/services/claude.ts (~line 91). Then sweep the other `src/services/*.service.ts` files for the same unsliced `${await res.text()}`-in-thrown-error pattern (the initial grep found only trading212, but confirm across the folder) and apply the same slice where found. Purely defensive; no behavior change for well-behaved upstreams.

## Done when
