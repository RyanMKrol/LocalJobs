## Overview

Two per-item catch blocks log their failure message with the text `"error: ..."` but forget to pass
`'error'` as `ctx.log`'s second argument, so the line is recorded (and coloured on the dashboard) at
`'info'` level instead of `'error'` — a real error is silently invisible in the run's log-level
filtering/colouring. Fix both call sites; a repo-wide grep found exactly these two.

## Do

`ctx.log`'s signature is `log(message: string, level?: LogLevel): void` (`src/core/types.ts`), and the
real child-process implementation defaults the level to `'info'` when the second argument is omitted
(`src/runJob.ts`, the `ctx.log: (message, level = 'info') => …` line). Two `catch` blocks build a
message whose TEXT starts with `error:` but never pass the actual level argument, so it silently
defaults to `'info'`:

1. `src/jobs/projects-sync/stages/project-summarize.ts` (around line 247), inside the per-repo
   `catch (e)` block:
   ```ts
   ctx.log(`error: failed to summarize ${entry.fullName}: ${String(e)}`);
   ```
   Change to:
   ```ts
   ctx.log(`error: failed to summarize ${entry.fullName}: ${String(e)}`, 'error');
   ```

2. `src/jobs/workouts-sync/stages/hevy-sync.ts` (around line 164), inside the per-workout
   `catch (err)` block:
   ```ts
   ctx.log(`error: failed to record workout ${workout.id}: ${msg}`);
   ```
   Change to:
   ```ts
   ctx.log(`error: failed to record workout ${workout.id}: ${msg}`, 'error');
   ```

**A repo-wide grep was already run during conversion of this idea** (the owner explicitly asked for
this to be checked) — `grep -rn "log(\`error\|log('error\|log(\"error" src --include="*.ts"` (excluding
the `src/jobs/projects-sync/data/` self-clone artifact and `*.test.ts` files) turned up **only these
two** call sites with the forgotten second argument. Every other `ctx.log(...)`/`addLog(...)`/
`log(...)` call whose message starts with an error-shaped string already passes `'error'` explicitly
(e.g. `src/core/executor.ts`, `src/core/workflow-executor.ts`, `src/jobs/places/stages/*.ts`). Do a
final confirming grep as part of this task (the codebase may have grown a new instance since this spec
was written) before considering it done, but do not go hunting for a broader class of logging bugs —
this task is scoped to the forgotten-second-argument pattern specifically, on `error:`-prefixed
messages.

Add a small regression-guarding unit test to each stage's existing test file (`project-summarize.test.ts`,
`hevy-sync.test.ts`) exercising the existing failure-path test case (both files already have a test
that forces the catch block via a rejecting/throwing stub) — extend the `fakeCtx()` `log()` stub in that
specific test to record `(message, level)` pairs and assert the resulting error log line's level is
`'error'`. Keep the change minimal — don't rewrite `fakeCtx` globally if a single test-local override is
cleaner.
