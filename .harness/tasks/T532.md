## Overview
authoriseMutation() in src/api/server.ts (line ~312) compares the presented token with `supplied === args.token || bearer === args.token` — a non-constant-time string compare, in principle timing-side-channel-able. Low practical risk on loopback but a trivial hardening.

## Do
In src/api/server.ts's `authoriseMutation` (~lines 300-313), replace the `===` token comparisons with a length-guarded `crypto.timingSafeEqual` helper: return false early if the presented value and `args.token` differ in byte length (timingSafeEqual throws on unequal-length buffers), else compare Buffers. Keep the exact same acceptance semantics (loopback bypass unchanged, empty configured token still rejects remote, X-LocalJobs-Token OR Authorization: Bearer accepted). Import `timingSafeEqual` from `node:crypto` (randomUUID is already imported elsewhere). This is a one-function change; the existing authoriseMutation tests in server.test.ts must still pass.

## Done when
authoriseMutation uses a length-guarded crypto.timingSafeEqual for both the X-LocalJobs-Token and Bearer comparisons; loopback + valid-token + wrong-token + empty-configured-token cases behave exactly as before (existing server.test.ts auth cases green); `npx tsc --noEmit` + `npm test` green.
