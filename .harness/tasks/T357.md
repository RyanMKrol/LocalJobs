## Overview

The `.movie-gaps-scroll` CSS class (used by the movie recs, TV recs, franchise-gaps, and generic
Output tables) caps its content at `max-height: 520px` with internal `overflow-y: auto` scrolling.
The owner wants these tables to display in full on the page instead — no internal scroll box.

## Do

1. In `dashboard/app/globals.css`, remove the scroll cap from `.movie-gaps-scroll`
   (currently around line 182):
   ```css
   .movie-gaps-scroll { max-height: 520px; overflow-y: auto; display: flex; flex-direction: column; gap: 12px; }
   ```
   becomes:
   ```css
   .movie-gaps-scroll { display: flex; flex-direction: column; gap: 12px; }
   ```
   (`display: flex; flex-direction: column; gap: 12px` is still needed — it's what stacks the
   per-collection/per-show `.panel` cards with spacing between them; only the height cap +
   scroll behavior are being removed.)
2. Reconsider the adjacent shrink-behavior rule + its comment (lines ~184-188):
   ```css
   /* Per-collection/per-show cards inside the scroll area: padding so content doesn't hug the border-radius. */
   /* flex-shrink: 0 is load-bearing (T333): .panel has overflow:hidden (for border-radius clipping),
      which per the flexbox spec collapses its automatic minimum size to 0 — without this the flex
      layout SHRINKS the panel to fit inside .movie-gaps-scroll's max-height instead of letting it
      grow past it, so the outer overflow-y:auto never has anything to scroll. */
   .movie-gaps-scroll .panel { padding: 12px 16px; flex-shrink: 0; }
   ```
   The comment's specific reasoning (preventing the panel shrinking to fit inside a max-height) no
   longer applies once the max-height is gone — but `flex-shrink: 0` is still a reasonable, harmless
   default for a `flex-direction: column` stack (it stops any per-card panel from being compressed
   below its content size in edge cases, e.g. a very long list rendered inside a constrained parent
   elsewhere). Keep `flex-shrink: 0` and the padding rule, but REWRITE the comment so it no longer
   references the now-removed max-height/scroll behavior — state plainly that it's a stack of
   full-height cards with no internal scrolling, and `flex-shrink: 0` just guards against unwanted
   compression in a flex column. Do not leave a stale comment describing behavior that no longer
   exists.
3. `dashboard/scripts/_dashboard-harness.mjs` has TWO `FLOWS` entries (a prior T333 task) that
   specifically exercise the removed scroll behavior via mouse-wheel + `scrollTop` assertions — these
   are now testing behavior that no longer exists and MUST be removed (not left to silently no-op or
   fail):
   - `movie-recs-scrolled` (currently ~line 664-685): drives a wheel scroll over `.movie-gaps-scroll`
     on `/workflows/movie-recommendations` and asserts `scrollTop > 0`.
   - `tv-recs-scrolled` (currently ~line 686-703): same pattern on `/workflows/tv-recommendations`.
   Delete both entries entirely from the `FLOWS` array (search for `T333` comments referencing
   `.movie-gaps-scroll` mouse-wheel scrolling to find them precisely — they may have shifted line
   numbers since this spec was written). Do NOT delete the underlying `movieRecs`/`tvRecs` fixture data
   that was bumped to ~15 rows for T333 (`_dashboard-harness.mjs`'s `movieRecs`/`tvRecs` constants) —
   that larger row count is still useful/realistic fixture data and is exactly what will now let a
   reviewer SEE the full-height, no-scroll table in the `workflow-movie-recs`/`workflow-tv-recs`
   baseline `PAGES` screenshots (those `PAGES` entries already exist and don't need new entries — the
   removed cap changes what they render, not whether they exist).
4. Grep the whole dashboard for any other reference to `.movie-gaps-scroll`'s scroll behavior (e.g. a
   comment elsewhere describing it as scrollable) and update/remove anything now stale. The class NAME
   itself (`movie-gaps-scroll`) is being left as-is even though it's no longer a "scroll" container —
   renaming it is out of scope for this task (would touch `page.tsx` in 2 places +
   `WorkflowOutputSection.tsx` for a cosmetic rename with no behavior change); leave a short comment
   above the CSS rule noting the name is now a historical misnomer if you think it aids future
   maintainers, but don't feel obligated to.
