## Overview
Adds a new `inputKeysService` field to `JobDefinition` so that any job declaring `inputKeys()` (making it a workflow's limitable root stage) must ALSO name the real, registered `ServiceDefinition` its candidate-enumeration call is routed through — formalizing "a root's `inputKeys()` must query the live external source, never this workflow's own prior output" as a structural, type-level contract instead of just a convention. Depends on T483 (fs service), T484 (places fix), T485 (plex-language-fix fix), T486 (projects-sync fix), and T474 (Plex callService migration, needed for `plex-profiles`'s retrofit) — all 5 jobs that will get the field must already have a live-service-backed `inputKeys()` implementation before this task retrofits the field onto them.

## Do
Add to `src/core/types.ts`, on `JobDefinition`, alongside the existing `inputKeys?(): string[] | Promise<string[]>`:
```ts
/** Required alongside inputKeys(): names the ServiceDefinition this root's
 *  candidate-enumeration call is routed through — proves it's a LIVE query
 *  against the true external source, never this workflow's own prior output.
 *  Enforced by src/core/root-inputkeys-service.test.ts (T488). */
inputKeysService?: string;
```
Retrofit the field onto every job in the registry that declares `inputKeys()` (5 total, confirmed this session — only 5 of 15 workflows declare `inputKeys()` at all):
- `perfumes`'s `find-url.job.ts` → `inputKeysService: 'dynamodb'` (already live via `callService('dynamodb', ...)` in `loadPerfumes()` — just add the field, no behavior change).
- `places`'s `resolve.job.ts` (the job wrapping `resolveInputKeys`) → `inputKeysService: 'fs'` (now live after T484).
- `plex-language-fix`'s `discover.job.ts` → `inputKeysService: 'plex'` (now live after T485 + T474).
- `projects-sync`'s `github-sync.job.ts` → `inputKeysService: 'github'` (now live after T486).
- `plex-profiles`'s `build.job.ts` → `inputKeysService: 'plex'` (its `plexGet` calls are wrapped in `callService('plex', ...)` by T474, which is already in this task's `dependsOn`).
Update each of the 5 touched workflows' own `CLAUDE.md` to note the `inputKeysService` value for its root stage, and update the root `src/workflows/CLAUDE.md` (or wherever the "root stage" convention is documented in root CLAUDE.md) to describe the new field and its enforcement.
