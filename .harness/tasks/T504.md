## Overview
Cache `google-places` Place Details reads (a paid API; a place's details are stable) for 22 hours so
re-running enrichment during development doesn't re-pay for the same place. Outcome of the T479
caching-adoption decision; the caching mechanism already exists (T451/T476).

## Do
1. Set `cacheTtlMs: 79_200_000` (22 hours) on the `google-places` `ServiceDefinition` in
   `src/services/google-places.service.ts`.
2. Add a stable `cacheKey` to the `callService('google-places', ...)` read site in
   `src/workflows/places/stages/enrich.ts`, keyed by the `place_id` being fetched:
   `` `google-places:details:${placeId}` ``.
3. Cache the read (Place Details GET) only.

## Done when
- `cacheTtlMs: 79_200_000` is set on the `google-places` service def.
- The call site passes `google-places:details:<place_id>` as its cacheKey.
- `npx tsc --noEmit` and `npm test` are green.
- Verified a repeated Place Details fetch for the same place_id within 22h is served from
  `service_cache` (hermetic scratch-DB check) rather than a second live paid call.
