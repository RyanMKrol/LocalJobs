## Overview
Clicking Run on the Workflows list page or the Overview page briefly shows 'Started…' then flashes back to plain 'Run' even though a real run just started, because the local busy flag clears the instant the fire-and-forget POST resolves — before the next 3s poll can pick up the new 'running' status. The workflow DETAIL page already fixed this exact race with a timed buffer; apply the same fix to the other two pages.

## Do
dashboard/app/workflows/page.tsx's `run()` (currently lines 30-43) and dashboard/app/page.tsx's `runWorkflow()` (currently lines 38-51) both follow the same pattern: `setBusyWorkflows(add name)`, `try { await api.runWorkflow(name) } catch {...} finally { setBusyWorkflows(delete name) }` — clearing busy immediately in the `finally`, which races the 3s `usePoll` refresh (`POST /api/workflows/:name/run` in src/api/server.ts is fire-and-forget: it kicks off `runWorkflow(...).catch(...)` and returns 202 almost instantly, well before the workflow_runs DB row's status is guaranteed to show up in a subsequent GET /api/workflows poll). dashboard/app/workflows/[name]/page.tsx's own `run()` (lines 306-309) already solved this: it holds `busy` for a fixed buffer after the POST resolves instead of clearing it immediately — `setBusy(true); try { await api.runWorkflow(...) } finally { busyTimerRef.current = setTimeout(() => setBusy(false), 1200); }` — using a `busyTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)` plus a mount-cleanup `useEffect(() => () => { if (busyTimerRef.current) clearTimeout(busyTimerRef.current); }, [])` (lines 203-206 in that file) to avoid leaking the timer across unmounts. Apply the SAME pattern to both `dashboard/app/workflows/page.tsx` and `dashboard/app/page.tsx`, adapted for their per-workflow `Set<string>` busy state (each page tracks busy-ness for potentially multiple workflow rows/cards at once, not a single `busy` boolean like the detail page) — i.e. keep a `Map<string, ReturnType<typeof setTimeout>>` (or equivalent per-name timer ref) keyed by workflow name, and in each page's run handler's `finally`, instead of immediately deleting the name from `busyWorkflows`, schedule a `setTimeout(() => { delete the name from busyWorkflows }, 1200)` (clearing/replacing any existing timer for that same name first, and clearing all pending timers in a page-level unmount `useEffect` cleanup — mirroring the detail page's single-ref cleanup pattern but for a keyed collection). Keep the 1200ms duration identical to the detail page's proven value for consistency across all three surfaces. Do not change `RunButton.tsx` itself, the `isRunning`/`busy` prop contract, or any other button/state logic on either page.
