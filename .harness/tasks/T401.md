## Overview

The `perfumes` workflow currently reads its list of perfumes-to-profile from a manually-maintained
local file, `src/jobs/perfumes/data/raw/perfumes.json` (42 hand-ported entries). The owner now
maintains perfume reviews directly in a live DynamoDB table (`PerfumeRatings`, populated by rating
perfumes on their website) and wants that table to become the workflow's actual source of truth,
with the local file retired. This task wires `perfumes-find-url` (the workflow's root stage) and
`loadPerfumes()` to read live from `PerfumeRatings` instead, seeds the URL-discovery stage from the
table's own `fragranticaUrl` field when present, and removes the local-file code path.

   (`src/services/dynamodb.service.ts`). The service currently exports `dynamoGet` / `dynamoPut` /
   `dynamoQuery` / `dynamoDelete` / `dynamoBatchWrite` but nothing that reads an entire table (a
   `Query` requires a known partition-key condition; we need every item). Add an exported
   `dynamoScan(table: string, params?: { filterExpression?: string; expressionAttributeNames?:

   Promise<Record<string, unknown>[]>` using `ScanCommand` from `@aws-sdk/lib-dynamodb` (already a

   `LastEvaluatedKey` is present, passing it as `ExclusiveStartKey`) so it returns the full table
   regardless of Scan's ~1MB-per-call limit — the table has 1 item today but will grow. Mirror the
   existing helpers' style (thin wrapper, `getClient().send(...)`, no service-name/cost logic here —
   that's owned by the caller via `callService`).
