## Do

Prevent the class of bug that just crash-looped the daemon (the T094 regression, fixed in commit 2748c58): a schema-bootstrap statement in schema.sql referencing a column that only a LATER additive migration adds. It was invisible to CI because the unit suite always starts from a FRESH/empty scratch DB — so schema.sql's CREATE TABLE already contained the new column — while the live/existing DB crashed in openDb(). Add a regression test (src/db/migrate-existing-db.test.ts) that exercises the REAL openDb() bootstrap+migration path against a PRE-POPULATED database seeded to an OLDER schema shape: stand up the core tables (work_items, workflow_runs, runs, jobs, services, etc.) in a pre-migration form WITHOUT the newer migration-added columns/indexes, insert a few representative rows, then run the full openDb() path against THAT file and assert it does not throw and ends correctly migrated (expected columns + indexes present, legacy rows backfilled e.g. root_key = item_key). openDb() currently uses config.dbPath and auto-opens at import — do the smallest enabling refactor to make it testable against a given path (e.g. let openDb(path?) accept an optional path) rather than fragile env juggling. Make the test strong enough that it would FAIL against the OLD buggy schema.sql (i.e. it genuinely catches the ordering bug), ideally by seeding from a realistic 'legacy' snapshot so ANY future ordering violation trips it, not just this one. Finally, codify the invariant: add a Conventions note in CLAUDE.md that schema.sql's bootstrap must NOT create an index/constraint on a column added by a later additive migration — such indexes belong IN the migration, after the ALTER (cross-reference the migrateRunLimitLineage pattern in index.ts).
