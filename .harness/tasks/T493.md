## Overview
The owner wants a place to jot free-form notes ('Top of mind') that shows up on the dashboard home page. This unit builds ONLY the persistence + a dedicated edit page — it does not touch the Overview page itself. Content expires 3 months after it was last written (it's no longer 'top of mind' after that) — expiry is computed at read time from `updated_at`, no background job needed.

## Do
Add a small, single-row settings mechanism for one piece of free-text markdown content, following this repo's existing conventions rather than inventing a new pattern: there is currently no generic key-value/settings table in `src/db/schema.sql`, so add a minimal dedicated table (e.g. `top_of_mind` with a single row — `id INTEGER PRIMARY KEY CHECK (id = 1)`, `content TEXT NOT NULL DEFAULT ''`, `updated_at TEXT NOT NULL DEFAULT (datetime('now'))`) via `CREATE TABLE IF NOT EXISTS` in `schema.sql` (fresh DBs only need this — no ALTER TABLE migration is needed since it's a brand-new table, per the T098 rule about NOT needing migrations for new tables). Add `getTopOfMind()` / `setTopOfMind(content: string)` functions to `src/db/store.ts` (all SQL goes through store.ts, per convention) — `setTopOfMind` should UPSERT the single row and bump `updated_at` to now (a fresh write always resets the TTL clock, since the owner re-writing something makes it 'top of mind' again). Add a **3-month TTL**: define a constant (e.g. `TOP_OF_MIND_TTL_DAYS = 90`, colocated with `getTopOfMind` in `store.ts`) and have `getTopOfMind()` compute whether `updated_at` is older than the TTL; when expired, treat the content as effectively empty for the reader (return `{ content: '', expired: true, updatedAt: <the stored timestamp> }` rather than the stale text) — the DB row itself is left untouched (don't delete/clear it on read; simplest and avoids surprising the edit page's own semantics). Add two endpoints in `src/api/server.ts`: `GET /api/top-of-mind` (read-only, returns `{ content: string, updatedAt: string | null, expired: boolean }` — `content` is `''` and `expired: true` once past the TTL, `content` is the real text and `expired: false` otherwise, both `content: ''`/`expired: false`/`updatedAt: null` when never set) and `POST /api/top-of-mind` (mutating — behind the existing global loopback/token mutation guard used by every other POST endpoint in the file — body `{ content: string }`, validate it's a string, reject non-string with 400; a save always clears the expiry by refreshing `updated_at`). Add matching client methods to `dashboard/app/lib/api.ts` (`api.topOfMind()` / `api.saveTopOfMind(content)`), following the existing `get<T>`/`post<T>` helper pattern already in that file, threading the new `expired` field through the TS response type. Build a new dashboard page at `dashboard/app/top-of-mind/page.tsx` — a simple form: a `<textarea>` bound to the current content (loaded via `api.topOfMind()` on mount — if the fetched content came back expired, load the textarea from the raw stored text so the owner can see/revive what they wrote rather than an empty box; fetch it via a small addition to the GET response or a second field, your call, but the edit page must be able to show the owner their expired text, not just blank), a live markdown preview using this repo's existing markdown-rendering precedent (`react-markdown`, matching the XSS-safe pattern in `dashboard/app/components/MarkdownModal.tsx` — no `rehype-raw`), an expiry indicator (e.g. a small muted line showing 'Last updated <relative time>' and, when applicable, 'Expires in N days' or 'Expired <relative time> ago — save again to keep it on the Overview page'), and a Save button that POSTs via `api.saveTopOfMind()` and shows a success/error message (mirror the busy/result/err state pattern already used in `dashboard/app/admin/page.tsx`). Add a link to this new page in the site nav in `dashboard/app/layout.tsx` (alongside Overview/Workflows/Integrations/Logs/Admin — the `<nav>` block already flex-wraps to accommodate more links, per the T323 comment there). Update `dashboard/scripts/_dashboard-harness.mjs`: add the new `/top-of-mind` route to the `PAGES` list with a synthetic fixture for `GET /api/top-of-mind` (per the file's living-artifact rule — a UI-surface change must update this in the same change).
