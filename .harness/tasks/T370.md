## Overview

The owner wants "every adjacent job pair in every workflow must have a validation gate between them"
to be a HARD, mechanically enforced rule — not just a documented convention — so that a future
workflow shipped without gates fails the test suite (`npm test`), not just a dashboard visual gap
someone happens to notice. Add that test.

## Do

Read root `CLAUDE.md`'s "Validation gates between workflow stages" section, `src/core/dag.ts` in full
(`Dag`, `buildDag`, `deriveGates` — deriveGates is pure: it takes a `Dag` plus per-job `produces`/
`consumes` key maps and returns one `Gate` per edge where a key matches on both sides), and
`src/jobs/registry.ts` (`export const jobs: JobDefinition[]` and `export const workflows:
WorkflowDefinition[]` — both fully loaded, auto-discovered arrays; see `src/jobs/registry.test.ts` for
the existing "smoke test the registry loads" pattern to mirror for imports).

By the time this task runs, tasks adding gates to `stocks-sync`, `projects-sync`, `workouts-sync`, and
`stock-digest` should already be merged (this task depends on them) — so ALL of `tv-recs`, `places`,
`perfumes`, `movies`, `plex`, `stocks-sync`, `projects-sync`, `workouts-sync`, `stock-digest` should
have full DAG-edge gate coverage. `listening-digest` and `claude-warmer` are single-stage (no edges to
check — they trivially pass).

**Create a new test file `src/core/gate-coverage.test.ts`** (sibling to `src/core/dag.test.ts` and
`src/core/contracts.test.ts`, which it complements — those test individual contracts/dag mechanics in
isolation; this one is the whole-repo invariant check):

1. Import `jobs, workflows` from `../jobs/registry.js`.
2. For each `workflow` in `workflows`:
   a. Build a `Map<string, JobDefinition>` (or just look up by name) from `jobs` for the workflow's
      member job names.
   b. Build `produces: Map<string, string[]>` and `consumes: Map<string, string[]>` keyed by job name,
      from each member job's `produces`/`consumes` arrays (`.map(c => c.key)` — `[]` if undeclared).
   c. Build the `Dag` via `buildDag(workflow.jobs)`.
   d. Compute the full edge set: for every job name in `dag.nodes`, for every `producer` in
      `dag.dependencies.get(job)!`, `(producer, job)` is one edge.
   e. Call `deriveGates(dag, produces, consumes)` and build a `Set` of `${gate.producer}->${gate.consumer}`
      strings from the result.
   f. **Assert every edge from (d) appears in the gate set from (e).** On failure, the assertion
      message must name the workflow, producer, and consumer explicitly (e.g. `"workflow 'foo-sync':
      edge bar-stage -> baz-stage has NO matching produces/consumes key — add a validation gate (see
      src/jobs/CLAUDE.md)"`) so a future author immediately knows what broke and where to look.
3. Run this check across **every** workflow in the loaded `workflows` array — do not hardcode a list
   of workflow names, and do not skip/allowlist any workflow. This is deliberate: a NEW workflow
   (including a future private, gitignored one under `src/jobs/<name>/`) that ships without gates on
   some edge must fail this test, per the owner's "enforce it everywhere" requirement. (On a public
   checkout with no private jobs, the test simply covers fewer workflows — that's fine, it still
   covers everything that's actually loaded.)
4. **Confirm the "trivial minimum bar" design goal explicitly with a dedicated unit test**, separate
   from the real-registry walk above: construct a small synthetic 2-node `Dag` (via `buildDag`) with a
   producer job declaring `produces: [{ key: 'x', check: () => ({ ok: true }) }]` (no `shape`, no rich
   checks — the bare minimum) and a consumer declaring `consumes: [{ key: 'x', check: () => ({ ok:
   true }) }]`, and assert this passes your coverage assertion — proving a deliberately trivial
   "previous job succeeded"-style gate satisfies the enforcement mechanism, exactly as the owner
   specified ("even a trivial one is acceptable, not everything needs a rich shape-check"). This test
   does NOT touch the registry — it's pure `dag.ts` mechanics, so keep it hermetic and fast.
5. Also add a **negative-case unit test** (again synthetic, not the real registry) proving the
   assertion actually catches a gap: a 2-node `Dag` where the producer declares NO `produces` and the
   consumer declares NO `consumes` — assert your coverage-check helper reports exactly one missing
   edge. This is the regression guard for the checker itself (a checker that can't fail is useless).

Structure the per-workflow coverage logic as a small reusable helper function in the test file itself
(e.g. `missingGateEdges(dag, produces, consumes): { producer: string; consumer: string }[]`) so both
the real-registry walk (step 2) and the synthetic pass/fail unit tests (steps 4-5) exercise the exact
same logic — don't duplicate the edge/gate-diff logic between them.
