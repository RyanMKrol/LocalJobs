## Do

Let the owner mark a **needs-human** task as **done** directly from the Backlog view, via a per-task
button (mirroring the old per-task "Mark as reviewed" affordance). Marking a needs-human task done
implies it is BOTH done AND reviewed (a human did the work and reviewed it).

Use an **owner-owned overlay file** — the same mechanism as the reviewed flag (T136), NOT a mutation
of TASKS.json status (which is shell/loop-owned; the loop skips needs-human tasks and must never race
the daemon over status). Concretely:
- **Overlay store.** Add a committed, owner-owned `.harness/human-done.json` — an `id → { done: true,
  at: ISO-8601 }` map (mirroring `.harness/reviews.json`). The daemon writes it atomically
  (temp-file + rename) and commits+pushes it `[skip ci]` under the SAME repo lock used for reviews
  (`src/core/repo-lock.ts`), exactly like `commitReviewsFile` (fetch+rebase+retry; failed push =
  non-fatal warning). It is a disjoint git path from TASKS.json and reviews.json, so it never
  conflicts with the loop.
- **API.** Add a mutating `POST /api/backlog/:id/done` behind the same loopback/token guard as
  `/api/backlog/:id/reviewed`. It sets `human-done.json[id] = { done: true, at }` (and commits as
  above). It should only apply to needs-human tasks (validate the task's `gate === 'needs-human'`;
  reject otherwise with 400). Add the `api.markBacklogDone(id)` client method.
- **Done implies reviewed.** `GET /api/backlog` must overlay BOTH flags so a human-done needs-human
  task shows as done AND reviewed: report `done` from `human-done.json`, and `reviewed = true` whenever
  `human-done.json[id].done` is true (in addition to the existing reviews.json overlay). (Derive
  reviewed from done — don't require a second write.)
- **UI.** On the Backlog page (`dashboard/app/backlog/page.tsx`), render a per-task **"Mark done"**
  button on needs-human tasks that aren't already human-done; clicking calls the endpoint and
  refreshes. A human-done needs-human task then renders as done (and reviewed). Keep it consistent
  with the existing backlog row styling.

Scope boundary (explicit non-goal): this records the task as done **for the backlog view/record** and
implies reviewed. It does NOT rewrite TASKS.json status and is NOT required to unblock the loop's
`dependsOn` for that task — leave loop/selection behaviour unchanged. Update CLAUDE.md + HARNESS.md
notes (the reviews.json/owner-owned-file section) to document the new human-done overlay alongside
reviewed.
