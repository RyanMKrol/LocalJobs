## Do

Add a `category` field to services — manifest-owned, code-synced, no dashboard edit UI — so a
later task can group the Services dashboard page by it. This mirrors T292 (which did the same
thing for `WorkflowDefinition`/`workflows`), applied to `ServiceDefinition`/`services`. This is the
backend/schema half; the dashboard-rendering half is a separate, dependent task (T306).

Controlled category values (exactly 3, decided by the owner): `'cli-tool'` | `'website-scrape'` |
`'api'`. A service with no `category` set falls back to `'uncategorized'` (same fallback shape as
workflow `category` in T292) — kept as a safety net for any future service the owner forgets to
categorize, not because any service should intentionally use it today.

1. **`src/core/types.ts`** — add an optional `category?: string` field to `ServiceDefinition`
   (~line 253), alongside the other fields, with a doc comment listing the 3 controlled values.
2. **`src/db/schema.sql`** — add a `category TEXT NOT NULL DEFAULT ''` column to the `CREATE TABLE
   IF NOT EXISTS services` block (~line 184), alongside `description` (the other always-refreshed,
   code-owned column — NOT `limits_overridden`-style).
3. **`src/db/index.ts`** — add an additive migration for EXISTING databases (mirror the workflows
   `category` migration at ~lines 79-84: a `PRAGMA table_info(services)` check + `ALTER TABLE
   services ADD COLUMN category TEXT NOT NULL DEFAULT ''`). Per the T098 rule, do **not** add any
   index on this column in `schema.sql`'s bootstrap path.
4. **`src/db/store.ts`** (~lines 1227-1273):
   - Add `category: string` to the `ServiceRow` interface.
   - `upsertServiceStmt`: add `category` to the `INSERT INTO services (...)` column list and its
     `VALUES (...)` placeholders, and set `category = excluded.category` **unconditionally** in the
     `ON CONFLICT ... DO UPDATE SET` clause — always refreshed from the manifest on every sync, like
     `description`/`paid`, NOT gated behind `limits_overridden` (no `_overridden` column for
     `category`, no reconciliation branch, no dashboard edit endpoint — matches how T292 treated
     workflow `category`).
   - `syncService(def)`: pass `category: def.category ?? 'uncategorized'` into the
     `upsertServiceStmt.run({...})` call.
   - `listServices()` / `getServiceRow()` already do `SELECT * FROM services` — no change needed;
     `category` flows through automatically once it's a real column.
5. **Set `category` on every existing service manifest** (`src/services/*.service.ts`, one line
   each):
   - `'cli-tool'` — `src/services/claude-cli.service.ts`
   - `'website-scrape'` — `src/services/fragrantica.service.ts`
   - `'api'` — `src/services/gemini.service.ts`, `src/services/github.service.ts`,
