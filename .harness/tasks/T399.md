## Overview

The owner wants a check-in confirming that nothing in this repo currently WRITES to
DynamoDB. `src/services/dynamodb.service.ts` reuses the `ryankrol.co.uk` website's AWS
key (not a scoped least-privilege key — see the service file's own header comment and
T251), so an accidental write from `local-jobs` would mutate data owned by that other
project. This is a read-only, human confirmation task, not a code change.

## Do

This is a `needs-human` check-in — there is no code to write. Present the following
re-verified static-analysis finding to the owner and have them confirm it (or flag a
gap) during review:

- `src/services/dynamodb.service.ts` DOES import and expose write-capable AWS SDK
  commands: `PutCommand`, `DeleteCommand`, `BatchWriteCommand` (from
  `@aws-sdk/lib-dynamodb`), wrapped as the exported functions `dynamoPut`,
  `dynamoDelete`, `dynamoBatchWrite`.
- A repo-wide, re-verified grep (excluding `data/` directories, which can contain
  stale cloned copies of this very repo via `projects-sync`'s clone-and-summarize
  stage — see the root `CLAUDE.md` "Never let `data/` folders be scanned" gotcha)
  confirms:
  - `dynamodb.service.ts` is the ONLY file under `src/` that imports `@aws-sdk/lib-dynamodb`
    at all.
  - `PutCommand` / `DeleteCommand` / `BatchWriteCommand` are referenced NOWHERE outside
    that one service file.
  - The exported write functions `dynamoPut` / `dynamoDelete` / `dynamoBatchWrite` are

    test file — no job or workflow currently reads OR writes via this service (the read
    helpers `dynamoGet`/`dynamoQuery` are equally unused right now).
  - No dynamic `import()` of the service was found anywhere outside the test file
    either.
- Conclusion: as far as static analysis can tell, **no job or workflow in `local-jobs`
  currently performs a DynamoDB write**, simply because the only write-capable code
  path (and indeed the whole service) is never invoked by anything yet.
- Caveat worth surfacing to the owner explicitly: this is a static, grep-based read —
  it would not catch code added after this check was performed, nor anything that
  reaches the service through an unusual indirection this search didn't anticipate.
  That's exactly why the owner asked for a `needs-human` check-in rather than treating
  the grep as fully conclusive.

Do not modify any code as part of this task. If, during review, the owner (or a fresh
