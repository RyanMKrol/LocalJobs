## Overview
The movies rec-merge stage silently swallows per-suggestion TMDB search failures, while its near-identical tv-recs twin (tv-rec-merge.ts) counts them and throws if any occurred this run. This is one of the concrete divergence defects R01 catalogs (the 'B13 rec-merge missing searchFailed throw' item). Land it as a small behavior-preserving fix FIRST, so the later shared-pipeline extraction can be a pure structural move with no behavior change.

## Do
Bring src/workflows/movies/stages/merge.ts to parity with src/workflows/tv-recs/stages/tv-rec-merge.ts on the searchFailed handling. In tv-rec-merge.ts the counters object has a `searchFailed: number` field (~line 105), it is incremented when a per-suggestion TMDB search errors (~line 169), initialized to 0 (~line 290), and at the end of the merge (~line 374-375) the stage throws an Error whose message is `TMDB search failed for N suggestion(s) this run — see warn logs above for titles.` when counters.searchFailed > 0. movies/merge.ts currently has NO searchFailed counter or throw. Add the identical mechanism to movies/merge.ts: count per-suggestion search failures, log a warn per failed title (matching tv's log style), and throw the same summarizing error at the end of the run when any occurred. Keep every other existing movies/merge.ts behavior identical (the resilient top-up loop, the per-item skip-and-continue, the always-write-a-list-then-succeed shape for downstream notify). This aligns the two stages so the R01 extraction of a single shared runMerge is a no-op behaviorally. Do NOT extract the shared module in this unit — that is a later unit; here just fix the movies stage in place. Update movies/CLAUDE.md's Stage 3 section to note the stage now fails the run when any TMDB search errored (mirroring tv-recs).
