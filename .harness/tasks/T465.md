## Overview
Add a per-service request-timeout value as a first-class, dashboard-editable ServiceDefinition field — following the EXACT same code-seeded-but-owner-overridable `_overridden` reconcile pattern the root CLAUDE.md already documents for `ratePerMinute`/`dailyCap`/`monthlyCap` — so a timeout change (like the historical T392 Plex-timeout bump) is discoverable/editable from the Integrations page instead of requiring a code or env-var change. Wire BOTH Plex's existing hard-coded `PLEX_REQUEST_TIMEOUT_MS` AND `src/services/claude.ts`'s `LOCALJOBS_CLAUDE_TIMEOUT_MS` through this new mechanism in the same pass — the owner confirmed both should migrate together rather than leaving claude-cli as a follow-up, since it has the exact same invisible-env-var gap.

## Do
Depends on `plex-service-a`. This is backend-only (db/api/core); the dashboard UI is a separate dependent task. Read the root `CLAUDE.md`'s `ratePerMinute`/`dailyCap`/`monthlyCap` `_overridden` convention description and `src/db/store.ts`'s `syncService`/`updateServiceLimits`/`ServiceLimits`/`ServiceRow` + `src/api/server.ts`'s `POST /api/services/:name/limits` as the exact precedent to extend, NOT reinvent: (1) add `timeoutMs?: number` to `ServiceDefinition` in `src/core/types.ts`, documented like the other limit fields; (2) add `timeout_ms INTEGER` (NULL = no override / use code default) and reuse the SAME `limits_overridden` flag (a service-wide 'user has customized this service's limits' flag already covers rate/daily/monthly — extend it to also cover timeout, rather than adding a second overridden flag, unless you find a concrete reason the two need independent reconcile semantics) to `services` in `src/db/schema.sql`, PLUS an additive `ALTER TABLE` migration in `src/db/index.ts` for existing DBs (per the repo's T098 rule — do NOT add any index on the new column in `schema.sql`'s bootstrap path); (3) extend `syncService`'s upsert to seed `timeout_ms` from `def.timeoutMs ?? null` the same CASE-preserving way the other three fields work, extend `ServiceRow`/`ServiceLimits` to include `timeout_ms`, and extend `updateServiceLimits` (and its `POST /api/services/:name/limits` request/response shape in `src/api/server.ts`) to accept/persist `timeout_ms` alongside the existing three fields — reuse the SAME endpoint rather than adding a new one, since it's the same 'edit this service's limits' concept; (4) add an exported helper in `src/core/services.ts`, e.g. `effectiveServiceTimeoutMs(name: string, fallbackMs: number): number`, mirroring the existing `effectiveLimits()` internal helper's override-wins-else-code-default logic (read the DB row's `timeout_ms` when `limits_overridden`, else the `ServiceDefinition.timeoutMs`, else the caller's `fallbackMs`) — export it since, unlike rate/quota (enforced generically inside `callService`), a timeout must be read by each service's OWN client code at the point of its real request/process-level timeout. The owner explicitly confirmed this design (Option 1: each service keeps its own REAL, cancelling timeout mechanism, just reading the value from this helper instead of a raw env var) over the alternative of a generic `Promise.race`-based soft-timeout built into `callService` itself (rejected: it can't truly cancel in-flight work, and would regress the real cancellation Plex/claude-cli already have). (5) In `src/core/plex-client.ts`, replace the module-load-time `PLEX_REQUEST_TIMEOUT_MS` constant's direct env read with a per-call (or lazily-cached-and-invalidatable) lookup via `effectiveServiceTimeoutMs('plex', Number(process.env.PLEX_REQUEST_TIMEOUT_MS ?? 300_000))` so a dashboard edit takes effect without a daemon restart, matching how every other user-editable service/workflow property in this repo is documented to behave; set `plex.service.ts`'s `timeoutMs` to the same 300_000 default so the two stay in sync. (6) Do the SAME for `src/services/claude.ts`'s `LOCALJOBS_CLAUDE_TIMEOUT_MS` — read the `claude-cli` service's existing `ServiceDefinition` (find it under `src/services/` — it may need a `timeoutMs` default added matching claude.ts's current env-var default) and replace claude.ts's direct env read with `effectiveServiceTimeoutMs('claude-cli', <its existing default>)`, keeping its real cancellation mechanism (whatever `runClaude`/the Claude CLI child-process invocation currently uses to enforce the timeout) unchanged, just sourcing the number from the new helper instead of the raw env var.
