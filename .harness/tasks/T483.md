## Overview
Part of a 6-task architecture fix for the "self-referential `inputKeys()` trap" (found in `places`, `plex-language-fix`, `projects-sync`): a limited-run root stage must query its TRUE live external source every time, never read back this workflow's own prior output. This task creates a new `fs` pseudo-service so a LOCAL file read (like `places`'s raw CSV parse, T484) can be governed by the same `ServiceDefinition`/`callService` contract as every real external dependency — deliberately embracing the conceptual stretch of treating a local file read as a "service," so the eventual "every root's `inputKeys()` must call a named service" rule (T487/T488) has zero carve-outs.

## Do
Create `src/services/fs.service.ts`, default-exporting a `ServiceDefinition` named `fs`. Model it directly on the existing `src/services/plex.service.ts` precedent (Plex is free, LAN-hosted, owner-controlled, "no meaningful rate limit or spend quota to model" — it deliberately declares NO `ratePerMinute`/`dailyCap`/`monthlyCap`/`minIntervalMs`/`maxJitterMs`). `fs.service.ts` follows the identical shape: `paid: false`, no rate/quota fields, a `rateLimitSource` string explaining there's no external limit to model because the "external" dependency here is the owner's own local disk (data/raw/** files the owner placed there themselves) — the service exists purely so a local-file-backed root's `inputKeys()` can satisfy the universal `inputKeysService` contract (T487), not because there's a real limit to govern. For `category`: the current controlled vocabulary is `cli-tool`, `website-scrape`, `api` (confirmed via a live grep of `src/services/*.service.ts` this session) — none fit "local filesystem." Add a new category value `local` (update the Services dashboard page's category grouping if it enumerates known categories explicitly — check `dashboard/app/integrations/page.tsx` for a hardcoded category list). Set `timeoutMs` short (a local file read is near-instant) or omit it, relying on the caller's own timeout. This task is scoped to JUST the service definition — it does NOT wire anything into `resolveInputKeys()` (that's T484, which depends on this task).

## Done when
`src/services/fs.service.ts` exists, default-exports a valid `ServiceDefinition` named `fs`, is auto-discovered by the registry (`src/workflows/registry.ts` scans `src/services/*.service.ts`), and appears on the Services/Integrations dashboard page after a daemon restart. It has `paid: false` and no rate/quota fields, matching the `plex.service.ts` "no real limit" pattern, with a `rateLimitSource` string explaining the local-file rationale. A new `category: 'local'` value is used (or `'uncategorized'` if the conversion-time investigation determines a new category needs more supporting changes than warranted — note whichever choice is made in the worklog). A unit test (`src/services/fs.service.test.ts`) confirms the service definition loads and has the expected shape (no rate limits configured). `npx tsc --noEmit` and `npm test` are green. This task does NOT modify `places`, `plex-language-fix`, or any workflow stage — it only adds the new service definition.
