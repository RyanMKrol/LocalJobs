## Overview

The dashboard's decoupled Input/Output display for `stock-digest` (`stageIoLists` in `src/db/store.ts:775`, `GET /api/workflow-runs/:id/stage-io` in `src/api/server.ts:1912`) currently only supports ONE job as the "outputs" side. Later tasks need it to support several jobs on EITHER side (a parallel DAG wave, or a workflow's whole root/terminal wave) — this task generalizes the function and endpoint to do that, with no change to today's single-job behavior.

## Do

In `src/db/store.ts`, generalize `stageIoLists` (currently `stageIoLists(jobName: string, predecessorJobNames: string[], workflowRunId: string): StageIoLists`, around line 775) to accept **arrays of job names on both sides** — e.g. `stageIoLists(outputJobNames: string[], inputJobNames: string[], workflowRunId: string): StageIoLists`. The `outputs` list must be the UNION of every `work_items` row any of `outputJobNames` recorded THIS run (via the existing `work_item_runs` linkage join — do not introduce any `root_key`-based collapsing or pairing), and `inputs` must be the UNION of every row any of `inputJobNames` recorded this run. Preserve ordering (currently `ORDER BY item_key`, or `ORDER BY job_name, item_key` for the multi-job input query) so results are deterministic.

Update the ONE existing caller in `src/api/server.ts` (`GET /api/workflow-runs/:id/stage-io?job=<job>`, around line 1912) to pass `[jobParam]` as the new `outputJobNames` array — the endpoint's request/response shape for the EXISTING single-job case must be byte-for-byte unchanged (still `{ inputs, outputs, predecessorJobs, job }`).

Update the `StageIoLists` TypeScript type / any other internal callers in `src/db/store.ts` to match the new signature. Do not touch the dashboard client (`dashboard/app/lib/api.ts`'s `workflowRunStageIo`) — its request/response shape is unaffected by this task.
