## Overview

`vercel-daily-redeploy`'s job (`vercel-redeploy`) spawns `vercel --prod --yes` directly via
`child_process.spawn`, with no service gating at all — no `vercel.service.ts` exists anywhere in the
repo. Found during a repo-wide rate-limiting audit: this is a genuine external side effect (a real
production deploy against Vercel's Hobby-plan quota — 100/day, 100/hour, 60/5min, per root
`CLAUDE.md`'s own account of why this workflow exists as a fallback) with zero mechanical protection
against, e.g., a manual "Run now" landing the same day as the scheduled run and firing two deploys in
quick succession. Add a real `vercel.service.ts` and route the CLI spawn through it.

## Context (confirmed current)

`src/jobs/vercel-daily-redeploy/vercel-redeploy.job.ts`:
- `defaultSpawn` (lines 11-13): `spawn('vercel', ['--prod', '--yes'], { cwd, stdio: [...] })`.
- `runVercelRedeploy` (lines 25-102) calls `spawnFn(checkoutPath)` (line 50) directly inside a
  `new Promise` that resolves/rejects based on the child process's exit code — no `callService`
  import, no service gating anywhere in this file.
- The job (`name: 'vercel-redeploy'`, lines 104-114) has `timeoutMs: 660_000` (11 min) and
  `maxRetries: 1`; the workflow (`vercel-daily-redeploy.workflow.ts`, not shown here — check it)
  schedules this once daily (`'0 23 * * *'`) but ALSO allows a manual "Run now" from the dashboard
  like any workflow, with no framework-level protection against that landing the same day.
- No `src/services/vercel.service.ts` exists (`find src/services -iname "*vercel*"` returns nothing).

## Do

### 1. New file: `src/services/vercel.service.ts`

Default-export a `ServiceDefinition`:
- `name: 'vercel'`
- `category: 'api'` (mirroring other CLI-driven services — actually check the existing
  `category` vocabulary in root `CLAUDE.md`'s "A service's `category` is manifest-owned only"
  section: controlled values are `'cli-tool'` (used by `claude-cli`), `'website-scrape'`
  (`fragrantica`), and `'api'` for the rest. Since this is a CLI tool invocation, not an HTTP API
  call, use `category: 'cli-tool'` to match `claude-cli`'s precedent.)
- `description`: something like "Vercel CLI — direct production deploy trigger for the
  ryankrol.co.uk checkout (a daily safety-net, see vercel-daily-redeploy)."
- Conservative caps — this only needs to run at most once a day by design, so err on the side of
  caps that would only ever trip if something is genuinely wrong (e.g. a scheduling bug firing
  repeatedly): `dailyCap: Number(process.env.VERCEL_DAILY_CAP ?? 3)` (covers the scheduled run plus
  a couple of manual re-runs the same day without ever blocking legitimate single-digit daily use),
  `monthlyCap: Number(process.env.VERCEL_MONTHLY_CAP ?? 30)`. Do NOT set `ratePerMinute` or
  `minIntervalMs` — a deploy takes minutes by itself (the job's own 11-minute timeout), so a
  per-minute/interval throttle is meaningless here; the caps are the only meaningful governor for
  this kind of call.
- `paid: false` (Vercel's Hobby plan is free; the concern is quota exhaustion, not billing).

### 2. Wrap the spawn call in `vercel-redeploy.job.ts`

Import `callService` from `../../core/services.js`. Wrap the deploy operation (the `new Promise`
block in `runVercelRedeploy`, or just the `spawnFn(checkoutPath)` call plus its promise — whichever
reads more naturally, but the WHOLE deploy-and-wait-for-exit operation should be the thing gated, not
just the initial spawn) in `callService('vercel', () => ...)`. `callService`'s quota check runs
BEFORE `fn()` is invoked, so this correctly blocks a call that would exceed the daily/monthly cap
before ever spawning `vercel` — it does not interact awkwardly with the deploy's own multi-minute
runtime (the `await fn()` inside `callService` just awaits however long the deploy takes, exactly
like any other long-running service call in this codebase).

Catch `QuotaExceededError` (from `../../core/services.js`) around the `callService` call and, if
thrown, `ctx.log` a clear warn (mirroring the soft-skip pattern this job already uses for
`RYANKWOL_CO_UK_PATH` not configured) instead of letting the run fail — a hit cap should read as "no
deploy attempted today, budget exhausted" not a job failure requiring investigation.

### 3. Tests

Add `vercel-redeploy.job.test.ts` (check whether one already exists) or extend the existing test
file: a test proving `runVercelRedeploy` calls through `callService('vercel', ...)` — inject a fake
`callService` (matching however other stage tests in this repo fake service-gating, e.g.
`hevy-sync.test.ts`'s pattern) and assert it's invoked with the right service name once per call to
`runVercelRedeploy`. Also add a test for the `QuotaExceededError` soft-skip path (a stubbed
`callService` that throws `QuotaExceededError` should result in a warn log, not a thrown error out of
`runVercelRedeploy`).

### 4. `.env.example` and root `CLAUDE.md`

Document `VERCEL_DAILY_CAP`/`VERCEL_MONTHLY_CAP` in `.env.example` (commented out, showing the
defaults, per this repo's convention for optional env-overridable limits). Update the
`vercel-daily-redeploy` section of root `CLAUDE.md` to mention the new service gating (a short
addition — the workflow's core behavior/design is otherwise unchanged).
