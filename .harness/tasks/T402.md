## Overview

The Plex DHCP-self-heal subnet scan (`src/jobs/plex/client.ts`, T149) wastes roughly half its fixed
20-second scan budget probing addresses on virtual/VPN network interfaces (observed: Tailscale) that
can never host the real Plex server, and has no way to prioritize the subnet the server was last
known to be on — so on a machine with more than one non-internal IPv4 interface, the scan can
plausibly exhaust its whole time budget without ever reaching the real LAN and report "Plex
unreachable" even though the server is up. Fix the candidate enumeration to exclude known-virtual
interfaces by name and to scan the previously-configured host's own subnet first.

## Do

**Confirmed root cause (verified against current code, 2026-07-04):**
- `enumerateSubnetHosts()` (`src/jobs/plex/client.ts:170-189`) iterates `Object.values(os.networkInterfaces())`
  and includes **every** non-internal IPv4 interface with no filtering on interface type/name — it
  reads only `ni.address`/`ni.family`/`ni.internal`, discarding the interface NAME entirely. For each
  qualifying interface's /24 it emits 254 hosts × 2 schemes (https + http) = 508 URL candidates. A
  real report of **1016** candidates is exactly `508 × 2` — two non-internal IPv4 interfaces were
  scanned, and since the dashboard was being accessed via a Tailscale address in that session, the
  second interface is very plausibly Tailscale's own virtual adapter (`utun*` on macOS), which can
  structurally never route to a LAN device — roughly half the scan budget is provably wasted.
- The scan budget is small and fixed: `SCAN_OVERALL_CAP_MS = 20_000` (`:28`, `PLEX_SCAN_CAP_MS`),
  `SCAN_CONCURRENCY = 32` (`:26`, `PLEX_SCAN_CONCURRENCY`), `PROBE_TIMEOUT_MS = 1_500` (`:24`,
  `PLEX_PROBE_TIMEOUT_MS`), enforced in `scanForPlexHost` (`:135-162`) via a wall-clock deadline
  checked per-worker. On a subnet where most addresses have no listening device, an unanswered probe
  commonly takes the full 1.5s timeout to resolve rather than an instant refusal — at 32 concurrent
  workers that's roughly `20_000 / 1_500 × 32 ≈ 426` probes completable before the cap fires, well
  under half of 1016. `prefixes` is a plain `Set<string>` built by iterating
  `Object.values(os.networkInterfaces())` in whatever order Node returns interfaces (`:171-180`) — NOT
  sorted or prioritized — so if the virtual interface's subnet happens to be enumerated first, the
  scan can exhaust its entire budget there and never reach the real LAN subnet at all. This is the
  shared client behind THREE production workflows (`missing-tv-seasons`, `tv-recommendations`,
  `movie-recommendations`), so this budget-exhaustion failure mode degrades all of them identically,
  not just `plex-space-saver`.
- Out of scope for this task (do not fold in): investigating why the specific configured
  `PLEX_HOST` in the observed run stopped answering in the first place — that's an orthogonal DHCP
  IP-change question (which the self-heal scan is *designed* to handle) and this scan-budget fix
  stands on its own regardless of that cause.

**Fix — two complementary, narrowly-scoped changes to candidate enumeration in `client.ts`:**

1. **Exclude known-virtual/VPN interface names.** Change `enumerateSubnetHosts()` to iterate
   `Object.entries(os.networkInterfaces())` so the interface NAME is available, and skip any
   interface whose name matches a documented "known-virtual" pattern before it contributes any
   candidate hosts — e.g. `/^(utun|tailscale|wg|ppp|ipsec|awdl|llw|bridge|docker|vmnet|zt)/i`
   (Tailscale on macOS shows as `utun*`, on Linux as `tailscale0`; also covers WireGuard, generic
   VPN/PPP/IPsec adapters, macOS AirDrop/peer-to-peer (`awdl0`/`llw0`), Docker bridges, VMware
   `vmnet*`, ZeroTier). Log which interfaces were included vs. excluded (name + address) at scan
   time — directly serving the owner's stated need to understand *why* a host wasn't found, not
   just that it wasn't.
2. **Prioritize the configured host's own subnet.** `doResolvePlexHost` already has `configuredHost`
   in scope when it falls back to a scan (`client.ts` around `:112-117`). Thread the configured
   host's `/24` prefix (parsed the same way `enumerateSubnetHosts` already derives a prefix from an
   address) into candidate enumeration so that subnet's hosts are placed FIRST in the returned
   candidate list, ahead of any other qualifying subnet. Rationale: a DHCP lease change alters the
   last octet, not the whole subnet, so the previously-known-good subnet is very likely still the
   right one even when the specific configured host no longer answers — trying it first means a
   second physical interface (if one is ever legitimately present) doesn't starve the real answer of
   scan budget.
   - Keep this additive/backward compatible: `enumerateSubnetHosts` gains an optional parameter
     (e.g. a preferred-prefix string, or an injectable `os.networkInterfaces()`-shaped map for
     testability) with a default that preserves current behaviour when no configured host is
     available (e.g. `PLEX_HOST` unset).
3. Do not change `SCAN_OVERALL_CAP_MS`/`SCAN_CONCURRENCY`/`PROBE_TIMEOUT_MS` defaults as part of this
   task — the candidate-list reduction from (1)+(2) is the fix; widening the budget is a separate
   lever the owner didn't ask for here.
4. Keep the existing `machineId` gating, caching (`resolvePlexHost`'s per-process cache), and the
   final "Plex unreachable — set PLEX_HOST…" error path exactly as they are — this task only changes
   *which* candidates are generated and in *what order*, not the resolution/probe/error logic around
   them.

## Done when

- `enumerateSubnetHosts()` (or its replacement) excludes hosts derived from interfaces whose name
  matches the documented known-virtual pattern, verified by a unit test that injects a
  `os.networkInterfaces()`-shaped fixture containing both a real LAN-like interface name (e.g.
  `en0`) and a virtual one (e.g. `utun3` and/or `tailscale0`) and asserts only the real interface's
  subnet produces candidates.
- When a configured host's subnet is among the qualifying candidate subnets, its hosts are ordered
  first in the returned candidate list — covered by a unit test with ≥2 qualifying subnets, one
  matching the configured host's prefix, asserting that subnet's hosts appear before the other's.
- With no configured host (or one whose subnet isn't among the candidates), behaviour matches
  today's (no prioritization applied, no crash).
- A scan run logs which interface(s) were included/excluded by name, so a future "why didn't it find
  Plex" investigation doesn't require reading the source.
- `resolvePlexHost`'s existing test suite (`src/jobs/plex/plex.test.ts`) — configured-host-reachable,
  scan-fallback, machine-id gating, caching, not-found error — all still pass unmodified in behaviour
  (only the candidate list generation changes).
- `npx tsc --noEmit` and `npm test` both pass.
- No new env var is required for this fix to take effect (the existing `PLEX_SCAN_*` env vars are
  unaffected); if the virtual-interface pattern is made env-overridable, document it in
  `.env.example` alongside the existing `PLEX_SCAN_*` vars — otherwise no `.env.example` change is
  needed.
