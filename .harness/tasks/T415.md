## Overview

Split `stock-digest`'s combined `stock-portfolio-snapshot` stage (which currently both fetches
Trading212 positions AND resolves each one's ISIN + real-world ticker via OpenFIGI) into two
separate stages, mirroring the same fetch/resolve split already done for `stocks-sync`. Unlike
that `stocks-sync` change, **this is a pure structural split — the ISIN/OpenFIGI resolution logic
itself is NOT being removed or changed**, only moved into its own stage: `stock-sector-lookup`
genuinely uses the resolved ticker (as its preferred Finnhub query symbol), so this resolution
stays load-bearing here, unlike in `stocks-sync` where it was purely cosmetic.

## Context (read before making any change)

Today `stock-digest` is a 3-job DAG: `stock-portfolio-snapshot` fans out to BOTH
`stock-sector-lookup` (`dependsOn: ['stock-portfolio-snapshot']`) AND `stock-digest-build`
(`dependsOn: ['stock-portfolio-snapshot', 'stock-sector-lookup']`, a genuine fan-in).
`stock-portfolio-snapshot` (`src/workflows/stock-digest/stages/stock-portfolio-snapshot.ts`) currently:
fetches Trading212 Invest (+ optional ISA) positions, normalizes them, resolves each one's ISIN +
real-world ticker via `src/services/trading212.service.ts`'s `resolveTickers`/
`resolveOpenFigiTickersBatched` (unchanged shared functions — also used independently by
`stocks-sync`, do not touch them), writes `data/out/portfolio.json`, and records ONE combined
`work_items` row per ISO week (`weekKey(now)`, `src/workflows/stock-digest/lib.ts`).

All three of today's stages deliberately share the SAME `weekKey(now)` as their lineage `rootKey`
(a documented, intentional choice in `stock-digest.workflow.ts`'s doc comment) — the new stage
added by this task must follow the same convention (no `rootKey`/`parentKey` args needed, since
using the same key on every stage makes them "same-key stages" per the standard root-key
resolution rule).

**This is a job-layer-only change** — no db/api/dashboard files need touching. This workflow's
`StageIoPanel` display (`run?.workflow_name === 'stock-digest'` gate in
`dashboard/app/workflow-runs/[id]/page.tsx`) already renders per-stage decoupled inputs/outputs
generically by job name, so a new stage automatically gets its own panel with zero dashboard code
changes — confirm this is still true rather than assuming.
