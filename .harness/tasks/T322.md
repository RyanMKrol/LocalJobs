## Do

Add a new mutating endpoint `POST /api/workflows/reset-output-all` in `src/api/server.ts` that runs the SAME per-workflow reset logic as the existing `POST /api/workflows/:name/reset-output` handler (around `src/api/server.ts:1666`), but across EVERY workflow in one call, so the owner can wipe all workflow output from one place instead of visiting each workflow's detail page.

Behaviour (mirror the existing per-workflow endpoint's building blocks — `findWorkflowDataOut`, `resetWorkflowOutput` from `src/db/store.ts`, `deleteDataOutContents` — and `listWorkflows()` from `src/db/store.ts` to enumerate every workflow):

1. Iterate every workflow returned by `listWorkflows()`.
2. For EACH workflow, independently:
   - If `workflowRunInProgress(name)` is true, SKIP that workflow (do NOT reset it) and record it in the response as skipped with a reason (e.g. `'active run in progress'`). **Do not let one active-run workflow block the reset of any other workflow** — this is a deliberate design decision: the bulk operation is best-effort across all workflows, not all-or-nothing. (Contrast with the single-workflow endpoint, which returns 409 and does nothing for that one workflow — here that per-workflow refusal becomes a per-workflow skip within an otherwise-successful bulk call.)
   - Otherwise, run the exact same steps the single-workflow endpoint runs: look up `findWorkflowDataOut(name)`, call `resetWorkflowOutput(name)`, then `deleteDataOutContents(outDir)` if `outDir` was found. Record the per-workflow result (itemsDeleted, runsDeleted, wfRunsDeleted, filesRemoved, outDir).
3. There is no request body required (no `{ name }` — it always targets ALL workflows). Reject with 400 only if the request has an unexpected body shape that looks like an accidental misuse (optional — keep it simple; a bare POST with no/empty body is the expected call).
4. Respond 200 with a structured summary, e.g.:
   ```json
   {
     "ok": true,
     "totalWorkflows": 12,
     "resetCount": 10,
     "skippedCount": 2,
     "results": [
       { "name": "places", "status": "reset", "itemsDeleted": 5, "runsDeleted": 3, "wfRunsDeleted": 1, "filesRemoved": 2, "outDir": "places/data/out" },
       { "name": "movie-recommendations", "status": "skipped", "reason": "active run in progress" }
     ]
   }
   ```
5. Log a single summary line via `console.log` (mirroring the existing endpoint's `[api] reset-output "<name>": …` log line) listing total reset vs skipped counts and which workflows were skipped and why.
6. This is a MUTATING, dangerous, whole-fleet-affecting endpoint — it MUST sit behind the exact same loopback/token mutation guard (`authoriseMutation`) as every other POST endpoint (`/toggle`, `/run`, `/schedule`, `/concurrency`, `/notify`, `/reset-output`). Do not add a separate auth mechanism.
7. Register the route BEFORE the existing `parts[2]`-based `/api/workflows/:name/reset-output` route match (since `reset-output-all` has no `:name` segment — `parts[1] === 'workflows'`, `parts[2] === 'reset-output-all'`, `parts.length === 3`), so it doesn't get shadowed or misrouted by the `:name`-based route's parts-length matching. Add a client helper `api.resetAllWorkflowsOutput()` in `dashboard/app/lib/api.ts` (mirroring the existing `api.resetWorkflowOutput(name)` at line ~599) that POSTs to the new endpoint with no body and returns the parsed JSON response — this task should add the client helper even though the UI that calls it is a separate follow-up task, so that follow-up can consume a ready-made typed helper.

Do not touch `dashboard/app/**` page components in this task (that's the paired UI task) — only `src/api/server.ts` (the new route) and `dashboard/app/lib/api.ts` (the new client helper), plus a unit test.
