## Do

Build the **movie-gaps workflow** — the DETERMINISTIC half of the movie audit (designed in T109;
the LLM recommendation layer is T146, built on top). A monthly workflow that scans the Plex movie
library, detects **franchise gaps** via the TMDB Collections API (films you own *some but not all*
of), and pushes a monthly digest. Validated live in T109 (1,538 movies → **167 real franchise
gaps** in ~46s, e.g. Saw X, Jurassic World Rebirth, Toy Story 5, Ghostbusters, Paddington). This
is a **public example workflow with private data/tokens** (like the TV workflow; the owner's
PlexUtils repo is already public) — code committed; tokens in `.env`, library data in `data/`,
both gitignored.

**Reuse the Plex + TMDB foundation from T144** (this task `dependsOn` T144): the Plex host
resolution + JSON section fetch + `tmdb://` GUID extraction, and the rate-limited `tmdb` service.
Factor/share rather than duplicate.

### Connectivity (same as T144)
- Env (NAMES + placeholders in `.env.example`): `PLEX_HOST`, `PLEX_API_TOKEN`, optional
  `PLEX_MACHINE_ID`, `TMDB_API_TOKEN` (Bearer; legacy `TVDB_API_TOKEN` fallback), and
  `PLEX_MOVIE_SECTION` (default `4`).
- Plex self-signed cert → scoped `https.Agent({ rejectUnauthorized: false })` for Plex only
  (never global). Request JSON (`Accept: application/json`); section items are under
  `MediaContainer.Metadata` (NOT `Directory`).

### Stages
1. **movie-snapshot** — GET `/library/sections/<PLEX_MOVIE_SECTION>/all?includeGuids=1`. Per
   movie capture `title`, `year`, `tmdbId` (from the `tmdb://` guid), `ratingKey`, and the taste
   metadata Plex already returns: `Genre[]`, `Director[]`, `Country[]`, `audienceRating`/critic
   `rating`. Build the **owned set** = all `tmdbId`s. Write the snapshot + a **taste profile**
   (per-genre / per-director / per-decade / per-country counts; T146 consumes this) to
   `data/out/`. Re-scans fresh every run.
2. **franchise-gaps** — the deterministic detector via the **TMDB Collections API**:
   - For each owned movie with a `tmdbId`: GET `/movie/{id}` → `belongs_to_collection`
     (`{id,name}` or null). Collect the **distinct** collection ids.
   - For each collection: GET `/collection/{id}` → `parts[]` (all franchise films). Keep only
     **released** parts (`release_date` present and ≤ today).
   - A **gap** = a released part whose tmdb id is NOT in the owned set. Emit per collection the
     missing films `{ collectionName, tmdbId, title, year, tmdbRating (vote_average) }`. **Dedupe
     collection fetches** (fetch each collection once even if you own several members).
   - Route every TMDB call through the `tmdb` service. Recompute fresh every run; write
     `data/out/franchise-gaps.json`.
   - **NO quality filter and NO "skipped on purpose" heuristics** (owner decision, T109): surface
     EVERY factual gap. `tmdbRating` is included for the owner's context ONLY — never used to hide
     anything.
3. **movie-gaps-notify** — build a monthly **markdown report** (`data/out/…md`, recorded via
   `detail.markdown` for the dashboard preview per T110) listing all current, non-ignored franchise
   gaps grouped by collection, each with its TMDB rating + a TMDB link. Push **ONE digest
   notification** (`core/notifier.ts` → ntfy + macOS) summarizing the newly-detected gaps.
   **First run = ONE big digest** of the whole current backlog (~167); thereafter only
   newly-detected gaps.

### Idempotency, dedup, and the ignore mechanism (load-bearing)
- Each surfaced gap is a `work_items` row keyed `(movie-gaps-notify, "<collectionMemberTmdbId>")`.
  Notify dedups per key so a given gap is announced **once** (the ledger is a "**notified**" log,
  same inversion as the TV workflow — snapshot/gaps RECOMPUTE every run; only notify uses the
  ledger to dedup).
- **Ignore-to-suppress (owner decision, T109): a gap stops being surfaced IN REPORTS only when the
  owner MANUALLY ignores it** — no assumptions about why a film was skipped. The existing `ignored`
  status acts only on `failed`/stuck items, so **extend "ignore" to apply to a surfaced
  (non-failed) gap**: a dashboard control + `POST` endpoint + store fn that marks a chosen gap
  `ignored`; an ignored gap is then excluded from BOTH future reports AND notifications and never
  resurfaces (even though the film is still un-owned). Owner action only — nothing auto-ignores.
  (If this surfaced-item-ignore capability is large, it MAY be split to its own follow-up — but at
  minimum the notify dedup must prevent monthly re-nagging.)

### Framework shape (same inversion as TV)
- Do NOT declare `inputKeys()` → not "limitable"; scheduled-only. All stages key by `tmdbId`, so
  `root_key = item_key` (no lineage args). Snapshot/gaps recompute each run; only notify uses the
  ledger.
