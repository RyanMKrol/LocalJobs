## Overview

A job's own read-only page (`/jobs/[name]`) and the page for any single run of that job
(`/runs/[id]`) should both show a real, paragraph-length, human-readable explanation of what the job
actually does — today only `/jobs/[name]` shows anything, and the `description` field it shows is
written repo-wide as a terse one-liner (e.g. "Stage 1: find each perfume's Fragrantica URL via Claude
Code (web search).") rather than a paragraph. This task builds the (small) mechanism change and, as
the worked example, writes real paragraph descriptions for the 4 `perfumes` jobs — the workflow the
owner was looking at when this idea came up.

## Context (confirmed)

- `JobDefinition.description` (`src/core/types.ts`) is already `description?: string` — no type
  change needed. The DB column (`jobs.description`, `src/db/schema.sql` line 7) is already an
  unconstrained `TEXT NOT NULL DEFAULT ''`. `syncJob` (`src/db/store.ts`) already round-trips whatever
  string the job manifest supplies. **No schema/migration/API-shape change is required at all** — this
  is purely (a) a content-length convention change for the same field, and (b) one new dashboard render
  site.
- Grepped every `.description` usage across `dashboard/app/**`: a JOB's `description` (distinct from a
  WORKFLOW's or SERVICE's own `description`, which are separate DB rows/fields) is rendered in exactly
  ONE place today: `dashboard/app/jobs/[name]/page.tsx` line 61, `<p className="sub">{job?.description}</p>`.
  The `.sub` CSS class (`dashboard/app/globals.css` line 85) is a plain `color: var(--muted); margin: 0
  0 20px` rule with no truncation/ellipsis/nowrap — it already wraps a long paragraph cleanly, so no CSS
  change is needed to hold longer text.
- `dashboard/app/runs/[id]/page.tsx` fetches only `api.run(id)` (line 12) and never fetches the job's
  own record — it has no description anywhere today.
- `GET /api/jobs/:name` (`src/api/server.ts` ~line 1456) already returns the job's `description` (via
  `listJobs()` + `jobView`), so the run page can get it with a plain extra fetch — no API change
  needed.
- `dashboard/scripts/_dashboard-harness.mjs`'s `job()` fixture (~line 77) already returns a non-trivial
  `description` string, and `/api/jobs/:name` in the synthetic router (~line 665) already returns
  `{ job: job() }` — so once the run page fetches+renders it, the existing hermetic visual-check should
  already show something on the `run` page (path `/runs/1`, already in `PAGES`) with no fixture change
  required. Confirm this yourself rather than assuming; only touch `_dashboard-harness.mjs` if you find
  it genuinely insufficient (e.g. you want more paragraph-realistic placeholder text).
