Context: today, log lines live in TWO separate tables with no cross-cutting view. `run_logs` (`src/db/schema.sql` ~line 36, `(id, run_id, ts, level, message)`) holds per-job-run logs, joined to `runs(job_name)` to know which JOB produced a line; `workflow_run_logs` (~line 166, `(id, workflow_run_id, ts, level, message)`) holds per-workflow-run FRAMEWORK logs, joined to `workflow_runs(workflow_name)` to know which WORKFLOW produced a line. The only existing readers are per-run: `getLogs(runId, afterId)` (`src/db/store.ts` ~line 187, backing `GET /api/runs/:id`) and `getWorkflowLogs(workflowRunId, afterId)` (~line 1222, backing `GET /api/workflow-runs/:id`) — both scoped to ONE run's id, ordered by `id ASC` for incremental tailing. There is currently no way to see logs across all runs/workflows at once, which is exactly the gap this task closes (the aggregate view a follow-up UI task will render as a new 'Logs' nav page).

Do:

1. **Schema (`src/db/schema.sql`)**: add two indexes to speed up ordering-by-timestamp across the whole table (today `run_logs` only has `idx_logs_run(run_id, id)` and `workflow_run_logs` only has `idx_workflow_run_logs(workflow_run_id, id)` — neither helps a global `ORDER BY ts DESC` scan): `CREATE INDEX IF NOT EXISTS idx_run_logs_ts ON run_logs(ts, id);` directly after the existing `idx_logs_run` index, and `CREATE INDEX IF NOT EXISTS idx_workflow_run_logs_ts ON workflow_run_logs(ts, id);` directly after `idx_workflow_run_logs`. This is SAFE in the bootstrap `schema.sql` (not a migration) because `ts`/`id` are original columns present on both tables since their `CREATE TABLE` — re-read root `CLAUDE.md`'s T098 rule if you're unsure why this matters: the trap is indexing a column a LATER migration adds, which this is not.

2. **Store layer (`src/db/store.ts`)**: add a new exported type and function near `getLogs`/`getWorkflowLogs`:
   ```ts
   export interface GlobalLogFilter {
     levels?: LogLevel[];      // omitted/empty = all levels
     job?: string;              // filter to run_logs from this job only
     workflow?: string;         // filter to workflow_run_logs from this workflow only (mutually exclusive with `job` — caller/route enforces this, not the store fn)
     q?: string;                 // free-text substring match against `message`, case-insensitive
     windowHours?: number;       // only applied when `before` is absent (first page); default 24 at the CALLER (route) level — this fn just takes whatever it's given
     before?: string;            // opaque cursor from a previous page's LAST returned row; when present, ignore windowHours entirely and return older rows
     limit?: number;             // default 200, caller (route) is responsible for clamping to a sane max
   }
   export interface GlobalLogLine {
     id: number; ts: string; level: LogLevel; message: string;
     source: 'job' | 'workflow';
     jobName: string | null; workflowName: string | null;
     runId: string | null; workflowRunId: string | null;
   }
   export function listGlobalLogs(filter: GlobalLogFilter): { logs: GlobalLogLine[]; nextCursor: string | null }
   ```
   Import `LogLevel` from `../core/types.js` (same source `addLog`/`addWorkflowLog` already use).

   **Recommended implementation shape** (two separate queries + an in-process merge — simpler and less error-prone than one dynamic UNION ALL SQL string with heterogeneous columns, and cheap at this project's personal-tool scale): build a `run_logs JOIN runs` query and a `workflow_run_logs JOIN workflow_runs` query, each applying the level/text/cursor filters that are relevant to it (the `job` filter only narrows the run_logs query — when `job` is set, skip the workflow_run_logs query entirely and vice versa for `workflow`; when NEITHER `job` nor `workflow` is set, run both and union the results), each `ORDER BY ts DESC, id DESC LIMIT <limit>`, then merge the (already individually sorted) result sets in JS by `ts` descending and slice to `limit`.

   **Cursor + tie-break — read this carefully, it is the easy way to get this subtly wrong.** `ts` is SQLite `datetime('now')` — SECOND granularity — so multiple log lines routinely share the exact same `ts` (root `CLAUDE.md`'s T112 note flags this exact pitfall for a different table: "`started_at` is second-granularity ... without a tiebreaker the ... status flicker"). A naive `before: ts` cursor will silently DROP or DUPLICATE same-second rows at every page boundary. Define the sort order as a genuine total order: `(ts DESC, source ASC, id DESC)` where `source` is compared as the literal strings `'job' < 'workflow'` — pick ANY consistent tiebreak, but make it total and consistent between the query and the cursor encoding. Encode the cursor as a plain string `` `${ts}|${source}|${id}` `` (no need for base64 — these three fields can't contain `|`). When `before` is supplied, filter each side's query to rows that sort strictly AFTER the cursor in this total order (i.e., `ts < cursorTs OR (ts = cursorTs AND (source, id) sorts after the cursor's (source, id) per the same ASC/DESC tiebreak)`). After merging + slicing to `limit`, set `nextCursor` = the encoded cursor of the LAST row in the returned page when `logs.length === limit` (there may be more), else `null` (we've reached the actual end of matching history — remember `before`-driven pages ignore the window, so `null` here genuinely means no more logs match the filters, not just 'end of window').
