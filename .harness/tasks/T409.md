## Overview

The `projects-sync` workflow's first stage, `github-sync`, catalogs every non-fork/
non-archived/non-private GitHub repo into `data/out/projects.json` — but its success
`markWorkItem` call records NO `detail` object at all, so the dashboard's Inputs &
Outputs panel can't show anything about what was cataloged for that repo, even though
the repo's description/language/topics/URL are all known at that point. Fix it per the
"every stage's success `detail` must describe what it produced" convention in the root
`CLAUDE.md` and `src/workflows/CLAUDE.md`.

## Do

- **`src/workflows/projects-sync/stages/github-sync.ts:175`** — inside `runGithubSync`'s
  per-entry loop, the success call is currently:
  ```ts
  markWorkItem(JOB_NAME, entry.repoId, 'success');
  ```
  Change it to record the catalog fields already computed on `entry` (the
  `CatalogEntry` produced by `repoToCatalogEntry`, defined at
  `src/workflows/projects-sync/stages/github-sync.ts:36-46`: `name`, `fullName`,
  `description`, `url`, `language`, `topics`, `pushedAt`, `defaultBranch`):
  ```ts
  markWorkItem(JOB_NAME, entry.repoId, 'success', {
    detail: {
      name: entry.fullName,
      description: entry.description,
      language: entry.language,
      topics: entry.topics,
      url: entry.url,
      pushedAt: entry.pushedAt,
    },
  });
  ```
  This is a catalog stage with one shared output file (`data/out/projects.json`), not
  a per-repo file — so per the root `CLAUDE.md`'s "no file, discovered/computed value"
  branch of the fix pattern, the right move is surfacing the repo's own catalog fields
  directly in `detail`, not inventing a `path`/`format` pointer at the shared file (that
  would be the same value repeated identically on every row and wouldn't tell you
  anything about THIS repo). Leave `defaultBranch` out if you judge `name`/
  `description`/`language`/`topics`/`url`/`pushedAt` already sufficiently describes the
  cataloged entry — use your judgement, but include at minimum `description` and
  `language` since those are the fields most likely to look "empty" in the panel
  otherwise.

- **`src/workflows/projects-sync/stages/project-summarize.ts`** — already compliant, no
  change needed. Its success call at line 241-244 already sets
  `detail: { name: entry.fullName, markdown: mdPath, lastCommitSha: marker }`, pointing
  at the real per-repo markdown artifact it writes (T110/T319 pattern). Confirmed by
  reading the file in full; do not touch it as part of this task.
