## Overview
Extend the Integrations page's existing per-service edit-limits row so the new timeout value (added by the dependent backend task) is visible and editable the same way rate/daily/monthly caps already are, and give each service a bit more visible identity/detail on the page — the idea's own text left the exact bar for 'more prominent' open, so this task takes the narrow, unambiguous interpretation (see the pending-questions relay for this cluster for the exact scoping question asked back).

## Do
Depends on `plex-service-c` (the backend `timeout_ms` field + `POST /api/services/:name/limits` support must exist first). In `dashboard/app/integrations/page.tsx`: add a 4th editable field alongside the existing Rate/min, Rate/day, Rate/month columns/inputs — a 'Timeout (ms)' column, using the same `toField`/`parseField`/`validField` blank-means-no-override idiom already used for the other three (extend those helpers or add a parallel one; blank = fall back to the code default), wired through `startEdit`/`save`/the `draft` state and `api.updateServiceLimits` (extend the `dashboard/app/lib/api.ts` `Service`/`updateServiceLimits` types to include `timeout_ms`, matching whatever field name the backend task lands on). Display the CURRENT effective timeout (not just editable input) somewhere sensible in each service row even when not editing — e.g. a small `mono` text under/beside the service name, similar in spirit to how `used`/`cap` are shown via `<Bar>`, since a bare number needs no bar visualization. For the 'more prominent' half: keep this modest per the narrow scope — e.g. always show the service's `description` (already shown) AND `rate_limit_source` inline in the row (today `rate_limit_source` is only visible after opening the Consumers popover) so a reader doesn't have to click through to see where a limit's numbers came from. Do not build a new dedicated per-service detail page/route in this task. Update `dashboard/scripts/_dashboard-harness.mjs`'s `/integrations` fixture data to include a `timeout_ms`/`rate_limit_source`-bearing 'plex' service entry (per the repo's living-artifact rule for the visual/mobile check harness), then build, run `node dashboard/scripts/mobile-check.mjs` and `node dashboard/scripts/visual-check.mjs`, and actually LOOK at the resulting screenshots to confirm the new timeout field and inline rate-limit-source render correctly and don't overflow at phone width.

## Done when
The Integrations page shows and allows editing a service's timeout the same way it already does for rate/daily/monthly caps (via the same edit-row affordance), each service row shows its `rate_limit_source` inline without requiring the Consumers popover, `dashboard/scripts/_dashboard-harness.mjs` fixtures are updated to exercise the new field, `npm --prefix dashboard run build` succeeds, `node dashboard/scripts/mobile-check.mjs` passes with no overflow at phone width, and `node dashboard/scripts/visual-check.mjs`'s screenshots (actually reviewed, not just run) show the new timeout field and inline rate-limit-source rendering correctly on the Integrations page.
