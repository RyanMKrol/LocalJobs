## Overview

`workouts-sync` (`src/jobs/workouts-sync/`) has 2 stages — `hevy-sync → workouts-progress` — and
currently has NO validation gate between them. Add a real `ArtifactContract` at the boundary,
mirroring the existing `places`/`perfumes` pattern.

## Do

Read root `CLAUDE.md`'s "Validation gates between workflow stages" section first, then
`src/core/types.ts` (`ArtifactContract`/`ArtifactShape`/`GateResult`/`ExpectationResult`) and
`src/core/dag.ts`'s `deriveGates`. Use `src/jobs/places/contracts.ts` as your primary worked example
for the factory-function + `fromChecks` idiom.

**Note: unlike every other workflow with contracts, `workouts-sync` has NO existing `config.ts`** —
its file paths are defined inline in the stage files: `defaultHistoryPath` is exported from
`src/jobs/workouts-sync/stages/hevy-sync.ts:79`
(`resolve(here, '../data/out/workouts-history.json')`), and `defaultOutDir` is exported from
`src/jobs/workouts-sync/stages/workouts-progress.ts:14`. You do not need to introduce a `config.ts`
for this task — it's fine for `contracts.ts` to `import { defaultHistoryPath } from './stages/hevy-sync.js'`
directly, matching how `workouts-progress.ts` itself already does
(`src/jobs/workouts-sync/stages/workouts-progress.ts:8`). If you judge a small root-level `config.ts`
would be cleaner and want to add one as part of this task, that's also fine — just keep it minimal and
don't scope-creep into unrelated refactors.

**Create `src/jobs/workouts-sync/contracts.ts`** with one factory function:

**`workoutsHistoryContract(file = defaultHistoryPath)`** — key `'workouts-history'`. Validates
`data/out/workouts-history.json` (written by `hevy-sync.ts` at
`src/jobs/workouts-sync/stages/hevy-sync.ts:93`): a plain JSON **array** of `HevyWorkout` records
(`src/jobs/workouts-sync/stages/hevy-sync.ts:38-47`):
`{ id: string, title: string, description: string, start_time: string, end_time: string, updated_at: string, created_at: string, exercises: HevyExercise[] }`.
Expectations: file exists + parses as JSON; top-level value is an array; IF non-empty, every entry has
a non-empty string `id` and an `exercises` array. **Do NOT hard-require non-empty** — a fresh install
before the owner has logged any Hevy workouts (or a very first run) can legitimately produce an empty
history array; treat that as `ok: true` with a `sample` noting "0 workouts", same leniency as
`stocks-sync`'s portfolio contract, not places' hard non-empty rule.

**Wire the contract into the 2 stage jobs:**
- `src/jobs/workouts-sync/stages/hevy-sync.job.ts` — add `produces: [workoutsHistoryContract()]`.
- `src/jobs/workouts-sync/stages/workouts-progress.job.ts` — add `consumes: [workoutsHistoryContract()]`.

Import from `../contracts.js` (relative from `stages/`, same pattern as
`src/jobs/perfumes/stages/fetch.job.ts` importing `../contracts.js`).

**Add a test file** `src/jobs/workouts-sync/contracts.test.ts` mirroring `src/core/contracts.test.ts`'s
structure (synthetic fixtures in a temp dir, NO live Hevy API call): cover the pass cases (well-formed
non-empty array, AND empty array — both must be `ok: true`) and fail cases (missing file, invalid
JSON, non-array top-level, an entry missing `id`). Also add a test that builds the real
`workouts-sync` job refs (or a synthetic 2-node chain) via `buildDag` and asserts `deriveGates`
produces exactly 1 gate for the `hevy-sync→workouts-progress` edge.
