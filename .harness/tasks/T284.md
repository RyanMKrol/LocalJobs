# T284 spec

## Do

Create a new slash command `.claude/commands/pre-loop-checkin.md` — a **read-only, advisory**
runbook that vets the harness backlog BEFORE the owner kicks off an unattended `.harness/loop.sh` /
`.harness/supervise.sh` run. It is the pre-run mirror of the existing `.claude/commands/loop-recover.md`
(read that file first for house style: YAML frontmatter with `description` + optional
`argument-hint`, a guardrails section, numbered steps with concrete `bash`/`jq` snippets, ending in a
consolidated report). Unlike `/loop-recover`, this command **never mutates anything** — it only reads
state and reports.

Implement four checks, each producing a short report section:

1. **Needs-human tasks blocking buildable work.** From `.harness/TASKS.json`, find every
   `gate:"needs-human"` task that appears in the `dependsOn` array of some `pending`, `gate:null`
   task. For each such blocker, report: how many/which buildable tasks it blocks, and whether it's
   already marked `done:true` in `.harness/human-done.json` (i.e. about to auto-reconcile to
   `status:"done"` on the loop's next pre-flight per the `reconcile_overlays` step in `loop.sh` — so
   it will unblock on its own) versus still fully open (owner action needed now to avoid the loop
   idling on that branch of work).
2. **Session hygiene / uncommitted or unpushed work.** Report `git status --porcelain`, the
   ahead/behind count vs `origin/main` (`git rev-list --count origin/main..HEAD` /
   `HEAD..origin/main`), and whether a `loop.sh`/`supervise.sh` process or the repo lock
   (`<git-common-dir>/<basename(repo-root)>-loop.lock`, same path `loop.sh`'s `acquire_lock` and
   `src/core/repo-lock.ts` use) is currently held. If a loop IS already running, say so plainly and
   recommend NOT starting a second one (T105's one-run guard covers workflows, not this).
3. **Dependency short-circuits.** Scan `dependsOn` edges for ones already effectively resolved: flag
   (informational only) any `dependsOn` id whose target task's `status` is already `"done"` — harmless
   but worth surfacing since it means that edge no longer constrains scheduling. This is a best-effort,
   advisory heuristic — do not attempt automatic graph rewrites.
4. **Per-task definition quality.** For every `pending`, non-`needs-human` task in TASKS.json, check:
   `facets` present with a `layer` + `workType` drawn from `.harness/facets.json`'s vocabulary (mirror
   `loop.sh`'s existing `_missing_facets` warning, ~line 831); the task's `spec` file
   (`.harness/tasks/TNNN.md`) exists on disk; that file has both a non-empty `## Do` and a non-empty
   `## Done when` section; `scope` is a non-empty array. Report every task that fails any of these.

Finish with ONE consolidated go/no-go report (mirroring `/loop-recover`'s final report style) so the
owner can read it in one glance before starting a run.

## Done when

- `.claude/commands/pre-loop-checkin.md` exists, with YAML frontmatter (`description` + an
  `argument-hint` if it accepts an optional focus argument) matching the style of
  `.claude/commands/loop-recover.md`.
- The command's body contains an explicit **guardrails section stating it is READ-ONLY /
  ADVISORY-ONLY**: it MUST NOT edit `.harness/TASKS.json`, MUST NOT write/commit/push any file, MUST
  NOT touch `.harness/reviews.json` / `.harness/human-done.json` / `.harness/manual-fail.json`, and
  MUST NOT run any destructive or mutating git command. This is the key contrast with `/loop-recover`
  and must be unambiguous.
- All four checks above are implemented as concrete, runnable steps (bash/jq snippets the assistant
  executes when the command runs), each producing its own report section:
  1. needs-human blockers of buildable work (with the human-done-overlay-about-to-reconcile
     distinction),
  2. session hygiene (working tree status, ahead/behind, loop process/lock state),
  3. dependency short-circuits (already-done `dependsOn` targets, informational),
  4. per-task definition quality (missing facets / missing or malformed spec file / empty scope) for
     every pending non-needs-human task.
- The command ends with one consolidated go/no-go report section a human can read before running the
  loop.
- The file is a plain Markdown prompt (no code to compile/lint beyond the repo's existing conventions)
  — verify by actually invoking `/pre-loop-checkin` once against this repo's current real state and
  confirming the four report sections render sensibly and no file under `.harness/` or `.git/` was
  modified (`git status --porcelain` unchanged before/after the dry invocation).
