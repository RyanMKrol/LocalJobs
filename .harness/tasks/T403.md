## Overview

`stocks-sync`'s `stocks-snapshot` stage currently records ONE `work_items` ledger row PER POSITION
(keyed `account:ticker`) every run, but all it actually produces is a single combined snapshot file
(`portfolio.json`/`portfolio.md`) — every one of those per-position rows points at the exact same
`detail.markdown` path. This is the same redundancy this repo already solved for `stock-digest-build`
(keyed by ISO week) and `plex-space-saver-scan` (keyed by ISO week): a workflow that produces ONE
combined artifact per period should record ONE ledger row per period, not one per underlying item.
Since `stocks-sync` runs daily, collapse `stocks-snapshot`'s ledger to ONE row per calendar day
(`YYYY-MM-DD`), overwritten (not duplicated) on a same-day re-run.

```ts
/** "2026-07-04" — the UTC calendar-day key, used as the single collapsed ledger key. */
export function dayKey(date: Date): string {

}
```

const key = dayKey(now);

markWorkItem(JOB_NAME, key, 'success', {
  detail: {

### 2. Drop `stocksSnapshotInputKeys()` and make the workflow non-limitable

Delete the `stocksSnapshotInputKeys` function (currently lines ~165-175 of `stocks-snapshot.ts`) and
remove `inputKeys: stocksSnapshotInputKeys` from `src/jobs/stocks-sync/stages/stocks-snapshot.job.ts`
(and its import). A daily portfolio snapshot is one atomic Trading212 fetch, not something that
meaningfully fans out per ticker — per-ticker `limit` never made sense here, and this repo's
established idiom for a single-combined-artifact stage is to declare NO `inputKeys()` at all (see
`src/jobs/plex-space-saver/stages/scan.job.ts` and `src/jobs/stock-digest/stages/
stock-portfolio-snapshot.job.ts` — neither declares `inputKeys`). This makes `stocks-sync` correctly
report `limitable: false` from `GET /api/workflows/stocks-sync` and the dashboard's "Limit" number box
no longer appears next to stocks-sync's "▶ Run now" — consistent with its sibling single-snapshot
workflows, none of which offer a limit control.

`positionKey` itself (in `src/services/trading212.service.ts`) stays — it's still used by `stocks-watch`
(step 3) and by `stocks-snapshot.ts`'s own per-position log lines/markdown builder.
