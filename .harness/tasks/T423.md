## Overview

`tv-rec-merge`'s `verifyInto()` helper (`src/jobs/tv-recs/stages/tv-rec-merge.ts`) TMDB-verifies each
pooled raw suggestion in a per-item loop. A genuine per-item failure (the `search()` call throwing
anything other than `QuotaExceededError` — a real network/API error, not a business-logic drop) is
caught, logged as a `warn`, and the loop simply `continue`s to the next suggestion — the failure is
never tallied and never surfaces to the run's outcome. `runTvRecMerge` always finishes normally and
writes `recommendations.json`, so a run that silently lost candidates to real search errors still
reports success. Per the T394 convention (item-loop jobs must hard-fail their own run when they had
genuine per-item failures this run, not swallow them), this stage needs to track that failure class
separately from the existing soft business-logic drops (hallucinated / owned / already-recommended /
below quality bar / quota-hit-and-stopped) and throw at the end if any occurred.

## Do

- In `src/jobs/tv-recs/stages/tv-rec-merge.ts`, extend `VerifyCounters` (or add a sibling counter) with
  a new tally, e.g. `searchFailed: number`, distinct from the existing soft-drop counters and from
  `quotaHit` (which is a deliberate graceful stop, not a failure).
- In `verifyInto`'s `catch (err)` block (the branch that currently only fires for non-
  `QuotaExceededError` errors — logs a `warn` and `continue`s), increment `counters.searchFailed`
  before continuing. Keep the existing per-item log line — this is additive tallying, not a
  behavior change to the drop/skip flow itself.
- At the end of `runTvRecMerge` (after the existing summary logging, once `recommendations.json`
  has been written — writing partial-but-valid output before failing the run is fine and matches
  how other stages in this repo behave), check the accumulated tally across the whole run
  (base verification pass + every top-up round uses the same `counters` object, so this naturally
  covers the whole run). If `counters.searchFailed > 0`, `throw` a summarizing `Error` — e.g.
  `TMDB search failed for N suggestion(s) this run — see warn logs above for titles.` — instead of
  letting `run()` return normally.
- Do NOT throw for the existing soft drops (hallucinated / owned / already-recommended / below
  quality bar) or for a `quotaHit` graceful stop with zero real search failures — those remain
  normal, successful outcomes exactly as today.
- Leave `tv-recs-notify.ts` alone — it was checked as part of this same audit and has no comparable
  per-item silent-swallow gap (its `markWorkItem` loop has no try/catch, and the one existing
  try/catch — `appendHistory`'s corrupt-JSON recovery — is not a per-item processing failure).

## Done when

- `verifyInto` increments a dedicated failure counter (not reused from an existing soft-drop
  counter) whenever `search()` throws something other than `QuotaExceededError`.
- `runTvRecMerge` throws a summarizing `Error` when that counter is `> 0` at the end of the run
  (after both the base verification pass and any top-up rounds), causing the `tv-rec-merge` job run
  to record `failed` and block `tv-recs-notify` from running that cycle.
- A run with zero real search failures (only soft drops, or a graceful `quotaHit` stop) still
  completes and returns normally, unchanged from current behavior.
- A new unit test in `src/jobs/tv-recs/stages/tv-rec-merge.test.ts` (or the existing test file for
  this stage, whichever already exists) covers: (a) a `searchTv` mock that throws a plain `Error` for
  one suggestion causes `runTvRecMerge` to reject/throw, and (b) a run with only soft drops (no
  throwing `searchTv` calls) still resolves normally.
- `npx tsc --noEmit` and `npm test` both green.
