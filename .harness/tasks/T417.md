## Overview

The workflow-run detail page's "Member runs" table shows only the LATEST run per stage, hiding
every earlier attempt/cycle within that same workflow run — including earlier FAILED attempts. Show
every attempt per stage instead, so a stage that failed once and then succeeded on retry (or a later
`repeatUntilStable` cycle) is visibly traceable, not silently collapsed into a single "success" row.

## Context (already confirmed — ground truth, not a guess)

`dashboard/app/workflow-runs/[id]/page.tsx`:
- Line 364: `const members = data?.jobs ?? [];` — `data.jobs` comes from `GET /api/workflow-runs/:id`,
  which is backed by `listRunsForWorkflowRun(id)` (`src/db/store.ts:1440-1443`) — this ALREADY
  returns **every** run for the workflow run (every attempt, every cycle), ordered by
  `(started_at, rowid)`. No new API endpoint or DB query is needed — the full data is already
  fetched client-side.
- Lines 13-23 (`latestByStage`): collapses this full list down to ONE row per `job_name` — "the
  final value per key is the genuinely-latest run" — for the "Member runs" table (lines 420-434).
  Any earlier attempt for that stage (e.g. a `repeatUntilStable` cycle 1 failure, or a
  `maxRetries`-driven retry) is discarded from this table entirely.
- The only place an earlier failure is visible today is the "Framework logs" section further down
  the page (a flat, chronological workflow-level log stream, line ~440) — which DOES include lines
  like `✗ perfumes-build → failed` from an earlier cycle, but with no visual link back to the
  (now-collapsed, showing "success") stage row above it. This is the exact source of confusion
  reported during T394's discussion: skimming the log feed surfaces a failure that the stage table
  no longer reflects.

This is a UI-only fix — the data is already there, `latestByStage`'s collapsing is purely a display
choice.
