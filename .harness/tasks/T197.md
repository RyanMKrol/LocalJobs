## Do

Harden ntfy notification sending against transient network blips. A real failure was observed: a
workflow-success push logged `notification FAILED (fetch failed)` — the Node `fetch` threw a
network-level error (DNS/connection/offline), which `_sendNtfyToUrl` in `src/core/notifier.ts`
catches at its `try/catch` (~line 198) and logs as just `e.message` ("fetch failed"), with no retry.
Two improvements (both in `src/core/notifier.ts`):

1. **Better diagnostics.** When the fetch throws, log the underlying cause, not just `e.message`.
   Node network errors carry the real reason on `e.cause` (e.g. `ENOTFOUND`, `ECONNREFUSED`,
   `ECONNRESET`). Include `e.cause` (its `code`/message) in the returned `error` string so the run
   log says WHY it failed, not the opaque "fetch failed".
2. **Retry network throws.** Today only HTTP 429 backs off; a thrown network error gets a single
   attempt then is dropped. Add a small bounded retry (1–2 quick re-tries with a short delay) around
   the `_fetch` call for THROWN errors only. Do NOT retry on a real HTTP status (4xx/5xx) — those
   aren't transient blips — and keep the existing 429 backoff path intact. The function must still
   **never throw** (a broken notifier must not affect job execution) and must still respect the
   `ntfyBackoffUntil` cooldown.

Keep the injectable `_fetch` seam so tests stay hermetic. Update the `src/core/notifier.ts` comment/
docs if they describe the single-attempt behaviour.
