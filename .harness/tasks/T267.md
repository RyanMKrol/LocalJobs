## Do

Build a tiny scheduled **"warm the Claude usage window"** workflow whose only job is to call Claude
with the smallest possible prompt every ~30 minutes, to START the account's 5-hour usage-limit window
early (so by the time the owner sits down to do real work, the window is closer to resetting rather
than a fresh 5h). Create a new tracked workflow folder `src/jobs/claude-warmer/` (no secrets; `data/`
gitignored). One stage, one minimal Claude call.

Decisions (settled):
- **Route through the existing `claude-cli` service** via the shared helper `src/services/claude.ts`
  (`runClaude`). NOTE on caps: the `claude-cli` service sets **no LOCAL cap** — not because usage is
  unlimited, but because the cap is enforced **UPSTREAM by the Claude plan**: hitting the plan's usage
  limit makes the CLI fail out, which surfaces gracefully (the harness's rate-limit detection backs
  off; jobs soft-fail). So there's no local service quota to raise or compete for; the warmer's
  ~48 calls/day just consume the real Claude account usage window — which IS the entire point. Handle
  the fail-out defensively: if `runClaude`/`callService` throws (a transient CLI error, or the
  upstream rate-limit/`QuotaExceededError`), catch it, log, and exit cleanly — never crash the workflow.
- **Minimal prompt / minimal tokens** — the smallest input that still elicits a response (e.g. "hi").
  Do nothing meaningful with the reply; this exists only to tick the usage window, not to do work.
  Keep `maxRetries` low and `timeoutMs` modest.
- **Schedule ~every 30 min** (`*/30 * * * *`) and **enabled by default**.
