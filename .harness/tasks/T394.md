## Overview

A real run reported `'succeeded'` while its own logs show it explicitly failed to summarize its only
repo, and produced 0 output. Run-level status and per-item work-item status are two disconnected
concepts in this codebase today, and the owner wants to sit down with an agent and design — not have
an agent unilaterally implement — how "soft-fail" (per-item resilience, the current default) and
"hard-fail" (a stage/run that should be reported as failed when its item-level work didn't
land) should be represented in job/run status. This task is the walkthrough + decision; a follow-up
implementation task should be filed once the shape is agreed (deliberately not pre-built here — see
"Why no implementation task is pre-authored" below).

## The incident (grounding, already root-caused)

Run: `http://100.112.137.1:4788/runs/5bed00ad-938b-4df2-a8d1-06657df2de06`, job `project-summarize`,
workflow `projects-sync`. Its logs show:

```
error: failed to summarize RyanMKrol/LocalJobs: Error: summary output missing template shape: leading "---" frontmatter marker
project-summarize complete — 0 summarized, 0 skipped
```

...yet the run's own recorded status is `'succeeded'`.

**Root cause** (`src/jobs/projects-sync/stages/project-summarize.ts`, the per-repo loop around lines
208-252): each repo's clone/summarize/write is wrapped in its own `try/catch` (lines 222-249). On
failure it logs the error, calls `markWorkItem(JOB_NAME, entry.repoId, 'failed', ...)` (line 248 —
correctly recording the PER-ITEM failure in the `work_items` ledger), then the loop just `continue`s.
The error is never rethrown, so `run()` returns normally, and `src/runJob.ts` reports `status:
'success'` to the parent executor, which records the RUN itself `'success'` — regardless of how many
(or what fraction) of the items it touched actually failed.

**This is a deliberate, common pattern in this codebase** — most item-loop jobs (places, perfumes,
workouts-sync, stocks-sync, etc.) catch per-item and continue, precisely so one bad item doesn't kill
an entire run's progress on every other item. The question is not "stop doing this" — it's "how do we
stop a run whose *every* (or *most*) item failed from reading identically to a run that succeeded."

## Why this isn't already caught

Grep/read `hasJobAdvancedAnyItem` (`src/db/store.ts`, used by `src/core/workflow-executor.ts` — the
T258 fix): it reclassifies a run from `'success'` to a `'skipped'`/noop annotation **only when the job
called `markWorkItem` ZERO times this run** (checked via the `work_item_runs` linkage table). That
does NOT apply here: `markWorkItem(..., 'failed', ...)` WAS called once, so a ledger write happened,
`hasJobAdvancedAnyItem` returns `true`, and no reclassification triggers. So today the codebase
distinguishes "did nothing" (caught) from "attempted and every attempt failed" (NOT caught) — and only
the first case is handled. This is a real gap, not a duplicate of T258.

There IS a `'partial'` workflow-run status already (`src/core/workflow-executor.ts`, near the bottom of
`runWorkflowInner`): if a workflow's member STAGES have mixed outcomes (`statuses.some(s => s ===
'success') && !statuses.every(...)`), the whole WORKFLOW RUN is recorded `'partial'`. But that operates
one level up, across DIFFERENT STAGES of the SAME workflow run — it has nothing to say about a SINGLE
stage/job run whose own `run()` returned success while 100% of the *items it processed* failed. A
1-stage workflow (or a stage whose sibling stages all genuinely succeeded) can never surface `'partial'`
via this mechanism, exactly the incident's shape (projects-sync's OTHER stage, `github-sync`, presumably
did succeed that run).

**Empirical check on notifications (done as part of shaping this task):** `notifyWorkflow` (the ONLY
push notification since T189 removed per-stage notifications) builds its body from
`run?.progress_msg` + duration ONLY — it does not read or mention any item-level success/failure
tally. There IS a `notifyStage` function in `src/core/notifier.ts` that DOES compute `workItemCounts(jobName)`
and include an "N ok, M failed" line in its body — but it is dead code, no longer called anywhere since
T189. So even today's push notification for this exact incident would have said something like "✅
projects-sync workflow — success" with no hint that project-summarize processed 0 items successfully.
This is squarely in scope for the architecture discussion: whatever status model is chosen, does the
notification body also need item-level failure visibility (independent of whether the run's `status`
itself changes)?

**Also worth confirming empirically during the walkthrough:** `GET /api/runs/:id` (`src/api/server.ts`)
returns the raw `run` row + logs only — no query currently joins in a per-run item-outcome tally (the
`work_items` ledger's `workItemCounts(jobName)` is CUMULATIVE across all runs, not scoped to one run;
the workflow-run Input→Output panel via `work_item_runs` linkage is the closest existing per-run view,
and even that surfaces items, not an aggregate pass/fail rate). So today a human reading a run's detail
page has no per-run item-tally to notice the mismatch either — they'd have to read the logs, as the
owner did here.

## Real options to ground the discussion (lay these out; do not pick one unilaterally)

1. **Item-failure-rate-aware run status.** Compute, at the point a job's `run()` returns successfully,
   the fraction of items this run *touched* (via `work_item_runs` linkage for this `workflowRunId` +
   `jobName`) that ended up `'failed'` in `work_items` vs `'success'`. Introduce a new run status (e.g.
   `'degraded'` or reuse `'partial'` at the RUN level, not just the workflow-run level) when that
   fraction crosses some threshold. Design questions: what threshold (any failure? >50%? 100%,
   i.e. only when literally everything failed, mirroring this incident exactly)? Does the run status
   enum (`RunStatus` in `src/core/types.ts`) need a new value, or does an existing status get
   reinterpreted? Does `'partial'`/`'degraded'` cascade up to the WORKFLOW run status the same way a
   `'failed'` stage does today (does a degraded member currently block downstream `dependsOn` stages —
   check `executeDag`'s semantics for `'success'` vs other returns before deciding)?
   **Implementation nuance to flag:** `work_item_runs` records ONLY linkage (workflow_run_id, job_name,
   item_key, root_key, at) — it does NOT store the outcome at the time. Computing "how many of the
   items this run touched are currently failed" means joining to `work_items.status` AS IT STANDS NOW,
   which is correct today only because of the "one active run per workflow" invariant (T105) — no other
   run mutates the same rows concurrently. Confirm this join approach (or an alternative — e.g. have
   `markWorkItem` ALSO write the per-attempt outcome onto `work_item_runs` directly, avoiding the
   join-through-current-state assumption) is sound before building it.

2. **Declarative soft-fail vs hard-fail per job/stage.** Add an opt-in `JobDefinition` property (e.g.
   `failOnAnyItemFailure: boolean`, default `false` = today's soft-fail behavior unchanged) that a stage
   can set when its purpose is genuinely all-or-nothing (arguably project-summarize IS this kind of
   stage when its catalog is small/empty — is "0 of 1 succeeded" fundamentally different from "0 of 50
   succeeded"?). This is more manual/curated (a human decides per-stage) vs. option 1's automatic
   detection — discuss whether one supersedes the other, or whether they're complementary (e.g. a
   declared hard-fail stage always uses a 0% failure threshold; other stages get an automatic
   `'degraded'` annotation without failing the run outright).

3. **Visibility-only fix (no status semantics change).** Leave `RunStatus`/workflow status exactly as-is,
   but surface a per-run item-outcome tally on `GET /api/runs/:id` (and the dashboard run detail page) —
   e.g. `{ itemsSucceeded, itemsFailed, itemsSkipped }` computed the same way option 1's threshold check
   would, and/or resurrect something like the dead `notifyStage` tally into the aggregate
   `notifyWorkflow` body. This directly addresses the owner's stated pain ("hard to understand the
   status of the site") without touching the status enum or cascade semantics at all — worth weighing as
   the lowest-risk option, possibly as a first increment even if 1 or 2 also get built later.

4. **Hybrid.** E.g. do (3) unconditionally (visibility), plus (1) or (2) gated behind a real threshold
   (only escalate run status on a TOTAL wipeout — 0 successes when items were attempted — which is
   exactly this incident's shape and arguably a much easier semantic to agree on than a graduated
   percentage threshold).

## Why no implementation task is pre-authored

The outcome of this discussion could be "just add visibility, no status change" (small), "add a new
run status + cascade rules" (touches `src/core/types.ts`, `src/core/workflow-executor.ts`,
`src/core/dag.ts`'s `executeDag`, `src/db/store.ts`, the dashboard's status badges/colours, and
possibly `notifier.ts`) or "add a per-job opt-in flag" (touches `JobDefinition` + every job that wants
it) — genuinely different scopes, files, and risk levels. Pre-authoring a speculative follow-up task
now would either be so vague it's useless or would bake in an unagreed design. File the follow-up
implementation task(s) — likely `layer: framework` and/or `risk: [touches-executor]` given the
`workflow-executor.ts`/`dag.ts` surface area involved in most of the options above — once this
walkthrough lands on a shape.

## Suggested reading order for the walkthrough

1. `src/core/types.ts` — `RunStatus`, `WorkflowRunStatus`, `JobContext.log`/`LogLevel`.
2. `src/jobs/projects-sync/stages/project-summarize.ts:208-252` — the incident's exact root cause.
3. `src/runJob.ts` — how a job's `run()` return/throw becomes the child's `result` event.
4. `src/core/executor.ts` — how the child's result event becomes a `runs` row status.
5. `src/db/store.ts` — `hasJobAdvancedAnyItem`, `workflowRunAdvancedAnyItem`, `markWorkItem`,
   `work_item_runs` schema (`src/db/schema.sql`).
6. `src/core/workflow-executor.ts` — the T258 noop-detection block (~line 362) and the existing
   `'partial'` workflow-status computation near the bottom of `runWorkflowInner` (~line 434-442).
7. `src/core/notifier.ts` — `notifyWorkflow` (live, called) vs `notifyStage` (dead code since T189) —
   confirm neither currently surfaces item-level failure counts.
8. `src/api/server.ts` — `GET /api/runs/:id` — confirm no per-run item tally is returned today.

## Done when

The owner and an agent have walked through the incident, the existing mechanisms (T258's
`hasJobAdvancedAnyItem`, the workflow-level `'partial'` status, the dead `notifyStage` tally), and the
options above, and reached a decision on ONE of: (a) do nothing beyond documenting the gap (a
conscious "acceptable as-is" call); (b) build visibility-only (option 3); (c) build a status-semantics
change (option 1, 2, or a hybrid) — with enough concreteness (which files, which threshold/flag
semantics, whether it cascades to workflow status) that a follow-up implementation task can be authored
directly from the decision. If a follow-up is warranted, file it as a new backlog task (via the normal
add-to-backlog flow) referencing this task's id. Either outcome is a valid completion — the bar is a
real decision was made and recorded, not that code changed.
