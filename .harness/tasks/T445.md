## Overview

A one-time, human sign-off checkpoint after the `plex-language-fix` workflow's apply stage has been
built: before leaving it running unattended forever on its weekly schedule, have a human actually watch
its first real live run and confirm it behaved correctly against the real Plex server — since it
mutates the default audio/subtitle selection for every user of the server, not just the owner, and
there is no per-run approval step by design (the owner explicitly declined manual sign-off on an
ongoing basis).

## Do

This is a **needs-human** task — not something the autonomous build loop builds. There is no code to
write. The person picking this up should:

1. Confirm `src/workflows/plex-language-fix`'s three stages (`plex-language-scan`,
   `plex-language-apply`, `plex-language-no-track-flag`) are built, `npm test`/`npx tsc --noEmit` are
   green, and the daemon has been restarted with the latest code.
2. Trigger ONE real run of the `plex-language-fix` workflow manually (dashboard "Run now", or `POST
   /api/workflows/plex-language-fix/run`) against the real Plex server.
3. Watch it end-to-end: confirm the scan stage's tallies look sane vs. the original exploration's
   numbers (2,184 titles; roughly 1,780 changes once former "ambiguous" ties are folded in, ~500
   already-correct, ~1,886 no-match — some drift from library changes since is expected, but a wildly
   different number is a red flag worth investigating before trusting it).
4. Confirm the apply stage actually triggered a real Butler backup (check Plex's own Activity/Butler
   log, or that a fresh dated backup file exists) BEFORE any real Plex mutation happened that run.
5. Spot-check a handful of applied files directly in a Plex client (at least one movie, one TV episode,
   and — if any exist that run — one that was previously a channel-count "tie": confirm the resolved
   choice actually plays with the correct original-language audio and, where applicable, English
   subtitles) — not just that the API says `selected: true`, but that playback genuinely reflects it.
6. Confirm the no-track-flag stage's digest/report correctly lists real no-match files (spot-check e.g.
   a known dub-only show like Pokémon appears) and that `ignoreSurfacedItem`/`unignoreSurfacedItem`
   actually suppress/restore an item end-to-end if exercised.
7. Decide whether the workflow is safe to leave running on its weekly schedule unattended, or whether
   something needs fixing first (file a follow-up task if so, describing exactly what broke).
8. Record the outcome — what was checked, what was found, go/no-go — in this task's worklog entry
   (`.harness/worklog/T<id>.md`, standard convention for any task).

## Done when

- A real live run of the full `plex-language-fix` workflow has been manually triggered and observed
  end-to-end by a human (not just CI/tests).
- Every checkpoint in steps 3-6 above has been confirmed (or a follow-up task filed for whatever
  didn't check out), and the go/no-go decision is recorded in the worklog.
- If the outcome is "go", the workflow is confirmed to be already running on its normal weekly schedule
  (no code change needed to make this true — just confirm the schedule is registered/enabled, e.g. via
  the dashboard's workflow detail page).
