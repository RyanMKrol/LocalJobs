## Overview

`stock-digest`'s Diversification section is empty even with a valid `FINNHUB_API_KEY` because
`stock-sector-lookup` sends Trading212's raw ticker (`AMD_US_EQ`) straight to Finnhub, which only
recognizes the bare symbol (`AMD`) — confirmed live: `GET .../profile2?symbol=AMD_US_EQ` returns `{}`
while `symbol=AMD` returns a full profile. Finnhub doesn't error on an unrecognized symbol, so the
code silently records every ticker as "resolved to no industry" — and, worse, records that null result
as `'success'` in the `work_items` ledger, so those tickers are treated as permanently done and never
retried even after the code is fixed. This task fixes both: translate the ticker before calling
Finnhub, and record an unresolved lookup as `'failed'` (retryable) instead of `'success'`, plus ships a
one-time script to clear the currently-poisoned production ledger rows.

## Do

1. **Translate the Trading212 ticker to a Finnhub-compatible symbol before the API call.** In
   `src/jobs/stock-digest/stages/stock-sector-lookup.ts`, add an exported pure function:
   ```ts
   export function toFinnhubSymbol(t212Ticker: string): string {
     return t212Ticker.replace(/_EQ$/, '').replace(/_[A-Z]{2}$/, '');
   }
   ```
   This strips a trailing `_EQ` (present on every Trading212 ticker) and, if what remains ends in a
   2-letter uppercase market/country code (e.g. `_US`), strips that too. **Do NOT use "everything from
   the first underscore" — verified unsafe:** the live portfolio holds `BRK_B_US_EQ` (Berkshire class B),
   where the underscore-separated `B` is part of the symbol itself, not a market code; stripping from the
   first underscore would wrongly produce `BRK` instead of `BRK_B`. The two-letter-suffix approach
   correctly handles all three shapes seen in this codebase's actual data/tests: `AMD_US_EQ` → `AMD`,
   `VUSA_EQ` → `VUSA` (LSE-primary instruments carry no market code, confirmed via the existing
   `stocks-snapshot.test.ts` fixture), and `BRK_B_US_EQ` → `BRK_B` (still not Finnhub's exact expected
   form for a class-share ticker — Finnhub likely wants `BRK.B` or `BRK-B` — but this is a known,
   documented residual edge case for class-share tickers, out of scope here; every other held ticker is
   fully fixed by this transform and this is a strict improvement over the current 100%-broken state).
   Use `toFinnhubSymbol(ticker)` to build the `symbol=` query param inside `fetchFinnhubProfile` — do NOT
   change what key `sectors[ticker]` / `markWorkItem` use (still the raw Trading212 ticker, so the map
   stays joinable with `portfolio.json` in `stock-digest-build.ts`).

2. **Record an unresolved lookup as `'failed'`, not `'success'`, so it retries automatically.** In
   `runStockSectorLookup`'s per-ticker loop, the `markWorkItem(JOB_NAME, ticker, 'success', ...)` call
   currently runs unconditionally before checking whether `industry` is null. Move it inside the
   `if (industry) { ... } else { ... }` branches:
   - `industry` truthy → unchanged: `markWorkItem(JOB_NAME, ticker, 'success', { detail: { name: ticker, industry } })`.
   - `industry` null → `markWorkItem(JOB_NAME, ticker, 'failed', { detail: { name: ticker, error: 'Finnhub returned no finnhubIndustry field (symbol not recognized, or genuinely unclassified)' } })`.
   Keep `sectors[ticker] = industry` (still written either way — `SectorMap`'s documented type is
   `string | null` and `stock-digest-build.ts`'s `sectorBreakdown` already treats a null/missing entry
   as "no sector", so this is unchanged). With `MAX_ATTEMPTS = 3` already defined in this file, a
   genuinely-unresolvable ticker naturally retries up to 3 times across runs, then becomes `isWorkItemDone
   = true` (failed + attempts >= max) — i.e. it surfaces on the dashboard's Stuck tile for manual
   unstick/ignore, exactly like every other stuck item in this codebase (see root `CLAUDE.md`'s "Stuck vs
   ignored" convention). No changes needed to `isWorkItemDone`/`unstickWorkItem` themselves — this reuses
   the existing mechanism as-is.

3. **Add a one-time production ledger cleanup script**, mirroring the existing
   `scripts/backfill-service-usage.ts` pattern (a committed, idempotent, manually-run-once tool — see its
   header comment for the precedent). Add `src/db/store.ts`:
   ```ts
   /**
    * One-time recovery helper: delete `success` rows for a job whose `detail` JSON has
    * `[field] == null` — for clearing rows a bug incorrectly recorded as succeeded (e.g.
    * stock-sector-lookup recording a null Finnhub result as 'success' before this fix).
    * Returns the item_keys removed so the caller can report exactly what was reset.
    */
   export function deleteNullDetailSuccessItems(jobName: string, field: string): string[] { ... }
   ```
   Implement it by reading all `status = 'success'` rows for `jobName`, parsing each row's `detail` JSON
   in JS (not `json_extract` SQL — keep it dependency-simple), filtering to rows where `parsed[field] ===
   null`, deleting those specific `(job_name, item_key)` pairs inside a `db.transaction`, and returning the
   removed `item_key`s.

   Then add `scripts/reset-stock-sector-lookup-null-successes.ts`:
   - Calls `deleteNullDetailSuccessItems('stock-sector-lookup', 'industry')`.
   - Logs each removed ticker and a final count (0 removed = "already reconciled — no-op", matching
     `backfill-service-usage.ts`'s idempotent-rerun messaging).
   - A header comment explaining WHY this exists (the ticker-format bug meant every currently-held
     ticker was wrongly recorded `'success'` with a null industry before this fix; this script clears
     those specific poisoned rows so the corrected code re-resolves them on the next `stock-digest` run)
     and how to run it: `npx tsx scripts/reset-stock-sector-lookup-null-successes.ts` (against the live
     `data/jobs.db`; `LOCALJOBS_DB=/tmp/foo.db` to dry-test against a scratch DB).
   - This script is **not** wired into any automated pipeline and is **not** run against the production
     DB as part of building/verifying this task (per root `CLAUDE.md`'s test/production-DB separation) —
     it's a one-time tool for the owner to run once, after this fix ships, to unstick the 7 currently
     mis-recorded tickers.

4. **Update root `CLAUDE.md`'s `stock-digest` paragraph** (Definition of Done — docs are part of every

   before the API call (stripping the `_EQ`/market-code suffix), an unresolved lookup is now recorded
