## Overview
Cache the `gemini` per-place summary generation for 22 hours so re-running `enrich-with-llm` during
development doesn't re-pay for the same place. NOTE: gemini's response here is a web-grounded,
non-deterministic generation (it live-fetches the site + web-searches reviews) deliberately FROZEN for
the TTL — fine for a summary. In production the places `work_items` ledger already makes gemini a
once-ever call per place, so this cache is primarily a dev-convenience on a paid API. Outcome of the
T479 caching-adoption decision; the caching mechanism already exists (T451/T476).

## Do
1. Set `cacheTtlMs: 79_200_000` (22 hours) on the `gemini` `ServiceDefinition` in
   `src/services/gemini.service.ts`.
2. Add a `cacheKey` to the `callService('gemini', () => researchPlace(ai, place))` site in
   `src/workflows/places/stages/llm-enrich.ts` (around line 125), keyed by the place id AND a
   prompt-version tag: `` `gemini:place:${place.placeId}:v1` ``.
3. The `v1` version tag MUST be bumped whenever `buildPrompt(place)` (same file,
   `src/workflows/places/stages/llm-enrich.ts`) changes — otherwise a stale cached summary built from
   the OLD prompt would be served. Add a short code comment at `buildPrompt` stating "if you change
   this prompt, bump the vN tag in the gemini cacheKey above so the cache invalidates."
4. Cache the generation call only.

## Done when
- `cacheTtlMs: 79_200_000` is set on the `gemini` service def.
- The `researchPlace` call passes `gemini:place:<placeId>:v1` as its cacheKey.
- A code comment at `buildPrompt` documents the version-bump rule.
- `npx tsc --noEmit` and `npm test` are green.
- Verified a repeated enrich of the same place within 22h is served (the frozen summary) from
  `service_cache` (hermetic scratch-DB check) rather than a second live paid call.
