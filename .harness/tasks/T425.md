## Overview

Trading212's instruments-metadata endpoint (`fetchInstrumentsMetadata`, used to resolve ISIN/
real-world tickers) is documented — in the code's OWN comment — as a separate, much more tightly
rate-limited endpoint than the portfolio one (1 request per 50 seconds, per Trading212's OpenAPI
spec), but that limit is enforced ONLY by a code comment ("callers must call this AT MOST ONCE per
stage run"), not mechanically. Add a real, mechanical `minIntervalMs` enforcement for this specific
endpoint.

## Context (confirmed current)

`src/services/trading212.service.ts`:
- The single exported `ServiceDefinition` (lines 16-26, `name: 'trading212'`) sets only
  `ratePerMinute: Number(process.env.TRADING212_RATE_PER_MIN ?? 10)` plus `dailyCap`/`monthlyCap` —
  no `minIntervalMs` anywhere in this file.
- `fetchPortfolio` (the portfolio endpoint) and `fetchInstrumentsMetadata` (lines 119-141, the
  instruments-metadata endpoint) BOTH currently route through the SAME `callService('trading212',
  ...)` name at every call site — meaning they share one `ratePerMinute` budget. The portfolio
  endpoint's own documented limit (per this file's top-of-file comment) is roughly 1 req/sec —
  much looser than instruments-metadata's 1-per-50-seconds.
- `callService` (`src/core/services.ts`) enforces `minIntervalMs` (fixed spacing) and `ratePerMinute`
  (rolling per-minute count) as MUTUALLY EXCLUSIVE per service name — when `minIntervalMs` is set,
  the `ratePerMinute` branch never runs for that service. So a single `minIntervalMs: 50_000` added
  directly to the existing `trading212` service definition would ALSO force the portfolio-fetch
  calls (Invest + optional ISA, made back-to-back in `stocks-fetch.ts`/`stock-portfolio-fetch.ts`)
  to wait 50 seconds apart from each other and from any instruments-metadata call — an
  over-tight, incorrect constraint on the portfolio endpoint, which has its own looser real limit.
- Current call sites of `fetchInstrumentsMetadata`: `src/jobs/stocks-sync/stages/stocks-snapshot.ts`
  and `src/jobs/stock-digest/stages/stock-portfolio-snapshot.ts` (as of this task's authoring —
  two OTHER pending backlog tasks, T413/T414/T415, may restructure these files further; if either
  has landed by the time you build this, find wherever `fetchInstrumentsMetadata` is called from at
  that point and update those call sites instead — the principle doesn't change).
