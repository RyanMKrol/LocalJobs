## Do

On the workflow-run page, the **Input → Output mapping** panel (`IoPanel` in
`dashboard/app/workflow-runs/[id]/page.tsx`) renders a `<table>` of every row with no height
limit, so a large workflow (the places run has **~1,766 rows**) makes the section dominate the
page — you have to scroll past all of it to reach anything below. Cap its height and make the
table itself scroll.

- **Hard-cap the height** of the IO table area and make it **vertically scrollable inside the
  panel** (the page stops growing with row count). Use a sensible cap — e.g. `max-height`
  around `~60vh` (or a fixed value in the spirit of the existing `.logs`/`code-block`
  `max-height: 480px` pattern) — wrapping the `<table>` in a scroll container
  (`overflow-y: auto`) rather than capping the whole `.panel` (the "no work items recorded"
  empty-state `<p>` should stay un-capped). New CSS goes in `dashboard/app/globals.css`.
- **Keep the column header visible while scrolling** — make the `<thead>` sticky
  (`position: sticky; top: 0`) with a solid background so it doesn't disappear as you scroll
  the rows.
- **Preserve horizontal scrolling** — the existing `.panel:has(> table){ overflow-x: auto }`
  behaviour must still work; the new vertical scroll container must not break it (both axes
  scroll independently as needed).
- Consider showing the **row count** near the "Input → Output mapping" heading (e.g.
  "Input → Output mapping · 1,766 items") so it's obvious the list is long and scrollable —
  nice-to-have.
- Must stay **mobile-responsive** (~402px): the capped, scrollable table works at phone width
  with no page-level horizontal overflow.

Confine changes to the `IoPanel` markup in `dashboard/app/workflow-runs/[id]/page.tsx` and the
relevant CSS in `dashboard/app/globals.css`.

## Done when

The Input → Output mapping table is height-capped and scrolls vertically within its panel (the
page no longer grows unbounded with row count — a 1,766-row run is contained in a fixed-height
scroll area), its column header stays visible (sticky) while scrolling, horizontal scrolling
still works, and it remains usable at phone width. `npm --prefix dashboard run build` passes and
`node dashboard/scripts/mobile-check.mjs` stays green.
