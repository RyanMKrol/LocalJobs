## Do

Fix a run-limit (T094) bug: a **limited manual run** of a multi-stage workflow can select **zero
roots** and do nothing while reporting success, even when there's plenty of outstanding work.

**Reproduced:** a places run with `limit=300` froze `selected_roots=[]` and every stage reported
"nothing to do / success". The ledger had **1,753 resolved** places but only **131 enriched** —
so ~1,622 places were resolved-but-not-yet-enriched (the Google Places daily cap throttled
enrichment over time). Those 1,622 ARE the outstanding work, but the selector couldn't see them.

**Root cause.** `selectPendingRoots` (`src/db/store.ts`) marks a candidate root *pending* iff
`!isWorkItemDone(entryStage, root) || rootHasOutstandingWork(members, root)`. For a
resolved-but-not-enriched root: the entry stage (the resolver) IS done (first clause false), and
`rootHasOutstandingWork` is **false** — and that's the bug: it only finds **existing** not-done
work_items rows (`SELECT 1 … WHERE root_key=? AND NOT (done)`). A downstream stage that simply
**hasn't been attempted yet has NO row at all**, and a *missing* row isn't counted as outstanding.
So a root whose later stages haven't run looks "fully done" → not pending → not selected → with an
empty `selected_roots`, every stage's `ctx.rootAllowed()` is false → nothing runs.

(Unlimited/scheduled runs are unaffected — they don't use `selectPendingRoots`; each stage just
processes its own not-done items. Only the LIMIT path is broken, which makes the run-limit unusable
for catching up a backlog — its main use.)

Fix:
1. **Redefine "pending" so a root counts as outstanding until it has propagated all the way through
   the pipeline — not just past the entry stage.** Suggested approach: a root is *fully done* only
   when the **terminal stage(s)** (the last DAG wave) have a *done* (`success`/`ignored`) work_item
   for that `root_key` AND no stage has an outstanding (not-done) row for it; otherwise it's
   pending. For the places case, a resolved CID with no `enrich-with-llm` row for its `root_key` is
   then correctly pending. Thread the terminal-wave job names from the DAG into `selectPendingRoots`
   (the caller `runWorkflow` in `src/core/workflow-executor.ts` has the `dag.waves`).
   - **Handle the "never reaches the terminal" edge case:** a root that legitimately can't progress
     to the terminal stage (e.g. its only further work is a stage that filtered it out / a stuck
     `failed`-past-`maxAttempts` row) must NOT be re-selected forever. Treat a root as done if every
     stage that has a row for it is done AND it has no *retryable* outstanding work — i.e. a root
     stuck below the terminal stage with all its rows in a terminal state (success/ignored/
     exhausted-failed) is done, not perpetually pending. Cover this with a test so the selector
     can't livelock on an unprogressable root.
2. **Guard against the silent no-op:** when a limited run computes an empty `selected_roots` while
   the candidate set is non-empty, log a clear WARNING on the workflow run (e.g. "limit N requested
   but 0 originating inputs were selectable — N candidates, all already complete") instead of
   quietly succeeding with nothing.
3. **Fix the misleading stage log:** the places stages log "Nothing to enrich — every resolved
   place is already done" when the real reason is the empty allowlist. Reword so "0 to do"
   distinguishes "nothing allowed by the run limit" from "all done" (a small clarity fix in the
   places enrich/llm stages, or a generic note).
4. Update `CLAUDE.md`'s run-limit / pending-root description to the corrected semantics (a root is
   pending until it's fully processed through the terminal stage, not just past the entry stage).

Tests (hermetic, scratch DB): the regression — a root that's `success` at the entry stage with NO
row at a downstream/terminal stage IS selected as pending (model resolved-but-not-enriched);
a fully-propagated root (terminal-stage done, no outstanding) is NOT selected; an existing
retryable-failed downstream row still makes the root pending (preserve current behaviour); an
unprogressable/stuck root is NOT re-selected forever; and a limited run with 0 selectable roots
emits the warning.

## Done when

A limited run of a multi-stage workflow with a backlog (early stage done, a later stage not yet
attempted) **selects those roots and actually processes them** — a `limit=300` places run with
~1,622 resolved-but-unenriched places no longer selects 0 and no-ops; "pending" is defined by
whether a root has propagated through the **terminal** stage (with the unprogressable-root edge case
handled so it can't livelock); a limited run that selects 0 roots despite candidates logs a clear
warning; the misleading "every resolved place is already done" message is corrected; CLAUDE.md
documents the corrected semantics; hermetic tests cover the regression + the edge cases; `npx tsc
--noEmit` and `npm test` are green.
