## Overview

**The original question is already answered — this is not a live incident.** The owner suspected
`stock-digest` "hallucinated" a YNDX holding they didn't believe they owned. Investigation confirmed
YNDX is a genuinely real Trading212 position: the `stock-digest-build` stage's existing per-holding
`ctx.log` line (`info: [isa] YNDX_US_EQ: qty 27.96..., value 6066.80...`) recorded it in that run's
own logs **before Claude was ever called**, meaning it came straight from `stocks-sync`'s live
Trading212 API snapshot, not from Claude's narration. No hallucination occurred. This task is a
purely forward-looking observability improvement so that a **genuine** future hallucination (if one
ever happens) is caught automatically instead of requiring manual log archaeology through past runs.

Two gaps remain in `src/jobs/stock-digest/stages/stock-digest-build.ts` (`runStockDigestBuild`):
1. The exact `facts` object (`StockDigestFacts`) passed to Claude is only ever logged as individual
   formatted-string lines (one per holding) — the raw JSON blob itself is never persisted anywhere.
2. Claude's raw narrated response is written straight to the output `.md` file with **no check**
   that the tickers/numbers it mentions actually match `facts` — so if a real hallucination ever did
   happen, nothing would flag it.

## Do

1. **Persist the raw `facts` JSON per week**, alongside the digest markdown:
   - Add a `factsPathFor(weekKey, outDir)` helper to `src/jobs/stock-digest/config.ts`, mirroring the
     existing `reportPathFor` helper immediately above it: returns
     `resolve(outDir, \`stock-digest-facts-${weekKey}.json\`)`.
   - In `runStockDigestBuild` (`src/jobs/stock-digest/stages/stock-digest-build.ts`), after `buildFacts`
     is called and BEFORE the Claude call (`claudeRunner(...)`) — not after — write
     `JSON.stringify(facts, null, 2)` to `factsPathFor(key, outDir)` via `mkdirSync(outDir, {recursive:
     true})` + `writeFileSync`. Writing it before the Claude call (not after) means the raw facts are
     on disk for debugging even if the Claude call itself throws.
   - Log the write with `ctx.log(\`info: wrote raw facts JSON to ${factsPath}\`)`.
   - This file's lifecycle mirrors the digest markdown: idempotent per ISO week (a re-run the same
     week overwrites, not duplicates — matches the existing markdown behaviour, no ledger changes
     needed since `work_items` idempotency is already handled elsewhere in the function).

2. **Add a soft ticker cross-check between Claude's narration and `facts`.** Add two small, exported,
   pure helper functions (so they're independently unit-testable, matching this file's existing
   convention of exported pure helpers like `gainPct`/`pickMovers`):
   - `extractCandidateTickers(text: string): string[]` — scans the narrated markdown text for
     ticker-shaped tokens using a regex tuned to this repo's actual Trading212 ticker style (e.g.
     `YNDX_US_EQ`, `AAPL`): something like `/\b[A-Z][A-Z0-9]{1,9}(?:_[A-Z0-9]{1,9}){0,3}\b/g`
     (uppercase-leading, letters/digits/underscore, 2–15 chars). Dedupe the matches.
   - `findUnknownTickers(candidates: string[], facts: StockDigestFacts): string[]` — returns every
     candidate that is NOT in `new Set(facts.holdings.map(h => h.ticker))` AND is not in a small,
     exported stoplist constant (e.g. `KNOWN_NON_TICKER_TOKENS`) of common uppercase acronyms that
     legitimately appear in the prose but aren't tickers — seed it with at least `ISA`, `USD`, `GBP`,
     `EUR`, `ETF`, `N/A`, `ID`, `URL`, `API` and add any others you find false-positiving during
     testing/manual sample runs. Document in a code comment that this stoplist is a living, adjustable
     heuristic, not an exhaustive list.
   - In `runStockDigestBuild`, right after `ctx.log('info: Claude narration received')` and before
     `writeFileSync(mdPath, ...)`, run `findUnknownTickers(extractCandidateTickers(result.text), facts)`.
     If the result is non-empty, log **one** `ctx.log(\`warn: narrated digest mentions ticker-like
     token(s) not found in facts.holdings: ${unknown.join(', ')} — possible narration drift, verify
     manually\`)` line. This is explicitly a **soft signal, not a hard failure**: do NOT throw, do NOT
     skip writing the markdown, do NOT change the `markWorkItem` outcome — narration formatting can
     plausibly produce a false positive (e.g. a heading acronym), and the point is to make a genuine
     drift visible in the run's logs, not to block the pipeline on a heuristic.

3. **Update the root `CLAUDE.md`'s `stock-digest` paragraph** (Definition of Done in this repo — docs
   are part of every change) to mention: the raw `facts` JSON is now persisted per week to
   `data/out/stock-digest-facts-<weekKey>.json` (alongside the markdown report), and
   `stock-digest-build` performs a soft ticker cross-check between Claude's narration and `facts`,
   logging a WARN (not a failure) if the narration mentions a ticker-like token absent from the
   structured facts it was given.
