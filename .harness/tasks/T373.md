## Overview

Resolve each Trading212 position's ISIN and a current, real-world ticker symbol (via the OpenFIGI service added in T372) alongside the existing Trading212-native ticker, so downstream consumers get accurate ticker data instead of Trading212's own possibly-stale internal ticker symbols — e.g. `YNDX_US_EQ`, which is actually Nebius Group NV (ISIN `NL0009805522`); Trading212 updated the company's `name` field after its 2024 rename but never updated the `ticker` field to match.

## Do

This is a strictly READ-ONLY change to `src/jobs/stocks-sync/stages/stocks-snapshot.ts` — the broker-API read-only rule (root `CLAUDE.md`: "Broker / trading APIs are READ-ONLY, always") applies in full. Every new call added here must be a GET request; never POST/PUT/PATCH/DELETE.

1. Add a new injectable fetcher (mirror `fetchPortfolio`'s shape/pattern in the same file — same HTTP Basic auth, key id as username, secret as password) calling Trading212's instruments-metadata endpoint: `GET https://live.trading212.com/api/v0/equity/metadata/instruments`. This is a SEPARATE, more tightly rate-limited endpoint than the portfolio one — confirmed via Trading212's OpenAPI spec (`https://docs.trading212.com/_bundle/api.yaml`): 1 request per 50 seconds. Call it **at most once per stage run** (not once per position) and route it through `callService('trading212', ...)` exactly like the existing `fetchPortfolio` call, so the shared rate-limit/quota meter in `src/services/trading212.service.ts` covers it too. The response is an array of instrument objects, each with (at minimum) `ticker`, `name`, `isin`, `currencyCode`, `type`. Build a `ticker -> isin` lookup map from this response — you only need entries whose `ticker` matches something in `rawInvestPositions`/`rawIsaPositions`, no need to retain the full instrument list.

2. For each resolved ISIN, call OpenFIGI (via the fetch helper added in T372's `src/services/openfigi.service.ts`, gated through `callService('openfigi', ...)`) to resolve it to a real, current ticker. Batch ISINs into OpenFIGI mapping requests respecting its per-request job-count limit (10 without `OPENFIGI_API_KEY` set, 100 with one) rather than issuing one HTTP call per ISIN.

3. Extend `NormalizedPosition` (currently: `ticker`, `quantity`, `averageBuyPrice`, `currentPrice`, `currentValue`, `account`) with two new **optional** fields: `isin?: string` and `resolvedTicker?: string`. They must be optional because EITHER lookup can miss (the Trading212 ticker isn't found in the instruments-metadata response, or OpenFIGI returns `{ "warning": "No identifier found." }` instead of `data` for that ISIN) — a miss must NOT fail the whole snapshot. On a miss, `ctx.log('warn: ...')` naming the Trading212 ticker that couldn't be resolved and leave both new fields `undefined` for that position — the same soft-skip philosophy this repo already uses elsewhere (e.g. `FINNHUB_API_KEY` unset in `src/jobs/stock-digest/stages/stock-sector-lookup.ts`). **Keep the existing `ticker` field exactly as-is** (Trading212's own ticker stays the join key back to broker/ledger data — do not remove, rename, or repurpose it).

4. `writePortfolio` (same file) should include the two new fields in the written `data/out/portfolio.json`; the markdown table (`data/out/portfolio.md`) should show the resolved real ticker as a new column when present, falling back to showing the Trading212 ticker alone when a position has no `resolvedTicker`.
