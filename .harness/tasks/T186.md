## Do

On the Services page, let the owner **click a service** (gemini, google-places, fragrantica,
claude-cli) and see **which workflows/jobs use it**. Today there's no view of a service's consumers.

- **Determine the mapping by recording it at runtime** (accurate + self-maintaining): when a job
  calls `callService(name, fn)` (`src/core/services.ts`), record the calling **job** for that
  service. The calling job's name is available to the child process (the executor sets job/run
  context in the env for every child, like `LOCALJOBS_WORKFLOW_RUN_ID` — thread a job-name env the
  same way if one isn't already present). Persist **distinct `(service_name, job_name)` pairs** via
  `src/db/store.ts` — a small new table (e.g. `service_consumers (service_name, job_name, last_used)`,
  UNIQUE on the pair). Being a brand-new table, its columns/index live in `schema.sql` (no T098
  migration trap). (If a static declaration — jobs declaring the services they use — turns out
  cleaner, that's acceptable, but runtime-recorded ACTUAL usage is preferred.)
- Resolve each consuming job to its workflow(s) via `workflow_jobs`, so the view can show **service →
  workflow → job**.
- **API:** add `GET /api/services/:name/consumers` returning the workflows + jobs that use the
  service (grouped by workflow). Read-only.
- **Dashboard:** make each service on the Services page **clickable / expandable** (an expandable row
  or a `/services/[name]` detail view) showing the list of consuming workflows + jobs, each linking
  through to its workflow/job page. Keep the existing limits-editing UI intact.
- Mobile-responsive. Update CLAUDE.md (the new table + endpoint) and the README Services section.

## Done when

Clicking a service on the Services page reveals which workflows/jobs use it (service → workflow →
job), backed by a runtime-recorded `(service, job)` mapping captured in `callService` + a read-only
`GET /api/services/:name/consumers` endpoint, with links through to the workflow/job pages; existing
