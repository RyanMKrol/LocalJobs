## Overview
Clicking Ignore/Un-ignore on any of the 4 recs/gaps output managers (TV recommendations, movie recommendations, movie franchise gaps, missing TV seasons) currently does two annoying things: the item doesn't visibly move between the active/ignored lists until the next 5-second poll tick fires, and every click (except missing-tv-seasons' single-item Ignore) is gated behind a browser confirm() popup the owner wants gone. Fix both in the same pass since they touch the exact same handler functions.

## Do
In dashboard/app/ui.tsx, extend the `usePoll<T>(fn, intervalMs, deps)` hook (currently returns only `{ data, error }`, lines ~282-305) to also return a `refetch` function that immediately re-invokes `fn` and updates `data`/`error` outside the interval timer, without changing existing callers' behavior (usePoll is used elsewhere in the codebase for read-only polling — e.g. the workflow detail page's own `usePoll(() => api.workflow(name), 3000, [name])` at page.tsx line ~856 — those call sites can simply ignore the new return value, so this must be additive/backward-compatible). Then, in dashboard/app/workflows/[name]/page.tsx, update all 4 manager components — TvRecsManager (ignore L84, unignore L97, unignoreAll bulk L111), MovieRecsManager (ignore L259, unignore L272, unignoreAll bulk L286), MovieGapsManager (ignore L428, unignore L441, ignoreCollection bulk L455, unignoreCollection bulk L469), and MissingSeasonsManager (ignore L656 — no confirm() here already, unignore L671, ignoreShow bulk L686, unignoreShow bulk L700) — to: (1) call the new `refetch()` from their own `usePoll` call immediately after each successful `await api.*(...)` mutation call (inside the `try` block, right after the await, so a thrown error skips it and the existing catch/finally still runs), and (2) delete the `if (!confirm(...)) return;` guard line that currently opens every one of these handlers except MissingSeasonsManager's single-item `ignore()`, so the mutation fires immediately with no dialog. Do NOT touch the unrelated `resetOutput()` confirm() in the same file (~line 876, the 'Clear output data' danger-zone action) — that one stays exactly as-is.

## Done when
usePoll returns a working `refetch` alongside `data`/`error` and every existing usePoll call site still compiles/works untouched. All 13 ignore/un-ignore-related confirm() guard lines across the 4 managers are removed (grep -n "confirm(" dashboard/app/workflows/[name]/page.tsx returns exactly one match: the resetOutput() call). Every ignore/unignore/bulk handler in all 4 managers calls the manager's own usePoll refetch immediately after its mutation succeeds, so clicking Ignore/Un-ignore (single or bulk) moves the item between the active/ignored lists without waiting for the next poll tick — confirm this by reading the code path (no dialog blocks the click; refetch is called on the success path) and, since this is a layer=ui change, via the project's visual-confirmation step (build the dashboard, run node dashboard/scripts/visual-check.mjs and node dashboard/scripts/mobile-check.mjs, and look at the screenshots to confirm the ignore/ignored-section states render correctly with no confirm dialog and no layout regression). npx tsc --noEmit, npm test, and npm --prefix dashboard run build all stay green.
