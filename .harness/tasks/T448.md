## Overview

On a workflow-run detail page, the "Inputs & Outputs" panel's default **Overall** tab shows the
Inputs column and the Outputs column with the byte-for-byte SAME row — same key, same label, same
status — for `plex-space-saver` (and, more generally, for ANY workflow whose DAG has only one wave,
i.e. every member job is independent with no `dependsOn` edges: `claude-warmer`, `vercel-daily-redeploy`,
`listening-digest`, and any future single-stage workflow). This is confirmed live: a screenshot of
`http://localhost:4788/workflow-runs/<plex-space-saver run id>` shows "INPUTS · 1" and "OUTPUTS · 1"
each showing "2026-W27 / Size breakdown — 2026-W27 — click to preview [success]" — the identical
ledger row rendered on both sides.

## Confirmed root cause (file:line)

`src/api/server.ts`'s `stage-io?overall=true` handler (~line 2036-2053):

```ts
let rootWave: string[] = [];
let lastWave: string[] = [];
try {
  const dag = buildDag(refs);
  rootWave = dag.waves[0] ?? [];
  lastWave = dag.waves[dag.waves.length - 1] ?? [];
} catch { ... }
const def = getWorkflowDefinition(run.workflow_name);
const outputJobs =
  def?.outputJob && refs.some((r) => r.job === def.outputJob) ? [def.outputJob] : lastWave;
const { inputs, outputs } = stageIoLists(outputJobs, rootWave, run.id);
return json(res, 200, { inputs, outputs, predecessorJobs: rootWave, outputJobs, job: '__overall__' });
```

For a DAG with exactly one wave (`dag.waves.length === 1`), `rootWave` and `lastWave` are the SAME
array — `dag.waves[0]` and `dag.waves[dag.waves.length - 1]` are literally index 0 both times. So
`outputJobs` and the `rootWave` passed as the "inputs" job list are the identical job-name set (for
`plex-space-saver`, both are `['plex-space-saver-scan']`). `stageIoLists(outputJobNames, inputJobNames,
workflowRunId)` (`src/db/store.ts` ~line 777-799) runs the SAME SQL query (`queryJobRows`) against the
SAME job name for both `outputs` and `inputs`, returning identical rows in both. This is a deterministic
computation bug, not a data/timing issue — it will reproduce on every run of every single-wave
workflow, and is unrelated to the separate stale-absolute-output-path issue found for the SAME
workflow (see the companion `plex-space-saver-output-bug-1` task) — that one causes the row's preview
to fail to load; this one causes the row to be shown twice.

The owner's framing (start of this idea) already correctly intuited the fix direction: "the input and
output should be fetched in a programmatic way" — the Overall tab's Inputs side is conceptually meant
to represent "genuine external predecessor(s) of the terminal stage(s)", which for a single-wave DAG
is structurally NOT a thing — a wave-0 job that IS ALSO the terminal job has no real "input stage" to
show, exactly like `StageIoBlock`'s own per-stage tab already handles correctly today (a root stage
with no predecessors renders `emptyText="No inputs — this is the root stage."`, since
`predecessors = buildDag(refs).dependencies.get(jobParam) ?? []` is legitimately `[]` for a root job —
see the non-overall branch a few lines below, ~line 2060-2066). The Overall tab's bug is that it
computes its "inputs" job list from a *wave position* (`rootWave`) rather than from *genuine DAG
dependency edges*, so it never notices when the "root wave" and the "terminal wave" are the same jobs.

## Do

Fix `src/api/server.ts`'s `overall=true` branch so the Inputs side only shows jobs that are genuinely
distinct predecessors of the output jobs — never the same job(s) as `outputJobs` itself. Concretely:
compute `const inputJobs = rootWave.filter((j) => !outputJobs.includes(j));` and pass `inputJobs`
(not raw `rootWave`) as the `inputJobNames` argument to `stageIoLists`, and return
`predecessorJobs: inputJobs` in the JSON response instead of `rootWave`. For a single-wave workflow
this yields `inputJobs = []`, so `stageIoLists([...outputJobs], [], run.id)` returns `inputs: []` —
the Overall tab then renders its existing `emptyText="No inputs recorded this run."` (already correct
copy for an empty array, no dashboard component change needed) instead of a duplicated row. For every
existing MULTI-wave workflow (places, perfumes, movie-recommendations, stock-digest, etc.) `rootWave`
and `outputJobs` are already disjoint job sets, so `inputJobs === rootWave` and behavior is completely
unchanged — verify this explicitly for at least one multi-wave workflow as part of testing, not just
the single-wave fix.

Add a unit test (likely in `src/api/server.test.ts` or wherever the existing `stage-io` endpoint tests
live) covering: (1) a single-wave (all-independent-jobs) workflow's `overall=true` response has an
EMPTY `inputs` array even when the output job(s) have recorded ledger rows for the run, and (2) a
genuine multi-wave workflow's `overall=true` response is unaffected (inputs still reflects the real
root-wave predecessors, unchanged from today).
