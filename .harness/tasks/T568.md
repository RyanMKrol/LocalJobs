## Overview
The 16 service-definition files are all the same shape with tiny variation, and the boilerplate is exactly where the bugs live -- ~30 unvalidated Number(process.env...) reads (a poisoned '2,000' silently becomes NaN) and drift like vercel missing rateLimitSource. This unit adds a defineService() factory plus a validating envInt() so each definition can shrink to ~10 declarative lines with env reads that fail loud at load. It bundles the separate B19 env-validation fix -- author envInt here so B19 is not written twice.

## Do
In src/services/lib.ts add envInt(varName, fallback): reads process.env[varName]; when unset returns fallback; when set, strictly parse and THROW an Error NAMING varName on empty-string / non-finite / NaN / negative (so a poisoned env fails at module load). Add defineService(d): ServiceDefinition accepting { name, category, description, paid, rateLimitSource (REQUIRED), envPrefix?, ratePerMinute?, dailyCap?: number|'monthly/30', monthlyCap?, minIntervalMs?, maxJitterMs?, timeoutMs?, cacheTtlMs? }. It resolves numeric fields through envInt: by default from envPrefix (PREFIX_RATE_PER_MIN etc.), but MUST also accept an explicit resolved number or an explicit per-field env-var-name override so services keeping bespoke names (gemini's PLACES_LLM_MONTHLY_CAP, fragrantica's PERFUMES_FETCH_DELAY_MS/JITTER_MS, trading212-instruments' TRADING212_INSTRUMENTS_MIN_INTERVAL_MS) end up with byte-identical effective limits. The 'monthly/30' dailyCap sentinel resolves to dailyFromMonthly(resolvedMonthlyCap). rateLimitSource being required makes vercel's current omission a type error (fixed in R04b). Keep it flexible enough for the in-flight plex service shape (T511-T517) -- services with only minIntervalMs/timeoutMs and no caps must be expressible. Do NOT migrate any *.service.ts here -- only add and unit-test the helper.

## Done when
envInt and defineService exist in src/services/lib.ts, exported. src/services/lib.test.ts asserts: envInt returns fallback unset, parses valid ints, and THROWS with the var name in the message for empty-string, '2,000', NaN, and negative inputs; defineService resolves numeric fields correctly via both envPrefix and bespoke env-name overrides; the 'monthly/30' sentinel yields dailyFromMonthly(monthlyCap); and a // @ts-expect-error case documents that omitting rateLimitSource fails to compile. npx tsc --noEmit and npm test pass. No service file is migrated yet (that's R04b) and the existing dailyFromMonthly / DAILY_SPEND_DIVISOR exports are unchanged.
