## Overview

Part of a repo-wide sweep giving every job a real paragraph-length `description` (surfaced on
`/jobs/[name]` and `/runs/[id]`) instead of a one-liner. This task covers the 4 jobs in the `places`
workflow. The mechanism (dashboard render site on `/runs/[id]`, the doc convention) was already built
by the foundational task this depends on — this is pure content authoring, no code/UI mechanism
changes.

## Do

Rewrite the `description` string of each of the following 4 job files to a real 3-6 sentence
paragraph — what the stage does step by step, what it reads/writes (files, services, APIs), and any
notable behavior (rate limiting, idempotency, gating). Ground each in `src/workflows/places/CLAUDE.md`
(if present) and the stage's own code — do not invent behavior that isn't actually implemented. Plain
prose only, no markdown syntax (this renders inside a plain `<p>`, not a markdown viewer):

- `src/workflows/places/stages/ingest.job.ts` (`places-ingest`) — currently: "Normalize Takeout
  saved-place CSVs into places.json + validation report."
- `src/workflows/places/stages/resolve.job.ts` (`cid-to-place-id-resolver`) — currently: "Resolve
  saved-place CIDs to Google place_ids via a headless browser."
- `src/workflows/places/stages/enrich.job.ts` (`places-enrich`) — currently: "Enrich resolved places
  via the Google Places API (free-tier budgeted, weekly)."
- `src/workflows/places/stages/llm-enrich.job.ts` (`enrich-with-llm`) — currently: "Research each place
  with Gemini (grounded search + website) into a second-brain profile."

Also read root `CLAUDE.md`'s `places` section (the "What this project is" summary) for the overall
DAG shape (ingest → resolve → enrich → llm-enrich) so each paragraph correctly places its stage in the
pipeline (what it consumes from the previous stage, what it hands to the next).

## Done when

- All 4 `places` jobs have a real, accurate paragraph description grounded in their actual behavior —
  not a one-liner.
- `npx tsc --noEmit` and `npm test` are green.
