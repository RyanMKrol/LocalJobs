## Do

Fix a visible icon flicker in the header's light/dark toggle button (`ThemeControls`, `dashboard/app/ui.tsx:494-509`) that happens on every full-page navigation. The nav links in `dashboard/app/layout.tsx` are plain `<a>` tags (not `next/link`), so every nav click is a real full browser page load, not a client-side SPA transition — this fully remounts `layout.tsx` and the `ThemeControls`/`useMode()` inside it from scratch each time.

Root cause: `useMode()` (`dashboard/app/ui.tsx:401-428`) initializes `const [mode, setMode] = useState<ModeId>('dark')` with a HARDCODED default, then only corrects `mode` to the real value inside a `useEffect` that runs AFTER the first paint/render. Meanwhile the DOM's `data-mode` attribute is already set correctly and synchronously by the pre-paint inline script in `dashboard/app/layout.tsx` (the `PREPAINT` script, ~lines 31-36) before React even mounts — so the CSS is correct from frame one, but the React-rendered icon (🌙 vs ☀️) is one render behind reality on every full navigation whenever the real mode differs from the hardcoded `'dark'` default.

Fix: change the `useState<ModeId>('dark')` initializer in `useMode()` to a LAZY init function that reads the ACTUAL current mode synchronously from the DOM at mount time, e.g.:
```ts
const [mode, setMode] = useState<ModeId>(() => {
  if (typeof document === 'undefined') return 'dark'; // SSR guard
  const attr = document.documentElement.getAttribute('data-mode');
  return attr === 'light' ? 'light' : 'dark';
});
```
The pre-paint script has already set `data-mode` correctly before this component's first render, so this lazy initializer gives the CORRECT icon on the very first render — no flicker, even before the existing `useEffect` runs. Keep the `useEffect` (it still needs to reconcile with localStorage and listen for OS `prefers-color-scheme` changes) but it is no longer responsible for the FIRST correct paint.

Constraints — keep this narrowly scoped: do NOT change `ModeId` (stays binary `'dark' | 'light'`), do NOT change `resolveMode`, do NOT change the pre-paint script in `layout.tsx`, do NOT change the toggle button's markup or its binary click-to-toggle interaction, and do NOT touch the theme-family/font system (already removed by T308). This is purely an initial-render-correctness fix.
