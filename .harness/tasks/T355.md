## Overview

On the dashboard's Logs page (`dashboard/app/logs/page.tsx`), info/warn/error log lines are too
visually similar at a glance — only warn/error get a distinct TEXT colour today
(`.logs .lvl-warn`/`.logs .lvl-error` in `dashboard/app/globals.css`), `.lvl-info` has no rule at
all, and there is no background-colour distinction or literal level word anywhere in a rendered
line. Add a literal `[INFO]`/`[WARN]`/`[ERROR]` bracket tag to every log line, and give each level
its own text colour AND a subtle background tint, so severity reads instantly without relying on
colour alone.

## Do

1. In `dashboard/app/logs/page.tsx`, inside the log-line render (~lines 151-163), add a level tag
   immediately before the message text (after the timestamp/source span), rendering the literal
   bracketed level word in uppercase, e.g. `[INFO]`, `[WARN]`, `[ERROR]`:
   ```tsx
   <span className={`log-lvl-tag log-lvl-tag--${l.level}`}>[{l.level.toUpperCase()}]</span>
   ```
   Place it between the existing source link/span and the `{' '}{l.message}` join, keeping the
   existing spacing convention. This must render as real text content (not a CSS `::before`
   generated-content trick) so it stays selectable/copyable and is visible to a screen reader.

2. In `dashboard/app/globals.css`:
   - Add a dedicated `--logs-info` colour variable to the FIXED `--logs-*` block in `:root`
     (~lines 44-50, alongside `--logs-err: #f85149` / `--logs-warn: #d29922`). Per this repo's
     convention ("logs keep a fixed dark-terminal palette … never overridden by a theme or by light
     mode"), this new var must live in that same fixed `:root` block, NOT be theme-overridden.
     Pick a colour visually distinct from both `--logs-err` (red) and `--logs-warn` (amber) — e.g.
     a blue/cyan in the family of `--accent` (`#58a6ff`).
   - Add `.logs .lvl-info { color: var(--logs-info); }` — currently there is NO rule for `.lvl-info`
     at all, so info lines render in the container's plain `--logs-text` colour, indistinguishable
     from unstyled text. This is the core of the owner's complaint.
   - Add a subtle per-level BACKGROUND tint (in addition to text colour) to all three
     `.logs .lvl-info` / `.logs .lvl-warn` / `.logs .lvl-error` rules — low-opacity `rgba(...)`
     backgrounds (roughly 8-12% alpha), following the same tinting approach already used in this
     file for `.io-filter-chip--success.active` / `.io-filter-chip--failed.active`
     (~lines 162-165, e.g. `background: rgba(63,185,80,.12)`). Use a small `padding`/`border-radius`
     and a matching negative `margin` so the tint reads as a clean, full-width row band rather than
     a tight text highlight — keep it subtle (this is a dense log feed, not a highlighted alert
     list) and do not change line spacing/alignment.
   - Style the new `.log-lvl-tag` span for readability (e.g. `font-weight: 600`, a touch of
     `margin-right`); `color: inherit` from the parent `.lvl-*` rule is the simplest way to keep the
     tag the same colour as its line.

3. This task is deliberately scoped to ONLY the global Logs page (`dashboard/app/logs/page.tsx`),
   per the idea's explicit ask. Do NOT add the `[LEVEL]` bracket tag to the run-detail
   (`dashboard/app/runs/[id]/page.tsx`) or workflow-run-detail
   (`dashboard/app/workflow-runs/[id]/page.tsx`) log views, even though they render logs using the
   same `.logs`/`.lvl-*` CSS classes. Those two views will automatically inherit the improved
   colour/background distinction from your `globals.css` changes — that's expected and fine, just
   don't add any JSX changes to those two files.

4. This task is independent of the companion "Logs page: fix stale log lines" bugfix task — they
   touch different parts of `dashboard/app/logs/page.tsx` (the render block here vs. a filter-change
   effect there) and can be built in either order without conflict.
