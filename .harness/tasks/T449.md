## Overview

Add a `rateLimitSource` field to `ServiceDefinition` so every shared service can record WHERE its
`ratePerMinute`/`dailyCap`/`monthlyCap`/`minIntervalMs` numbers actually came from — a documented API
limits page (with a link), an empirical/observed estimate, or an explicit guess — and populate it
honestly for all current services. This is the data/plumbing half; a separate dependent task renders
it on the Integrations page.

## Context (confirmed current)

- `ServiceDefinition` lives in `src/core/types.ts` (lines ~261-284). It already has a manifest-owned,
  no-dashboard-edit-UI field of exactly this shape to mirror: `category?: string` (line 283, added by
  T305) — "Manifest-owned only — no dashboard edit UI, refreshed from code on every sync." Follow the
  SAME pattern for `rateLimitSource`, not the user-overridable pattern the three limit fields use.
- `src/db/schema.sql`'s `services` table (~line 186) already has `category TEXT NOT NULL DEFAULT ''`
  as a fresh-DB column. `src/db/index.ts` has the matching additive migration for EXISTING DBs (search
  for "T305" — a `PRAGMA table_info(services)` check + `ALTER TABLE services ADD COLUMN category TEXT
  NOT NULL DEFAULT ''`, guarded so it only runs once and carries no index — this is the T098-safe
  pattern to copy verbatim for the new column).
- `src/db/store.ts`'s `upsertServiceStmt` (~line 1522) and `syncService` (~line 1534) refresh
  `description`/`category`/`paid` unconditionally from the manifest on every sync (unlike the three
  limit columns, which are preserved once `limits_overridden = 1`). `rateLimitSource` must be
  refreshed the SAME unconditional way `category` is — there is no `_overridden` column and no
  edit endpoint for it.
- `ServiceRow` (in `store.ts`) and the dashboard's `Service` type (`dashboard/app/lib/api.ts`, ~line
  265) both mirror the DB row shape — both need the new field added so it round-trips through
  `GET /api/services` (which just spreads `listServices()` rows unchanged in `src/api/server.ts`
  around line 2110 — no server.ts change needed beyond what the DB/type layer already provides).
- The 12 current service files under `src/services/*.service.ts`: `claude-cli.service.ts`,
  `dynamodb.service.ts`, `finnhub.service.ts`, `fragrantica.service.ts`, `gemini.service.ts`,
  `github.service.ts`, `google-places.service.ts`, `hevy.service.ts`, `lastfm.service.ts`,
  `openfigi.service.ts`, `tmdb.service.ts`, `trading212.service.ts`.
  **If `src/services/trading212-instruments.service.ts` already exists by the time you build this**
  (a separate pending task, T425, adds it — it may or may not have landed yet), give it a

  the field is optional and this task doesn't depend on T425.

## Do

### 1. Add the field

In `src/core/types.ts`, add to `ServiceDefinition` (mirror the `category` doc-comment style):

```ts
/** Where this service's rate/quota numbers (ratePerMinute/dailyCap/monthlyCap/minIntervalMs) came
 *  from — a documented API limits page (name + link when one exists), an empirical/observed
 *  estimate from testing, or an explicit conservative guess when nothing is published. Write this
 *  HONESTLY: say "documented at <url>" only when you can point at a real page stating the number;
 *  otherwise say so plainly ("no public rate-limit docs found; empirically tuned" / "no published
 *  limit; conservative estimate"). Manifest-owned only — no dashboard edit UI, refreshed from code
 *  on every sync, exactly like `category`. */
rateLimitSource?: string;
```

### 2. Schema + migration (mirror the `category` T305 pattern exactly)

- `src/db/schema.sql`: add `rate_limit_source TEXT NOT NULL DEFAULT ''` to the `services` CREATE
  TABLE (fresh DBs only need this).
- `src/db/index.ts`: add the additive migration for existing DBs, right next to the `category`
  migration, same shape:
  ```ts
  if (!svcCols.some((c) => c.name === 'rate_limit_source')) {
    db.exec("ALTER TABLE services ADD COLUMN rate_limit_source TEXT NOT NULL DEFAULT ''");
  }
  ```
  No index on this column (avoids the T098 bootstrap-index trap).

### 3. Store layer

In `src/db/store.ts`:
- Add `rate_limit_source` to the `upsertServiceStmt` INSERT column list + `VALUES`, and to the
  `ON CONFLICT ... DO UPDATE SET` clause as an UNCONDITIONAL refresh (same treatment as `description`/
  `category`/`paid` — NOT the `CASE WHEN limits_overridden = 1 ...` treatment the three limit columns
  get).
- Update `syncService(def)` to pass `rate_limit_source: def.rateLimitSource ?? ''`.
- Add `rate_limit_source: string` to the `ServiceRow` interface.

### 4. Dashboard type

In `dashboard/app/lib/api.ts`, add `rate_limit_source: string;` to the `Service` interface (no other
API changes needed — `GET /api/services` already spreads the full row).
