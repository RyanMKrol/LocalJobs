## Overview

There is currently no way to reverse an "ignore" on a surfaced movie franchise gap, movie
recommendation, TV recommendation, or missing-TV-season — once dismissed, an item is gone for good.
Add a genuine un-ignore capability (backend + API + UI button) for all four of these audit-style
ignore lists.

## Do

**Confirmed starting state.** `src/db/store.ts` has `ignoreSurfacedItem(jobName, itemKey)` /
`ignoreSurfacedItems(jobName, itemKeys)` (~line 944) which upsert a `work_items` row to
`status='ignored'`. There is NO counterpart that clears it. The one-way ignore API surface:
`POST /api/movie-gaps/:tmdbId/ignore` + `POST /api/movie-gaps/ignore-bulk`, `POST
/api/movie-recs/:tmdbId/ignore`, `POST /api/tv-recs/:tmdbId/ignore`, `POST
/api/missing-seasons/:tmdbId/:season/ignore` + `POST /api/missing-seasons/ignore-bulk` — all in
`src/api/server.ts`. This task depends on `ignored-section-unignore-1`, which extracts the shared
`IgnoredSection` component and fixes its styling — build on top of that component; do not re-fork the
duplicated markup it removed.

**Design decision already made (don't re-litigate): un-ignoring DELETES the ledger row, mirroring
`unstickWorkItem`'s delete-and-refresh pattern (`src/db/store.ts` ~line 908), not a reset to some
other status.** Rationale: root `CLAUDE.md`'s own "Stuck vs ignored" section documents
`unstickWorkItem`'s delete pattern as the established precedent for a reversible ignore/unignore UI
control, and semantically "un-ignore" reads as "forget my decision, treat this as if I never saw it" —
not "keep it marked notified but merely make it visible again". **Consequence to be aware of and
document, not surprised by:** because these are audit-style workflows using the "have I already
notified this?" ledger pattern (T144 — a row NOT yet `success` is treated as newly-detected), deleting
the row means an un-ignored item is genuinely treated as brand-new and CAN be re-notified/reappear in
a future digest, exactly like an un-stuck item is retried fresh. This is deliberate, not a bug.

**1. Backend (`src/db/store.ts`).** Add:
```ts
/**
 * "Un-ignore" a surfaced item: delete its `ignored` ledger row so it's treated as
 * brand-new on the next run — the OPPOSITE of {@link ignoreSurfacedItem}, mirroring
 * {@link unstickWorkItem}'s delete-and-refresh pattern rather than resetting to some
 * other status. Because the audit-style workflows' ledger means "have I already
 * notified this?" (T144), deleting the row means the item CAN be re-notified/reappear
 * in a future digest — this is the deliberate, documented behavior of un-ignoring.
 * MANUAL ONLY. Returns the number of rows removed (0 if it wasn't ignored).
 */
export function unignoreSurfacedItem(jobName: string, itemKey: string): number {
  return db.prepare("DELETE FROM work_items WHERE job_name = ? AND item_key = ? AND status = 'ignored'")
    .run(jobName, itemKey).changes;
}
```
Only per-item — do NOT add a bulk variant. Unlike the bulk-ignore case (T210, dismissing a whole
collection/show at once), there's no analogous "un-ignore a whole group" action requested or
naturally useful here (the Ignored table lists individually-dismissed items one at a time; the owner
reconsiders them one at a time too).

**2. API (`src/api/server.ts`).** Add 4 new endpoints, mirroring the exact validation/guard pattern of
their `ignore` counterparts (same positive-integer checks, same loopback/token mutation guard,
placed right next to each `ignore` handler):
- `POST /api/movie-gaps/:tmdbId/unignore` → `unignoreSurfacedItem(MOVIE_GAPS_JOB, gapKey(tmdbId))`
- `POST /api/movie-recs/:tmdbId/unignore` → `unignoreSurfacedItem(RECS_JOB, recKey(tmdbId))`
- `POST /api/tv-recs/:tmdbId/unignore` → `unignoreSurfacedItem(TV_RECS_JOB, tvRecKey(tmdbId))`
- `POST /api/missing-seasons/:tmdbId/:season/unignore` →
  `unignoreSurfacedItem(PLEX_SEASONS_JOB, pairKey(tmdbId, season))`

Response shape mirrors the ignore endpoints: `{ ok: true, unignored: <rows affected> }`.

**3. Client (`dashboard/app/lib/api.ts`).** Add matching functions: `unignoreMovieGap(tmdbId)`,
`unignoreMovieRec(tmdbId)`, `unignoreTvRec(tmdbId)`, `unignoreMissingSeason(tmdbId, season)` —
mirroring the existing `ignoreMovieGap` / `ignoreMovieRec` / `ignoreTvRec` /
`missingSeasonsIgnore` functions right next to them.

**4. UI (`dashboard/app/workflows/[name]/page.tsx`).** In each of the 4 managers' Ignored table
(now rendered via the `IgnoredSection` component from `ignored-section-unignore-1`), add an action
column: an extra `<th></th>` in the header and, per row, a button — e.g. `↺ Un-ignore` — that, after a
`confirm()` (mirroring the existing per-item Ignore button's confirm pattern in the same file), calls
the matching client function. Reuse each manager's existing `busy`/`err` state (keyed the same way the
active-list Ignore button already keys it — by `tmdbId`, or `${tmdbId}:${season}` for
`MissingSeasonsManager`) rather than inventing new state; a given key can't be both active and ignored
at once so there's no collision. No local list manipulation needed on success — the existing
`usePoll(..., 5000)` will pick up the item moving back into the active list on its next poll, exactly
like the existing ignore flow already relies on.

Keep mobile-responsive; the new action column should follow the same `width: 1%; white-space: nowrap`
convention the active-list action columns already use (see `.movie-gaps-scroll table th:last-child` /
`td:last-child` in `globals.css`) where applicable — for `MovieGapsManager`/`MissingSeasonsManager`,
whose Ignored tables aren't inside `.movie-gaps-scroll`, add an equivalent narrow/no-wrap rule scoped
to `.ignored-section table` if needed so the button doesn't force wrapping/overflow.
