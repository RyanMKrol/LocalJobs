## Overview

Add a new shared service definition for OpenFIGI (Bloomberg's free ISIN ‚Üí ticker resolution API), following this repo's existing `src/services/*.service.ts` pattern, so any job can resolve a broker's ISIN into a current, real-world ticker symbol.

## Do

Create `src/services/openfigi.service.ts`, default-exporting a self-contained `ServiceDefinition` ‚Äî mirror `src/services/finnhub.service.ts`'s exact shape (`name`, `category: 'api'`, `description`, `ratePerMinute`, `dailyCap`, `monthlyCap`, `paid: false`, all limits read from env with sensible defaults, nothing imported from any workflow's `config.ts`).

OpenFIGI's stated free-tier limits (confirmed live via https://www.openfigi.com/api/documentation): **without** an API key, 25 requests/minute, 10 "jobs" (ISINs) per request; **with** a free API key (optional, header `X-OPENFIGI-APIKEY`), 25 requests per 6 seconds, 100 jobs per request. Set `ratePerMinute` conservatively at 20 (env-overridable via `OPENFIGI_RATE_PER_MIN`); `dailyCap`/`monthlyCap` are defensive metering only (OpenFIGI is free either way) ‚Äî default something generous like 2000/day, 20000/month, both env-overridable, matching how other free services in this repo set caps.

Also add an exported, injectable fetch helper (mirroring `fetchFinnhubProfile`'s shape in `src/jobs/stock-digest/stages/stock-sector-lookup.ts`) that performs the actual mapping call: `POST https://api.openfigi.com/v3/mapping`, `Content-Type: application/json`, body `[{ "idType": "ID_ISIN", "idValue": "<isin>" }]` (an array ‚Äî supports batching multiple ISINs in one call, respecting the job-count limit above). Only set the `X-OPENFIGI-APIKEY` header when `process.env.OPENFIGI_API_KEY` is set ‚Äî it's optional. The response is an array with one entry per input `idValue`, in the SAME order; a successful resolution has a `data` array (take the first entry's `ticker` field); a genuinely unmapped ISIN returns `{ "warning": "No identifier found." }` instead of `data` for that entry ‚Äî treat this as "no resolution", not an error, and do not throw.

This helper function does NOT live in the service file itself (a `ServiceDefinition` only carries limits/metadata ‚Äî the actual fetch is gated via `callService('openfigi', () => ...)` from whatever job calls it, exactly like every other service in this repo). Put the fetch helper in the SAME file as the export (`openfigi.service.ts` can still export both the default `ServiceDefinition` and this named helper function ‚Äî `finnhub.service.ts` does NOT do this, its helper lives in the consuming job file instead; either placement is acceptable here, but keep the helper exported and injectable so the consuming job (a separate, dependent task) can unit-test against a stub).
