## Do

Add a second, terminal stage to the `projects-sync` workflow (depends on the catalog stage from
T287) that turns each cataloged GitHub repo into a one-project markdown report using a local
shallow clone + the shared Claude CLI helper. Name the stage/job e.g. `project-summarize` (a new
`src/jobs/projects-sync/stages/project-summarize.job.ts` + `.ts` impl + `.test.ts`), and wire it
into `projects-sync.workflow.ts` with `dependsOn: [{ job: 'github-sync' }]` (or T287's renamed
equivalent) so the workflow becomes a 2-stage DAG.

For each repo recorded by the catalog stage (read `data/out/projects.json` or re-derive from the
prior stage's ledger — pick whichever is simpler given T287's actual output):
1. **Skip check first (idempotency).** Look up this repo's `work_items` row for THIS stage (keyed by
   `repoId`) and read its `detail.lastCommitSha` (or equivalent last-processed marker, e.g. the
   catalog's `pushed_at` timestamp if a real commit sha isn't cheaply available). If it matches the
   repo's CURRENT value from the catalog, skip entirely — no clone, no Claude call — and log why
   (e.g. "repo X unchanged since last summary (sha abc123) — skipping"). This is the mechanism that
   avoids reprocessing a repo untouched in the last month (or ever) — there is no separate
   calendar-based check; an unchanged sha IS "not touched since last run".
2. **Clone/pull.** For a repo that needs (re-)processing, shallow-clone (`git clone --depth 1
   <html_url> <dest>`) into a job-local, gitignored directory (e.g.
   `src/jobs/projects-sync/data/repos/<repo-name>/` — add this data path to the existing job
   `data/` gitignore convention, don't hardcode any absolute path). If a previous clone already
   exists at that path, `git pull` (or `fetch` + reset) instead of re-cloning from scratch. Use
   `child_process` (spawn/exec), inject the git-invoking function so it's replaceable in tests
   (mirror the `ReposFetcher`-style injectable pattern already used in `github-sync.ts`).
3. **Read repo context.** Read the cloned repo's README (any common filename/casing:
   `README.md`/`Readme.md`/`README`) plus the metadata already available from the catalog
   (description, language, topics, last-pushed date).
4. **Summarize via Claude.** Call `runClaude(prompt, model)` from `src/services/claude.ts` (already
   service-gated via the `claude-cli` service — reuse it as-is, do not modify it) with a prompt
   asking for a short markdown summary covering what the project is, what it's for/covers, and its
   last-commit date. Default the model to a Sonnet 5 id, overridable via a new
   `PROJECTS_SYNC_CLAUDE_MODEL` env var (mirror the `MOVIES_RECS_MODEL` pattern in
   `src/jobs/movies/config.ts`); document the new var in `.env.example`.
5. **Write output + ledger.** Write the result to `src/jobs/projects-sync/data/out/<repo-name>.md`.
   Record the `work_items` row for this repo/stage as `success` with
   `detail: { name: <repo full name>, markdown: <absolute path to the .md file>, lastCommitSha:
   <the sha/marker used for the skip check above> }` — the `detail.markdown` path is what makes the
   existing unified workflow Output section (T205) pick this up automatically; no dashboard changes
   are needed.
6. Skip producing any separate roll-up/index markdown file — one file per project in `data/out/`
   only (explicitly out of scope per the owner).

Update root `CLAUDE.md`'s `projects-sync` file-map/job description and `README.md`'s `projects-sync`
bullet to describe the new 2-stage shape (catalog → clone+summarize) and the idempotency rule
(skip unchanged repos by commit sha, not by calendar time).
