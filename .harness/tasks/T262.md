## Do

Decouple the workflow detail page's **Output section from markdown-only**, so a workflow can declare
what render form its output should take instead of every output being treated as a markdown artifact.
Today the unified Output section (`WorkflowOutputSection` in `dashboard/app/workflows/[name]/page.tsx`,
T205) assumes each terminal-stage work item is a markdown file: it shows a "View" button that fetches
`GET /api/workflows/:name/output?job=&key=` (`safeOutputMarkdown` in `src/api/server.ts`) and renders
the content via `react-markdown` in a popover.

Refactor so an output item carries a **declared form** (e.g. a `format`/`kind` on the item's `detail`,
recorded via `markWorkItem`), and the Output section **dispatches rendering by that form** to the
appropriate renderer. `markdown` stays a form (the existing renderer, unchanged), and the mechanism
must make adding a NEW form (e.g. a structured size table — used by T263) straightforward and
documented. Keep the read-only, path-safe output endpoint(s) confined to each job's `data/out/` tree
(no traversal, no paid/remote calls) — extend them as needed to serve a non-markdown form's data.

This is a refactor of SHARED output components — **it must not regress existing markdown outputs**:
places and perfumes must render their `data/out/` markdown profiles in the Output section exactly as
they do now.

## Done when

- An output item can declare its render form; the Output section dispatches to the right renderer;
  `markdown` is one form with its current behaviour intact (places/perfumes output renders identically
  — verify both).
- Adding a new output form is a clean, documented extension point (so T263 can add a size view without
  re-plumbing) — a short note in `CLAUDE.md`'s Output-section convention describes how.
- The output API stays read-only + confined to `data/out/` (the `safeOutputMarkdown`-style guards
  remain; any new form's data path is equally guarded).
- `npm --prefix dashboard run build`, `npx tsc --noEmit`, `npm test`, and
  `node dashboard/scripts/mobile-check.mjs` are green; if the output endpoint behaviour changed, a
  `src/api/server.test.ts` case covers it (`expectsTest`).
- `node dashboard/scripts/visual-check.mjs` was run after the build; since the Output "View" opens a
  popover (an interaction), add/keep a `FLOWS` entry in `dashboard/scripts/_dashboard-harness.mjs`
  (with `viewport: true`) that opens an output preview, and its screenshot confirms output still
  renders. The harness (`PAGES`/fixtures/`FLOWS`) is updated to reflect the change.
