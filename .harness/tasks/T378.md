## Overview

Add a documented harness convention — reinforced by a non-fatal pre-flight check in `loop.sh`,
mirroring the existing missing-facets warning — that any backlog task whose `scope` touches
`.harness/**`, or whose `facets.layer == "harness"`, must be authored `gate: "needs-human"` and
never left buildable. This closes a real gap: the owner asked whether such a convention already
existed and it did not — nothing today stops the autonomous loop from picking up and
unsupervised-building a change to its own build/task/calibration machinery.

## Do

- **Add the rule to `.harness/CLAUDE.md`.** Add a new subsection (near "Adding a backlog task →
  invoke the add-to-backlog skill" / "The floor (holds even on a direct edit)") stating: any task
  whose `scope` array includes a path prefixed `.harness/`, OR whose `facets.layer == "harness"`,
  MUST be authored with `gate: "needs-human"` — never `gate: null`. State the rationale explicitly:
  self-modifying build/calibration machinery running unsupervised is uniquely dangerous — unlike an
  ordinary buildable task, a bad autonomous edit here can corrupt `TASKS.json`, break task
  selection/escalation, or silently defeat the loop's own safety rails (the exact thing the loop is
  supposed to be constrained by). Because `.harness/CLAUDE.md` auto-loads whenever Claude works with
  `.harness/` files, this single doc edit is the mechanism that reaches every authoring path — the
  `ralph-loop-add-to-backlog` skill, a `/local-jobs-convert-ideas` sweep, or a hand-edit — without
  needing to touch anything outside this repo (see the scoping note below).

- **Cross-reference in `.harness/docs/HARNESS.md` §8.** Right where `gate:"gate"` / `gate:"needs-human"`
  are defined (search for that line — currently in the "TASKS.json schema" section), add a one-line
  pointer to the fuller rule + rationale in `.harness/CLAUDE.md`, so a reader of the schema doc sees
  the constraint at the point where `gate` is explained, not only in CLAUDE.md.

- **Add a mechanical, NON-FATAL pre-flight check to `.harness/scripts/loop.sh`.** Grep for
  `_missing_facets` to find the existing pre-flight WARN pattern (a `jq`/`tj` query over
  `TASKS.json` run once at loop startup, before the main iteration loop, that logs a WARN line
  naming offending task ids — non-fatal, does not stop the loop). Add a sibling check, in the same
  style, that WARNs about any task that is currently buildable (`status != "done"`, `gate == null`)
  AND EITHER has a `scope` entry starting with `.harness/` (prefix match — mirror how `scope`'s
  directory-glob matching already treats a trailing `/**`/`/*`/`/` as a directory prefix elsewhere in
  this repo's conventions) OR has `facets.layer == "harness"`. Log format should mirror the
  `_missing_facets` WARN's style (a clear message + a comma-joined list of offending task ids).

- **Keep it a soft warning — do not add a hard block in this task.** This repo's established idiom
  for backlog-hygiene issues (missing facets, etc.) is a non-fatal pre-flight WARN, not a startup
  refusal or a `select_task` skip. Match that here: do NOT make `loop.sh` exit non-zero, and do NOT
  change `select_task()`'s selection logic to skip such a task in this task. If, after this ships,
  the warning proves insufficient in practice (a harness-touching task actually gets auto-built), a
  stronger mechanical block — e.g. `select_task()` treating a scope-violating buildable task as
  gate-blocked regardless of its declared `gate` — is a reasonable escalation; add it as a new dated
  bullet under `.harness/CLAUDE.md`'s existing "Known-but-deferred issues" log rather than building
  it speculatively now.

- **Scoping decision — do NOT attempt to edit the `ralph-loop-add-to-backlog` skill itself.**
  `ralph-loop-add-to-backlog` is NOT part of this repo: it lives in the `claude-skills` plugin
  marketplace (installed under `~/.claude/plugins/.../ralph-harness/skills/ralph-loop-add-to-backlog`,
  developed in a sibling `claude-skills`/`claude-skills-loop` repo, not this checkout). Propagating
  this convention into that skill's own instructions is explicitly OUT OF SCOPE for this task — it
  mirrors an existing deferred cross-repo item (T188, which defers folding `convert-ideas`'s poor-fit
  gate into the same plugin). Relying on `.harness/CLAUDE.md` being auto-loaded for any Claude
  session (including one running that skill) while working inside this repo's `.harness/` is
  sufficient coverage for now — do not attempt to `git` operate on the plugin's separate repo as
  part of this task.

- **Apply the new rule to ITSELF.** This very task edits `.harness/CLAUDE.md` /
  `.harness/docs/HARNESS.md` / `.harness/scripts/loop.sh` — all under `.harness/`. Per the rule it is
  authoring, it is therefore correctly gated `gate: "needs-human"` (never buildable). This is the
  intended, self-consistent outcome, not a contradiction to "fix" by loosening its own gate.
