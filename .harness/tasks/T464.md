## Overview
Route every current direct `plexGet`/`plexPutStreams`/`triggerButlerBackup` call across all 6 Plex-touching workflows through the new `plex` service in ONE coordinated pass (per the owner's explicit ask not to migrate piecemeal), so Plex calls get the same consumer-tracking + call-count metering as every other service — mirroring the IDENTICAL pattern these same files already use for `tmdbGet` (which is already wrapped in `callService('tmdb', ...)` everywhere it's called).

## Do
Depends on `plex-service-a` (the `plex.service.ts` definition must exist first). In each of the 7 files listed in scope, find every direct call to `plexGet(...)`, `plexPutStreams(...)`, or `triggerButlerBackup(...)` (imported from `../../core/plex-client.js` or `../../../core/plex-client.js`) and wrap it in `callService('plex', () => <the original call>)`, importing `callService` from `../../core/services.js` / `../../../core/services.js` as appropriate (matching the exact import depth already used for `callService('tmdb', ...)` in the SAME files where both exist, e.g. `missing-tv-seasons/stages/snapshot.ts` sits next to `season-check.ts` which already imports `callService` the same relative depth). Do not change any call's arguments, return-value handling, or error handling beyond the wrap — this is a mechanical routing change, not a logic change. `plex-language-fix/stages/apply.ts` additionally wraps `plexPutStreams`/`triggerButlerBackup` — since those are genuinely mutating Plex calls, the wrap doesn't change their mutation semantics, only adds metering. For each touched stage's existing unit test file, update any place that mocks/stubs the module's `plexGet`/`plexPutStreams`/`triggerButlerBackup` directly to also account for the new `callService` wrapper (since `callService('plex', fn)` with no def loaded is itself testable ungated — check whether tests already register a `plex` service def or need `registerService`/an equivalent test seam; follow whatever existing tests for `tmdbGet`-calling stages already do for `callService('tmdb', ...)` as the template, since those tests already solved this exact problem for TMDB in these same files). After migrating, manually confirm (via a quick grep) that `src/core/plex-client.ts`'s own `plexGet`/`plexPutStreams`/`triggerButlerBackup` implementations are UNCHANGED — this task only touches call SITES, not the shared client itself, exactly mirroring how `tmdbGet`'s own implementation was never touched when its callers were wired onto `callService('tmdb', ...)`. Update each of the 6 workflows' own `CLAUDE.md` (one line each) noting Plex connectivity is now also metered via the shared `plex` service, and add a one-line note to the root `CLAUDE.md`'s `src/core/plex-client.ts` file-map entry mentioning the `plex` service now sits in front of `plexGet`/`plexPutStreams`/`triggerButlerBackup` calls, per this repo's Definition-of-Done-includes-docs rule.

## Done when
Every one of the 7 files in scope wraps its direct `plexGet`/`plexPutStreams`/`triggerButlerBackup` call(s) in `callService('plex', ...)`, with no change to any other behavior; `npx tsc --noEmit` and `npm test` are green (including updated per-stage tests); a grep for `plexGet(` / `plexPutStreams(` / `triggerButlerBackup(` outside of `callService(...)` wrappers and outside `src/core/plex-client.ts` itself returns nothing; the 6 touched workflows' own `CLAUDE.md` files and the root `CLAUDE.md`'s file-map entry for `src/core/plex-client.ts` are updated to note the new `plex` service. A live end-to-end confirmation (running a real Plex-touching workflow and seeing a `plex` row with real consumers on the dashboard's Integrations page) is a manual owner follow-up post-merge, not required for this task, since the build sandbox has no LAN access to a real Plex server.
