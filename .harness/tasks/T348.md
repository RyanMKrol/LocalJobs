`stocks-sync`'s DAG is `stocks-snapshot → stocks-watch → stocks-notify` (see `src/jobs/stocks-sync/stocks-sync.workflow.ts` and its doc comment). The unified Output section (T205, `workflowTerminalItems` in `src/db/store.ts`, served by `GET /api/workflows/:name/output-items` in `src/api/server.ts` ~line 1510) always queries the DAG's TERMINAL WAVE job name(s). For `stocks-sync` that's `stocks-notify` — which, BY DESIGN, never calls `markWorkItem` at all (it's a pure notify-trigger: reads `fresh-breaches.json`, optionally sends one push, done). So the Output section for `stocks-sync` is structurally guaranteed to be empty forever, even though real, meaningful per-item output exists one stage earlier — `stocks-snapshot` writes a `work_items` row per ticker position (and, per a sibling task in this same batch, will start recording rich `detail` on those rows).

This is a structural mismatch between the T205 convention's assumption ("the terminal stage IS the meaningful output") and workflows shaped like `stocks-sync` (build stage first, notify-only stage last). Fix it generically so any future notify-terminated workflow benefits, not just `stocks-sync`:

1. In `src/core/types.ts`'s `WorkflowDefinition` interface (alongside the existing `category` field, ~line 215-221), add a new optional field:
   ```ts
   /**
    * Override which member job's work_items ledger the unified Output section
    * (T205) reads from, for workflows whose DAG terminal stage legitimately
    * records NO per-item ledger rows (e.g. a pure notify-trigger stage like
    * stocks-sync's `stocks-notify`). Must name an existing member job of this
    * workflow. When unset (the common case), the Output section reads the
    * DAG's terminal wave, exactly as before.
    */
   outputJob?: string;
   ```

In `src/api/server.ts`'s `GET /api/workflows/:name/output-items` handler, after computing `lastWave` from `buildDag(refs)`, look up the workflow's manifest via the already-imported `getWorkflowDefinition(name)` (from `../jobs/registry.js`).
