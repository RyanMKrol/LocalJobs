## Do

Add a second, terminal stage to the `workouts-sync` workflow (depends on the stage redesigned in
T298, which accumulates the owner's full Hevy workout history to a local
`data/out/workouts-history.json`), that computes a per-exercise 6-month progress comparison and uses
Claude to write a narrative markdown report — covering the owner's explicit ask: "examine my
workouts over the last six months and produce a per-exercise breakdown showing progress from six
months ago to today", across ALL of: best single set, total volume, and estimated 1-rep-max.

1. Add `src/jobs/workouts-sync/stages/workouts-progress.job.ts` + `workouts-progress.ts` (+
   `.test.ts`), wired into `workouts-sync.workflow.ts` as a member `dependsOn` the sync stage from
   T298 (use the same `dependsOn` member-edge shape another multi-stage workflow manifest already
   uses).
2. Read `data/out/workouts-history.json` (the full accumulated history from T298: an array of
   workouts, each with `start_time` and `exercises[]`, each exercise with a `title`,
   `exercise_template_id`, and `sets[]` of `{ weight_kg, reps, set_type }`).
3. Define two calendar-month windows relative to the run: **current period** = the most recently
   completed calendar month (i.e. last month relative to the run date); **baseline period** = the
   calendar month exactly 6 months before the current period (e.g. if current period is July 2026,
   baseline is January 2026). Bucket every SET from `workouts-history.json` into whichever period its
   parent workout's `start_time` falls into (sets outside both periods are ignored for this report).
4. **Group by exercise** (key by `exercise_template_id`, the canonical Hevy exercise identifier, with
   the human-readable `title` carried along for display), and for EACH exercise that has at least one
   set logged in either period, compute, per period:
   - **Best single set**: the highest `weight_kg` among that period's sets for the exercise (ties
     broken by higher `reps`).
   - **Total volume**: `sum(weight_kg * reps)` across all of that period's sets for the exercise.
   - **Estimated 1-rep-max**: for each set compute `weight_kg * (1 + reps / 30)` (the Epley formula),
     and take the MAX across that period's sets for the exercise.
   Skip sets with a null `weight_kg` or null `reps` (e.g. duration/distance-based exercises) from all
   three calculations — note in the output when an exercise had sets that were skipped for this
   reason.
5. Write the full structured comparison (every qualifying exercise's baseline vs current values for
   all three metrics, plus the deltas) to `data/out/progress-data.json` — this is the "raw data" the
   owner asked for as a separate, inspectable artifact.
6. Feed `progress-data.json`'s content into `runClaude(prompt, model)` (the shared helper in
   `src/services/claude.ts` — reuse as-is, do not modify it) with a prompt asking for a markdown
   report that, per exercise, narrates the 6-month change across all three metrics (improved /
   plateaued / declined, with the actual numbers), and write the result to
   `data/out/workouts-progress.md`.
7. **Idempotency**: this is a monthly report, like `listening-digest` — use the SAME "idempotent per
   calendar month via the work_items ledger" pattern documented in root `CLAUDE.md` for that
   workflow: key the ledger row by the current period's `YYYY-MM`; a manual re-run within the same
   month regenerates that month's report rather than duplicating it.
8. Update root `CLAUDE.md`'s `workouts-sync` paragraph (from T298) and `README.md`'s bullet to
   describe the second stage: a monthly per-exercise 6-month progress report (best set / volume /
   estimated 1RM), written via Claude to `data/out/workouts-progress.md`, plus the raw
   `progress-data.json`.
