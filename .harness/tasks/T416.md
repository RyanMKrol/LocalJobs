## Overview

Formalize a new job-authoring convention, decided during T394's owner walkthrough: **an item-loop
job must fail its own run when ANY item it touched this run ended up failed — not silently return
success.** This resolves the T394 incident (a `project-summarize` run reported `'succeeded'` while
its own logs showed it failed to summarize its only repo, producing 0 output) without requiring any
new `RunStatus` value, threshold config, or executor change — it reuses machinery that already
exists.

## Context (read before writing anything)

During T394's walkthrough, the owner decided: **"if any part of a job fails, the entire job
fails"** — with the important caveat that already-succeeded items must NOT be lost or reprocessed;
the job should simply resume from wherever it left off on a subsequent run. Verifying this against
the real code confirmed the ENTIRE mechanism already exists:

1. **`work_items` (the per-item idempotency ledger) is completely independent of the job's overall
   return/throw.** Each `markWorkItem('success', ...)` call is a synchronous DB write as soon as
   that item finishes — a later `throw` at the end of a processing loop does not undo any of them.
2. **`src/core/dag.ts`'s `executeDag` already treats ANY status other than exactly `'success'` as
   blocking every dependent stage** (`const failedDep = dag.dependencies.get(job)!.find((d) =>
   status.get(d) !== 'success')`, `executeDag`'s ready-stage loop) — a genuinely `'failed'` run
   ALREADY halts the whole downstream pipeline today, cascading to every dependent, with no new
   mechanism required.
3. **`src/core/executor.ts`'s `runAttempts` already re-invokes a job fresh on retry** (up to
   `maxRetries`), and every item-loop job in this repo already follows the existing "idempotent /
   safe to re-run" convention (root `CLAUDE.md`'s Job conventions) — filtering its work-list by
   `!isWorkItemDone(...)` at the top of its loop. So a retry (or a fresh scheduled run after a
   failure) naturally resumes only the not-yet-done items; nothing new is needed there either.

So the actual gap is a **job-authoring pattern**, not a framework feature: jobs like
`project-summarize` (and `perfumes-build`, observed live failing-then-recovering across
`repeatUntilStable` cycles during this same session) catch each item's failure, log it, correctly
record it in the ledger via `markWorkItem(..., 'failed', ...)`, and then just `continue` all the
way to a clean `return` — nothing ever signals to the framework that the run itself should be
`'failed'`.

## Do

Add a new bullet to root `CLAUDE.md`'s "Job conventions" section (near the existing "Jobs must be
idempotent / safe to re-run" bullet), titled something like **"An item-loop job must fail its own
run if it failed ANY item this run (2026-07)"**, stating:

- The per-item `try/catch`/`markWorkItem(..., 'failed', ...)`/`continue` pattern inside a
  processing loop is UNCHANGED and still correct — one bad item must not stop the loop from
  attempting every other item this run.
- BUT at the END of that loop, if this run's own tally shows `failed > 0` (a genuine `'failed'`
  outcome — NOT a `'skipped'` soft-stop like a quota/rate-limit pause, which is an intentional
  defer-to-later-run, not a failure), the job's `run()` function MUST `throw` a summarizing `Error`
  (e.g. `` throw new Error(`${failed}/${processed} item(s) failed this run — see logs above`) ``)
  instead of returning normally.
- This reuses 100% existing machinery: the thrown error correctly marks the RUN `'failed'`
  (`src/runJob.ts` → `src/core/executor.ts`), which correctly blocks every downstream DAG dependent
  (`executeDag`, unchanged), and correctly triggers the job's own retry budget
  (`runAttempts`/`maxRetries`, unchanged) — a retry (or the next scheduled run) naturally skips
  already-`isWorkItemDone` items and resumes at the first not-yet-done one, since that idempotency
  filtering is already how every compliant item-loop job works.
- Most jobs already compute an `{ ok, failed, ... }` tally for their own logging/return value (e.g.
  `perfumes`'s `runFetch`/`runParse` already return `{ ok, failed, pending, rateLimited }`) — for
  those, the fix is usually just a couple of added lines at the very end: `if (failed > 0) throw
  new Error(...)`.
- This convention applies ONLY to genuine item-PROCESSING loops where a per-item failure is
  currently silently swallowed. It does NOT apply to stages with no per-item "did work" concept
  (e.g. a pure notify-trigger that either sends a push or doesn't).

Also add a short pointer/reinforcement in `src/workflows/CLAUDE.md` (mirroring how the existing
gate-coverage and detail-convention rules are reinforced there), since that file auto-loads when
working directly inside `src/workflows/`.

Do NOT retrofit any actual job in this task — that is separate follow-up work, tracked as its own
per-workflow backlog tasks (referencing this task's id). This task is documentation only.
