## Do

A workflow must only ever have ONE active run at a time (different workflows MAY still run concurrently — this is per-workflow, not a global lock). Today the Run button stays active on the Workflows list even while that workflow is running. Enforce the constraint at all three layers: (1) UI — disable/relabel the Run button (the per-row button on the Workflows LIST page workflows/page.tsx AND the 'Run now' button on the workflow DETAIL page workflows/[name]/page.tsx) whenever that workflow has an active run (the workflows API already exposes last_run; treat last_run.status === 'running' as active), e.g. show 'Running…' disabled. (2) API — POST /api/workflows/:name/run should reject a duplicate start with a clear response (e.g. 409 Conflict + message) instead of returning ok/202 and silently skipping, so callers know it didn't start. (3) EXECUTOR — the workflow executor is the AUTHORITATIVE guard: it must atomically refuse to start a second concurrent run of the same workflow (there is already a hasActiveWorkflowRun check / overlap-prevention — make it the single source of truth and race-safe, so two near-simultaneous POSTs can't both start). Add unit tests for the API reject and the executor guard (and that a DIFFERENT workflow can still start while one is running). Document the constraint in CLAUDE.md (executor/overlap section) + README.

## Done when

Starting a workflow that already has an active run is prevented at every layer: the Run button (list + detail) is disabled/shows 'running' while a run is active; POST /api/workflows/:name/run returns a clear conflict (e.g. 409) rather than appearing to start a second run; the executor atomically refuses a concurrent run of the SAME workflow so a race can't start two, while DIFFERENT workflows can still run concurrently; unit tests cover the API reject + executor guard + the different-workflow case; CLAUDE.md + README document the constraint; npx tsc --noEmit, npm test, the dashboard build, and mobile-check are all green.
