## Overview
scripts/*.ts (including the Plex undo recovery tool) is neither typechecked nor tested, and the test runner has three defects: it prints its pass/fail verdict before node:test drains, has no per-file timeout, and its npm script hardcodes a shared scratch-DB path that concurrent runs mutually delete. Make scripts/ visible to tsc and the runner, and fix all three defects.

## Do
1) Extend tsconfig.json `include` to ["src/**/*.ts","scripts/**/*.ts"] (keep existing excludes). 2) Extend scripts/run-tests.ts walk to [...walk('src'), ...walk('scripts')] — this is the base the dashboard-suite task extends, so land it first. 3) Fix the verdict-ordering defect: emit the final PASS/FAIL banner AFTER node:test drains (process.on('beforeExit') hook, or per-file child processes). 4) Add a per-file wall-clock timeout so a hanging test fails within it instead of the 6h Actions kill (per-file child-process isolation is the idea's preferred shape; a single-process per-file timeout is acceptable if isolation is too invasive). 5) Fix the scratch-DB collision: drop the hardcoded `rm -f /tmp/lj-test.db*; LOCALJOBS_DB=/tmp/lj-test.db` from package.json's `test` script and rely on config.ts's isTestEnv/resolveDbPath per-PID scratch path (or mktemp -d); each parallel worker must still get its own per-PID scratch DB.

## Done when
Self-demonstrating per criterion: breaking plex-language-undo.ts's imports fails `npx tsc --noEmit`; scripts/plex-language-undo.test.ts is discovered and run by `npm test`; a deliberately-failing node:test file makes the runner's LAST printed line the failure verdict; a deliberately-hanging test file fails within the per-file timeout; two concurrent `npm test` runs both pass. The existing suite stays green (`npm test` and `npx tsc --noEmit` both pass). Whether runner-defect fixes may be split into a follow-up is an open owner question (pending-questions #17).
