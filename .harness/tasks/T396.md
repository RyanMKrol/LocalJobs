## Overview

Add a fleet-wide "run everything" admin endpoint that iterates every workflow and kicks off a
manual run of each (defaulting to a 3-originating-input limit where the workflow supports limiting),
so the owner can trigger a "run all and see what happens" sweep from one action instead of visiting
every workflow's detail page and clicking Run individually. This mirrors the shape of the existing
bulk `POST /api/workflows/reset-output-all` (T322/T323) but triggers runs instead of resetting output.

## Do

Add a new mutating endpoint `POST /api/workflows/run-all` (optional body `{ limit }`, default `3`) in
`src/api/server.ts`, registered near the existing single-workflow `POST /api/workflows/:name/run`
handler (around `src/api/server.ts:1586`) and following the same enumeration pattern the bulk
`reset-output-all` endpoint uses (around `src/api/server.ts:1690`, via `listWorkflows()` from
`src/db/store.ts`).

**This endpoint is fire-and-forget per workflow, NOT synchronous like `reset-output-all`.** A workflow
run can take minutes; the endpoint must not block the HTTP response waiting for every workflow to
finish. Mirror the EXISTING single-workflow `/run` endpoint's behaviour (`runWorkflow(def, 'manual',
{ limit }).catch((e) => console.error(...))`, responding `202` immediately) — just do it in a loop
across every workflow, synchronously checking/starting each one (the check-and-fire has no `await`
between them per workflow, same as the single-workflow endpoint), then return once every workflow has
been either started or skipped.

Behaviour, per workflow returned by `listWorkflows()`:

1. Validate the optional `limit` from the request body: if provided, it must be a positive integer
   (same validation as the single-workflow `/run` endpoint at `src/api/server.ts:1590-1595`) — else
   400. If omitted, default to `3` (the idea's stated default).
2. Look up the workflow's `WorkflowDefinition` via `getWorkflowDefinition(wf.name)` (from
   `src/jobs/registry.ts`, already imported in `server.ts`).
3. If `workflowRunInProgress(wf.name)` is true (the T105 one-active-run-per-workflow guard, already
   imported/used elsewhere in `server.ts`), SKIP that workflow — do NOT treat it as an error for the
   whole bulk call. Record it in the response as `{ name, status: 'skipped', reason: 'already running'
   }`. **One busy workflow must never block starting any other workflow** — same "best-effort across
   all, not all-or-nothing" principle `reset-output-all` already established.
4. Otherwise, determine whether the workflow is *limitable* — some member job declares `inputKeys()`
   — using the SAME expression the `GET /api/workflows` list endpoint already computes (`
   def.jobs.some((j) => getJobDefinition(j.job)?.inputKeys)`, see `src/api/server.ts:965`).
   - If limitable: call `runWorkflow(def, 'manual', { limit })` (the resolved/defaulted limit).
   - If NOT limitable: call `runWorkflow(def, 'manual', {})` — i.e. run it UNLIMITED. Per T094, `limit`
     is documented as a no-op for a workflow with no `inputKeys()`-declaring stage, so there is no
     reason to force a limit that would do nothing; running it unlimited is simplest and matches how a
     non-limitable workflow behaves on every other manual run today.
   - `.catch((e) => console.error('[api] run-all: workflow run error', wf.name, e))` — same
     fire-and-forget error handling as the single-workflow endpoint; a later failure must not throw out
     of the loop or affect any other workflow's dispatch.
   - Record `{ name, status: 'started', limited: <boolean>, limit: <the limit used, or null when
     unlimited> }`.
5. Respond `202` (not `200` — nothing has *finished*, only been dispatched) with a structured summary:
   ```json
   {
     "ok": true,
     "totalWorkflows": 13,
     "startedCount": 11,
     "skippedCount": 2,
     "limit": 3,
     "results": [
       { "name": "places", "status": "started", "limited": true, "limit": 3 },
       { "name": "claude-warmer", "status": "started", "limited": false, "limit": null },
       { "name": "movie-recommendations", "status": "skipped", "reason": "already running" }
     ]
   }
   ```
6. Log a single summary line via `console.log` (mirroring `reset-output-all`'s `[api] reset-output-all:
   … reset, … skipped` line), e.g. `[api] run-all: N started (limit 3), M skipped (already running:
   ...)`.
7. **Include DISABLED workflows too.** A workflow's `enabled` flag only gates the *scheduler* firing it
   automatically — manual runs (the single-workflow `/run` endpoint) already bypass it entirely, and
   `reset-output-all` already enumerates every workflow regardless of `enabled`. Do the same here: don't
   filter by `enabled`.
8. This is a MUTATING, fleet-wide endpoint — it MUST sit behind the exact same loopback/token mutation
   guard (`authoriseMutation`) as every other POST endpoint (`/toggle`, `/run`, `/schedule`,
   `/concurrency`, `/notify`, `/reset-output`, `/reset-output-all`). Do not add a separate auth
   mechanism.
9. Register the route with an explicit `parts[2] === 'run-all' && parts.length === 3` match (mirroring
   `reset-output-all`'s `parts[2] === 'reset-output-all' && parts.length === 3` guard) so it can never
   be confused with the `:name`-based `/api/workflows/:name/run` route (which additionally requires
   `parts[3] === 'run'` — a genuinely different shape, so ordering relative to it doesn't matter for
   correctness, but place it near that route for readability).
10. Add a client helper `api.runAllWorkflows(limit?: number)` in `dashboard/app/lib/api.ts` (mirroring
    `api.resetAllWorkflowsOutput()` at line ~632 and `api.runWorkflow(name, limit?)` at line ~566) that
    POSTs to `/api/workflows/run-all` with `{ limit }` in the body when a limit is passed (omit/undefined
    to let the server default to 3) and returns the parsed JSON response, typed to match the response
    shape above.

Do not touch `dashboard/app/**` page components in this task (that's the paired UI task) — only
`src/api/server.ts` (the new route) and `dashboard/app/lib/api.ts` (the new client helper), plus a unit
test.

## Done when

- `POST /api/workflows/run-all` exists in `src/api/server.ts`, behind the same `authoriseMutation`
  loopback/token guard as other mutating endpoints, accepting an optional `{ limit }` body (default `3`,
  must be a positive integer when provided, else 400).
- It iterates every workflow from `listWorkflows()`. A workflow with `workflowRunInProgress(name) ===
  true` is SKIPPED (not started, no error) while every other workflow has `runWorkflow(def, 'manual',
  { limit })` fired for it (fire-and-forget, matching the single-workflow `/run` endpoint's pattern —
  the request does NOT wait for any workflow run to finish).
- A non-limitable workflow (no member declares `inputKeys()`) is started UNLIMITED (no `limit` passed to
  `runWorkflow`), not rejected and not force-limited.
- Response is `202` with a per-workflow `results[]` array (`name`, `status: 'started' | 'skipped'`, and
  either `{ limited, limit }` or a skip `reason`) plus aggregate `totalWorkflows` / `startedCount` /
  `skippedCount` / the resolved `limit`.
- A summary line is logged via `console.log` naming started vs skipped counts.
- `dashboard/app/lib/api.ts` gets a new `runAllWorkflows(limit?)` client helper that POSTs to the new
  endpoint.
- A new unit test in `src/api/server.test.ts` (or the closest existing server-route test file) exercises
  the endpoint against a scratch DB seeded with at least 3 real fixture workflows registered via
  `syncJob`/`syncWorkflow` (mirroring the existing T105 guard test at
  `src/api/server.test.ts:479-507`, which registers a trivial no-op `JobDefinition` — `run: async () =>
  {}` — and a matching one-stage `WorkflowDefinition`, so `runWorkflow` genuinely executes it end-to-end
  fast with no real work): at least one workflow with a job declaring `inputKeys()` (limitable), at
  least one plain workflow with no `inputKeys()` (non-limitable), and one workflow made to look
  "already running" via `createWorkflowRun(name, 'manual')` (same helper the T105 guard test uses,
  cleaned up afterwards via `finishWorkflowRun`). Assert: the already-running workflow is reported
  `skipped`/`'already running'`, the limitable workflow's result shows `limited: true` with the resolved
  `limit`, the non-limitable workflow's result shows `limited: false` and `limit: null`, and the overall
  response shape matches (`results[]`, `startedCount`, `skippedCount`, `totalWorkflows`). Clean up all
  registered fixtures afterwards so other tests are unaffected, matching the existing convention in this
  file.
- `npx tsc --noEmit` and `npm test` are green (repo-wide Definition of Done). No dashboard build needed
  for this task (no `dashboard/app/**` page changes — only the `lib/api.ts` client helper, which is plain
  TypeScript with no visual surface).
