## Do

The `rec-merge` **top-up loop** re-prompts the 8 recommender branches **sequentially** — each Claude
CLI call takes ~80s, so a single top-up round takes ~11 minutes (8 × ~80s), and with up to
`recsTopUpRounds` (3) rounds the job can run 30+ minutes. The branch re-prompts are **independent**,
so run them **concurrently** (bounded) to collapse each round's wall-time.

- In the top-up loop (`src/jobs/movies/stages/merge.ts`), fire the branch re-prompts in parallel
  (e.g. a concurrency-limited `Promise.all` over the branches) instead of `await`-ing each in turn.
- **Bound the concurrency.** Each branch is a `claude` CLI **subprocess**, so running all 8 at once
  is heavy on the Mac Mini (CPU/memory). Cap it (e.g. 3–4 concurrent) so a round runs in ~2–3
  batches rather than 8 serial calls. Make the cap a config/env value with a sensible default.
- Calls still route through the shared `claude-cli` service via `callService` (rate-limit/quota
  coordinated) — concurrency just lets the long calls **overlap** instead of running strictly
  one-after-another; the service still paces the starts. A hit quota / rate-limit must still
  **soft-fail that branch gracefully** (the round continues with whatever the other branches
  returned), as today.
- Keep the per-round logging (T176) correct under concurrency — attribute each branch's results to
  the right branch even when they complete out of order.
- **Performance only — no change to WHAT gets recommended** (same pooling / dedup / quality bar /
  target logic), just how fast the branches are re-prompted.

## Done when

The `rec-merge` top-up rounds re-prompt the branches concurrently (bounded to a safe configurable N),
so a round's wall-time drops from ~8×call to ~ceil(8/N)×call; the `claude-cli` rate/quota
coordination and graceful per-branch soft-fail still hold; per-round logging stays correct under
out-of-order completion; the recommendation output is unchanged; `npx tsc --noEmit` and `npm test`
are green.
