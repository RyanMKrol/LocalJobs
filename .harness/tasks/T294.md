## Do

Surface a task's **build-attempt failure history** on the Backlog dashboard page
(`dashboard/app/backlog/page.tsx`) — this is a DIFFERENT concept from the existing red "✗ failed"
`Pill` (which reflects the owner-driven `.harness/manual-fail.json` overlay on a `done` task).
The gap: a task that is still `status: "pending"` (fully buildable — all its `dependsOn` are done)
but has ALREADY been attempted by the loop and hit a blocker or failure (recorded in
`.harness/failures.jsonl`, e.g. `kind: "agent-blocked"`, `"audit-fail"`, `"ci-red"`,
`"local-dod"`, `"scope-creep"`) currently shows up in the "ready"/buildable section looking
IDENTICAL to a task that has never been attempted at all — there is no visual signal that it kept
failing to build. A concrete real example at time of writing: **T282**
(`.harness/tasks/T282.md`) is `status: "pending"` with its dependency (T262) done, so it shows as
ready/buildable, yet `.harness/failures.jsonl` already has one row for it
(`kind: "agent-blocked"`, reason: scope excludes a file the render logic moved into). Use it as a
manual test case.

1. **Backend** (`src/api/server.ts`): add a small helper, e.g. `readTaskBuildFailures(id, baseDir)`
   (mirror the shape/safety style of the existing `readWorklogContent(id, baseDir)` a few lines
   above it — same `id` validation, same "confined to a specific file under the backlog's directory"
   pattern) that reads `.harness/failures.jsonl` (JSON-Lines; may not exist — treat as no failures,
   not an error), filters lines whose `.id` matches the task id, and returns `null` when there are
   none, or `{ count: number, latestKind: string, latestDetail: string, latestAt: string }`
   (aggregated from ALL matching lines: `count` = number of matching lines, `latestKind`/
   `latestDetail`/`latestAt` from the line with the greatest `ts`) when there are.
2. Wire this into `readBacklog()` (~line 308-345): for each task, call the new helper (using the
   same `baseDir` already computed there) and, when non-null, spread `{ buildFailures: {...} }` onto
   the returned task object — mirroring exactly how `specContent`/`worklogContent` are conditionally
   spread in the same function. Terminal tasks (`done`/`failed` status) don't need this (their
   outcome is already shown some other way) — only compute/attach it when useful, but it's fine to
   compute it for every task and let the frontend decide where to render it if that's simpler.
3. **Frontend** (`dashboard/app/backlog/page.tsx`): in the ready/buildable list (and the human/waiting
   list, if relevant — anywhere a still-pending task can appear OUTSIDE the terminal "Done" section),
   when a task has a non-null `buildFailures`, render a distinct warning-style `Pill` next to its
   existing `buildable`/`human`/`dep-waiting` pills — e.g. `⚠ {count} failed attempt(s)` — with a
   `title` tooltip showing `buildFailures.latestKind` + `buildFailures.latestDetail` (mirror the
   existing `<Pill kind="failed" title={t.failReason ?? undefined}>` tooltip pattern used for the
   terminal failed pill). Use a NEW `Pill` `kind` (do not reuse `"failed"`, which visually means
   "owner-confirmed done-but-actually-failed" — a different, terminal state) — add it to
   `globals.css`'s pill kind rules (e.g. `.pill.blocked` or similar, an amber/warning color distinct
   from both the green "done"/"buildable" and red "failed" pills) per the T268 shared-component
   convention (document the new kind where the other pill kinds are documented).
