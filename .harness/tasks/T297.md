## Do

existing user-editable + code-reconciled override pattern used for a workflow's `schedule` (T135)
and `maxConcurrency` (T169) — but applied to a JOB row instead of a workflow row (a genuinely new,
but structurally identical, override). **Scope decision already settled with the owner: there is NO
workflow-level timeout — only the existing per-job `timeoutMs` becomes editable, on the job's own
page, NOT the workflow page. The dashboard displays and accepts the value in SECONDS, converting
to/from milliseconds at the API boundary** (internal storage/enforcement stays milliseconds,
unchanged).

1. **`src/db/schema.sql`** — add `timeout_ms_overridden INTEGER NOT NULL DEFAULT 0` to the `CREATE
   TABLE IF NOT EXISTS jobs` block (~line 5-11), alongside the existing `timeout_ms` column.
2. **`src/db/index.ts`** — add an additive migration for existing databases: `PRAGMA table_info(jobs)`
   check + `ALTER TABLE jobs ADD COLUMN timeout_ms_overridden INTEGER NOT NULL DEFAULT 0` (mirror the
   `notify_enabled`/`max_concurrency` migrations already in this file). Per the T098 rule, add NO
   index on this column.
3. **`src/db/store.ts`** — `upsertJobStmt`/`syncJob` (~lines 20-36): change so `timeout_ms` is
   PRESERVED across a code re-sync when overridden, exactly like `schedule`/`max_concurrency` on
   workflows: `timeout_ms = CASE WHEN timeout_ms_overridden = 1 THEN timeout_ms ELSE
   excluded.timeout_ms END` (leave `description`/`max_retries` unconditionally refreshed, as today).
   Add `timeout_ms_overridden: number` to the `JobRow` interface (~line 38-44). Add a new exported
   function `updateJobTimeout(name: string, timeoutMs: number): JobRow | undefined` that runs `UPDATE

4. **`src/api/server.ts`** — add `POST /api/jobs/:name/timeout` accepting `{ timeoutMs: number }`
   (milliseconds — the API stays in ms; only the DASHBOARD converts to/from seconds). Validate:
   integer, `>= 0` (0 = no timeout, matches the existing convention), else **400**. Call
   `updateJobTimeout`; **404** if the job doesn't exist. This is a MUTATING endpoint — put it behind
   the SAME loopback/token guard every other mutating endpoint uses (`/toggle`, `/run`, `/schedule`,
   `/concurrency`, `/notify`, `/limits`, …). NOTE: per T070 there is deliberately no `/run` or
   `/toggle` for jobs — this endpoint does NOT reintroduce those; `timeoutMs` is a genuinely

   change is needed for the override to take effect on the next run.
5. **`dashboard/app/jobs/[name]/page.tsx`** (~line 41, currently `<div>{job?.timeout_ms ?
