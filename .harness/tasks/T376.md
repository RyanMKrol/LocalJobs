## Overview

Each stage box in a workflow's DAG graph has uneven vertical padding — the job-name text sits
noticeably above center, leaving a big empty gap at the bottom of the box. Vertically center the
node's content so the padding is even top and bottom.

## Do

**Confirmed empirically (screenshots taken via `dashboard/scripts/visual-check.mjs` before authoring
this task) that this is a structural-view-only bug, not present on run-scoped graphs:**

- On `/workflows/<name>` (a definition-view graph — no run, e.g. `/workflows/places` or
  `/workflows/stock-digest`), each `StageNode` renders ONLY the job-name line (`showStatus` is
  `false` since there's no run to show a status for — see `dashboard/app/components/DagFlow.tsx`
  line 288: `showStatus: !!statusByJob`, and `statusByJob` is never passed on this page). With a
  single short line of content and no vertical-centering rule, the text sits flush at the top
  (right after the box's top padding) and the box's `min-height: 58px` forces the leftover ~20-30px
  of empty space to accumulate entirely at the bottom.
- On `/workflow-runs/<id>` (a run-scoped graph, `showStatus` is `true`, both the job-name AND a
  status line like `PENDING`/`FAILED` render), the two lines already fill the box much more evenly
  and the imbalance is barely noticeable — this case is not the primary target of this task, but
  must not regress.

**Root cause:** `dashboard/app/globals.css:235`'s `.dag-node` rule:
```css
.dag-node { min-width: 156px; min-height: 58px; padding: 10px 12px 10px 14px; border-radius: 9px; background: var(--panel); border: 1px solid var(--border); border-left-width: 3px; box-shadow: 0 1px 3px rgba(0,0,0,.3); transition: border-color .15s, box-shadow .15s; box-sizing: border-box; }
```
has no flexbox/vertical-centering rule — plain block layout, so content is always top-anchored and
any leftover `min-height` space accumulates at the bottom.

**Fix — add flexbox vertical centering to `.dag-node`:**
```css
.dag-node { ...existing properties...; display: flex; flex-direction: column; justify-content: center; }
```
This vertically centers whatever content is present, correctly handling BOTH the one-line structural
case (name only) and the two-line run-scoped case (name + status) without needing to special-case
either — a rendered-prototype check during conversion of this task (built the dashboard, ran
`visual-check.mjs`, applied this exact CSS change, rebuilt, re-ran `visual-check.mjs`, and viewed both
`workflow.png` and `workflow-run.png`) confirmed: the structural-view nodes (`places-resolve`,
`places-enrich`, `places-enrich-with-llm`) are now vertically centered with even top/bottom padding,
and the run-scoped nodes (already fairly balanced) show no visible regression — still centered, no
overlap or clipping. `node dashboard/scripts/mobile-check.mjs` also stayed green with the change
applied (18/18 pages pass, 0 boundary-crossing elements) — the flex change doesn't interact with the
mobile `.dag-node { min-width: 0; }` override (`dashboard/app/globals.css:617`).

**Do NOT touch** the outer graph container/panel padding (`.dag`, `.dag-panel`, `.dag-flow-wrap`) —
that's the different, already-addressed target of T250/T259 (padding of the container that WRAPS the
whole DAG). This task is scoped strictly to the padding WITHIN each individual node box
(`.dag-node`/`.dag-node-name`/`.dag-node-status`).

Apply the fix in `dashboard/app/globals.css` (the `.dag-node` rule around line 235). No change to
`dashboard/app/components/DagFlow.tsx` is expected — this is a pure CSS fix — but check the rendered
result and adjust `.dag-node-name`/`.dag-node-status` margins if centering exposes any awkward
spacing between the two lines in the run-scoped case (e.g. the `.dag-node-status`'s `margin-top: 3px`
might want a small tweak once the block is centered rather than top-anchored — judge this from the
actual rendered screenshot, don't guess blind).
