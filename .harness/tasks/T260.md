## Do

**Follow-up to T238 (built but the owner marked it FAILED — the popup still uses a code font).** When
you click a service on the Services page, the **"Consumers of …"** modal shows the service name, the
workflow names, and the job names in a blocky monospace/code font instead of the site's theme font.
Root cause (confirmed): in `dashboard/app/services/page.tsx` the consumers modal renders these with
the **`.mono` class** — the service name (`<span className="mono">{name}</span>`, ~line 42), each
workflow link (`<a … className="mono">`, ~line 55), and each job link (`<a … className="mono">{j.job_name}</a>`,
~line 65). `.mono` deliberately maps to the fixed code/log face and is NOT overridden by the theme
font (per CLAUDE.md, `.mono`/log surfaces intentionally stay monospace), so these stay blocky
regardless of the Theme Picker.

Make **every text element inside the Consumers modal use the site's theme font** (the same
`--font-body`/`--font-display` the rest of the site uses, honouring whatever the Theme Picker
selects) — i.e. remove `.mono` from the service name, workflow names, and job names in that modal (and
any other text in it), replacing with normal theme-font text. Do NOT change `.mono` globally or other
pages' mono usage (the limit-input fields elsewhere on the Services page may legitimately stay mono);
scope the change to the consumers modal's display text only.

**Make the screenshot script actually capture this** (the owner explicitly asked): add a `serviceConsumers`
fixture to `dashboard/scripts/_dashboard-harness.mjs` (the `/api/services/:name/consumers` route is
currently NOT mocked — it falls through to the empty `{}` catch-all, so the popup renders empty) AND a
new viewport/modal `FLOW` that opens the Consumers modal on the Services page, so `visual-check.mjs`
produces a screenshot of the populated popup for verification.
