## Overview
Turn plex-space-saver into a safety-net against silent library data loss: after it computes the library's total on-disk size each run, compare that total to the PRIOR run's recorded total and fire ONE critical push alert if the library has shrunk by more than a configurable threshold. A stable or growing library sends nothing. This is a guard against another job (or a failing drive) silently deleting content without the owner noticing.

## Do
Extend the existing single-stage plex-space-saver scan so that, in addition to writing the biggest-first size breakdown, it detects a significant DROP in the library's total on-disk size run-over-run and alerts once per drop.

1. PERSIST A BASELINE. plex-space-saver does NOT currently keep any prior-size baseline — data/out/size-breakdown.json is overwritten each run and the work_items ledger is keyed by ISO week purely for idempotency. Add a small persisted baseline recording each run's total library size in bytes plus its ISO timestamp. Prefer a dedicated JSON file under the workflow's data/out/ (e.g. size-baseline.json, resolved in config.ts alongside breakdownOut) holding at least { totalBytes, at } for the last recorded run; write it at the END of every successful scan so the NEXT run has something to diff against. Keep the baseline read/write in lib.ts (pure-ish helpers, unit-testable) with the path in config.ts.

2. DIFF AND DECIDE. In scan.ts, after buildBreakdown yields this run's totalBytes: read the prior baseline (if any); compute the drop as (prior.totalBytes - current.totalBytes). Trigger the alert only when a prior baseline EXISTS and the drop exceeds the configured threshold. Default the threshold to a configurable percentage drop (env-overridable, e.g. PLEX_SIZE_DROP_PCT default 10 — read in config.ts); if the owner's answer in the paired question selects an absolute-GB or 'either' rule instead, implement that instead, still env-overridable. On the FIRST run (no baseline yet) send nothing — just seed the baseline. On a stable or GROWING library (current >= prior, or a drop under threshold) send nothing.

3. ALERT ONCE, HOUSE-STYLE. When the threshold is exceeded, send exactly ONE critical push via core/notifier's push() (high priority + a rotating_light/warning tag, consistent with how other workflows signal a critical condition). Guard against re-alerting for the same drop using the notify-once work_items ledger pattern (isWorkItemDone / markWorkItem, keyed by a stable drop-event key — e.g. the baseline timestamp being alerted against — mirroring missing-tv-seasons/stages/notify.ts) so a manual re-run the same week does not double-notify. Log verbosely: prior total, current total, absolute + percentage drop, the threshold, and whether an alert fired and why (per the repo's verbose-logging convention). Always still write the size breakdown as before — the drop check is additive and never replaces the report.

4. DO NOT opt these Plex reads into any service-layer response caching (the point is a fresh read every run); the current plex.service.ts declares no cacheKey/TTL, so no change is needed there — just confirm you are not adding one.

Update src/workflows/plex-space-saver/CLAUDE.md in the same change to document the new baseline file, the drop threshold env var, and the shrink-guard alert behaviour. Keep it a single-stage workflow (no new DAG edge, so no gate needed). THRESHOLD DECIDED BY OWNER (this overrides the percentage-default wording above): use an ABSOLUTE GB drop threshold, env PLEX_SIZE_DROP_GB, default 1 GB — the library should never meaningfully shrink, so alert on ANY drop beyond a ~1 GB fluctuation buffer. Do not implement the percentage rule.
