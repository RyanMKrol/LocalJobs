## Overview

Replace `plex-language-fix`'s current single `plex-language-scan` stage with four stages —
`plex-language-discover` → `plex-language-resolve` → `plex-language-evaluate` →
`plex-language-apply` — switching the workflow's idempotency model from "re-scan the whole library
fresh every run" to "process each file/episode exactly once, ever, then skip it forever" (the same
per-item work-ledger pattern `places`/`perfumes` already use), and using the new
`callService(..., { cacheKey })` option (from the task this depends on) to avoid a TMDB call per
episode.

## Do

This task assumes T452 has already removed `plex-language-no-track-flag`, so today's workflow is
`plex-language-scan` → `plex-language-apply`. Replace `plex-language-scan` with THREE new stages, and
rewrite `plex-language-apply`'s eligibility logic — ending with this DAG in
`src/workflows/plex-language-fix/plex-language-fix.workflow.ts`:

```
plex-language-discover → plex-language-resolve → plex-language-evaluate → plex-language-apply
```

(`resolve` depends on `discover`; `evaluate` depends on BOTH `discover` and `resolve`; `apply` depends
on `evaluate`.)

**Item key — the SAME key at every stage, no lineage args needed:**
`` `${itemRatingKey}::part${partId}` `` (this exact key shape is already used today by
`no-track-flag.ts`'s ledger key, deleted in T452 — reuse that shape).

**1. `plex-language-discover`** (new files: `stages/discover.ts`, `stages/discover.job.ts`,
`stages/discover.test.ts`) — root stage, read-only, no dependencies. Declares `inputKeys()` on its
`JobDefinition` (this workflow becomes limitable for the first time — return the current set of
`${itemRatingKey}::part${partId}` keys the ledger already knows about, following this repo's
`inputKeys()` convention used elsewhere, e.g. in `places`). Reuse the existing Plex-section-walking
logic currently in `scan.ts` (`fetchSections`, `fetchSectionItems`, `fetchAllLeaves`, from
`src/workflows/plex-language-fix/lib.ts` — unchanged) to enumerate every file (a movie, or an episode)
across the configured `PLEX_MOVIE_SECTION`/`PLEX_TV_SECTION`/optional `PLEX_DOWNLOADABLE_SECTION`
sections (same env vars, unchanged). For each file: if `isWorkItemDone('plex-language-discover', key)`
is already true, skip it (no-op, already known); otherwise call
`markWorkItem('plex-language-discover', key, 'success', { detail: { name: "<Show> — S04E12" or
"<Movie Title>", file: <path> } })`.

**2. `plex-language-resolve`** (new files: `stages/resolve.ts`, `stages/resolve.job.ts`,
`stages/resolve.test.ts`) — read-only, `dependsOn: ['plex-language-discover']`. For each file discover
recorded that this stage hasn't yet resolved (`isWorkItemDone('plex-language-resolve', key)` false):
resolve its show/movie's true original language + candidate languages via TMDB, routing the TMDB call
through `callService('tmdb', () => lookupLanguageDetail(tmdbId, type), { cacheKey:
\`tmdb-language:${type}:${tmdbId}\` })` (the `cacheKey` option added by T451) —
`lookupLanguageDetail` and `buildCandidateLanguages` already exist in `lib.ts`, reuse them unchanged.
Because the cache is keyed by `tmdbId` (not by file), every OTHER file belonging to the same show
resolved within the same 5-minute window reuses the cached TMDB response instead of making a redundant
call — but this stage still records its OWN per-file ledger row:
`markWorkItem('plex-language-resolve', key, 'success', { detail: { name, originalLanguage,
candidateLanguages } })`. Catch `QuotaExceededError` per-file (log + continue, matching `scan.ts`'s
existing quota-handling pattern) so a hit TMDB cap stops gracefully rather than failing the whole run.

**3. `plex-language-evaluate`** (new files: `stages/evaluate.ts`, `stages/evaluate.job.ts`,
`stages/evaluate.test.ts`) — read-only, `dependsOn: ['plex-language-discover', 'plex-language-resolve']`.
For each file not yet evaluated (`isWorkItemDone('plex-language-evaluate', key)` false): reuse the
existing `evaluatePart`/`pickAudioCandidate` logic from `lib.ts` (currently called inline inside
`scan.ts`'s per-episode loop) to compare the file's current Plex stream selection against `resolve`'s
recorded `candidateLanguages`, and decide a 2-value status: `'change'` or `'skip'` — this REPLACES the
current 3-value `FileStatus` (`'change' | 'already-correct' | 'no-match'` in
`src/workflows/plex-language-fix/types.ts`); collapse the old `'already-correct'` and `'no-match'`
outcomes into `'skip'` (no separate no-match handling — that behavior was intentionally dropped along
with `no-track-flag` in T452). Update `FileStatus` in `types.ts` to `'change' | 'skip'`. Record
`markWorkItem('plex-language-evaluate', key, 'success', { detail: { name, status: 'change' | 'skip',
proposedAudio, proposedSubtitle } })`.

**4. `plex-language-apply`** (rewrite existing `stages/apply.ts`/`.job.ts`/`.test.ts`) — mutating,
`dependsOn: ['plex-language-evaluate']` (was `['plex-language-scan']`). STOP reading the whole-library
`data/out/language-scan.json` file (delete that read path). Instead, for each file `evaluate` marked
`status: 'change'` that this stage hasn't yet applied (`isWorkItemDone('plex-language-apply', key)`
false): call the EXISTING `plexPutStreams` (`src/core/plex-client.ts`, unchanged) exactly as today,
then `markWorkItem('plex-language-apply', key, 'success', { detail: { name, audio: "<from label> →
<to label>", path: <appliedLogPath>, format: 'json' } })`. Files with `status: 'skip'`, or already
marked done by `plex-language-apply` from a prior run, are never touched — this is the PERMANENT
per-file idempotency: once a file is applied, it is NEVER automatically re-touched by a future run,
even if `evaluate` somehow re-flags it as `'change'` again (re-applying requires the operator to
manually "unstick" that `work_items` row via this framework's existing unstick mechanism —
`POST /api/stuck/unstick` — no new mechanism needed; do not build any automatic re-apply path). Keep
the existing Plex Butler backup-before-first-change (`triggerButlerBackup`) and the self-contained
per-run applied-changes log + `scripts/plex-language-undo.ts` exactly as they are today — only the
file-selection/eligibility logic changes (reading from `evaluate`'s ledger instead of the scan JSON
file), not the safety mechanism.

**Delete** `src/workflows/plex-language-fix/stages/scan.ts`, `stages/scan.job.ts`,
`stages/scan.test.ts`, and the `data/out/language-scan.json` write path (and the
`LanguageScanFile`/`ShowOrMovieEntry` types in `types.ts` if now unused — grep to confirm before
deleting).

**Update `src/workflows/plex-language-fix/contracts.ts`** for the new edges (`discover→resolve`,
`discover→evaluate`, `resolve→evaluate`, `evaluate→apply`) — every DAG edge needs a matching
`produces`/`consumes` `ArtifactContract` pair per this repo's non-negotiable gate-coverage rule
(`src/core/gate-coverage.test.ts`); a trivial `check(): { ok: true }` is an acceptable minimum where a
richer shape-check isn't warranted, but prefer checking that the recorded `detail` has the expected
shape (e.g. `evaluate`'s contract can check `detail.status` is `'change'` or `'skip'`) where cheap to
do.

**Update `src/workflows/plex-language-fix/CLAUDE.md`** to fully replace its description of the
workflow's DAG, idempotency model, and stages with the new 4-stage shape: the per-item-ledger
idempotency model (replacing "re-scan fresh every run"), the TMDB response cache usage via
`cacheKey`, and the fact that `no-track-flag` was removed (T452) with no replacement. If the root
`CLAUDE.md`'s workflow table one-line description of `plex-language-fix` (in the "Shipped example
workflows" table) is no longer accurate given the new stage names, update it too — check first, since
the description may still read correctly at a high level.

## Done when

- `plex-language-fix.workflow.ts`'s `jobs` array is exactly: `plex-language-discover` (no deps) →
  `plex-language-resolve` (deps: discover) → `plex-language-evaluate` (deps: discover, resolve) →
  `plex-language-apply` (deps: evaluate).
- A test simulating two consecutive runs confirms: on the SECOND run, a file already processed in the
  first run is NOT reprocessed by ANY of the four stages (assert via `isWorkItemDone`/ledger row counts
  staying unchanged, or via a call-counter on the injected Plex/TMDB fetchers not incrementing for
  already-done files).
- A test confirms `plex-language-resolve` calls TMDB (the injected fetcher) exactly ONCE for a show
  with multiple not-yet-resolved episodes processed in the same run (proving the `cacheKey` dedup
  works end to end), by asserting the injected TMDB fetcher's call count.
- A test confirms `plex-language-apply` never re-touches (never calls the injected `plexPutStreams`) a
  file already marked done by `plex-language-apply` from a prior run, even if
  `plex-language-evaluate`'s ledger row for that file is manually set back to `status: 'change'`.
- `src/core/gate-coverage.test.ts` passes (every new edge has a matching gate).
- `npx tsc --noEmit` and `npm test` both pass.
- `src/workflows/plex-language-fix/CLAUDE.md` accurately describes the new 4-stage DAG and idempotency
  model (no stale references to `plex-language-scan` or the old re-scan-every-run behavior remain in
  that file).
