## Overview
Three unguarded channels each let one localized failure kill the whole daemon (and, via launchd, group-kill every running child): (1) a cron callback that rejects becomes an unhandled rejection because croner defaults catch:false; (2) daemon.ts installs no global unhandledRejection/uncaughtException handlers; (3) synchronous store throws (SQLITE_BUSY) inside the executor's rl.on('line')/errRl.on('line') log-stream handlers are uncaught. Guard all three so one localized failure logs loudly and the daemon keeps running.

## Do
Three independent guards.

(1) src/core/scheduler.ts — scheduleWorkflow (~lines 36-41). The cron callback is an async function passed to new Cron(...); croner's default catch:false means a rejecting runWorkflow (or a throw before it) surfaces as an unhandledRejection. Wrap the callback body in a try/catch that logs the error loudly (console.error with the workflow name and the error) and swallows it so one workflow's fire failure can never take down the scheduler / daemon, OR equivalently pass croner's { catch: (err) => { ...log... } } option — pick the try/catch approach for explicit, testable logging. Keep the existing enabled-toggle check and the skipped-result log. rescheduleWorkflow reuses the same callback via scheduleWorkflow, so fixing it once covers both the initial and re-registered crons.

(2) src/daemon.ts — main() (~lines 31-40, alongside the existing SIGTERM/SIGINT handlers). Install last-resort global handlers: process.on('unhandledRejection', ...) and process.on('uncaughtException', ...). Each must log LOUDLY (console.error with a clear '[daemon] uncaught…' / '[daemon] unhandled rejection…' prefix, the reason/error, and its stack). Make a DELIBERATE, documented decision about exit-vs-continue: an uncaughtException leaves the process in a possibly-corrupt state so it should log and then process.exit(1) (letting launchd restart a clean daemon — the safer choice), whereas an unhandledRejection is usually a localized async fault and should log-and-continue (do NOT exit) so a single bad promise doesn't kill everything. Put a short comment explaining the chosen policy for each. This is the true safety net for anything the per-channel guards miss.

(3) src/core/executor.ts — the rl.on('line') and errRl.on('line') handlers (~lines 161-189). These call addLog / setProgress (SQLite writes) directly; a synchronous store throw (e.g. SQLITE_BUSY under contention) inside the handler is uncaught and crashes the daemon. Wrap the store-touching body of each line handler in a try/catch so a failed log/progress write drops at most that ONE line (log the failure to console.error as a fallback — being careful the fallback itself can't throw) instead of killing the process. The child keeps running; only that single log/progress line is lost.
