## Overview
Raised during the split-movie-workflows clarification: today every DAG is strictly per-workflow, and the established precedent for splitting a combined workflow (tv-recommendations/missing-tv-seasons, and now movie-recommendations/missing-movies) is to DUPLICATE a shared upstream job (e.g. the Plex snapshot fetch) rather than share it across two independent workflow manifests. This is cheap today because Plex/TMDB reads are free/unmetered, but the pattern would recur for any future workflow split and could matter more for a paid/rate-limited upstream. This is a standalone research/decision task, independent of and non-blocking for the movie/missing-movies split, which proceeds with duplication regardless of this evaluation's outcome.

## Do
Survey every place in the codebase where two or more workflows currently duplicate a job/stage that reads the same underlying data (at minimum: tv-snapshot vs plex-tv-snapshot for TV; movie-snapshot vs missing-movies' new snapshot job once split-movie-workflows-backend lands). Assess whether a cross-workflow job-sharing mechanism is worth building — e.g. a job that multiple `*.workflow.ts` manifests can declare as a member, with the executor recognizing 'already run this cycle by another workflow, reuse the output' instead of re-running it — versus just continuing the duplicate-job convention indefinitely. Weigh: how much duplication actually costs today (mostly free Plex/TMDB reads — low), how much complexity a sharing mechanism would add to the DAG/scheduler/executor model (jobs are currently owned by exactly one workflow; a shared job breaks that assumption in ways that touch `src/core/dag.ts`, `src/core/workflow-executor.ts`, and the registry's orphan-job check), and whether any UPCOMING or hypothetical future workflow split would involve a genuinely expensive or rate-limited upstream (unlike Plex/TMDB) where duplication would actually hurt. Record a recommendation: build it, don't build it (and why), or revisit if/when a concrete expensive-duplication case arises.

## Done when
A recorded decision (in the task worklog) on whether to build a cross-workflow job-sharing mechanism, backed by a survey of current duplication instances and an explicit cost/complexity assessment — either declining for now with a clear reason to revisit later, or scoping out what a follow-up buildable task would need to change in the executor/DAG/registry. No code changes in this task.
