## Overview

Add a new, simple `local-jobs` workflow, `vercel-daily-redeploy`, that once a day POSTs to a Vercel
Deploy Hook for the separate `ryankrol.co.uk` repo — a safety net so the live site is never more than
24h behind the latest pushed commit, even if that repo's autonomous build harness pushed enough
commits in a day to exceed Vercel's per-commit auto-deploy rate limits and some deploys got dropped.

## Do

**Context (do not re-research — already confirmed against Vercel's official docs by the idea's
author):** Vercel Deploy Hooks (Project Settings → Git → Deploy Hooks in the Vercel dashboard) are a
unique, unauthenticated URL bound to one branch; a plain `POST` with no auth header and no body
re-triggers a full build of that branch's latest commit. This is a documented, Vercel-sanctioned
mechanism for exactly this "scheduled/cron redeploy" use case (see
https://vercel.com/docs/deploy-hooks). No Vercel API token or CLI is needed — just the hook URL,
which the owner will provision manually (that step is a SEPARATE, `needs-human`-gated task — see the
note at the end of this Do section; do NOT block this task on it).

1. Create a new workflow folder `src/jobs/vercel-daily-redeploy/` (this is a PUBLIC example-style
   workflow, not a private one — it doesn't touch any of the owner's private data, only triggers a
   redeploy of the public `ryankrol.co.uk` site — so do NOT add it to `.gitignore`; follow the
   standard tracked-workflow layout used by `claude-warmer`, `stock-digest`, etc.).
2. `src/jobs/vercel-daily-redeploy/vercel-redeploy.job.ts` — a single `JobDefinition` (see root
   `CLAUDE.md`'s "How to add a job" section and `src/jobs/claude-warmer/claude-warm.job.ts` as the
   structural template — this job is comparably simple: one HTTP call, defensive error handling, no
   retries needed since tomorrow's run will just try again):
   - Reads the deploy hook URL from `process.env.VERCEL_DEPLOY_HOOK_URL`.
   - **Soft-skip when unset**: if the env var is empty/undefined, `ctx.log(...)` a clear message
     (e.g. `"VERCEL_DEPLOY_HOOK_URL not configured — skipping redeploy trigger (see .env.example)"`,
     level `'warn'`) and return cleanly — do NOT throw. This mirrors how `FINNHUB_API_KEY` being unset
     soft-skips `stock-digest`'s `stock-sector-lookup` stage (documented in root `CLAUDE.md`'s
     `stock-digest` section) — an optional/not-yet-provisioned credential is never a hard failure.
   - When set, `POST` to the URL with no auth header and no body (Vercel Deploy Hooks require
     neither — do not add any). Use the platform's native `fetch` (already used elsewhere in this
     codebase for HTTP calls to services — check e.g. `src/services/*.service.ts` for the existing
     idiom rather than adding a new HTTP library dependency).
   - Log verbosely per this repo's logging convention (root `CLAUDE.md`'s "Logging — be verbose by
     default" section): log that the redeploy is about to be triggered (masking or omitting the full
     hook URL from logs is NOT required — this codebase already logs full paths/config elsewhere, but
     do NOT log it as a "secret was exposed" concern; it's fine to log the hostname/prefix only if you
     prefer extra caution, your call), the HTTP response status, and a clear success/failure summary
     line.
   - Treat a non-2xx response as a job failure (`throw`) so it shows up as a failed run in the
     dashboard — this is a safety-net job; a silently-swallowed failure would defeat its whole
     purpose. A network error (fetch throws) should also propagate as a failure, not be swallowed.
   - This job does NOT need the `work_items` ledger / idempotency machinery — it's a pure
     fire-and-forget trigger with no "items" to track, exactly like `claude-warm.job.ts`. Keep
     `timeoutMs` short (e.g. `30_000` — a Deploy Hook POST just enqueues a build, it doesn't wait for
     the build itself) and `maxRetries: 0` or a small number (your call — a failed POST due to a
     transient network blip retrying once is reasonable, but don't over-engineer this).
3. `src/jobs/vercel-daily-redeploy/vercel-daily-redeploy.workflow.ts` — a one-stage
   `WorkflowDefinition` (mirror `src/jobs/claude-warmer/claude-warmer.workflow.ts`'s shape exactly):
   `name: 'vercel-daily-redeploy'`, a `description`, `category: 'regular-maintenance'` (matches the
   vocabulary used by `workouts-sync`/`listening-digest`/`projects-sync`/`claude-warmer`/`stocks-sync`
   per root `CLAUDE.md`'s category note), `schedule: '0 23 * * *'` (23:00 daily — deliberately late in
   the day, after a typical day's harness push activity on `ryankrol.co.uk`, same reasoning
   `stock-digest`'s schedule uses relative to `stocks-sync`), and `jobs: [{ job: 'vercel-redeploy' }]`.
4. Add a `vercel-daily-redeploy-1.test.ts`-style unit test (or name it to match this repo's
   `<stage>.test.ts` convention) that mocks the HTTP POST (do NOT make a real network call in tests)
   and covers: (a) env var unset → job returns cleanly without throwing and without attempting a
   fetch, (b) env var set + mocked 2xx response → job succeeds, logs a success line, (c) env var set +
   mocked non-2xx response → job throws, (d) mocked fetch rejection (network error) → job throws.
5. Document the new env var in `.env.example` (names/placeholders only, never a real value — this
   repo's hard secrets rule): add `VERCEL_DEPLOY_HOOK_URL=` near the other per-workflow optional
   integration creds (e.g. alongside the `HEVY_API_KEY`/`FINNHUB_API_KEY` block), with a one/two-line
   comment explaining what it is, that it's optional (unset = the job soft-skips), and where to get it
   (Vercel dashboard → the `ryankrol.co.uk` project → Settings → Git → Deploy Hooks). Mention that
   provisioning the actual hook URL is tracked as a separate task (reference it as "the
   vercel-daily-redeploy-2 needs-human task" in the comment if you like, or just "see the workflow's
   needs-human companion task" — the exact wording is your call).
6. **Update docs in the SAME change (this repo's Definition of Done — root `CLAUDE.md`):**
   - Add a new bullet for `vercel-daily-redeploy` to `README.md`'s workflow list (the section
     containing the `claude-warmer`/`stocks-sync`/`projects-sync` bullets, roughly README.md:150-235) —
     mirror the existing bullets' style: what it does, why (the Vercel rate-limit safety-net
     rationale), its schedule, and that `VERCEL_DEPLOY_HOOK_URL` is optional/soft-skips when unset.
   - Add a corresponding paragraph to root `CLAUDE.md`'s "What this project is" section describing the
     new workflow (mirror the existing per-workflow paragraphs' density and style — see the
     `claude-warmer`/`stock-digest` paragraphs as the closest structural analogues: short, explains the
     WHY, the schedule, the credential + its soft-skip behavior).
   - Add `vercel-daily-redeploy` to the file-map table's job-folder list
     (`src/jobs/<workflow>/` row) alongside the other workflow folder names.
7. Do NOT attempt to actually create the Vercel Deploy Hook or provision `VERCEL_DEPLOY_HOOK_URL` in
   `.env` yourself — that requires the owner's own Vercel account access and is intentionally a
   separate `needs-human` task (see that task's spec for what the owner needs to do). This task is
   complete once the workflow soft-skips cleanly with the env var unset and is verified to correctly
   POST when a URL IS present (mock this in the test — do not fabricate a real hook URL).
