## Overview

`projects-sync` has two external-call sites that aren't properly service-gated, found during a
repo-wide rate-limiting audit: (1) GitHub REST API pagination is metered with ONE `callService`
call wrapping the entire multi-page fetch, not once per page fetched; (2) `git clone`/`git fetch`
operations against GitHub make zero external calls through any service at all — completely
unmetered. Both should route through the existing `github` service (same external host/budget,
no need for a second service name).

## Context (confirmed current)

`src/workflows/projects-sync/stages/github-sync.ts`:
- `fetchAllRepos` (lines 54-80) paginates GitHub's `/users/:username/repos` endpoint in a
  `while (true)` loop (lines 67-77), issuing one raw `fetch(url, { headers })` per page (line 69).
- `runGithubSync` (line 130) wraps the ENTIRE `fetchAllRepos` call in one `callService('github', ...)`
  (line 142: `const fetchRepos = opts.fetchRepos ?? ((u, t) => callService('github', () =>
  fetchAllRepos(u, t)));`). So however many pages GitHub returns, only ONE rate/quota reservation is
  made — page 2+ (if the owner's repo count ever exceeds 100) would fetch with zero additional
  rate-limit check or quota increment. Not currently triggered (repo count is under 100), but
  structurally wrong.

`src/workflows/projects-sync/stages/project-summarize.ts`:
- `runGit` (lines 60-70) spawns `git` as a child process (clone/fetch/reset).
- `cloneOrPullRepo` (lines 77-85) calls `runGit(['fetch', ...])` + `runGit(['reset', ...])` for an
  existing checkout, or `runGit(['clone', ...])` for a fresh one — genuine external network
  operations against GitHub (git-over-HTTPS), once per changed repo per run.
- `runProjectSummarize`'s injectable `cloneOrPull` (line 179/188) defaults to `cloneOrPullRepo`
  directly (line 224: `await cloneOrPull(entry.url, dest);`) — no `callService` wrapper anywhere in
  this file; it doesn't even import `callService`.
- `src/services/github.service.ts` (`name: 'github'`) already sets `ratePerMinute: 30` plus
  `dailyCap`/`monthlyCap` — a real, working rate/quota governor, just not applied to these two call
  sites correctly.

### 1. `src/workflows/projects-sync/stages/github-sync.ts` — fix pagination granularity

Move the `callService('github', ...)` wrap from around the whole `fetchAllRepos` call to around each
individual page fetch, inside the `while (true)` loop. Concretely: change `fetchAllRepos` itself (or
the `fetchRepos` default in `runGithubSync`) so that the `fetch(url, { headers })` call on line 69 is
made inside `callService('github', () => fetch(url, { headers }))`, called once per loop iteration —
not once for the whole function. Keep `ReposFetcher`'s existing signature/injectability for tests
(the default implementation changes internally; the injected-stub shape used by tests should stay the
same, or be minimally adjusted — your call, whichever keeps the tests clearest).

### 2. `src/workflows/projects-sync/stages/project-summarize.ts` — gate git operations through `github`

Wrap `cloneOrPullRepo`'s git operations in `callService('github', ...)` — reuse the SAME `github`
service (same external provider/budget as the REST API calls in `github-sync.ts`), rather than
inventing a new service name; a `git clone`/`git fetch` against GitHub consumes the same underlying
"how hard are we hitting GitHub" budget conceptually, and one shared cap is simpler than tracking two.
Wrap each `runGit` invocation individually (or wrap the whole `cloneOrPullRepo` call once per repo,
per your judgment — either is a single external "unit of work" per repo per run, so either granularity
is defensible; just make sure it's INSIDE the main per-repo loop in `runProjectSummarize`, not once for
the whole run). Import `callService` from `../../../core/services.js` in this file.
