## Overview

On the dashboard's Overview page, whichever workflow card happens to be the very
FIRST card in the "Workflows" grid renders visibly taller than its row-mates — for
the owner's real dashboard that's `claude-warmer` (alphabetically first among the
current workflow names). This is a pure CSS side effect: a `.panel + .panel`
sibling-margin rule meant for stacked, block-flow panel lists elsewhere in the app
is unintentionally also matching inside the Overview's `.grid.cards` grid, and
grid's default stretch alignment then compresses every OTHER card to compensate.
It has nothing to do with `claude-warmer`'s own content (cron string, name length,
or stage count).

## Do

- **Root cause (confirmed live, via a real Playwright render against the hermetic
  dashboard harness — not just reading the JSX):**
  - `dashboard/app/page.tsx`'s Overview "Workflows" grid (`<div className="grid
    cards">` around line 172, `visibleWorkflows.map(...)` around line 178) renders
    every card as a sibling `<div className="panel" style={{ padding: 16 }}>` —
    confirmed the JSX/outerHTML is BYTE-IDENTICAL between any two cards (only the
    text content differs); there is no conditional block that gives one card more
    content than another.
  - `dashboard/app/globals.css` has `.panel + .panel { margin-top: 28px; }` (around
    line 89) — an adjacent-sibling rule that exists to space out STACKED panels in
    ordinary block-flow layouts (e.g. the category-grouped panel list on the
    `/workflows` and `/integrations` pages, where each category is its own
    `<div className="panel">` stacked vertically). CSS's adjacent-sibling
    combinator only cares about DOM adjacency, not layout mode — so it ALSO matches
    every non-first `.panel` inside `.grid.cards` (`.cards { grid-template-columns:
    repeat(auto-fill, minmax(220px, 1fr)); }`, around line 98), which is a `display:
    grid` container, not block flow.
  - `.grid` has no explicit `align-items`, so the default `normal` applies, which
    computes to `stretch` for grid items. This means each row's TRACK height is
    sized to the tallest item's MARGIN box, and then every item in that row is
    stretched to fill that track height. The very FIRST `.panel` card in the whole
    list has no preceding `.panel` sibling, so it gets NO top margin — its full,
    un-eaten-into content height sets the row's height. Every OTHER card in the
    list (including every later row, since DOM adjacency doesn't reset per grid
    row) DOES have a preceding `.panel` sibling, so it gets the 28px top margin;
    when stretched to the same row height, that margin eats into its own content
    box, compressing it by 28px and pushing it down within its cell.
  - **Verified empirically:** two cards with entirely different names but
    otherwise-default fixture data rendered at 182px vs 154px tall, with
    IDENTICAL outerHTML — a 28px difference, exactly the `.panel + .panel`
    margin-top value. Their bottoms aligned exactly (both `top + height` summed to
    the same value), confirming the shorter card was being stretched-and-compressed,
    not independently shorter. Reordering the fixture array so a
    `claude-warmer`-shaped entry (single stage, `schedule: '*/30 * * * *'`) was
    FIRST in the list reproduced the exact reported symptom: it became the 182px
    "tall" card while its row-mates compressed to 154px. This confirms the bug is
    about LIST POSITION, not `claude-warmer`'s specific content.
  - **Ruled out:** the `*/30 * * * *` cron string wrapping onto a second line. Test
    render at viewport widths from 700px to 1440px (down to ~220-230px actual card
    width, i.e. the grid's own minimum) never wrapped the cron text — the
    `.cron-badge` span has `white-space: nowrap`, and even the full `"1 stages ·
    */30 * * * * ?"` line fits on one line at the grid's narrowest column width.
- **Fix:** neutralize the leaked block-flow margin specifically inside the
  Overview's card grid, since `.grid { gap: 14px; }` already provides the correct
  inter-item spacing there (the sibling-margin rule is redundant, and wrong, in a
  grid context). Add a scoped override near the existing `.panel + .panel` rule
  and the `.grid`/`.cards` rules in `dashboard/app/globals.css`, e.g.:
  ```css
  /* .grid layouts already space items via `gap` — the block-flow sibling margin
     above is for stacked panel lists (e.g. the category-grouped Workflows/
     Integrations pages) and must not leak into a grid, where it combines with
     grid's default stretch alignment to make whichever card is literally first
     in the DOM list render taller than every other card in its row. */
  .grid.cards > .panel + .panel { margin-top: 0; }
  ```
  Confirm `dashboard/app/workflows/page.tsx` and `dashboard/app/integrations/page.tsx`
  do NOT use `.grid.cards` for their category-grouped panel lists (they don't — they
  stack bare `.panel` divs in block flow) so this scoped override cannot affect
  their intentional 28px category spacing. Exact selector wording is up to you as
  long as the net effect is: cards inside `.grid.cards` are never given an
  unintended top margin, and the 28px stacked-panel spacing elsewhere is
  unaffected.
- **Fixture (required, per the LIVING ARTIFACT rule — this workflow doesn't exist
  in the hermetic harness yet):** add a `claude-warmer`-shaped entry to the
  `/api/workflows` fixture in `dashboard/scripts/_dashboard-harness.mjs` (a single-
  job workflow, e.g. `workflow({ name: 'claude-warmer', category:
  'regular-maintenance', schedule: '*/30 * * * *', jobs: [{ job_name:
  'claude-warm', depends_on: [] }] })`), inserted as the FIRST entry in the array
  returned for `/api/workflows` — this is what actually exposes the bug in the
  `overview` baseline screenshot (a card elsewhere in the array would just show the
  general mechanism, not the specific real-world case). This is fixture DATA, not a
  new interactive state, so it's a `PAGES`-level baseline change, not a new `FLOWS`
  entry.
