## Overview
Every 'Started'-style run timestamp in the dashboard currently shows a relative, referential time ('2h ago', '1d ago'). The owner wants these changed to an absolute date+time everywhere — no relative time at all for this column.

## Do
In dashboard/app/ui.tsx, add a new exported formatter, `fmtAbsolute(t: string | null): string`, placed next to the existing `fmtTime`/`fmtRelative` (around lines 73-92). It must follow the SAME UTC-parsing convention as `fmtTime`/`fmtRelative` (SQLite datetime() is UTC without a trailing 'Z' — reuse the same `t.includes('Z') || t.includes('T') ? t : t.replace(' ', 'T') + 'Z'` parse guard) and return '—' for a null/falsy input. OWNER-CONFIRMED FORMAT: a compact, FIXED (not browser/locale-dependent) style — day + abbreviated month name + full year + 24-hour time, e.g. '15 Jul 2026, 16:00'. Build this with explicit `Intl.DateTimeFormat` options (day: 'numeric', month: 'short', year: 'numeric', hour: '2-digit', minute: '2-digit', hour12: false) rather than a bare `d.toLocaleString()` call, so the output is consistent regardless of which browser/OS locale renders it (the owner explicitly rejected reusing the existing `fmtTime`, which IS a bare `toLocaleString()` and would vary by locale, e.g. '7/15/2026, 4:00:00 PM' in en-US). Do NOT remove or repurpose `fmtTime`/`fmtRelative` — they may still be used elsewhere (e.g. `fmtTime` for 'Next run', T208's `human-done` overlay dates, etc.); only replace call sites that specifically render a 'Started' / run-start column. Then, in each of these exact call sites, replace `fmtRelative(...)` with `fmtAbsolute(...)`: dashboard/app/page.tsx (Overview page, 'Recent workflow runs' table, 'Started' column, currently `fmtRelative(r.started_at)`); dashboard/app/workflows/page.tsx (Workflows list page, inline last-run-time span next to each row's status badge, currently `fmtRelative(p.last_run.started_at)`); dashboard/app/workflows/[name]/page.tsx ('Runs' table, 'Started' column, currently `fmtRelative(r.started_at)`); dashboard/app/workflow-runs/[id]/page.tsx (BOTH the 'Member runs' table's 'When' column AND the nested earlier-attempts sub-rows, both currently `fmtRelative(r.started_at)` / `fmtRelative(e.started_at)`); dashboard/app/jobs/[name]/page.tsx ('Run history' table, 'Started' column, currently `fmtRelative(r.started_at)`). Re-grep for `fmtRelative(` across `dashboard/app/**` before finishing to confirm every Started/When/last-run-time call site was caught and none were missed due to line-number drift from this spec. If a UI-surface change requires it, also update `dashboard/scripts/_dashboard-harness.mjs` (the PAGES/FLOWS living-artifact fixture list) per the repo's mobile/visual-check convention if the change alters what a screenshot would show, and update the relevant workflow-doc mention in CLAUDE.md's 'Gotchas'/UI conventions section only if the timestamp-format convention needs recording there (optional — judge whether it rises to a documented convention; a plain formatter swap likely does not need a new CLAUDE.md rule beyond noting the new fmtAbsolute helper if a future author would otherwise duplicate it).
