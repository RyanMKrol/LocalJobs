## Overview

Add a "Run all workflows" action to the existing Admin page, so the owner can kick off a manual
"run everything and see what happens" sweep (each workflow started with a 3-originating-input limit
where it supports limiting) from one button, consuming the bulk `POST /api/workflows/run-all` endpoint
(and its `api.runAllWorkflows(limit?)` client helper) added in the dependency task.

## Do

Add a new section to `dashboard/app/admin/page.tsx` (which currently has exactly one section, the
"Delete all workflow output" danger zone — see `dashboard/app/admin/page.tsx:39-122`), for the new
fleet-wide RUN action. This is a SEPARATE, additive section — do not remove or restructure the
existing delete-all section.

1. **Section placement and framing.** Add a new section (e.g. `<h2>Run everything</h2>` above or below
   the existing "Danger zone" heading, whichever reads better) explaining what it does in plain
   language: it starts a manual run of EVERY workflow at once, each capped to 3 originating inputs
   where the workflow supports a limit (workflows with no limitable input just run normally/unlimited),
   mirroring what clicking "Run now" on every workflow's own page would do — this is NOT destructive
   (unlike the delete-all section below it — no data is deleted), but it DOES trigger real work,
   including workflows that make paid/metered API calls (those calls are still governed by each
   service's own spend cap, same as any other manual run — do not build any new spend-limiting
   mechanism here).
2. **Confirmation — an inline confirm panel, not a native `confirm()` and not a typed-phrase input.**
   This action is not destructive the way "Delete all workflow output" is (nothing is lost — worst
   case is wasted API quota, already governed by existing per-service caps), so the heavier
   type-to-confirm pattern used for the delete-all section is NOT required here — but a bare native
   `window.confirm()` should also be avoided so the confirmation step is actually visible in a
   screenshot (native dialogs cannot be captured this way). Implement a simple **inline two-step
   confirm**: clicking the primary "Run all workflows" button reveals an inline confirmation panel (not
   a native dialog) stating something like "This will start a manual run of every workflow (limit 3
   inputs where supported). Continue?" with two buttons, "Confirm" and "Cancel" — Cancel collapses the
   panel back with no call made; Confirm calls the API and collapses/replaces the panel with the busy
   state.
3. **Optional limit input.** Include a small number input (default `3`, matching the API's default) so
   the owner can override the per-workflow limit before confirming, but keep it optional/simple — no
   need for per-workflow overrides, just one shared limit value passed to `api.runAllWorkflows(limit)`.
4. **Calling the endpoint.** On confirmed click, call `api.runAllWorkflows(limit)` (the dependency
   task's client helper), show a busy state on the button (disable it, label change, mirroring the
   existing delete-all section's `busy`/`result`/`err` state pattern in the same file), and on success
   render the per-workflow `results[]` returned — a small table showing workflow name, status
   (`started`/`skipped`), and either the limit used (or "unlimited" when `limited` is false) or the skip
   reason — plus the aggregate `startedCount`/`skippedCount`/`totalWorkflows`. On error, show the error
   message inline (no silent failure). Note in the UI copy that "started" only means the run was
   DISPATCHED, not that it has finished — link or point the owner to the Workflows page to watch
   progress (a plain sentence is enough, no need to build live status polling into this page).
5. **Update the living-artifact test harness** `dashboard/scripts/_dashboard-harness.mjs` in the SAME
   change (non-negotiable per repo convention): add a synthetic fixture response for `POST
   /api/workflows/run-all` (a realistic `{ ok: true, totalWorkflows, startedCount, skippedCount, limit,
   results: [...] }` payload covering both a started-limited, a started-unlimited, and a skipped
   workflow, so the results table has something to render) to the fixture dispatcher (see how
   `/api/workflows/reset-output-all` is already stubbed nearby, added in the T322/T323 work); and add a
   `FLOWS` entry (e.g. `admin-run-all-confirm`) that navigates to `/admin`, clicks the "Run all
   workflows" button to reveal the inline confirm panel, and screenshots that revealed state — so the
   confirmation UI is actually visible in a captured screenshot, per the `layer:ui` floor rule in
   `.harness/CLAUDE.md` (any interactive state only reachable via a click needs a `FLOWS` entry, not just
   a baseline `PAGES` shot).
6. Ensure the new section is mobile-responsive per the repo's `@media (max-width: 640px)` convention in
   `dashboard/app/globals.css` — no horizontal overflow, reuse `.panel:has(> table)` for the results
   table scroll if needed (same as the existing delete-all results table on this page).
7. Reuse existing shared components/classes from `dashboard/app/ui.tsx` / `dashboard/app/components/`
   per the repo's "reuse, don't re-implement" convention (e.g. `<RunButton>` styling idioms, `<Pill>` for
   the started/skipped status chips in the results table, matching how the delete-all section already
   uses `pill on`/`pill off`).
