## Overview

A one-off, human-driven review pass: read the past ~7 days of `local-jobs` PRODUCT runtime logs
(the dashboard's Logs page / `run_logs`) for recurring or notable errors, and author well-specified
follow-up backlog tasks for anything genuinely actionable. This task is itself a live test case of
the harness's new "tasks touching `.harness/**` must be `needs-human`" convention (see the paired
task in this same conversion cluster): because carrying this out means authoring new
`TASKS.json`/`.harness/tasks/**` entries, this task is correctly gated `needs-human` rather than
buildable.

## Do

- **Scope of "logs" — PRODUCT runtime logs, not harness build logs.** Review is limited to the
  dashboard's Logs page (`dashboard/app/logs/page.tsx`), backed by `GET /api/logs` →
  `listGlobalLogs()` in `src/db/store.ts` — i.e. the `run_logs` table, covering every job/workflow
  run, over the past ~7 days (a rolling window measured from whenever this task is actually picked
  up, not a fixed calendar date). This explicitly EXCLUDES the harness's own build-time logs
  (`.harness/worklog/*.md` per-task worklogs, `.harness/ledgers/outcomes.jsonl`/`failures.jsonl`) —
  those describe the autonomous loop's OWN task-building attempts, a different concern from product
  runtime health, and already have their own review surfaces (`local-jobs-review-failed`, the
  "Known-but-deferred issues" log in `.harness/CLAUDE.md`). This is a judgment call: the owner said
  "logs" without qualifying which; "errors" + "tasks to address those errors" reads most naturally
  as product defects, so product logs are the default scope here. If, having done the product-log
  review, the harness's own logs seem worth a similar pass too, note that as a possible follow-up
  idea rather than silently expanding this task's scope mid-review.

- **One-off, not recurring.** The owner described this as wanting it "at some point in the near
  future," not on a schedule — treat this as a SINGLE ad-hoc review pass, not a new cron'd product
  workflow. **Note the overlap with T062** ("Daily self-inspection workflow: review run logs →
  propose/fix framework issues", already pending in the backlog): T062 is a materially LARGER,
  separate idea — a new automated, scheduled `local-jobs` workflow where Claude runs headlessly every
  day and, per its own spec, "proposes and, where safe, applies" framework fixes directly. This task
  is deliberately narrower and lower-ceremony: a single manual read-and-triage pass, with a human in
  the loop throughout, that results ONLY in newly-authored backlog tasks — never a direct framework
  edit, and never a new scheduled workflow. Do not build any automation as part of this task; if a
  recurring cadence or the automated-fix-proposal idea (T062) is wanted, that stays T062's separate
  scope — don't merge the two.

- **Review procedure:**
  1. Open the dashboard's Logs page (or query `GET /api/logs` directly) and scan entries from the
     last ~7 days across all jobs/workflows.
  2. Note every `error`-level entry, and any `warn`-level entry that recurs across multiple runs (a
     single transient warning is not automatically actionable; the same warning repeating across runs
     likely is).
  3. For each candidate, judge whether it reflects a genuine product/framework defect worth fixing
     (a bug, a misconfigured service limit, a recurring external-API failure, an item that keeps
     getting stuck/re-surfacing) versus expected/benign noise (an already-known, accepted soft-fail;
     a quota warning that is working exactly as designed).
  4. For each genuinely actionable finding, author a properly-specified backlog task via the
     `ralph-loop-add-to-backlog` skill (capture it with `/idea` first if it needs more exploration
     before it's ready to shape). Do not hand-roll a `TASKS.json` entry directly — that bypasses the
     harness's own authoring conventions (facets, spec MD, scope, the chooser/review pairing rule,
     etc.).
  5. If nothing actionable turns up, that is a valid, complete outcome for this pass — record that
     conclusion rather than manufacturing busywork tasks just to have something to show.
- Record the review's outcome in this task's worklog: what time window was actually covered, what
  was found (or that nothing was), and which new task id(s) — if any — were authored as a result.

## Done when

- The past ~7 days of PRODUCT `run_logs` (across all jobs/workflows, via the Logs page / `GET
  /api/logs`) have been reviewed by a human, with Claude assisting, per the procedure above.
- Every genuinely actionable error/warning pattern found has a corresponding, properly-specified new
  backlog task (authored via `ralph-loop-add-to-backlog`, with real facets/scope/spec — this task
  itself does not expand scope to fix anything directly).
- The worklog records what window was reviewed, what was found, and which new task ids (if any)
  resulted — or explicitly records that nothing actionable was found.
- No new scheduled/automated workflow was built as a side effect of this task (that remains T062's
  separate, larger, already-tracked scope).
