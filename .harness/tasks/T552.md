## Overview
A latent reproduction of the already-shipped-once T098 crash-loop class (heavily documented in the root CLAUDE.md). Two tiny fixes in one unit: (1) two stale column-value comments in src/db/schema.sql drifted from what the code actually writes — runs.status omits 'skipped' (written by recordSkippedRun/setRunNoop in store.ts) and runs.trigger omits 'workflow' (written by member skip/gate rows); (2) the real hardening — src/db/index.ts caches three PRAGMA table_info results (svcCols line ~42, wfCols line ~75, jobCols line ~117) ONCE and keeps consulting the stale arrays after intervening ALTER TABLEs (wfCols from line 75 is still used at line 153). This is correct only because every migration today touches a distinct column name; the day two migrations touch the same column, the stale check produces a duplicate ALTER → throw → daemon crash-loop at startup, the exact T098 failure mode. Replacing the arrays with an always-fresh hasColumn(db,table,col) helper removes the sharp edge.
