## Do

Make a workflow's cron **schedule editable from its detail page** (`/workflows/<name>`),
so the owner can type a new cron expression to change when the workflow fires — without
editing code or restarting the daemon. Today the schedule comes only from the
`WorkflowDefinition` in code: the scheduler registers crons from `def.schedule` at
startup, and the `workflows.schedule` DB column is just a display mirror. Implement a
real, persisted, user-owned override that takes effect live. Mirror the EXISTING
user-owned-but-code-reconciled patterns already in this repo — the `enabled` toggle
(`setWorkflowEnabled`, preserved across `syncWorkflow`) and the editable service limits
(`updateServiceLimits` + the `limits_overridden` reconcile in the services upsert). Do
all of:

1. **DB + reconcile.** Add a `schedule_overridden INTEGER NOT NULL DEFAULT 0` column to
   the `workflows` table. Put it in `src/db/schema.sql`'s `CREATE TABLE` (fresh DBs) AND
   add an additive `ALTER TABLE` migration in `src/db/index.ts` for existing DBs (follow
   the T098 rule: do NOT reference the new column from a bootstrap index in schema.sql;
   any index goes inside the migration after the ALTER). Update `upsertWorkflowStmt` in
   `src/db/store.ts` so a code-sync PRESERVES the user's schedule once overridden — i.e.
   `schedule = CASE WHEN schedule_overridden = 1 THEN schedule ELSE excluded.schedule END`
   (exactly the reconcile `enabled`/service limits already get). Add
   `updateWorkflowSchedule(name, schedule: string | null)` to `store.ts` that sets
   `schedule` and flips `schedule_overridden = 1`; an empty/blank input clears it to
   `null` = manual-only. Surface `schedule_overridden` on the workflow row type.

2. **Scheduler reads the EFFECTIVE schedule + can reschedule live.** Change
   `src/core/scheduler.ts` so `startScheduler` registers each workflow's cron from its
   EFFECTIVE schedule — the DB `workflows.schedule` value (which, after the reconcile
   above, is the override when set else the code default) — rather than `def.schedule`
   directly. (The daemon syncs workflows to the DB before `startScheduler`, so the rows
   exist.) Export a `rescheduleWorkflow(name, schedule: string | null)` that: stops +
   removes the existing `Cron` from the `workflowCrons` map, and if the new schedule is
   non-null registers a fresh `Cron` (reusing the same fire callback — look the
   `WorkflowDefinition` up by name from the registry); if null, leaves it unregistered
   (manual-only). This makes an edit take effect WITHOUT a daemon restart, the same way
   the live `enabled` check does. Keep `nextWorkflowRun(name)` working (it reads the live
   cron) so the detail page's "Next run" updates after an edit.

3. **API.** Add `POST /api/workflows/:name/schedule { schedule }` to `src/api/server.ts`
   (a MUTATING endpoint — it MUST go through the same loopback/token guard as
   `/toggle`, `/run`, `/limits`). VALIDATE the cron expression server-side: an empty/blank
   value is allowed (→ manual-only, null); a non-empty value must be a valid croner
   pattern (validate by attempting `new Cron(expr, { paused: true })` and catching) — an
   invalid expression returns **400** with a clear error, never crashing the daemon. On
   success call `updateWorkflowSchedule` then `rescheduleWorkflow`, and return
   `{ ok: true, schedule, next_run }` (the recomputed next fire). Unknown workflow → 404.

4. **Dashboard.** On the workflow detail page (`dashboard/app/workflows/[name]/page.tsx`)
   make the Schedule row editable: an "Edit" affordance reveals a text input prefilled
   with the current cron (blank if manual-only), with Save / Cancel. Save calls a new
   `api.updateWorkflowSchedule(name, schedule)` client method (in
   `dashboard/app/lib/api.ts`), shows the server's 400 validation error inline on a bad
   expression, and on success refetches so the existing `CronBadge` and "Next run" update.
   Keep it understated and consistent with the existing toggle styling; any new CSS goes
   in `dashboard/app/globals.css`. Must stay phone-width responsive (no overflow).

5. **Docs + tests.** Update `README.md` (the workflow-detail-page description — the
   schedule is now editable, empty = manual-only, edits take effect live) and `CLAUDE.md`
   (the schedule is user-owned + reconciled across code-sync via `schedule_overridden`,
   exactly like `enabled`/service limits; the scheduler uses the effective schedule and
   reschedules live). Add unit tests for: the `syncWorkflow` reconcile preserving an
   overridden schedule across a re-sync; `updateWorkflowSchedule` (set + clear-to-null);
   the API validation (valid → 200 + next_run, invalid → 400, unknown → 404); and
   `rescheduleWorkflow` re-registering the cron with a new next-run. Ensure the existing
   `src/db/migrate-existing-db.test.ts` regression guard still passes with the new column.

## Done when

From a workflow's detail page the owner can enter a new cron expression and save it; the
change persists, is preserved across daemon restarts and code-syncs (via
`schedule_overridden`, like the `enabled` flag), and takes effect on the live scheduler
WITHOUT a restart — the "Next run" and `CronBadge` update to the new schedule, and the
workflow subsequently fires on the new cron. An empty value sets the workflow to
manual-only (no schedule); an invalid cron expression is rejected with a clear 400 and
never reaches the scheduler. `POST /api/workflows/:name/schedule` is guarded like the
other mutating endpoints (loopback/token). README + CLAUDE.md document the new
user-owned-schedule behaviour; unit tests cover the store reconcile, `updateWorkflowSchedule`,
the API validation paths, and live rescheduling; the migrate-existing-db regression guard
stays green; and `npx tsc --noEmit`, `npm test`, and `npm --prefix dashboard run build`
all pass.
