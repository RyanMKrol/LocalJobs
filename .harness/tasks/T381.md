## Overview

A human walkthrough of how `stocks-sync` and `stock-digest` each use Trading212, OpenFIGI, and
Finnhub ‚Äî prompted by the owner's own words, "I'm not entirely sure that the Trading 212, OpenFIGI,
or Finnhub APIs are being used correctly." Three recently-`done` tasks changed this exact surface in
one session ‚Äî T372 (new `openfigi.service.ts`), T373 (wire ISIN + `resolvedTicker` into
`stocks-snapshot.ts`), and T352 (fix `stock-sector-lookup.ts`'s Trading212‚ÜíFinnhub ticker-format bug)
‚Äî and none of it has been reviewed by the owner yet. This task walks through both workflows' relevant
code + real output side by side (they're covered together because `stocks-snapshot.ts` ‚Äî the file
T373 changed ‚Äî is the shared upstream input both workflows' entry points ultimately depend on), and
flags one concrete, evidence-backed question worth the owner's attention: whether Finnhub's
sector lookup should be using OpenFIGI's *resolved* ticker instead of Trading212's raw one for
renamed/rebranded companies. No code changes are expected from this task unless the owner's review
surfaces something to fix, in which case file that as a separate follow-up task.

This is distinct from **T302** ("Human review: stocks-sync workflow end-to-end", still pending) ‚Äî
T302 is a broader, open-ended review of the whole `stocks-sync` DAG (snapshot ‚Üí watch ‚Üí notify
gain-alerting), predating T352/T372/T373. This task is narrowly scoped to the Trading212 / OpenFIGI /
Finnhub data-sourcing question the owner just raised, with a concrete checklist and real evidence
already gathered below ‚Äî not a re-run of T302.

### Section A ‚Äî `stocks-sync`: Trading212 ‚Üí OpenFIGI ISIN/ticker resolution (T373)

This is the workflow named in idea #25 ("the other stock workflow") ‚Äî `src/jobs/stocks-sync/`,
3-stage DAG `stocks-snapshot ‚Üí stocks-watch ‚Üí stocks-notify`. All the OpenFIGI logic lives in stage 1.

1. Read `src/jobs/stocks-sync/stages/stocks-snapshot.ts` in full, focusing on:
   - `fetchPortfolio` (~L85) and `fetchInstrumentsMetadata` (~L108) ‚Äî confirm both are GET-only, use
     HTTP Basic auth (key id as username, secret as password), and are routed through
     `callService('trading212', ...)` (the read-only rule + the shared rate/quota meter).
   - `resolveTickers` (~L151) ‚Äî confirm the flow: Trading212 ticker ‚Üí ISIN (via the
     instruments-metadata lookup map) ‚Üí real-world ticker (via OpenFIGI). Confirm a miss at either
     step is soft (logs `warn:`, leaves `isin`/`resolvedTicker` undefined) and never throws.
   - `resolveOpenFigiTickersBatched` (~L187) ‚Äî confirm ISINs are batched (10/request without
     `OPENFIGI_API_KEY`, 100/request with one) rather than one HTTP call per ISIN, and routed through
     `callService('openfigi', ...)`.
   - `runStocksSnapshot`'s `try`/`catch` around the whole resolution block (~L290-302) ‚Äî confirm a
     failure fetching instruments-metadata itself doesn't fail the run; it just logs a `warn:` and
     continues without ISIN/ticker data for any position that run.
   - **A specific thing to check carefully:** `fetchInstrumentsMetadataFn` is called ONCE per run
     (~L293), using the *Invest* account's credentials (`apiKeyId`/`apiSecretKey`) ‚Äî never the ISA
     credentials, even when `hasIsaCreds` is true and ISA positions are present. Confirm from
     Trading212's own docs (or by comparing the instrument list against ISA holdings) whether
     `GET /equity/metadata/instruments` returns a **universal** catalog (in which case Invest-only
     credentials are fine for looking up ISA tickers too) or an **account-scoped** list (in which case
     an ISA-only ticker could silently fail to resolve an ISIN, purely because the call used the wrong
     account's key). The real portfolio.json below shows ISA-account ISINs resolving fine, which is
     evidence *for* "universal catalog" ‚Äî but worth confirming, not assuming.

### Section C ‚Äî the cross-cutting question (why these two are reviewed together)

6. Confirm the conceptual separation the owner's phrasing ("Trading212, OpenFIGI, or Finnhub... used
   correctly") blurs together: **OpenFIGI resolves a real-world ticker SYMBOL** (used in
   `stocks-sync`'s output only ‚Äî `stocks-watch`/`stocks-notify` never read `resolvedTicker`, and
   `stock-digest-build.ts`'s `buildFacts`/`HoldingSummary` never read it either, only the raw
   `ticker`), while **Finnhub resolves an industry/SECTOR classification** (used only by
   `stock-digest`'s Diversification section). They are two independent lookups off the same raw
   Trading212 ticker, feeding two unrelated purposes ‚Äî confirm this separation is correct and not
   confused anywhere in the code (it currently reads as correct; the risk was only in the assumption,
   not the code).

7. **The concrete question worth the owner's judgment call:** `stock-sector-lookup.ts` derives its
   Finnhub symbol from the RAW, stale Trading212 ticker (`toFinnhubSymbol("YNDX_US_EQ")` ‚Üí `"YNDX"`,
   Yandex's old/defunct symbol) ‚Äî it never looks at `resolvedTicker` (`"NBIS"`, Nebius Group's
   current, real symbol) even though `stocks-snapshot.ts` already resolved it one stage earlier via
   OpenFIGI. Given Finnhub's `profile2` endpoint returned nothing for `YNDX` (hence the `null` in
   `sectors.json`), it's plausible Finnhub *would* resolve an industry for `NBIS` if that's what was
   queried instead ‚Äî but this is unverified, not asserted as fact. Similarly for `BRK_B_US_EQ`:
   OpenFIGI resolved `BRK/B` (its own slash convention), which still likely isn't Finnhub's expected
   form either ‚Äî feeding Finnhub the OpenFIGI-resolved ticker might not fully fix that one. Decide:
   is it worth a follow-up task to try Finnhub lookups with `resolvedTicker` (falling back to the
   current `toFinnhubSymbol(ticker)` translation when no `resolvedTicker` exists), or is the current
   ~2-ticker gap in the Diversification section acceptable as-is?
