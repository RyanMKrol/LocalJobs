## Overview

Two stages in the `places` workflow record a success `detail` on `markWorkItem` that only restates
the item's identity, even though both stages compute/discover real values — violating the "every
stage's success `detail` must describe what THAT STAGE produced" convention (root `CLAUDE.md`
"Job conventions" section, mirrored in `src/workflows/CLAUDE.md`). Fix `places-ingest` and
`places-enrich`'s success `detail` to surface the values each stage actually computes, so the
dashboard's `StageIoPanel` (Inputs & Outputs) shows real evidence of work instead of a bare name.

## Do

- **`src/workflows/places/stages/ingest.ts:38`** (inside `recordIngestLedger`) — currently:
  ```ts
  if (p.cid && rootAllowed(p.cid)) markWorkItem(INGEST_JOB, p.cid, 'success', { detail: { name: p.name } });
  ```
  `ingest` computes, per place, which of the owner's saved Google Maps lists it belongs to
  (`p.lists: string[]`) and the canonical `?cid=` Maps URL it derived (`p.cidUrl: string | null`) —
  both real, stage-produced values (see `NormalizedPlace` in `src/workflows/places/types.ts:11-26`).
  Change the `detail` to:
  ```ts
  detail: { name: p.name, url: p.cidUrl, lists: p.lists }
  ```
  This mirrors the "stage discovers/computes a value, writes no file → put the value itself in
  `detail`" branch of the convention — `url` here is the same shape as `perfumes-find-url`'s
  `{ name, url }` worked example in the root CLAUDE.md, and `lists` shows list-membership (a
  count of 1 = only ever in one list; >1 = multi-list) which was previously only visible in logs.
  - `src/workflows/places/stages/ingest.test.ts` asserts on ledger counts and `isWorkItemDone` only — it
    does NOT assert on `detail` shape, so no test changes are required for this file, but re-run it
    after the change to confirm it still passes untouched.

- **`src/workflows/places/stages/enrich.ts:156`** (inside `runEnrich`, the success branch) — currently:
  ```ts
  markWorkItem(JOB_NAME, placeId, 'success', { attempts, rootKey: place.cid, parentKey: place.cid, parentJob: 'cid-to-place-id-resolver', detail: { cid: place.cid, name: dn } });
  ```
  `cid` here is just the item's own identity (its lineage `rootKey`, already recorded separately as
  a ledger column) — it adds nothing describing what the Google Places API call actually returned.
  The surrounding code (lines 154-165) already extracts real discovered values from the API response
  (`d`) that are logged but never recorded in `detail`: `d.rating`, `d.userRatingCount`, `ptype`
  (primary type display name), and `d.formattedAddress`. Change the `detail` to surface these instead
  of the redundant `cid`:
  ```ts
  markWorkItem(JOB_NAME, placeId, 'success', {
    attempts, rootKey: place.cid, parentKey: place.cid, parentJob: 'cid-to-place-id-resolver',
    detail: { name: dn, rating: d.rating ?? null, type: ptype, address: d.formattedAddress ?? null },
  });
  ```
  (`ptype` is already computed one line above the existing `markWorkItem` call at line 158 in the
  current file — reorder so it's computed before the `markWorkItem` call, or inline the same
  expression: `(d.primaryTypeDisplayName as { text?: string } | undefined)?.text ?? '—'`.)
  - There is no `enrich.test.ts` in `src/workflows/places/stages/` — no existing test asserts on this
    `detail` shape, so no test file needs updating. If you add a new test for `runEnrich`'s success
    detail as part of `expectsTest`, assert the new fields are present.

- **Do NOT touch `resolve.ts` or `llm-enrich.ts`** — both are already compliant:
  - `src/workflows/places/stages/resolve.ts:104-106` already records the discovered `placeId` in
    `detail` on success (`{ name: place.name, placeId: rec.placeId }`).
  - `src/workflows/places/stages/llm-enrich.ts:128` already references the written markdown artifact
    (`{ name, markdown: mdPath }`), matching the T110 pattern exactly.
  - Leave the `'skipped'`/soft-stop `markWorkItem` calls in `enrich.ts` and `llm-enrich.ts`
    (quota/rate-limit graceful-stop markers, e.g. `enrich.ts:127,139` and `llm-enrich.ts:136,146`)
    and the `'failed'` calls alone — they already explain *why* (status/error message) and are out
    of scope for this task, which is about SUCCESS `detail` only.
