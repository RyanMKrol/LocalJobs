## Overview
POST /api/workflows/:name/reset-output (and /reset-output-all) locate a workflow's data/out folder via a bespoke filesystem walk (findWorkflowDataOut / walkForWfFiles in src/api/server.ts) that recurses with NO exclusion of directories named 'data'. Because projects-sync shallow-clones whole repos (including THIS one) into src/workflows/projects-sync/data/repos/, full COPIES of every workflow manifest exist on disk. The walk (1) await-imports and thus EXECUTES those stale cloned manifests in the live daemon, and (2) returns on the FIRST name-match — the clone, which sorts before tv-recs/stocks-sync/etc. and has no data/out — so existsSync fails, the function returns null, and reset-output reports ok:true/filesRemoved:0 while the REAL data/out is never deleted AND the DB ledger is still wiped. This is the exact incident class the root CLAUDE.md 'Never let data/ folders be scanned for code' Gotchas section documents; the registry, tsconfig, and test runner were all fixed for it — this API walk was missed.

## Do
Fix the manifest-discovery walk in src/api/server.ts. Do ALL of the following, not just the first:

(a) In walkForWfFiles (inside findWorkflowDataOut, ~lines 110-120), SKIP any directory named exactly 'data' before recursing into it — mirror src/workflows/registry.ts's findFiles, which does `if (entry.name === 'data') continue;` (registry.ts line 53). This prevents descending into projects-sync/data/repos/ clone trees entirely, so no cloned manifest is ever discovered, imported, or executed in the daemon process.

(b) In findWorkflowDataOut (~lines 121-132), do NOT `return null` on the first name-match whose data/out is missing. Keep walking the remaining discovered files, and only return null after the FULL walk finds no name-match with an EXISTING data/out. (This is a belt-and-braces robustness fix so that any future generated tree — even one NOT under a directory literally named 'data' — cannot make the endpoint pick a candidate that lacks data/out over the real one. Return the FIRST match whose data/out actually exists.)

(c) OPTIONAL / preferred-if-cheap: the spec's preferred structural fix is to drop the bespoke walk and resolve the workflow's source directory from the registry's already-loaded definitions (which already excludes data/). The registry does NOT currently export per-workflow file paths, so wiring this would require changing src/workflows/registry.ts — which is OUT of this task's scope. Do NOT expand scope to do this; implement the in-server fix via (a)+(b)+(d). (Recorded only so a reviewer knows the structural option was considered and deliberately deferred.)

(d) While here, tighten deleteDataOutContents's data/out path guard (~line 143): it currently checks `outDir.includes(`${sep}data${sep}out`)`, which also matches paths like .../data/output-x. Require a trailing separator (or end-of-string) so it matches only a genuine .../data/out directory. (Unreachable with hostile input today — the candidate is always constructed as joinPath(dir, 'data', 'out') — but cheap correctness.)

Keep the existing behaviour otherwise: deleting a workflow's OWN data/out is correct and intended — do NOT change the deletion step or the endpoints' response shape. Both POST /api/workflows/:name/reset-output and POST /api/workflows/reset-output-all call findWorkflowDataOut, so both are fixed by this change.

Restart note for the human: this is src/ (daemon) code — it goes live only after `launchctl kickstart -k gui/$(id -u)/com.ryankrol.localjobs`. (The loop itself doesn't restart the daemon; note it in the worklog.)
