## Overview

On a workflow-run page, the "Input → Output mapping" panel's stage-filter pill bar ("All stages |
stocks-notify | stocks-snapshot | stocks-watch") lists stages alphabetically instead of in the
workflow's real DAG order (snapshot → watch → notify), even though the DAG diagram and the "Member
runs" table on the same page both show the correct order. Root cause: `getWorkflowJobs()` in
`src/db/store.ts` runs `SELECT job_name, depends_on FROM workflow_jobs WHERE workflow_name = ?` with
**no `ORDER BY`**, and SQLite satisfies that query via the table's `PRIMARY KEY (workflow_name,
job_name)` index — which is sorted by `job_name` alphabetically within a workflow, not by
declaration/dependency order. Fix it once, at the source, so every consumer of `getWorkflowJobs()`
gets correctly-ordered members, not just this one filter bar.

## Do

1. In `src/db/store.ts`, change `getWorkflowJobs()` to return its rows in **topological (DAG wave)
   order** instead of raw query order. Reuse the existing, tested Kahn's-algorithm topological sort in
   `buildDag()` (`src/core/dag.ts`) rather than re-implementing a sort — `buildDag` takes
   `{ job: string, dependsOn?: string[] }[]` and returns `{ waves: string[][], ... }` where
   `waves.flat()` is a valid topological order (jobs within the same wave have no ordering constraint
   between them, so their relative order from the flattened array is fine to keep as-is).
   Concretely:
   ```ts
   import { buildDag } from '../core/dag.js';

   export function getWorkflowJobs(name: string): { job_name: string; depends_on: string[] }[] {
     const rows = db.prepare('SELECT job_name, depends_on FROM workflow_jobs WHERE workflow_name = ?')
       .all(name) as { job_name: string; depends_on: string }[];
     const members = rows.map((r) => ({ job_name: r.job_name, depends_on: JSON.parse(r.depends_on) as string[] }));
     // Order topologically (DAG wave order) instead of SQLite's unordered/PK-index-driven row
     // order, so every consumer — e.g. the workflow-run IO panel's stage filter bar
     // (dashboard/app/workflow-runs/[id]/page.tsx) — reflects the workflow's actual dependency
     // order. Falls back to the raw query order if the DAG is malformed (duplicate member /
     // depends_on referencing a non-member) so this function itself never throws — callers that
     // care about a malformed DAG already guard `buildDag` themselves (e.g. `gatesForWorkflow`
     // catches and returns `[]`).
     try {
       const dag = buildDag(members.map((m) => ({ job: m.job_name, dependsOn: m.depends_on })));
       const order = new Map(dag.waves.flat().map((job, i) => [job, i]));
       return [...members].sort((a, b) => (order.get(a.job_name) ?? 0) - (order.get(b.job_name) ?? 0));
     } catch {
       return members;
     }
   }
   ```
   `src/core/dag.ts` only imports from `./types.js` and `src/db/store.ts` does not currently import it
   — confirmed no circular-import risk before writing this task.
