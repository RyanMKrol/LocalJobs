## Do

The `rec-merge` job's **top-up rounds** are opaque in the logs. The first merge pass is well-narrated
(pools the branch suggestions, dedups, and logs each drop with a reason — *"⨯ Barry Lyndon → already
OWNED — dropped"*), but once it enters top-up (*"Top-up round 1/3: have 4/15 — re-prompting branches
for more…"*) there's **no visibility** into what happens — you can't see which branches were
re-prompted, what they returned, what got dropped, or how the count progressed. Make each top-up
round as verbose as the first pass.

- For each top-up round, log: **which branches are being re-prompted** (and with what exclusion
  context — e.g. "excluding the N already-collected / owned / previously-recommended"), **how many
  NEW suggestions each branch returned**, the **dedup + drop decisions** (named items + reason, same
  style as the first pass), and the **running total toward the target** (e.g. "round 1: +6 → 10/15").
- Log **why the loop stops** — target reached, max rounds hit, or a round added nothing ("round 2:
  +0 new — stopping early").
- Reuse the first-pass logging style/helpers so the top-up rounds read consistently. Keep the
  existing summary lines.
- **Logging only — no behaviour change** to the top-up logic itself (built in T162).

## Done when

Each `rec-merge` top-up round narrates which branches were re-prompted, how many new suggestions came
back, the named dedup/drop decisions with reasons, and the running count toward the target — plus the
reason the loop stops — consistent with the first-pass log style; no behavioural change; `npx tsc
--noEmit` and `npm test` are green.
