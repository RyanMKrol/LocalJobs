## Overview
An exception escaping runWorkflowInner permanently wedges a workflow: the workflow_runs row is left stuck status='running' forever, so future manual runs 409, scheduled fires are skipped, and it can't be cancelled — only a daemon restart recovers. The realistic trigger is a SQLITE_BUSY write error under child/daemon contention. Wrap the run body so any escape marks the run failed, aborts in-flight children, and cleans up; and make executeDag treat a rejecting runOne as a normal 'failed' settle rather than a poison pill.

## Do
Two coordinated fixes.

(1) src/core/workflow-executor.ts — runWorkflowInner. Today createWorkflowRun runs at ~line 296 and the AbortController is registered at ~298, but the try (~323) only wraps the repeatUntilStable cycle loop and its finally (~419-421) merely does activeWorkflowRuns.delete(workflowRunId) — it never marks the run failed. So ANY throw between createWorkflowRun and finishWorkflowRun (~443) — e.g. a SQLITE_BUSY thrown by rollUpWorkflowProgress in onSettle, or a rejecting runOne bubbling out of executeDag — escapes with the run row still status='running'. Widen the try to begin immediately AFTER createWorkflowRun (from ~296 onward, so the AbortController registration, the cycle loop, and finalisation are all inside it) and add a catch that: (a) aborts the AbortController so any in-flight member child is hard-killed via the existing SIGTERM→SIGKILL abort path; (b) logs the error loudly to the workflow run log (addWorkflowLog(..., 'error')); (c) calls finishWorkflowRun(workflowRunId, 'failed') so the run row leaves 'running' and the workflow is immediately re-runnable — being careful NOT to double-finish if finishWorkflowRun already ran on the normal path (guard with a boolean flag, or ensure the catch only finishes when the run is still 'running'); (d) returns { workflowRunId } like the normal path. Keep the existing finally that removes the run from activeWorkflowRuns (move/keep it so it always runs). Do NOT swallow the error silently beyond logging + failing the run — the goal is a durable failed state, not a hidden one.

(2) src/core/dag.ts — executeDag. Today the ready-launch block does hooks.runOne(job).then((s) => settle(job, s, false)).finally(() => inflight.delete(job)) (~lines 264-268) with NO .catch, so a REJECTING runOne rejects the whole executeDag promise — a single member's unexpected throw poisons the entire DAG run (nothing else settles; the promise rejection is what escapes runWorkflowInner in bug 1). Add a .catch on the runOne chain that settles that node as 'failed' (settle(job, 'failed', false)) instead of letting the rejection propagate — so a member that throws (rather than resolving 'failed') is treated identically to one that returns 'failed': its dependents cascade-skip, other independent stages keep running, and executeDag resolves normally with that node marked 'failed'. Preserve the existing .finally(() => inflight.delete(job)) ordering. This makes runOne rejection non-fatal at the DAG level; combined with fix (1) it's also caught at the workflow level as defence in depth.

Update any CLAUDE.md prose that describes executeDag's settle behaviour or runWorkflowInner's finalisation if it now contradicts the code (Definition of Done requires docs stay accurate).
